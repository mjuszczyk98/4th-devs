# S01E05 — Limity modeli i architektura produkcyjnego agenta

## Realia produkcyjnych aplikacji generatywnych

Aplikacja z AI to ~80% klasycznego software'u. Pozostałe 20% przenika wszystko: UX, bezpieczeństwo, koszty, skalowalność. Większość problemów ujawnia się dopiero na produkcji, gdy użytkownicy (nieświadomi limitów LLM) wrzucają wielosetstronicowe PDF-y, aktywują setki narzędzi MCP i oczekują nieskończonych konwersacji.

Ograniczenia nie dotyczą tylko modeli — równie dotkliwe są realia biznesowe: rozproszone bazy wiedzy, nieustrukturyzowane procesy manualne, narzędzia bez API, brak dostępu do aktualnych danych. Rzadko mówimy o pełnej automatyzacji — częściej o optymalizacji procesu o kilka-kilkanaście procent, co może być w pełni uzasadnione.

Zasada czerwonej linii: wszędzie gdzie agent ma dostęp do danych i akcji, zakładaj że informacje z jednego źródła trafią do drugiego. Jeśli przypadkowy wyciek jest niedopuszczalny, a akceptacja użytkownika niewystarczająca — **LLM nie powinien być tam wdrożony**.

## Human-in-the-Loop

Agent sterujący logiką aplikacji **będzie** popełniał błędy. System musi umożliwiać samodzielną naprawę (error recovery) albo zaangażować człowieka — przed akcją lub po niej.

**Deterministyczność potwierdzeń.** Decyzję o wyświetleniu confirmation UI podejmuje kod, nigdy model. Model nie wie, że użytkownik zobaczy dialog — po prostu wywołuje tool, system przechwytuje wywołanie. Potwierdzenie musi pokazywać **wszystkie detale** akcji (np. cały szkic emaila, nie sam adres), bo treść może pochodzić z innego narzędzia — to wektor wycieku między źródłami.

**Trust Escalation.** Ręczne zatwierdzanie każdego kroku staje się uciążliwe, stąd trzy poziomy: jednorazowe tak, zaufanie na sesję, odrzucenie. Wymagania produkcyjne:
- Identyfikacja narzędzia bezkolizyjnie jako `server__action` (`resend__send`), nie sama nazwa.
- **Auto-revoke** zaufania przy każdej zmianie nazwy, opisu lub schematu narzędzia — krytyczne dla MCP, gdzie serwer może zmienić interfejs bez wiedzy użytkownika.
- Komendy `untrust` i `clear` (czyści jednocześnie konwersację i trust).

**Trzy warstwy obrony**: statyczna whitelist (blokuje niedozwolonych odbiorców, obsługuje wzorce domen `@example.com`), dynamiczne confirmation UI z pełnym podglądem, oraz trust escalation jako warstwa UX zmniejszająca friction.

## Limity kontekstu

**Jawne limity.** Okno to input + output razem. GPT-5.2 z oknem 400k i max output 128k daje realnie 272k na input — nie 400k. Estymacja: `chars / 4`, bufor ~20%, doprecyzowanie po odpowiedzi z `usage`. Kompresję uruchamiać **wcześnie**, już przy ~30% zużycia, z konfiguracją per-model (próg pruningu, target po kompresji, min zachowanych tur).

**Strategia 3-etapowa:**

1. **Truncation** — outputy narzędzi >10k znaków obcinane middle-cut (początek: nagłówki/struktura, koniec: wyniki/podsumowania). Lepsze niż right-cut.
2. **Turn dropping** — konwersacja dzielona na tury (user message + wszystko do następnego); zawsze zachowywana pierwsza tura (setup) i ostatnie N; usuwane od najstarszych.
3. **LLM Summarization** — usunięte tury podsumowywane przez model, wstawiane jako `[Context Summary]` na początku. Summary jest **inkrementalne**: nowy dropped content merge'owany z poprzednim. Sugerowana struktura: Goal / Progress (Done/In Progress/Blocked) / Key Decisions / Critical Context.

DB przechowuje pełną historię (audit trail); model widzi tylko to, co mieści się w oknie.

**Niejawne limity.** Gwarancja struktury ≠ gwarancja wartości — poprawny JSON z fałszywymi danymi nie zostanie wykryty przez system. Modele Flash potrafią w pełni halucynować treść strony WWW na podstawie samego URL, podczas gdy mocniejsze modele przyznają brak dostępu. Model może "domyślić się" brakujących parametrów (email z nazwiska) zamiast dopytać.

Redukcja ryzyka: informowanie modelu o jego ograniczeniach (czy web search jest aktywny), instrukcje zachowania przy braku danych ("dopytaj" zamiast "zgadnij"), zmniejszenie złożoności zadania, **zmniejszenie objętości kontekstu** (za duży kontekst rozprasza uwagę). Halucynacje często wynikają z błędów aplikacji: źle wczytane instrukcje, model informowany o narzędziach do których nie ma dostępu — trudne do zauważenia w systemach wieloagentowych. Stąd obowiązek: **zapisywać i monitorować wszystkie zdarzenia**, łącznie z instrukcjami systemowymi i ich zmianami.

## Wydajność i UX

Wydajność modeli jest niska — adresujemy ją na poziomie architektury, nie modelu.

- **Heartbeat** — wgląd w bieżące kroki agenta i w budowany rezultat (nie loading spinner). Nie poprawia wydajności realnej, zmienia postrzeganą.
- **Wielowątkowość** — użytkownik musi móc wysłać wiadomość do kolejki i otworzyć nowy wątek bez czekania. Stan UI oddzielony od akcji backendu.
- **Przetwarzanie w tle** — zadanie trwa minutami, użytkownik może zamknąć przeglądarkę.
- **Wznawianie** — agent może oczekiwać na więcej niż jedną rzecz jednocześnie; event-driven to fundament.
- **Zasada "czy AI tu jest niezbędne?"** — jeśli da się kodem, zrób kodem.

Gdy to nie wystarczy: fine-tuning mniejszych modeli lub destylacja większych. Rzadko stosowane, bo modele Flash/Haiku zwykle wystarczają.

## Koszty

- LRM (Large Reasoning Models) mają niższą cenę/token, ale generują znacznie więcej tokenów (thinking tokens).
- Proporcja nawet **1:50** — jedna wiadomość użytkownika : pięćdziesiąt zapytań do modelu (multi-turn agent loop).
- Tańszy model ≠ tańsze rozwiązanie: GPT-4.1-nano okazał się droższy niż GPT-4.1-mini, bo wymagał więcej kroków (paper: *Solving A Million-step LLM Task With Zero Errors*).
- AI ambientowe: małe zadania w krótkich interwałach sumują się do dużych liczb.

Podstawa optymalizacji to **obserwowalność** (Langfuse, Confident AI), nie zgadywanie. Mała grupa testerów pozwala oszacować skalę przed wdrożeniem. OpenRouter programistycznie zarządza kluczami per user z indywidualnymi limitami wydatków.

## Rate limiting i moderacja

Limity providera (RPM/TPM) są widoczne w nagłówkach odpowiedzi — reaguj na zbliżający się limit, informując użytkownika o wydłużonym oczekiwaniu. **Każdy endpoint AI musi mieć własny rate limiting per user** (zalogowany i anonimowy), niezależnie od limitów providera.

**Niestosowanie OpenAI Moderation API może prowadzić do zablokowania konta organizacji** — nie klucza, całego konta. Zagrożenia: dane poufne trafiające do modelu, treści nielegalne, nadużycia zasobów, nieintencjonalne pętle, wczytanie strony WWW z prompt injection.

Filtrowanie wykracza poza prawo — obejmuje sam zakres systemu. Weryfikacja: prompt zwraca ustrukturyzowaną odpowiedź o wejściu, a **kod** decyduje o dalszym przebiegu. Moderacja może też fałszywie blokować legalne akcje (generowanie reklamy z interfejsem Messengera blokowane przez Gemini).

Niewłaściwych zachowań modelu nie da się wyeliminować — aplikacja musi jasno o tym informować, a produkt musi być zabezpieczony prawnie (regulamin, polityka prywatności, umowy z dostawcami).

## Fundamentalne zasady architektoniczne

**Wspólny interfejs providerów.** Format `provider:model` (`openai:gpt-5.4`, `gemini:gemini-3-pro-preview`) rozdziela wybór modelu od implementacji. Adapter per provider tłumaczy wspólny interfejs (wzorowany na Responses API) na natywne API. Ten sam agent loop działa z dowolnym providerem. OpenRouter wystarcza jako uproszczenie, ale **nie wspiera wszystkich funkcji** — providerzy coraz częściej wprowadzają API-specific features blokujące pełną translację. W systemach wieloagentowych każdy agent może używać innego providera.

**Brak frameworków AI.** Langchain, CrewAI — dynamiczny rozwój API sprawia, że frameworki szybko stają się obciążeniem.

**Niezależność.** Ograniczanie natywnych funkcji API, unikanie platform z lock-in (wektorowe DB bez eksportu). Rozwiązanie dzisiejsze zostanie z nami znacznie krócej niż w klasycznych aplikacjach.

**Event-driven jako fundament.** To nie feature — to warunek możliwości monitorowania, kompresji kontekstu, moderacji w locie, heartbeat i niezależności komponentów. Każdy event niesie **pełny kontekst** (self-contained) — subscriber nigdy nie sięga do runnera po dodatkowe dane. Każde zdarzenie zawiera `EventContext` z `traceId`, `sessionId`, `agentId`, `rootAgentId`, `depth` — umożliwia korelację w systemach wieloagentowych. Monitorowanie dwupoziomowe: logi aplikacji + śledzenie agenta (Langfuse traces z pełnym input/output na każdym kroku).

## Non-blocking execution — wzorzec `waiting/deliver`

Agent nie musi zakończyć pracy w jednym request-response cycle. Maszyna stanów: `pending → running → waiting → running → completed` (lub `failed`/`cancelled`).

Przepływ: agent napotyka tool wymagający czekania (human input, delegacja, zewnętrzne API), przechodzi w `waiting` z listą `waitingFor[]`, HTTP zwraca **202 Accepted** (semantyczne rozróżnienie "zrobione" vs "czekam"), zewnętrzny system dostarcza wynik przez `POST /agents/:agentId/deliver`, agent wznawia pracę od momentu zatrzymania.

Agent może czekać na **wiele rzeczy jednocześnie** — jak `Promise.all`. Status wraca do `running` dopiero gdy wszystkie `waitingFor` zostaną spełnione. Gdy child-agent kończy pracę i ma `parentId` + `sourceCallId`, wynik automatycznie propaguje się rekurencyjnie do korzenia.

## Typy narzędzi — model nie wie

Cztery typy po stronie runnera: `sync`, `async`, `agent`, `human`. Model widzi **tylko** definicje funkcji z `description` — typ istnieje wyłącznie po stronie infrastruktury.

| Typ | Zachowanie | Przykład |
|-----|-----------|----------|
| `sync` | Natychmiastowy wynik | `calculator`, `send_message` |
| `async` | Background, wynik przez `deliver` | zadania tła |
| `agent` | Spawn child-agenta | `delegate` |
| `human` | Zamrożenie, czeka na człowieka | `ask_user` |

Handler typu `human` **nic nie robi** — runner rozpoznaje typ i dodaje wpis do `waitingFor` zamiast wywoływać handler. Narzędzia MCP rozpoznawane po konwencji `server__tool`. Celowy design: model wie *co* narzędzie robi, nie *jak*.

## Multi-agent — delegacja i komunikacja

**Agent jako plik markdown** z frontmatter (`name`, `model`, `tools`) i treścią jako system prompt. Zero kodu do stworzenia agenta; pliki czytane z dysku na każde wywołanie — hot-reload bez restartu. Narzędzia rozwiązywane dynamicznie z trzech źródeł: registry, MCP, native.

**Hierarchia.** Delegacja przez tool `delegate`. Runner sprawdza `MAX_AGENT_DEPTH = 5` (guard przed nieskończoną rekurencją), ładuje template child-agenta, wstawia zadanie od rodzica jako user message, rekurencyjnie uruchamia `runAgent()`, a wynik wraca jako `function_call_output` w historii rodzica. Z perspektywy rodzica delegacja wygląda jak zwykłe wywołanie narzędzia — nie wie, że dziecko uruchomiło własny web search.

**Dwa modele komunikacji:**
- `delegate` — synchroniczny request-response, parent czeka na child, blokujący.
- `send_message` — asynchroniczny fire-and-forget, wpisuje system message do historii targeta, non-blocking, **nie wznawia** agenta. Jeśli target jest `running` — zobaczy wiadomość na następnym turnie; jeśli `completed` — nikt jej nie przeczyta. To "karteczka na biurku", nie "pukanie do drzwi".

## Bezpieczeństwo produkcyjne

- Bearer auth z kluczami hashowanymi SHA-256, nigdy plaintext.
- Rate limiting per user z cleanup stale windows.
- Graceful shutdown z hard timeout.
- `AbortSignal` propagation przez cały stack — od HTTP przez runner do provider calls.
- HTTP 202 vs 200 jako semantyczny sygnał stanu.
- Moderacja na entry point.

## Deployment

VPS (DigitalOcean / Mikr.us) → Ubuntu z SSH key auth (wyłączyć hasła) → git + node + nginx + ufw → DNS A na Cloudflare → TLS przez `certbot --nginx` → GitHub Actions self-hosted runner → reverse proxy nginx na `127.0.0.1:3000` → secrets w ustawieniach repo → workflow `.yml` triggerowany na push do main. Ostrzeżenie: wyłączyć udostępnianie ekranu przy dodawaniu kluczy API.

## Przykłady kodu — progresja

### `01_05_confirmation` — REPL agent z potwierdzaniem

Separacja walidacji od LLM. System prompt zawiera `"DO NOT ask for confirmation - the system handles that automatically"` — model decyduje **co** zrobić, kod decyduje **czy** pozwolić. Confirmation callback jako dependency injection do agent loop. Walidacja whitelisty `isEmailAllowed()` obsługuje dokładne adresy i wzorce domen, jest po stronie kodu i niezależna od modelu. Hybrydowa lista narzędzi: MCP (pliki) + native (email) mergowane w jeden array, dispatch przez `isNativeTool()`.

Trusted tools: `Set` per sesja, auto-approve jeśli narzędzie w secie, reset przez `clear`/`untrust`. Zaufanie jest **sesyjne, nie permanentne**.

### `01_05_agent` — production-grade agent API

Delta względem `01_05_confirmation`:

**Persystencja przez Repository pattern** — `UserRepository`, `SessionRepository`, `AgentRepository`, `ItemRepository` z identycznym interfejsem dla in-memory i SQLite (Drizzle). Swap backendu bez zmiany logiki.

**Domain model jako czyste funkcje** — `startAgent()`, `completeAgent()`, `failAgent()` zwracają nowy obiekt, nigdy nie mutują. Result type `TransitionResult = { ok: true; agent } | { ok: false; error }` zamiast wyjątków.

**Items jako polimorficzny log** — `MessageItem | FunctionCallItem | FunctionCallOutputItem | ReasoningItem` z type guardami. Sequence number per agent zapewnia porządek.

**Provider abstraction** — `Provider` interface z `generate()` i `stream()`, `ProviderInputItem`/`ProviderOutputItem` jako wspólny format, `mapItemsToProviderInput()` tłumaczy domain items na format providera. Każdy provider ma własny adapter. OpenRouter automatycznie dokleja `:online` gdy agent ma tool `web_search`.

**Streaming** — `executeTurnStream()` jako `AsyncGenerator<ProviderStreamEvent, TurnResult>`: yielduje eventy w trakcie, zwraca finalny `TurnResult` na końcu. SSE na poziomie HTTP. Oba tryby (`runAgent`, `runAgentStream`) dzielą tę samą logikę pętli; stream kończy się gdy agent wejdzie w `waiting`.

**Langfuse subscriber** — mapuje eventy na traces: `agent.started` → Span (z nestingiem dla child-agentów), `generation.completed` → Generation z `input/output/usage`. Eventy niosą `startTime` + `durationMs` — timeline odtwarzany nawet przy opóźnionym przetwarzaniu. `agentObsMap` + `agentSpanCtxMap` do korelacji parent-child.

**MCP Manager** — stdio i streamable HTTP transport, OAuth 2.1 flow z `finishAuth()`, namespacing `server__toolName`, timeout 30s per wywołanie przez `AbortSignal.any([signal, timeout])`.

**Pruning w akcji** — `needsPruning()` → `pruneConversation()` → jeśli włączone, `generateSummary(provider, model, droppedItems, session.summary)` aktualizuje session summary kaskadowo, a przy każdym turn `[Context Summary]` wstrzykiwany jako pierwsza systemowa wiadomość.
