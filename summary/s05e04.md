# S05E04 — Produkcja

## MVP → produkcja w gen-AI

W klasycznych aplikacjach produkcja różni się skalą i CI/CD. W gen-AI różni się **fundamentalnie** — detale pomijane w MVP determinują użyteczność i bezpieczeństwo na produkcji. Budowanie nawet prostych narzędzi AI na własne potrzeby może zająć kilkanaście minut, ale pierwsze wdrożenie wielodostępnego systemu ujawnia przepaść. Modele kodują szybko, ale **pomijają** to, czego nie znają — różnicę robi to, co zostanie pominięte. Umiejętność rozpoznawania takich detali bierze się wyłącznie z doświadczenia.

## Delikatna struktura konwersacji

Trzy pozornie drobne problemy, które łącznie definiują użyteczność produktu:

**Usuwanie wiadomości.** ChatGPT, Claude, Cursor nie oferują delete. Usunięcie ze środka łamie invarianty API — Anthropic/Gemini wymagają user message jako ostatniej, inaczej zwracają błąd. Usuwanie zaburza historię, zarządzanie kontekstem i może uniemożliwić dalszą konwersację. Zamiast delete: **rozgałęzianie** (branching) i **przywracanie** do punktu.

**Edycja wiadomości agenta.** Agent posługuje się narzędziami — edycja jego odpowiedzi otwiera ścieżkę do **many-shot jailbreak**: użytkownik może zasugerować, że agent wykonał coś, czego nie powinien. Rozwiązanie: agent messages nieedytowalne w UI.

**Długi tekst w prompcie.** Textarea z wielkim tekstem zabija wydajność edycji. Rozwiązanie: detekcja rozmiaru → automatyczna konwersja do file attachment. W rozmowach z LLM duże treści wkleja się regularnie.

## Halucynacje w niszowych domenach

Whisper halucynuje na ciszę ("Thanks for watching!", "Subtitles by the Amara.org community") — artefakt treningu na napisach filmowych. Mieszanie języków w jednym zdaniu ("Turn left and walk straight down **Piotrkowska** Street"). Nie jest to błąd naprawialny promptem — wymaga świadomego projektowania pipeline'u audio.

## Narzędzia wspomniane, nieaktywne

Samo wspomnienie o możliwości w system prompt, bez aktywacji narzędzia, wystarczy by model **symulował** jego użycie. Ryzyko: niejawne halucynacje funkcjonalne — model opisuje wynik akcji, która nigdy nie nastąpiła. Mitygacja: `isAvailable(context)` per-run — każde narzędzie ma callback sprawdzający dostępność w kontekście konkretnego runa.

## Agent Runtime — architektura produkcyjna

Wiadomość użytkownika traktowana jako **zadanie** (job), nie event. Job → Run (próba wykonania). Oddzielenie "co zrobić" od "kto próbuje" umożliwia retry, delegację i monitoring.

### Cykl życia

`Request → Persist → Queue → Schedule → Claim → Turn Loop → [Done | Delegate | Wait] → Persist → Events`

1. **Żądanie** — HTTP z uwierzytelnieniem i walidacją
2. **Inicjalizacja** — wpisy w DB: sesja, wątek, wiadomość, job, run. Wyłącznie zapis danych
3. **Kolejka** — job `queued`, run `pending`. Gotowe do wznowienia po błędzie
4. **Scheduler** — cykliczne odpytywanie o pary job/run wymagające działania
5. **Claim** — worker rezerwuje run z czasem wygaśnięcia + heartbeat
6. **Pętla agenta** — model nie wskaże narzędzi → odpowiedź zapisana, run zakończony. W przeciwnym razie → wykonanie narzędzi + kolejna iteracja
7. **Delegacja** — `delegate_to_agent` tworzy child run z własnym agentem i kontekstem
8. **Dostarczenie wyniku** — child zakończony → scheduler dostarcza wynik do parenta
9. **Zapis** — items (reasoning, wywołania narzędzi, wyniki, delegowanie)
10. **Odzyskiwanie** — scheduler sprawdza "martwe" runy co cykl

### Readiness engine

Deterministyczny scheduler odpytujący DB o pary job/run. **Stała kolejność priorytetów:**

1. Dostarczenie wyników od child agents
2. Wznowienia po wait / timeout
3. Odzyskiwanie po crashu (stale recovery)
4. Nowe zadania

Root runs mają priorytet nad child runs — zagnieżdżone delegacje nie zagładzą nowych interakcji. Sortowanie wewnątrz grup: priorytet → queuedAt → runCreatedAt → runId.

### Durable-first execution

**Persist first, execute second.** Komenda zapisuje stan do DB *zanim* wywoła model. Konsekwencje: model fail → work queued (nie lost), crash recovery z durable state, route handler zwraca persistent data, nie ephemeral output.

### Lease-based crash recovery

Worker claims run → heartbeat co `leaseTtlMs/3` → crash = heartbeat stopuje → claim expires → scheduler detect stale → requeue z exponential backoff (`baseDelayMs * 2^(count-2)`) → po `maxStaleRecoveries` → permanent fail. `staleRecoveryCount` inkrementuje per run, zapobiega rapid re-crash loops.

### Delegacja jako tool call

`delegate_to_agent` — natywne narzędzie. LLM sam decyduje o delegacji. Mechanika w jednej transakcji: child run + child job + prywatny wątek (`threadId: null`) + dependency edge + linkowanie plików + handoff item + eventy. Parent → `waiting` → child complete → readiness engine delivers result → parent resumes. Zagnieżdżenie wielopoziomowe (Alice → Tony → Claire). Child run jest prywatny — tylko root run pisze do widocznego threada. Delegacja jest narzędziem, więc infrastruktura orkiestracji = ta sama co dla innych tool calls.

### Event sourcing z outbox

Domain events → `event_outbox` (SQLite) → dedykowane workery rozsyłają do SSE (realtime), projections (read models), Langfuse (traces). Rozdziela writes od side effects, gwarantuje at-least-once delivery.

### Narzędzia z wynikiem `waiting`

Narzędzie nie musi zwrócić natychmiastowego wyniku — może zwrócić `{ kind: 'waiting', wait: {...} }`, zawieszając run. Obsługuje asynchroniczne interakcje: czekanie na człowieka, MCP, upload.

### Współdzielenie root job między turami

`reopenThreadRootJobForNewMessage` — ten sam root job reopenowany dla kolejnych wiadomości z nowym currentRunId. Job = jednostka szeregowalna, nie jednorazowa.

## Model danych

Pełny schema: 23 migracje SQLite z WAL mode, foreign keys, busy timeout. Kluczowe encje:

| Encja | Rola |
|-------|------|
| **Tenant** | Grupowanie kont, workspace'ów, agentów, plików, konfiguracji MCP |
| **Account** | Konto użytkownika z personalizacją i uprawnieniami |
| **Workspace** | Przestrzeń zasobów — system plików, ustawienia, interakcje |
| **Session** | Grupowanie wątków ze współdzieleniem zasobów |
| **Thread** | Interakcja z agentem, może mieć wątek nadrzędny (rozgałęzienia) |
| **Job** | Zadanie agenta ze statusem — analogia do Issue w Linear/Jira |
| **Run** | Aktywność agenta ze stanem + snapshot konfiguracji modelu |
| **Item** | Element aktywności: reasoning, tool calls, wyniki |
| **Dependency** | Mechanizm oczekiwań i blokad |
| **Execution** | Historia wywołań narzędzi — argumenty, wyniki, błędy |

Foreign keys zawsze obejmują `(id, tenant_id)` — gwarancja na poziomie DB, że wiersz z jednego tenanta nigdy nie odwoła się do wiersza z innego. Agenci mają "rewizje" z ustawieniami w chwili zapytania + checksum SHA256.

## Clean Architecture w agentic runtime

Rygorystyczny podział: `app/` (konfiguracja), `adapters/` (HTTP, AI, MCP), `application/` (komendy, runtime), `domain/` (encje, repozytoria), `db/` (Drizzle ORM), `shared/` (typy). Adaptery zależą od domeny, nie odwrotnie — wymiana komponentów infrastruktury bez dotykania logiki biznesowej.

## Bezpieczeństwo

| Wektor | Mitigacja |
|--------|-----------|
| Many-shot jailbreak przez edycję | Agent messages nieedytowalne w UI |
| Narzędzie wspomniane, nieaktywne | `isAvailable(context)` per-run |
| MCP STDIO na serwerze | Dopuszczalne **tylko** na desktop, proces lokalny użytkownika |
| Timing attack na hasła | scrypt + timingSafeEqual |
| Sekrety OAuth w DB | AES-256-GCM encryption at rest |
| Double execution na retry | Idempotency keys z progress snapshots |

Brak skutecznej obrony przed prompt injection — świadomość tego faktu jest częścią architektury.

## Frontend produkcyjny

### Streaming z rAF batching

SSE eventy nie triggerują re-render indywidualnie. `requestAnimationFrame` batch = jedno DOM update na klatkę. Buforowanie: pending events → flush na rAF (+ setTimeout(100) fallback).

### Inkrementalny rendering Markdown

Treść nie re-renderowana w całości. `committedSegments` + `liveTail` = tylko ogon parsowany na deltę. Zatwierdzone segmenty zostają w DOM bez dotykania. Podział frozen/live — fundament wydajnego streaming renderowania.

### Trójwarstwowa projekcja stanu

`durableMessages` (backend confirmed) + `optimisticMessages` (pending user) + `liveAssistantMessage` (streaming) → scalane w projected view. Gdy serwer zwróci trwałą wiadomość, optimistic usuwana, live zastępowane przez durable. Eliminuje flicker przy stream→durable transition.

Stable UI keys: store mapuje lokalne ID na prawdziwe przez `stableUiKeyByMessageId` — zapobiega remounting komponentu i utracie stanu scroll.

### Typewriter effect z gatingiem

Bloki odkrywane po jednym z konfigurowalną prędkością. Buforowanie: nowe bloki z backendu czekają aż obecny się wyświetli. Durable handoff: gdy stream się skończy, typewriter pomija replay.

### Projected sync batching

Batch/transaction system dla state updates. `projectedSyncBatchDepth` liczy zagnieżdżenie, flush na depth=0. Eliminuje redundant DOM updates przy wielu zmianach w jednym ticku.

### Virtual scrolling

Wiadomości chunkowane po 12, overscan ±3 chunks, `ResizeObserver` do pomiaru rzeczywistych wysokości, `@chenglou/pretext` do pre-layout text measurement — oblicza ile linii zajmie tekst bez renderowania DOM.

### SSE z reconnect

Exponential backoff (250ms base, 2s max, 6 attempts), event ID tracking dla cursor-based resume, abort signal forwarding.

### Kompozytor

Tiptap v3 z custom extensions: `@` mention agentów, `#` mention plików, image paste, inline markdown. Command palette z provider-based architekturą — każdy feature dostarcza itemy. Shortcut Layer Stack — gdy palette otwarty, skróty globalne dezaktywowane.

### Submit button jako state machine

4 stany (idle/active/cancelling/finishing) przez CSS grid stacking. Wszystkie labelki na tej samej komórce, widoczność przez opacity.

### Flash-less theme switching

Theme aplikowany synchronicznie w `<head>` przed CSS — eliminuje FOUC. CSS containment (`contain: layout style`) izoluje layout do własnej warstwy GPU.

## Sugestie ogólne

- **Jakość > szybkość** — gdy każdy może generować kod, wyróżnikiem jest dbałość o detale
- **Użytkownik nie wie czym jest LLM** — projektuj dla minimalnej wiedzy. Skuteczny agent jest niewidoczny
- **Agent bez czatu** — reagowanie na akcje (upload, klik) daje większą kontrolę niż open-ended chat. Czat sensowny przy złożonych akcjach lub skali
- **Budowanie dla agentów** — aplikacje dostępne nie tylko dla ludzi, ale i dla agentów działających w ich imieniu
- **Kwestionuj wzorce** — po poznaniu zasad, łam je celowo. Koszt eksperymentu jest dziś minimalny
- **Wszyscy eksplorujemy** — przekonania o AI aktualizować bardzo często lub zakładać, że mogą być niepoprawne

## Przykłady kodu

### Readiness Engine — deterministyczny scheduler (05_04_api)

**Delta:** pure function — snapshot stanu z DB → deterministyczna decyzja. Stała kolejność priorytetów.

```typescript
// 1. deliver_resolved_child_result
// 2. recover_timed_out_wait
// 3. requeue_waiting_job
// 4. resume_waiting_run
// 5. requeue_stale_running_run (crash recovery)
// 6. execute_pending_run (root przed child)
// Sort: priorytet → queuedAt → runCreatedAt → runId
```

### Claim & Heartbeat — fault tolerance (05_04_api)

**Delta:** lease-based crash recovery — martwy proces nie blokuje zadań. Exponential backoff na stale recovery.

```typescript
// claim: runId → workerId, acquiredAt, expiresAt, renewedAt
// heartbeat co leaseTtlMs/3
// crash → heartbeat wygasa → scheduler detect stale
// → requeue: staleRecoveryBaseDelayMs * 2^(count-2)
// Limit: maxStaleRecoveries (5), po czym run = failed
```

### Delegation — child run w jednej transakcji (05_04_api)

**Delta:** delegacja jako transakcja tworząca pełną infrastrukturę child runa — prywatny wątek, własne narzędzia, zależności, eventy.

```typescript
// delegation-service.ts — w jednej transakcji:
// 1. Walidacja parent revision ma alias subagenta
// 2. Pobranie child agent + aktywnej rewizji
// 3. Utworzenie child Run (threadId=null — prywatny)
// 4. Job dependency edge + run dependency
// 5. Linkowanie plików parent → child
// 6. Handoff item + task item w items childa
// 7. Eventy: run.created, child_run.created, delegation.started
```

### Event Sourcing + Outbox (05_04_api)

**Delta:** każda zmiana domeny emituje event zapisywany w tej samej transakcji co dane. Background worker dostarcza asynchronicznie.

```typescript
// domain_events — event_no, aggregateId, aggregateType, payload
// event_outbox — asynchroniczna dostawa
```

### Result Monad + Branded Types (05_04_api)

**Delta:** domena zwraca `Result` zamiast rzucać wyjątki. Każdy ID ma swój typ — kompilator nie pozwala pomylić RunId z JobId.

```typescript
export type Result<TValue, TError> =
  | { ok: true; value: TValue }
  | { ok: false; error: TError }

export type Brand<TValue, TName extends string> = TValue & { readonly __brand: TName }
export type RunId = Brand<string, 'RunId'>
export const createPrefixedId = <P extends string>(prefix: P) =>
  `${prefix}_${crypto.randomUUID().replace(/-/g, '')}`
```

### rAF Batching + Event Sourcing po stronie klienta (05_04_ui)

**Delta:** SSE events buforowane i flushowane na requestAnimationFrame — max jeden update na klatkę (~16ms). Client-side event sourcing.

```typescript
export const withRafBatching = <T>(onBatch: (batch: T[]) => void) => {
  let pending: T[] = []
  let scheduled = false
  const flush = () => {
    scheduled = false
    if (pending.length === 0) return
    const batch = pending
    pending = []
    onBatch(batch)
  }
  return {
    push(item) {
      pending.push(item)
      if (scheduled) return
      scheduled = true
      requestAnimationFrame(flush)
    },
  }
}

// applyEvent — client-side event sourcing
// stream.delta | tool.called | tool.completed | reasoning.summary.delta
```

### Inkrementalny Markdown — frozen/live segmenty (05_04_ui)

**Delta:** committedSegments + liveTail = re-render tylko ogona. Zatwierdzone segmenty w DOM bez dotykania.

```typescript
const syncIncrementalMarkdownView = (current, { content, streaming }) => {
  const delta = content.slice(current.processedContent.length)
  const candidateTail = current.liveTail + delta
  const blocks = parseMarkdownIntoBlocks(candidateTail)
  const { finalizedSources, liveTail } = splitBlocks(blocks, streaming)
  return {
    committedSegments: current.committedSegments.concat(
      toSegments(blockId, current.nextSegmentIndex, finalizedSources)
    ),
    liveTail,
    processedContent: content,
  }
}
```
