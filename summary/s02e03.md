# S02E03 — Dokumenty i pamięć długoterminowa jako narzędzia

## Baza wiedzy: budowana dla agentów vs podłączana do nich

Agent przeszukujący istniejące dokumenty (pisane dla ludzi) przez chunki i embeddingi — **ma szansę** trafić na właściwe źródła, ale nie ma gwarancji. Pełny kontekst może wymagać dziesiątek dokumentów, a trafiony dokument często wygląda na kompletny, ale wymaga pogłębienia w powiązanych źródłach. Prompt nie rozwiąże tego problemu.

Odwrócone podejście: baza wiedzy **zbudowana dla agentów**. Agent nie szuka — wie, gdzie patrzeć. Nie generyczna instrukcja "przeszukaj bazę", lecz "instrukcje obsługi narzędzi znajdziesz w `./workflows`". Dokument prowadzi do kolejnego dokumentu — agent podąża za wskazówkami jak za importami w kodzie.

Meta-rozróżnienie: **łączenie ze źródłem** (klasyczny RAG — chunki + embeddingi, model widzi fragmenty pozbawione powiązań) vs **nauka ze źródła** (dane prezentowane tak, aby agent po nich nawigował — odnośniki wewnątrz dokumentów, struktura katalogów, importy w kodzie). Kod źródłowy jest naturalnym przykładem drugiego — coding agenty skutecznie nawigują po projektach korzystając tylko ze ścieżki + podpowiedzi w plikach. Poza kodem rzadkość — wyjątki: Wikipedia (bogate linkowanie), Digital Garden (Obsidian, Roam, Logseq).

Konsekwencje budowanej KB: eliminacja problemu odnajdywania (agent widzi tylko to, czego potrzebuje), brak problemów z formatami/rozmiarem (dokumenty generowane pod agenta), kompozycja (zmiana jednego wpisu propaguje się przez proces), wymaga dyscypliny w utrzymaniu struktury.

## Cztery wymiary nawigacji po wiedzy

Kod źródłowy naturalnie posiada te cechy. Dokumenty biznesowe — prawie nigdy.

| Wymiar | Opis | W kodzie | W dokumentach |
|--------|------|----------|---------------|
| **Perspektywa** | Widok z lotu ptaka | `ls`, struktura katalogów | Rzadko istnieje |
| **Nawigacja** | Przeszukiwanie nazw i treści | `grep`, search | Wymaga RAG |
| **Powiązania** | Odnośniki między zasobami | Importy, ścieżki | Prawie brak |
| **Szczegóły** | Oryginalna treść | Odczyt pliku | Odczyt dokumentu |

Ekspozycja kontekstu opiera się na **informacjach wewnątrz dokumentów** (linki, referencje), nie na zewnętrznym indeksie. Wyświetlanie drzewa katalogów pomaga tylko gdy jest stałe i agent na nim operuje.

## Observational Memory — kompresja zamiast wyszukiwania

Podejście Mastra.ai — ~95% na LongMemEval z gpt-5-mini — bez wektorów, bez grafów, bez wyszukiwania semantycznego. Zamiast rozwiązywać problemy wyszukiwania, eliminuje je przez architekturę pamięci.

Okno kontekstowe podzielone na dwa bloki:
- **Observations block** — skompresowane obserwacje z przeszłych interakcji
- **Raw messages block** — nowe, nieprzetworzone wiadomości

Dwa agenty zarządzają pamięcią:
- **Observer** (próg: 30k tokenów raw messages) — kompresuje wiadomości do zwięzłych obserwacji, czyści raw block, appenduje do observations
- **Reflector** (próg: 40k–60k tokenów obserwacji) — garbage collection obserwacji, usuwa nieistotne

Dziennik przetrwa pojedynczą sesję — nieskompresowane wiadomości + najnowsza wersja dziennika przechodzą do nowej sesji. Koszt: stopniowa utrata szczegółów (kompresja stratna), ale wyniki benchmarkowe pokazują akceptowalność dla pamięci konwersacyjnej.

Obserwacje to sformatowany tekst z **temporal model** (trzy daty per obserwacja: data obserwacji, referencyjna, relatywna), **priorytetyzacją** (🔴 critical, 🟡 potentially important, 🟢 informational), grupowaniem po dacie i inline timestamps.

Filozofia: "Text is the universal interface" — LLM lepiej operuje na tekście niż na grafach/wektorach.

### Prompt caching jako ukryty killer feature OM

Wiadomości appendują do stałego prefixu → **full cache hit** przez większość czasu. Po Observer: nowe obserwacje appendują do istniejących → **partial cache hit**. Cache invalidacja tylko przy Reflector (rzadko). Grafowy RAG generuje nowy kontekst per zapytanie — nie korzysta z prompt cache.

## Graph RAG — hybrydowa pamięć strukturalna

Neo4j — grafy właściwości: wierzchołki z etykietami i właściwościami + krawędzie z kierunkiem, typem i właściwościami. Każda krawędź łączy dokładnie dwa wierzchołki. Łączy: indeksowanie treści + wyszukiwanie pełnotekstowe (BM25) + semantyczne (vector) + nawigację po relacjach.

Uzasadnione zastosowanie: **wielopoziomowe powiązania** między informacjami rozproszonymi w wielu dokumentach. Koszt (infrastruktura, złożoność, czas odpowiedzi) sprawia, że nie jest to domyślny wybór.

### OM vs Graph RAG

| Kryterium | Observational Memory | Graph RAG |
|-----------|---------------------|-----------|
| Infrastruktura | Zero (tylko LLM) | Neo4j + embedding API + LLM |
| Powiązania | Implicit (w tekście) | Explicit (krawędzie) |
| Skalowalność wiedzy | Ograniczona (kompresja = utrata) | Nieograniczona (graf rośnie) |
| Prompt caching | Natywne | Brak |
| Pamięć konwersacyjna | Doskonała | Nie do tego służy |
| Multi-document navigation | Słaba | Doskonała |
| Temporal reasoning | Wbudowane | Brak |

Nie wykluczają się — OM dla pamięci konwersacyjnej, graf dla współdzielonej bazy wiedzy.

## Dekompozycja na sesje

Złożone procesy dekomponuje się na osobne sesje agentów — ten sam agent, ta sama instrukcja, różne źródła. Wyniki do współdzielonego systemu plików. Przykład: Researcher (3 sesje gromadzenia) → Writer (synteza) → Sender (dystrybucja). Instrukcje określają zasady zapisu, agenci niezależnie produkują spójny wynik.

Agent skupiony na jednym zadaniu = wyższa jakość. Optymalne kosztowo — płacimy za wielokrotne wczytanie instrukcji, nie za ogromny kontekst. Naturalna kompozycja — każdy etap niezależny, łatwy do debugowania i wymiany.

**Krytyczna zasada:** agent generujący treść do wysyłki musi mieć uprawnienia tylko do tworzenia szkicu. Halucynacje + prompt injection = ryzyko przekierowania odbiorców.

## Deep Research / Deep Action

Wzorzec iteracyjnego pogłębiania: doprecyzowanie zapytania → parafraza → dekompozycja → szukaj → analizuj → identyfikuj braki → kolejna iteracja → synteza. Możliwe wzbogacenie: wstępne przeszukiwanie do zawężenia kontekstu przed właściwym zapytaniem.

Zmiana nazwy z "deep research" na **"deep action"** rozszerza zastosowania — audyty, generowanie kodu, analiza logów. Każdy proces wymagający eksploracji przez pogłębianie zapytań i niebędący natychmiastowy.

## Przykłady kodu — Graph RAG Agent

Przykład `02_03_graph_agents` realizuje pełen pipeline: indeksowanie → ekstrakcja encji/relacji → hybrydowe wyszukiwanie → eksploracja grafu przez agenta z narzędziami.

### Architektura grafu (schema)

Trzy typy węzłów, trzy typy relacji:

| Węzeł | Klucz | Relacja | Opis |
|---|---|---|---|
| `Document` | `source` (unique) | `HAS_CHUNK` | Dokument źródłowy |
| `Chunk` | `source + chunkIndex` | `MENTIONS` | Fragment tekstu z embeddingiem |
| `Entity` | `name + type` (unique) | `RELATED_TO` | Encja wyekstrahowana przez LLM |

Indeksy: full-text na `Chunk.content` i `Entity.name`, wektorowe na `Chunk.embedding` i `Entity.embedding`.

### Pipeline indeksowania

```
chunk → embed chunks → extract entities/rels via LLM → embed entities → write to Neo4j
```

- Chunking: 4000 znaków, 500 overlap, rekursywny podział po hierarchii separatorów (`\n## ` → `\n### ` → `\n\n` → `\n` → `. ` → ` `). Metadane `section` z nagłówków markdown.
- Hash SHA256 — skip-if-unchanged. Zmiana = usunięcie starego + pełne reindeksowanie.
- Auto-czyszczenie: po reindeksacji porównanie źródeł w grafie z plikami na dysku, usuwanie osieroconych wpisów. `reindex --force` czyści cały graf.
- Ekstrakcja (gpt-5-mini): 3–15 encji per chunk, enumy typów (concept/person/technology/organization/technique/other) i relacji (relates_to/uses/part_of/created_by/influences/contrasts_with/example_of/depends_on). Wymóg: source i target obecne w liście encji, brak self-refs.

### Deduplikacja encji — czterowarstwowa

LLM generuje warianty tego samego konceptu ("LLM", "Large Language Model"). Bez deduplikacji graf = szum.

1. **Normalizacja per chunk** — title case z akronimami, clamp typów do dozwolonego zbioru, filtracja self-ref i wiszących krawędzi.
2. **Globalna deduplikacja** — klucz `singularize(name.trim().toLowerCase())`, dłuższa nazwa + dłuższy opis wygrywają. Remap relacji na kanoniczne nazwy. Deduplikacja krawędzi po `source→type→target`.
3. **Neo4j MERGE** — `ON MATCH` zachowuje dłuższy opis. Constraint `(name, type) UNIQUE` jako ostatnia linia obrony.
4. **Runtime kuracja** — `audit` substring matchingiem + sieroty. `merge_entities` transakcyjnie rewiruje krawędzie i kasuje duplikat.

### Hybrydowe wyszukiwanie — RRF Fusion

Dwa niezależne strumienie: BM25 (full-text) + cosine (vector). Fuzja przez **Reciprocal Rank Fusion**:

```
score(chunk) = Σ 1/(k + rank + 1)  // k=60, sumowane po obu strumieniach
```

Overfetch: każdy strumień zwraca `limit * 3` wyników przed fuzją. Obcięcie do żądanego limitu dopiero po RRF — zwiększa szansę trafienia z obu strumieni. Graceful degradation — błąd vector search nie crashuje, FTS działa samodzielnie.

Po search: entity enrichment — chunki wzbogacane o encje przez `MENTIONS`. Agent dostaje tekst + "uchwyty" do nawigacji po grafie.

### Narzędzia agenta

| Kategoria | Narzędzie | Rola |
|---|---|---|
| Retrieval | `search` | Hybrydowy retrieval + odkrywanie encji w chunkach |
| | `explore` | Sąsiedztwo encji w grafie |
| | `connect` | Najkrótsza śceżka między encjami |
| | `cypher` | Read-only (odrzuca write keywords, appenduje LIMIT) |
| Curation | `learn` | Plik z workspace lub surowy tekst z etykietą źródła |
| | `forget` | Usunięcie dokumentu + orphaned entities |
| | `merge_entities` | Merge duplikatów encji |
| | `audit` | Diagnostyka: orfany, duplikaty, statystyki |

Strategia retrievalowa: `search` → `explore` dla ciekawych encji → `connect` dla relacji → `cypher` tylko dla zapytań strukturalnych. Nigdy dla small talk.

`learn` w dwóch trybach: plik (`{filename: "article.md"}`) i surowy tekst z etykietą (`{text: "...", source: "note:meeting-2024-02-11"}`) — dynamiczne budowanie grafu z treści konwersacji, nie tylko plików. Źródło identyfikowane przez etykietę, umożliwia `forget`.

### Pętla agenta

Maksymalnie 30 kroków: `user input → LLM → [tool calls → results → LLM]* → tekstowa odpowiedź`. Parallelne wykonanie tool calls (`Promise.all`). Historia zachowana między zapytaniami w ramach sesji REPL.
