# S02E01 — Zarządzanie kontekstem w konwersacji

## System prompt jako mapa terenu

Instrukcja systemowa daje agentowi orientację przed uruchomieniem narzędzi, ale nie zastępuje eksploracji. Kluczowe pytanie projektowe: **o czym agent musi wiedzieć PRZED sięgnięciem po narzędzia, żeby skrócić ścieżkę do celu?** Wszystko inne odkrywane dynamicznie.

Cztery warstwy treści:

| Warstwa | Zawartość | Zmienność |
|---|---|---|
| Uniwersalne instrukcje | Rola pamięci, generalne zasady zachowania, obszary wiedzy | Statyczna |
| Otoczenie | OS, profil użytkownika, typ interfejsu (głos/CLI/CRON), dostępne zasoby | Niska (per sesja) |
| Sesja | Podsumowania wcześniejszej rozmowy, wskazówki do skompresowanego kontekstu | Średnia |
| Zespół agentów | Reguły komunikacji, współdzielone fragmenty promptów | Statyczna (per system) |

Opisy narzędzi **nie należą** do system promptu, jeśli to samo narzędzie pełni różne role u różnych agentów. W prompcie siedzą **zgeneralizowane** opisy ról (pamięć, notatki, wyszukiwanie); specyfika żyje w schemacie narzędzia.

**Ostrzeżenie o chronologii**: "postępy realizacji zadania" w system prompcie zaburzają porządek czasowy — prompt staje się starszy niż dalsze wiadomości i dezorientuje model.

## Sygnał vs szum — warunki, nie kontrola

Nie da się w pełni kontrolować proporcji sygnału. Cel: **stworzyć warunki** maksymalizujące go przez detale systemu. Obszary, które ją kształtują:

- **Dostarczanie kontekstu** — niekompletne/błędne dane z narzędzi (konflikty, mylne API) to częste źródło problemów
- **Jakość logiki aplikacji** — mniej kodu w systemach agentowych, ale musi być niezwykle dopracowany; agent nie "obejdzie" błędów swojego harnessu (w przeciwieństwie do błędów obsługi narzędzi)
- **Prompt engineering jako projektowanie komponentów** — instrukcje pojawiające się dynamicznie, działające niezależnie od momentu dołączenia do kontekstu
- **Generyczne mechanizmy** — kompresja, planowanie, monitorowanie postępów projektowane uniwersalnie
- **Przestrzeń na doprecyzowanie** — interwencje człowieka/innego agenta gdy główny agent utknie

Przesunięcie balansu kod vs AI: workflow → agent → multi-agent. Mniej kodu w multi-agent nie oznacza spadku roli kodu — oznacza wyższe wymagania jakościowe.

## Agentic RAG — kontekst przez obserwację

Agent domyślnie **nie wie, o czym wie**. Sam system prompt nie wystarczy; agent musi iteracyjnie budować kontekst. Czterofazowa, **generyczna** strategia nawigacji:

1. **Skanowanie** — struktura folderów, nazwy plików, nagłówki. Rozpoznanie terenu zanim pojawi się konkretna ścieżka.
2. **Pogłębianie** — pętla: szukaj → czytaj → zbierz nowe terminy → szukaj ponownie. Warianty słów kluczowych, synonimy, terminologia z wcześniejszych wyników.
3. **Eksplorowanie** — tropienie powiązań: przyczyna/skutek, część/całość, problem/rozwiązanie, ograniczenia/obejścia, wymagania/konfiguracja.
4. **Weryfikacja pokrycia** — czy mam dość danych (definicje, liczby, edge case'y, kroki, wyjątki)? Jeśli nie — powrót do pogłębiania.

Te zasady działają niezależnie od źródła danych (markdown, vector DB, API). Jedyny specyficzny fragment promptu: charakter danych (np. język dokumentów). Przykład: agent szukający po angielsku w polskich dokumentach nic nie znajdzie — zamiast logiki translacji zapytań wystarczy **jedna linia** w prompcie o języku treści.

**Dlaczego większość strategii promptu nie działa:**

| Podejście | Problem |
|---|---|
| Specyficzne reguły ("szukaj limitów przy pytaniach o okno kontekstowe") | Działa dla jednego zapytania, nie generalizuje |
| "Szukaj powiązanych zagadnień" | Zbyt niejednoznaczne → nadinterpretacja → zalew nieistotnych treści |
| Few-shot przykłady | Obszerne, użyteczne tylko dla wąsko wyspecjalizowanych agentów |
| Opis wieloetapowego procesu | Zwiększa skuteczność, ale wymusza kroki nawet przy prostych zapytaniach |

Operujemy w obszarze prawdopodobieństwa, nie pewności — brak gwarancji 100% pokrycia.

## Cache hit jako priorytet architektoniczny

Definicje narzędzi leżą **pod** system promptem — każda zmiana promptu invaliduje cache narzędzi. Konsekwencje:

- System prompt **statyczny** w ramach sesji
- Dynamiczne dane (git status, otwarte pliki, data) → **wiadomość użytkownika**, nie system prompt
- Kolejne wiadomości zawierają tylko to, co zmieniło się od ostatniej
- Starsze metadane odświeżane przez narzędzia, nie modyfikację promptu
- Tagi XML-like oddzielają właściwe zapytanie od dołączonych metadanych

Wiadomość użytkownika nie musi zawierać wyłącznie tego, co użytkownik napisał — programistyczne doklejanie kontekstu, który użytkownik **mógłby** dostarczyć, jest uzasadnione.

## Sterowanie uwagą przez powtórzenia

Model gubi fakty w długich konwersacjach. Dwa mechanizmy bez modyfikacji system promptu:

**Listy zadań** — model zapisuje plan, odznacza zrobione, **przepisuje pozostałe** w kolejnych wiadomościach. Powtórzenie utrzymuje uwagę. Treść generowana przez sam model może mieć silniejszy wpływ na jego zachowanie niż treść zewnętrzna (por. many-shot jailbreaking). Wymaga programistycznego wsparcia — bez niego model zapomina aktualizować listę.

**Tryb planowania** — informacja o trybie dołączana do wiadomości użytkownika, nie do system promptu. Cache bezpieczny, starsze wiadomości nietknięte.

Oba mechanizmy mają odbicie również w UI, nie tylko w kontekście modelu.

## Kontekst poza oknem — Agent Harness

Infrastruktura wokół modelu umożliwiająca funkcjonowanie poza bieżącym oknem:

- **Sesja** — hooki monitorujące interakcje, częściowe podsumowania w tle, kolejki aktualizacji z zewnętrznych systemów
- **Pamięć** — budowanie wspomnień asynchronicznie (nawet po sesji, np. raz/dobę), Batch API dla obniżenia kosztów
- **Pliki** — wyniki narzędzi, komunikacja między agentami, załączniki, notatki; powstają bez udziału agenta i są wykorzystywane później
- **Otoczenie** — dane spoza urządzenia użytkownika, aktualizowane poza kontekstem, wstrzykiwane warunkowo

## Workspace agentów

Przestrzeń plików per sesja z programistyczną kontrolą dostępu:

- **Katalogi dat → sesje → agenci** — izolacja sesji = izolacja danych użytkowników
- Role katalogów: `notes/` i `outbox/` (zapis agenta), `inbox/` (zapis **wyłącznie** przez root)
- Sub-agenci nie komunikują się bezpośrednio — root routuje dokumenty między nimi
- Zawartość: załączniki użytkownika, notatki sesyjne, dokumenty "publiczne" dla usera oraz "wewnętrzne" (inbox/outbox)

## Maskowanie kontekstu (Manus)

Uzupełnianie początku wypowiedzi modelu tokenami wymuszającymi konkretne narzędzie: `<|im_start|>assistant<tool_call>{"name": "browser_`. Dopóki przeglądarka jest aktywna, agent fizycznie nie może wywołać niczego innego. Ograniczenie zdejmowane deterministycznie po zakończeniu sesji.

Deprecated w API Anthropic, ale ilustruje zasadę: **niekonwencjonalne podejścia potrafią adresować całe klasy problemów**.

## Generalizowanie instrukcji z pomocą LLM

Generalizacja = najważniejsza umiejętność przy budowie agentów. Brak walidatorów; tylko obserwacja i iteracja. Model potrafi **uzasadnić** swoje zachowanie i zasugerować zmiany, ale pierwsza propozycja jest zwykle zbyt bezpośrednia (łata konkretny przypadek zamiast kategorii).

Proces iteracji:
1. **Analiza** — opisz problem, podaj kontekst (narzędzia, prompt, zachowanie), poproś o diagnozę przyczyn
2. **Generalizacja** — żądaj uniwersalnych przyczyn, nie łatek
3. **Korekta** — odrzucaj ~60% sugestii bezwartościowych, ~30% wymagających poprawek; podkreślaj niezależność promptu od konkretnych narzędzi i unikanie "przesterowania"
4. **Iteracja** — proste wskazówki, z których model wyciąga esencję

Efekt: zwięzłe, precyzyjne sformułowania ("provide clear, concise answers", "ask or pick safest option"), do których trudno dojść samodzielnie.

## Przykłady kodu — `02_01_agentic_rag`

Interaktywny REPL chatbot przeszukujący lokalne pliki markdown przez MCP. Jedyna zależność: `@modelcontextprotocol/sdk`.

```
app.js          → MCP client + REPL + shutdown
src/agent.js    → pętla agentowa (chat → tool calls → results → chat)
src/config.js   → system prompt + parametry modelu
src/mcp/client.js → MCP stdio transport, konwersja narzędzi MCP → OpenAI
src/repl.js     → readline z historią konwersacji
```

### 1. System prompt: zgeneralizowane zasady wyszukiwania

Prompt uczy *jak szukać*, nie *co znaleźć*. Cztery sekcje w `config.js`:

```
## SEARCH GUIDANCE → Scan / Deepen / Explore / Verify (uniwersalne)
## EFFICIENCY     → Nigdy nie czytaj całych plików; najpierw szukaj, potem czytaj fragmenty
## RULES          → Odpowiadaj tylko na podstawie źródeł, cytuj, przyznawaj braki
## CONTEXT        → Jedyna specyficzna sekcja: język dokumentów (PL), źródło (AI_devs)
```

Sekcja `EFFICIENCY` wymusza wyczerpanie wariantów słów kluczowych **przed** sięganiem po pełną treść — ochrona przed marnowaniem tokenów.

### 2. Pętla agentowa z MCP

MCP jako abstrakcja: agent nie zna narzędzi w compile-time, pobiera je z serwera i konwertuje na format OpenAI. Zmiana źródła danych = zmiana `mcp.json`, zero zmian w kodzie agenta.

```js
const messages = [...conversationHistory, { role: "user", content: query }];
for (let step = 1; step <= MAX_STEPS; step++) {
  const response = await chat({ input: messages, tools });
  const toolCalls = extractToolCalls(response);
  if (toolCalls.length === 0) return { response, conversationHistory: messages };
  messages.push(...response.output);
  messages.push(...await runTools(mcpClient, toolCalls));  // Promise.all wewnątrz
}
```

Limit 50 kroków. Konwersja narzędzi MCP → OpenAI:
```js
mcpTools.map(t => ({ type: "function", name: t.name, description: t.description, parameters: t.inputSchema, strict: false }))
```

### 3. Reasoning z widocznością myślenia

`reasoning: { effort: "medium", summary: "auto" }` — reasoning automatycznie podsumowywany (zmniejsza zużycie kontekstu). Logger wyświetla summaries: widać jak model planuje wyszukiwanie przed wywołaniem narzędzi.

### 4. REPL i tracking kosztów

REPL utrzymuje `conversation.history` między zapytaniami; komenda `clear` resetuje historię i statystyki. `stats.js` zlicza input/output/reasoning/cached tokens per sesja, logger pokazuje breakdown po każdym wywołaniu API. Trójpodział odpowiedzialności: **REPL** zarządza cyklem życia sesji, **agent loop** pojedynczym zapytaniem, **system prompt** zachowaniem modelu.
