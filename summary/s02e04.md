# S02E04 — Organizowanie kontekstu dla wielu wątków

## Architektury wieloagentowe

| Architektura | Koordynacja | Zależności | Kiedy |
|---|---|---|---|
| **Pipeline** | Sekwencyjne przekazanie wyniku dalej | Wynik N = wejście N+1 | ETL, generowanie + redakcja |
| **Blackboard** | Współdzielony stan, niezależni agenci | Odczyt/zapis wspólnej przestrzeni | Równoległe gromadzenie danych |
| **Orchestrator** | Agent-koordynator deleguje i odbiera | Manager steruje przepływem | Złożone wieloetapowe zadania |
| **Tree** | Orchestrator + managerowie pośredni | Hierarchia delegowania | Duża skala, podział odpowiedzialności |
| **Mesh** | Komunikacja adresowana agent↔agent | Luźne powiązania, routing | Stałe role, bezpośrednia komunikacja |
| **Swarm** | Rozproszona, self-organizing | Selekcja/agregacja wyników | Rzadko w LLM — trudne do debugowania |

W praktyce dominują pierwsze cztery, często łączone. Mesh i Swarm — trudne do kontroli i debugowania.

## delegate i message — dwa prymitywy komunikacji

**delegate** — otwiera nowy wątek: własny system prompt + własny zestaw narzędzi + niezależna pętla agentowa. Wynik sub-agenta staje się wynikiem narzędzia `delegate` w kontekście agenta nadrzędnego. Fire-and-collect: zlecaj i zbierz rezultat.

**message** — dwukierunkowa komunikacja z wstrzymaniem. Pętla sub-agenta **zawieszona** do czasu dostarczenia danych. Realizowalne przez generatory JS (`yield` = czekaj na dane, `next()` = dostarcz). Scenariusz: sub-agent potrzebuje doprecyzowania → nadrzędny kontaktuje użytkownika → przekazuje odpowiedź → praca sub-agenta wznowiona.

Różnica: `delegate` = jednokierunkowe zlecenie, `message` = dwukierunkowy dialog z zawieszeniem.

## Architektury zdarzeniowe

Powiązania między agentami **bezpośrednie** (A deleguje do B) lub **pośrednie** — przez event bus. Zdarzenia rozbijają sprzężenie: agent A nie musi wiedzieć, że agenty C i D zareagują na jego wynik.

Przepływ obsługi zgłoszenia klienta:
1. `user.message` → Intent Agent klasyfikuje zgłoszenie
2. `ticket.classified` → Tracker i Credit reagują **niezależnie** (to samo zdarzenie, dwa agenty)
3. `customer.lookup_requested` → Draft Agent nie ma pełnych danych, emituje zdarzenie żądania i **zawiesza się**
4. Po dostarczeniu danych → Draft Agent pisze szklic → człowiek weryfikuje

Agent może nasłuchiwać, emitować i zawieszać się czekając na dane — wszystko przez zdarzenia. Koszt: złożoność debugowania i trudniejszy tracing.

## Współdzielony kontekst — konflikty

Jedna instancja agenta uruchomiona wielokrotnie = system wieloagentowy. Agent A czyta plik → agent B zapisuje → agent A zapisuje = utrata danych B (ostatni zapis wygrywa).

| Strategia | Mechanizm | Kiedy |
|---|---|---|
| **Wykrywanie** | Checksumy/hash'e linii między odczytem a zapisem | Dwa agenty modyfikują ten sam plik |
| **Unikanie** | Izolacja zasobów per sesja, uprawnienia RO, przynależność | Projektowanie architektury |
| **Agent zarządzający** | Memory Manager z pełnym wglądem + kontakt z człowiekiem | Pamięć długoterminowa |
| **Historia zmian** | Append-only zamiast overwrite | Wiedza temporalna |
| **Eskalacja** | Człowiek rozstrzyga (jak merge conflict w Git) | Gdy automatyzacja zawodzi |

Zewnętrzny kontekst (baza wiedzy, pamięć) **odseparowany od logiki agentów** — dokumenty opisują co, system decyduje kiedy i kto. Dostęp per agent + per użytkownik, nie globalny. Dokumenty dostępne także dla ludzi = kolaboracja, nie tylko automatyzacja.

## Wyzwania współdzielonego kontekstu

1. **Sesja vs pamięć** — informacja tymczasowa vs trwała. Kto decyduje co utrwalić? Agenci w tle potrzebują zgeneralizowanych zasad, balansowanych deterministycznymi ograniczeniami (np. dostęp do katalogów).

2. **Degradacja komunikacji** — każdy przekaz między agentami = potencjalna utrata/zniekształcenie. Instrukcje `delegate` i `message` muszą być precyzyjne. System musi zakładać niekompletne dane na wejściu sub-agenta.

3. **Własna interpretacja** — te same dane, inna interpretacja. Ryzyko minimalne dla zadań zamkniętych ("zaktualizuj X"), rośnie dla otwartych ("znajdź wszystko o X").

4. **Kontekst oryginalny** — notatka "Anna" stworzona w rozmowie o Ani K. → pomylona z Anną M. w innej sesji. Metadane (źródło, data, powiązania) zapobiegają odrywaniu informacji od kontekstu powstania.

5. **Duplikowanie** — nieuniknione, ale wykrywalne. Mniejsze modele mogą skanować zmiany i identyfikować duplikaty (technika znana z grafów w S02E03).

6. **Metadane** — wykorzystywane nie tylko w UI, ale **w komunikacji między agentami**. Pytanie "o czym rozmawialiśmy w drodze do Warszawy" wymaga metadanych o czasie i lokalizacji, nie tylko treści.

Zasada: projektować system tak prosty jak to możliwe i utrzymywać w tej formie najdłużej.

## Agent zarządzający

Zwykle brak narzędzi operacyjnych (poza `delegate`, `message`, `search_memory`), ale najszerszy dostęp do informacji i uprawnień:

- **Wiedza o systemie** — role agentów, zakres odpowiedzialności, dane użytkownika
- **Dostęp do informacji** — wyniki agentów + workspace + pamięć (min. RO)
- **Delegowanie** — plan działania, postępy w kontekście głównego wątku
- **Transport wiedzy** — przekazywanie rezultatów między agentami, dostarczanie brakujących danych
- **Decyzyjność** — limity uprawnień, eskalacja do człowieka
- **Weryfikacja** — ocena rezultatów przed akceptacją (samodzielnie lub przez delegację)

Ograniczanie roli konieczne (modele zawodzą), ale z umiarem — każdy dodatkowy etap komunikacji komplikuje zarządzanie kontekstem. Część błędów niezauważalna od razu (pominięcie etapu z powodu braku danych, nie zatrzymanie systemu).

## Agenci vs kod

| Kryterium | Agenci uzasadnieni |
|---|---|
| Zadania otwarte | Cel jasny, ale reagowanie na dane z otoczenia w trakcie |
| Dynamiczne dane | Input/output nie z góry określony, transformacje poza kodem |
| Dynamiczne zależności | Zależności na poziomie semantyki, nie struktury danych |
| Iterowanie | Kryteria w języku naturalnym, kroki niemożliwe do zdefiniowania z góry |
| Elastyczna architektura | Nowe obszary bez zmian w logice |
| Dopasowanie wyniku | Personalizacja wykraczająca poza szablony |

Gdzie wymagania wykluczają LLM (koszt zero, odpowiedź w ms, pełna przewidywalność) — tylko deterministyczny kod.

## Wyzwalacze i nadzór

**CRON / wyzwalacze** — zadanie agenta nie wymaga ręcznego promptu. Daily Ops startuje z CRONa wysyłającego powiadomienie w imieniu użytkownika z odniesieniem do pliku workflow. Wzorzec: trigger + odniesienie do instrukcji w pliku, nie hardkodowana logika.

**Panel zarządzania** — agenci sprawiają, że pojedyncze okno czatu nie wystarcza. Potrzebne dedykowane interfejsy: statystyki, aktywne sesje, harmonogram zadań, obszary wymagające uwagi. Człowiek pozostaje kluczowym koordynatorem systemu agentowego — dashboard, nie czat.

## Przykłady kodu: Daily Ops (`02_04_ops`)

Minimalny system: 4 pliki TypeScript (~270 linii) + workspace z plikami markdown/JSON. Delta względem S02E01 (`02_01_agentic_rag`): agent loop **bez MCP** — narzędzia zdefiniowane inline w `tools.ts`, nie pobierane z serwera. Celowo prostsze, pokazuje czystą mechanikę delegacji.

### Agent-as-Markdown (frontmatter pattern)

Definicja agenta = plik `.agent.md` z YAML frontmatter + treść jako system prompt:

```markdown
---
model: openai:gpt-4.1-mini
tools: [get_mail]
---
You scan the inbox and return a structured summary.
```

Nowy agent = nowy plik, zero zmian w kodzie. `gray-matter` parsuje frontmatter, treść staje się system promptem. Podmioty — osobne szablony, osobne narzędzia, osobne modele.

### delegate jako rekurencyjne runAgent()

Definicja `delegate` w `tools.ts` to tylko JSON Schema — handler zwraca surowe args. Rzeczywista logika w `agent.ts`:

```typescript
if (name === 'delegate') {
  result = await runAgent(agent, delegatedTask, depth + 1)
}
```

Rekurencyjne wywołanie z `depth + 1`. Guardrails: `MAX_DEPTH = 3`, `MAX_TURNS = 15`. Sub-agent dostaje własną konwersację z własnym system promptem i narzędziami. Wynik sub-agenta = wynik narzędzia dla orchestratora. Brak `message` — komunikacja jednokierunkowa.

### Workflow-as-Data

Orchestrator dostaje: "przeczytaj `workflows/daily-ops.md` i wykonaj". Sam odczytuje instrukcje przez `read_file`, interpretuje kroki i egzekwuje. Zmiana workflow = zmiana markdowna, bez rekompilacji. **Podział odpowiedzialności przez narzędzia**: sub-agent widzi tylko swoje (np. `get_mail`), orchestrator widzi `delegate` + `read_file` + `write_file`.

### Bezpieczeństwo ścieżek

`isPathSafe()` — walidacja konieczna bo LLM generuje ścieżki:

```typescript
const rel = relative(workspaceResolved, fullPath)
return !rel.startsWith('..') && rel !== '..'
```

Path traversal (`../../etc/passwd`, `../../.env`) zablokowany. Wszystkie operacje scoped do `workspace/`.

### Kontekst warstwowy (agent-driven context assembly)

Workspace dzieli dane na cztery kategorie dostępne przez `read_file`:

| Katalog | Zawartość | Rola |
|---|---|---|
| **goals/** | Długoterminowe cele | Alignment decyzji z priorytetami |
| **history/** | Wczorajszy output | Dedup + escalation z licznikiem dni |
| **memory/** | Preferencje usera | Format, styl, reguły priorytetyzacji |
| **sources/** | Dane wejściowe | Mock API, statyczne JSON |

Agent sam decyduje co przeczytać, w jakiej kolejności i jak połączyć. To nie RAG — to **agent-driven context assembly**. Synteza z historią, celami i preferencjami = rola orchestratora, nie mechaniki.

### Uproszczenia (świadome, PoC)

- Źródła danych: statyczne JSON, nie API
- Delegacja: sekwencyjna, nie równoległa (`Promise.all` prychnęłoby koszty)
- Brak weryfikacji outputu przez orchestratora
- Brak conflict detection przy zapisie
- Wszystkie agenty na `gpt-4.1-mini` mimo mechanizmu pozwalającego na różne modele
