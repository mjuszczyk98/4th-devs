# S04E04 — Projektowanie własnej bazy wiedzy dla AI

## Perspektywa: baza wiedzy = kod źródłowy agenta

Poza programowaniem agent nie ma naturalnego źródła kontekstu — użyteczność zależy wyłącznie od instrukcji, bieżącej interakcji i tego, co sam znajdzie. Baza wiedzy wypełnia tę lukę, stając się ekwiwalentem kodu źródłowego dla domeny pozaprogramowej. Pytanie nie "jak budować bazę?", lecz "co agent musi wiedzieć, żeby być użyteczny w MOIM kontekście?". Odpowiedź jest indywidualna i wymaga iteracji.

W architekturze z lekcji baza wiedzy i pamięć długoterminowa to ta sama przestrzeń — funkcjonują w tych samych plikach, tworzonych przez człowieka i AI.

## Pięć stref odpowiedzialności

| Katalog | Właściciel | Cel |
|---|---|---|
| `me/` | człowiek | tożsamość, preferencje, proces osobisty |
| `world/` | człowiek + agent | ludzie, miejsca, narzędzia, źródła, zdarzenia |
| `craft/` | człowiek + agent | idee, projekty, wiedza, eksperymenty, publikacje |
| `ops/` | agent | workflowy, research, dane operacyjne |
| `system/` | człowiek | profile agentów, szablony, reguły |

Podział funkcjonalny, nie techniczny. Kluczowe rozróżnienie: kto pisze (człowiek vs agent) i kto czyta (człowiek, agent, obaj). **Człowiek odpowiada za treść i zasady, AI za organizację.** Agent nie przekracza strefy człowieka bez wyraźnego pozwolenia — gdyby sam tworzył treść w sekcjach ludzkich, tracimy sens budowania bazy.

Wyróżniona trzecia strefa: obszary **niezautomatyzowane**, ale obecne w bazie jako kontekst dla agentów. Przykład: tworzenie treści — człowiek pisze, ale agent ma dostęp do materiałów bez ręcznego przekazywania.

## Notatki samowystarczalne

Zasada: **notatki piszemy tak, jakby czytelnik nie miał żadnego dodatkowego kontekstu.** Fundamentalna zmiana nawyku — z kodowania implikacji do ich jawnej ekspozycji.

| Luk kontekstowy | Przykład | Rozwiązanie |
|---|---|---|
| Niejawne referencje | "ostatnia rozmowa", "w poprzedniej wersji" | jawny link do powiązanej notatki |
| Nierozpoznawalne linki | skrócony URL bez opisu treści | pełny URL z opisem |
| Nadpisane wersje | edycja usuwa poprzednią wiedzę | wersjonowanie lub explicit "zastąpiono przez X" |
| Brak powtórzeń | powiązanie istnieje tylko w jednym dokumencie | linki w obu kierunkach |

Rozwiązanie: jawne linkowanie, pełne nazewnictwo, notatki indeksowe (MoC), zasady wersjonowania.

## Frontmatter — wieloosiowa kontrola metadanych

Osy niezależne od siebie:

- **publish** (`draft` → `review` → `live` → `updated`) — cykl publikacji
- **status** (`seed` → `growing` → `evergreen` → `archived`) — dojrzałość treści
- **access.read / access.write** — dziedziczone z sekcji, nadpisywane per notatka
- **attention** (`who` + `reason`) — sygnał handoffu między człowiekiem a agentem
- **tags** — płaskie (bez hierarchii `ai/llm`), opisują **co**, nie **gdzie** (folder to załatwia)

Domyślnie `seed` + `draft` + dostęp zgodny z sekcją. Jawne ustawienie tylko przy nadpisaniu.

## Markdown jako format

AI natywnie generuje i rozumie markdown. Pliki tekstowe = swobodna transformacja, wyszukiwanie, wersjonowanie. YAML frontmatter dodaje maszynowo-czytelny layer metadata.

Ograniczenia: praca wieloosobowa na jednym dokumencie w czasie rzeczywistym, złożone uprawnienia → Notion/Google Docs. Konwersja między formatami traci informacje — lepiej podjąć jedną decyzję per obszar. Obrazy muszą być zewnętrznymi URL — lokalne pliki są "niewidoczne" dla agenta.

## Role AI w zarządzaniu bazą

| Rola | Zakres |
|---|---|
| Transformacja | formatowanie, korekta, OCR — źródło = człowiek, AI = formatter |
| Szablony | utrzymanie struktury — szablony w `system/templates/` |
| Organizacja | sugestia placement'u, nie automatyczna akcja |
| Linkowanie | wikilinkowanie między powiązanymi notatkami z regułami w `system/rules/` |
| Walidacja | sprawdzanie zgodności z zasadami (struktura, tagi, linki) |
| Indeksowanie | generowanie Map of Content (MoC) |
| Audytowanie | wykrywanie braków, szumu, brakujących linków |

## Szablony notatek

Szablony w `system/templates/` definiują: docelowy katalog, konwencję nazwy pliku, sekcje frontmatteru, sekcje treści. Agent przed utworzeniem notatki: czyta `index.md` → czyta odpowiedni szablon → tworzy plik we właściwym miejscu **lub aktualizuje istniejący**.

Efekt: im większa baza, tym większa wartość szablonów — spójność struktury bez ręcznej dyscypliny. Koszt: kilkanaście wywołań LLM na jedną notatkę (orientacja + szablon + decyzja + zapis).

## mind.md — konstytucja bazy wiedzy

Jeden plik referencyjny (`mind.md`, ~258 linii) definiujący pełną architekturę vaultu:
- Struktura katalogów z uprawnienami dziedziczonymi per sekcja (np. `Craft/Knowledge` → write: `[adam, ellie, tony]`)
- ~15 scenariuszy agentów pokazujących nawigację po secjach (np. Ellie: `Ops/Research` → `World/Sources` → `Craft/Knowledge` → `Craft/Projects`)
- Wikilink conventions: ścieżki vault-root absolutne (`[[Craft/Knowledge/AI/transformers]]`, nie `[[../../transformers]]`), aliasy przez pipe
- Tag reuse: agenci sprawdzają tagi w sąsiednich notatkach przed tworzeniem nowych

## Workflow jako preskryptywne fazy

Workflow = katalog w `ops/` z plikami: `_info.md` (metadata, źródła, fazy, `depends_on`) + pliki fazowe `01-*.md`, `02-*.md`...

Kluczowa decyzja: **instrukcje preskryptywne, nie goal-oriented.** Każda faza = dokładna lista kroków + lista "Do NOT". Vague cele dają nieprzewidywalne rezultaty; tight playbook = przewidywalność.

Podział ról: orchestrator (Alice) czyta `_info.md`, deleguje fazy sekwencyjnie — jedna faza na czas, każda zależy od plików wygenerowanych przez poprzednią. Każdy agent dostaje swój plik fazowy jako instrukcję. Sekwencyjność obowiązkowa — nigdy równoległe.

## Profile agentów jako markdown

Agent = plik `system/agents/<name>.md` z YAML frontmatter (`title, description, model, tools`) + treść jako system prompt. Loader parsuje gray-matter. Każdy agent ma ograniczony zestaw narzędzi (np. Ellie: `files, web`; Tony: `files`; Rose: `files, send_email`) i jawne ograniczenia w prompcie.

### Dual-mode prompting

Każdy agent ma dwa tryby z osobną inicjalizacją:
1. **Workflow mode** — agent czyta TYLKO plik fazy, wykonuje kroki. Nie czyta templates, indexów, nie eksploruje filesystemu.
2. **Standalone mode** — agent działa samodzielnie (czyta template, researchuje, zapisuje).

Każdy tryb czyta inne pliki — unika niepotrzebnych odczytów i scope creep.

## Delegacja z kontrolą głębokości

`delegate` tool pozwala agentowi przekazać zadanie sub-agentowi. Dziecko dostaje własny model, narzędzia, instrukcje — działa niezależnie, zwraca tekstowy wynik. **Depth limit** (`MAX_DEPTH`): przy maksymalnej głębokości tool `delegate` jest fizycznie usuwany z listy dostępnych narzędzi — agent nie może złamać reguły, bo jej nie widzi.

Constraint enforcement przez architekturę, nie prompt — instrukcje można zignorować; brak narzędzia — nie.

### Ścieżki docelowe

Agenci delegowani otrzymują **dokładne ścieżki** do odczytu i zapisu — nie eksplorują filesystemu samodzielnie. Orchestrator jest jedynym agentem z orientacją w strukturze vaultu. Zmniejsza liczbę kroków, eliminuje błędy nawigacji.

## MCP — namespacing i dynamiczna rejestracja

Serwery narzędziowe jako child processes przez stdio, konfigurowane w `mcp.json`. Narzędzia odkrywane runtime: `client.listTools()`. Nazwy prefiksowane serwerem: `files__fs_read`, `web__search`. Prefix matching (`name.split("__")[0]`) pozwala agentowi deklaratywnie wybrać serwer — `tools: [files]` daje wszystkie toolle z serwera `files`. Scoped access: filesystem MCP rootowany w `./workspace`.

Dodanie nowego serwera = jedna linijka w `mcp.json`, zero zmian w kodzie agenta.

## Wzorce operacyjne

### Scatter/gather w researchu

Agent przy web scrapingu używa `outputMode: "file"` — wyniki zapisywane jako pliki, nie inline w kontekście. Następnie czyta tylko potrzebne fragmenty. Pełne artykuły nigdy nie trafiają do conversation history — kontrola wielkości kontekstu.

### Wielo-turnowa akumulacja

Po każdej odpowiedzi conversation aktualizowana o `[...result.conversation, { role: "assistant", content: result.text }]` — agent pamięta poprzednie notatki w sesji. Kontrastuje z single-shot, gdzie każde zapytanie zaczyna nową konwersację.

### dryRun przy aktualizacji

Rozróżnienie tworzenia i aktualizacji: "When creating new files, write directly — never use dryRun. Only use dryRun when updating existing files." — mechanizm bezpieczeństwa przed nadpisaniem istniejącej wiedzy.

## Odkrywanie zakresu i start praktyczny

Zamiast zakładać, że AI zautomatyzuje wszystko — przeskanować rzeczywiste otoczenie: regularnie odwiedzane domeny, powtarzalne aktywności, narzędzia. Lista okaże się krótsza, niż się wydaje. Podzielić na: obszary do automatyzacji, do kontekstowego wsparcia, wyłączone z AI.

"Najlepsze pomysły nie są powodem wykonanej pracy, lecz jej rezultatem" — trzeba budować i popełniać błędy, żeby dotrzeć do zastosowań, których planowanie nie wyprodukuje.

Nie trzeba budować całej struktury naraz. Wybrać jeden obszar — najlepiej coś, co się podoba, niekoniecznie "użytecznego". Można zacząć bez kodu: podłączyć katalog markdown bezpośrednio do Claude Code, ukształtować pierwsze notatki i procesy, a dopiero potem myśleć o runtime, MCP i logice.

## Przykłady kodu

### 1. Loader — profil agenta z pliku markdown

Delta względem hardkodowanych konfiguracji: agent = plik `.md`, nie klasa. Frontmatter = konfiguracja, treść = prompt.

```js
export const loadAgent = async (name) => {
  const raw = await readFile(join(AGENTS_DIR, `${name}.md`), "utf-8");
  const { data, content } = matter(raw);
  return {
    name: data.title ?? name,
    model: data.model ?? "gpt-4.1-mini",
    toolNames: data.tools ?? [],
    instructions: content.trim(),
  };
};
```

### 2. Delegate — factory z depth awareness

Delta: handler nie wykonuje logiki, uruchamia osobny agent loop. `depth + 1` propagowany do dziecka.

```js
export const createRun = (depth, chatFn) => async ({ agent: name, task }) => {
  const child = await loadAgent(name);
  const { text } = await chatFn(
    [{ role: "user", content: task }],
    { ...child, depth: depth + 1 },
  );
  return { agent: child.name, result: text };
};
```

### 3. Agent loop z delegacją i filtracją tooli

Delta względem prostych loopów: depth-aware tool availability, `Promise.all` na parallel tool calls, conversation accumulowana przez spread.

```js
const canDelegate = depth < MAX_DEPTH;
const availableTools = canDelegate
  ? [...tools(toolNames), delegateDefinition]
  : tools(toolNames);

for (let step = 0; step < MAX_STEPS; step++) {
  const response = await complete(current, availableTools, { instructions, model });
  const calls = response.output.filter((item) => item.type === "function_call");
  if (calls.length === 0) return { text: response.output_text, conversation: current };
  const results = await Promise.all(calls.map(executeTool));
  current = [...current, ...calls, ...results];
}
```

### 4. MCP registry — dynamiczne łączenie z prefix matching

Delta: narzędzia wczytywane z serwerów MCP i prefiksowane. Deklaracja `tools: [files]` daje wszystkie toolle z serwera.

```js
export const tools = (names) => {
  if (!names || names.length === 0) return allDefinitions;
  const nameSet = new Set(names);
  return allDefinitions.filter((def) =>
    nameSet.has(def.name) || nameSet.has(def.name.split("__")[0]),
  );
};
```

### 5. Workflow phase file — proces jako instrukcja

Delta: definicja procesu operacyjnego jako plik markdown z frontmatter `{agent, depends_on}` i preskryptywnymi krokami + listą zakazów.

```markdown
---
title: "Phase 1 — Research"
agent: ellie
depends_on: []
---
## Steps
1. Run `web__search` with queries "{topic} news {yyyy-mm-dd}"...
2. Read the first 50 lines of each search result file...
3. For each topic with results, write to `ops/daily-news/{yyyy-mm-dd}/{topic}.md`...
Do NOT: browse the workspace tree, read templates, scrape full articles...
```
