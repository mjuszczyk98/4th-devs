# S05E02 — Zestaw narzędzi

## Architektura Event Sourcing w UI agenta

Zdarzenia SSE to źródło prawdy, bloki UI (text, thinking, tool_call, artifact, error) to projekcja. Dwie ścieżki materializacji:

- **Pełna rekonstrukcja** z historii — O(n), hydratacja przy ładowaniu konwersacji
- **Przyrostowa** `applyEvent` — O(1), mutacja bloków in-place. `toolIndexById` daje O(1) resolwację tool_result → istniejący ToolBlock

Typy zdarzeń: `text_delta`, `thinking_start/delta/end`, `tool_call`, `tool_result`, `artifact`, `error`, `complete`. Każdy → dedykowany blok UI.

Serwer tłumaczy format providerowy na zunifikowany model: `response.output_text.delta` → `text_delta`, `response.reasoning_summary_text.delta` → `thinking_delta`, `response.function_call_arguments.done` → `tool_call`.

Deduplikacja klient-side: `eventIdsByMessageId: Map<string, Set<string>>` + `lastSeq` per wiadomość. Zdarzenie ignorowane gdy `knownIds.has(event.id)` lub `event.seq <= lastSeq` — ochrona przed retransmisjami i wyścigami sieciowymi.

`store: false` w Responses API — konwersacja nie persystowana u providera, pełna kontrola nad kontekstem na serwerze.

## Inkrementalny Streaming Markdown

Pipeline pięciu warstw (kolejność krytyczna — pominięcie = konkretna awaria):

1. **remend** — naprawa niekompletnej składnię (niezamknięte code fences, partial emphasis) przed parsowaniem. Bez tego pękają code blocki mid-stream
2. **marked (Lexer)** — tokenizacja na niezależne bloki (akapit, nagłówek, lista). Bez tego pełny re-render co token
3. **markdown-it** — tokeny → HTML
4. **highlight.js** — kolorowanie składni
5. **DOMPurify** — usuwanie złośliwego HTML z outputu LLM (XSS). Niewyłączny element pipeline'u

Podział treści na `committedSegments` (sfinalizowane, cachowane, nie przebudowywane) + `liveTail` (ostatni niekompletny blok, re-renderowany co nowy token). Promocja z liveTail do committedSegments = zamrożenie. Można odroczyć (`compactionDeferred`) gdy DOM niegotowy.

Edge case'y `parseMarkdownIntoBlocks` — bez nich inkrementalny rendering rozbiłby struktury wymagające kontekstu:
- Zagnieżdżone tagi HTML — `htmlStack` śledzi otwarte tagi, bloki scalane dopóki stos niepusty
- Niezamknięte bloki matematyczne `$$` — parzystość decyduje o kompletności
- Footnotes — zawsze scalane jako jeden blok (ref + definicja w jednym drzewie parsowania)
- Zabezpieczenie `*` na końcu elementu listy przed pomyleniem z list markerem — zero-width space

Optymalizacje: cache `Map<string, string>` z limitem 400 wpisów; `highlight: false` podczas streaming, pełne highlight po zakończeniu; kursor blink wstrzykiwany jako `<span class="caret-blink">` przed ostatni zamykający tag blokowy.

**Streamdown** (Vercel) — potencjalna alternatywa całego pipeline'u, nieprzetestowana produkcyjnie.

## Wydajność UI

### Wirtualizacja chunkowa

Grupowanie po `CHUNK_SIZE` (12 wiadomości) — jednostka montowania/demontażu w DOM. Per-element zbyt granularny.

- `ResizeObserver` mierzy wysokość chunka; niezmierzone estymowane ze średniej
- `visibleRange` z overscan (3 chunki marginesu) + top/bottom spacer absolutny, `overflow-anchor: none`
- **Scroll anchoring:** chunk powyżej viewport zmienia wysokość → `scrollTop` kompensuje deltę

**Pin/unpin asymetryczny** z histerezą: `PIN_RELEASE_DISTANCE` (80px) do odpinania, `PIN_REACQUIRE_DISTANCE` (4px) do przypinania. Czasowe blokady (`scrollGuardUntil`, `bottomLockUntil`) chronią przed wyścigami.

CSS uzupełniający: `content-visibility: auto` (pomijanie rendering poza viewport), `overscroll-behavior: contain` (blokada scroll chaining), `scrollbar-gutter: stable` (eliminacja layout shift).

### Batchowanie SSE — requestAnimationFrame

Bufor `pending[]`, flush synchronizowany z `requestAnimationFrame` — DOM aktualizuje się maks. 60fps niezależnie od tempa streamingu. Bez tego: render thrashing.

### Typewriter

Cztery tryby (`off/fast/normal/slow`) — `burst` (tokenów na flush) + `interval` (ms). **Gating:** kolejny blok UI czeka na zakończenie animacji poprzedniego. Bez gating interfejs "wyskakuje" zamiast płynnie odkrywać treść.

Zaznaczenie tekstu: jeśli użytkownik zaznaczył w live tail, render odkładany (`pendingRenderState`) by nie zepsuć selekcji.

### Mock-first development

Serwer wspiera tryb `mock` z predefiniowanymi scenariuszami (sales, email, artifact, research). `ScenarioBuilder` z fluent builderem generuje sekwencję `StreamEvent`ów z konfigurowalnymi opóźnieniami. `detectScenario()` dopasowuje prompt regexem. UI rozwijane z 1500+ wiadomościami bez kosztów API — deterministyczne testy.

## Agenci głosowi

### Dwie architektury

| | STT → LLM → TTS | Realtime (Gemini Live) |
|---|---|---|
| **Modele** | 3 osobne | 1 multimodalny natywny |
| **Latencja** | Wyższa (3 skoki pipeline) | Najniższa (native audio I/O) |
| **Modalności** | Tekst jako pośrednik | Audio/tekst/obraz/wideo jednocześnie |
| **Koszt** | Niższy | Wyższy |
| **Zastosowanie** | Proste interakcje, polecenia | Zaawansowane interakcje wielomodalne |

Realtime overkill dla prostych zastosowań — koszt i złożoność bez proporcjonalnej korzyści.

Kluczowe zjawiska głosowe: detekcja ciszy (VAD), przerywanie agenta (barge-in), rozpoznawanie końca wypowiedzi użytkownika.

### LiveKit Agents — prewarm/entry

- **prewarm** — ładuje ciężkie zasoby RAZ: Silero VAD (neural network do detekcji końca mowy), połączenia MCP, modele. Eliminuje latencję startową
- **entry** — per sesja użytkownika, korzysta z załadowanych zasobów (warm start)

Transport: WebRTC. Token JWT krótkożyciowy (10 min) z grantami `roomJoin + canPublish + canSubscribe`. Osobny mikroserwer (Hono) generuje tokeny per sesja. Browser blokuje autoplay — `room.startAudio()` musi być w click handlerze.

### Integracja MCP

Narzędzia MCP konwertowane dynamicznie: JSON Schema → Zod (rekursywna konwersja `toZod` z enum, anyOf/oneOf, nested objects). `mcp.callTool()` owinięte w `llm.tool()`. Prefiksowanie `serverName__toolName` — bezkolizyjne scalenie z wielu źródeł.

`Promise.allSettled` — graceful degradation: jeden MCP pada, agent działa dalej bez jego narzędzi.

### Multi-provider voice stack

Mode resolve z env → wybór komponentów: Gemini (jeśli klucz) > OpenAI+ElevenLabs > OpenAI fallback. Determinowany dostępnością kluczy, nie konfiguracją.

### Custom TTS pipeline

ElevenLabs zwraca MP3, LiveKit wymaga PCM. Wzorzec: WebSocket → ffmpeg stdin → PCM stdout → `AudioByteStream`. `FLUSH_SENTINEL` wymusza wysłanie tekstu oczekującego w buforze. Klasyczny process pipeline w Node.js.

### Gemini Live workaround

`autoToolReplyGeneration: true` powoduje deadlock w pętli speech-queue drain LiveKit. Wymagany override gettera `capabilities` z `autoToolReplyGeneration: false` — ręczne generowanie tool-reply omija problem. Ilustruje, że integracje z beta API wymagają takich sztuczek.

## Spektrum wyszukiwania — drabina decyzyjna

Od najprostszego do najbardziej zaawansowanego, każdy poziom usprawiedliwiony konkretną potrzebą:

| Podejście | Kiedy | Złożoność |
|---|---|---|
| Wczytanie do kontekstu | Małe korpusy, priorytet szybkości | Minimalna |
| Pliki + grep/ripgrep | Agent nawigujący po FS, jeden język/format | Niska |
| Hybrydowe (FTS + semantyczne) | Grep niewystarczający, synonimy/rozumienie | Średnia |
| Grafy | Zaawansowane indeksowanie, mapowanie relacji | Wysoka |

Samodzielne bazy wektorowe **nierekomendowane** — łączyć FTS + semantyczne. Claude Code = grep. Cursor = grep + semantyczne. Nie ma uniwersalnego podejścia.

Rozszerzenia vs dedykowane bazy:
- SQLite fts5 + sqlite-vec / PostgreSQL — wystarczają na większość projektów
- Qdrant, Chroma — dopiero przy skali, wydajności, zaawansowanym filtrowaniu
- Algolia, Elasticsearch — pełnoprawny silnik wyszukiwania

Rozszerzenia i dedykowane rozwiązania mają niemal identyczny proces, ale drugie = nieporównywalnie większa złożoność architektury. Musisz mieć realny powód.

## Budowanie własnych komponentów

Powtarzalne schematy w pracy z AI do zamknięcia w reużywalne formy:

- **Prompty** — biblioteka instrukcji na skróty/makra. Nawet kilka szablonów zwraca inwestycję
- **CLI/MCP** — integracje w standardową formę = przenaszalność między agentami. Zacznij od jednej
- **Zarządzanie plikami** — FS, Notion, Google Drive. Agent bez dostępu do dokumentów drastycznie ograniczony
- **Dostęp do chmury** — upload/download obrazów i plików (Uploadthing lub własny serwer)
- **Sandbox** — izolowane środowiska dla ryzykownych akcji (alternatywa: dedykowany fizyczny komputer)
- **Generowanie dokumentów** — szablony auto-uzupełniane przez LLM
- **Przetwarzanie media** — obraz/audio/wideo (np. Audiopen — prosty, dopracowany przykład)
- **Interfejs** — własny chat UI jako panel zarządzania, sensowny przy wysokiej personalizacji

Brakujące elementy UI w produkcji: załączniki, kopiowanie z formatowaniem, rozgałęzianie/edycja wiadomości, usuwanie, wstrzymywanie generacji, przeszukiwanie konwersacji, wysyłanie podczas inferencji, LaTeX (KaTeX), formatowanie (TipTap), diagramy (Mermaid), mapy myśli (Markmap), udostępnianie wątków, subagenci, skróty klawiszowe.

### 05_02_ui — Chat UI z event sourcing

**Delta:** pierwszy przykład produkcyjnego streaming UI z pełnym pipeline'em Markdown i wirtualizacją.

Serwer: czysty `node:http`, routing po pathname + method, SSE z abort/close. Agent loop server-side: query → text + tool_calls → execute tools → results → continue (max 6 kroków), `parallel_tool_calls: true`.

```ts
// materialize.ts — inkrementalna materializacja O(1)
export const applyEvent = (blocks: Block[], event: StreamEvent, toolIndexById: Map<string, number>): void => {
  switch (event.type) {
    case 'text_delta': {
      const lastBlock = blocks[blocks.length - 1]
      if (lastBlock?.type === 'text') { lastBlock.content += event.textDelta; break }
      blocks.push(createTextBlock(event.id, event.at, event.textDelta))
      break
    }
    case 'tool_call': {
      toolIndexById.set(event.toolCallId, blocks.length)
      blocks.push({ type: 'tool_interaction', status: 'running', ... })
      break
    }
    case 'tool_result': {
      blocks[toolIndexById.get(event.toolCallId)!].status = event.ok ? 'complete' : 'error'
      break
    }
  }
}
```

```ts
// streaming-markdown.ts — committedSegments + liveTail
export const syncIncrementalMarkdownView = (current, { content, streaming }) => {
  const delta = content.slice(current.processedContent.length)
  const candidateTail = current.liveTail + delta
  const blocks = parseMarkdownIntoBlocks(candidateTail)
  const { finalizedSources, liveTail } = splitBlocks(blocks, streaming)
  return {
    committedSegments: current.committedSegments.concat(
      toSegments(blockId, current.nextSegmentIndex, finalizedSources)
    ),
    liveTail, processedContent: content,
    nextSegmentIndex: current.nextSegmentIndex + finalizedSources.length,
  }
}
```
