# S01E03 — Projektowanie narzędzi i MCP dla agentów

## Audyt API przed projektowaniem narzędzi

Zanim powstanie pierwsze narzędzie, zweryfikuj API pod kątem:

- **Kompletność akcji** — czy CRUD pokrywa operacje potrzebne agentowi
- **Identyfikacja zasobów** — model operuje nazwami ("etykieta Priorytet"), API identyfikatorami. Mapowanie nazwa → ID leży po stronie narzędzia, nie modelu
- **Spójność kontraktu** — `content` vs `body`, `id` vs `ID` w różnych endpointach wymaga normalizacji
- **Kompletność odpowiedzi** — samo `201 Created` bez zwrotu zasobu zmusza model do dodatkowego GET
- **Złożoność wieloetapowa** — jedno zadanie użytkownika wymagające 3+ wywołań kandyduje do agregacji w jedno narzędzie (np. promocja = create + assign)
- **Asynchroniczność, rate limit, paginacja** — polling, kolejki, retry, precyzyjne filtry obsługuj kodowo; nie zmuszaj agenta do pętli oczekiwania

Najlepsze podejście: wgraj SDK/dokumentację API do kontekstu agenta kodującego i wspólnie zaplanuj strukturę narzędzi.

## Konsolidacja narzędzi

Nie mapuj 1:1 endpointów API. Grupuj po domenie działania. Parametr `mode`/`operation` konsoliduje powiązane akcje; wspólna struktura parametrów (część ignorowana w zależności od trybu) upraszcza schemat.

Przykład: oficjalny Filesystem MCP (13 narzędzi) → zoptymalizowana wersja (4):

| Narzędzie | Odpowiedzialność |
|---|---|
| `fs_search` | Przeszukiwanie po nazwie/treści, drzewo katalogów, metadane |
| `fs_read` | Tekst/binarne/wiele plików, zakresy linii, automatyczne rozwiązywanie ścieżek |
| `fs_write` | Zapis i edycja z checksum, dryRun, historią zmian |
| `fs_manage` | mkdir, move, rename, delete (ograniczone do pustych katalogów/pojedynczych plików) |

Cel to balans, nie minimalizacja. Za mało narzędzi = wielkie schematy. Za dużo = zaśmiecenie kontekstu. Smoke test: mały model lokalny (Qwen, GPT-OSS 20B, GLM) — jeśli poprawnie obsługuje zestaw, interfejs jest dobrze zaprojektowany.

## Projektowanie interfejsu z perspektywy LLM

Wymiary kształtujące schemat narzędzia:

1. **Fundament API** — możliwości i ograniczenia podstawowe
2. **Przejrzystość** — proste nazwy, minimalna liczba parametrów, opcjonalne `details: true/false` zamiast osobnych wariantów verbose/compact
3. **Dopasowanie złożoności** do możliwości modelu — unikaj wieloetapowej orkiestracji w jednym handlerze
4. **Kompensacja ograniczeń modelu** — halucynacje, zapominanie → walidacja i checksumy po stronie kodu; ograniczenia uprawnień (`FS_ROOT`) egzekwowane programistycznie, nie promptem
5. **Use case** — adresuj potrzeby użytkownika, nie możliwości API

### Zabezpieczenia dla akcji destrukcyjnych

- **Checksum** — agent nie nadpisze pliku zmienionego w międzyczasie
- **dryRun** — podgląd wyniku bez wykonania
- **Historia zmian** — rollback bez angażowania modelu
- **Usuwanie** ograniczone do pojedynczych plików i pustych katalogów; koncepcja "kosza"

### Dynamiczne odpowiedzi (hints)

Każda odpowiedź narzędzia — sukces i błąd — niesie informację kontekstową, której model nie może wydedukować:

- **Błąd + recovery**: "Plik został zmieniony. Przeczytaj go ponownie."
- **Status zasobu**: "Dokument istnieje, ale jest chroniony przed zapisem."
- **Sugestia następnego kroku**: "Znaleziono 3 pliki. Wczytaj je przed edycją."
- **Dostępne opcje przy błędzie walidacji**: "Dostępne etykiety: X, Y, Z."
- **Korekta automatyczna**: "Żądano linii 48–70, plik ma 59. Wczytano 48–59."

Hinty mają sens nawet przy zaawansowanych modelach — niosą informację kontekstową (uprawnienia, stan zasobu), której model nie ma skąd poznać.

## Agent vs Workflow

| Wymiar | Workflow | Agent |
|---|---|---|
| Kontrola | Pełna, zdefiniowane kroki | Model decyduje o kolejności |
| Elastyczność | Niska | Wysoka — koryguje, weryfikuje, poprawia |
| Ryzyko | Niskie | Wyższe, ale z szansą na lepszy rezultat |

Kryterium: jeśli proces **zawsze** musi iść według ustalonych kroków → workflow. Jeśli kolejność zależy od kontekstu → agent.

## Model Context Protocol — architektura

### Role

- **Host** — aplikacja zarządzająca połączeniami (Claude Desktop, Cursor, backend bez UI)
- **Client** — pojedyncze połączenie z serwerem, część hosta
- **Server** — proces udostępniający capabilities (files-mcp, uploadthing MCP)

MCP jest komplementarny wobec natywnego function-callingu, nie alternatywny. Z perspektywy modelu schematy MCP i natywne scalają się w jedną listę narzędzi — źródło jest niewidoczne. MCP definiuje jedynie spójny sposób dostarczania i odkrywania narzędzi; raz stworzony serwer podłącza się do dowolnego hosta.

### Capabilities

| Capability | Kierunek | Opis |
|---|---|---|
| **Tools** | klient → serwer | Akcje wywoływane przez LLM |
| **Resources** | klient → serwer | Dane read-only, statyczne lub dynamiczne |
| **Prompts** | klient → serwer | Szablony wiadomości z parametrami, aktywowane przez użytkownika |
| **Sampling** | serwer → klient | Serwer prosi klienta o completion LLM |
| **Elicitation** | serwer → klient | Serwer prosi klienta o input użytkownika (formularz, potwierdzenie) |

Sampling i Elicitation odwracają kierunek komunikacji. Kluczowa konsekwencja Samplingu: **serwer MCP nie potrzebuje własnych kluczy API** — leverage'uje relację klienta z providerem AI. Klient deklaruje obsługiwane capability przy połączeniu.

### Transporty

| | STDIO | Streamable HTTP |
|---|---|---|
| Mechanizm | Subprocess, stdin/stdout | HTTP endpoint |
| Sesje | Nowy proces per połączenie | Wielosesyjny, multi-user |
| Auth | Klucze API w env | OAuth 2.1 (PKCE, RS tokens, auto-refresh) |
| Deployment | Lokalnie | VPS + nginx, Cloudflare Workers |

STDIO dla lokalnego systemu (pliki, ffmpeg). Streamable HTTP jako domyślny wybór dla serwerów produkcyjnych. Format MCPB pakietuje serwer STDIO w jeden plik z auto-konfiguracją.

### Routing wielu serwerów

Przy wielu serwerach jednocześnie host prefixuje nazwy narzędzi nazwą serwera, co eliminuje kolizje (np. `gmail__search` vs `resend__search`):

```
files__fs_read, uploadthing__upload_files
```

Routing: split po `__`, pierwsza część = nazwa serwera, reszta = oryginalna nazwa narzędzia.

### Deklaratywna konfiguracja (`mcp.json`)

Produkcyjny wzorzec: jeden JSON definiuje wszystkie serwery z mieszanymi transportami, komendami, argumentami, env vars i URL-ami. Klient wczytuje config, tworzy odpowiedni transport per serwer i prefixuje nazwy.

## Bezpieczeństwo i prywatność

- **Narzędzia wewnętrzne** — wysoka kontrola: znane zasoby, akcje, użytkownicy, przepływ danych
- **Publiczne serwery MCP** — brak widoczności otoczenia: nieznany host, pozostałe narzędzia, procesy agenta. Wymagane programistyczne ograniczenia: blokady dostępu, limity zapytań, walidacje, anonimizacja. Niektóre zasoby w ogóle nie powinny trafiać do agentów AI
- **Prompt injection** pozostaje otwartym problemem bez uniwersalnego rozwiązania

## Proces budowy serwera MCP

1. Pobierz szablon (np. Streamable MCP Template)
2. Utwórz `API.md` z dokumentacją docelowego API
3. Agent czyta `README.md` + `manual.md` szablonu
4. Agent proponuje listę narzędzi → zastosuj konsolidację
5. Zaprojektuj input/output z perspektywy LLM (hints, walidacja, sugestie)
6. Implementacja → weryfikacja małym modelem → iteracja

## Przykłady kodu — progresja

### `mcp_native` — fundament unifikacji

Uczy: MCP tools i natywne JS tools są izomorficzne z perspektywy agenta. Jedna mapa handlerów, jedna lista schematów, jedna pętla agentic (chat → extract tool calls → execute parallel → append results → repeat). Najprostsza forma transportu to `InMemoryTransport` — serwer i klient w jednym procesie. Konwersja schematów MCP → OpenAI function-calling format przez `mcpToolsToOpenAI`.

### `mcp_core` — delta: pełne capability MCP

Nowe: `StdioTransport` (subprocess jako wzorzec produkcyjny), Resources statyczne (`config://project`) i dynamiczne (`data://stats`), Prompts z parametrami i Zod schema. Kluczowa demonstracja — jeden handler łączy Elicitation i Sampling w jednym flow:

```js
const confirmation = await extra.sendRequest(confirmationForm(text), elicitationResponseSchema);
if (confirmation.action !== "accept") return textResult("Cancelled");
const result = await extra.sendRequest(samplingMessage(text, style), samplingResponseSchema);
```

Separacja: serwer = logika domenowa, klient = infrastruktura AI. Klient konwertuje MCP sampling request na Responses API.

### `mcp_translator` — delta: realny agent autonomiczny

Nowe: podłączenie do **zewnętrznego** serwera MCP (`files-mcp`) zamiast własnego, konfiguracja z `mcp.json`, file watcher jako źródło zdarzeń (polling katalogu co 5s), HTTP API jako alternatywny interfejs (`/api/translate`, `/api/chat`), token tracking z Responses API (`input_tokens`, `cached_tokens`, `output_tokens`). Agent dostaje cel ("przetłumacz plik") i decyduje jak — scan → plan → translate → verify zdefiniowane w system prompcie, ale wykonanie należy do modelu. Pętla agentic z **max iterations jako safety guard** (80 kroków), niezależny stan per plik (Set inProgress). `strict: false` na tool definitions, bo zewnętrzne schematy MCP mogą nie spełniać wymagań strict mode.

### `upload_mcp` — delta: multi-server, multi-transport

Nowe: jednoczesne połączenie STDIO (`files`) + StreamableHTTP (`uploadthing`), routing po prefiksie:

```js
const [serverName, ...toolParts] = name.split("__");
const result = await clients[serverName].callTool({ name: toolParts.join("__"), arguments: args });
```

Wzorzec `{{file:path}}` — model deklaruje intencję odczytu pliku placeholderem zamiast kodować base64 w argumentach; resolver rekurencyjnie podmienia placeholdery na base64 z dysku przed wywołaniem MCP tool. Fallback na raw request gdy `listTools()` failuje na niestandardowym `outputSchema` (bypass Zod). `ConfigurationError` z instrukcjami naprawczymi dla błędów konfiguracji HTTP.
