# S03E05 — Niedeterministyczna natura modeli jako przewaga

## Paradoks determinizmu

LLM nie jest ani deterministyczny (te same dane wejściowe → różne wyniki), ani wystarczająco losowy (często powtarza wzorce). `temperature`/`top_p` ma znaczenie marginalne — zachowanie jest zdeterminowane przez **poprzedzającą treść**. Różnorodność wzrasta dopiero gdy do gry wchodzi pamięć, otoczenie i stan sesji.

LLM to "śniąca maszyna" (Karpathy) — wszystko jest "halucynacją" która czasem sprzyja. Precyzyjne podążanie za instrukcjami i generowanie zróżnicowanych odpowiedzi to dla modelu **równie trudne**. Zamiast walczyć z niedeterminizmem — zaprojektować przestrzeń, w której staje się zaletą.

## Od poleceń do warunków

| Wymiar | Agent skryptowy | Agent "świadomy" |
|--------|----------------|------------------|
| Trigger | bezpośrednie polecenie | niejawne potrzeby, kontekst sytuacyjny |
| Decyzje | reguły w kodzie | model ocenia luki i dobiera akcje |
| Kontekst | wczytywany eagerly | lazy loading — co potrzebne, kiedy potrzebne |
| Identyczny input | przewidywalny | bardzo niskie prawdopodobieństwo powtórzenia |

Oddelegowanie decyzji o *kiedy* i *jak* działać, przy zachowaniu **wyznaczonej przestrzeni** (narzędzia, dane, metadane). Ryzyko "utraty kontroli" jest pozorne — model operuje w ramach mechanik, przestrzeń jest większa ale nie nieograniczona.

## Cztery obszary oddelegowane do modelu

| Obszar | Istota oddelegowania |
|--------|---------------------|
| **Proaktywność** | Model sam określa kiedy działać — na podstawie podejrzeń, nie poleceń |
| **Synteza** | Łączenie informacji bez "jeśli X to Y" — dynamiczne dopasowanie do sytuacji |
| **Wnioskowanie** | Narzędzia (`think`, `recall`) tworzą przestrzeń na refleksję; obecność wpływa na zachowanie |
| **Dopasowanie** | Model decyduje o fokusie uwagi, stylu, formie — mechaniki wzmacniają kategorie, nie akcje |

## Architektura kognitywna agenta

Przestajemy projektować zachowania. Stwarzamy **warunki**, w których powstają samoistnie — odchodzi to od prompt engineering w stronę cognitive architecture (ref. ["Cognitive Architectures for Language Agents"](https://arxiv.org/pdf/2309.02427)). Nie wyklucza kierunku branży (eliminacja halucynacji, śledzenie instrukcji) — wprowadza **zmienność** pozwalającą na odkrywanie ścieżek nieprzewidzianych na etapie projektowania.

Warstwy budowane w instrukcjach:

- **Tożsamość** — agent wie że ma osobowość/nastrój/opinie, ale pozostają **rozmyte** dopóki nie zostaną odkryte
- **Zdolności poznawcze** — wpływ na zachowanie przez zadawanie pytań, nie przez reguły
- **Inteligencja emocjonalna** — odczytywanie stanu rozmówcy, dopasowanie gdy użytkownik **jeszcze nie wie** czego szuka
- **Sposoby ekspresji** — dopasowanie stylu sterowane **metadanymi** wstrzykiwanymi do wiadomości, nie promptem
- **Mechaniki wzmacniające** — w metadanych i wynikach narzędzi pojawiają się **postawy** (nie akcje), które model bierze pod uwagę

Instrukcje **kierują "snem" modelu** bez bezpośrednich poleceń. Krytyczna cecha: są **zgeneralizowane** — opisują postawy i zasady, nie procedury. Modele słabiej radzą sobie z tym stopniem abstrakcji (nawet modele z rozwiniętym PE). Rola człowieka jako projektanta warunków pozostaje kluczowa — LLM jest przydatny jako partner myślowy, ale generalizacja pozostaje zadaniem projektanta.

Podstawa neurokognitywna: modele wykorzystują [Theory of Mind](https://arxiv.org/pdf/2505.00026) — obserwowaną od GPT-4, znacznie rozwiniętą w obecnych modelach. Umożliwia "czytanie między słowami", odczytywanie intencji, dopasowywanie odpowiedzi.

## Narzędzia jako cognitive scaffolding

### `think` — wymuszona pauza na refleksję

Zero I/O. Zmusza model do sformułowania pytań do samego siebie — zauważa luki **zanim** odpowie. Realizacja chain-of-thought jako explicit tool, nie prompting technique. Wynik wraca do kontekstu jako `function_call_output`, wpływając na kolejne kroki. Mechanizm typu "nudge" — obecność zmienia zachowanie nawet gdy nie zostanie wywołane w danej turze.

### `recall` — delegacja do scout sub-agenta

Główny agent nie wie jakie pliki istnieją — scout odkrywa je na podstawie celu (goal). Scout ma własną sesję, system prompt i MCP file tools. **Oddzielenie wnioskowania (main agent) od eksploracji danych (scout)** — celowe ograniczenie: max 8 tur, czyta tylko to co odpowiada celowi. Sesja scouta persistuje across turns, parametr `new_session` resetuje przy zmianie tematu.

## Temporal grounding

Metadane czasowe wstrzykiwane w **każdą** wiadomość — fundamentalny building block dla agentów "świadomych":

```xml
<metadata>
  now_iso: 2026-03-30T10:53:19.897Z
  weekday: Monday
  local_time: 12:53:19
  timezone: Europe/Warsaw
  recallable: persona, user_identity, user_preferences, ...
  nudge: think before you respond; recall when the topic shifts...
</metadata>
```

Agent wie **kiedy** jest, **co może odkryć**, dostaje **nudge** zachęcający do aktywności. Metadane sterują uwagą bez modyfikacji promptu systemowego — bezpieczne dla cache, zmieniają się co turę.

## Workspace jako pamięć kognitywna

Trzy kategorie pamięci analogiczne do kognitywnych systemów:

| Kategoria | Lokalizacja | Zawartość |
|-----------|-------------|-----------|
| **Epizodyczna** | `memory/episodic/` | Oznaczone czasem snapshoty interakcji |
| **Faktyczna** | `memory/factual/` | Trwałe fakty persystentne między sesjami |
| **Proceduralna** | `memory/procedural/` | Reguły behawioralne wyuczone w czasie |

Ponadto `profile/user/` (identity, preferences, important-dates) i `profile/agent/` (persona) jako warstwa tożsamości. `system/index.md` to mapa nawigacyjna dla scouta — agent główny nigdy nie widzi struktury plików bezpośrednio.

### External signal enrichment

Scout integruje pogodę (Open-Meteo API) gdy cel lub wiadomość zawiera weather keywords. Wzorzec: scout nie tylko czyta pliki, ale **wzbogaca kontekst o sygnały z zewnętrznych API**. Lokalizacja z `profile/user/identity.md`, wynik jako `Weather hint` w kontekście scouta.

## Spektrum generowania interfejsów przez LLM

| | Artifacts | Render | MCP Apps |
|---|---|---|---|
| **Co generuje model** | Dowolny HTML/CSS/JS | JSON spec (katalog komponentów) | Nic — UI predefiniowany |
| **Kontrola** | Niska | Średnia (guardrails prompt + validation) | Pełna |
| **Stan** | Efemeryczny | Inicjalizowalny, zapisywalny, wczytywalny | Dwukierunkowa synchronizacja |
| **Interakcja** | Tylko w iframe | Tylko w iframe | Akcje usera → MCP tools → system |
| **Widoczność dla modelu** | Pełna (widzi kod) | Pełna (widzi spec) | Brak — widzi tylko udostępnione dane |
| **Rola hosta** | Standardowa | Standardowa | **Zwiększona** — obsługuje UI + agenta |
| **Produkcyjna gotowość** | Niska | Średnia | Wysoka |

Dyskusję "X vs Y" zamień na "kiedy X, kiedy Y, kiedy X+Y". Tryby rozbudowanego rozumowania (`reasoningEffort`) pozwalają modelom radzić sobie ze złożonymi interfejsami generowanymi w locie.

**Przesunięcie roli projektanta:** z ręcznego budowania detali w stronę projektowania **struktur**, po których poruszają się agenci. Balansowanie przewidywalności (stabilne realizowanie procesów) z dynamiką (możliwości generatywnego AI).

### Artifacts — pełna swoboda generatywna

- **Capability packs** — 11 pakiety (Preact+HTM, Chart.js, D3, Tailwind, Zod, Day.js, DOMPurify, PapaParse, XLSX, SheetJS, core helpers). Podwójny tryb: inline (self-contained) vs external (`<script src>` z 24h cache). Katalog packs jawny w prompcie — model **sam wybiera** które wykorzystać
- **4-fazowy pipeline** — dane testowe → rozpoznanie (model decyduje o wizualizacji) → generowanie (HTML w iframe) → optymalizacja (search/replace zamiast regeneracji)
- **Deterministic editing** — `edit_artifact` wykonuje search/replace przez regex. Model decyduje CO, system deterministycznie aplikuje. Szybsze, tańsze, przewidywalne niż regeneracja
- **CSP** — strict: `default-src 'none'`, inline tylko z preview server
- **ArtifactKit** — capability pack `core` wstrzykuje `window.ArtifactKit` z `createStore` (reaktywne store), `q`/`qa` (querySelector helpers) — interaktywne UI bez frameworka

### Render — deterministyczny rendering ze specyfikacji

- Model generuje **JSON** (`{title, spec, state}`) — spec definiuje drzewo komponentów, state dostarcza dane. 19 komponentów (Stack, Grid, Card, Heading, Metric, LineChart, BarChart, Table, Alert, Accordion, Input, Select, itd.)
- **Podwójna walidacja** — manifest w system prompt + Zod schema + `normalizeSpec()`. Limit elementów (120), detekcja cykli, depth limit (40). Nieznane komponenty → odrzucenie
- **JSON Pointer state binding** — `{"$state": "/kpis/0/value"}` deklaratywne bindowanie, resolve'owane server-side
- **Edit-as-regenerate** — cały dokument serializowany do prompt, model regeneruje z modyfikacjami. Prostsza logika, więcej tokenów

### MCP Apps — dwukierunkowa komunikacja przez protokół

- Model nie widzi kodu UI — wywołuje narzędzia, serwer MCP obsługuje z wbudowanym interfejsem (`registerAppResource`)
- **Tool visibility scoping** — narzędzia `get_lists_state`/`save_lists_state` mają `visibility: ["app"]` — ukryte przed modelem, dostępne tylko dla UI
- **Structured content** — narzędzia zwracają `content` (dla agenta) + `structuredContent` (maszynowy, dla UI)
- **Dual-mode UI** — jeden HTML bundle działa standalone (REST) i embedded w MCP host (iframe). Detekcja: `window.parent === window`
- **Single-file bundle** — `vite-plugin-singlefile` inlinuje JS/CSS do jednego HTML serwowanego jako MCP resource z MIME `text/html;profile=mcp-app`

## Wybór technologii pod kątem capabilities modelu

Tailwind v3 celowo zamiast v4 — modele znacznie lepiej się nim posługują. Przy artefaktach nie podążamy za "najlepszymi standardami", lecz wybieramy narzędzia **optymalne dla capabilities modelu**.

## Progresja wzorców

1. **Awareness** — metacognitive tools, scout sub-agent, workspace jako cognitive memory. Brak narzędzi operacyjnych — model sam decyduje o potrzebie kontekstu
2. **Artifacts** — pełny HTML z capability packs. Edit przez search/replace (delta, nie regeneracja). Live preview przez WebSocket
3. **Render** — JSON spec z katalogu komponentów. Server-side rendering. Stan zapisywalny/odczytywalny
4. **MCP Apps** — gotowy UI sterowany przez protokół MCP. Model izolowany od kodu. Dwukierunkowa synchronizacja

Każdy krok ogranicza niedeterminizm inaczej: awareness — przez generalizację instrukcji, artifacts — przez capability packs, render — przez katalog komponentów i walidację, apps — przez predefiniowany UI.

## Przykłady kodu — delta względem poprzednich lekcji

### extractJsonCandidates — resilentne parsowanie outputu LLM

3 strategie, pierwsza która parsuje wygrywa: raw text → fenced code block → first `{` to last `}`. Reusable pattern dla każdego pipeline'u z LLM.

### YAML+MD template agenta

```yaml
---
name: awareness
model: gpt-5.2
tools: [think, recall]
---
[system prompt as markdown body]
```

Narzędzia deklarowane we frontmatter resolve'owane — tylko wymienione trafiają do modelu. Scout dostaje narzędzia MCP dynamicznie (brak deklaratywnych).

### Observability — JSONL trace

`responses-loop` zapisuje trace do `workspace/traces/{date}.jsonl` — request, response, tool result z timestampem, modelem, statusem, preview. Label odróżnia sesje agenta od scouta.

### Fallback document (render)

Gdy API key brakuje, render generuje lokalny dokument z syntetycznymi danymi — system degraduje gracefully, nie crashuje.
