# s01e04 — Multimodalność i załączniki w agentach

## Problem załączników między narzędziami

LLM dostaje obraz/audio/wideo jako Base64 lub URL, ale **nie widzi samego adresu** — nie potrafi go przekazać do kolejnego narzędzia ani do innego agenta. Dotyczy wszystkich modalności i nie ma branżowego standardu rozwiązania.

Autorska konwencja: w wiadomości użytkownika obok tekstu i pliku umieszczamy programistycznie trzeci element — tag referencyjny (`<media src="path/to/file">`). Instrukcja systemowa tłumaczy agentowi jak go używać. Przed wywołaniem narzędzia resolver zamienia referencję na Base64 lub publiczny URL zależnie od wymagań odbiorcy.

## Agent vs Workflow — kiedy co

| Kryterium | Workflow | Agent |
|---|---|---|
| Dane | Statyczne, znane z góry | Zmienne, niepewne |
| Klasyfikacja | Prosta | Dynamiczna |
| Instrukcja | Sztywna sekwencja kroków | Cel + limity + zasady rozumowania |
| Skuteczność | Wysoka dla przewidywalnych | <100%, elastyczna |

Instrukcja agenta **nie może** zależeć od konkretnego zestawu danych — tylko od **klasy problemów**. Workflow i agent to nie wybór "albo-albo": workflow może być jednym z narzędzi agenta, agent może delegować do workflow gdy wykryje stabilny podproces.

Przeniesienie logiki na model oznacza ryzyko błędów — weryfikacja przez człowieka jest wpisana w model kosztu.

## Vision jako narzędzie w pętli

Agent nie widzi natywnie outputu narzędzi wizualnych — wygenerowanych obrazów, wykresów, PDF-ów, również tych które tworzy pisząc kod (matplotlib, graphviz). Potrzebuje dedykowanego narzędzia `analyze_image` (Base64 + mimeType + question → tekst) jako osobnego kroku.

Pętla: **generuj → analizuj → accept/retry**. Analiza sprawdza prompt adherence, artefakty, anatomię, text rendering, spójność stylu, kompozycję. Werdykt RETRY tylko przy blocking issues (zła poza, złamana kompozycja, nieczytelny tekst); drobne niedoskonałości = ACCEPT — bez tego agent wpada w nieskończoną pętlę poprawek.

## Halucynacje wizualne

Modele generujące obrazy halucynują analogicznie do tekstowych: ~95% poprawnych detali, ~5% subtelnych glitchy (np. błędny napis na ostatniej klatce komiksu). Subtelne błędy są trudniejsze do wyłapania niż oczywiste. Skuteczność agenta nigdy nie będzie 100% — weryfikacja ludzka jest konieczna, ale zawsze łatwiejsza niż praca od zera.

## Szablony JSON jako prompty

Strukturyzowanie promptu do generowania obrazów jako JSON daje:

- **Precyzyjną kontrolę** — agent edytuje tylko sekcję `subject`, reszta (styl, paleta, oświetlenie, kompozycja, negative prompt) zostaje nietknięta
- **Powtarzalność stylu** między generacjami i scenami
- **Oszczędność tokenów** — agent przekazuje ścieżkę do pliku zamiast przepisywać pełny prompt
- **Wersjonowanie** — każda generacja ma własny plik

Workflow: **klonuj template → edytuj fragment kopii → przekaż ścieżkę do narzędzia**. Szablon jest master reference i nigdy nie jest edytowany bezpośrednio. Agent może też generować szablon JSON z obrazów referencyjnych, "ucząc się" stylu.

## Obrazy referencyjne

Referencje przekazywane jako `reference_images` sterują kompozycją, pozą, kadrem. Zastosowania: spójność postaci między scenami, in/out-painting, zachowanie pozy. Połączenie z JSON promptem — JSON określa styl i scenę, referencja determinuje kompozycję.

Agent **wnioskuje** odpowiednią referencję z opisu użytkownika ("warrior charging" → `running-pose.png`) zamiast wymagać dokładnej nazwy pliku. W tworzeniu dłuższych filmów: ostatnia klatka jednego segmentu staje się pierwszą klatką następnego.

## Audio

Trzy podejścia architektoniczne: rozdzielone TTS/STT, modele multimodalne, interakcja real-time. Wybór zależy od ceny, jakości i latencji — nie ma jednego najlepszego.

Modele audio rozumieją nie tylko treść, ale ton, emocje, dźwięki otoczenia; **diaryzacja** (rozpoznawanie rozmówców) działa bez dedykowanego treningu. Gemini obsługuje transkrypcję z timestampami i emocjami, analizę strukturyzowaną przez `response_schema`, TTS single/multi-speaker z kontrolą stylu przez naturalny język wstawiany w tekst ("Say cheerfully:", "In a whisper:").

Pliki >20 MB wymagają resumable upload (Gemini Files API); YouTube URL-e działają bezpośrednio jako `file_uri`.

**UX audio**: styl odpowiedzi agenta musi być dostosowany do medium — bez dyktowania URL-i, tabel, formatowania markdown.

## Wideo — kontrola kosztów i generowanie

Najdroższa modalność: ~300 tokenów/sekundę, 1h filmu ~1M tokenów. Mechanizmy redukcji:

- **Clipping** (`start_time`, `end_time`)
- **FPS** — mniej klatek = mniej tokenów
- **Przyspieszenie nagrania** gdy nie tracą się istotne detale

Analiza (Gemini) działa bezpośrednio na pliku lub YouTube URL, bez podziału na klatki. Tryby ekstrakcji: sceny, keyframes, obiekty, tekst — każdy ze swoim schematem JSON.

**Generowanie** (Kling przez Replicate) — wzorzec maksymalnej kontroli:
1. Wygeneruj klatkę startową z JSON promptu
2. Wygeneruj klatkę końcową z referencją do startowej (spójność postaci)
3. Przekaż obie + prompt opisujący ruch do `image_to_video`

Kling ograniczony do ~10s/segment, ale mechanizm klatka końcowa → klatka początkowa pozwala łączyć dowolnie długie sekwencje. Polling asynchroniczny — operacje trwają minuty.

## Dokumenty PDF

Asymetria: **generowanie** HTML → Puppeteer → PDF jest relatywnie proste, **przetwarzanie** istniejących PDF-ów to osobny poważny problem (dedykowane API, LlamaIndex, Claude PDF support).

Wzorzec generowania: agent czyta `template.html` + `style-guide.md`, klonuje szablon, modyfikuje `<body>`, osadza wygenerowane obrazy (ścieżki absolutne), konwertuje. Może iteracyjnie poprawiać fragmenty bez przerabiania całości.

## Konfiguracja runtime przez pliki

Zasady stylu nie są zaszyte w instrukcji systemowej — agent czyta `style-guide.md` z systemu plików przed pierwszą akcją. Ten sam wzorzec dla `template.html` i `template.json`. Oddziela reguły od kodu agenta; użytkownik zmienia zachowanie edytując markdown/HTML/JSON bez dotykania instrukcji.

## Architektura przykładów

Wszystkie 8 przykładów współdzielą strukturę: `agent.js` (pętla), `api.js` (klient Responses API), `config.js` (modele + system prompts), `mcp/` (klient MCP dla FS), `native/` (vision, generowanie, audio, wideo, HTML→PDF), `repl.js`. Dyspozytor: `isNativeTool(name)` → handler natywny, reszta → MCP. Z perspektywy modelu schematy scalają się w jedną listę.

## Przykłady — progresja

### `image_recognition` — fundament i filozofia instrukcji
Bazowa architektura: `chat → extractToolCalls → runTools → repeat` z `MAX_STEPS`. Narzędzie `understand_image` czyta plik → Base64 → Vision API. Brak REPL — jedno query uruchamia klasyfikację.

Demonstruje **instrukcję opartą na regułach rozumowania, nie krokach procesu**: jak zbierać dowody (tylko obserwowalne cechy), jak dopasowywać (profile = wymagania minimalne), jak radzić sobie z niejednoznacznością (brak dopasowania → `unclassified`), jak oceniać kompozyty (każdy obiekt osobno). Realizacja zasady "instrukcja zależna od klasy problemów".

### `image_editing` — pętla weryfikacji
REPL z historią konwersacji. Narzędzie `analyze_image` zwraca ustrukturyzowany werdykt parsowany z tagów:

```javascript
verdict: rawVerdict === "RETRY" ? "retry" : "accept",
blockingIssues: extractBulletSection(analysis, "BLOCKING_ISSUES"),
nextPromptHints: extractBulletSection(analysis, "NEXT_PROMPT_HINT")
```

Style guide jako plik czytany przez agenta przed akcją (augmented function call).

### `json_image` — szablony jako prompt
Dwa szablony (`template.json`, `character-template.json`) z locked sections. Agent edytuje wyłącznie `subject.main` i `subject.details`. Workflow: `COPY → EDIT subject only → READ full JSON → PASS path to tool`.

### `image_guidance` — referencje wizualne
Mandatory reference — agent odmawia generowania bez grafiki referencyjnej. Inferencja pozy z opisu użytkownika, dopasowanie do pliku w `workspace/reference/`. Najczystsza struktura kodu: każde narzędzie w osobnym folderze (`definition.js`, `handler.js`).

### `reports` — kompozycja narzędzi w dokument
Trzecie narzędzie `html_to_pdf` (Puppeteer). Szablon HTML z embedded CSS (dark theme, Google Fonts, komponenty na metryki/statusy/tagi). Agent klonuje, modyfikuje, osadza obrazy, konwertuje.

### `audio` — nowa modalność
Cztery dedykowane narzędzia: `transcribe_audio`, `analyze_audio`, `query_audio`, `generate_audio`. Resumable upload >20MB. Multi-speaker TTS z przypisywaniem głosów (30 dostępnych, kontrola stylu naturalnym językiem).

### `video` — analiza
YouTube URL jako źródło bezpośrednie. Cztery typy ekstrakcji (scenes/keyframes/objects/text) z własnymi schematami. `video_metadata` (fps, start/end) sterowane parametrami dla kontroli kosztów.

### `video_generation` — kulminacja
Pipeline łączący wszystko: JSON template → klatka start → klatka end → animacja Kling. Agent sam decyduje czy klatki wymagają poprawek przed animacją. Polling asynchronicznej operacji z retry do `maxAttempts`.
