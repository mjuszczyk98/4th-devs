# S02E05 — Projektowanie agentów

## Architektura instrukcji systemowej

Instrukcja agenta to zbiór sekcji o odmiennych odpowiedzialnościach i cyklach życia — od statycznego (identity) po dynamicznie generowany w runtime (workspace).

| Sekcja | Cel | Cykl życia |
|--------|-----|------------|
| **Identity** | Persona — charakter, styl, zachowanie | Stały |
| **Protocol** | Zasady działania w środowisku | Stały |
| **Voice** | Ton wypowiedzi | Stały (rozbudowany) |
| **Tools** | Narzędzia i zachowania specyficzne | Dynamiczny |
| **Workspace** | Kontekst sesji, observational memory | Dynamiczny |
| **CTA** | Sygnał zakończenia promptu | Stały |

**Identity** — motyw przewodni łączący charakter, styl i zachowanie. Szeroka perspektywa bez detali technicznych. Celowe użycie słów o zabarwieniu emocjonalnym ("instynkt", "wyczucie") kieruje uwagę modelu analogicznie do "nie myśl o niebieskich motylach" — fokusuje na określonych wzorcach zachowania silniej niż dosłowne instrukcje. Paradygmat: zaprojektować przestrzeń, by model **pozytywnie zaskoczył**. Halucynacje jako potencjalny atut — prompting przez skojarzenia na swoją korzyść, nie przez bezpośrednie instrukcje.

**Protocol** — zarządzanie kontekstem, wspomnieniami, relacjami, reakcje na błędy. Może zawierać ścieżki/pliki, ale z balansem — nie wiąże agenta ze strukturą katalogów na stałe. Kluczowe: co gdy agent nie potrafi coś zrobić? Jak prosić o pomoc?

**Voice** — wymaga rozbudowanej sekcji, bo LLM szybko zapominają ton i wracają do domyślnego. Narzędzia: przykłady wyrażeń, serie skojarzeń, antywzorce, few-shot. Środowisko determinuje styl — agent głosowy unika dyktowania linków czy "wyświetlania" obrazów.

**Tools** — schematy narzędzi opisują się same, sekcja niemal w pełni dynamiczna. Wyjątek: zachowania specyficzne (np. nie mylić powiadomienia z komunikatem). Przy progressive disclosure — agent musi wiedzieć, jak szukać narzędzi.

**Workspace** — observational memory i dane sesyjne generowane w runtime. Na końcu — jedno zdanie CTA zapobiegające "doklejaniu" przez model.

## Sześć obszarów projektowania

| Obszar | Co definiuje | Cykl życia |
|--------|-------------|------------|
| Ustawienia | Nazwa, opis (po nim inni agenci wywołują), narzędzia, model, uprawnienia | Stały |
| Profil | Osobowość, złożoność, format, techniki rozumowania | Stały |
| Zasady | Komunikacja, obsługa błędów, dostęp do wiedzy, zachowania awaryjne | Stały |
| Limity | Aktualność wiedzy ("kiedy jest teraz?"), dynamiczne uprawnienia | Mieszany |
| Styl | Sposób wypowiedzi zależny od środowiska (tekst/głos/GUI) | Stały |
| Sesja | Użytkownik, preferencje, aktywność agentów, dane dynamiczne | Dynamiczny |

## Strategia narzędzi

"10-15 narzędzi to max" to mit. Liczba zależy od złożoności integracji — jedno narzędzie (CLI do GitHub) zastępuje kilkanaście API calls, trzy do prostego serwisu mogą nie wystarczyć.

**Progressive disclosure** — agent dostaje meta-narzędzia do odkrywania reszty w runtime. Nie ładuje wszystkiego do kontekstu na start. Oszczędza okno kontekstowe.

**Współdzielenie narzędzi** między agentami zmniejsza wymianę informacji, ale rodzi problem: agent z ograniczonym dostępem do pamięci, nie znajdując informacji, **zakłada że nie istnieje** zamiast poprosić o dostęp.

**Pytanie filtrujące**: czy system stanie się lepszy z lepszymi modelami? Jeśli nie — budujemy niewłaściwą rzecz.

## Sandbox z progresywnym odkrywaniem

Agent startuje z minimum meta-narzędzi, odkrywa narzędzia docelowe dynamicznie, **pisze kod** i uruchamia w izolowanym środowisku. Dane funkcjonują jako zmienne w kodzie — nie trafiają do kontekstu agenta.

**Workflow:**
1. `list_servers` → co jest dostępne
2. `list_tools("todo")` → jakie ma metody
3. `get_tool_schema("todo", "create")` → **TypeScript type definition** (nie JSON Schema)
4. `execute_code` → agent pisze JS korzystający z załadowanych API

**Zalety:** elastyczne łączenie narzędzi bez wrapperów, duże zbiory danych bez powiększania kontekstu, ograniczony dostęp do świata zewnętrznego, izolacja (limit pamięci, timeout, brak filesystem/network).

**Koszty:** wzrost złożoności architektury, dodatkowa warstwa abstrakcji, problemy specyficzne dla izolowanego środowiska.

## Organizacja wiedzy

| Kategoria | Charakter | Dostęp |
|-----------|-----------|--------|
| Dokumenty sesji | Załączniki + pliki wygenerowane w sesji | Agenci w bieżącej interakcji |
| Wiedza publiczna | Pamięć długoterminowa współdzielona | Agenci + użytkownicy |
| Wiedza prywatna | Dane i kontekst użytkownika | Per-user |
| Wiedza agentów | Instrukcje, wspomnienia, obserwacje | Per-agent |
| Pamięć podręczna | Wyniki wyszukiwania, treści stron | Tymczasowa |
| Runtime | Baza sesji, interakcji, harmonogramów | Niewidoczna dla agentów |

Kategoryzacja niejednoznaczna — ta sama informacja może należeć do wielu kategorii zależnie od kontekstu. **Proste zasady organizacji > złożone systemy kategoryzacji.** Czasem wystarczą proste dokumenty zamiast zaawansowanej pamięci długoterminowej.

## Ograniczenia i ryzyka

- **Połączenia między narzędziami** — wektor obejścia restrykcji (filesystem + Jira = dane w sekcji nieobjętej limitami)
- **Instrukcja nie gwarantuje przestrzegania** — model może ignorować lub źle interpretować
- **Złożoność rośnie szybko** — kilkanaście iteracji to norma nawet z pomocą LLM
- **Nadmierne limity** — mogą zredukować użyteczność do zera
- **Zagnieżdżone delegowanie** — subagent może sam wzywać innych agentów; instrukcje muszą uwzględniać reguły przekazywania w głąb hierarchii

## Przykłady kodu

### 02_05_agent — Observational Memory (Observer/Reflector)

**Delta:** kompresja pamięci konwersacyjnej z priorytetyzacją i kalibracją tokenów.

**Observer** — gdy nieskompresowane wiadomości przekroczą threshold, LLM ekstrahuje strukturalne obserwacje z priorytetami:

```
* 🔴 [user] User's name is Adam, developer from Poland
* 🟡 [tool:write_file] Created notes/adam-profile.md
* 🟢 [assistant] Acknowledged preferences
```

Tagi źródłowe (`[user]`, `[assistant]`, `[tool:name]`) determinują priorytet kompresji — `[user]` fakty najwyżej, `[assistant]` elaboracje najniżej.

**Reflector** — gdy obserwacje przekroczą budżet, kompresuje w 3 poziomach agresywności. Zatrzymuje się gdy tokens ≤ target. `[user]` nigdy usuwane, `[assistant]` kondensowane jako pierwsze.

**Layout kontekstu:**

```
┌──────────────────────────────────────────────────┐
│ System prompt + Observations │ Unobserved tail   │
│ (skompresowana historia)     │ (raw messages)    │
└──────────────────────────────────────────────────┘
```

Ogon = 30% budżetu, minimum 120 tokenów. `splitByTailBudget` cofa indeks przy `function_call_output` — nigdy nie odetnie wyniku narzędzia od wywołania.

**Kalibracja estymacji** — heurystyka `chars/4` z runtime'ową kalibracją: porównuje szacunki z rzeczywistym użyciem API, buduje ratio do korekty. Safety margin 1.2x.

**Generation count** — śledzi ile razy pamięć była kompresowana. Pozwala ocenić "stratę informacyjną" i dostosować zachowanie.

**Guard** — `processMemory` uruchamia observer co najwyżej raz per request HTTP (`_observerRanThisRequest`). `flushMemory` wymusza obserwację wszystkich pozostałych wiadomości na koniec sesji.

**Persystencja** — obserwacje i refleksje jako `.md` z frontmatterem (sekwencja, generacja, tokeny, zakres indeksów). Debugowalne bez narzędzi.

**Konfiguracja agenta** — `.agent.md` z gray-matter frontmatter (model, tools) + body (system prompt). Zachowanie konfigurowalne bez zmiany kodu.

### 02_05_sandbox — MCP + QuickJS + Tool Discovery

**Delta względem 02_05_agent:** dynamiczne odkrywanie narzędzi, generowanie kodu zamiast bezpośrednich tool calls, izolowane wykonanie w sandboxie.

**Kluczowa decyzja:** narzędzia opisane TypeScriptem, nie JSON Schema. LLM naturalnie rozumie typy TS — niższe ryzyko błędnej interpretacji parametrów.

**QuickJS sandbox** — każdy egzekucja dostaje nowy runtime/context: 128 MB RAM, 5s timeout, interrupt handler sprawdza czas na żywo. Moduł cached (singleton `modulePromise`).

**Asyncify** — host functions pozwalają narzędzia MCP (asynchroniczne w Node) wywoływać **synchronicznie** z perspektywy kodu w piaskownicy:

```js
const result = todo.create({title: "Buy milk"}); // sync, nie async
```

Wrappery muszą być synchroniczne — `async` stworzyło JS Promises, których microtask queue może nie zostać opróżniony. Po eval: `runtime.executePendingJobs()` jako flush.

**MCP over stdio** — serwer jako osobny proces, `StdioClientTransport` + `McpServer`. Nazewnictwo: `__call_${serverName}_${toolName}`, kod agenta odnosi się przez wrappery (`todo.create()`).

**Rejestr narzędzi** — statyczny registry z pełnymi sygnaturami TS. Agent ładuje tylko potrzebne (`loadedTools` Map). Reszta nie zajmuje kontekstu.
