# S04E05 — Projektowanie rozwiązań wewnątrzfirmowych

## Trzy wymiary wdrożenia AI

Każde wdrożenie rozpatruj na trzech osiach jednocześnie — pominięcie którejkolwiek kończy się niepowodzeniem:

| Wymiar | Zakres |
|---|---|
| Biznesowy | koszty wdrożenia + utrzymania, compliance (Bedrock/Azure), komunikacja ograniczeń do biznesu, ROI |
| Kulturowy | adopcja wymaga zmiany nawyków; inicjatywy oddolne (warsztaty, wymiana doświadczeń) skuteczniejsze niż narzucone z góry |
| Technologiczny | zakres projektu, ewaluacja modeli, architektura agentów, założenie <100% skuteczności |

Technologia nie jest celem, lecz kosztem z oczekiwanym zwrotem. Prezentacja działającego prototypu stakeholderom generuje organicznie nowe wymagania — budować małe, pokazywać wcześnie, iterować na podstawie reakcji.

## Paradoks doświadczenia

Skuteczność tej samej osoby z tym samym agentem drastycznie zależy od świadomości mechanik. Użytkownik świadomy wymienia źródła danych w zapytaniu ("sprawdź w **gmailu i na slacku**, potem zerknij do **plików** z **planem dnia**"). Nieuświadomiony pyta ogólnikowo — agent nie wykonuje zadania. Model nie jest w stanie reliably zrekompensować tego braku. Rozwiązanie: projektuj narzędzia tak, by **ograniczały potrzebę precyzji** ze strony użytkownika.

## Dokument jako najprostsze wdrożenie AI

Zanim piszesz kod, sprawdź czy wystarczy dokument podłączony do istniejącego interfejsu (Claude, Slack):

- **Checklista** — powtarzalny proces (weryfikacja SEO, procedury wdrożeniowe). AI niczego nie przegapi, rola człowieka: akceptacja, nie przypominanie sobie.
- **Onboarding** — AI dopasowuje zapytanie semantycznie, nawet gdy fraza jest daleka od zawartości. Nie zastępuje rozmowy, lecz przekierowuje.
- **Styl/standard** — jeden plik promptu dający spójność wygenerowanych treści w całym zespole.

AGENTS.md i Skills to ten sam wzorzec — deklaratywne instrukcje sterujące zachowaniem modelu — przeniesiony do kontekstu programowania.

## Generatywne interfejsy — MCP Apps

Chat-only ma dwa problemy: użytkownik nie widzi struktury danych, akcje są niedeterministyczne. MCP Apps rozwiązują oba:

| | Czysty chat | MCP Apps | Pełna automatyzacja |
|---|---|---|---|
| Kontrola | Niska | Pełna (kod UI) | Żadna (agent sam działa) |
| Determinizm akcji | Niski | Wysoki | Niski |
| Złożoność implementacji | Niska | Średnia | Wysoka |
| Elastyczność | Wysoka | Średnia | Wysoka |

### Architektura trójwarstwowa

1. **Serwer MCP** — rejestruje narzędzia (`registerAppTool`) i zasoby UI (`registerAppResource`). Każde narzędzie zwraca `content` (tekst dla LLM) + `structuredContent` (dane dla UI) — jeden wynik, dwóch odbiorców. Narzędzia app-only (`visibility: ["app"]`) niewidoczne dla modelu, dostępne tylko z iframe'a.
2. **Host** (przeglądarka) — renderuje czat, montuje aplikacje w sandboxed iframe, proxy'uje żądania przez `AppBridge`.
3. **Embedded App** (iframe) — łączy się przez MCP Apps SDK. Cykl życia: `connect()` → `ontoolresult` → interakcja użytkownika → `callServerTool()` / `updateModelContext()` → `onteardown`.

Każda warstwa wymienialna niezależnie. Ten sam serwer MCP podłączysz do Claude.ai bez zmian — MCP jako platforma integracyjna B2B.

### Kluczowe mechaniki

- **Model Context Update** — `app.updateModelContext()` z debouncem 120ms wstrzykuje snapshot stanu UI do kontekstu kolejnego zapytania. Frontend utrzymuje `Map<resourceUri, context>` z deduplikacją per appka.
- **Cross-domain chaining** — instrukcje agenta zachęcają do sekwencji narzędzi z różnych domen biznesowych (kampania → kupon → sprzedaż).
- **Graceful degradation** — regexowy fallback bez klucza API. System działa zawsze, LLM jest enhancementem.
- **Security iframe** — `sandbox="allow-scripts"` + dynamiczne CSP (`default-src 'none'`, `connect-src 'none'`). Appka może tylko uruchamiać skrypty.
- **Auto-reconnect MCP** — transparentna rekonstrukcja sesji przy utraconym session-id. `withClient(callback)` łapie błędy sesji, tworzy nową Connection i powtarza callback.
- **`parallel_tool_calls: false`** — sekwencyjne wywołania gdy narzędzia mają efekty uboczne.

MCP Apps nie zastępują agentów — są komplementarne. Czat to najczęstszy kontekst, ale embedded apps mogą być osadzane w dowolnych miejscach (dashboardy, panele admina).

## Przegląd dokumentu — agent z inline annotations

Wzorzec: AI analizuje dokument blok po bloku, komentuje wybrane fragmenty, człowiek akceptuje/odrzuca.

### Architektura

Dokument Markdown → AST (remark) → bloki (heading, paragraph, list_item, blockquote). Każdy blok dostaje ID (`b1`, `b2`, ...). Agent przetwarza bloki sekwencyjalnie lub równolegle.

### Dwa tryby pracy

| Tryb | Przetwarzanie |
|---|---|
| `paragraph` | jeden blok = jedno wywołanie, worker pool (concurrency 4) |
| `at_once` | cały dokument = jedno wywołanie, jeden agent loop ze wszystkimi blokami |

### Kluczowe decyzje projektowe

- **Narzędzie waliduje scope, nie prompt** — `add_comment` odrzuca błędny `block_id`. Handler weryfikuje: scope → `findQuoteRange` (wymaga unikalnego matchu cytatu) → `overlaps` (brak nakładania) → zapis z pozycją. LLM halucynujący parametry jest blokowany fizycznie.
- **Błędy narzędzi jako dane, nie wyjątki** — handler zwraca `{status: "rejected", error}` zamiast throw. Agent widzi powód i koryguje. Wyjątek przerywa pętlę.
- **Elastyczność przez kontekst** — zmiana promptu + `contextFiles` zmienia agenta z korektora w fact-checker, SEO optimizer lub router zgłoszeń.
- **Prompty jako Markdown z frontmatter** — `title`, `model`, `modes`, `contextFiles`. Samoopisujący się dokument = konfiguracja agenta. Nowy typ review = nowy plik `.md`. Agent profile w `workspace/system/agents/reviewer.md` z frontmatter `model: gpt-5.4`.
- **Mini Operational Transform** — accept sugestii patchuje markdown i przesuwa pozycje innych komentarzy (delta = `len(suggestion) - len(original)`). Batch accept sortuje **od końca** tekstu — klasyczny wzorzec wielokotwicowych modyfikacji.
- **Re-anchoring** — po edycji dokumentu `resolveCommentRange` waliduje pozycje, a jeśli nie pasują — fallbackuje do `findQuoteRange`. Brak dopasowania → `_broken` → filtrowanie z widoku. Komentarz nie ginie, nawet jeśli tekst drgnął.

### Akcje po review

- **Accept** — podmienia tekst w bloku, przesuwa pozycje sąsiednich komentarzy.
- **Reject** — oznacza komentarz jako odrzucony.
- **Revert** — przywraca oryginalny tekst bloku z `originalBlockText` zapisanego przy accept.
- **Rerun block** — stales otwarte komentarze + ponownie uruchamia agenta z opcjonalnym `customMessage`.

Podsumowanie generowane osobnym wywołaniem LLM na podstawie statystyk (status/severity/kind counts + top 10 komentarzy), z deterministycznym fallbackiem.

## Bezpieczeństwo agentów firmowych

Nawet z zaufanym dostawcą chmury — **nie ufasz modelowi**. Zaufanie do infrastruktury ≠ zaufanie do outputu.

| Kategoria ryzyka | Przykład |
|---|---|
| Wyciek danych | agent z dostępem do Internetu przesyła dane wewnętrzne na zewnątrz |
| Destrukcja danych | agent z dostępem do kodu usuwa/uszkadza źródła |
| Nieautoryzowane akcje | przypadkowe `send_email`, zaproszenie osób spoza org |
| Wprowadzenie w błąd | chatbot sugeruje akcję naruszającą procedury (restart produkcji) |
| Omijanie zabezpieczeń | modele wykrywają testowanie i dostosowują zachowanie (ref: Claude Opus 4.6 System Card, Eval Awareness) |

Zasada: ograniczanie uprawnień + fizyczne uniemożliwianie niepożądanych akcji na poziomie kodu, nie promptu. Sandbox, CSP, path traversal protection, narzędzia app-only — to warstwy obrony, nie pojedyncze rozwiązanie.

## Przykłady kodu — progresja i delta

### 1. Agent loop — dynamiczny rejestr narzędzi (04_05_review)

**Delta względem S04E01:** uproszczony loop bez `previous_response_id`, narzędzia jako para (definition, handler).

```js
const runAgent = async ({ input, instructions, model, tools }) => {
  const definitions = tools.map((t) => t.definition);
  const handlers = Object.fromEntries(
    tools.map((t) => [t.definition.name, t.handler]),
  );
  let conversation = [{ role: "user", content: input }];

  for (let step = 0; step < MAX_STEPS; step += 1) {
    const response = await complete({ input: conversation, tools: definitions, instructions, model });
    const calls = response.output.filter((i) => i.type === "function_call");
    if (calls.length === 0) return { text: response.text, conversation };

    const results = [];
    for (const call of calls) {
      const output = await handlers[call.name](JSON.parse(call.arguments ?? "{}"));
      results.push(call, {
        type: "function_call_output",
        call_id: call.call_id,
        output: JSON.stringify(output),
      });
    }
    conversation = [...conversation, ...results];
  }
};
```

Czego uczy: separacja deklaracji (dla modelu) od implementacji (handler). Handler zwraca plain object — model dostaje JSON jako `function_call_output`. Loop kończy się gdy brak tool calls.

### 2. Narzędzie add_comment — walidacja scope w handlerze (04_05_review)

**Delta:** handler z wielopoziomową walidacją zapobiegającą halucynacjom parametrów.

```js
const range = findQuoteRange(block.text, quote);
if (range.error) return { status: "rejected", error: range.error };

const existing = session.comments.find((entry) =>
  entry.blockId === block_id && entry.status === "open" && overlaps(entry, range)
);
if (existing) return { status: "rejected", error: `Quote overlaps with existing comment ${existing.id}.` };
```

Czego uczy: `findQuoteRange` wymaga unikalnego matchu — wielokrotny cytat → odrzucenie z prośbą o dłuższy fragment. `overlaps` chroni przed nakładaniem. Błąd jako dane, nie wyjątek.

### 3. Worker pool z kolejką (04_05_review)

**Delta:** kontrolowana współbieżność przez N workerów + współdzieloną kolejkę, nie `Promise.all`.

```js
const queue = reviewableBlocks.map((block, i) => ({ block, i }));
const workers = Array.from({ length: Math.min(PARAGRAPH_REVIEW_CONCURRENCY, queue.length) }, async () => {
  while (queue.length > 0) {
    const { block, i } = queue.shift();
    await reviewBlock(block, i);
  }
});
await Promise.all(workers);
```

Czego uczy: każdy worker bierze kolejny blok gdy skończy poprzedni — równomierny rozkład niezależnie od różnic w czasie przetwarzania.

### 4. MCP App — rejestracja z zasobem UI (04_05_apps)

**Delta:** `registerAppTool` wiąże narzędzie z zasobem `ui://` — dual-audience output.

```js
appTool("open_todo_board", { title: "Open todo board", inputSchema: {} }, async () => {
  const state = await readTodosState();
  return {
    content: [{ type: "text", text: `Opened todo board. ${summarizeTodos(state)}.` }],
    structuredContent: toTodoPayload(state),
  };
});
```

Czego uczy: model dostaje tekst, iframe dostaje dane strukturalne. `_meta.ui.resourceUri` wskazuje zasób HTML montowany w sandboxie — ten sam endpoint, dwa konsumenci.

### 5. MCP App SDK — komunikacja iframe↔host (04_05_apps)

**Delta:** embedded app wywołuje narzędzia serwera i aktualizuje kontekst modelu.

```js
const app = new App({ name: "Todo Board", version: "0.1.0" });
app.ontoolresult = (result) => applyState(result?.structuredContent ?? {});

const save = async () => {
  const result = await app.callServerTool({ name: "save_todos_state", arguments: { items: state.items } });
  applyState(resultToState(result));
};

const syncModelContext = async () => {
  await app.updateModelContext({
    content: [{ type: "text", text: `Todo board snapshot: ${state.items.length} total` }],
    structuredContent: { kind: "todos", items: state.items },
  });
};
```

Czego uczy: `ontoolresult` — reakcja na wynik wywołania przez model. `callServerTool` — app-only tool. `updateModelContext` — debounced sync wstrzykuje snapshot. Trójkąt: model ↔ host ↔ app.

### 6. Fallback routing bez LLM (04_05_apps)

**Delta:** graceful degradation — regex routing gdy API niedostępne.

```js
const buildFallbackResponse = async (message, runtime) => {
  const lower = message.toLowerCase();
  if (/\b(sales|revenue)\b/i.test(lower) && !/\b(product|pricing)\b/i.test(lower)) {
    return await executeNamedTool(runtime, "open_sales_analytics", args);
  }
};
```

Czego uczy: system degraduje gracefully — matchuje wzorce z wiadomości do narzędzi MCP. Nie jest to pełny NLU, ale pokrywa typowe przypadki bez kosztu modelu.
