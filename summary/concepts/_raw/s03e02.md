# s03e02 — koncepty

## Context Engineering
S03e02 wprowadza **izolację kontekstu** (scoped access) jako wzorzec zarządzania widocznością danych: agent pracujący na danych wielu kont musi mieć programistycznie wymuszone ograniczenia w fazie generowania treści. Fazowa kontrola: kategoryzacja = pełna widoczność, szkicowanie = KB zablokowany do konta + dopasowanego typu kontaktu. Runtime mutex z `try/finally` gwarantuje izolację. AI nie może eskalować uprawnień — system determinuje scope na podstawie klasyfikacji, nie żądania modelu.

## Security & Sandboxing
S03e02 rozszerza o **architekturę zero-trust** z sześcioma warstwami obrony: publiczny system prompt, programistyczna kontrola dostępu, capability stripping (narzędzia tylko na czas potrzeby), izolacja danych (warstwy konta + contact type), runtime mutex na KB, audit trail każdego dostępu. Filtr prompt injection jako osobny, odseparowany zapytanie zwracające konkretny ciąg znaków weryfikowany programistycznie. Privacy by design: `store: false` na wywołaniach AI. Adwersarz testowy jako część testów — celowy social engineering email weryfikujący poprawność izolacji.

## Prompt Injection
S03e02 dodaje konkretną implementację bariery: **osobny prompt w izolowanym zapytaniu**, całkowicie odseparowany od głównego wątku. Zwraca ustalony ciąg znaków ("bezpieczne"/"niebezpieczne") weryfikowany programistycznie, nie przez LLM. Atakujący nie ma fizycznej możliwości poznania frazy weryfikacyjnej. Ominięcie trudne ale możliwe — dodatkowo zabezpieczenie prawne i jawne informowanie o AI.

## Capability Stripping
Wzorzec fazowego usuwania narzędzi z agenta na podstawie etapu wykonania. Faza triage: 12 narzędzi, pełny dostęp KB. Faza draft: 0 narzędzi, KB zablokowany na konkretne konto. AI nie może sięgnąć po więcej danych niż upieczone w prompcie — fizycznie brak narzędzi do zapytań. Kluczowe: ograniczenia wymuszane kodem (`access-lock.ts`), nie instrukcją. Agent dostaje narzędzia TYLKO na czas ich potrzeby.

## Heartbeat Orchestration
Pull-based task queue z round-based reconciliacją jako alternatywa dla sztywnego DAG-a. Trzy komponenty: kontrakty (struktura planu, zadań, zależności), heartbeat (manager przydzielający zadania po każdym cyklu), pamięć (filesystem-based). Agenci **claimują** zadania (pull), nie są do nich przypychani (push). Matching po capabilities, nie nazwach agentów. Cykl: reconcile dependencies → resolve waiting → claim → run session → persist status → flush events. Daje elastyczność dynamicznych procesów przy zachowaniu kontroli — nowe zadania mogą powstać w trakcie.

## Goal Contracts
Strukturyzowana definicja celu w `goal.md` z YAML frontmatter: objective, must_have, forbidden, budżety (kroków, replanowania), max zadań, warunki wymagające aprobaty człowieka. LLM planner generuje plan z zadaniami → walidacja strukturalna (deps, capabilities, cycles) → repair loop (max 2 próby) → materializacja jako pliki markdown. Gdy cel nieosiągalny → `no-go.md` z powodami i sugerowanymi akcjami. Zapobiega cichym fiaskom.

## Sandbox Code Execution
Przeniesienie obliczeń z kontekstu LLM do deterministycznego kodu w izolowanym środowisku (Deno). Architektura 3-procesowa: proces główny ↔ MCP (STDIO) ↔ Sandbox. HTTP Tool Bridge pozwala kodowi w sandboxie wywoływać narzędzia hosta przez localhost. 4 poziomy uprawnień (`safe`/`standard`/`network`/`full`) mapowane na flagi Deno. Kluczowe optymalizacje: redukcja halucynacji, kosztów tokenów i czasu — 150K+ linii danych → 6-10 kroków zamiast setek. Dual runtime (Bun główny, Deno sandbox). Skalowanie produkcyjne: Cloudflare Sandbox, Daytona.

## Observational Memory
S03e02 rozszerza zastosowanie do kontekstu multi-agent: Observer/Reflector cycle działa per-agent-session, nie globalnie. Umożliwia komunikację między agentami przez system plików — obserwacje jednego agenta stają się dostępne dla innych. Progresywna kompresja z priorytetyzacją (🔴/🟡/🟢). Zastosowanie: długotrwałe sesje agentowe w systemie heartbeat bez przepełnienia context window.

## Event-driven Architecture
S03e02 dodaje wzorzec **kaskadowania zdarzeń** z outputu agenta: etykieta przypisana przez agenta = zdarzenie obserwowalne = trigger dla downstream (deterministycznych lub agentowych). Przykład: label "Bug Report" → tworzenie issue w Jira → inny agent ocenia złożoność. Człowiek też może przypisywać etykiety, uruchamiając te same kaskady. Rozszerza wartość systemu bez zwiększania ryzyka.

## Human-in-the-Loop
S03e02 rozszerza o wzorzec **człowieka jako final gate**: agent nigdy nie wykonuje nieodwracalnych akcji (wysyłanie maili, publikowanie, postowanie). Agent kategoryzuje, etykietuje, szkicuje, filtruje — człowiek decyduje. `request_human` jako narzędzie pauzujące system do czasu decyzji. Kaskadowanie zdarzeń rozszerza wartość bez zwiększania ryzyka — etykieta agenta wyzwala downstream, ale akcja finalna wymaga człowieka.

## Knowledge Anti-patterns
Pliki knowledge zawierają **anty-wzorce i edge cases** (co idzie źle), nie instrukcje "jak zrobić". Wynik trial-and-error: "blank pages w PDFKit", "cursor drift po positioned blocks". Prompt wymusza czytanie tych plików jako pierwszy krok. To RAG, ale retrieval domain to failure modes, nie surowa wiedza.

## Multi-agent / Delegation
S03e02 rozszerza o orkiestrację wielu agentów przez heartbeat z **capability-based task assignment**: zadania deklarują `required_capabilities`, agenci deklarują `capabilities` w szablonach `.agent.md`. Matching po capabilities, nie nazwach. Agent Templates jako deklaratywne pliki markdown z YAML frontmatter (model, tools, capabilities, system prompt). Bounded replanning: patche (add/split/reassign/cancel task) walidowane i ograniczone budżetem. Event sourcing: JSONL + per-round markdown dla audytowalności.

## Observability / Token Tracking
S03e02 dodaje: **eval suite** (Langfuse) z 4 suitami testowymi (triage, izolacja, język, malicious) i konkretnymi metrykami. **Audit trail dostępu do wiedzy** (`knowledgeAccessLog`): każdy dostęp loguje kto, o co, co zwrócono/zablokowano i dlaczego. Token estimation z autokalibracją: estymacja `chars/4` kalibruje się na bazie rzeczywistego API-reported usage — precyzja rośnie w czasie.

## Autonomy Gap
Przejście od systemów z człowiekiem w pętli do autonomicznych to **nowa klasa problemów programistycznych**, nie inkrementalny wzrost trudności. Błąd co piąty raz przy ustalonym procesie = system bezużyteczny. Odchylenia tolerowane w trybie wsparcia przekreślają sens istnienia systemu autonomicznego. Regularna aktualizacja przekonań o możliwościach modeli — to co niemożliwe dziś może być trywialne jutro.
