# s01e05 — koncepty

## Human-in-the-Loop
Kontrola nad akcjami agenta przez deterministyczne potwierdzenia (kod, nie LLM) zawierające pełne detale akcji. Trzy warstwy obrony: statyczna whitelist, dynamiczne confirmation UI, trust escalation jako warstwa UX. Jeśli przypadkowe wycieki są niedopuszczalne, a potwierdzenie niewystarczające — LLM nie powinien być wdrożony.

## Trust Escalation
Mechanizm redukujący friction przy powtarzalnych akcjach. Trzy poziomy: jednorazowe tak, zaufanie na sesję, odrzucenie. Identyfikacja narzędzia jako `server__action` (bezkolizyjnie). **Auto-revoke** trust przy każdej zmianie nazwy/opisu/schematu — krytyczne dla MCP, gdzie serwer może zmienić interfejs bez wiedzy użytkownika. Zaufanie jest sesyjne, nie permanentne.

## Context Engineering
Zarządzanie oknem kontekstu w agentach długożyciowych. Jawne limity (input + output razem; GPT-5.2: 400k okno - 128k output = 272k input). Strategia 3-etapowa: middle-cut truncation dużych tool outputów, turn dropping (zachowaj pierwszą i ostatnie N tur), LLM summarization usuniętych tur z inkrementalnym summary (`[Context Summary]` jako systemowa wiadomość). Kompresja uruchamiana wcześnie, przy ~30% zużycia. Konfiguracja per-model. DB trzyma pełną historię (audit), model widzi tylko to co mieści się w oknie.

## Observability / Token Tracking
Podstawa optymalizacji kosztowej = obserwowalność, nie zgadywanie. Langfuse, Confident AI. Langfuse subscriber mapuje eventy agenta na traces: `agent.started` → Span (z nestingiem), `generation.completed` → Generation z input/output/usage. Eventy niosą `startTime` + `durationMs` — timeline odtwarzalny przy opóźnionym przetwarzaniu. Dwupoziomowe: klasyczne logi + śledzenie agenta. Mała grupa testerów przed wdrożeniem pozwala oszacować skalę.

## Rate Limiting
Limity providera (RPM/TPM) widoczne w nagłówkach — programistyczna reakcja na zbliżający się limit. Każdy endpoint AI musi mieć własny rate limiting per user (zalogowani i anonimowi), niezależnie od providera. OpenRouter programistycznie zarządza kluczami per user z indywidualnymi limitami wydatków.

## Moderacja treści
**Niestosowanie OpenAI Moderation API może prowadzić do zablokowania konta organizacji** (nie klucza). Zakres moderacji wykracza poza prawo — obejmuje sam zakres działania systemu. Weryfikacja przez prompt zwracający ustrukturyzowaną odpowiedź, decyzję o dalszym przebiegu podejmuje kod. Może fałszywie blokować legalne akcje (reklama z interfejsem Messengera blokowana przez Gemini).

## Halucynacje / Niejawne limity
Gwarancja struktury ≠ gwarancja wartości — poprawny JSON z fałszywymi danymi nie zostanie wykryty. Modele Flash potrafią halucynować treść strony WWW z samego URL, podczas gdy mocniejsze modele przyznają brak dostępu. Mitygacje: informowanie modelu o jego ograniczeniach, instrukcje przy braku danych, zmniejszenie złożoności, **zmniejszenie objętości kontekstu**. Halucynacje często wynikają z błędów aplikacji (źle wczytane instrukcje, model informowany o niedostępnych narzędziach) — trudne do zauważenia w systemach multi-agent.

## Provider Abstraction
Format `provider:model` (`openai:gpt-5.4`, `gemini:gemini-3-pro-preview`). Adapter per provider tłumaczy wspólny interfejs (wzorowany na Responses API) na natywne API. `ProviderInputItem`/`ProviderOutputItem` jako wspólny format, `mapItemsToProviderInput()` konwertuje. OpenRouter jako skrót, ale nie wspiera wszystkiego — providerzy wprowadzają API-specific features blokujące translację. W multi-agent każdy agent może używać innego providera. Argument przeciw frameworkom AI (Langchain, CrewAI) — szybko stają się obciążeniem.

## Event-driven Architecture
Fundament, nie feature. Warunek możliwości monitorowania, kompresji kontekstu, moderacji w locie, heartbeat, niezależności komponentów. Każdy event **self-contained** — subscriber nigdy nie sięga do runnera po dane. Eventy niosą `EventContext` (`traceId`, `sessionId`, `agentId`, `rootAgentId`, `depth`) do korelacji w multi-agent. Zdarzenia pętli: `agent.started/completed/failed/waiting/resumed/cancelled`, `turn.started/completed`, `tool.called/completed/failed`, `generation.completed`.

## Non-blocking Agent Execution (waiting/deliver)
Maszyna stanów `pending → running → waiting → running → completed/failed/cancelled`. Agent w `waiting` ma listę `waitingFor[]`, HTTP zwraca **202 Accepted** (semantyka "czekam"), zewnętrzny system dostarcza przez `POST /deliver`. Agent może czekać na wiele rzeczy jednocześnie — jak `Promise.all`. Status `running` dopiero gdy wszystkie `waitingFor` spełnione. Auto-propagacja wyniku child → parent przez `parentId` + `sourceCallId` rekurencyjnie do korzenia.

## Typy narzędzi (sync/async/agent/human)
Cztery typy po stronie runnera, model widzi tylko definicje z `description`. Typ istnieje wyłącznie w infrastrukturze. `sync` — natychmiastowy wynik; `async` — background + deliver; `agent` — spawn child; `human` — zamrożenie + `waitingFor`. Handler typu `human` nic nie robi — runner rozpoznaje typ i omija handler. MCP po konwencji `server__tool`. Celowy design: model wie co, nie jak.

## Multi-agent / Delegation
Agent jako plik markdown (frontmatter: name, model, tools; body: system prompt). Hot-reload bez restartu. Delegacja przez tool `delegate`: guard `MAX_AGENT_DEPTH = 5`, child z `parentId/sourceCallId/depth+1`, zadanie rodzica jako user message dziecka, rekurencja, wynik jako `function_call_output` w historii rodzica. Dwa modele komunikacji: `delegate` (sync, blocking, request-response) vs `send_message` (async, fire-and-forget, "karteczka na biurku" — nie wznawia targeta).

## Agent Memory Schema
Model danych produkcyjnego agenta: `users`, `sessions`, `agents`, `items` (Drizzle/SQLite). Items jako polimorficzny log: `MessageItem | FunctionCallItem | FunctionCallOutputItem | ReasoningItem` z type guardami i sequence number per agent. Repository pattern z identycznym interfejsem dla in-memory i SQLite — swap backendu bez zmiany logiki. Domain model jako czyste funkcje zwracające `TransitionResult = { ok; agent } | { ok: false; error }` (Result type zamiast wyjątków).

## Wydajność / UX agenta
Wydajność modeli adresowana przez architekturę, nie model. **Heartbeat** — wgląd w kroki i budowany rezultat (zmienia postrzeganą wydajność). **Wielowątkowość** — kolejka wiadomości, otwieranie nowych wątków bez czekania, stan UI oddzielony od backendu. **Przetwarzanie w tle** — zadanie musi przeżyć zamknięcie przeglądarki. **Wznawianie** — agent czeka na wiele rzeczy jednocześnie. **Zasada "czy AI tu jest niezbędne"** — jeśli da się kodem, zrób kodem. Ostateczność: fine-tuning małych / destylacja dużych modeli.

## Koszty LLM
LRM mają niższą cenę/token ale generują znacznie więcej (thinking tokens). Proporcja 1:50 — jedna wiadomość user : 50 zapytań model. Tańszy model ≠ tańsze rozwiązanie (GPT-4.1-nano okazał się droższy niż GPT-4.1-mini bo wymagał więcej kroków — paper "Solving A Million-step LLM Task With Zero Errors"). AI ambientowe: małe zadania w krótkich interwałach sumują się drastycznie.

## Security & Sandboxing
Bearer auth z kluczami SHA-256 (nigdy plaintext). Rate limiting per user z cleanup stale windows. Graceful shutdown z hard timeout. `AbortSignal` propagation przez cały stack od HTTP do provider calls. HTTP 202 vs 200 jako semantyczny sygnał stanu. Moderacja na entry point. Niewłaściwych zachowań modelu nie da się wyeliminować — aplikacja musi o tym informować i być zabezpieczona prawnie (regulaminy, polityka prywatności, umowy).

## Streaming
`executeTurnStream()` jako `AsyncGenerator<ProviderStreamEvent, TurnResult>` — yielduje eventy w trakcie, zwraca finalny `TurnResult`. SSE na poziomie HTTP. `runAgent` i `runAgentStream` dzielą tę samą logikę pętli, różnią się tylko sposobem zwracania rezultatów. Stream kończy się gdy agent wchodzi w `waiting` — wznowienie osobnym requestem.

## Deployment produkcyjny
VPS (DigitalOcean/Mikr.us) → Ubuntu z SSH key auth (wyłączyć hasła) → git, node, nginx, ufw → DNS A na Cloudflare → TLS przez `certbot --nginx` → GitHub Actions self-hosted runner → reverse proxy nginx na `127.0.0.1:3000` → secrets w repo settings → workflow `.yml` na push do main.

## Realia biznesowe aplikacji AI
Aplikacja AI to ~80% klasycznego software. Ograniczenia nie tylko modeli: rozproszone bazy wiedzy, nieustrukturyzowane procesy manualne, narzędzia bez API, brak dostępu do aktualnych danych. Rzadko pełna automatyzacja — częściej optymalizacja procesu o kilka-kilkanaście procent. Użytkownicy nieświadomi limitów LLM (wrzucają PDF-y, aktywują setki MCP, oczekują nieskończonych rozmów). Wymóg projektowy, nie zarzut.
