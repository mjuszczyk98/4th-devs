# s02e03 — koncepty

## Knowledge Base Design
Dwa podejścia do dostarczania wiedzy agentom: **podłączana** (istniejące dokumenty pisane dla ludzi → chunki, embeddingi, RAG, problem odnajdywania) vs **budowana** (dokumenty tworzone dla agentów ze strukturą i linkowaniem wewnętrznym, agent nawiguje jak po kodzie źródłowym). Meta-rozróżnienie: "łączenie ze źródłem" (fragmenty pozbawione powiązań) vs "nauka ze źródła" (odnośniki wewnątrz dokumentów prowadzą agenta krok po kroku). Kod źródłowy jako naturalny przykład drugiego. Cztery wymiary nawigacji: perspektywa, nawigacja, powiązania, szczegóły — kod posiada wszystkie, dokumenty biznesowe prawie nigdy.

## Observational Memory
Architektura pamięci długoterminowej oparta na kompresji tekstu, nie wyszukiwaniu. Okno kontekstowe podzielone na observations block (skompresowane) i raw messages block (nowe). Dwa agenty: Observer (kompresuje po 30k tokenów raw) i Reflector (garbage collection po 40k–60k tokenów obserwacji). Obserwacje z temporal model (trzy daty) i priorytetyzacją. Dziennik przetrwa pojedynczą sesję. ~95% LongMemEval bez wektorów/grafów. Filozofia: "Text is the universal interface". Natywne wykorzystanie prompt caching (append-only, full cache hit przez większość czasu) — grafowy RAG tego nie potrafi.

## Graph RAG
Baza grafowa (Neo4j) jako pamięć agenta — nie tylko chunki z embeddingami, ale sieć encji z relacjami umożliwiająca nawigację po powiązaniach. Łączy wyszukiwanie pełnotekstowe (BM25), semantyczne (vector) i nawigację po relacjach w jednej infrastrukturze. Uzasadnione przy wielopoziomowych powiązaniach między rozproszonymi dokumentami. Koszt infrastruktury i złożoności sprawia, że nie jest domyślnym wyborem. Komplementarny z Observational Memory: OM dla pamięci konwersacyjnej, graf dla współdzielonej bazy wiedzy.

## Chunking
Rekursywny podział dokumentów z hierarchią separatorów (nagłówki → akapity → zdania → słowa). Konkretne parametry: 4000 znaków, 500 overlap, automatyczna detekcja sekcji z nagłówków markdown jako metadane. Zbalansowany punkt wyjścia — docelowy rozmiar zależy od domeny i modelu.

## Hybrid Search / RRF
Łączenie wyników z wielu strumieni wyszukiwania (BM25 + vector) przez Reciprocal Rank Fusion: `Σ 1/(k + rank + 1)` — bez wag, bez trenowania, degraduje graceful gdy jeden strumień niedostępny. Overfetch (`limit * 3`) przed fuzją zwiększa szansę trafień z obu strumieni. Kluczowanie po unikalnym identyfikatorze dokumentu.

## Entity Deduplication
Wielowarstwowa deduplikacja encji ekstrahowanych przez LLM: normalizacja per chunk (title case, clamp typów) → globalna deduplikacja przez klucz (lowercase + singularize, dłuższa nazwa/opis wygrywa) → merge w bazie (ON MATCH, najbogatszy opis zachowany) → runtime kuracja (audit + merge_entities). Bez deduplikacji graf staje się szumem — LLM generuje wiele wariantów tego samego konceptu.

## Session Decomposition
Złożone procesy dekomponowane na osobne sesje agentów z tą samą instrukcją, różnymi źródłami, i wynikami do współdzielonego systemu plików. Agent skupiony na jednym zadaniu = wyższa jakość. Optymalne kosztowo — płacimy za wielokrotne wczytanie instrukcji, nie za ogromny kontekst. Krytyczne: agent generujący treść do wysyłki = tylko szkic, nigdy auto-wysyłka (halucynacje + prompt injection).

## Deep Research / Deep Action
Wzorzec iteracyjnego pogłębiania zapytań: doprecyzowanie → parafraza → dekompozycja → pętla szukaj/analizuj/identyfikuj braki → synteza. "Deep action" rozszerza zastosowania poza research — audyty, generowanie kodu, analiza logów. Każdy proces wymagający eksploracji i niebędący natychmiastowy. Wstępne przeszukiwanie jako opcjonalne zawężenie kontekstu.

## Prompt Caching
Observational Memory jako architektoniczny killer feature dla prompt caching: append-only observations → pełny cache hit przez większość czasu, partial hit po Observer, invalidacja tylko przy rzadkim Reflector. Kontrast z grafowym RAG generującym nowy kontekst per zapytanie — nie korzysta z cache.
