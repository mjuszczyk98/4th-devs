# s02e01 — koncepty

## System Prompt Design
Cztery warstwy promptu systemowego: uniwersalne instrukcje, otoczenie (OS/interfejs/zasoby), sesja (kompresja historii), zespół agentów. W prompcie umieszczaj **zgeneralizowane** opisy ról narzędzi/pamięci, nie szczegóły — to samo narzędzie może pełnić różne role u różnych agentów. Kluczowe pytanie projektowe: o czym agent musi wiedzieć PRZED uruchomieniem narzędzi? Ostrzeżenie: "postępy zadania" w system prompcie zaburzają chronologię sesji.

## Context Engineering
Sygnał vs szum to nie jednorazowa optymalizacja promptu, lecz wypadkowa detali systemu: jakość danych wejściowych, dopracowanie logiki aplikacji, dynamiczne komponenty promptu, generyczne mechanizmy (kompresja/planowanie/monitorowanie), przestrzeń na interwencję człowieka. Przesunięcie balansu kod vs AI: workflow → agent → multi-agent (mniej kodu, ale wyższe wymagania jakościowe — błędy głównego systemu nie są do ominięcia).

## Agentic RAG / Agentic Search
Agent domyślnie nie wie, o czym wie — buduje kontekst przez iteracyjne wyszukiwanie. Czterofazowa generyczna strategia: **Skanowanie** (struktura, nagłówki), **Pogłębianie** (pętla szukaj→czytaj→zbierz terminy→szukaj), **Eksplorowanie** (tropy przyczyna/skutek, część/całość, problem/rozwiązanie), **Weryfikacja pokrycia**. Zasady działają niezależnie od źródła danych — jedyny specyficzny fragment promptu to charakter danych (np. język). Operujemy w obszarze prawdopodobieństwa, nie pewności.

## Query Transformation
Rozszerzenie konceptu: agent generuje warianty słów kluczowych, synonimy i terminy z wcześniejszych wyników w pętli pogłębiania. Problem języka (agent szukający po angielsku w polskich dokumentach) rozwiązywany jedną linią w prompcie o języku treści, nie logiką translacji.

## Prompt Caching
Definicje narzędzi siedzą pod system promptem — każda zmiana promptu invaliduje cache narzędzi. Implikacja: prompt statyczny w ramach sesji, dynamiczne dane (git status, data, otwarte pliki) trafiają do wiadomości użytkownika oddzielone tagami XML, kolejne wiadomości zawierają tylko delty. Starsze metadane odświeżane przez narzędzia, nie modyfikację promptu.

## Chain-of-Thought / Reasoning Control
Techniki sterowania uwagą bez modyfikacji system promptu: **lista zadań** (model zapisuje, odznacza, **przepisuje pozostałe** — treść generowana przez model wpływa silniej niż zewnętrzna, por. many-shot jailbreaking) oraz **tryb planowania** (sygnał w wiadomości użytkownika, cache bezpieczny). Reasoning model z `summary: "auto"` oszczędza kontekst przy widocznym planowaniu.

## Agent Harness
Cztery kategorie infrastruktury poza oknem kontekstu: **sesja** (hooki, podsumowania w tle, kolejki zewnętrznych aktualizacji), **pamięć** (asynchroniczne budowanie wspomnień, Batch API), **pliki** (komunikacja między agentami, załączniki, notatki powstające bez agenta), **otoczenie** (dane spoza urządzenia, wstrzykiwane warunkowo).

## Workspace / Agent File Organization
Przestrzeń plików organizowana per data → sesja → agent. Role katalogów: `notes/` i `outbox/` z prawem zapisu agenta, `inbox/` zapisywany wyłącznie przez root. Sub-agenci nie komunikują się bezpośrednio — root routuje dokumenty. Izolacja sesji = izolacja danych użytkowników, egzekwowana programistycznie.

## Context Masking (Manus technique)
Uzupełnianie początku wypowiedzi modelu tokenami wymuszającymi konkretne narzędzie (prefill tool call) — ogranicza dostępne akcje bez usuwania definicji z kontekstu. Deterministyczne zdjęcie po zakończeniu sesji (np. przeglądarki). Deprecated w API Anthropic, ale ilustruje zasadę: niekonwencjonalne podejścia adresują całe klasy problemów.

## Prompt Iteration with LLM
Model potrafi uzasadnić swoje zachowanie i zasugerować zmiany, ale pierwsza propozycja łata konkretny przypadek, nie kategorię. Proces: analiza → generalizacja (żądaj uniwersalnych przyczyn) → korekta (~60% sugestii bezwartościowych, ~30% wymaga zmian; podkreśl niezależność od narzędzi i unikanie przesterowania) → iteracja. Efekt: zwięzłe sformułowania, do których trudno dojść samodzielnie. Generalizacja = najważniejsza umiejętność projektanta agenta.

## MCP (Model Context Protocol)
MCP jako abstrakcja narzędzi w agencie — narzędzia niedostępne w compile-time, pobierane z serwera MCP i konwertowane na format OpenAI (`type: "function"`, `parameters: inputSchema`). Zmiana źródła danych = zmiana `mcp.json`, zero zmian w kodzie agenta. StdioClientTransport uruchamia serwer jako proces potomny.

## Agent Loop / Agentic Loop
Konkretna implementacja pętli z MCP: chat → extract tool calls → parallel execute przez `Promise.all` → append → repeat, z `MAX_STEPS = 50`. Historia konwersacji przekazywana między zapytaniami. Trójpodział odpowiedzialności: REPL zarządza cyklem sesji, agent loop jednym zapytaniem, system prompt zachowaniem modelu.

## Observability / Token Tracking
Breakdown kosztów per sesja: input/output/reasoning/cached tokens, wyświetlany po każdym wywołaniu API. Komenda `clear` w REPL resetuje historię konwersacji i statystyki jednocześnie.

## Prompt Organization
Struktura promptu w `config.js` jako cztery sekcje o jasnych rolach: `SEARCH GUIDANCE` (generyczne zasady nawigacji), `EFFICIENCY` (zakaz czytania całych plików przed wyszukiwaniem), `RULES` (cytowanie źródeł, przyznawanie braków), `CONTEXT` (jedyna sekcja specyficzna dla domeny). Wzorzec: sekcja `EFFICIENCY` chroni przed marnowaniem tokenów przez wymuszenie wyszukiwania przed czytaniem.
