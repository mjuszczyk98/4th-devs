# s02e04 — koncepty

## Multi-agent / Delegation
S02e04 rozszerza koncept o architektury systemów wieloagentowych: Pipeline, Blackboard, Orchestrator, Tree, Mesh, Swarm — pierwsze cztery dominują, często łączone. Dwa prymitywy komunikacji: **delegate** (fire-and-collect, nowy wątek z własnym promptem i narzędziami) vs **message** (dwukierunkowy dialog z wstrzymaniem pętli sub-agenta, realizowalny przez generatory JS). Agent zarządzający: minimalne narzędzia (delegate/message/search_memory), maksymalny dostęp do informacji, odpowiedzialny za wiedzę o systemie, delegowanie, transport wiedzy, decyzyjność i weryfikację. Ograniczanie roli konieczne ale z umiarem — każdy etap komunikacji komplikuje zarządzanie kontekstem.

## Multi-agent Architectures
Sześć wzorców koordynacji: **Pipeline** (sekwencyjne przekazanie wyniku), **Blackboard** (współdzielony stan, niezależni agenci), **Orchestrator** (centralny koordynator deleguje), **Tree** (hierarchia orchestrator + managerowie pośredni), **Mesh** (komunikacja adresowana między agentami), **Swarm** (rozproszona self-organizacja). Mesh i Swarm rzadko produkcyjnie z LLM — trudne do debugowania i kontroli.

## Event-driven Architecture
Zdarzenia między agentami rozbijają sprzężenie — agent A nie musi wiedzieć, że C i D zareagują na jego wynik. Agent może nasłuchiwać, emitować i zawieszać się czekając na dane. Przykład: obsługa zgłoszenia klienta przez łańcuch zdarzeń (`user.message` → klasyfikacja → niezależne reakcje tracker/credit → draft z zawieszeniem na dane → weryfikacja człowieka). Koszt: trudniejszy tracing i debugowanie.

## Context Engineering
Wieloagentowy kontekst wprowadza nowe wyzwania: konflikty współbieżne (ostatni zapis wygrywa), degradacja komunikacji (każdy przekaz = potencjalna utrata), własna interpretacja danych przez agentów, utrata kontekstu powstania informacji, duplikowanie, rola metadanych w komunikacji między agentami. Pięć strategii konfliktów: wykrywanie (checksumy), unikanie (izolacja), agent zarządzający, append-only, eskalacja do człowieka. Zewnętrzny kontekst odseparowany od logiki — dokumenty mówią co, system decyduje kiedy i kto. Zasada: najprostszy możliwy system, utrzymywany w tej formie najdłużej.

## Workflow-as-Data
Workflow zdefiniowany jako plik markdown, nie hardkodowana logika. Orchestrator odczytuje instrukcje przez `read_file` i interpretuje je jako kroki do wykonania. Zmiana procesu = zmiana pliku, bez rekompilacji i bez zmian w kodzie agenta. CRON jako wzorzec inicjacji: trigger + odniesienie do pliku workflow.

## Agent-driven Context Assembly
Układ workspace z danymi pogrupowanymi w cztery warstwy: **goals/** (długoterminowe cele), **history/** (poprzedni output), **memory/** (preferencje usera), **sources/** (dane wejściowe). Agent sam decyduje co przeczytać, w jakiej kolejności i jak połączyć — nie jest to RAG (wyszukiwanie + wstrzyknięcie), lecz agent-driven assembly. Synteza z korelacją warstw = rola orchestratora.

## Security & Sandboxing
Path traversal w kontekście multi-agent: LLM generuje ścieżki w narzędziach plikowych, `isPathSafe()` weryfikuje czy resolved path nie wychodzi poza `workspace/`. Bez tego agent może czytać `../../etc/passwd` lub `../../.env`. Wszystkie operacje plikowe scoped do katalogu roboczego.

## Workflow vs Agent
Rozszerzenie o tabelę kryteriów uzasadniających agentów: zadania otwarte, dynamiczne dane/zależności, iterowanie z kryteriami w NL, elastyczna architektura, personalizacja. Gdzie wymagania wykluczają LLM (koszt zero, ms response, pełna przewidywalność) — tylko deterministyczny kod. Rola człowieka: dashboard (statystyki, sesje, harmonogram, obszary wymagające uwagi), nie okno czatu.

## Agent Loop / Agentic Loop
Implementacja rekurencyjnej delegacji: `delegate` = rekurencyjne wywołanie `runAgent(agent, task, depth + 1)` z guardrails `MAX_DEPTH = 3` i `MAX_TURNS = 15`. Narzędzie zdefiniowane jako JSON Schema, logika interceptowana w pętli agenta — handler zwraca surowe args, pętla rozpoznaje `delegate` i wywołuje rekurencję.
