# s01e02 — koncepty

## Function Calling / Tool Use
Podstawowy mechanizm łączenia LLM z otoczeniem: model generuje JSON (nazwa + argumenty), aplikacja wykonuje funkcję, wynik wraca do kontekstu. Jedno wywołanie = minimum dwa round-tripy do LLM. Schematy wszystkich narzędzi kosztują kontekst przy każdym zapytaniu. Zalecany limit 10–15 narzędzi na agenta.

## Tool Design
Narzędzia LLM to nie mapowanie API 1:1 — API jest pisane dla programistów, narzędzia dla modelu bez dokumentacji. Zasady: unikalne nazwy, wysokie signal-to-noise w opisach, podział pól na obowiązkowe/programowe/zabronione, aliasy i defaulty, łączenie zasobów API (`workspace_metadata`), konfigurowalne połączenia gdzie model wybiera podzbiór danych, operacje batchowe, pomijanie hashy i wewnętrznych IDs.

## Agent Loop
Pętla `while (steps-- > 0)` z `MAX_TOOL_STEPS` jako obowiązkowym guardem. Po wyczerpaniu kroków rzuć błąd, nie zwracaj cicho. Wywołania niezależne parallelize przez `Promise.all` (parallel function calling). Conversation state to `[...messages, ...toolCalls, ...toolResults]`.

## Error Handling w narzędziach
Błąd NIE przerywa pętli — serializuj wyjątek do `output` i zwróć modelowi jak normalny wynik. Standard wyższy niż dla API: komunikaty muszą być opisowe i sugerować następny krok (`"team_id nieprawidłowy. Pobierz przez workspace_metadata"`). Obsługa literówek i wariantów.

## Prompt Cache
Priorytet optymalizacji prędkości i kosztu. Stabilny system prompt + stabilny początek wątku = automatyczny cache u OpenAI/Gemini. Dynamiczne dane (data, godzina) w system prompcie niszczą cache przy każdym zapytaniu — wprowadzaj je przez wyniki narzędzi na końcu wątku.

## Context Management
System plików jako pamięć zewnętrzna: narzędzia zapisują wyniki do plików, agent wczytuje fragmenty na żądanie. Mapy treści (`_index.md`) zamiast instrukcji nawigacji w system prompcie. Pływające okno kontekstu jest gorsze od pełnego wątku z cache. Auto-compact = lossy, ważne rzeczy zapisuj ręcznie do plików. Dynamiczne listy narzędzi bez psucia cache — obecnie tylko Anthropic; alternatywy: sub-agenci, Progressive Disclosure przez code execution sandbox.

## Context Engineering
Dwa znaczenia: (1) poziom użytkownika — zarządzanie kontekstem w Claude Code/Cursor, (2) poziom aplikacji — kontrola interakcji, narzędzi, komunikacji między agentami. Kurs skupia się na (2).

## Agent Harness
Pełna obudowa dla agenta: system plików, sandbox do kodu, zarządzanie pamięcią, komunikacja między agentami, observability. ~80% klasycznej inżynierii + ~20% nowej klasy problemów.

## MCP (Model Context Protocol)
Pozycjonowane jako jeden z wzorców proxy między istniejącym API a LLM — sposób na adaptację API bez jego modyfikacji. Porównanie z bezpośrednim API i CLI pod kątem kontroli, skalowalności, dokumentacji i elastyczności.

## Security & Sandboxing
Każda granica dostępu wymuszona przez kod, nie przez prompt. Path traversal blokowany w handlerze. Model nie ustala `user_id`, uprawnień, ścieżek poza sandbox. Akcje nieodwracalne → potwierdzenie przez UI (przycisk), nie przez tekst. Wzorzec dry-run: agent testuje przed wykonaniem.

## Prompt Injection
Problem otwarty, brak skutecznej obrony. Agent czytający email/web może wykonać instrukcje z treści. Przykład: agent z kalendarzem + pocztą wysyła plan spotkań obcej osobie na prośbę z emaila. Jedyne mitygacje: ograniczenia środowiskowe (sandbox, whitelisty, izolacja), nieudostępnianie zasobów wrażliwych.

## Workflow vs Agent
Workflow — sztywne etapy, wysoka przewidywalność, dobre dla ustabilizowanych procesów. Agent — dynamiczne decyzje, elastyczny, ale mniej przewidywalny, dla otwartych problemów. Heurystyka: najpierw próbuj zamknąć problem w workflow; 100% skuteczności wymaga człowieka w pętli.

## Reasoning Control
Wbudowany reasoning (LRM) pomaga trudnym, szkodzi prostym zadaniom ("Don't Overthink it"). Techniki sterowania uwagą agenta: Planowanie (lista zadań jako narzędzie), Odkrywanie (prowadź przez eksplorację pamięci — "model nie wie że wie"), Przekierowanie (focus na stanie środowiska), Uśrednianie (wiele modeli → głosowanie). Premise Order Matters (arxiv 2402.08939) — sama kolejność informacji wpływa na skuteczność nawet o 40%.

## Query Transformation
Zapytanie użytkownika rzadko pasuje 1:1 do zasobów. Ekspansja przez synonimy i powiązane zagadnienia zwiększa hit rate. Agent musi mieć ogólną mapę bazy wiedzy (np. `_index.md`) żeby nawigować. Pytania doprecyzowujące jako pierwszy krok.

## Augmented Function Calling
Instrukcje kontekstowe doklejane do wywołania narzędzia (np. stały styl obrazów). Tryby: statyczny (user-driven), dynamiczny (model-driven przez nazwę/opis), hybrydowy. Może zarządzać skillami (aktywacja/tworzenie) i sekwencjami akcji.

## Natywne vs własne narzędzia
Natywne (web search, code execution) — dokleja się jedną linią obok własnych, wygoda kosztem konfigurowalności. Własne — pełna kontrola. Można mieszać w jednym zapytaniu.
