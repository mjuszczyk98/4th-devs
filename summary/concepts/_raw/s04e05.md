# s04e05 — koncepty

## Enterprise AI Deployment
Wdrożenie AI w organizacji wymaga trzech osi jednocześnie: biznesowej (koszty, compliance, ROI), kulturowej (adopcja, oddolne inicjatywy > narzucone z góry) i technologicznej (ewaluacja, architektura, założenie <100% skuteczności). Paradoks doświadczenia: skuteczność użytkownika drastycznie zależy od świadomości mechanik agenta — projektuj narzędzia ograniczające potrzebę precyzji. Prototypowanie generuje organicznie nowe wymagania — budować małe, pokazywać wcześnie.

## Document-as-Tool
Najmniej inwazyjne wdrożenia AI to dokumenty/prompty podłączane do istniejących interfejsów. Trzy wzorce: checklista (powtarzalny proces, AI weryfikuje kompletność), onboarding (przekierowanie semantyczne, AI radzi sobie z niedokładnymi zapytaniami), styl (jeden prompt = spójność w zespole). AGENTS.md i Skills to ten sam wzorzec przeniesiony do programowania.

## Dual-audience Output
Każde narzędzie MCP App zwraca `content` (tekst dla LLM) i `structuredContent` (dane dla embedded UI). Jeden wynik narzędzia, dwóch odbiorców — model dostaje tekst, iframe dostaje dane strukturalne. `_meta.ui.resourceUri` wskazuje zasób HTML montowany w sandboxie.

## Model Context Update from UI
Embedded app wywołuje `app.updateModelContext()` z debouncem 120ms — wstrzykuje snapshot stanu UI do kontekstu kolejnego zapytania modelu. Frontend utrzymuje `Map<resourceUri, context>` z deduplikacją per appka, serializowaną w każdej wiadomości do API. Model "widzi" stan UI bez pytania.

## Multi-anchor Text Editing
Wzorzec edycji z wieloma zakotwiczeniami: accept sugestii patchuje markdown i przesuwa pozycje sąsiednich komentarzy (delta = `len(suggestion) - len(original)`). Batch accept sortuje od końca tekstu — klasyczny wzorzec zachowania poprawności niezmnodyfikowanych pozycji. Re-anchoring: `resolveCommentRange` waliduje pozycje, fallbackuje do wyszukiwania tekstu, brak dopasowania → marked as broken, nie usunięte.

## Worker Pool Pattern
N workerów z jedną współdzieloną kolejką zamiast `Promise.all` na wszystkich elementach. Każdy worker bierze kolejny element gdy skończy poprzedni — równomierny rozkład niezależnie od różnic w czasie przetwarzania.

## NDJSON Streaming
POST zwraca `application/x-ndjson` — każda linia to osobny JSON. Frontend czyta przez `ReadableStream` z buforem na niekompletną linię (`buf = lines.pop()`). Prostsze niż WebSocket, działa z HTTP, istotne dla długotrwałych zadań agentowych z natychmiastową informacją zwrotną.

## Tool Scoping
`createReviewTools(allowedBlockIds)` tworzy narzędzia z wbudowanym scope — agent nie może dodać komentarz poza dozwolonym blokiem. `visibility: ["app"]` ukrywa narzędzia persystencji przed modelem — dostępne tylko z wnętrza iframe'a. Ograniczenia uprawnień wymuszane kodem w handlerze, nie instrukcją w prompcie.

## Graceful Degradation (Agent Fallback)
Gdy API niedostępne, agent degraduje do regex-based routingu — matchuje wzorce z wiadomości do konkretnych narzędzi MCP. Real-world pattern: system działa zawsze, LLM jest enhancementem. Podobnie re-anchoring komentarzy: tekst drgnął → poszukiwanie na nowo, nie usunięcie.

## Enterprise Agent Security
Pięć kategorii ryzyka: wyciek danych, destrukcja danych, nieautoryzowane akcje, wprowadzenie w błąd, omijanie zabezpieczeń (modele wykrywają testowanie i dostosowują zachowanie). Zasada: zaufanie do infrastruktury ≠ zaufanie do outputu. Ograniczanie uprawnień + fizyczne uniemożliwianie na poziomie kodu, nie promptu. Sandbox, CSP, path traversal protection, narzędzia app-only jako warstwy obrony.

## Prompts as Markdown with Frontmatter
Prompty review jako pliki `.md` z frontmatter: `title`, `model`, `modes`, `contextFiles`. Samoopisujący się dokument = konfiguracja agenta. Agent profile w `workspace/system/agents/reviewer.md`. Nowy typ review = nowy plik `.md`, bez zmiany kodu. `contextFiles` jako mechanizm przełączania ról agenta (korektor → fact-checker → SEO optimizer).
