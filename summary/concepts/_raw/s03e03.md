# s03e03 — koncepty

## Agent Triggers
Pięć typów wyzwalaczy akcji agenta: wiadomości (człowiek/agent), hooki wewnętrzne, webhooki zewnętrzne, cron (harmonogram), heartbeat (regularna kontrola stanu). Wszystkie mogą trafiać do jednego punktu wejścia — agent dynamicznie interpretuje zadanie w NL i dobiera akcję. Kluczowa różnica względem klasycznego event-driven: system agentowy rozumie intencję, nie wymaga mappingu event→handler.

## Proactive Agents
Agent działa niezależnie od bezpośrednich działań człowieka. Heartbeat cyklicznie sprawdza listę zadań (`tasks.md`) — gdy nic nie wymaga akcji, agent jest invisible. Gdy tak — proaktywnie działa na bazie metadanych (czas, lokalizacja, pogoda, status urządzenia). Przejście od L02E01 (otoczenie tylko podczas interakcji) do pełnej integracji z otoczeniem.

## Runtime Feedback
Trzy warstwy informacji zwrotnej wewnątrz tool loop: (1) statyczne sugestie wstrzykiwane do wyniku narzędzia po błędzie (nie zmieniają promptu, wzbogacają output), (2) dynamiczne interwencje — wstrzyknięcie dodatkowej wiadomości `role: 'user'` po 2+ porażkach z rzędu (screenshot, zmiana strategii), sugestia zapisu discoveries po odzyskaniu, (3) hook lifecycle z `beforeFinish` jako deterministyczny guard wymuszający ukończenie pełnego flow. `FeedbackTracker` rejestruje sukcesy/porażki per narzędzie i generuje hinty specyficzne dla typu błędu. Runtime intervention ≠ prompt engineering — modyfikuje input do modelu, nie instrukcję.

## Hook Lifecycle
Funkcje wywoływane na etapach pętli agenta: `beforeToolCall` (nagranie stanu), `afterToolResult` (aktualizacja flag fazowych, detekcja błędów), `beforeFinish` (guard wymuszający ukończenie kroków). Pełna sekwencja AI SDK: `onStart → onStepStart → onToolCallStart → onToolCallFinish → onStepFinish → onFinish`. Trzy role: przechwycenie (modyfikacja I/O), śledzenie stanu (flagi fazowe z auto-resetem po `tryCompletePhase`), strażnik procesu (`beforeFinish` blokuje zakończenie gdy wymagane kroki nie ukończone). `beforeFinish` wstrzykuje komunikat z listą brakujących kroków — wymusza dokończenie z wyjątkiem błędów narzędzi lub limitu turns.

## Persistent Learning
Dwa wzorce nauki między sesjami. **Instruction files**: agent zapisuje discoveries (co zadziałało na danej stronie) do `{site}-discoveries.md`, korzysta z gotowych recipes w `{site}.md`. System prompt: "najpierw sprawdź instruction file". Agent uczy się struktury świata, nie historii konwersacji. **Profile accumulation**: profil ucznia akumuluje odkryte `trait_id` między sesjami — ustandaryzowana trait taxonomy (`grammar.articles`, `pronunciation.stress`, `fluency.fillers`). Profil małe (role, goals, weakAreas), pełne sesje w osobnych plikach.

## Agent Phasing
Architektura agenta z fazami: każda faza z własnym promptem i zestawem narzędzi. Model dostaje tylko to co potrzebuje — redukcja complexity i halucynacji. Brak overlapu narzędzi między fazami = model nie próbuje akcji z innej fazy. State machine z `tryCompletePhase()` resetuje się po ukończeniu — wielokrotne tury analizy w jednej sesji. Komplementarne z Capability Stripping: phasing organizuje dostępne narzędzia, stripping je usuwa.

## Tool-chaining
Narzędzia mogą samoistnie wywoływać dodatkowe zapytania do modeli/API — nie są tylko proxy do danych. `listen` wywołuje osobne zapytanie (ASR + analiza), `feedback` wewnętrznie wywołuje `speak` (TTS). Jedno narzędzie enkapsułuje cały sub-pipeline modeli i API, ukrywając złożoność przed pętlą agenta. W MCP odpowiada za to mechanizm Sampling — serwer MCP odwraca komunikację i prosi klienta o wywołanie API.

## Context Engineering
S03e03 rozszerza o **pasywne odkrywanie otoczenia** — metadane (czas, lokalizacja, pogoda, status urządzenia) wstrzykiwane programistycznie do wiadomości zamiast wymagania aktywnego wywołania narzędzi. `buildMetadata()` generuje blok `<metadata>` doklejany do każdej wiadomości. Klucz: scope'ować do potrzeb bieżącej interakcji — niewymagane zdarzenia nie zanieczyszczają kontekstu. Content overflow: pełna treść strony zapisywana do pliku, agent przeszukuje przez `fs_search` zamiast ładować całość do konwersacji.

## Structured Outputs
S03e03 pokazuje wymuszanie JSON Schema na **całej odpowiedzi modelu** przez Gemini `response_format` — nie tylko na argumentach narzędzi (jak `strict: true` w OpenAI). Model zwraca dokładnie strukturę np. `{ transcript, confidence, strengths, issues, segments }`. Gemini Interactions API wymusza to natywnie na poziomie zapytania.

## MCP Sampling & Elicitation
S03e03 pokazuje praktyczne zastosowanie Sampling — narzędzia wywołują dodatkowe API/model przez odwrócenie komunikacji. Serwer MCP prosi klienta o wywołanie. Rzadko wspierane przez implementacje, ale adresuje przypadek narzędzi orkiestrujących sub-pipelines modeli.

## Defensive JSON Parsing
Dwuetapowe parsowanie argumentów narzędzi: `JSON.parse` → fallback `jsonrepair` (trailing commas, brakujące cudzysłowy, markdown fences). Przy sukcesie naprawy: hint doklejany do wyniku ("Keep arguments strict JSON next time"). `parseJsonLoose()` — wyciąganie JSON z markdown fences, fragmentów tekstowych, błędnych struktur. Kolekcja kandydatów (full string, `{`→`}`, `[`→`]`) → parse first success. Must-have w każdym agentic system — LLM regularnie generuje niepoprawny JSON.

## Enrichment
Agent jako węzeł łączący szczątkowe informacje z wielu źródeł (kontakty, miejsca, kalendarz, mapa, pogoda) w ustandaryzowany output. Oryginalna wiadomość wzbogacana o dane z narzędzi. Wartość biznesowa w standaryzacji procesów, nie pojedynczych interakcjach. Bezpieczeństwo: deterministyczne interfejsy dla wrażliwych operacji (np. programistyczne zarządzanie kontaktami) — etap nie przez agenta.

## Web Scraping Evolution
Przesunięcie w traktowaniu agentów: od blokowania (AI Labyrinth) do współpracy — Cloudflare Markdown for Agents (strukturyzowana treść dla agentów), Chrome WebMCP (natywny protokół komunikacji). Nie eliminuje scrapowania — platformy bez API nadal wymagają agenta z przeglądarką. Gradacja automatyzacji: klasyczny bot → agent AI z Playwright → Browserbase/kernel.sh na większą skalę.

## Offensive Design
Projektowanie z perspektywy "co dodać" nie tylko "co naprawić". Przykład: interfejs głosowy (nawet prosty STT→LLM→TTS) w proaktywnym systemie okazuje się bardzo użyteczny. Nie każdy agent musi być uniwersalny — wysoka specjalizacja jest akceptowalna. Onboarding i komunikacja produktu = element architektury na równi z kodem.

## Provider Abstraction
S03e03 porównuje OpenAI Responses API vs Gemini Interactions API: różnice w kontynuacji sesji (`previous_response_id` vs `previous_interaction_id` — API trzyma kontekst po stronie serwera), wymuszaniu schema (tool params vs cała odpowiedź), audio output, thinking control, output types. Argument za multi-provider: każdy API ma unikalne capabilities (audio natywne, response_format na całą odpowiedź, thinking levels).
