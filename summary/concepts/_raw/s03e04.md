# s03e04 — koncepty

## Tool Design
Lekcja rozbudowuje koncept o Tool Envelope `{ data, hint }` jako architekturę odpowiedzi narzędzi, Next Actions jako guided exploration z confidence scoringiem, kontrolę szczegółowości przez parametr `details`, auto-detekcję typu zasobu (message vs thread), wzorzec modify zwracający stan po zmianie, `.describe()` na każdym polu Zod jako jedyne źródło semantyki parametru, oraz zawężanie pełnego API do dedykowanych narzędzi per przypadek użycia.

## Tool Response Envelope
Standaryzowana koperta odpowiedzi narzędzia `{ data: T, hint: ToolHint }` z polami status, reasonCode, summary, nextActions, recovery, diagnostics. LLM dostaje kontekst decyzyjny zamiast gołych danych — nie zgaduje co zrobić po błędzie/braku wyników, dostaje propozycję kolejnej akcji. Zmniejsza halucynacje i bezcelowe tury agenta.

## Error Handling w narzędziach
Lekcja dodaje 4-warstwową walidację (registry → JSON parse → Zod → handler) z ustandaryzowanym błędem na każdej warstwie, klasyfikację błędów przez pattern matching (AUTH_REQUIRED, RATE_LIMITED, TRANSIENT_FAILURE, NOT_FOUND, INVALID_ARGUMENT) ze strategią recovery per kategorię, oraz Zod `.refine()` dla reguł zależnościowych między polami.

## Schema-driven Development
Zod jako single source of truth: definicja raz → automatyczna konwersja do JSON Schema (function calling), `.describe()` dla semantyki widocznej dla modelu, `.refine()` dla walidacji biznesowej. Eliminuje podwójne definicje parametrów.

## Security & Sandboxing
Send whitelist jako guardrail wymuszony w kodzie handlera: adresat poza whitelistą → automatyczny draft (`policy.enforcedDraft=true`), nie wysyłka. Zasada "AI proposes, code decides" — blokada nie do obejścia przez prompt injection.

## Ewaluacja (Evals)
Architektura z Promptfoo: stateless provider (subprocess per query) i stateful provider (serializacja historii sesji z SHA256 key, promise chain dla sekwencyjności). 7 warstw asercji od is-json po llm-rubric. Trzy kategorie scenario evalów (readonly, safety, actions). `expected_outcome` jako semantyczna asercja rezultatu. Scenariusze ewaluacyjne bez system promptu jako walidacja samo-opisowości narzędzi.

## Prompt Iteration with LLM
Iteracyjne projektowanie schematów: model daje ~60-70%, typowe patologie (brak stronicowania, base64, brak metadanych). Metoda: kilka ogólnych zasad rozszerzanych na cały kontekst integracji. Trik: gdy brakuje pomysłów → generowanie przykładowych interakcji ujawnia problemy niewidoczne w abstrakcji. Workflow odkrywania: API docs + SDK → lista akcji → pytania filtrujące.

## Model Selection Strategy
Porównanie modeli na tych samych ewaluacjach: pass/fail to za mało — modele różnią się liczbą kroków, stylem komunikacji, stabilnością. Optymalizacja schematów pod mniejsze modele opłacalna nawet gdy docelowo najsilniejszy. Automatyzacja ewaluacji możliwa po ustabilizowaniu zestawu testów.

## System Prompt Design
Prompt jako kompaktowa lista zasad (19 linii) referująca do mechanizmu hintów zamiast dublować logikę. Zasady opisują kiedy użyć narzędzia, nie jak działa.

## LLM-generated Test Data
Syntetyczne dane testowe generowane przez LLM z kodu narzędzi. Kategoryzacja dwuwymiarowa: happy path / edge case × pojedyncze interakcje / scenariusze wieloetapowe. LLM domyślnie generuje płytkie testy — wymaga kierowania i weryfikacji kompletności. Scenariusze testowe służą też do oceny jakości zestawu testowego.

## Mock → Real Strategy
Jedna zmienna env przełącza między realnym API a in-memory mockiem z pełną funkcjonalnością (parser zapytań, paginacja, mutowalny stan). Mock nie jest statycznym fixturem — deterministyczny, pozwala testować edge case'y bez dostępu do backendu.

## Reasoning Models Compatibility
Modele reasoning emitują reasoning items z server-side IDs — przy `store=false` IDs nie replikowalne w kolejnych turach (404). Rozróżnienie parametrów reasoning/non-reasoning konieczne w Responses API.

## Context Engineering
Kontrola szczegółowości odpowiedzi (parametr `details`) jako mechanizm zarządzania oknem kontekstowym. Domyślnie compact metadata, pełne dane na żądanie. Agent nie zużywa kontekstu na niepotrzebne detale.

## Agent Loop / Agentic Loop
Pętla z `maxTurns` guard — agent wykonuje tury aż do braku tool calls lub limitu. Wynik jako struktura (finalText, turns, reachedMaxTurns, toolCalls) gotowa do eval. Trace zapisuje turn, name, callId, rawArguments, parsedArguments, output/error.
