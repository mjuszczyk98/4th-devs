# s01e01 — koncepty

## Context Engineering
Sterowanie zachowaniem modelu = zarządzanie kontekstem. Lekcja pozycjonuje to jako fundamentalną zasadę: programista kontroluje, co trafia do okna kontekstowego (prompt systemowy, historia, wynik poprzednich zapytań), i w ten sposób programuje zachowanie LLM.
- API jest bezstanowe — klient musi dostarczyć pełny kontekst przy każdym żądaniu.
- Okno kontekstowe obejmuje wszystko razem: system prompt, historia, input, output.
- Routing kontekstu: wynik jednego zapytania kształtuje kolejne (np. klasyfikacja intencji → dopasowanie promptu).

## Structured Outputs
Wymuszanie odpowiedzi w formie JSON zgodnej z JSON Schema. Lekcja pokazuje dwa wymiary schemy (struktura vs wartości), wzorce projektowe i praktyczny przykład ekstrakcji danych.
- `strict: true` + `additionalProperties: false` gwarantują strukturę.
- Nazwy i opisy pól są "mikropromptami" wpływającymi na wartości.
- Kolejność pól ma znaczenie przez autoregresję (reasoning → decyzja → confidence).
- Wartości neutralne (`null`, `"unknown"`) jako mechanizm redukcji halucynacji.

## Function Calling / Tool Use
Drugi (obok Structured Outputs) sposób wymuszania strukturalnej odpowiedzi; tu wspomniany jako alternatywa. Natywne narzędzia providerów (web search, file search, code execution) są łatwe w użyciu, ale słabo konfigurowalne i wiążą z platformą.

## Multi-step Pipelines
Rozbijanie zadania na kaskadę zapytań LLM. Lekcja pokazuje pipeline `extract → dedupe → search → ground` jako wzorzec: każdy etap ma dedykowany prompt + schemę, wynik etapu N zasila etap N+1.
- Fragmentowanie inputu (per akapit) dla skupienia uwagi modelu, nawet gdy całość mieści się w oknie.
- Separation of concerns — jedno zapytanie = jedna odpowiedzialność.
- Kaskadowa inwalidacja cache'u przez hashe.

## Parallelism & Batching
Równoległe wywołania API w grupach (`Promise.all`) w obrębie etapu pipeline'u, dla skrócenia czasu wykonania i unikania rate-limit.

## Prompt Caching
Mechanizm natywny providerów redukujący koszt i latencję przy powtarzających się prefiksach promptów. Lekcja pozycjonuje cache jako krytyczny dla skali i agentów. Dodatkowo na poziomie aplikacji: caching wyników etapów pipeline'u do JSON przez hashe inputu.

## Rate Limiting
Limity wywołań API jako wąskie gardło na produkcji. Obejścia: planowanie zwiększenia limitów z wyprzedzeniem, użycie OpenRouter, retry z exponential backoff (429/500/503).

## Model Selection Strategy
Cztery strategie: jeden główny / główny + alternatywny / główny + specjalistyczne / zespół małych. Kluczowa teza: nie ma "najlepszego modelu", jest tylko "najlepszy w tej sytuacji". Jedyna miarodajna metoda wyboru to testy na własnych zadaniach.

## Provider Abstraction (anti vendor lock-in)
Aplikacja nie powinna być ściśle powiązana z jednym providerem. Lekcja pokazuje abstrakcję warstwy API (OpenAI vs OpenRouter) przez warunkowe budowanie requestów. Frameworki typu AI SDK obecnie nierekomendowane — narzucają ograniczenia bez adekwatnej wartości.

## Prompt Organization
Cztery techniki organizacji promptów w kodzie: inline / oddzielne pliki z kompozycją / systemy zewnętrzne (Langfuse) / markdown + YAML frontmatter. Preferowane: markdown + frontmatter — dostępne z filesystemu w runtime, edytowalne przez samych agentów jako narzędzie.

## System Prompt Design
Składowe: tożsamość, strukturyzacja (tagi XML), limity, ograniczenia, styl, adaptacja, kalibracja. Zasada **"generalizowanie generalizacji"**: zamiast reguły per błąd, kształtuj proces myślowy obejmujący całą kategorię błędów. Instrukcje systemowe nie są mechanizmem bezpieczeństwa — jailbreaking jest możliwy, bezpieczeństwo implementuj w kodzie.

## In-context Learning / Few-shot
Model uczy się wzorców z przykładów w kontekście. Sekcja `<examples>` działa lepiej niż opis zasad słowami, szczególnie dla klasyfikacji/formatowania. Obecny trend: zgeneralizowane zasady w prompcie + dynamiczne ładowanie przykładów/dokumentów przez narzędzia agenta zamiast statycznego wbijania.

## Chain-of-Thought / Reasoning Tokens
Generowane tokeny rozumowania traktowane jako osobny sygnał (`reasoning: { effort: "medium" }`) i mierzone w `usage.output_tokens_details.reasoning_tokens`. Teza lekcji: "modele myślą generując tokeny" — nie należy agresywnie optymalizować zwięzłości.

## Semantic Events & Rendering
Interakcja z LLM to seria zdarzeń (tekst, tool calls, reasoning, obrazy, błędy, potwierdzenia), nie prosta para Q&A. Architektura: oddziel LLM ↔ warstwę stanu aplikacji ↔ UI. Używaj semantycznych zdarzeń z ID/typem/metadanymi zamiast przesyłania surowego tekstu. Streaming markdown→HTML wymaga obsługi niekompletnych fragmentów (code blocks, LaTeX, tabele).

## Dynamic / Generative UI
Kierunek rozwoju: od tekstu do dynamicznie generowanych interfejsów (MCP Apps, Apps SDK, JSON Render). AI generuje interaktywne komponenty i wizualizacje zamiast tylko tekstu. Proces może być wstrzymywany na potwierdzenie użytkownika lub wynik innego agenta.

## Multimodality
Zdolność modeli do obsługi różnych modalności (tekst, obraz, audio). Gemini Interactions API i OpenAI Responses API prowadzą w tej kategorii. Argument za projektowaniem architektury na wielu providerów.

## Agent State / Memory Schema
Modele danych dla czatbota (`conversations` + `messages`) vs systemu wieloagentowego (`sessions` + `agents` + `items`). Ta trzecia tabela `items` unifikuje wiadomości, tool calls i załączniki jako atomy interakcji. Struktura umożliwia dwukierunkową komunikację między agentami i zatwierdzanie akcji przez użytkownika.

## Web Search Grounding
Natywne narzędzie providerów (`tools: [{ type: "web_search_preview" }]` w OpenAI, suffix `:online` w OpenRouter) pozwala modelowi odpytać web w trakcie generowania. Lekcja pokazuje ekstrakcję źródeł z odpowiedzi (`web_search_call`, `url_citation`) i owijanie fraz w `<span class="grounded">` jako wzorzec groundingu odpowiedzi.

## Open-source Models & Quantization
Formaty GGUF (cross-platform, llama.cpp) vs MLX (Apple Silicon). Kwantyzacja jako kompresja wag (Q2-Q8); Q4/Q5 = rozsądny balans. Wymagania VRAM/Unified Memory jako główne ograniczenie praktyczne. Testowanie przez LM Studio lokalnie lub OpenRouter zdalnie. Open-source bije komercyjne w wybranych zadaniach specjalistycznych i przy wymogach prywatności.

## Tokenization Costs
Token = fragment tekstu (część słowa, słowo, znak). Język polski zużywa ~50-70% więcej tokenów niż angielski — realny wpływ na koszt, latencję i efektywne wykorzystanie okna kontekstowego.
