# s02e02 — koncepty

## Prompt Injection
Zewnętrzne dokumenty wczytane do kontekstu jako wektor ataku — email z instrukcją może zostać wykonany dosłownie. Filtrowanie przez LLM niewiarygodne. Skuteczna obrona wyłącznie środowiskowa: ograniczenie narzędzi, potwierdzenia UI, zawężenie źródeł. Analogia XSS.
- **s01e02** — problem + mitygacje środowiskowe
- **s01e03** — brak widoczności otoczenia w publicznych MCP
- **s02e02** — zewnętrzne dane jako wektor ataku, walidacja plików, XSS jako analogia

## Query Transformation
Zapytanie użytkownika rzadko pasuje 1:1 do zasobów. W kontekście RAG: generowanie synonimów, cross-language retrieval (polskie zapytanie → angielskie wyniki, embedding wielojęzyczny łaczy gdy FTS zawodzi), rozbijanie na pod-zapytania. `_index.md` jako mapa treści czytana przed eksploracją — daje orientację gdy wiedza bazowa modelu niewystarczająca.
- **s01e02** — ekspansja, mapa wiedzy, pytania doprecyzowujące
- **s02e01** — warianty słów kluczowych z wyników, problem języka w prompcie
- **s02e02** — cross-language retrieval, `_index.md`, synonimy i pod-zapytania w RAG

## Security & Sandboxing
Walidacja plików na wejściu do systemu RAG: rozmiar, format, mime-type, źródło. Moderacja treści. Linki tymczasowe. Programistyczna kontrola dostępu do dokumentów. Agent otrzymuje tylko błędy i wskazówki, nigdy nie steruje bezpieczeństwem.
- **s01e02** — bezpieczeństwo przez kod, dry-run, potwierdzenia UI
- **s01e03** — destructive action safeguards, FS_ROOT egzekwowany programistycznie
- **s01e05** — Bearer auth, AbortSignal, graceful shutdown
- **s02e02** — walidacja plików wejściowych, moderacja treści, wygasające linki

## Agentic RAG / Agentic Search
Fundamenty infrastruktury RAG wspierające strategię wyszukiwania: indeksowanie, chunking, embedding, hybrid search, RRF. Architektura od prostego (filesystem + grep) do złożonego (dedykowane silniki). Kluczowe: agent generuje dwa zapytania (keywords + semantic) dopasowane do mocnych stron każdego silnika.
- **s02e01** — czterofazowa strategia, niezależna od źródła danych
- **s02e02** — infrastruktura RAG, hybrid search FTS+semantic+RRF, progresja architektury

## Chunking
Cztery strategie podziału dokumentów na przeszukiwalne fragmenty (200-500 słów): znaki (stałe okno), separatory (rekurencyjna hierarchia), kontekst (separatory + LLM prefix per chunk), tematy (LLM dzieli od podstaw). Każda strategia to inny tradeoff jakości/metadanych/kosztu API. Heading index z dual detection + sampling ze środka chunka. Resilient JSON parsing dla outputu LLM.
- **s02e02** — cztery strategie, tabelaryczne porównanie, implementacje na jednym dokumencie

## Embeddings
Model embeddingowy zamienia tekst na wektor liczb opisujący znaczenie. Wybór modelu: wielkość, wymiary, okno kontekstowe, dane treningowe. Ograniczenie: terminy spoza treningu (żargon, nazwy wewnętrzne) — model nie opisze ich poprawnie. Relevant ≠ Similar — podobieństwo wektorowe nie oznacza istotności dla zapytania, co uzasadnia hybrydę z FTS.
- **s02e02** — wektory znaczeniowe, cosine similarity, ograniczenia modeli, relevant vs similar

## Hybrid Search
Łączenie FTS5 (BM25, leksykalny) i wyszukiwania wektorowego (cosine, semantyczny) przez Reciprocal Rank Fusion. RRF: `score = Σ 1/(k + rank)`, k=60 — dokument wysoko w obu listach awansuje. Agent generuje dwa zapytania: keywords dla FTS, natural language dla embeddingu. Cross-language: gdy FTS zawodzi na różnych językach, embedding ratuje wynik. Graceful degradation: błąd API → FTS-only.
- **s02e02** — FTS5+sqlite-vec+RRF, dual queries, cross-language, graceful degradation

## RAG
Retrieval-Augmented Generation — pipeline od surowych dokumentów do odpowiedzi agenta. Komponenty: indeksowanie, chunking, embedding, wyszukiwanie, prezentacja kontekstu. Sześć wyzwań skuteczności: wiedza bazowa przesłania search, niekompletne wyniki, brak świadomości zasobów, brak kontekstu użytkownika, multimodalność, instruction dropout od masy wyników. Inkrementalna indeksacja (SHA256), FTS triggers, batch embedding.
- **s02e02** — pełny pipeline w SQLite, architektura od filesystem do dedykowanych silników, wyzwania skuteczności

## Instruction Dropout
Zjawisko: rozbudowany kontekst z zewnętrznych źródeł powoduje pomijanie instrukcji systemowych. Uwaga modelu przesuwa się na nowe treści kosztem zasad. Potwierdzone w "How Many Instructions LLMs Follow at Once" i "Reasoning on Multiple Needles In A Haystack". Kontrmiary: krótkie chunki, subagenci z własnymi oknami, powtarzanie kluczowych instrukcji w wynikach narzędzi.
- **s02e02** — definicja zjawiska, paper evidence, kontrmiary środowiskowe

## Context Engineering
Instruction dropout jako konsekwencja przeładowania kontekstu — kompromis między ilością dostarczonych danych a jakością instruction following. W RAG: masowe wyniki wyszukiwania pogłębiają problem. Dekompozycja na subagentów jako strategia zarządzania oknem.
- **s01e01** — API bezstanowe, routing kontekstu
- **s01e02** — system plików jako pamięć, kontrola okna
- **s01e05** — zarządzanie oknem, 3-etapowa kompresja
- **s02e01** — sygnał vs szum
- **s02e02** — instruction dropout od zewnętrznych danych, subagenci jako dekompozycja

## Observability / Token Tracking
Token accounting w systemie RAG — śledzenie input/output/reasoning/cached tokenów per request jako osobna statystyka, nie ad hoc logowanie. Istotne przy wielokrotnych wywołaniach search i embedding API w pętli agenta.
- **s01e03** — token usage tracking w pętli
- **s01e05** — Langfuse traces/spany/generacje
- **s02e01** — breakdown kosztów per sesja
- **s02e02** — token accounting w pipeline RAG, osobna statystyka
