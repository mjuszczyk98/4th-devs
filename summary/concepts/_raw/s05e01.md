# s05e01 — koncepty

## Architecture Primitives
Projektowanie na prymitywach (zdarzenia, artefakty, items) zamiast na funkcjonalnościach (czat, pliki, obrazy). Prymitywy to najprostsze elementy z których buduje się struktury wyższego rzędu — pozwalają rozbudowę bez przebudowy gdy czatbot ewoluuje w system wieloagentowy.

## Provider Abstraction
Warstwa tłumaczeń między providerami w tej samej logice: różnice w strukturze API (system message), ustawieniach (reasoning), thought signatures (Gemini 3+), ograniczeniach (Anthropic). Konkretne przykłady: web search realizowany różnie per provider, automatyczne wykrywanie modeli reasoningowych. Własna logika + SDK rekomendowana nad OpenRouter/bibliotekami dla projektów produkcyjnych.

## Multi-agent Architectures
Łączenie czterech wzorców w jednym systemie: Orchestrator (dynamiczne tworzenie aktorów), Blackboard (współdzielony stan, brak bezpośredniej komunikacji między agentami), DAG (deterministyczne rozwiązywanie zależności), Events (SSE z buforem i replay). System ~1500 linii TS bez frameworka agentowego.

## DAG Task Scheduling
Deterministyczny scheduler zarządzający cyklem życia zadań na dynamicznym DAG-u. Stany: todo → in_progress → done|waiting|blocked. `findReadyTasks` filtruje po statusie + zależnościach, `unblockParents` kaskadowo promuje po zakończeniu dzieci. Stale task recovery resetuje `in_progress` do `todo` po awarii. Trójpoziomowa obsługa błędów: API → Actor → Task z recovery state.

## Agent Configuration as Markdown
Agent jako dane (`AgentDefinition` — instrukcje + narzędzia + maxSteps), nie klasa. Zachowanie wyłania się z konfiguracji + uniwersalnego runtime'u. Rejestr agentów mapuje nazwę → definicję; `create_actor` szuka w rejestrze, wymaga jawnej konfiguracji dla ad-hoc. Różnicowanie wyłącznie zestawem narzędzi.

## Observational Memory
Per-task-thread kompresja w dwufazowym cyklu. Observer: raw items → split head/tail (~30% chronione) → LLM → obserwacje XML z priorytetami 🔴🟡🟢. Reflector: wielopoziomowa kompresja (gentle → aggressive → heavy) gdy obserwacje przekroczą threshold. Generation counter śledzi stratę kompresji. Adaptacja Mastra Observational Memory.

## Context Engineering
Czterowarstwowa kompozycja promptu aktora: (1) prompt prefix (rola, zasady, rejestr), (2) task memory (skompresowane obserwacje, sealed), (3) task snapshot (pełny stan sesji — aktorzy, zadania, zależności, artefakty), (4) raw items od `lastObservedSeq`. Actor nie zarządza kontekstem — scheduler wstrzykuje pełny snapshot. LLM nigdy nie dostaje pełnej historii.

## Agent Memory Schema
Model domenowy: Session, Actor, Task, Item, Artifact, Relation. Generyczny `Relation` z `from_kind/from_id/relation_type/to_kind/to_id` jako polimorficzny join zastępujący specjalistyczne tabele. Graf zadań nad FileStore (JSON) — DAG nie wymaga dedykowanej bazy grafowej.

## Heartbeat Orchestration
Delta vs S03E02: plan zadań kształtowany dynamicznie przez agenta zarządzającego, nie określony z góry. Elastyczność rośnie, przewidywalność maleje. Scheduler pull-based z max 20 rund.

## Event-driven Architecture
Każda zmiana stanu emituje event przez SSE z buforem max 500. Replay na nowe połączenie — klient dostaje całą historię. Fundament dla observability, ewaluacji, guardrails, UI.

## Agent Loop / Agentic Loop
Actor loop z maxSteps iteracji: processTaskMemory → buildTaskRunInput → LLM call → tool execution. Narzędzia terminalne (`complete_task`, `block_task`) przerywają loop. Pozostałe zwracają `continue`. Invocation+result jako Items z sekwencją.

## Prompt Caching
SHA-256 z `session_id:task_id:actor_id` jako `prompt_cache_key` w Responses API — stabilny klucz per task-actor daje maksymalne wykorzystanie cache. Cache hit rate raportowany w podsumowaniu sesji.

## Error Handling w narzędziach
Trójpoziomowa obsługa: API (retry z exponential backoff na 429/5xx) → Actor (`RecoverableActorError`) → Task (recovery state z `explicit_block`/`llm_transient`/`runtime_error`). Stale task recovery resetuje zawieszone zadania.

## File Reference Resolver Pattern
Agenci referencjonują artefakty przez `{{file:path}}` zamiast kopiować treść — oszczędność tokenów + unikanie niespójności.

## Observability / Token Tracking
`TokenUsage` akumulowany na sesji z dwóch źrórzeł: actor loop + operacje pamięci (observer + reflector). Cache hit rate: `cachedTokens / inputTokens * 100%`. Podsumowanie sesji z pełnym breakdown.

## Security & Sandboxing
Scope uprawnień systemu plików dla agentów — ryzyko nieprzewidzianych akcji destrukcyjnych (usunięcie katalogu) wymaga sandboxingu innego niż w klasycznej aplikacji.
