# s03e01 — koncepty

## Observability / Token Tracking
S03e01 rozszerza koncept o pełną architekturę observability: hierarchia zdarzeń (Session → Trace → Span → Generation → Tool → Event), scentralizowana rura monitorująca, kontekst propagowany przez AsyncLocalStorage bez jawnych parametrów. Dekorator `withGenerationTracing` opakowuje adapter LLM — transparentny tracing bez modyfikacji logiki agenta. Graceful degradation: zero overhead gdy tracing wyłączony, z wyjątkiem propagacji kontekstu agenta potrzebnej logice biznesowej. Kontekst vs metadata: dane specyficzne dla projektu (licencje, uprawnienia, CRON) vs informacje o połączeniu (wersja, środowisko). Wersjonowanie promptów przez hash-gated sync (SHA256) z jednostronną synchronizacją kod → platforma.

## Ewaluacja (Evals)
Ustrukturyzowany test oceniający zachowanie modelu/agenta pod kątem metryk. Nie gwarantuje poprawności — weryfikuje stopień dopasowania, nie deterministyczną zgodność. Trzy poziomy oceny: programistyczna (regex, schema), LLM-as-judge, człowiek. Składniki: zadanie (input + expected output), dataset (syntetyczny + produkcyjny, iteracyjny), evaluator (funkcja → score 0–1). Offline eval (przed publikacją, CI/CD) vs online eval (w trakcie działania, automatyczna ocena naruszeń). Projektowanie datasetów: pokrycie, różnorodność, balans. Odpowiedzi narzędzi szybko stają się większością kontekstu — uwzględniać w testach. Strategia kosztowa: dane strukturalne → kod (darmowe), oceny subiektywne → LLM (zoptymalizowane). Decyzja biznesowa: evals kosztują czas i tokeny, na wczesnych etapach tempo zmian je wyklucza, przy stabilnym produkcie — trudno bez.

## Guardrails
Trzeci filar obok observability i evals: moderacja, filtrowanie, blokowanie niepożądanych zapytań na wejściu i wyjściu. Niezależna warstwa od ewaluacji, ale korzystająca z tych samych sygnałów. Reaguje w czasie rzeczywistym, complementuje mechanizmy wbudowane w system. Online eval może pełnić rolę dodatkowego safety-netu.

## Agent Debugging
Debugowanie agentów różni się fundamentalnie od debugowania kodu — zmiana instrukcji naprawiająca jeden przypadek może psuć inne, brak determinizmu uniemożliwia gwarancję braku regresji. Playground na platformach pozwala re-execucję z pełnym kontekstem (system prompt + messages + tools) i manipulację parametrami. Technika uzasadnienia: poproszenie modelu o wyjaśnienie wyboru narzędzi ujawnia przyczyny problemu. AI wspiera analizę — eksport JSON interakcji z platformy i omówienie z modelem.

## Prompt Versioning
Wersjonowanie promptów systemowych: prompty w kodzie (Git trackuje) + korelacja ze statystykami i historią uruchomień na platformie observability. Jednostronna synchronizacja (kod → platforma) przez hash SHA256 treści — push tylko gdy treść się zmieni. Lokalny stan w JSON trzyma mapping name→{hash, version}. Wynik: każda generacja ma referencję PromptRef (name + version). Traci się łatwe przełączanie z UI, zachowuje spójność z kodem i kompletność statystyk.
