# s01e03 — koncepty

## Tool Design for LLMs
Lekcja koncentruje się na projektowaniu interfejsu narzędzi pod ograniczenia i cechy LLM-ów. Kluczowe zasady: proste nazwy, minimalna liczba parametrów, opcjonalny `details`, automatyczna resolucja niepełnych danych, konsolidacja powiązanych akcji przez parametr `mode`/`operation`. Interfejs kształtują cztery perspektywy: fundament API, przejrzystość schematu, dopasowanie złożoności do modelu, realny use case.

## API Audit for Agents
Przed projektowaniem narzędzi należy zaudytować API pod kątem kompletności akcji (CRUD), identyfikacji zasobów (nazwa vs ID), spójności kontraktu (content vs body), kompletności odpowiedzi (nie `201` bez zasobu), złożoności wieloetapowej (agregacja w jedno narzędzie) oraz asynchroniczności/rate limitu/paginacji (obsługa kodowa, nie agentowa).

## Tool Consolidation
Zamiast mapować 1:1 endpointy API, grupuj narzędzia po domenie działania. Przykład: 13 narzędzi Filesystem MCP → 4 (`fs_read`, `fs_write`, `fs_search`, `fs_manage`). Cel to balans między liczbą schematów a przejrzystością, nie minimalizacja. Smoke test: mały model lokalny.

## Dynamic Tool Responses (Hints)
Odpowiedź narzędzia powinna nieść kontekst którego model nie może wydedukować: recovery dla błędów, status zasobu, sugestie następnego kroku, dostępne wartości przy walidacji, automatyczne korekty. Hinty mają sens niezależnie od jakości modelu.

## Destructive Action Safeguards
Mechanizmy chroniące przed błędnymi akcjami modelu: checksum (wykrywanie współbieżnych zmian), dryRun (podgląd bez wykonania), historia zmian (rollback bez udziału modelu), ograniczenia zakresu usuwania (pojedyncze pliki, puste katalogi), koncepcja "kosza". Uprawnienia (`FS_ROOT`) egzekwowane programistycznie, nie przez prompt.

## Model Context Protocol (MCP)
Szczegółowe wprowadzenie: trzy role (Host, Client, Server), pięć capabilities (Tools, Resources, Prompts, Sampling, Elicitation), dwa transporty (STDIO, Streamable HTTP). MCP jest komplementarny wobec natywnego function-callingu — schematy scalają się w jedną listę, model nie widzi różnicy. Serwer raz zbudowany podłącza się do dowolnego hosta.

## MCP Sampling & Elicitation
Odwrócenie kierunku komunikacji: serwer → klient. Sampling pozwala serwerowi MCP korzystać z LLM bez własnych kluczy API (leverage relacji klienta z providerem). Elicitation pozwala serwerowi prosić użytkownika o dane/potwierdzenie przez formularz JSON Schema. Demo: `summarize_with_confirmation` łączący oba w jednym handlerze.

## MCP Transports
STDIO (subprocess, jeden użytkownik per proces, lokalny system, klucze w env) vs Streamable HTTP (multi-user, OAuth 2.1 z PKCE, VPS/Cloudflare Workers). STDIO dla narzędzi lokalnych, HTTP jako domyślny wybór produkcyjny. MCPB pakietuje serwer STDIO w jeden plik z auto-konfiguracją.

## Multi-Server Tool Routing
Przy wielu serwerach MCP host prefixuje nazwy narzędzi nazwą serwera (`files__fs_read`). Routing: split po `__`, pierwsza część = klient docelowy. Eliminuje kolizje nazw między serwerami.

## Declarative MCP Configuration
Wzorzec `mcp.json` — jeden plik deklaruje wszystkie serwery z mieszanymi transportami, komendami, argumentami, env vars, URL-ami. Klient wczytuje konfig, tworzy odpowiedni transport per serwer, prefixuje nazwy narzędzi.

## Agent vs Workflow
Kryterium wyboru: proces o stałej, zdefiniowanej sekwencji → workflow (kontrola, niskie ryzyko, brak elastyczności). Proces gdzie kolejność zależy od kontekstu → agent (elastyczność, autokorekcja, wyższe ryzyko, lepsze rezultaty).

## Agentic Loop
Podstawowa pętla agenta: chat → extract tool calls → execute parallel → append results → repeat. Kluczowe elementy produkcyjne: limit iteracji jako safety guard (np. 80), historia tool calls w kontekście, niezależne zarządzanie stanem per zadanie (Set inProgress). Model sam decyduje o kolejności mając zdefiniowany cel i schemat pracy w system prompcie.

## MCP Security
Narzędzia wewnętrzne mają pełny kontekst (znane zasoby/użytkownicy/przepływ). Publiczne serwery MCP — brak widoczności otoczenia wymaga programistycznych ograniczeń: blokady, limity, walidacje, anonimizacja. Niektóre akcje w ogóle nie powinny być dostępne dla agentów. Prompt injection pozostaje otwartym problemem.

## MCP Server Development Process
Ustrukturyzowany proces budowy serwera z AI: szablon → `API.md` z dokumentacją → agent czyta manual szablonu → propozycja narzędzi → konsolidacja → projekt schematów z perspektywy LLM → implementacja → weryfikacja małym modelem lokalnym jako smoke test.

## File Reference Resolver Pattern
Wzorzec `{{file:path}}` — zamiast kodować binaria base64 w argumentach narzędzia, model deklaruje intencję placeholderem. Resolver rekurencyjnie podmienia placeholdery na base64 z dysku tuż przed wywołaniem MCP tool. Redukuje użycie kontekstu i koszty.

## Token Usage Tracking
Kumulacja `input_tokens`, `cached_tokens`, `output_tokens` z Responses API w pętli agenta jako podstawowa obserwowalność kosztów i cache hit rate.
