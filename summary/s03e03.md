# S03E03 — Kontekstowy feedback wspierający skuteczność agentów

## Wyzwalacze autonomii

Agent reaguje na pięć typów triggerów: **wiadomości** (człowiek/agent), **hooki** (zdarzenia wewnętrzne), **webhooki** (zdarzenia zewnętrznych usług), **cron** (harmonogram), **heartbeat** (regularna kontrola stanu). Wszystkie mogą trafiać do **jednego punktu wejścia** — agent dynamicznie interpretuje zadanie w NL i dobiera akcję do otoczenia.

Każda akcja to domyślnie nowa sesja. Sesja użytkownika może być "nieskończona" z kompresją kontekstu (Observational Memory). Sesje niezależne = domyślność; współdzielenie = decyzja architektoniczna.

## Proaktywność

L02E01: otoczenie wpływa na agenta **tylko podczas interakcji** z użytkownikiem. S03E03: pełna integracja + **uniezależnienie od człowieka**. Heartbeat cyklicznie sprawdza `tasks.md` — gdy nic nie wymaga akcji, agent się nie udźwiękawia (invisible). Gdy tak — proaktywnie działa na bazie metadanych (czas, lokalizacja, pogoda, status urządzenia). Treść `tasks.md` ustalana ręcznie lub zarządzana przez agenta.

## Odkrywanie otoczenia

Dwa tryby:
- **Aktywny** — agent wywołuje narzędzia (przeglądarka, search, API)
- **Pasywny** — metadane programistycznie wstrzykiwane do wiadomości (czas, lokalizacja, pogoda, otwarte aplikacje)

Klucz: **nie wysyłać wszystkiego za każdym razem** — scope'ować do potrzeb bieżącej interakcji. Niewymagane zdarzenia nie zanieczyszczają kontekstu.

## Feedback w pętli narzędzi

Najważniejszy koncept lekcji: **informacja zwrotna wewnątrz tool loop podnosi skuteczność agenta w obrębie sesji i między sesjami.**

Trzy warstwy, rosnąco zaawansowane:

### 1. Statyczne sugestie po błędach

Po porażce narzędzia — wstrzyknięcie contextual hintu do wyniku. Nie zmienia promptu, wzbogaca output:

```
Error: querySelector returned null
[feedback] The expected DOM element is missing. The page structure may have changed.
```

### 2. Dynamiczne interwencje

Gdy 2+ porażki z rzędu → wstrzyknięcie dodatkowej wiadomości `role: 'user'` do konwersacji (screenshot, inny selector). Po odzyskaniu → sugestia zapisania discoveries. To **runtime intervention** modyfikujący input do modelu, nie prompt engineering.

### 3. Hook lifecycle z guard railsem

| Hook | Sytuacja | Akcja |
|------|----------|-------|
| `beforeToolCall` | Przed wywołaniem narzędzia | Nagranie stanu (np. ścieżka pliku audio) |
| `afterToolResult` | Po wyniku narzędzia | Aktualizacja flag fazowych, detekcja błędów |
| `beforeFinish` | Agent chce zakończyć | **Blokada** jeśli wymagane kroki nie ukończone |

`FeedbackTracker` rejestruje każdy wynik wywołania narzędzia (sukces/porażka), liczy kolejne porażki, generuje hinty specyficzne dla typu błędu (timeout → scroll, null selector → page structure changed, 3+ porażki → zmiana strategii).

Pełna lista hooków (AI SDK): `onStart → onStepStart → onToolCallStart → onToolCallFinish → onStepFinish → onFinish`. Projekty produkcyjne rozszerzają o streaming, context building, session compression, user activity, agent statuses. Najprostszy przykład: generowanie nazwy sesji (auto-rename "New Chat" → opisowy tytuł).

`beforeFinish` to deterministyczny guard nad niedeterministycznym agentem — wymusza pełny flow przed zakończeniem. Wyjątek: błędy narzędzi lub przekroczenie max turns.

## Nauka między sesjami

Dwa komplementarne wzorce:

**Instruction files** — agent zapisuje "discoveries" do `{site}-discoveries.md`. System prompt: "najpierw sprawdź czy istnieje instruction file". `{site}.md` zawiera gotowe recipes (JS snippets). Agent uczy się struktury świata, nie historii konwersacji.

**Profile accumulation** — profil ucznia z `weakAreas` akumuluje odkryte `trait_id` między sesjami. Trait taxonomy: `grammar.articles`, `pronunciation.stress`, `fluency.fillers` — ustandaryzowana klasyfikacja błędów. Profil małe (role, goals, weakAreas), pełne sesje w osobnych plikach `sessions/*.json`.

## Fazowanie agenta

Agent z fazami: każda faza z **własnym promptem i zestawem narzędzi**. Model dostaje tylko to co potrzebuje — redukcja complexity i halucynacji. Faza 1 (tworzenie eventów): narzędzia do kontaktów, miejsc, search, kalendarza. Faza 2 (powiadomienia): kalendarz, trasy, powiadomienia. **Brak overlapu** = model nie próbuje wysyłać powiadomień w fazie tworzenia.

State machine z automatycznym resetem po ukończeniu fazy (`tryCompletePhase`) — wielokrotne tury analizy w jednej sesji.

## Narzędzia wywołujące dodatkowe API

Narzędzia nie są tylko proxy do danych — mogą **samoistnie wywoływać dodatkowe zapytania do modeli**. `listen` wywołuje osobne zapytanie (ASR + analiza), `feedback` wewnętrznie wywołuje `speak` (TTS). Agent widzi tylko końcowy wynik — narzędzie enkapsułuje sub-pipeline modeli i API.

MCP przewiduje to przez **Sampling** — serwer MCP odwraca komunikację i prosi klienta o wywołanie API. Rzadko wspierane, ale adresuje dokładnie ten przypadek.

## Enrichment — łączenie rozproszonych danych

Agent jako węzeł łączący szczątkowe informacje z wielu źródeł (kontakty, miejsca, kalendarz, mapa, pogoda). Oryginalna wiadomość wzbogacana o dane z narzędzi — adresy, maile, trasy, warunki pogodowe. Wartość biznesowa w standaryzacji procesów (np. sprzedażowych), nie pojedynczych interakcjach.

Bezpieczeństwo: agent tworzący eventy może pomylić adresy / wpuścić poufne dane do opisu. Rozwiązanie: deterministyczne interfejsy dla wrażliwych operacji (np. programistyczne zarządzanie kontaktami) — etap nie przez agenta.

## OpenAI Responses API vs Gemini Interactions API

| Cecha | OpenAI Responses | Gemini Interactions |
|-------|-----------------|---------------------|
| Kontynuacja sesji | `previous_response_id` | `previous_interaction_id` |
| Wymuszanie schema | `strict: true` na tool params | `response_format` jako JSON Schema na **całą odpowiedź** |
| Audio output | Nie natywnie | `response_modalities: ['AUDIO']` — base64 PCM |
| Thinking control | Brak | `thinking_level: 'low'/'medium'/'high'` |
| Output types | `text`, `function_call` | `text`, `function_call`, `thought`, `audio` |
| Store | `store: true/false` | Brak odpowiednika |

`previous_response_id` / `previous_interaction_id` — API trzyma kontekst po stronie serwera. Klient nie wysyła historii — drastyczna redukcja token usage.

Gemini `response_format` wymusza schema na **całej odpowiedzi modelu**, nie tylko argumentach narzędzi — gwarantuje strukturę np. `{ transcript, confidence, strengths, issues, segments }`.

Gemini TTS: model `gemini-2.5-flash-preview-tts` z `speech_config: { voice, language }`. Output: raw PCM → ręczna konwersja do WAV (44-bajtowy header w kodzie).

## Techniki defensywne

**JSON auto-naprawa** — `jsonrepair` jako fallback po `JSON.parse` (trailing commas, brakujące cudzysłowy, markdown fences). Przy sukcesie: hint doklejany do wyniku ("Keep arguments strict JSON next time"). `parseJsonLoose()` — wyciąganie JSON z markdown fences, fragmentów tekstowych, błędnych struktur. Kolekcja kandydatów (full string, substring `{`→`}`, substring `[`→`]`) → parse first success. **Must-have** w każdym agentic system.

**Session persistence (Playwright)** — login flow z `headless: false` → `context.storageState()` → zapis do JSON. Kolejne uruchomienia w headless z załadowanymi cookies. Custom User-Agent + disable automation detection.

**Content overflow** — treść strony przekraczająca context window → zapis do pliku (`pages/*.txt` + `pages/*.struct.txt` — pełny tekst z numeracją linii + uproszczona struktura DOM). Agent przeszukuje przez `fs_search` na `.struct.txt` zamiast zgadywać z HTML. Alternatywa: subagent. Zasada: **nigdy nie ładuj pełnego tekstu strony do konwersacji**.

## Krajobraz webowy

Narracja przesuwa się od blokowania agentów (AI Labyrinth) do współpracy: Cloudflare [Markdown for Agents](https://blog.cloudflare.com/markdown-for-agents/) (strukturyzowana treść dla agentów), Chrome [WebMCP](https://developer.chrome.com/blog/webmcp-epp) (natywny protokół komunikacji). Nie eliminuje scrapowania — platformy bez API (Goodreads po zamknięciu rejestracji) nadal wymagają agenta z przeglądarką.

Gradacja automatyzacji: powtarzalne na dużej skali → klasyczny bot (bez LLM). Gdy niewystarczające → agent AI z Playwright/Puppeteer. Na większą skalę → Browserbase, kernel.sh.

## Filozofia projektowa

**Ofensywne projektowanie** — zamiast tylko "gdzie system ma problemy", pytać "jakie opcje przeniosą go na wyższy poziom". Interfejs głosowy (nawet prosty STT→LLM→TTS) w proaktywnym systemie okazuje się bardzo użyteczny.

**Rola człowieka** — agent w tle musi mieć interfejs dla sytuacji gdy akcja autonomiczna niemożliwa (konflikty nazw, brakujące dane, błędy integracji). Hooki mogą obejmować weryfikację — nie tylko potwierdzenie niezaufanej akcji, ale uzupełnienie brakujących informacji.

Problemy z agentami często leżą po stronie użytkowników. Onboarding i komunikacja produktu = element architektury na równi z kodem. Nie każdy agent musi być uniwersalny — wysoka specjalizacja jest akceptowalna.

## Przykłady kodu — progresja wiedzy

### 03_03_calendar — Fazowanie + symulowane dane + webhooki

**Delta względem S03E02:** Agent reagujący na webhooki zewnętrznych usług, z fazową architekturą i w pełni symulowanym otoczeniem.

- **Dual-phase agent** — dwie fazy, każda z własnym promptem i zestawem narzędzi. Brak overlapu.
- **Symulowane dane relacyjne** — kontakty z preferences, miejsca z tags i hours, trasy z 3 opcjami (walk/drive/transit), pogoda po godzinach. Zero zewnętrznych API — pełne testowanie end-to-end.
- **Mutowalny environment state** — `setTime()` + `setUserLocation()` + `buildMetadata()` generujący blok `<metadata>` wstrzykiwany do każdej wiadomości.
- **Fuzzy search ze scoringiem** — `find_event` weighted: title match (token overlap + substring) + time proximity (odległość w minutach). Handles rozbieżności między webhook title a calendar title.
- **Webhook payloads jako trigger** — `event.upcoming` z `eventTitle`, `startsAt`, `minutesUntilStart` → agent sam decyduje o trasie i pogodzie → powiadomienie.
- **Rich terminal observability** — ~290 linii loggera z kolorami, tabelami, ikonami. Nie cosmetic — debugowalność systemu agentowego w batch mode.

```ts
export const buildMetadata = (): string => {
  const weather = getWeatherAt(date, hour);
  const lines = [
    `<metadata>`,
    `Current time: ${dayName}, ${date} ${timeStr} CET`,
    `Location ID: ${state.userLocation.placeId}`,
    `Location: ${state.userLocation.name}`,
    `Weather: ${weather.description} — ${weather.tempC}°C`,
    `</metadata>`,
  ];
  return lines.join('\n');
};
```

### 03_03_browser — Feedback-driven interventions + self-learning

**Delta względem `calendar`:** Agent z runtime feedback system, interwencjami, i persistent learning między sesjami.

- **FeedbackTracker** — per-tool history sukcesów/porażek. `consecutiveFailures()` + `generateHints()` produkują contextual porady na podstawie typu błędu (timeout, null selector, invalid JSON).
- **Runtime interventions** — dynamiczne wstrzykiwanie wiadomości do konwersacji: screenshot tip po 2+ porażkach, discovery tip po odzyskaniu. Nie modyfikują promptu — modyfikują input do modelu.
- **Instruction files** — `instructions/{site}.md` z gotowymi recipes (JS snippets), `instructions/{site}-discoveries.md` zapisywane przez agenta po odzyskaniu z porażek.
- **Structural DOM dump** — po nawigacji: pełny tekst z numeracją linii + uproszczona struktura DOM (tag, klasy, data-testid, role, aria-label). Agent szuka selectorów przez `fs_search` na `.struct.txt`.
- **MCP integration z tool coalescing** — narzędzia z MCP file server (`fs_read`, `fs_search`, `fs_write` via stdio) wrzucane do tego samego tool registry co browser tools. Model nie wie o źródle narzędzi.
- **`previous_response_id` chaining** — kontynuacja konwersacji bez resendowania historii. API trzyma kontekst.

```ts
const generateHints = (tool: string, outcome: ToolOutcome, error?: string): string[] => {
  if (tool === 'click' && error?.includes('timeout'))
    return ['The element may not be visible. Try scrolling first.'];
  if (recentOf(tool, 3).filter(e => e.outcome === 'fail').length >= 3)
    return ['Multiple failures detected. Consider changing strategy.'];
  return [];
};
```

### 03_03_language — Hook lifecycle + multimodalność + Gemini API

**Delta względem `browser`:** Agent z hook-based lifecycle control, multimodalnym I/O (audio in → text+audio out), i Gemini Interactions API.

- **Hook lifecycle z phase tracking** — `beforeToolCall` nagrywa stan, `afterToolResult` aktualizuje flagi (`listen_done`, `feedback_done`, `session_saved`), `beforeFinish` wymusza pełny flow. `tryCompletePhase()` snapshottuje ukończoną fazę i resetuje state machine.
- **Gemini Interactions API** — raw fetch do `/v1beta/interactions`. `response_format` jako JSON Schema wymuszający strukturę **całej odpowiedzi**. `response_modalities: ['AUDIO']` dla TTS. `thinking_level` kontrolujący głębokość reasoning.
- **Gemini TTS** — model `gemini-2.5-flash-preview-tts` z `speech_config: { voice: 'kore', language: 'en-us' }`. Output: raw PCM → ręczna konwersja do WAV.
- **Tool-chaining** — `feedback` wewnętrznie wywołuje `speak` (kolejne zapytanie API). Jedno narzędzie enkapsułuje sub-pipeline modeli.
- **Trait taxonomy** — ustandaryzowana klasyfikacja błędów: `grammar.articles`, `grammar.verb_tense`, `pronunciation.stress`, `pronunciation.vowels`, `fluency.fillers`. Akumulacja `weakAreas` w profilu między sesjami.
- **`parseJsonLoose()`** — wyciąganie JSON z: markdown fences, fragmentów tekstowych, błędnych struktur. Kolekcja kandydatów → parse first success.
- **Multi-phase w jednej sesji** — po ukończeniu fazy state machine się resetuje, agent analizuje kolejny plik audio. Aktualizacja profilu raz na koniec wszystkich faz.

```ts
beforeFinish: (finalText) => {
  const missing: string[] = [];
  if (!state.phase_flags.listen_done) missing.push('listen to audio');
  if (!state.phase_flags.feedback_done) missing.push('generate feedback');
  if (!state.phase_flags.session_saved) missing.push('save session');
  if (!state.profile_updated) missing.push('update profile.json');
  if (missing.length === 0) return { allow: true, missing: [] };
  return {
    allow: false,
    missing,
    inject_message: 'You must complete these before finishing:\n'
      + missing.map(m => `- ${m}`).join('\n'),
  };
},
```
