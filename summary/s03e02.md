# S03E02 — Ograniczenia modeli na etapie założeń projektu

## Wspieranie, nie automatyzacja

Pełna automatyzacja procesów przez AI = zły pomysł niezależnie od obszaru. Narzędzia wspierające ludzi (etykietowanie, priorytetyzacja, filtrowanie, szkicowanie), nie zastępujące ich. Błędy w trybie wsparcia mają ograniczone konsekwencje — człowiek pozostaje w pętli.

Wąskie, wyspecjalizowane narzędzia > all-in-one agenci dla całej organizacji. Prompt injection i halucynacje uniemożliwiają bezpieczne systemy autonomiczne o szerokim zakresie.

| Pułapka | Dlaczego nie działa | Alternatywa |
|---------|---------------------|-------------|
| RAG na "całą bazę wiedzy" | Koszt zebrania danych z różnych źródeł przewyższa wartość | Prototyp → zawęź zakres (onboarding, sprzedaż, managerowie) |
| Czatbot na stronę | Budowanie relacji przez bota = wizerunkowe ryzyko | Agenci w tle zintegrowani z panelem obsługi |
| Auto-email marketing | Skuteczność niska, ryzyko wizerunkowe wysokie | Wsparcie istniejących procesów |
| Agent na całą organizację | Prompt injection + halucynacje = brak kontroli | Wyspecjalizowane narzędzia per obszar |

**Przepaść autonomii:** odchylenia tolerowane gdy człowiek jest w pętli → przekreślają sens systemu autonomicznego. Błąd co piąty raz przy ustalonym procesie = system bezużyteczny. Przejście do autonomicznych agentów to nowa klasa problemów programistycznych, nie inkrementalny wzrost trudności.

Regularnie aktualizuj przekonania — to co niemożliwe dziś, jutro może być trywialne. Kluczowa umiejętność: określanie, co w danej chwili nie ma znaczenia.

## Podział odpowiedzialności: model vs kod

Model nie musi nieść całej logiki. **AI proposes, system decides** — model proponuje akcje, security properties (uprawnienia, scope danych, klasyfikacja kontaktów) determinuje kod.

Jeśli model musi "przeliczać w pamięci" — przenieś do kodu. Obliczenia w sandboxie są deterministyczne, w kontekście LLM — nie. Czasem właściwa odpowiedź to wycofanie się z pomysłu i oddanie procesu ludziom.

### Sandbox code execution

Dane zbyt duże na kontekst LLM (np. 240 plików, 150K linii) → agent generuje kod, który przetwarza dane deterministycznie. 6-10 kroków zamiast setek zapytań.

Architektura 3-procesowa:
```
Proces główny (agent loop) ↔ MCP server (STDIO) ↔ Sandbox (Deno)
```

Sandbox nie ma bezpośredniego dostępu do narzędzi hosta. **HTTP Tool Bridge** — lokalny HTTP server mapujący tool names na endpointy. Prelude wstrzykiwany przed kodem sandboxa daje czysty async API (`tools.fs_read({path: ...})`).

4 poziomy uprawnień sandboxa (`safe` / `standard` / `network` / `full`) mapowane na flagi Deno — sandbox dostaje tylko uprawnienia niezbędne dla danego poziomu.

Skalowanie architektury 3-procesowej (główny + MCP STDIO + sandbox) na produkcję z tysiącami użytkowników = wyzwanie. Rozwiązania rosnące: Cloudflare Sandbox, Daytona. Koszt komplikacji uzasadniony — precyzja, szybkość, koszty i redukcja halucynacji dramatycznie lepsze niż w standardowym podejściu agentowym.

Ostrzeżenie: kalkulacje w kodzie są poprawne, ale poprawność wczytania danych nie jest gwarantowana. Krytyczne dane = ścisły nadzór + deterministyczne procesy wszędzie gdzie możliwe.

### 5 osi optymalizacji wydajności

1. **Tokeny wejściowe** — prompt systemowy, definicje narzędzi, kontekst
2. **Cache** — dotyczy tylko tokenów wejściowych
3. **Tokeny wyjściowe** — skrócenie wypowiedzi + ograniczenie kroków agenta
4. **Liczba zapytań** — szczególnie sekwencyjnych nieobjętych cache
5. **Mniejsze modele** — szybsze, tańsze, ale z mniejszymi możliwościami

Sandbox + generowanie kodu łączy wszystkie osie jednocześnie.

## Architektura bezpieczeństwa

**Zero-trust:** nie ufamy agentom w zakresie udostępniania informacji i pracy na zewnętrznych źródłach. Każdy punkt kontaktu ze światem zewnętrznym = niebezpieczny.

### Warstwy obrony

1. **Prompt systemowy = publiczny** — żadnych sekretów, bo zostanie wydobyty
2. **Programistyczna kontrola dostępu** — zmiana zachowania agenta przez input nie może prowadzić do eskalacji uprawnień
3. **Capability stripping** — agent dostaje narzędzia TYLKO na czas ich potrzeby. Faza kategoryzacji = pełne narzędzia, faza szkicowania = zero narzędzi
4. **Izolacja danych** — scope'owanie kontekstu przez warstwy: Layer 1: izolacja konta (tylko `shared` + matching account). Layer 2: scope'owanie contact type (`internal` > `trusted_vendor` > `client` > `untrusted`)
5. **Runtime mutex** — blokada KB na konkretne konto na czas sesji draft, z `try/finally` gwarantującym zwolnienie
6. **Audit trail** — każde wywołanie narzędzi KB loguje: kto pytał, o co, co zwrócono (z id i scope), co zablokowano i dlaczego

### Izolacja kontekstu (scoped access)

Agent pracujący na danych wielu kont musi mieć programistycznie wymuszone ograniczenia widoczności wiedzy **w momencie generowania treści**:

- **Kategoryzacja** — brak ograniczeń, agent widzi wszystko
- **Szkicowanie odpowiedzi** — dostęp do KB zablokowany do konta + kategorii dopasowanych do typu kontaktu nadawcy

Lock binarny, egzekwowany kodowo — agent nie może go ominąć przez prompt. `mark_for_reply` to wirtualne narzędzie: AI mówi "odpowiedz", SYSTEM determinuje `contactType` i scope KB. AI nie może eskalować uprawnień.

### Filtr prompt injection

Osobny prompt oceniający intencje inputu, **całkowicie odseparowany** od głównego wątku konwersacji. Zwraca ustalony ciąg znaków ("bezpieczne"/"niebezpieczne"), weryfikowany **programistycznie**, nie przez LLM. Atakujący nie ma fizycznej możliwości poznania frazy weryfikacyjnej. Ominięcie możliwe, ale trudne dla typowego użytkownika. Dodatkowo: zabezpieczenie prawne + jawne informowanie użytkownika o użyciu AI.

## Heartbeat orkiestracja

Zamiast sztywnego DAG-a — **pull-based task queue** z round-based reconciliacją. Proces jest dynamiczny: zadania mogą wymagać dodatkowych kroków, nowe zadania mogą powstać w trakcie.

Trzy komponenty: **kontrakty** (struktura planu, zadań, zależności, statusów, wymaganych capabilities), **heartbeat** (manager przydzielający zadania), **pamięć** (filesystem-based, per-agent i komunikacja między agentami).

### Kontrakt celu (Goal Contract)

`goal.md` z YAML frontmatter: objective, must_have, forbidden, budżet kroków, budżet replanowania, max zadań, warunki wymagające aprobaty człowieka. LLM planner → strukturyzowany plan z zadaniami. Jeśli cel nieosiągalny → `no-go.md` z powodami, brakującymi capabilities i sugerowanymi akcjami. Zapobiega cichym fiaskom.

### Planowanie + walidacja

Plan walidowany: deps, capabilities, forbidden phrases, cycle detection. Niepoprawny → do 2 prób naprawy przez LLM (repair loop). Materializacja jako pliki markdown (task files) — czytelne, wersjonowalne, debugowalne bez bazy danych.

### Task state machine

```
open → in-progress → done
                   → blocked → open (retry)
                   → waiting-human → open (z human_answer)
```

Każdy cykl heartbeat: reconcile dependencies → resolve waiting tasks → dla każdego agenta: claim task → run session → persist status → flush events.

Agenci claimują zadania, nie są do nich przypychani. Zadania deklarują `required_capabilities`, agenci deklarują `capabilities`. Matching po capabilities, nie nazwach agentów. CapabilityMap skanuje szablony agentów (`*.agent.md`) budując mapę umiejętności.

### Replanowanie

ReplanPatch: add_task, split_task, reassign_owner, change_dependencies, de_scope_task, cancel_open_task. Ograniczone budżetem. Uruchamia się automatycznie gdy: blocked ratio ≥ 35% zadań lub dowolne zadanie ma attempt ≥ 2.

Każdy patch walidowany: ID/filename conflicts, istnienie agenta, cykle w zależnościach, pokrycie capabilities. Patche `add_task`/`split_task`/`cancel_open_task` wymagają manual approval (chyba że `autoHuman=true`). Wynik zapisywany jako snapshot JSON.

## Observational Memory

Dla długotrwałych sesji agentowych — kompresja historii konwersacji do ustrukturyzowanych obserwacji z priorytetyzacją (🔴/🟡/🟢). Dwa komponenty: **Observer** (ekstrakcja po >30K nieobserwowanych tokenów) i **Reflector** (kompresja po >40K tokenów obserwacji, progresywny poziom kompresji). Zapobiega context window exhaustion bez utraty kluczowych informacji. Per-agent-session, umożliwia komunikację między agentami przez system plików. (Szczegóły architektury: s02e03.)

## Knowledge anti-patterns

Zamiast uczyć AI "jak zrobić", powiedz mu "co idzie źle". Pliki knowledge zawierają wyselekcjonowane zasady będące wynikiem trial-and-error (np. "blank pages w PDFKit", "cursor drift po positioned blocks"). Prompt wymusza czytanie tych plików jako pierwszy krok.

To jest RAG, ale retrieval domain to **anty-wzorce i edge cases**, nie surowa wiedza.

## Człowiek jako final gate + kaskadowanie zdarzeń

Agent nigdy nie wykonuje nieodwracalnych akcji. Wysyłanie maili, publikowanie dokumentów, postowanie — domena człowieka. Agent kategoryzuje, etykietuje, szkicuje, filtruje, wzbogaca. Człowiek decyduje.

Etykieta przypisana przez agenta = zdarzenie obserwowalne → wyzwalacz dla downstream. Label "Bug Report" → tworzenie issue w Jira → inny agent ocenia złożoność → automatyczne rozwiązanie. Człowiek też może przypisywać etykiety, uruchamiając te same kaskady. Wzorzec: **output agenta = event = trigger**. Rozszerza wartość bez zwiększania ryzyka.

## Przykłady kodu — progresja

### 03_02_code — Sandbox execution

**Czego uczy:** Przeniesienie ciężaru obliczeniowego z LLM do deterministycznego kodu w izolowanym środowisku.

Agent ma dwa narzędzia: `fs_read`/`fs_write` (przez MCP) i `execute_code` (sandbox Deno). Kod generowany przez LLM działa na surowych danych, nie w kontekście promptu.

Kluczowe elementy:
- HTTP Tool Bridge z prelude generującym obiekt `tools` z async proxy do MCP
- 4 poziomy uprawnień mapowane na flagi Deno (`--allow-read`, `--allow-net`, itd.)
- Deterministyczny generator danych: FNV-1a + mulberry32 PRNG seedowany stringiem — identyczne dane co uruchomienie
- Knowledge-guided generation: wymuszenie czytania plików z anty-wzorcami przed generowaniem kodu
- Dual runtime: Bun jako proces główny, Deno jako sandbox — osobne zarządzanie zależnościami

### 03_02_email — Izolacja danych + capability stripping

**Delta:** Dwufazowa architektura z programistyczną kontrolą dostępu i ewaluacją. Zamiast sandboxa — kontrola dostępu do wiedzy.

Dwie fazy:
- **Triage**: 12 narzędzi, pełny dostęp KB, etykietowanie maili, planowanie odpowiedzi
- **Draft**: 0 narzędzi, `lockKnowledgeToAccount(account)` — KB zablokowany do jednego konta + kategorii dopasowanych do typu kontaktu. Lock w kodzie (`access-lock.ts`), nie w prompcie

Kluczowe elementy:
- Runtime mutex na KB z `try/finally`, moduł-level state z error throwing na naruszenia
- System-determined classification — AI nie może eskalować uprawnień
- Adwersarz testowy: celowy email social engineering — runtime guard blokuje dostęp do danych drugiego konta
- Eval Langfuse: 4 suite'y testowe (triage, izolacja, język, malicious)
- Emergent language matching — KB zawiera politykę "odpowiadaj w języku nadawcy", eval weryfikuje bez library do detekcji języka
- Privacy by design: `store: false` na wszystkich wywołaniach AI
- Responses API zamiast Chat Completions

### 03_02_events — Multi-agent z heartbeat i kontraktami

**Delta:** Autonomiczny system multi-agentowy z heartbeat loop, memory management i goal contracts. Od pojedynczego agenta do orkiestracji.

Trzy nowe mechaniki:
- **GoalContract** → LLM planner → walidacja + repair → materializacja jako pliki zadań
- **CapabilityMap** skanuje `*.agent.md` budując mapę umiejętności do przypisania zadań
- **Observational Memory** kompresuje historię sesji — długotrwała praca bez przepełnienia kontekstu

Dodatkowe elementy:
- Agent Templates jako `.agent.md` — deklaratywne definicje w markdown z YAML frontmatter (model, tools, capabilities, system prompt)
- Event sourcing: JSONL + per-round markdown — pełna audytowalność
- Bounded replanning z walidacją patchy i snapshotami
- Token estimation z kalibracją: estymacja `chars/4` autokalibruje się na bazie rzeczywistego API-reported usage
- `request_human` — system pauzuje i czeka na decyzję człowieka
