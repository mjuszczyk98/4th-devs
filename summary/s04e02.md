# S04E02 — Aktywna współpraca z AI

## Wybór interfejsu jako decyzja architektoniczna

Interfejs determinuje model, integracje, personalizację i autonomię. Cztery komplementarne (nie wykluczające się) ścieżki — system produkcyjny to często kompozycja kilku.

| Ścieżka | Kontekst | Personalizacja | Trudność |
|---|---|---|---|
| CLI (Claude Code, Open Code) | Osobisty / sandbox | Najwyższa | Wymaga kompetencji technicznych |
| Serwery MCP podłączone do gotowego klienta | Most do istniejących interfejsów (Claude.ai, ChatGPT) | Ograniczona przez hosta | Niska |
| Komunikatory (Slack, Discord, Telegram) | Zespołowy | Umiarkowana | Niska |
| Dedykowane rozwiązanie | Dowolny | Pełna | Wysoka |

**Kryteria decyzyjne:**
- Grupa docelowa: techniczna → CLI/MCP; nietechniczna → web/komunikator/dedykowany UI
- Ekonomia: subskrypcje (ChatGPT, Claude) nieporównywalnie tańsze niż API przy dużym wolumenie — sprawdź istniejące umowy korporacyjne
- Złożoność systemu: im więcej autonomii w tle, uprawnień i dwukierunkowej interakcji z użytkownikiem, tym bliżej dedykowanego interfejsu

**ACP (Agent Client Protocol)** — protokół łączący IDE (JetBrains, Zed) z agentami (Codex, Cursor). Trend zacierania granic między narzędziami.

## Ograniczenia gotowych platform MCP

Integracja MCP z Claude.ai obnaża konkretne luki:

- **Brak samplingu** — serwer MCP nie może inicjować komunikacji z modelem ani użytkownikiem po stronie klienta
- **Ograniczona personalizacja instrukcji** — system prompt sterowany głównie przez opisy narzędzi i ich wyniki, brak dedykowanego panelu konfiguracji
- **Brak dwukierunkowej komunikacji** — agent nie może prosić o dodatkowe informacje ani raportować postępu w tle
- **Uprawnienia i wielowątkowość** — rozbudowane reguły dostępu (różne kampanie, różni klienci) trudne do zaimplementowania
- **Pełne wsparcie protokołu MCP rzadkie** — z [listy klientów MCP](https://modelcontextprotocol.io/clients) pełne wsparcie deklaruje niewiele pozycji

**Lista brakujących funkcjonalności** weryfikowana przy wyborze klienta: dostęp do historii wielu użytkowników (kanały Slack), profile asystentów per obszar, automatyczna konfiguracja (współdzielenie skilli), elastyczny interfejs (Artifacts), opcja wielu modeli (Raycast), różne formaty treści (NotebookLM), interakcje audio/wideo (Google AI Studio live), dostęp do funkcji systemowych (desktop), pełna prywatność (LM Studio).

Decyzja: podłączyć się do ekosystemu (szybkie wdrożenie, ograniczenia) czy zbudować własne (pełna kontrola, koszt budowy).

## Cztery filary personalizacji interfejsu

Jakość implementacji tych mechanik (nie ich obecność) decyduje o użyteczności.

1. **Profile (subagenci)** — dedykowane konteksty z własnymi ustawieniami, modelem i zasobami wiedzy. Przełączanie: menu, skróty, mention (`@`), automatyczne przez AI. Wzorzec z Claude Code.
2. **Umiejętności (skills)** — predefiniowane instrukcje wstrzykiwane intencjonalnie lub przez decyzję modelu. Przy dużej liczbie — konieczne wyszukiwanie i grupowanie (np. po agencie).
3. **Narzędzia (tools)** — nie samo wywołanie, lecz pełny cykl UI: prezentacja danych wejściowych → potwierdzenie → postęp → błąd → wynik + możliwość wstrzymania/anulowania.
4. **Workflow** — serie powtarzalnych akcji, hooki, zaplanowane zadania. Obecnie rzadkie w gotowych interfejsach.

## Mikro-akcje

Pojedyncze akcje AI przypisane do skrótów klawiszowych, gestów lub wyzwalaczy (Keyboard Maestro, BetterTouchTool, Siri Shortcuts). Niski koszt implementacji, najwyższy dzienny ROI.

| Typ akcji | Przykład |
|---|---|
| Odczyt zaznaczenia | TTS zaznaczonego tekstu z kontrolą tempa |
| Wyjaśnienie zaznaczenia | Definicja / kontekst z wiedzy modelu lub wyszukiwarki |
| Transformacja zaznaczenia | Korekta, tłumaczenie, parafraza, ekstrakcja |
| Transformacja kontekstowa | Auto-dopasowanie do domeny (np. github.com → styl dokumentacji) |
| Wizualizacja zaznaczenia | Tekst → precyzyjna grafika |
| Odnalezienie powiązań | Zaznaczenie → zapytanie do bazy wiedzy → deep-link do notatki |
| Opisanie zawartości schowka | Auto-prompt do generowania obrazu na podstawie zdjęcia w schowku |
| Lokalizacja zdjęcia | Zdjęcie z telefonu + metadane GPS → kontekstowa akcja |

Realizacja: skrypt + skrót / automatyzacja, lub lekka natywna aplikacja (Swift, Tauri, Electron, React Native) dla dostępu do funkcji systemowych.

## Meta-prompt jako wzorzec projektowy

Meta-prompt to instrukcja dla modelu, której celem jest **wygenerowanie innej instrukcji** przez rozmowę z użytkownikiem. Człowiek dostarcza dane, model wnosi wiedzę o dobrych praktykach prompt engineeringu. Rola człowieka fundamentalna — proces nie jest autonomiczny.

### Trójczłonowa struktura

1. **Dane** — kategorie informacji do zgromadzenia: cel, zakres, styl, narzędzia, wzorce, modele mentalne, ograniczenia, format, wyjątki. Wszystko, czego model nie wie z natury.
2. **Generator** — wiedza o prompt engineeringu: struktury, zasady specjalizacji/generalizacji, przydatne wyrażenia. Model stosuje automatycznie.
3. **Rezultat** — szablon końcowej instrukcji ze stałymi sekcjami: tożsamość, proces myślowy, zasady i ograniczenia, wiedza zewnętrzna, styl wypowiedzi.

### Sekcje implementacyjne

| Sekcja | Rola |
|---|---|
| **Proces** | Model informowany o celu: przeprowadzenie użytkownika przez pytania do instrukcji |
| **Strategia** | Zasady realizacji celu z dopasowaniem do sytuacji |
| **Dopasowanie** | Reguły specjalizacji pod domenę (kod, marketing, support...) + obsługa wymagań wykraczających poza domyślną strukturę |
| **Format** | Szablon końcowej instrukcji + sposób prezentacji (np. blok markdown z opcją „instalacji") |
| **Zasady** | Conducting rozmowy: czekaj na odpowiedź, pogłębiaj niejasności, **model nie zgaduje** |
| **Natywne funkcjonalności** | Generowanie nie tylko promptu, ale też ustawień agenta (model, tryby, integracje) jako wartości domyślne |
| **Generowanie** | Selekcja — brane pod uwagę wyłącznie elementy istotne dla bieżącego kontekstu |
| **Krytyczne zasady** | Podkreślenie najważniejszych reguł na końcu; część reguł powinna trafiać do metadanych wiadomości |

### Zastosowania

- **Onboarding marketplace** — użytkownik dodaje produkty → z opisów, strony i zdjęć automatycznie generowane instrukcje personalizujące (opisy produktów, kreacje reklamowe dopasowane do tonu marki)
- **Generowanie obrazu** — meta-prompt tworzy instrukcję edycji dopasowaną do konkretnego produktu i tonu, nie generyczny prompt
- **Subagenci (Claude Code, Cursor)** — automatyczne generowanie skilli i subagentów przez meta-prompt
- **Optymalizacja promptów** — automatyczne doszlifowywanie instrukcji agentów klasyfikujących, transformujących

Jakość promptów wciąż bezpośrednio przekłada się na skuteczność systemu. Meta-prompt **automatyzuje proces optymalizacji**, ale nie eliminuje potrzeby wiedzy o konstrukcji dobrych promptów.
