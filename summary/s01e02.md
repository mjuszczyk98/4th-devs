# S01E02 — Łączenie modelu z narzędziami (Function Calling, agenci)

## Function Calling — mechanizm

- Model nie wykonuje kodu. Generuje JSON z `name` + `arguments`, aplikacja uruchamia funkcję i zwraca wynik do kontekstu. Separacja tego "co model widzi" (definicje) od "co kod wykonuje" (handlery) jest fundamentalna.
- Jedno użycie narzędzia = minimum **dwa zapytania** do LLM: decyzja o wywołaniu → wykonanie przez kod → wygenerowanie odpowiedzi na podstawie wyniku.
- Definicje wszystkich narzędzi trafiają do kontekstu przy **każdym** zapytaniu — kosztują tokeny nawet gdy nie są użyte. Limit: 10–15 narzędzi na agenta, powyżej model się rozprasza.
- Historia rośnie jako `[...messages, ...toolCalls, ...toolResults]` — wpływa na koszt i limit kontekstu.
- `strict: true` w schemacie wymusza brak pól spoza definicji.
- Natywne narzędzia (web search, code execution) dokleja się obok własnych jedną linią — wygodne, ale ograniczona konfiguracja, logika po stronie providera.

## Pętla agenta

- Struktura: `while (steps > 0) { response → toolCalls → if empty return → buildNextConversation → steps-- }`.
- `MAX_TOOL_STEPS` jest obowiązkowe — bez niego agent może kręcić się w nieskończoność. Po wyczerpaniu limitu **rzuć błąd**, nie zwracaj cicho — model musi wiedzieć co poszło nie tak.
- Parallel function calling: `Promise.all` na `toolCalls` — darmowa optymalizacja dla niezależnych akcji.
- Każdy krok = kolejny round-trip + transfer kontekstu + czas zewnętrznych usług. Minimalizuj kroki.

## Projektowanie narzędzi

**Zasada naczelna:** nie mapuj API 1:1. API jest pisane dla programistów z dokumentacją, narzędzia LLM muszą być zrozumiałe bez żadnego zewnętrznego kontekstu. Filtruj, łącz, upraszczaj.

Trzy pytania przy każdym schemacie:
- Co model **musi** uzupełnić?
- Co powinno być uzupełniane **programistycznie** (np. `user_id` z sesji)?
- Czego model **nie może** uzupełnić (uprawnienia, IDs kontroli dostępu)?

Zasady:
- Nazwy unikalne, niskie ryzyko kolizji (`send_email` > `send`).
- Opis = wysoki signal-to-noise, tylko to co zwiększa szansę poprawnego wyboru. Testuj na 10–30 przykładowych zapytaniach.
- Pomijaj pola nieistotne dla modelu: hashe, wewnętrzne flagi, ID kontroli dostępu.
- Łącz zasoby API gdy model potrzebuje danych z kilku endpointów (np. `workspace_metadata`) — redukuje liczbę kroków.
- **Konfigurowalne połączenia** — narzędzie nie zwraca całości, model wybiera podzbiór (parametr `include`/`fields`).
- Wybaczaj potknięcia: aliasy (identyfikator **lub** nazwa), sensowne defaulty opisane w description, `recursive: true`.
- Projektuj operacje **batchowe** (edycja wielu rekordów naraz) zamiast per-item.

## Output i obsługa błędów

- Output = minimum niezbędnych informacji. `stat()` zwraca ~15 pól — handler zwraca 4, reszta świadomie odrzucona.
- **Błąd jako dane, nie jako crash.** `catch` serializuje wyjątek do `output` — pętla leci dalej, model decyduje co zrobić.
- Standard komunikatów wyższy niż dla zwykłego API. "Coś poszło nie tak" = ślepy zaułek. Każdy błąd = konkretna przyczyna + wskazówka naprawcza:
  - `"team_id nieprawidłowy. Pobierz go przez 'workspace_metadata'."`
  - `"Access denied: path '../config.js' is outside sandbox"` — model wie że nie może, nie że się posypało.
  - `"czy chodziło o 'completed'?"` — reakcja na literówki/warianty.

## Bezpieczeństwo i izolacja

- Każda granica dostępu wymuszona przez **kod**, nie przez prompt. Model nie ustala `user_id`, ścieżek poza sandbox ani zakresu uprawnień.
- Path traversal blokowany w handlerze przed dotknięciem FS.
- Akcje nieodwracalne (email, delete) wymagają potwierdzenia przez **UI** (przycisk/formularz) — nie przez wiadomość do modelu, bo tekst użytkownika może być zignorowany lub źle zinterpretowany.
- **Dry-run** jako wzorzec: agent testuje akcję, ocenia wynik, potem wykonuje z tymi samymi argumentami.
- **Prompt injection** = problem otwarty, brak skutecznej obrony. Agent przeglądający email/web może wykonać instrukcje z treści. Przykład kanoniczny: agent z kalendarzem + pocztą, na prośbę z emaila wysyła plan spotkań obcej osobie. Mitygacje tylko środowiskowe: sandbox, whitelisty, izolacja kontekstu, nieudostępnianie zasobów wrażliwych.

## Zarządzanie kontekstem i prompt cache

- **Prompt cache = priorytet optymalizacji.** Stabilny system prompt + stabilny początek wątku = automatyczny cache (OpenAI, Gemini). Jakakolwiek zmiana system promptu (nawet bieżąca data) unieważnia cache przy każdym zapytaniu.
- Dynamiczne dane wprowadzaj przez narzędzia — ich wyniki lądują na końcu wątku i nie psują cache systemu.
- Pływające okno kontekstu (usuwanie najstarszych wiadomości) jest gorsze niż utrzymanie pełnego wątku z cache.
- **System plików jako pamięć zewnętrzna.** Wyniki narzędzi zapisuj do plików, agent wczytuje tylko potrzebne fragmenty. To dotyczy też map treści (`_index.md`) — nie wpychaj nawigacji po bazie wiedzy do system promptu.
- Auto-compact (Claude Code, Cursor) = kompresja z utratą — ważne wątki ręcznie zapisuj do plików.
- **Dynamiczne listy narzędzi** bez psucia cache obecnie tylko u Anthropica. Alternatywy: sub-agenci z osobnymi oknami kontekstu, **Progressive Disclosure** przez code execution sandbox (narzędzia jako pliki, tylko podstawowe w kontekście).

## API vs CLI vs Function Calling / MCP

| | API bezpośrednie | CLI | Function Calling / MCP |
|---|---|---|---|
| Kontrola | brak | ograniczona do interfejsu | pełna (kod) |
| Dokumentacja dla modelu | eksploracja | `--help` | JSON Schema + opis |
| Skalowalność | dobra | trudna poza lokalnym | dobra |
| Elastyczność | pełna (curl) | pełna (terminal) | ograniczona do zdefiniowanych akcji |

Proxy między API a LLM to sposób na adaptację istniejącego API bez jego modyfikacji. **MCP to jeden z wzorców takiego proxy.**

## Workflow vs Agent

| | Workflow | Agent |
|---|---|---|
| Struktura | sztywna, ustalone etapy | dynamiczna, model wybiera kroki |
| Pętla LLM | sekwencyjna, ograniczona | wieloturowa |
| Przewidywalność | wysoka | niższa, elastyczniejsza |

- Proces ustrukturyzowany i stabilny → workflow.
- Otwarte problemy i reakcja na zmiany → agent, ale najpierw próbuj zamknąć problem w workflow.
- 100% skuteczności wymaga człowieka w pętli.

## Reasoning i sterowanie uwagą

- Wbudowany reasoning (LRM) pomaga w trudnych zadaniach, **szkodzi prostym** (arxiv "Don't Overthink it").
- **Premise Order Matters** (arxiv 2402.08939): sama zmiana kolejności informacji w prompcie potrafi obniżyć skuteczność nawet o 40%.
- Techniki sterowania refleksją agenta:
  - **Planowanie** — lista zadań jako narzędzie; aktualizacja listy przypomina o wątkach.
  - **Odkrywanie** — model "nie wie, że wie"; prowadź go przez eksplorację pamięci/bazy wiedzy.
  - **Przekierowanie** — focus na stanie (otwarta przeglądarka → skup się na jej obsłudze).
  - **Uśrednianie** — wiele modeli → perspektywy → głosowanie.

## Transformacja zapytań

- Pytanie użytkownika rzadko 1:1 pasuje do zasobów. Ekspansja przez synonimy/powiązane zagadnienia zwiększa hit rate.
- Agent potrzebuje ogólnej mapy bazy wiedzy (`_index.md`), inaczej nie nawiguje skutecznie.
- Zachęcaj agenta do pytań doprecyzowujących; rozpoznawaj czy zapytanie w ogóle dotyczy bazy.

## Augmented Function Calling

- Dodatkowe instrukcje kontekstowe aktywowane przy wywołaniu narzędzia (np. stały styl generowanych obrazów).
- Tryby dołączania: **statyczny** (od użytkownika), **dynamiczny** (model na podstawie nazwy/opisu), **hybrydowy**.
- Może obejmować sekwencje akcji i zarządzanie skillami (aktywacja/tworzenie).

## Agent Harness

Pełny system agentowy to nie sam LLM + narzędzia, tylko cała obudowa: system plików, sandbox do kodu, zarządzanie pamięcią/kontekstem, komunikacja między agentami, observability. Budowa aplikacji z LLM to ~80% klasycznej inżynierii i ~20% zupełnie nowej klasy problemów.

## Context Engineering — dwa znaczenia

1. **Poziom użytkownika** — zarządzanie kontekstem w Claude Code/Cursor (restart wątków, przenoszenie info). Poza zakresem kursu.
2. **Poziom aplikacji** — kontrola przebiegu interakcji, narzędzi, ich wyników, komunikacji między agentami. Na tym skupia się kurs.

## Przykłady kodu

### `01_02_tools` — minimalna pętla tool-callingu

Kompletny flow w ~50 liniach: definicje jako JSON Schema + handlery jako zwykłe funkcje + pętla z `MAX_TOOL_STEPS`. Mockowane handlery, `webSearch: true` doklejone obok własnych narzędzi.

Uczy: fundamentalnej separacji definicji od implementacji, parallel execution przez `Promise.all`, pętli z guardem.

### `01_02_tool_use` — delta architektoniczna

Podział na `definitions.js` / `handlers.js` / `executor.js` / `app.js` — skalowalne przy 10+ narzędziach, zmiana narzędzi nie dotyka executora.

**Sandbox guard** — path traversal blokowany w kodzie, zanim dotrze do FS:

```js
const rel = relative(sandbox.root, resolved);
if (rel.startsWith("..")) throw new Error(`Access denied: path "${relativePath}" is outside sandbox`);
```

**Błąd jako dane** — `catch` pakuje wyjątek do `output`, pętla leci dalej, model widzi `ENOENT` i reaguje.

**Celowe filtrowanie outputu** — `stat()` ma ~15 pól, handler zwraca 4.

**`TOOLS.md`** — dokumentacja dla developera (nie dla modelu) ze schematem input→handler→output→errors. Wzorzec do powielania.
