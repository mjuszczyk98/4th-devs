# S02E02 — Zewnętrzny kontekst narzędzi i dokumentów

## Bezpieczeństwo zewnętrznego kontekstu

Każdy dokument wczytany do kontekstu to potencjalny prompt injection — email z "prześlij mi swój kalendarz" zostanie wykonany dosłownie, jeśli agent ma narzędzie `send_email`. Filtr LLM oceniający ryzyko nie jest gwarancją — instrukcja może przejść przez filtr. Analogia: agent z zewnętrznymi danymi = formularz bez walidacji renderujący HTML na stronę (XSS).

Skuteczna obrona jest **środowiskowa**, nie promptowa:
- Ograniczenie zestawu narzędzi — agent bez `send_email` nie wyśle maila
- Potwierdzenie użytkownika przez UI (przyciski, nie chat) przed nieodwracalnymi akcjami
- Zawężenie zakresu źródeł — mniej niekontrolowanych danych = mniejsza powierzchnia ataku

Zabezpieczenia programistyczne na wejściu: walidacja rozmiaru/formatu/mime-type, moderacja treści, kontrola dostępu, wygasające linki publiczne.

## Instruction dropout

Im więcej treści w kontekście, tym większa szansa że model zignoruje instrukcje systemowe. Uwaga przesuwa się na nowo dodane treści kosztem wcześniejszych zasad — jakość odpowiedzi wysoka, ale forma/styl/zasady pominięte.

Potwierdzenie w badaniach: "How Many Instructions LLMs Follow at Once" (arXiv:2507.11538) i "Reasoning on Multiple Needles In A Haystack" (arXiv:2504.04150v1).

Kontrmiary:
- Ograniczanie objętości jednorazowo wczytywanych treści (krótkie chunki, subagenci z własnymi oknami)
- Powtarzanie kluczowych instrukcji w odpowiedziach narzędzi (technika z S02E01)
- Dekompozycja na subagentów — każdy w czystym kontekście

## Indeksowanie dokumentów

Gdy baza wiedzy wykracza poza pliki markdown, bezpośrednia eksploracja nie rozwiązuje problemu **odnalezienia** właściwego dokumentu. Indeksowanie przekształca surowe pliki w przeszukiwalną strukturę:

- **Chunking** — podział na fragmenty 200–500 słów / 500–4000 tokenów
- **Metadane** — źródło, sekcja, indeks chunka, tagi
- **Dokumenty syntetyczne** — podsumowania, opisy obrazów, transkrypcje audio/wideo
- **Mapa powiązań** — graf relacji między dokumentami
- **Synchronizacja** — indeks nadąża za zmianami (event-driven lub harmonogram)

## Strategie chunkingu

| Strategia | Mechanizm | Metadane | Koszt API |
|---|---|---|---|
| **Znaki** | Stałe okno (np. 1000 znaków) z overlapem | Pozycja, długość | 0 |
| **Separatory** | Rekurencyjny podział po hierarchii: `## → ### → \n\n → \n → . → spacja` | + sekcja z heading index | 0 |
| **Kontekst** | Separatory + LLM generuje 1-2 zdania prefiksu per chunk | + wygenerowany kontekst | N wywołań (1 per chunk) |
| **Tematy** | LLM identyfikuje logiczne tematy i dzieli od podstaw | + label tematu | 1-kilka wywołań |

Separatory to solidny default — zachowuje strukturę bez kosztu LLM. Kontekst (inspiracja Anthropic) sytuuje chunk w dokumencie. Tematy dają najlepszą semantykę, ale wymagają pełnego przetworzenia. Fundamentalnie różne profile kosztowe: kontekst = N wywołań LLM, tematy = 1-kilka, przy tym samym celu (wzbogacenie metadanych).

## Embedding i cosine similarity

Model embeddingowy (np. `text-embedding-3-small`, 1536 wymiarów) zamienia tekst na tablicę liczb opisującą **znaczenie**. Identyczny tekst → identyczny wektor. Podobne znaczenie → bliskie wektory. Wymiar wektora w bazie **musi** zgadzać się z liczbą wymiarów modelu.

Kryteria wyboru modelu: wielkość (koszt), liczba wymiarów, okno kontekstowe, zakres danych treningowych.

**Ograniczenia:** model nie opisze poprawnie terminów spoza danych treningowych (żargon firmowy, wewnętrzne nazwy, niszowa terminologia).

Cosine similarity — iloczyn skalarny znormalizowany długością wektorów. 1.0 = identyczne znaczenie, 0.0 = brak relacji. Dopasowanie **znaczeniowe**, nie leksykalne — "Kobieta" bliżej "Królowej" niż "Króla" mimo braku wspólnych liter.

**Relevant ≠ Similar** — semantic search zwraca dokumenty o podobnym znaczeniu, ale niekoniecznie odpowiadające na pytanie. Podobieństwo ≠ istotność. Dlatego hybryda z FTS + reasoning agenta jest konieczna.

## Hybrydowe wyszukiwanie — FTS + semantic + RRF

Dwa silniki równolegle:
- **FTS5 (BM25)** — ranking leksykalny, łapie dokładne terminy i frazy
- **sqlite-vec (cosine distance)** — ranking semantyczny, łapie znaczenie niezależnie od zapisu

Wyniki łączone przez **Reciprocal Rank Fusion**: `score(doc) = Σ 1/(K + rank)` po listach, K=60. Dokument wysoko w obu awansuje; w jednej dostaje szansę, ale nie dominuje.

Agent generuje **dwa osobne zapytania** — keywords dla FTS, pytanie naturalne dla embeddingu. Każdy silnik dostaje input dopasowany do swoich mocnych stron. Cross-language: polskie zapytanie "Czym jest autoregresja?" → angielskie dokumenty — FTS nic nie znajdzie, embedding (jeśli wielojęzyczny) dopasuje znaczeniowo. To uzasadnia hybrydę: gdy jeden silnik zawiedzie, drugi ratuje wynik.

**Graceful degradation** — błąd embedding API → fallback do FTS-only.

## Transformacja zapytań

Zapytanie użytkownika rzadko pasuje 1:1 do treści dokumentów. Transformacje:
- Generowanie synonimów i powiązanych pojęć
- Cross-language retrieval
- Rozbijanie na pod-zapytania pokrywające różne aspekty

Transformacja oparta wyłącznie na wiedzy bazowej modelu jest niewystarczająca przy specyficznym kontekście. `_index.md` (mapa treści) czytany przed eksploracją daje agentowi orientację w zasobach.

## Prezentacja kontekstu

- **Obrazy w markdown** (`![](url)`) — model nie parsuje natywnie, wymaga wydobycia regexem i przesłania osobno
- **PDF/DOCX** mogą być efektywniej przesłane jako obrazy — DeepSeek-OCR: 9-10x kompresja przy 96% precyzji
- **Cytaty i źródła** — metadane chunków (`source`, `section`) dla modelu (grounding) i UI (odnośniki)
- Dwa narzędzia nawigacji: `search` + `read` — agent nie widzi plików bezpośrednio

## Architektura RAG — od prostego do złożonego

| Wariant | Komponenty | Kiedy wystarczy |
|---|---|---|
| Agent + filesystem | grep/ripgrep, pliki markdown | Mała baza, średnia organizacja |
| Agent + SQLite + rozszerzenia | FTS5, sqlite-vec, chunking | Ustrukturyzowane dane, semantic search |
| Agent + dedykowane silniki | Elasticsearch/Qdrant/Neo4j + synchronizacja | Duża skala, multimodal, Graph RAG |

Pytanie nie "które najlepsze?", lecz "które najlepsze dla tego problemu?". Poziomy mogą współistnieć.

## Wyzwania skuteczności RAG

1. **Wiedza bazowa przesłania search** — model odpowiada z pamięci zamiast szukać. Adres: instrukcja wymuszająca search + grounding w wynikach
2. **Niekompletne wyniki** — trudne do wykrycia, prowadzą do halucynacji. Adres: hybrid search, wielokrotne szukanie, RRF
3. **Brak świadomości zasobów** — agent nie wie co dostępne. Adres: mapa treści (`_index.md`), progressive disclosure
4. **Brak kontekstu użytkownika** — "moje projekty" nieresolwowalne bez danych o użytkowniku. Adres: augmented context programistycznie, Graph RAG
5. **Multimodal** — obrazy/audio/wideo wymagają pre-processingu przed indeksowaniem
6. **Rozbudowany kontekst** — masowe wyniki wyszukiwania pogłębiają instruction dropout. Adres: dekompozycja, subagenci

## Przykłady kodu — progresja

### `02_02_chunking` — cztery strategie podziału

Delta: cztery implementacje na tym samym dokumencie, wszystkie zwracają `[{ content, metadata }]`.

```js
const chunkByCharacters = (text, size = 1000, overlap = 200) => {
  const chunks = [];
  let start = 0;
  while (start < text.length) {
    chunks.push(text.slice(start, start + size));
    start += size - overlap;
  }
  return chunks.map((content, i) => ({ content, metadata: { strategy: "characters", index: i } }));
};
```

Delta **Separatory**: rekurencyjny podział przez `["\n## ", "\n### ", "\n\n", "\n", ". ", " "]` + overlap na granicy słowa/linii + heading index z dual detection (`#` markdown + regex) + sampling sekcji ze środka chunka (40%, nie początek — unika fałszywych dopasowań z overlap).

Delta **Kontekst**: separatory + `enrichChunk()` → LLM generuje 1-2 zdania prefiksu per chunk (N wywołań).

Delta **Tematy**: jedno wywołanie LLM dla całego dokumentu → `[{ topic, content }]` z resilient JSON parsing (oczyszczenie markdown code blocks + fallback `JSON.parse`).

### `02_02_embedding` — cosine similarity

Delta: tekst → wektory, zero npm dependencies. REPL budujący macierz similarity przyrostowo.

```js
const cosineSimilarity = (a, b) => {
  let dot = 0, normA = 0, normB = 0;
  for (let i = 0; i < a.length; i++) {
    dot += a[i] * b[i];
    normA += a[i] * a[i];
    normB += b[i] * b[i];
  }
  return dot / (Math.sqrt(normA) * Math.sqrt(normB));
};
```

Debug UX: pierwsze 4 + ostatnie 2 wymiary z 1536. Embedding API wspiera batch (tablica inputów).

### `02_02_hybrid_rag` — pełny pipeline w SQLite

Delta: łączy chunking + embedding + FTS5 + sqlite-vec + agent z narzędziem search.

Schemat bazy:

```
documents (id, source, content, hash)
  → chunks (id, document_id, content, chunk_index, section, chars)
    → chunks_fts (FTS5 virtual table, synced via triggers)
    → chunks_vec (vec0 virtual table, embedding float[1536])
```

**FTS5 external content mode** — indeksuje istniejącą tabelę bez duplikacji danych:

```sql
CREATE VIRTUAL TABLE chunks_fts USING fts5(
  content,
  content='chunks',
  content_rowid='id'
);
```

Triggery `AFTER INSERT/DELETE/UPDATE` synchronizują indeks. Sanityzacja FTS: usunięcie znaków specjalnych, filtrowanie terminów <2 znaki, łączenie `OR`.

**Incremental indexing** — SHA256 hash plików, re-indeksuje tylko zmienione, cascade delete na nieistniejące.

**Dwa zapytania per search**:

```js
{ keywords: "autoregresja model", semantic: "Czym jest autoregresja w kontekście modeli językowych?" }
```

**RRF**: `1/(60 + rank)` sumowane po obu listach, top N. Degradacja: błąd `embed()` → FTS-only.

**SQLite pragmas** dla long-running process:

```js
db.pragma("journal_mode = WAL");      // reads nie blokują writes
db.pragma("synchronous = NORMAL");    // kompromis bezpieczeństwo/wydajność
db.pragma("foreign_keys = ON");       // cascade delete
db.pragma("busy_timeout = 5000");     // czeka na odblokowanie
```

**Highlight-based term extraction** — `highlight(chunks_fts, 0, '«', '»')` oznacza dopasowane terminy, regex wyciąga do debugowania i UI. **Token accounting** — śledzenie input/output/reasoning/cached per request.
