# S05E05 — Nowa Rzeczywistość

## Odwrócenie ról: człowiek jako supervisor AI

IntelliSense → AI Completion → AI generujące 90%+ kodu z człowiekiem jako supervisorem. Kluczowe kompetencje: dostarczanie kontekstu, kierunek rozwoju, decyzje architektoniczne, quality assurance — nie pisanie logiki.

AI kompresuje czas implementacji, nie czas zrozumienia domeny. Wonderlands powstał w kilkanaście dni przy pomocy AI, ale na bazie 3 lat doświadczeń z podobną aplikacją. Projektowanie i kształtowanie wymaga czasu niezależnie od szybkości generowania kodu.

## Technologia bez procesu = imponujący chatbot

System agentowy z MCP, sandboxem, przeglądarką i multi-agent delegation nie ma wartości bez zdefiniowanych procesów. Podłączenie Google Calendar by zapytać o dodanie wpisu jest mniej efektywne niż zrobienie tego ręcznie. Wartość pojawia się dopiero przy:
- spersonalizowanych procedurach opisujących jak agent ma używać narzędzi
- skryptach automatyzujących powtarzalne sekwencje
- cyklicznych wyzwalaczach uruchamiających agentów w tle

Zacznij od jednego powtarzalnego procesu, który działa — nie od architektury wszechświata. Taki struktury nie buduje się "w jeden wieczór". Jeśli po uruchomieniu agenci realizują jeden powtarzalny proces — to już sukces.

## Cyfrowy ogród jako baza wiedzy agenta

Strona www generowana z systemu plików tekstowych (markdown + frontmatter + wikilinks) pełni cztery role jednocześnie:

1. **Wiedza długoterminowa** — notatki, procedury, skille agenta
2. **Publikacja** — wybrane treści dostępne jako strona www
3. **Obszar roboczy** — agent czyta i modyfikuje pliki, tworzy nowe notatki
4. **Organizacja** — tagi, katalogi, wikilinks łączą wiedzę w sieć

Pliki z `visibility: private` w frontmatter ukryte w menu i chronione hasłem. Folder workspace kompatybilny z Obsidian — edycja tekstowa i agentowa uzupełniają się.

### Build pipeline

1. **Collect** — skanuj `.md` pliki
2. **Parse** — frontmatter (`gray-matter`) → exposure: public/protected/private/hidden
3. **Rewrite links** — `[[slug]]` → relative hrefs, detekcja broken links
4. **Render** — HTML z inline CSS, sidebar, TOC, listing, pagination
5. **Search** — Pagefind indeksy (osobne public + protected)

Auto-build worker fingerprintuje source (SHA-256 hash), debounce po ostatniej zmianie, automatycznie publikuje po udanym build.

## Daily Ops — wzorzec cyklicznej orkiestracji asynchronicznej

Agenci uruchamiani **równolegle** cyklicznie, każdy w własnej sesji, każdy zbiera dane z własnych integracji:

| Agent | Źródło | Akcja |
|-------|--------|-------|
| Calendar | Google Calendar | Wydarzenia nadchodzącego dnia i kluczowe z kolejnych dni |
| Tasks | Linear | Nieukończone + aktywne zadania |
| Mail | Gmail | Kluczowe powiadomienia wymagające uwagi |
| Newsfeed | Firecrawl / YouTube | Filtrowane informacje wg kryteriów, zestawienie z ostatnimi dniami |

Wyniki → jeden folder → agent agregujący → transkrypt + wersja audio → urządzenie mobilne. Agenci bez zależności między sobą — uruchamiani równolegle na godzinę przed deadline.

Uruchomienie przez `POST /v1/sessions/bootstrap` z API key + tenant ID + `initialMessage` kierujący agenta do plików opisujących proces. Brak wbudowanego crona — harmonogram z zewnątrz (cron, GitHub Actions, dowolny scheduler).

## MCP Code Mode — narzędzia przez kod, nie function calling

Gdy sandbox aktywny, agent nie ładuje definicji narzędzi do kontekstu. Zamiast tego **pisze i wykonuje kod**, który odkrywa i wywołuje MCP tools dynamicznie. Definicje narzędzi nie zajmują kontekstu — agent szuka ich na żądanie przez file-based IPC bridge (`mcp-bridge.mjs`): write request JSON → poll response JSON. Zero network, zero sockets.

Konfigurowalne per agent: wyspecjalizowany agent z małą liczbą narzędzi powinien mieć je w kontekście od razu (function calling). Agent z wieloma integracjami — Code Mode.

## Instrukcje dla agentów

Pułapki: zbyt ogólne instrukcje (agent się "domyśli"), założenie bezbłędności, oczekiwanie jakości bez doprecyzowania, długie interakcje (koszty tokenów!), oczekiwanie determinizmu.

Praktyki:
- Wskazuj konkretne **pliki** i **nazwy narzędzi** — foldery, kategorie
- Utrzymuj interakcje **tak krótkie jak możliwe** — koszty tokenów w własnych systemach agentowych są realne i przewyższają subskrypcje Claude Code
- Procedury proste, łączone wikilinkami — agent odkrywa je stopniowo
- Doprecyzowanie to jednorazowy wysiłek zwracający się wielokrotnie

## Nawyk > technologia

Techniczne możliwości bez nawyku korzystania = zero wartości. Klucz: dopasowanie kanału do istniejących zachowań:
- Discord → tam trafiają powiadomienia
- Telefon → prosta aplikacja mobilna / shortcut do czatu
- Poranny spacer → audio wygenerowane przez system ładuje się automatycznie

Wyrabianie nawyku = łączenie nowej aktywności z tym, co już robimy.

## Referencje — przepływ artefaktów między agentami

Agent A generuje obraz → referencja do agenta B → agent B nakłada znak wodny → publikuje w cyfrowym ogrodzie. Referencje integrują sandbox z systemem plików — wynik wykonania kodu staje się plikiem dostępnym dla innych agentów.

## Wyjątki produkcyjne

- **STDIO** — rejestracja serwerów STDIO nie może być dostępna w produkcyjnej aplikacji webowej (chyba że desktopowa z bezpośrednim kontaktem z komputerem)
- **Sandbox** — na produkcji można mieć więcej niż jeden (np. just-bash do nawigacji w FS, lo do wykonywania kodu)
- **Uprawnienia** — przed publikacją dodatkowa weryfikacja mimo zarządzania uprawnieniami i wymogu potwierdzeń

## Przykłady kodu — delta względem S05E04

### 1. Monorepo ze współdzielonymi kontraktami

Delta: od osobnych apps do **monorepo** z `packages/contracts` jako source of truth. Zod schemas → inferred types → `Replace<>` utility overlayuje string na branded IDs. Runtime validation + compile-time safety z jednego źródła. Zero build step — surowe `.ts` przez `exports` map.

```
wonderlands/
├── apps/
│   ├── client/          # Svelte 5 + Vite
│   └── server/          # Hono + TypeScript
├── packages/
│   ├── contracts/       # Shared API contract types (Zod schemas)
│   └── sandbox-runtime-lo/
├── scripts/
└── setup/
```

### 2. Agent Markdown — pełny DSL konfiguracyjny

Delta: znacznie bardziej zaawansowane niż proste `.agent.md` — pełny DSL z wersjonowaniem rewizji, checksumami SHA-256 i sandbox/kernel/memory/garden policy.

```yaml
slug: alice
kind: primary
model:
  provider: openai
  alias: gpt-5.4
  reasoningEffort: medium
sandbox:
  mode: enabled
  runtime: lo
  mountGarden: readWrite
memory:
  observationScope: thread_shared
garden:
  preferredSlugs: [my-garden]
subagents:
  - alias: researcher
    mode: async_join
tools:
  mcpMode: code
  nativeTools: [web_search, delegate_to_agent]
```

Agent nie jest tylko promptem — to pełna konfiguracja runtime z bezpieczeństwem, pamięcią, narzędziami i relacjami. Każda zmiana tworzy nową rewizję.

### 3. Context Compaction z Boundary Integrity

Delta: nowy mechanizm kompakcji kontekstu zachowujący integralność par tool call/response i pending waits. Granica tail iteracyjnie dostosowywana — kompakcja nie może ciąć w środku interakcji narzędzia.

### 4. Sandbox Writeback — Explicit Consent Model

Delta: sandbox edits domyślnie ephemeral — muszą być jawnie zatwierdzone przez `commit_sandbox_writeback`. Dwa tryby: `bash` (just-bash inline) i `script` (JavaScript ES module). Warstwa bezpieczeństwa między "agent wykonuje kod" a "agent modyfikuje pliki".

### 5. Polling Worker z wake support

Delta: zamiast skomplikowanych message queues, proste polling workers z adaptacyjnym opóźnieniem.

```typescript
createPollingWorker<T>({
  computeNextDelay: ({ result, wakeRequested }) =>
    wakeRequested || result ? 0 : pollIntervalMs,
  runOnce: async () => { ... },
  supportsWake: true,
})
```

Jeśli była praca → natychmiast sprawdź ponownie (delay=0). Jeśli nie → czekaj `pollIntervalMs`. `wake()` przerywa oczekiwanie. Wystarczające dla single-server SQLite deployment.

### 6. Multi-Lane Event Outbox

Delta: każdy event dispatchowany do 4 niezależnych lanes z osobną retry/quarantine logiką.

| Lane | Cel |
|------|-----|
| realtime | SSE do klientów |
| projection | Read model rebuild |
| background | Side effects (garden build, email) |
| observability | Langfuse tracing |

Failed observability events po max retries → quarantine. Duże payloady kompresowane i przechowywane osobno (sidecars).

### 7. Delegation Handoff Envelope

Delta: strukturalny kontekst przekazania między agentami — child dostaje pełny envelope z informacjami o parent, target, plikach.

```typescript
interface DelegationHandoffEnvelope {
  kind: 'delegation_handoff'
  parent: { agentId, agentRevisionId, runId, sourceCallId }
  target: { agentId, agentName, agentRevisionId, alias, delegationMode, inputFileIds, runId }
  version: 1
}
```

Delegacja to nie tylko zadanie — to pełny kontekst relacji, wersji agenta i plików wejściowych.

### 8. Streaming Markdown — Block Settling

Delta: w trakcie streamingu bloki renderują się flat. Po zakończeniu: faza `settling` → bloki grupowane w `ToolChain` (sekwencyjne) i `ToolGroup` (równoległe) z animacją collapse. Split na `committedSegments` (pełne renderowanie) i `liveTail` (lekkie, re-render tylko taila).

### 9. Setup Wizard

Delta: interaktywne CLI z TTY raw mode (password masking), spinner, structured prompts. Generuje `.env`, seed'uje DB, konfiguruje API keys.

### 10. Readiness Engine — heterogenne decyzje

Delta: zamiast hardcoded DAG walker, readiness engine podejmuje heterogenne decyzje w jednym unified loop: resume waits, requeue stale runs, start sandbox executions, queue pending jobs. Priorytetyzacja deterministyczna. Na startupie `reconcileOnStartup()` sprawdza uruchomione runy z `claim_expired` i requeue'uje.

### 11. Coordinator Pattern na froncie

Delta: chat store (~2000 linii) komponuje ~20 fokusowanych modułów (run-lifecycle, run-bootstrap, leases, event-engine, materialize, rendered-message-state). Dependencje wstrzykiwane przez config objects (getter functions), nie importowane bezpośrednio.
