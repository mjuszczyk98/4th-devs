# S01E01 — Programowanie interakcji z modelem językowym

## Natura LLM jako komponentu systemu

- Model to funkcja: `tekst → rozkład prawdopodobieństwa tokenów → wybór (z losowością) → powtórz`. Generowanie jest **autoregresywne** — każdy kolejny token zależy od wszystkich poprzednich (prompt systemowy, historia, wcześniej wygenerowane tokeny), a raz wygenerowanego tokenu nie można "cofnąć".
- **Niedeterminizm**: te same dane wejściowe mogą dać różny wynik. Programowanie generatywne = deterministyczny kod + niedeterministyczny LLM.
- **API jest bezstanowe** — każde żądanie musi zawierać komplet danych (prompt systemowy, historia, input). Pamięć rozmowy to odpowiedzialność klienta.
- **Okno kontekstowe** obejmuje wszystko naraz: prompt systemowy + historię + input + output. Język polski zużywa ~50-70% więcej tokenów niż angielski — realny wpływ na koszt i latencję.

## Sterowanie modelem = zarządzanie kontekstem

To kluczowa zasada całej lekcji. Kontrola nad tym, co trafia do modelu i jak łączą się kolejne wywołania, daje programistyczną kontrolę nad zachowaniem. Możliwości:

- kontrola treści zapytania,
- łączenie wyników wielu zapytań (routing kontekstu — np. pierwszy call klasyfikuje intencję, drugi dostosowuje prompt do kategorii),
- wieloetapowe pipeline'y, gdzie output jednego LLM staje się inputem kolejnego.

Koszt większej kontroli: wyższa latencja i więcej zapytań API.

## Structured Outputs

Dwa mechanizmy wymuszania JSON w odpowiedzi: **Structured Outputs** (JSON Schema) oraz **Function Calling**. Schema ma dwa wymiary:

1. **Struktura** — gwarantowana w trybie `strict: true` + `additionalProperties: false`.
2. **Wartości** — generowane przez model na podstawie wejścia oraz **nazw i opisów** pól. Nazwy/opisy to de facto mikroprompty — muszą być precyzyjne i zwięzłe.

Zasady projektowania schemy:
- **Kolejność pól ma znaczenie** — wcześniejsze tokeny wpływają na późniejsze. Układaj: `reasoning` → `sentiment` → `confidence`.
- **Wartości neutralne** (`null`, `"unknown"`, `"mixed"`) zmniejszają halucynacje — nie zmuszaj modelu do decyzji bez podstaw.

Typowe transformacje realizowane tą drogą: ekstrakcja, klasyfikacja, kompresja, wzbogacenie, synteza, tłumaczenie, weryfikacja, parafraza, generowanie.

## Semantyczne zdarzenia i renderowanie odpowiedzi

Interakcja z LLM to nie para pytanie-odpowiedź, lecz **seria zdarzeń**: tokeny rozumowania, tool calls, żądania potwierdzenia, błędy, obrazy, strumieniowany tekst. Implikacje architektoniczne:

- Oddziel trzy warstwy: (1) LLM generuje tekst/JSON, (2) aplikacja reaguje na zdarzenia i zarządza stanem, (3) użytkownik widzi reprezentację stanu — nie surowy output modelu.
- Używaj **semantycznych zdarzeń** (ID, typ, metadane) zamiast surowego tekstu. Rozbudowa systemu = dodanie właściwości do zdarzenia, nie migracja architektury.
- Streaming markdown→HTML wymaga obsługi niekompletnych fragmentów (code block, LaTeX, tabele). Biblioteki: Streamdown, markstream-vue.
- Kierunek: od tekstu do **dynamicznie generowanych interfejsów** (MCP Apps, Apps SDK, JSON Render). Proces może być wstrzymywany na potwierdzenie lub wynik innego agenta.

## Strategie doboru modeli

| Strategia | Opis | Kiedy |
|---|---|---|
| Jeden główny | Jeden model na wszystko | Proste aplikacje |
| Główny + alternatywny | Duży do trudnych, mały/szybki/tani do reszty | Optymalizacja kosztu i szybkości |
| Główny + specjalistyczne | Różne modele do różnych formatów/zadań | Maksymalna skuteczność per zadanie |
| Zespół małych | Dekompozycja + głosowanie | Rzadkie, eksperymentalne |

- Pytanie "jaki model jest najlepszy" jest błędne — poprawne brzmi "jaki model jest najlepszy **w tej sytuacji**". Jedyna miarodajna metoda wyboru to **testy na własnych zadaniach**.
- Unikaj vendor lock-in — abstrahuj providera od logiki aplikacji.
- Frameworki typu AI SDK obecnie nierekomendowane — narzucają ograniczenia bez adekwatnej wartości.

## Funkcjonalności natywne API providerów

- **Multimodalność** — Gemini Interactions API, OpenAI Responses API. Projektuj architekturę pod wielu providerów.
- **Natywne narzędzia** (web search, file search, code execution) — łatwe w użyciu, słabo konfigurowalne, silnie wiążą z platformą.
- **Cache** — krytyczny przy skali; obniża koszt i latencję przy powtarzających się promptach, kluczowy dla agentów.
- **Rate limit** — szybko staje się wąskim gardłem; planuj zwiększenie limitów z wyprzedzeniem. Obejście na start: OpenRouter.

## Organizacja promptów w kodzie

| Podejście | Zalety | Wady |
|---|---|---|
| Inline | Prostota | Brak kompozycji, trudna konserwacja |
| Oddzielne pliki | Kompozycja, zmienne | Monitoring finalnego promptu po kompozycji |
| Zewnętrzne (Langfuse) | Wersjonowanie i monitoring poza kodem | Dodatkowa zależność |
| **Markdown + YAML frontmatter** | Dostępne z FS w runtime, edytowalne przez agentów, dynamicznie modyfikowalne | Brak wbudowanego wersjonowania |

Markdown + frontmatter to preferowany format — łączy zalety pozostałych i daje agentom dostęp do instrukcji w czasie działania (np. dynamiczne ładowanie umiejętności). Alternatywa: deklaratywne sygnatury `input → output` (DSPy, Ax).

## Projektowanie instrukcji systemowych

Składowe dobrego promptu systemowego:

- **Tożsamość** — kieruje uwagę modelu ku właściwym skojarzeniom, nie nadaje magicznych umiejętności.
- **Strukturyzacja** — tagi XML-like wyznaczają granice sekcji.
- **Limity** i **ograniczenia** — zmniejszają ryzyko, nie dają gwarancji; warunkowe reguły ("edytuj zadanie dopiero po jego odnalezieniu").
- **Styl i format** — zwięzłość przyspiesza, ale "modele myślą generując tokeny", więc nie skracaj agresywnie.
- **Adaptacja** — instrukcje na sytuacje, w których model naturalnie sobie nie radzi (brak danych/narzędzi).
- **Kalibracja** — konkretne zasady formatowania ("używaj Mermaid dla diagramów").

**Generalizowanie generalizacji**: zamiast dodawać regułę per każdy zaobserwowany błąd, kształtuj proces myślowy modelu obejmujący całą kategorię. Przykład: zamiast "nie używaj `add_event` gdy użytkownik prosi o zadanie" → "zastanów się głośno nad wyborem narzędzia, określ pewność, doprecyzuj w razie wątpliwości".

Instrukcje systemowe **nie są zabezpieczeniem** — można je ominąć (jailbreaking). Bezpieczeństwo implementuje się na poziomie kodu i założeń projektowych.

## In-context learning (few-shot / many-shot)

Model korzysta z wzorców obecnych w kontekście. Sekcja `<examples>` z pokazanymi interakcjami > opis zasad słowami — zwłaszcza dla klasyfikacji i formatowania. Obecny trend: zgeneralizowane zasady w prompcie systemowym + **dynamiczne wczytywanie** przykładów/dokumentów przez narzędzia agenta zamiast statycznych przykładów wbitych na sztywno.

## Struktury danych dla interakcji

**Chatbot** — `conversations` (id, tytuł, status) + `messages` (id, conv_id, role, content). Wystarczy dla prostej interakcji pytanie-odpowiedź.

**System wieloagentowy** — trzy tabele:
- `sessions` — sesja powiązana z użytkownikiem i agentem koordynującym (nazwa, status).
- `agents` — instancje agentów tworzone w sesji z szablonów; każdy z zadaniem, statusem, postępem, powiązaniem z sesją i agentem zlecającym.
- `items` — atomy interakcji: wiadomości, tool calls, załączniki.

Taka struktura umożliwia dwukierunkową komunikację między agentami i zatwierdzanie akcji przez użytkownika (np. Alice deleguje do Claire, Claire zwraca wynik, Alice potwierdza wykonanie u użytkownika).

## Modele open-source

- **Formaty**: GGUF (llama.cpp, cross-platform) vs MLX (Apple Silicon).
- **Kwantyzacja**: kompresja wag (Q2→Q8). Q4/Q5 = rozsądny balans jakość/rozmiar; Q2 = szybko, słaba jakość.
- **Pamięć**: model powinien zmieścić się w VRAM/Unified Memory. Praktyczne minimum ~32 GB; największe modele wymagają ~512 GB (np. Mac Studio).
- Lokalnie przez LM Studio, zdalnie bez instalacji przez OpenRouter. Open-source bije komercyjne w wybranych zadaniach specjalistycznych i daje pełną prywatność, ale większość firm pozostaje przy komercyjnych API.

## Przykłady kodu

### 1. `01_01_interaction` — bezstanowość i budowanie historii

Wieloturnowa rozmowa przez Responses API, kontekst budowany ręcznie w kodzie.

```js
body: JSON.stringify({
  model: MODEL,
  input: [...history, toMessage("user", input)],
  reasoning: { effort: "medium" },
});
```

Czego uczy: każde zapytanie dostaje **pełną historię** jako `input: [...history, nowy_msg]` — model nie pamięta nic. W odpowiedzi `usage.output_tokens_details.reasoning_tokens` mierzy ukryty koszt chain-of-thought.

### 2. `01_01_structured` — JSON Schema i obsługa nieznanych wartości

Delta względem #1: dodanie `text.format` zamienia swobodny tekst w parsowalny JSON o gwarantowanej strukturze.

```js
const personSchema = {
  type: "json_schema",
  strict: true,
  schema: {
    type: "object",
    properties: {
      name: { type: ["string", "null"], description: "Full name. Null if not mentioned." },
      skills: { type: "array", items: { type: "string" }, description: "List of skills. Empty array if none." },
    },
    required: ["name", "skills"],
    additionalProperties: false,
  },
};
// później: JSON.parse(extractResponseText(data))
```

Czego uczy: typy unii (`["string", "null"]`) + opisy instruujące "use null if..." to wzorzec obsługi braków danych. `strict: true` + `additionalProperties: false` gwarantuje strukturę. Output to string do sparsowania, nie gotowy obiekt.

### 3. `01_01_grounding` — wieloetapowy pipeline z web searchem

Delta względem #2: kaskadowy pipeline **extract → dedupe → search → generate HTML**. Każdy etap = osobny LLM call z dedykowanym promptem i schemą (separation of concerns na poziomie zapytań).

- **Etap 1 — Extract**: per akapit wydobywa koncepty (`label`, `category`, `needsSearch`, `searchQuery`, `surfaceForms`).
- **Etap 2 — Dedupe**: scala synonimiczne koncepty w grupy (`canonical`, `ids`, `aliases`); schema wymusza `rationale`.
- **Etap 3 — Search**: natywny `tools: [{ type: "web_search_preview" }]`; schema wymusza `summary`, `keyPoints`, `sources`. Parser wyciąga metadane z `web_search_call` i `url_citation`.
- **Etap 4 — Ground**: generuje HTML z `<span class="grounded" data-grounding="...">` owijającym frazy.

Wzorce, których nie ma w poprzednich przykładach:
- **Fragmentowanie dla skupienia uwagi** — akapit jako jednostka przetwarzania, mimo że cały dokument mieściłby się w oknie. Mniejszy zakres = wyższa skuteczność + tańszy model.
- **Równoległe batchowanie** — `Promise.all()` w grupach po 5, skraca czas i omija rate limit.
- **Caching przez hashe** — każdy etap zapisuje wynik do JSON i sprawdza `sourceHash`/`conceptsHash`/`dedupeHash`. Zmiana upstream automatycznie unieważnia downstream; możliwość wznowienia po błędzie bez powtarzania wcześniejszych etapów.
- **Prompty jako moduły** — `buildExtractPrompt({ paragraph, paragraphType, index, total })` kompozycyjnie składa sekcje z parametrów.
- **Retry z exponential backoff** — `api.js` obsługuje 429/500/503.
- **Abstrakcja providera** — ta sama logika dla OpenAI i OpenRouter dzięki warunkowemu budowaniu requestu (web search: `tools: [{ type: "web_search" }]` vs suffix `:online` w nazwie modelu).
