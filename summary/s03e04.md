# S03E04 — Budowanie narzędzi na podstawie danych testowych

## Narzędzia jako zawężony interfejs API

Pełne podłączenie agenta do zewnętrznego API to antywzorzec. Agent obsługujący wybraną kategorię maili nie potrzebuje `gmail_search` — potrzebuje `gmail_search_support` z domyślnym zawężeniem. Agent generujący grafiki nie potrzebuje pełnego MCP Replicate — potrzebuje narzędzia ze ściśle określonymi ustawieniami. Pełne API = pełna powierzchnia ataku. Zawężone narzędzie = ograniczona przestrzeń na błędy agenta.

## Workflow projektowania: od API do narzędzi

Proces zaczyna się od zgrania dokumentacji API (jako Markdown, z pominięciem niechcianych obszarów) i kodu SDK do kontekstu agenta. Agent generuje zwięzłą listę dostępnych akcji. Na tej liście odpowiadasz na trzy pytania filtrujące: (1) które akcje są potrzebne, (2) które połączyć w jedno narzędzie, (3) które zablokować. Granice narzędzi są zdefiniowane przed napisaniem linijki kodu.

## Iteracyjne projektowanie schematów z LLM

LLM generuje użyteczny punkt startowy (~60–70%), ale pierwsza wersja to iluzja poprawności. Typowe patologie: brak stronicowania, pola o niejednoznacznej semantyce (`date` zamiast `receivedAt`), base64 załączników zabijające context window, brak metadanych (labels, attachments, read status).

**Metoda:** kilka ogólnych zasad/wskazówek → model rozszerza je na cały kontekst integracji. Nie trzeba iterować po każdym narzędziu osobno — zasady generują spójną implementację we wszystkich schematach.

**Gdy brakuje pomysłów:** poproś o wygenerowanie przykładowych interakcji (scenariuszy użycia). Scenariusze ujawniają problemy niewidoczne w abstrakcyjnych schematach.

Każde pole Zod ma `.describe(...)` — ten opis trafia do JSON Schema jako `description` i jest jedynym źródłem wiedzy modelu o semantyce parametru. Bez tego model zgaduje.

## Tool Envelope `{ data, hint }`

Narzędzia nie zwracają "gołych" danych. Standaryzowana koperta:

```
{ data: T, hint: ToolHint }
```

| Pole hinta | Znaczenie |
|------------|-----------|
| `status` | `success` / `empty` / `partial` / `error` |
| `reasonCode` | `OK`, `NO_RESULTS`, `AUTH_REQUIRED`, `RATE_LIMITED`, `NOT_FOUND`, `INVALID_ARGUMENT`, `POLICY_BLOCKED`, `TRANSIENT_FAILURE` |
| `summary` | Opis wyniku czytany przez LLM |
| `nextActions` | Proponowane kolejne kroki (tool + args + confidence) |
| `recovery` | `{ retryable, backoffMs, maxAttempts }` — czy i jak ponawiać |
| `diagnostics` | Scope (tool/global), httpStatus, rawMessage |

LLM dostaje nie tylko dane, ale kontekst decyzyjny. Agent nie zgaduje co zrobić po pustym wyniku — dostaje propozycję szerszego zapytania z `reasonCode: NO_RESULTS`. Zmniejsza halucynacje i bezcelowe tury.

## Next Actions — guided exploration

Każdy hint może zawierać `nextActions` — proponowane wywołania z gotowymi argumentami i scoringiem confidence:

- Po `gmail_search` z wynikami → `gmail_read` z ID pierwszej wiadomości (0.76)
- Po `gmail_search` z pustym wynikiem → `gmail_search` z szerszym zapytaniem (0.63)
- Po `gmail_read` z załącznikiem → `gmail_attachment` (0.72)
- Po `gmail_send` → `gmail_read` do weryfikacji (0.58)

Agent nie błądzi — dostaje mapę.

## Klasyfikacja błędów z recovery

Pattern matching na wiadomości + HTTP status, klasyfikacja w jednym miejscu (`buildErrorHint`):

| Kategoria | Patterny | Strategia |
|-----------|----------|-----------|
| `AUTH_REQUIRED` | `invalid_grant`, `unauthorized`, 401/403 | "uruchom `bun run auth`" |
| `RATE_LIMITED` | `rate limit`, `quota exceeded`, 429 | retry 3s backoff, max 3 |
| `TRANSIENT_FAILURE` | `timeout`, `econnreset`, 5xx | retry 1.5s backoff, max 2 |
| `NOT_FOUND` | 404 | "sprawdź ID" |
| `INVALID_ARGUMENT` | `invalid`, `required`, `must be`, `mutually exclusive` | komunikat walidacji |

Każda warstwa walidacji toola (registry → JSON parse → Zod → handler) używa tego samego mechanizmu. Agent nigdy nie dostaje surowego exception.

## 4-warstwowa walidacja; Zod jako single source of truth

Każde wywołanie toola:

1. **Registry** — czy narzędzie istnieje (scope: `global`)
2. **JSON parse** — czy argumenty to poprawny JSON (scope: `global`)
3. **Zod validate** — czy dane pasują do schematu (scope: `tool`)
4. **Handler** — czy wykonanie się powiodło (scope: `tool`)

Każda warstwa zwraca `{ data: null, hint: buildErrorHint(...) }`.

Zod schema = single source of truth. Definiowana raz (input validation), konwertowana do JSON Schema przez `z.toJSONSchema()` dla function calling. Reguły zależnościowe między polami żyją w `.refine()`, nie w handlerze — błędy trafiają do agenta jako structured hint.

## Kontrola kontekstu w narzędziach

**Detail parameter:** narzędzia search/read przyjmują `details` (default false). Compact metadata domyślnie (subject, sender, date), pełne dane (recipients, labels, attachments) tylko na żądanie. Agent nie wczytuje 50 linii metadanych by odpowiedzieć "masz 3 nieprzeczytane maile".

**Auto-detekcja typu zasobu:** agent nie decyduje czy ID to message czy thread — implementacja rozwiązuje to programistycznie (`messages.get` → jeśli 404 → `threads.get`). Jeden parametr `id`, oba przypadki.

**Modify zwraca stan po zmianie:** nie "success", ale zmienione właściwości (`isRead`, `inInbox`, `inTrash`, `isDraft`). Agent wie czy akcja przyniosła efekt bez ponownego odczytu. Wzorzec: zwracaj to co zmieniłeś, nie potwierdzenie że zmiana nastąpiła.

## Send policy — guardrails w kodzie nie w prompcie

Wysyłka maili weryfikuje adresatów względem whitelist (właściciel skrzynki + `GMAIL_SEND_WHITELELIST` z env). Każdy adresat poza whitelistą → automatyczny draft zamiast wysyłki, `policy.enforcedDraft=true`. Blokada jest w kodzie handlera, nie w instrukcjach systemowych — agent nie może obejść przez prompt injection.

**Zasada: AI proposes, code decides.** Prompt może prosić o wysyłkę, ale kod decyduje czy pozwoli.

## System prompt jako lista zasad

19 linii: kompaktowa lista 5 narzędzi z jednozdaniowymi opisami + 9 zasad routingu i obsługi błędów. Prompt referuje do mechanizmu hintów (`"Each tool response is shaped as { data, hint }"`) zamiast dublować logikę. Zasady opisują **kiedy** użyć narzędzia, nie **jak** działa.

## Mock → Real strategy pattern

Jedna zmienna env (`EVAL_MOCK_GMAIL`) przełącza między realnym API a in-memory mockiem. Mock nie jest statycznym fixturem — w pełni funkcjonalny in-memory Gmail z parserem zapytań, paginacją cursorową, mutowalnymi labelkami i generowanymi ID. Stan zmienia się między wywołaniami. Pozwala na ewaluację bez dostępu do backendu + testowanie edge casów deterministycznie.

## Generowanie i kategoryzacja danych testowych

LLM generuje syntetyczne dane testowe na podstawie kodu źródłowego narzędzi. Dwa wymiary kategoryzacji:

| | Pojedyncze interakcje | Scenariusze wieloetapowe |
|---|---|---|
| **Happy path** | tools/gmail_search, tools/gmail_send | scenarios/actions (szukaj → odpowiedz) |
| **Edge case / error** | tools/errors | scenarios/safety (external → enforced draft) |
| **Readonly** | tools/gmail_read, tools/gmail_attachment | scenarios/readonly |

Kategorie muszą być jawne od początku — bez nich ewaluacja niezarządzalna. LLM domyślnie generuje płytkie testy — trzeba pokierować: wymusić kategoryzację, dodawać edge case'y. Nie warto dopracowywać każdego testu na etapie tworzenia — użyteczność ujawnia się przy uruchamianiu.

Scenariusze testowe służą też do **oceny samych testów** — model ocenia nie tylko wyniki, ale jakość i kompletność zestawu.

## Ewaluacja z Promptfoo — architektura

**Promptfoo** = ewaluacja offline (development/publikacja). **Langfuse** = ewaluacja online (obserwacja/produkcja). Komplementarne, integracja opcjonalna.

**Dwa custom Providery:**
- **Stateless** — każde zapytanie uruchamia subprocess z pełnym agentem
- **Stateful** — serializuje historię sesji do JSON z SHA256 key (`evaluationId:conversationId:model`), kolejkuje zapytania z tego samego `conversationId` sekwencyjnie przez promise chain

**Warstwy asercji:**
1. `is-json` — poprawność JSON
2. `valid_eval_payload` — struktura wyniku (finalText, turns, reachedMaxTurns, toolCalls)
3. `required_tools` / `forbidden_tools` — weryfikacja użycia narzędzi
4. `expected_primary_tool` — kluczowe narzędzie wywołane
5. `expected_outcome` — semantyczny rezultat (`sent`, `drafted`, `enforced_draft`, `search`, `read`)
6. `turn_budget` — limit tur
7. `llm-rubric` — LLM ocenia jakość odpowiedzi

**Trzy kategorie scenario evalów:**
- **readonly** — blokuje send/modify, testuje nawigację
- **safety** — weryfikuje policy enforcement + assert że agent nie twierdzi "wysłałem" gdy policy zablokowało
- **actions** — pełne akcje multi-step (reply, forward)

Każda kategoria: osobny YAML config + dataset + dedykowane asercje.

## Scenariusze ewaluacyjne bez system promptu

Scenariusze uruchamiane **niemal bez system prompt**. Cel: sprawdzenie czy opisy narzędzi i schematy są wystarczające same w sobie. Jeśli agent bez instrukcji radzi sobie poprawnie — z wyspecjalizowanym promptem poradzi lepiej. Dolna granica jakości interfejsu. Jednocześnie walidacja `.describe()` w schematach.

## Porównanie modeli jako decyzja architektoniczna

| Model | Z ewaluacji |
|-------|-------------|
| gpt-5.2 | Perfekcyjna skuteczność, najwolniejszy, wypowiedzi zbyt techniczne |
| gpt-5-mini | Środek — zaliczył testy, rozsądny balans |
| gpt-4.1 | Najmniej kroków, profesjonalne, ale niepełne odpowiedzi i problemy stabilności |

Zaliczenie testów nie jest wystarczającym sygnałem — gpt-4.1 zaliczył wszystko, ale ujawnił problemy przewidujące awarie na produkcji. Ewaluacje muszą być na tyle wnikliwe, by wyłapać te różnice. Wyniki (HTML/JSON) przekazywane do LLM z prośbą o interpretację ujawniają niuanse niewidoczne w surowych liczbach pass/fail.

Wybór modelu: jeśli zakres agenta wykracza poza jedną integrację — mniejsze modele odpadają. Jeśli integracja prosta (pojedyncze akcje) — można rozważyć tańsze alternatywy (OpenRouter, Qwen, GLM). Ewaluacje pozwalają to sprawdzić natychmiast.

## Optymalizacja schematów pod mniejsze modele

Nawet przy docelowo najsilniejszym modelu — optymalizacja interfejsu pod mniejsze modele jest opłacalna. Lepsze opisy, ściślejsze schematy, precyzyjne hinty = lepsza skuteczność z każdym modelem.

Po ustabilizowaniu ewaluacji możliwa **automatyzacja** — agenci czytają pliki narzędzi + testy → wykonują ewaluację → interpretują wyniki → sugerują usprawnienia. Uruchamiać dopiero gdy ewaluacje są zrozumiane na tyle dobrze, by wyłapać błędy logiczne AI.

## Reasoning models w OpenAI Responses API

Modele reasoning (gpt-5, o1, o3, o4) emitują reasoning items z server-side IDs. Przy `store=false` te IDs nie mogą być replikowane w kolejnych turach → 404 na turn 2+. Non-reasoning modele odrzucają parametry reasoning — rozróżnienie konieczne.

## Kontekst z repozytorium

Pobranie repozytorium Promptfoo do zasobów projektu + zapytanie LLM o dostępne typy asercji. LLM znają Promptfoo ogólnie, ale kontekst z repozytorium zapewnia dostęp do najnowszych funkcjonalności, podnosząc jakość sugestii.

## Przykłady kodu — progresja

### Delta 1: ToolDefinition — zunifikowany interfejs rejestracji

Czego uczy: narzędzie to name + description + schema Zod + handler. Rejestr to `Map<string, ToolDefinition>`. Zod schema serializuje się do JSON Schema.

```ts
interface ToolDefinition {
  name: string;
  description: string;
  schema: z.ZodType;
  handler: (args: any) => Promise<unknown>;
}
```

### Delta 2: Zod `.refine()` — walidacja biznesowa w schema

Czego uczy: reguły zależnościowe między polami żyją w schemacie, nie w handlerze.

```ts
.refine(d => !(d.replyToMessageId && d.forwardMessageId), {
  message: 'replyToMessageId and forwardMessageId are mutually exclusive.',
})
.refine(d => d.mode !== 'new' || ((d.to ?? []).length > 0 && d.subject), {
  message: 'to and subject are required when mode="new".',
})
```

### Delta 3: Konwersja Zod → JSON Schema dla Responses API

Czego uczy: single source of truth — jedna definicja Zod serwuje validation i function calling.

```ts
const asResponseTools = (definitions: ToolDefinition[]): Tool[] =>
  definitions.map(tool => {
    const { $schema: _, ...parameters } = z.toJSONSchema(tool.schema);
    return { type: 'function', name: tool.name, description: tool.description, parameters };
  });
```

### Delta 4: Agent runtime — pętla tur z limitem

Czego uczy: agent wykonuje tury aż do braku tool calls lub limitu. Wynik = struktura gotowa do eval.

```ts
for (let turn = 1; turn <= maxTurns; turn++) {
  const result = await executeTurn(turn, params);
  if (result.done) return { finalText, turns: turn, reachedMaxTurns: false, toolCalls };
}
return { finalText: MAX_TURN_FINAL_TEXT, turns: maxTurns, reachedMaxTurns: true, toolCalls };
```

### Delta 5: Reasoning models compatibility

Czego uczy: reasoning modele wymagają `include: ['reasoning.encrypted_content']`; non-reasoning odrzucają te parametry.

```typescript
reasoning ? { reasoning: { effort: 'high' },
  include: ['reasoning.encrypted_content' as ResponseIncludable] } : {}
```
