# S03E01 — Obserwowanie i ewaluacja

## Observability — fundament przed evals

Monitorowanie wdrożone **przed** ewaluacją: logowane zdarzenia są materiałem do evals, a bez widoczności zachowania agenta nie wiadomo co testować.

Cztery zastosowania: **analiza błędów** (agent przeszukuje niewłaściwy obszar z powodu pokrywających się opisów — problem niewidoczny w kodzie), **estymacja kosztów** (tokeny z instrukcji, narzędzi, kontekstu, dynamicznych danych — przybliżone estymacje bywają niezwykle trudne), **wydajność** (niektórzy użytkownicy osiągają limity szybciej bez intencjonalnego działania), **obsługa klienta** (bezpośredni wgląd w aktywności zgłaszającego problem).

Architektura: scentralizowana logika interakcji z LLM API + narzędzia, z jedną rurą monitorującą. Brak centralizacji = fragmentaryczne dane.

### Hierarchia obserwacji

| Typ | Zakres | Kluczowe dane |
|-----|--------|---------------|
| Session | Wątek czatu / zadanie agenta | Id sesji, użytkownik |
| Trace | Pojedyncza interakcja użytkownika | Input, output, metadane |
| Span | Czas trwania akcji | Durations, fazy przetwarzania |
| Generation | Zapytanie do LLM | Model, prompt, tools, usage, output |
| Tool | Wywołanie narzędzia | Input/output narzędzia |
| Event | Zdarzenie aplikacji niezwiązane z LLM | Niezależne od LLM |

Span i Event to opcjonalna inicjatywa. Po minimalnej integracji szybko ujawniają się brakujące i nadmierne logi.

Nie logujemy "co się stało", ale **w jakim kontekście**. Dwa typy danych:
- **Kontekst** — specyficzny dla projektu: licencje, uprawnienia, preferencje, rodzaj zapytania (CRON vs user)
- **Metadata** — o połączeniu: interfejs, wersja aplikacji, środowisko

## Tracing — implementacja

**AsyncLocalStorage** — propaguje kontekst (agentName, turnNumber, toolIndex) przez async granice bez jawnych parametrów. Pozwala generować hierarchiczne span names (`alice/generation#2`) z poziomu funkcji tracingowych, nie logiki biznesowej.

```ts
export const withAgentContext = (name, id, fn) =>
  storage.run({ agentName: name, agentId: id, turnNumber: 0, toolIndex: 0 }, fn);

export const formatToolName = (tool) =>
  `${storage.getStore().agentName}/${tool}#${nextToolIndex()}`;
```

**Dekorator `withGenerationTracing`** — opakowuje `Adapter.complete()` w generation span. Agent i adapter nie wiedzą o tracingu, tracing nakładany transparentnie przez fabrykę adapterów. Loguje messages + tools jako input, text + tool_calls jako output, usage per generation.

```ts
export const withGenerationTracing = (adapter: Adapter): Adapter => ({
  complete: async (params) => {
    if (!isTracingActive()) return adapter.complete(params);
    const gen = startGeneration({ model, input: buildGenerationInput(params) });
    const result = await adapter.complete(params);
    result.ok ? gen.end({ output, usage }) : gen.error({ code, message });
    return result;
  },
});
```

**Graceful degradation** — gdy tracing wyłączony (brak credentials), wszystkie wrappery wywołują oryginalną logikę bez overheadu. Wyjątek: `withAgent` **zawsze** propaguje kontekst przez `AsyncLocalStorage` — bo kontekst agenta potrzebny jest samej logice biznesowej. `flush()` wywoływany w `finally` każdego żądania HTTP.

Wymaga zaplanowania na etapie architektury, nie retrofitu.

## Wersjonowanie promptów

Prompty w kodzie = automatyczna wersjonowanie przez Git, ale niewystarczające — potrzebna korelacja treści ze statystykami, historią uruchomień i evals.

**Jednostronna synchronizacja** (aplikacja → platforma): hash SHA256 treści promptu determinuje czy pushować nową wersję. Stan lokalny w `.langfuse-prompt-state.json`. Wynik: każda generacja ma referencję PromptRef (name + version).

Tracimy łatwe przełączanie wersji z UI, zachowujemy kompletność statystyk i spójność z kodem.

```ts
const contentHash = createHash('sha256').update(source.content).digest('hex');
if (cached?.contentHash === contentHash) { return; }
const version = await client.prompts.create({ name, prompt: content, labels: ['production'] });
state.prompts[name] = { contentHash, version };
```

## Debugowanie agentów ≠ debugowanie kodu

Zmiana instrukcji lub opisów narzędzi naprawiająca jeden przypadek może psuć inne. Brak determinizmu = brak gwarancji braku regresji. Poprawa nie jest punktem końcowym — trzeba weryfikować wpływ na resztę systemu.

Platformy (Langfuse, Confident AI) oferują **Playground** do re-execucji zapytań z modyfikowanymi parametrami. Wymaga przesyłania instrukcji systemowej, historii i listy narzędzi — nie tylko promptu użytkownika.

Technika: poproszenie modelu o **uzasadnienie** wyboru narzędzi często ujawnia przyczyny problemu.

## Ewaluacja — struktura

Eval = ustrukturyzowany test oceniający zachowanie modelu/agenta pod kątem metryk. **Nie gwarantuje** poprawnego działania — aplikacja nadal może zachowywać się niezgodnie z oczekiwaniami. 100% pass rate nie jest celem.

Częściowa zgodność jest normą — "Mam na imię Adam" vs "Jestem Adam" = ten sam sens, różne cechy wypowiedzi.

Trzy poziomy oceny: **programistyczna** (regex, JSON schema), **LLM-as-judge** (model ocenia model), **człowiek** (manualna weryfikacja). Czasem to nie system, lecz **testy wymagają optymalizacji** — wynik oceniony gorzej może w praktyce być lepszy.

### Składniki evala

- **Zadanie** — dane wejściowe + oczekiwane dane wyjściowe
- **Dataset** — przykłady syntetyczne i/lub z produkcji, iteracyjnie rozwijany
- **Evaluator** — funkcja `(input, output, expectedOutput) → score[]` (0–1), deterministyczna lub LLM-based

### Offline vs Online

**Offline eval** — przed publikacją, w CI/CD lub dev. Weryfikuje skuteczność promptów, agentów, RAG na zdefiniowanych datasetach.

**Online eval** — w trakcie działania aplikacji. Interakcje oceniane automatycznie pod kątem naruszeń, których nie wyłapały wbudowane mechanizmy.

## Projektowanie datasetów testowych

| Zasada | Opis |
|--------|------|
| **Pokrycie** | Pozytywne i negatywne scenariusze |
| **Różnorodność** | Szeroki obraz warunków działania, edge case'y |
| **Balans** | Równomierne rozłożenie — skrzywienie zaburza wyniki |

Odpowiedzi narzędzi szybko stają się większością promptu agenta w interakcji — uwzględniać je w testach (Braintrust zwraca na to uwagę).

LLM pomocny przy generowaniu scenariuszy i weryfikacji poprawności, ale człowiek weryfikuje rezultaty. Datasety mają charakter iteracyjny.

## Kryteria oceny

**Deterministyczne:** `contains`, `is-json`, `equals`, regex, asercje programistyczne (JS/Python). Darmowe.

**Model-based:** `llm-rubric` (ocena wg kryteriów w języku naturalnym), `conversation-relevance`, `context-recall`. Kosztują tokeny.

| Obszar | Co weryfikuje | Typowa metoda |
|--------|--------------|---------------|
| Skuteczność promptów | Jakość instrukcji, umiejętności agenta | LLM-as-judge |
| Wybór narzędzi | Czy agent sięga po właściwe narzędzia | Deterministyczna |
| Obsługa narzędzi | Poprawność użycia, liczba powtórzeń/błędów | Deterministyczna + LLM |
| Satysfakcja użytkownika | Aktywny feedback (łapki) | Człowiek |
| Naruszenia | Dane wejściowe i odpowiedzi modelu | Deterministyczna + LLM |

## Koszty i strategia podziału pracy

Koszty ewaluacji = koszty API (tokeny na wykonanie zadań + tokeny na LLM-as-judge). Deterministyczne asercje darmowe i często wystarczające.

Strategia: dane strukturalne/numeryczne → **kod** (darmowe, deterministyczne), język naturalny/subiektywne oceny → **LLM** (zoptymalizowane: batch, deduplikacja identycznych inputów, minifikacja outputu).

Promptfoo: input tani, output drogi — wysyłaj dużo danych, wymagaj minimalnej odpowiedzi.

Eksperyment z N przypadkami = N wywołań agenta. `confirmExperiment()` wymaga interaktywnego "yes" przed API calls — dobra praktyka. Evals mogą być tymczasowe (test porównawczy modeli przed decyzją o downgrade).

## Guardrails — trzeci filar

Pełny obraz monitorowania systemów agentowych: **Observability** (monitoring) + **Evals** (ewaluacja) + **Guardrails** (moderacja/filtrowanie treści, blokowanie niepożądanych zapytań).

Guardrails działają jako mechanizm ochronny na wejściu i wyjściu, reagując na naruszenia w czasie rzeczywistym. Niezależna warstwa od ewaluacji, ale korzystająca z tych samych sygnałów.

## Anonimizacja danych

Obserwowanie zachowań agentów = przetwarzanie danych użytkowników. Nawet przy self-hostingu — wymaga anonimizacji: nazwy własne, adresy, dane kontaktowe, dane osobowe. Dane logowane to istotne źródło do evals — nie można ich pominąć, ale muszą być zgodne z zasadami ochrony danych.

## Decyzja strategiczna — czy evals?

Evals nie gwarantują poprawności, kosztują czas i tokeny. Skrajne opinie: "evals are the real moat" (YCombinator) vs "no evals" (twórca Claude Code). Analogia do testów E2E — część zespołów uważa za krytyczne, inni opcjonalne.

Decyzja zależy od **tempa zmian** i **priorytetów**. Na wczesnych etapach rozwiązania zmieniają się zbyt szybko by uzasadnić inwestycję. Przy produkcie gdzie stabilność lub efektywność kosztowa są priorytetem — trudno bez.

Bezwzględnie warto uwzględnić observability i evals na etapie planowania architektury, nawet jeśli same evals wprowadzisz później.

## Platformy

| Platforma | Siła |
|-----------|------|
| **Langfuse** | Rekomendowany, open source, pełny stack (observability + evals + prompt management) |
| **Promptfoo** | Rekomendowany, silny w konfigurowalnych kryteriach oceny |
| **Confident AI** | Wbudowany Playground do debugowania |
| **Braintrust** | Perspektywa: anatomia promptu w kontekście eval |
| **Grafana** | Opcja dla projektów już korzystających ze stacku |

Integracja z platformami może być zrealizowana przez AI, ale detale zależą od nas: kontekst specyficzny dla projektu i struktura obserwacji. Zawsze rozważyć, czy wystarczą proste wewnętrzne skrypty.

AI odnajdzie się w analizie wyników — w Langfuse: eksport JSON interakcji/eksperymentu i omówienie z modelem.

## Przykłady kodu

### `03_01_observability` — Agent z Langfuse tracing

Minimalny agent HTTP (Hono) z dwoma narzędziami (`get_current_time`, `sum_numbers`), pełną instrumentacją Langfuse i wersjonowaniem promptów.

**Delta względem S01E05 (`01_05_agent`):** S01E05 pokazywało Langfuse jako subscriber eventów w architekturze event-driven. Tutaj Langfuse zintegrowany bezpośrednio przez OpenTelemetry SDK + dekorator adaptera.

- **OTel + Langfuse** — `NodeSDK` z `LangfuseSpanProcessor`. Tracing inicjalizowany na starcie, grzecznie degraduje bez poświadczeń
- **ReAct agent loop** — pętla do `MAX_TURNS=8`, wyjście gdy model nie produkuje tool calls. Usage akumulowany przez wszystkie tury. Hierarchia spanów: trace → agent → generation#N / tool#N
- **Prompt sync** — content-hash w `.langfuse-prompt-state.json`. Push tylko gdy hash się zmieni
- **Result type** — `Result<T, E>` z discriminated union. `if (result.ok)` daje type narrowing. Mapowanie błędów OpenAI SDK na typed `CompletionError` z kodami

### `03_01_evals` — Ewaluacja agenta

Tożsama struktura `src/` + katalog `experiments/` z frameworkiem ewaluacyjnym. Migracja z Chat Completions na Responses API.

**Delta względem `03_01_observability`:** dodaje eval framework — dataset management, experiment runner, deterministyczne scorery.

- **Dataset CRUD** — `ensureDataset()` (get-or-create) + `syncDatasetItems()` (upsert by ID). Idempotentne. Dane z plików `.synthetic.json`
- **Tool-use eval** — dekompozycja na 4 niezależne wymiary: decision (czy użyć), required (czy wszystkie), forbidden (czy żadne niechciane), call count (czy w zakresie). Każdy 0 lub 1, średnia = overall. Całkowicie deterministyczna, zero LLM calls

```ts
const decision = expected.shouldUseTools ? (count > 0 ? 1 : 0) : (count === 0 ? 1 : 0);
const required  = expected.requiredTools.every(n => unique.has(n)) ? 1 : 0;
const forbidden = expected.forbiddenTools.every(n => !unique.has(n)) ? 1 : 0;
const callCount = inRange(count, expected) ? 1 : 0;
return { overall: (decision + required + forbidden + callCount) / 4 };
```

- **Response correctness eval** — dispatch po typie oczekiwanego wyniku: `exact_number` (ekstrakcja regex z tolerancją 0.01), `contains_iso_timestamp` (regex), `relevance` (keywords). Deterministyczna

```ts
if (expected.type === 'exact_number')  return scoreExactNumber(response, expected.value);
if (expected.type === 'contains_iso_timestamp') return scoreTimestamp(response);
if (expected.type === 'relevance')     return scoreRelevance(response, expected.topic);
```

- **Safety gate** — `confirmExperiment()` wymaga "yes" przed API calls
- **Run-level evaluators** — `createAvgScoreEvaluator` agreguje per-item scores w średnią dla całego runu. Langfuse `dataset.runExperiment()` wspiera per-case i run-level scoring
