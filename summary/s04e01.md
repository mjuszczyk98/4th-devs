# S04E01 — Wdrożenia rozwiązań AI

## Punkt startowy

Brak ustalonych najlepszych praktyk — tempo zmian to miesiące, nie lata. Każde wdrożenie = eksperyment z ograniczoną zdolnością prognozowania. Strategia: **zbuduj rozwiązanie, z którego sam korzystasz** — bezpośrednie doświadczenie ograniczeń buduje intuicję skalowalną na innych.

MVP celowo minimalny (`04_01_garden`: brak UI, kompresji kontekstu, mało narzędzi) — punkt odniesienia do iteracji, nie kompletny system. Decyzja o **niestosowaniu AI** tam, gdzie korzyści marginalne, jest równie ważna jak decyzja o zastosowaniu. Ramka: „nie możemy zrobić wszystkiego, ale możemy zrobić cokolwiek".

## Synchroniczna vs asynchroniczna współpraca

Dwie kategorie problemów projektowych, nie tylko tryby pracy.

| Wymiar | Synchroniczna | Asynchroniczna |
|---|---|---|
| Interfejs | centralny — edytor, czat, CLI | pominięty lub zminimalizowany |
| Personalizacja | bieżące budowanie kontekstu | predefiniowane procesy z góry |
| Feedback | nadzór człowieka na każdej akcji | raportowanie, minimalne zaangażowanie |
| Uprawnienia | szersze (człowiek nadzoruje) | wąskie, sandboxowane |
| Zmienność | elastyczna, iteracyjna | ustalona, osadzona w codzienności |

System hybrydowy — cztery elementy: **edytor** (bezpośrednia edycja markdown), **interfejs** (proste polecenia), **workflow** (procesy w plikach, rezultaty w ustalonych strukturach), **agenci** (dedykowane narzędzia, brak bezpośredniej współpracy — tylko wymiana informacji). Klucz: agent nie powinien przeszkadzać.

## Mapowanie decyzji projektowych

Trzy pytania naraz: **co robimy**, **czego nie robimy**, **jak to robimy**.

| Obszar | Decyzja | Uzasadnienie |
|---|---|---|
| Użytkownik | programista → złożone narzędzia + sandbox | zmniejszenie limitów agenta |
| Treść | ręczna → AI wzbogaca, nie generuje | pełen dostęp, ale według zasad |
| Format | markdown → HTML generowany przez kod | HTML niepraktyczny do edycji |
| Integracje | skills + code mode | CLI, MCP, narzędzia natywne bez ograniczeń |
| Publikacja | GitHub Pages, statyczny HTML | public-by-design = brak problemu prywatności |
| Dostępność | API + zdalny serwer + Mutagen | dostęp z dowolnego miejsca |

Balans kod vs AI przesuwa się z 90/10 w wybranych scenariuszach, ale większość architektury to nadal zwykły kod.

## Prototypowanie z AI

AI obniża koszt prototypowania do poziomu **równoległego** testowania wielu wariantów — zamiast sekwencyjnego sprawdzania po jednym. Każdy test = odpowiedź na konkretne pytanie: „czy model sam wie co dodać, czy potrzebuje wskazówki w frontmatterze?", „czy agent sam decyduje o umiejscowieniu notatki?". Założenia początkowe będą błędne — liczy się szybkość korekty.

## Architektura Cyfrowego Ogrodu

Trzy subsystemy: **Vault** (Markdown jako single source of truth), **Agent** (OpenAI Responses API + Daytona Sandbox), **Grove** (custom static site generator). Agent traktuje vault jednocześnie jako pamięć i przestrzeń roboczą.

Przepływ: lokalny vault ↔ sandbox vault (dwukierunkowa synchronizacja), agent operuje na sandboxie, sync z powrotem + git push = deploy (GitHub Actions/Pages).

Celowe ograniczenia: dane publiczne by design, treść tworzy człowiek a AI wzbogaca, konwersja do HTML wyłącznie przez kod.

## Agent Loop

```
userMessage → loadTemplate → resolveSkillContext → [loop max N turns]
  → completion(model, instructions, tools, previousResponseId)
  → if function_call → executeTool (try/catch) → feed result back
  → if text only → return
```

- **`previous_response_id`** — server-side kontekst bez re-sendu pełnej historii (redukcja tokenów)
- **Reasoning effort** z fallbackiem (`xhigh` → `high` gdy model nie wspiera)
- Błąd narzędzia nie przerywa loopa — `function_call_output` z errorem wraca do modelu
- Paralelne wykonanie tool calli w każdej turze

## Tool Registry

`Map<string, Tool>` — `findTool(name)` resolwuje handler, `definitions(names?)` filtruje dla modelu. Nowe narzędzie = obiekt z `definition` (OpenAI schema) + `handler`. Rejestr ogranicza narzędzia per skill.

| Narzędzie | Rola |
|---|---|
| `terminal` | shell w sandboxie, CWD = vault root |
| `code_mode` | TypeScript z helperami `codemode.vault.*` + `runtime.exec` |
| `git_push` | sync → commit → push w jednym wywołaniu |

## Code Mode

Agent przekazuje inline script LUB script_path do pliku w skills. Runner wstrzykuje `codemode` helper (vault read/write/list/search/move), `input` z env, przechwytuje wynik przez marker `__CODE_MODE_RESULT__=` / `__CODE_MODE_ERROR__=`. Output parsowany od końca — ostatni marker wygrywa. Wzorzec podobny do Code Mode Anthropic.

## System Skilli

Katalogi `vault/system/skills/<name>/` z `SKILL.md` (YAML frontmatter + instrukcje). Frontmatter:

- `allowed-tools` — ogranicza zestaw narzędzi per skill
- `runtime-scripts` — deterministyczne skrypty (code_as_skill: transformacje delegowane do kodu, nie do LLM step-by-step)
- `user-invocable`, `disable-model-invocation`, `argument-hint` — kontrola aktywacji

Rezolucja: komunikat parsowany jako `/skill-name args` → do wiadomości doklejany `<metadata>` z konfiguracją + lista narzędzi ograniczana do `allowed-tools`. Auto-discovery: system skanuje `scripts/` w folderze skilla (walidacja ścieżki, rozszerzenie `.ts/.js/.mjs`).

## Workflows

Pliki `.md` w `vault/system/workflows/` z frontmatter `{name, description}`. Ładowane do instrukcji agenta — „MUSISZ podążać za workflow gdy request pasuje". Deklaratywny opis kroków, agent wykonuje w swoim loopie. Zmiana procesu = zmiana pliku.

## Template Agenta

`vault/system/<name>.agent.md` — frontmatter `{model, tools}` + treść jako instrukcje systemowe. Dynamicznie ładowany, wzbogacany o workflows, skills, interpolację `{{date}}`. Wiele agentów z różnymi konfiguracjami w jednym systemie.

## Sandbox (Daytona)

LazySandbox tworzy instancję dopiero na `get()`. Background loop (700ms) śledzi zmiany i syncuje delta (sygnatura `size:mtimeMs`). `syncVaultBackNow()` pobiera zmienione pliki z wykluczeniem `system/`. `syncInFlight` flag zapobiega race conditions. Cykl: create → init → [sync loop] → sync back → delete.

## Grove — generator strony

`collectMarkdown` pomija `system/`, `parse` z gray-matter + marked, wiki-links `[[x]]` resolwowane do hrefów. Dodatkowo: comparison template z interaktywnym UI, draft mode (`draft: true`), pełne SEO (OG tags, Twitter cards, canonical).

## Ograniczenia AI w praktyce

Agent: nie przetwarza długich dokumentów, nie widzi obrazów osadzonych w tekście, nie czyta PDF, nie ma dostępu do niektórych stron, nie korzysta z funkcji niedostępnych w API, generuje koszty i czas. Każdy problem rozwiązywalny z odpowiednią architekturą, ale wymaga świadomego wyboru obszaru.

## Implementacje — delta kodu

### Delta 1: Server-side kontekst (response chaining)

```typescript
const response = await completion({ ..., previousResponseId });
previousResponseId = response.id;
```

Referencja do poprzedniej odpowiedzi zamiast re-sendu pełnej historii. Zmienia koszt z liniowego do stałego per tura.

### Delta 2: Skill invocation jako routing

```typescript
const match = trimmed.match(/^\/([a-z0-9][a-z0-9-]*)(?:\s+([\s\S]*))?$/i);
```

Skill scope'uje narzędzia — agent fizycznie nie może użyć narzędzia spoza `allowed-tools`.

### Delta 3: Frontmatter jako runtime config

```yaml
---
model: gpt-5.2
tools: [terminal, code_mode, git_push, web_search]
---
```

Markdown + YAML oddziela definicję zachowania od implementacji. Zmiana modelu/narzędzi = edycja pliku.

### Delta 4: Snapshot-based incremental sync

```typescript
const signature = `${stat.size}:${stat.mtimeMs}`;
if (previousSnapshot.get(rel) === signature) continue;
```

Sygnatura pliku = rozmiar + timestamp. Prostsze niż hashowanie zawartości, wystarczające do detekcji zmian.

### Delta 5: Marker-based IPC

```typescript
console.log(`__CODE_MODE_RESULT__=${JSON.stringify(result)}`);
```

Protokół tekstowy sandbox→host przez stdout. Parsowanie od końca — ostatni marker wygrywa.
