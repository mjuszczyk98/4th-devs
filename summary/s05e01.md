# S05E01 — Architektura

## Prymitywy architektoniczne

Projektowanie na prymitywach: zamiast budować struktury pod konkretną funkcjonalność, buduje się je pod **możliwość przekształcenia**. "Prosty czatbot" szybko staje się agentem, potem systemem wieloagentowym — prymitywy pozwalają rozbudowę bez przebudowy.

| Funkcjonalność (sztywna) | Prymityw (elastyczny) |
|--------------------------|-----------------------|
| Wiadomości user ↔ asystent | Zdarzenia między aktorami (user, agent, system) |
| Osobne tabele na pliki/obrazy/dokumenty | Artefakty — metadane dowolnej treści z polem `kind` |
| Tabela `messages` z polem `text` | Tabela `items` z typami (message, tool_call, invocation, result) |

Zdarzenia mogą obejmować nie tylko tekst — reasoning modelu, kompresję kontekstu, prośbę o potwierdzenie subagenta. Aktor może być użytkownikiem, agentem, systemem.

## Decyzje architektoniczne — 5 obszarów

| Obszar | Kluczowa decyzja |
|---|---|
| **Gateway** | Scentralizowana komunikacja z AI. Swobodne przełączanie modeli/providerów przez AI SDK, LiteLLM lub własny adapter. |
| **API** | Klient nie ma bezpośredniego dostępu do modelu. Wyspecjalizowane endpointy (`/product/review`) zamiast generycznych (`/api/chat`). |
| **System plików** | Scope uprawnień i sandboxing dla agentów. Ryzyko nieprzewidzianych akcji (usunięcie katalogu) wymaga innych zabezpieczeń niż klasyczna aplikacja. |
| **Baza danych** | Dodatkowe struktury: interakcje, zaplanowane zadania, definicje agentów i narzędzi. |
| **Zależności** | Biblioteki: ewaluacja, observability, transformacja dokumentów, SSE, RAG. |

### Pewniki (zawsze warto zaimplementować)

- Centralizacja interakcji z AI — rozproszone wywołania blokują zarządzanie ustawieniami i przełączanie modeli
- Obsługa wielu providerów — lepsze modele pojawiają się u różnych dostawców
- Strumieniowanie (SSE) — informowanie o postępach, mniejszy czas reakcji
- Multimodalność — struktury DB gotowe na obraz/audio nawet jeśli teraz tylko tekst
- Logika agentów od dnia zero — `items` zamiast `messages`, by przechowywać akcje między wiadomościami
- Zadania długoterminowe — użytkownik zamyka kartę, czas przekracza limit połączenia

## Elastyczność architektury w dobie AI

System powinien być zaprojektowany tak, by **rozwój modeli wzmacniał jego możliwości**:

- Nie buduj na obszarach, gdzie LLM szybko dogoni funkcjonalność — produkt stanie się niepotrzebny przed zwrotem z inwestycji
- Dynamika iteracji rzędu tygodni, nie kwartałów — architektura musi być bardziej podatna na modyfikacje
- Ostrożność przy frameworkach AI — fundamenty wciąż się zmieniają, oparcie na nich aplikacji = ryzyko blokady

## Integracja wielu providerów

Problem: więcej niż jeden provider **w tej samej logice** (np. agenci z różnymi modelami). Wymaga warstwy tłumaczeń:

- **Struktura API:** system message — OpenAI (element listy) / Anthropic (oddzielne pole) / Gemini (`system_instruction`)
- **Ustawienia:** reasoning — OpenAI (`reasoning_effort`) / Anthropic+Gemini (`budget_tokens`)
- **Thought Signatures:** Gemini 3+ wymaga przy tool calls w bieżącej turze; Anthropic ma analogiczny mechanizm; brak obsługi = utrata reasoning tokens
- **Ograniczenia:** Anthropic nie przyjmuje wiadomości asystenta jako pierwszej w historii

| Podejście | Kiedy | Wady |
|-----------|-------|------|
| **OpenRouter** | Tylko LLM, bez zaawansowanych funkcji API | Brak wsparcia nowszych funkcji, sporadyczne błędy mapowania |
| **Biblioteki** (AI SDK, LiteLLM) | Provider dobrze wspierany, szybki start | Blokada najnowszych API, bugi o niskim priorytecie naprawy |
| **Własna logika + SDK** | Większość projektów produkcyjnych | Pełen ciężar utrzymania — z AI do kodowania koszt akceptowalny |

Rekomendacja: własna logika z domyślnym formatem **Responses API**. Kluczowe decyzje: (1) domyślny kształt zapytania/odpowiedzi/streamingu, (2) mechanizm wyboru providera + mapowanie ustawień, (3) zakres obsługiwanych ustawień.

### Warstwa tłumaczeń w praktyce

Pojedynczy klient OpenAI SDK z konfigurowalnym `baseURL`/`apiKey`/`headers` — przełączanie między OpenAI a OpenRouter bez zmiany kodu. `resolveModelForProvider()` mapuje nazwę modelu na format providera. Automatyczne wykrywanie reasoningowych modeli (`isReasoningModel` — dopasowanie `/^o\d/i`) i warunkowe dodawanie `reasoning: { effort, summary: 'auto' }`. Retry z exponential backoff (`MAX_RETRIES = 2`) na 429/5xx i błędy sieciowe.

Web search realizowany różnie: OpenAI — `{ type: 'web_search_preview' }` w narzędziach API; OpenRouter — suffix `:online` na nazwie modelu.

## System wieloagentowy — 4 połączone wzorce

System `05_01_agent_graph` łączy cztery wzorce:

**Orchestrator** — agent z `delegate_task` i `create_actor`. Nie wykonuje pracy. Tworzy aktorów dynamicznie (nie predefiniowanych). Reguła: nie blokuj żeby czekać — scheduler wznowi automatycznie.

**Blackboard** — współdzielony stan (sesja, zdarzenia, zadania, relacje, artefakty). Agenci nie komunikują się bezpośrednio — komunikują się przez stan za pomocą narzędzi (`write_artifact`, `read_artifact`).

**DAG** — relacje `depends_on` tworzą graf zadań. Scheduler deterministycznie rozwiązuje zależności: promuje gotowe, wstrzymuje czekające, wznawia po zakończeniu dzieci.

**Events** — każda zmiana stanu → event → SSE. Używane do observability, ewaluacji, guardrails, UI. Bufor max 500 eventów, replay na nowe połączenie.

### Agent jako dane, nie klasa

Agent = obiekt konfiguracyjny `AgentDefinition` (instrukcje + narzędzia + maxSteps). Zachowanie wyłania się z konfiguracji + uniwersalnego runtime'u. Różnicowanie agentów wyłącznie zestawem narzędzi (researcher nie ma `read_artifact`, writer nie ma `web_search`). Rejestr agentów mapuje nazwę → definicję; `create_actor` najpierw szuka w rejestrze, potem wymaga jawnej konfiguracji ad-hoc. Bootstrap: tylko `['orchestrator']` tworzony przy starcie sesji.

### Delta vs S03E02 (heartbeat)

W `03_02_events` plan zadań był statyczny — określony z góry. Tutaj plan jest **kształtowany dynamicznie** przez agenta zarządzającego. Elastyczność rośnie, przewidywalność maleje.

## Scheduler — cykl życia zadań

Stany: `todo → in_progress → done | waiting | blocked`

```
while (ready_tasks exist):
  for each ready_task:
    set in_progress → find assigned actor → runActorTask()
    → completed? set done, unblock parents
    → waiting?    keep waiting (children still active)
    → blocked?    set blocked with recovery info
```

- **todo**: brak zależności lub zależności spełnione → `findReadyTasks` promuje do `in_progress`, sortuje po `priority`
- **waiting**: actor kończy pracę, ale scheduler wykrywa aktywne subzadania → czeka, potem promuje z powrotem do `todo`
- **blocked**: nieusuwalna przeszkoda (runtime error, actor bez narzędzi) lub limit prób
- **Recovery**: transient errors (rate limit, 5xx) → automatyczny retry z exponential backoff (max 3 próby), zapisywane jako `TaskRecoveryState`

`recoverStaleTasks()` na początku `processSession()` — resetuje `in_progress` do `todo`. Odpowiada za awarię procesu w trakcie zadania.

Po zakończeniu/zablokowaniu zadania `updateActorStatus()` sprawdza czy aktor ma aktywne zadania → `active` lub `idle`. Zapobiega przypisywaniu zadań do nieaktywnych aktorów.

Maksymalnie 20 rund scheduler'a, po czym pętla się kończy.

## Aktor i jego kontekst

Actor loop: `maxSteps` iteracji, każda = processTaskMemory → buildTaskRunInput → LLM call → tool execution. Narzędzia terminalne (`complete_task`, `block_task`) przerywają loop. Pozostałe zwracają `continue` — LLM dostaje wynik i decyduje o kolejnym kroku. Invocation + result zapisywane jako Items z sekwencją.

### Kompozycja promptu aktora (4 warstwy)

1. **Prompt prefix** — stały: rola, zasady, rejestr agentów, instrukcje użycia narzędzi
2. **Task memory** — skompresowane obserwacje z poprzednich kroków (sealed history)
3. **Task snapshot** — aktualny stan sesji: lista aktorów, wszystkie zadania, zależności, artefakty
4. **Raw items** — nieobserwowane zdarzenia (wiadomości, decyzje, invocations, results) od `lastObservedSeq`

Actor nie zarządza swoim kontekstem — scheduler wstrzykuje pełny snapshot sesji.

### Rejestr narzędzi

| Narzędzie | Rola | Wynik |
|---|---|---|
| `create_actor` | Utworzenie/aktualizacja aktora z rejestru lub ad-hoc | `continue` |
| `delegate_task` | Sub-zadanie z przypisaniem aktora + opcjonalne `dependsOnTaskIds` | `continue` |
| `complete_task` | Zamknięcie zadania | `completed` (terminal) |
| `block_task` | Zablokowanie zadania z powodu | `blocked` (terminal) |
| `write_artifact` | Zapis artefaktu do FS + metadane w store | `continue` |
| `read_artifact` | Odczyt artefaktu po ID lub ścieżce | `continue` |

## Pamięć observational per-task

Dwufazowy cykl (adaptacja Mastra Observational Memory), per-task-thread:

**Observer** — uruchamiany gdy raw items > `observationThresholdTokens` (800). `splitByTailBudget` dzieli na *head* (do kompresji) i *tail* (~30% zachowane raw, chroni ostatnie interakcje). Head → LLM → ustrukturyzowane obserwacje XML z priorytetami 🔴🟡🟢. Invocation+result nigdy nie są rozdzielone.

**Reflector** — uruchamiany gdy skumulowane obserwacje > `reflectionThresholdTokens` (1500). Kompresuje do `reflectionTargetTokens` (800) wielopoziomowo: level 0 (gentle) → 1 (aggressive) → 2 (heavy).

Stan trzymany na `Task.memory: MemoryState` — observations, lastObservedSeq, observationTokens, generation (ile razy reflector był uruchamiany).

## Model domenowy

| Encja | Kluczowe pola |
|---|---|
| **Session** | id, title, status (active/paused/done), usage |
| **Actor** | id, session_id, type (user/agent), name, status (active/idle), capabilities |
| **Task** | id, session_id, parent_task_id, owner_actor_id, title, status, priority, memory, recovery |
| **Item** | id, session_id, task_id, actor_id, type (message/decision/invocation/result), content, sequence |
| **Artifact** | id, session_id, task_id, kind (file/plan/diff/image), path, version |
| **Relation** | id, from_kind, from_id, relation_type (depends_on/assigned_to/produces/uses), to_kind, to_id |

Generyczny `Relation` z `from_kind/from_id/relation_type/to_kind/to_id` — jeden typ relacji obsługuje wszystkie powiązania między encjami. Polimorficzny join pozwala zapytania typu "znajdź wszystkie depends_on dla taska X" bez dedykowanych tabel per związek.

Graf zadań nad FileStore (JSON) — relacyjna warstwa nad systemem plików. Task DAG nie wymaga dedykowanej bazy grafowej.

## Dodatkowe wzorce implementacyjne

**Artifact placeholders** — agenci referencjonują artefakty przez `{{file:path}}` zamiast kopiować treść. Oszczędność tokenów + unikanie niespójności.

**Trójpoziomowa obsługa błędów** — API (retry z exponential backoff) → Actor (`RecoverableActorError`) → Task (recovery state z `explicit_block` / `llm_transient` / `runtime_error`).

**Prompt caching per task-actor** — `sha256(session_id:task_id:actor_id).slice(0,48)` jako `prompt_cache_key` w Responses API. Maksymalne wykorzystanie cache'u.

**Write queue** — sekwencyjne zapisy bez mutexów: `this.writeQueue = this.writeQueue.then(async () => ...)`. Łańcuch Promisów gwarantuje kolejność.

**Akumulacja tokenów** — `TokenUsage` akumulowany na sesji z dwóch źródeł: actor loop (kumulatywnie za wszystkie kroki) + operacje pamięci (observer + reflector). Cache hit rate: `cachedTokens / inputTokens * 100%`.

**Brak frameworka agentowego** — ~1500 linii TS, zero LangChain/AutoGen. Separation of concerns: domain (typy) → store (persystencja) → runtime (kontekst) → tools → scheduler (orkestracja) → memory (kompresja) → agents (konfiguracja).

### 1. Relation — generyczny model relacji

Delta: jeden typ relacji zastępuje specjalistyczne tabele joinujące.

```ts
export type RelationType = 'depends_on' | 'assigned_to' | 'produces' | 'uses'
export type EntityKind = 'session' | 'actor' | 'task' | 'item' | 'artifact'

export interface Relation {
  id: string; session_id: string
  from_kind: EntityKind; from_id: string
  relation_type: RelationType
  to_kind: EntityKind; to_id: string
  created_at: string
}
```

### 2. findReadyTasks — deterministyczne rozwiązywanie DAG

Delta: scheduler to czysty kod na structured data, nie LLM. Rozdzielenie deterministycznej logiki od niedeterministycznej.

```ts
const findReadyTasks = async (sessionId: string): Promise<Task[]> => {
  const candidates = await rt.tasks.find(
    t => t.session_id === sessionId && (t.status === 'todo' || t.status === 'waiting' || t.status === 'blocked'),
  )
  const ready: Task[] = []
  for (const task of candidates) {
    if (task.status === 'todo' && !await areDependenciesMet(task)) continue
    if (task.status === 'waiting' && await areDependenciesMet(task) && !await hasUnfinishedChildren(task)) { ... }
    if (task.status === 'blocked' && shouldAutoRetryTask(task)) { ... }
    ready.push(task)
  }
  return ready.sort((a, b) => a.priority - b.priority)
}
```

### 4. Observational Memory — observer + reflector

Delta: dwufazowa kompresja per-task-thread. Split head/tail chroni ostatnie interakcje.

```
raw items (sequence > lastObservedSeq)
  → estimate tokens → above threshold?
    → split: head (compress) + tail (keep raw, ~30%)
    → Observer: LLM → 🔴🟡🟢 observations
    → above reflection threshold?
      → Reflector: multi-level (0=gentle, 1=aggressive, 2=heavy)
    → persist on Task.memory
```
