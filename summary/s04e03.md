# S04E03 — Kontekstowa współpraca z AI

## API jako warunek sine qua non integracji

Wspólny mianownik wszystkich integracji z AI: **dostępność API**. Decyzja zaczyna się od weryfikacji: czy usługa wystawia endpointy, webhooki, x-scheme-url, CLI, czy pozwala na scrapowanie. Struktura zwracanych danych bywa niekompletna lub zbyt ograniczona — weryfikacja konieczna przed projektowaniem scenariusza.

Cztery mechanizmy integracji:
- **API** — kluczowy wymóg; endpointy mogą być niekompletne z perspektywy agenta
- **Webhooki** — mechanizm reaktywny: agent nie odpytuje, odpowiada na zdarzenia
- **Deep-linki** (x-scheme-url) — URL-y wywołujące akcje w aplikacjach desktopowych bez API
- **Powiadomienia mailowe** — wyzwalacze akcji (parsowanie powiadomienia o zgłoszeniu)

## Izolacja agentów działających w tle

Główny błąd wieloagentowych systemów: **nadmiar zależności** powstający samoistnie przy rozbudowie. Problemy ujawniają się z opóźnieniem — nawarstwienie informacji lub rozbudowa odkrywa ukryte sprzężenia.

Komplikacja rośnie **nieliniowo** z liczbą połączeń. Każde połączenie = potencjalna pętla, wyścig, niespójność.

Zasada: projektuj tak, aby **konflikty nie powstawały**, a nie by były rozwiązywane:
- Agent klasyfikujący zgłoszenia nie musi wiedzieć o agencie robiącym zestawienia
- Agent weryfikujący treści nie musi znać agentów robiących research
- Pełna izolacja nie zawsze możliwa — ale dążenie do niej jest domyślnym trybem
- Gdy współpraca niezbędna → mechaniki orkiestracji z zaangażowaniem człowieka (ref. s02e04)

## Samoobserwacja systemu (Self-observing Systems)

Agenci obserwują skuteczność przepływu informacji — analogia do LLM-as-a-judge z ewaluacji:

- Newsletter, którego nikt nie czyta → wyłącz lub zmień częstotliwość
- Źródło danych niedostępne od tygodnia → oznacz i wyrejestruj
- Agent generujący powiadomienia bez reakcji → podnieś próg istotności

To mechanizm **samooczyszczania** — bez niego automatyzacje degradują w szum. LLM nie powinny generować kompleksowych raportów dla kluczowych decyzji; wartość: transformacje i klasyfikacje niedostępne przez kod + równoległa analiza z odpowiednim kontekstem.

## Kontekst urządzenia jako sygnał sterujący

Agent z dostępem do stanu urządzenia (aktywna aplikacja, lokalizacja, tryb DND) **dopasowuje formę komunikacji**:
- Praca w programie → auto-włącz DND, blokuj powiadomienia
- Sytuacja krytyczna mimo DND → eskalacja przez SMS/telefon
- Lokalizacja jako metadane notatek → automatyczna organizacja pamięci

Wymaga natywnych aplikacji (Swift / Electron / Tauri / React Native) — generowanych z pomocą AI, niekoniecznie publikowanych.

## Dwa pytania filtrujące

Przy każdym potencjalnym zastosowaniu AI:

1. **Jak AI może tu pomóc?** — czy zmniejsza zaangażowanie, podnosi jakość, odkrywa to, czego byśmy nie zauważyli?
2. **Czy powinniśmy tu angażować AI?** — czy automatyzacja nie zniszczy relacji, nie wygeneruje szumu, nie wprowadzi ryzyka (prompt injection, wyciek danych).

Modele są **słabymi doradcami** w identyfikacji scenariuszy — sugestie generyczne. Inicjatywa musi przyjść od człowieka, z obserwacji własnych procesów roboczych.

## Kluczowe napięcia projektowe

| Napięcie | Opis |
|---|---|
| Ograniczenie zaangażowania człowieka vs rozsądek w uprawnieniach | Minimum potrzebnych uprawnień dla agentów |
| Elastyczność automatyzacji vs ryzyko zapętlenia | Klasyczne automatyzacje zbyt sztywne, LLM-owe zbyt elastyczne |
| ROI z automatyzacji vs ryzyko zniszczenia relacji | Szczególnie sprzedaż i komunikacja z klientami |
| Autonomia agenta vs oczekiwanie na decyzje człowieka | Każda manualna akcja obniża efektywność systemu w tle |

## Wzorce scenariuszy kontekstowych

| Wzorzec | Mechanika | Kluczowe ryzyko |
|---|---|---|
| Katalogi aktywne | Plik w folderze → trigger → transformacja → przeniesienie (`concept/ → review/ → ready/ → published/`) | Założenia niepokrywające się z rzeczywistością (np. klient rezerwuje wiele sesji) |
| Przekierowania | Formularz → LLM klasyfikuje → przekierowanie do właściwej osoby | Niepełne dane, błędna klasyfikacja priorytetu |
| Monitorowanie wskaźników | Agent czyta analitykę + feedback → priorytetyzuje (MRR, Churn, NPS) | Fałszywe pozytywy, ignorowanie kontekstu biznesowego |
| Szablony projektowe | Framework + lista aktywności → auto-przydzielanie zadań | Sztywność szablonu w zderzeniu z nietypowym projektem |
| Sugestie wydarzeń | Notatki + obserwacja otoczenia → propozycje w **oddzielnym kalendarzu** | Przeładowanie propozycjami — akceptacja przed przeniesieniem do głównego |
| Kontrola jakości | Treść w kolejce publikacji → agent weryfikuje linki, poprawność, spójność | Fałszywe negatywy blokujące publikację |

## Wzorce bezpieczeństwa per obszar

**Oddzielny kalendarz** — agent tworzy sugestie w osobnym kalendarzu (nie głównym). Dopiero po akceptacji przenoszone do głównego. Rozwiązuje problem zaśmiecania.

**Osobny e-mail agenta** — własny adres odbierający newslettery, powiadomienia systemowe, przekierowane wątki. Ograniczenia: wysyłka wyłącznie do właściciela, dostęp read-only do bazy wiedzy. Na główny adres trafiają tylko najistotniejsze informacje.

**Manager schowka** — obserwowanie schowka systemowego pod kątem zasobów wiedzy. Kluczowe: zawartość schowka to ostatnia rzecz do przesyłania na serwery dostawców — wymaga lokalnych modeli.

## Nasłuchiwanie sygnału — ranking częstościowy

Agenci monitorujący źródła (YouTube, X, newslettery, blogi) rankingują wydarzenia po częstotliwości pojawiania się w danym okresie. Narzędzie wspomniane wielokrotnie w miesiącu = sygnał, że warto zwrócić uwagę. Konkretna technika priorytetyzacji, nie tylko "monitorowanie".

## Baza wiedzy — powiązania i duplikacja

Z czasem pojawia się duplikacja informacji i potrzeba powiązania wpisów. Agenci obecnie nie radzą sobie wystarczająco skutecznie by oddać proces w całości, ale generowanie sugestii powiązań na podstawie najnowszych zmian ma duży sens.

## Panel zarządzania (Second Brain)

Koncepcja spersonalizowanej "ściany" agregującej kontekst zewnętrzny (nowości branżowe) i wewnętrzny (obszary wymagające uwagi). Poprzedniczka LLM-era: "Second Brain". Przegląd materiałów z tego obszaru daje inspiracje rozszerzalne przez AI.

## Projektowanie pod przypadki brzegowe

Źle zaprojektowany system agentowy przynosi więcej problemów niż wartości. Przykład: proces konsultacji (rezerwacja → przygotowanie → realizacja → rozliczenie) — klient chcący zarezerwować wiele sesji i opłacić z góry rozbija założenia uproszczonego flow. Analogia do programowania: założenia adresowane w projekcie okazują się niewystarczające w zderzeniu z rzeczywistością.
