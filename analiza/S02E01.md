# S02E01 — Zarządzanie kontekstem w konwersacji

## Instrukcja systemowa jako mapa

Instrukcja systemowa to **mapa terenu, nie teren** — daje agentowi orientację, ale nie zastępuje eksploracji. Zawiera to, co agent musi wiedzieć **zanim** sięgnie po narzędzia, żeby skrócić ścieżkę do celu.

Cztery warstwy treści w system prompcie:

| Warstwa | Co zawiera | Zmienność |
|---|---|---|
| **Uniwersalne instrukcje** | Rola pamięci, generalne zasady zachowania, główne obszary wiedzy | Statyczna |
| **Otoczenie** | OS, ustawienia, profil użytkownika, typ interfejsu, dostępne zasoby | Niska (per sesja) |
| **Sesja** | Co wydarzyło się wcześniej, jak dotrzeć do skompresowanego kontekstu | Średnia |
| **Zespół agentów** | Zasady komunikacji, współdzielenia kontekstu, wspólne fragmenty promptów | Statyczna (per system) |

Kluczowe pytanie projektowe: **o czym agent musi wiedzieć PRZED uruchomieniem narzędzi?** Wszystko inne powinno być odkrywane dynamicznie.

Opisy narzędzi nie należą do system promptu, jeśli to samo narzędzie może pełnić różne role u różnych agentów. System prompt zawiera **zgeneralizowane** opisy; specyfika żyje w schemacie narzędzia.

## Sygnał vs szum

Proporcja sygnału do szumu wynika z **detali systemu**, nie z jednorazowej optymalizacji promptu. Pięć obszarów, które ją kształtują:

- **Poprawne dostarczanie kontekstu** — niekompletne lub błędne dane z narzędzi zdarzają się często (konflikty narzędzi, mylne odpowiedzi z API)
- **Jakość logiki aplikacji** — mniej kodu w systemach agentowych, ale musi być niezwykle dopracowany. Agent nie "obejdzie" błędów w swoim harnesie
- **Dopracowane instrukcje i schematy** — prompt engineering to teraz projektowanie **komponentów** promptu pojawiających się dynamicznie. Muszą działać niezależnie od momentu dołączenia do kontekstu
- **Generyczne mechanizmy** — kompresja kontekstu, planowanie, monitorowanie postępów. Projektowane uniwersalnie, żeby dawały wartość w każdej sytuacji
- **Przestrzeń na doprecyzowanie** — interwencje człowieka lub innych agentów dostarczające "sygnał" gdy agent nie może kontynuować

## Agentic RAG — budowanie kontekstu przez obserwację

Agent domyślnie **nie wie, o czym wie**. Mapa (system prompt) nie wystarczy — nie uwzględnia detali ujawniających się dopiero podczas eksploracji. Rozwiązanie: agent sam buduje kontekst przez iteracyjne wyszukiwanie.

Czterofazowa strategia nawigacji po zasobach:

1. **Skanowanie** — struktura folderów, nazwy plików, nagłówki dokumentów. Gdy brak konkretnej ścieżki, zaczynaj od rozpoznania terenu
2. **Pogłębianie** (iteracyjne) — seria zapytań z wariantami słów kluczowych, synonimami, terminami odkrytymi w poprzednich wynikach. Nie jedno wyszukiwanie — pętla: szukaj → czytaj → zbierz nowe terminy → szukaj ponownie
3. **Eksplorowanie** — tropienie powiązań: przyczyna/skutek, część/całość, problem/rozwiązanie, ograniczenia/obejścia, wymagania/konfiguracja. Każde powiązanie to osobny trop
4. **Weryfikacja pokrycia** — czy mam dość danych na definicje, liczby, edge case'y, kroki, wyjątki? Jeśli nie, powrót do pogłębiania

Te zasady są **generyczne** — działają niezależnie od tego, czy agent przeszukuje pliki markdown, bazę wektorową, czy API. Jedyny specyficzny fragment promptu to informacja o charakterze danych (np. język dokumentów).

Problem: agent szukający po angielsku w polskich dokumentach nic nie znajdzie. Rozwiązanie nieoczywiste — zamiast budować logikę translacji zapytań, wystarczy jedna linia w prompcie o języku treści. Agent sam skoryguje zachowanie.

## Cache hit jako priorytet architektoniczny

Narzędzia w oknie kontekstowym siedzą **pod** system promptem. Każda zmiana system promptu invaliduje cache definicji narzędzi.

Konsekwencje projektowe:
- System prompt musi być **statyczny** w ramach sesji
- Dynamiczne dane (git status, otwarte pliki, data) trafiają do **wiadomości użytkownika**, nie do system promptu
- Kolejne wiadomości użytkownika zawierają coraz mniej metadanych — tylko to, co zmieniło się od ostatniej wiadomości
- Agent odświeża starsze informacje (np. git status) przez narzędzia, nie przez modyfikację promptu

Wiadomość użytkownika nie musi zawierać wyłącznie tego, co użytkownik napisał. Programistyczne dołączenie kontekstu, który użytkownik **mógłby** dostarczyć, jest uzasadnione. Tagi XML oddzielają treść zapytania od dołączonych metadanych.

## Powtórzenia jako sterowanie uwagą

Model gubi fakty w długich konwersacjach. Dwa mechanizmy przeciwdziałania:

**Listy zadań** — model zapisuje co ma zrobić, po każdym kroku odznacza i **przepisuje** pozostałe punkty. Powtórzenie treści w kolejnych wiadomościach utrzymuje uwagę. Treść generowana przez model może mieć silniejszy wpływ na jego zachowanie niż treść zewnętrzna (por. many-shot jailbreaking). Problem: bez programistycznego wsparcia model zapomina o aktualizowaniu listy.

**Tryb planowania** — informacja o trybie i zasadach dołączana do wiadomości użytkownika (nie do system promptu — cache). Nie modyfikuje wcześniejszych wiadomości, jedynie sygnalizuje w najnowszej.

## Kontrola stanu poza oknem kontekstu

Agent Harness — infrastruktura wokół modelu umożliwiająca skuteczne funkcjonowanie:

- **Sesja** — hooki monitorujące interakcje, częściowe podsumowania w tle, aktualizacje z zewnętrznych systemów czekające w kolejce
- **Pamięć** — budowanie wspomnień w tle, nawet po sesji. Batch API dla obniżenia kosztów
- **Pliki** — wyniki narzędzi, komunikacja między agentami, załączniki, tymczasowe notatki. Mogą powstawać bez udziału agenta i być wykorzystane później
- **Otoczenie** — dane spoza urządzenia użytkownika, aktualizowane poza kontekstem, wstrzykiwane warunkowo

## Maskowanie kontekstu (technika Manus)

Uzupełnianie początku wypowiedzi modelu tokenami wymuszającymi wywołanie konkretnego narzędzia: `<|im_start|>assistant<tool_call>{"name": "browser_`. Dopóki przeglądarka jest aktywna, agent fizycznie nie może wywołać innego narzędzia. Ograniczenie zdejmowane deterministycznie po zakończeniu sesji przeglądarki.

Technika deprecated w API Anthropic, ale ilustruje ważną zasadę: **nawet niekonwencjonalne podejścia mogą adresować całe klasy problemów**.

## Workspace agentów

Przestrzeń plików w ramach sesji:

- Załączniki użytkownika
- Notatki z sesji
- Dokumenty "publiczne" (dla użytkownika)
- Dokumenty "wewnętrzne" — **outbox** (od agenta dla innych) i **inbox** (od innych agentów)

Katalogi organizowane per data, per sesja, per agent. Programistyczne ograniczenie dostępu: agent modyfikuje tylko swój `notes/` i `outbox/`. `inbox` zapisywany wyłącznie przez root agenta. Izolacja sesji = izolacja danych użytkowników.

## Generalizowanie instrukcji z pomocą LLM

Model skutecznie **uzasadnia** swoje zachowanie i sugeruje zmiany w promptach, ale pierwsza propozycja jest zwykle zbyt bezpośrednia (naprawia przypadek, nie kategorię). Proces iteracji:

1. **Analiza** — opisz problem, podaj kontekst (narzędzia, prompt, zachowanie)
2. **Generalizacja** — żądaj uniwersalnych przyczyn, nie łatek na konkretny przypadek
3. **Korekta** — ~60% sugestii modelu jest bezwartościowych, ~30% wymaga zmian. Podkreśl niezależność promptu od narzędzi i unikanie "przesterowania"
4. **Iteracja** — proste wskazówki, z których model wyciąga esencję

Wynikowy prompt ma zwięzłe, precyzyjne sformułowania ("provide clear, concise answers", "ask or pick safest option"), do których trudno dojść samodzielnie.

## Przykład kodu: `02_01_agentic_rag`

Interaktywny agent RAG — REPL chatbot przeszukujący lokalne pliki markdown przez MCP.

### Architektura

Jedyna zależność: `@modelcontextprotocol/sdk`. Reszta to Node.js stdlib + `fetch`.

```
app.js          → MCP client + REPL + shutdown
src/agent.js    → pętla agentowa (chat → tool calls → results → chat)
src/config.js   → system prompt + parametry modelu
src/mcp/client.js → MCP stdio transport, konwersja narzędzi na format OpenAI
src/repl.js     → readline z historią konwersacji
```

### Delta względem wcześniejszych przykładów

**MCP jako abstrakcja narzędzi** — agent nie zna swoich narzędzi w compile-time. Pobiera je z MCP servera, konwertuje na format OpenAI, przekazuje do API. Zmiana źródła danych = zmiana `mcp.json`, zero zmian w kodzie agenta.

**Wielokrokowa pętla agentowa z limitem** — max 50 kroków. W każdym: wywołanie LLM → jeśli tool calls, wykonaj równolegle (`Promise.all`) → wrzuć wyniki → powtórz. Gdy brak tool calls = odpowiedź końcowa.

**Reasoning model z widocznością myślenia** — `reasoning: { effort: "medium", summary: "auto" }`. Logger wyświetla reasoning summaries — widać jak model planuje wyszukiwanie przed wywołaniem narzędzi.

**Statyczny prompt, dynamiczna konwersacja** — prompt i narzędzia ładowane raz. Jedyna zmienna to rosnąca `conversationHistory`. Prefiks kontekstu nie zmienia się = wysoki cache hit.

**Tracking kosztów** — `stats.js` zlicza input/output/reasoning/cached tokens per sesja. Logger pokazuje breakdown po każdym wywołaniu API. Komenda `clear` resetuje i historię, i statystyki.
