# S05E03 — Rozwój funkcjonalności

## Fundamenty vs ekosystem — asymetria zmian

Architektura transformerów, autoregresja, tokenizacja, limity kontekstu, halucynacje — **stabilne od 3 lat**. Postęp na tym poziomie to skalowanie + syntetyczne dane treningowe + reasoning w inferencji. Nie przełom rangi następcy transformerów.

Dynamiczne zmiany dotykają wyższych warstw: modele, techniki pracy, narzędzia, otwartość API na agentów. Konsekwencja projektowa:

- Logika oparta o **fundamentalne mechaniki** (tokenizacja, okno kontekstu, determinizm outputu) — mniejsze ryzyko deprecjacji
- Logika oparta o **konkretne API/frameworki** — planuj wymienność, multi-provider

## Dwuznaczność architektury

**Uproszczenie:** Oddelegowanie logiki do modelu. Systemy RAG przechodzą od wieloetapowych pipeline'ów do agenta z serią prostych narzędzi. Agentic RAG < klasyczny RAG pod względem ilości kodu.

**Komplikacja:** Środowisko w którym agent operuje — sandboxy, multimodalność, długie horyzonty, integracja z lokalnym systemem, bezpieczeństwo. Fabryka zdolna do kształtowania linii produkcyjnej, nie sama linia.

Analogia: programowanie klasyczne = budowanie elementów linii produkcyjnej. Programowanie generatywne = budowanie fabryki, która je kształtuje.

## Strategia migracji modeli i API

- **Responses API** przyjęło się jako standard branżowy w ~4 miesiące. **Assistants API** — deprecated po 15 miesiącach. Różnica: Responses API nie uzależnia od provider-specific abstrakcji (threads, runs), lecz rozszerza uniwersalny format wiadomości o function calling i structured output.
- Mniejsze wersje modeli (mini/nano) **sprawiają wrażenie** większych możliwości niż realnie mają — wymagają dokładniejszej ewaluacji.
- Nowsze modele mogą **zmiękczyć** wcześniej rekomendowane praktyki — np. agresywny ton (CRITICAL, MUST) w Opus 4.5 przestał być rekomendowany przez Anthropic.
- Migracja to nie tylko zmiana identyfikatora modelu — w systemach agentowych może wymagać reewaluacji instrukcji, uproszczenia promptów, lub odwrotnie — umożliwić zwiększenie złożoności.

Sygnały nowości wartej uwagi: sam sprawdzaj, zachowaj dystans do szumu marketingowego, daj drugą szansę gdy temat wraca wielokrotnie. Obserwuj prace badawcze providerów i Model Spec jako wskaźnik kierunku.

## Rozwój agentów przez narzędzia, nie kod

Agent z dostępem do systemu plików + pamięć = podstawa zdolna do samodzielnego programowania aplikacji. Rozszerzenia to dodanie serwerów MCP, zero zmian w logice agenta:

| Rozszerzenie | Zysk | Koszt implementacji |
|---|---|---|
| Lepszy model | Natychmiastowa poprawa jakości | Zmiana stringa |
| Terminal (bash) | Build, test, git, deploy | Nowy MCP server |
| Przeglądarka | Interakcja z web UI | Nowy MCP server |
| Gmail/Calendar/Todoist | Automatyzacja workflow | Nowy MCP server |

Wniosek: architektura agenta powinna być **agnosticzna** wobec narzędzi — dynamiczne odkrywanie (`listTools()`), transformacja schematów MCP → function calling, zero hardcodowania.

## Samorozwój agentów

Automatyczna optymalizacja promptów to koncepcja produkcyjnie realizowana przez DSPy (Python) i Ax (TypeScript). Dwa mechanizmy:

1. **Optymalizacja instrukcji** — hill-climbing: generuj kandydatów → ewaluuj → zachowaj lepszego. Metaprompt instruuje LLM jak modyfikować inne prompty.
2. **Optymalizacja demonstracji** — BootstrapFewShot: uruchom na danych treningowych → zbierz traces z udanych predykcji → użyj jako few-shot. Prompt się nie zmienia, zmieniają się przykłady.

Oba podejścia wymagają: metryki ewaluacyjnej, zbioru treningowego + walidacyjnego, mechanizmu ochrony przed overfittingiem.

Sygnatury Ax eliminują prompt z kodu aplikacji — deklarujesz `input:type -> output:type "opis"`, framework generuje prompt. To przeskok od "prompt engineering" do "prompt programming".

## Pułapki wdrożeń produkcyjnych

| Obszar | Ryzyko | Mitigacja |
|---|---|---|
| Rate limit | Unieruchomienie przy skali | OpenRouter, rotacja kluczy |
| Moderacja | Blokada konta za treści userów | Moderation API, filtry wejścia/wyjścia |
| Wydajność | Spowolnienie przy równoległych zapytaniach | Unikanie >100 równoległych calls na usera |
| Koszty | 1-3% userów generuje większość kosztów | Twarde limity per user, dedykowane klucze |
| Skuteczność | Demo działa, userzy robią coś innego | Obserwacja aktywności, przyciski zamiast czatu gdy możliwe |
| Szybkość | LLM zabija UX | Cache, równoległość, streaming, cząstkowe odpowiedzi |
| Użyteczność | Agent zarządzający todo przez czat < dobry UI | Czat sensowny na skali lub przy złożonych akcjach |

## Przykłady kodu — progresja i delta wiedzy

### 05_03_coding — Minimalny agent z pamięcią i narzędziami

**Delta:** Kompletny agent w ~100 linii core logic (REPL + agent loop + MCP + memory). Zero frameworków AI.

- **Agent loop** — `while(tool_calls) { LLM → execute → LLM }` z `MAX_TURNS=30`. Model decyduje o zakończeniu — brak tool calls = odpowiedź końcowa. To wystarczający wzorzec dla większości zastosowań.
- **Rolling memory** — sliding window zachowuje K ostatnich wiadomości, starsze kompresuje przez osobny tańszy model. Kompresja proaktywna: wyzwalana *zanim* kontekst wyjdzie poza limit (>18 wiadomości lub >18K znaków). Podsumowanie zapisywane do pliku, wstrzykiwane do system prompt.
- **Dynamiczne odkrywanie narzędzi** — MCP `listTools()` → transformacja schematów → function calling. Zmiana serwera MCP nie wymaga zmiany agenta.
- **`parallel_tool_calls: false`** — operacje na plikach muszą być sekwencyjne. Agent kodujący to nie chatbot.
- **`store: false`** — prywatność przez design, OpenAI nie loguje konwersacji.

### 05_03_autoprompt — Autonomiczna optymalizacja promptów

**Delta względem coding:** Agent nie wykonuje zadań — optymalizuje instrukcje. Architektura 3-rolowa: execution (tani model) → judge (mocny model) → improver (mocny model).

- **Noise floor** — `delta > max(spread) / 2`. LLM jest niedeterministyczny — poprawa o 0.01 nie jest istotna statystycznie. Spread obliczany z wielu runów; gdy `runs=1`, każda poprawa jest akceptowana (tryb demo).
- **Stuck detection** — 3 odrzucone próby z tą samą operacją → wymuszona zmiana strategii z zabronioną operacją. Zapobiega cyklicznemu próbowaniu tego samego.
- **Metaprompt** — 137 linii instrukcji dla LLM optymalizującego inne prompty. 5 atomowych operacji (REWORD/ADD/REMOVE/MERGE/REORDER), jedna na iterację. Strategia zależna od wyniku: <0.5 napraw fundamenty, >0.85 tylko usuwaj.
- **Anti-verbosity creep** — największy bias improver LLM: domyślnie dodaje reguły. Ręczne wymuszanie "fewer rules = more compliance" w instrukcjach.
- **Train/verify split** — 2 transkrypcje do optymalizacji, 1 holdout do weryfikacji. ML 101 zastosowane do prompt engineeringu.
- **LLM-as-judge z konfigurowalną polityką** — sekcje z wagami (suma=1.0), matchBy do identyfikacji encji, exact vs semantic matching.

### 05_03_ax — Prompt programming przez sygnatury

**Delta względem autoprompt:** Zamiast optymalizować prompt, optymalizuj **demonstracje**. Prompt w ogóle nie istnieje w kodzie — generowany z deklaratywnej sygnatury.

```
emailFrom:string, emailBody:string -> labels:string[], priority:class "high,medium,low"
```

- **BootstrapFewShot** — uruchom klasyfikator na training set → zbierz traces z udanych predykcji → zapisz jako dema. Następne uruchomienie używa tych dem jako few-shot. Zero zmian w sygnaturze.
- **Consistency bonus w metryce** — ocenia nie tylko vs ground truth, ale też spójność wewnętrzną predykcji (`needsReply=true` ↔ label `needs-reply`). Wzorzec "self-consistency scoring".
- **Jaccard similarity** — ocena częściowej poprawności zbiorów etykiet. Model znajduje 3 z 4 etykiet → dostaje częściową ocenę.
- **Dwa tryby** — fallback (ręczne examples) + opcjonalna optymalizacja (demos.json). System działa od razu, może się poprawić.
- **Duplikaty w wygenerowanych demach** — BootstrapFewShot nie deduplikuje traces, 3 identyczne wpisy w demos.json zożerają tokeny. Praktyczna uwaga przy stosowaniu optimizerów.
