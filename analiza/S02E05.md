# S02E05 — Projektowanie agentów

## Architektura instrukcji systemowej

Instrukcja agenta to **zbiór sekcji o wyraźnie odmiennych odpowiedzialnościach**, nie jeden blok tekstu. Każda sekcja ma inny cykl życia — od statycznego (identity) po dynamicznie generowany w runtime (workspace/knowledge).

### Identity

Motyw przewodni łączący charakter, styl i zachowanie. Szeroka perspektywa bez detali technicznych. Celowe użycie nietypowego słownictwa i skojarzeń nie mówi modelowi co robić — **pokazuje** to przez kierowanie uwagi. Analogia do "nie myśl o niebieskich motylach": sekcja identity kieruje uwagę modelu w określone obszary, co wpływa na jakość pracy silniej niż dosłowne instrukcje.

Paradygmat: zamiast wymagać dokładnego wykonania, zaprojektować przestrzeń, by model **pozytywnie zaskoczył**. Halucynacje jako potencjalny atut, nie problem.

### Protocol

Zasady działania osadzone w środowisku: zarządzanie kontekstem, wspomnieniami, relacjami. Może zawierać wzmianki konkretnych ścieżek/plików, ale z balansem — nie wiąże agenta ze strukturą katalogów na stałe. Kluczowe elementy: co gdy agent nie potrafi coś zrobić? Jak prosić o pomoc?

### Voice

Ton wypowiedzi wymaga **rozbudowanej sekcji**, bo LLM szybko zapominają o tonie i wracają do domyślnego. Narzędzia: przykłady wyrażeń, serie skojarzeń, antywzorce, few-shot. Krytyczny przy multimodalności — agent głosowy musi unikać dyktowania linków czy wyświetlania obrazów. **Środowisko determinuje styl**, nie narzędzie.

### Tools

Niemal w pełni wygenerowana dynamicznie — schematy narzędzi opisują się same. Wyjątek: wskazówki o specyficznym zachowaniu (np. nie mylić powiadomienia z komunikatem). Przy progressive disclosure — agent musi wiedzieć, jak szukać narzędzi.

### Sekcja dynamiczna (workspace)

Tu trafia Observational Memory i inne dane sesyjne/wiedzy generowane w runtime. Na końcu instrukcji — CTA sygnalizujące zakończenie.

## Obszary projektowania agenta

| Obszar | Co definiuje | Cykl życia |
|--------|-------------|------------|
| Ustawienia | Nazwa, opis (po nich inni agenci wywołują), narzędzia, model, uprawnienia | Stały, z domyślnymi |
| Profil | Osobowość, złożoność, długość, format, techniki rozumowania | Stały |
| Zasady | Komunikacja, obsługa błędów, dostęp do wiedzy, zachowania awaryjne | Stały |
| Limity | Aktualność wiedzy ("kiedy jest teraz?"), dynamiczne uprawnień | Mieszany |
| Styl | Sposób wypowiedzi zależny od środowiska (tekst/głos/GUI) | Stały |
| Sesja | Użytkownik, preferencje, aktywność agentów, dane dynamiczne | Dynamiczny |

## Strategia przypisywania narzędzi

"10-15 narzędzi to max" to mit. Liczba narzędzi zależy od złożoności integracji, nie od reguły. Jedno narzędzie (CLI do GitHub) może zastąpić kilkanaście API calls. Trzy narzędzia do prostego serwisu mogą nie wystarczyć.

**Progresywne odkrywanie (progressive disclosure)** — agent dostaje meta-narzędzia do odkrywania reszty w runtime. Nie ładuje wszystkiego do kontekstu na start.

**Współdzielenie narzędzi** między agentami zmniejsza wymianę informacji, ale rodzi problem: agent z ograniczonym dostępem do pamięci, nie znajdując informacji, **zakłada że nie istnieje** zamiast poprosić o dostęp.

**Pytanie filtrujące**: czy system stanie się lepszy z lepszymi modelami? Jeśli nie — prawdopodobnie budujemy niewłaściwą rzecz.

## Sandbox

Agent w sandboxie startuje z minimum meta-narzędzi, odkrywa narzędzia docelowe dynamicznie, **pisze kod** wykorzystujący je i uruchamia w izolowanym środowisku. Dane funkcjonują jako zmienne w kodzie — nie trafiają do kontekstu agenta.

**Co to rozwiązuje:**
- Agent łączy narzędzia elastycznie bez indywidualnych wrapperów
- Duże zbiory danych przetwarzane bez powiększania kontekstu
- Ograniczenie kontaktu ze światem zewnętrznym
- Izolacja: limit pamięci, timeout, brak filesystem/network

**Koszty:** wzrost złożoności architektury, dodatkowa warstwa abstrakcji, nowe problemy specyficzne dla izolowanego środowiska.

## Organizacja wiedzy

| Kategoria | Charakter | Dostęp |
|-----------|-----------|--------|
| Dokumenty sesji | Załączniki + pliki wygenerowane w sesji | Agenci w bieżącej interakcji |
| Wiedza publiczna | Pamięć długoterminowa współdzielona | Agenci + użytkownicy |
| Wiedza prywatna | Dane i kontekst użytkownika | Tylko właściciel |
| Wiedza agentów | Instrukcje, wspomnienia, obserwacje | Konkretny agent |
| Pamięć podręczna | Wyniki wyszukiwania, treści stron | Tymczasowa |
| Runtime | Baza sesji, interakcji, harmonogramów | Niewidoczna dla agentów |

Kategoryzacja jest niejednoznaczna — ta sama informacja może należeć do wielu kategorii. Proste zasady organizacji > złożone systemy kategoryzacji. Czasem wystarczą proste dokumenty zamiast zaawansowanej pamięci długoterminowej.

## Ograniczenia i ryzyka

- **Połączenia między narzędziami** — agent z bezpiecznym filesystemem może przesłać informację do niezabezpieczonej lokalizacji przez inne narzędzie
- **Instrukcja nie gwarantuje przestrzegania** — model może ignorować lub źle interpretować
- **Złożoność rośnie szybko** — nawet z pomocą LLM w tworzeniu promptów, kilkanaście iteracji to norma
- **Ograniczanie użyteczności** — nadmierne limity mogą sprawić, że system traci sens

## Przykłady kodu

### 02_05_agent — Observational Memory (Observer/Reflector)

**Delta względem poprzednich lekcji:** kompresja pamięci konwersacyjnej z priorytetyzacją i kalibracją tokenów.

**Observer** — gdy nieskompresowane wiadomości przekroczą threshold, LLM ekstrahuje strukturalne obserwacje z priorytetami:

```
* 🔴 [user] User's name is Adam, developer from Poland
* 🟡 [tool:write_file] Created notes/adam-profile.md
* 🟢 [assistant] Acknowledged preferences
```

Tagi źródłowe (`[user]`, `[assistant]`, `[tool:name]`) determinują co reflektor kompresuje najpierw — `[user]` fakty najwyżej, `[assistant]` elaboracje najniżej.

**Reflector** — gdy obserwacje przekroczą budżet, kompresuje w 3 poziomach agresywności. Zwiększa generation count. Każdy level to osobny LLM call — zatrzymuje się gdy tokens <= target.

**Token estimation** — heurystyka `chars/4` z runtime'ową kalibracją: porównuje szacunki z rzeczywistym użyciem API, buduje ratio do korekty. Safety margin 1.2x.

**Context window layout:**

```
┌──────────────────────────────────────────────────┐
│ System prompt + Observations │ Unobserved tail   │
│ (skompresowana historia)     │ (raw messages)    │
└──────────────────────────────────────────────────┘
```

Observer "zapieczętowuje" starsze wiadomości (sealed), agent dostaje observations w system prompt + raw recent messages jako input.

**Persistence** — obserwacje i refleksje zapisywane jako `.md` z frontmatterem (sekwencja, generacja, tokeny, zakres indeksów). Debugowalne bez narzędzi.

**Config` `.agent.md`** — gray-matter frontmatter (model, tools) + body (system prompt). Zachowanie agenta konfigurowalne bez zmiany kodu.

### 02_05_sandbox — MCP + QuickJS + Tool Discovery

**Delta względem 02_05_agent:** dynamiczne odkrywanie narzędzi, generowanie kodu zamiast bezpośrednich tool calls, izolowane wykonanie w sandboxie.

**Workflow odkrywania:**
1. `list_servers` → co jest dostępne
2. `list_tools("todo")` → jakie ma metody
3. `get_tool_schema("todo", "create")` → **TypeScript type definition** (nie JSON schema) ładowana do kontekstu
4. `execute_code` → agent pisze JS korzystający z załadowanych API

**Kluczowa decyzja:** narzędzia opisane TypeScriptem, nie JSON Schema. LLM naturalnie rozumie typy TS — niższe ryzyko błędnej interpretacji parametrów.

**QuickJS sandbox** — asyncified host functions pozwalają narzędzia MCP (asynchroniczne w Node) wywoływać **synchronicznie** z perspektywy kodu w piaskownicy:

```js
const result = todo.create({title: "Buy milk"}); // sync, nie async
```

QuickJS asyncify wstrzymuje/wznowi wykonywanie JIT. Wrapper w IIFE — narzędzia dostępne jako obiekty (`todo.create()`, `todo.list()`).

**MCP over stdio** — serwer narzędzi jako osobny proces, komunikacja stdin/stdout. `StdioClientTransport` + `McpServer`. Konwencja nazewnictwa: `server__tool` (podwójny underscore).

**Rejestr narzędzi** — statyczny registry z pełnymi sygnaturami TS. Agent ładuje tylko te, których potrzebuje (`loadedTools` Map). Reszta nie zajmuje kontekstu.
