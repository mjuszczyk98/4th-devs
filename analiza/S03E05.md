# S03E05 — Niedeterministyczna natura modeli jako przewaga

## Niedeterminizm jako zasada, nie błąd

LLM to "śniąca maszyna" — wszystko jest "halucynacją", która czasem nam sprzyja. Precyzyjne podążanie za instrukcjami i generowanie zróżnicowanych odpowiedzi to dla modelu **równie trudne**. Zmiana `temperature` czy `top_p` nie rozwiązuje problemu powtarzalności — kluczowa jest **zawartość kontekstu**, a parametry mają znaczenie marginalne.

Dla tych samych danych wejściowych model **nie zachowuje się jak funkcja czysta**. Zamiast z tym walczyć, należy zaprojektować system tak, by niedeterminizm był **źródłem wartości**.

## Od poleceń do przestrzeni interpretacji

Klasyczny agent: polecenie → akcja. Agent "świadomy": kontekst → model sam decyduje czy/i co działać.

| Wymiar | Agent skryptowy | Agent świadomy |
|--------|----------------|----------------|
| Trigger | bezpośrednie polecenie | niejawne potrzeby, kontekst sytuacyjny |
| Podejmowanie decyzji | reguły w kodzie | model ocenia luki w wiedze i sam dobiera akcje |
| Kontekst | wczytywany eagerly | lazy loading — tylko co potrzebne, kiedy potrzebne |
| Zachowanie dla "cześć" | brak akcji lub stała odpowiedź | model sam ocenia czy potrzebuje kontekstu |
| Wynik dla identycznego inputu | przewidywalny | **bardzo niskie prawdopodobieństwo powtórzenia** |

**Nie straciliśmy kontroli** — agent porusza się po wyznaczonej przez nas przestrzeni. Przestrzeń jest po prostu znacznie większa niż w przypadku sztywnych reguł.

## Cztery obszary oddelegowane do modelu

1. **Proaktywność** — model sam określa **kiedy** działać i jakiej wiedzy potrzebuje. Nie reaguje na polecenia, lecz na **podejrzenia** i otwarte pytania.
2. **Synteza** — brak instrukcji "jeśli X to Y". Dynamiczne łączenie informacji z dostępnych i **potencjalnie** posiadanych źródeł.
3. **Wnioskowanie** — dedykowane narzędzia (`think`) zachęcające do samorefleksji. Ich rolą jest **stworzenie przestrzeni na zastanowienie**, sama obecność zmienia zachowanie modelu.
4. **Dopasowanie** — model "decyduje" o fokusie uwagi, tonie, stylu. Mechaniki wzmacniają **kategorie zachowań**, nie konkretne akcje.

## Architektura kognitywna agenta

Przestajemy projektować zachowania. Zaczynamy **stwarzać warunki**, w których zachowania powstają samoistnie. To odchodzi od prompt engineering w stronę **cognitive architecture** (ref. ["Cognitive Architectures for Language Agents"](https://arxiv.org/pdf/2309.02427)).

Podejście to stoi w kontrze do kierunku, w którym zmierza branża (podążanie za instrukcjami, eliminacja halucynacji). Nie wyklucza go jednak — wprowadza **zmienność** pozwalającą na odkrywanie ścieżek, o których trudno pomyśleć na etapie założeń projektu.

Warstwy budowane w instrukcjach agenta:

- **Tożsamość** — agent wie, że ma osobowość, nastrój, opinie, ale na początku interakcji pozostają one **rozmyte** dopóki nie zostaną **odkryte**
- **Zdolności poznawcze** — instrukcje wpływające na zachowanie przez zadawanie pytań, nie przez reguły
- **Inteligencja emocjonalna** — odczytywanie stanu rozmówcy, dopasowanie do celu, w tym gdy użytkownik **jeszcze nie wie** czego szuka
- **Sposoby ekspresji** — dopasowanie stylu wymuszone przez **metadane** wstrzykiwane do wiadomości, nie przez prompt
- **Mechaniki wzmacniające** — w metadanych i wynikach narzędzi pojawiają się **postawy** (nie akcje), które model powinien brać pod uwagę

### Kluczowe instrukcje kształtujące zachowanie

- Wiedza o sobie, rozmówcy i otoczeniu pochodzi z **zewnętrznego kontekstu**, nie z bazowej wiedzy modelu
- Wczytane informacje nie zawsze zawierają dane do przekazania — stają się elementem **zrozumienia sytuacji**
- Agent ma pozwolenie na **luźne łączenie faktów** i wnioskowanie na podstawie potencjalnych powiązań
- Każdy nowy temat = potencjalnie nowa przestrzeń wymagająca odkrycia

Wszystkie powyższe **kierują "snem" modelu** bez bezpośrednich poleceń prowadzących w konkretną stronę.

## Spektrum generowania UI przez AI

Trzy podejścia do tego samego problemu — wykorzystania LLM do tworzenia interfejsów — na różnym poziomie kontroli:

| | Artifacts | Render | MCP Apps |
|---|---|---|---|
| **Co generuje model** | Dowolny HTML/CSS/JS | JSON spec (katalog komponentów) | Nic — UI jest predefiniowany |
| **Kontrola** | Niska (CSP jako jedyna bariera) | Średnia (guardrails prompt + validation) | Pełna |
| **Stan** | Efemeryczny | Inicjalizowalny, zapisywalny, wczytywalny | Dwukierunkowa synchronizacja |
| **Interakcja** | Tylko wewnątrz iframe | Tylko wewnątrz iframe | Akcje usera → narzędzia MCP → system |
| **Widoczność dla modelu** | Pełna (model widzi kod) | Pełna (model widzi spec) | Brak — model widzi tylko udostępnione dane |
| **Rola hosta** | Standardowa | Standardowa | **Zwiększona** — obsługuje nie tylko user↔agent, ale też interfejs |
| **Produkcyjna gotowość** | Niska | Średnia | Wysoka |

**Lekcja:** dyskusję "X vs Y" zamień na "kiedy X, kiedy Y, kiedy X+Y". Koncepcje się nie wykluczają.

Tryby **rozbudowanego rozumowania** (`reasoningEffort`) pozwalają modelom radzić sobie ze złożonymi interfejsami generowanymi "w locie" — nie mówimy już o prostych wykresach, ale o panelach pokazujących dane z wielu perspektyw.

**Przesunięcie roli projektanta:** z ręcznego budowania detali w stronę projektowania **struktur**, po których poruszają się agenci. Sztuka **balansowania** przewidywalności (stabilne realizowanie procesów) z dynamiką (możliwości generatywnego AI).

## Narzędzia jako cognitive scaffolding

Narzędzie `think` nie robi I/O. Jego obecność zmienia zachowanie modelu — zmusza do **pauzy i samorefleksji** przed odpowiedzią. To realizacja chain-of-thought jako explicit tool, a nie prompting technique.

```
think → "Jakie założenia robię? Czego nie sprawdziłem?"
      → nudge: "Now decide: which of these could recall help answer?"
      → model sam decyduje czy wywołać recall
```

Narzędzie `recall` deleguje do **scout sub-agenta** — osobnego agenta z własnym system promptem i własną sesją, z dostępem do filesystem przez MCP. Główny agent nie czyta plików sam. Separation of concerns: conversationalist + retrieval specialist.

## Temporal grounding

Metadane czasowe wstrzykiwane w **każdą** wiadomość — fundamentalny building block dla agentów "świadomych":

```
<metadata>
now_iso: 2026-03-30T10:53:19.897Z
weekday: Monday
local_time: 12:53:19
timezone: Europe/Warsaw
recallable: persona, user_identity, user_preferences, ...
nudge: think before you respond; recall when the topic shifts...
</metadata>
```

Agent wie **kiedy** jest, **co może odkryć**, i dostaje **nudge** zachęcający do aktywnego zachowania.

## Pragmatyzm technologiczny

Tailwind v3 celowo zamiast v4 — modele znacznie lepiej się nim posługują. Przy artefaktach nie podążamy za "najlepszymi standardami", lecz wybieramy narzędzia **optymalne dla capabilities modelu**. Biblioteki podłączane jako capability packs — model wybiera z manifestu, system resolwuje i wstrzykuje.

## Modele a generalizacja

Claude Opus 4.6 (silny w prompt engineering) **niemal w pełni zawodzi** przy generalizowaniu zadań na tym poziomie. Dobrze sprawdza się jako towarzysz **procesu myślowego** stojącego za kształtowaniem takich instrukcji. Rola człowieka pozostaje kluczowa.

Podstawa neurokognitywna: modele wykorzystują [Theory of Mind](https://arxiv.org/pdf/2505.00026) — obserwowaną od GPT-4, znacznie rozwiniętą w obecnych modelach. To ona umożliwia "czytanie między słowami", odczytywanie intencji i dopasowywanie odpowiedzi.

## Przykłady kodu — progresja i delta wiedzy

### 03_05_apps — MCP Apps

**Delta:** Serwer MCP wystawia interaktywny HTML jako resource + narzędzia operujące na stanie.

- **Dual-mode UI** — jeden HTML bundle działa standalone (REST) i embedded w MCP host (iframe). Detekcja: `window.parent === window`
- **Tool visibility scoping** — narzędzia `get_lists_state` / `save_lists_state` mają `visibility: ["app"]` — ukryte przed modelem, dostępne tylko dla UI. Model widzi tylko `manage_lists` otwierający interfejs
- **Structured content** — narzędzia zwracają `content` (czytelny dla modelu) + `structuredContent` (maszynowy, dla UI)
- **Single-file bundle** — `vite-plugin-singlefile` inlinuje JS/CSS do jednego HTML serwowanego jako MCP resource z MIME `text/html;profile=mcp-app`

### 03_05_artifacts — AI-generowane artefakty z live preview

**Delta:** LLM generuje dowolny HTML w sandboxie. Kluczowe: capability packs + deterministic editing + WebSocket state sync.

- **Capability pack system** — 11 pakiety (Preact+HTM, Chart.js, D3, Tailwind, Zod, Day.js, DOMPurify, PapaParse, XLSX, SheetJS, core helpers). Podwójny tryb: inline (self-contained artifact) vs external (`<script src>` z serwera z 24h cache)
- **4-fazowy pipeline artefaktów** — dane testowe → rozpoznanie (model decyduje o sposobie wizualizacji na podstawie treści danych) → generowanie (HTML w izolowanym iframe) → optymalizacja (search/replace zamiast regeneracji)
- **Deterministic editing bez LLM** — `edit_artifact` wykonuje search/replace przez regex. Model decyduje CO, system deterministycznie aplikuje. Szybsze, tańsze, przewidywalne niż regeneracja
- **extractJsonCandidates()** — 3 strategie parsowania outputu LLM: raw text → fenced code block → first `{` to last `}`. Pierwsza która parsuje wygrywa. Reusable pattern
- **CSP jako safety boundary** — każdy artefakt dostaje strict CSP: `default-src 'none'`, inline tylko z preview server

### 03_05_render — Guardrail rendering

**Delta:** LLM nie generuje HTML — generuje JSON spec z katalogu 19 komponentów. Server-side rendering deterministycznie zamienia spec w HTML.

- **Podwójna walidacja** — manifest w system prompt (prompt-side) + Zod schema + `normalizeSpec()` (validation-side). Model nie może wygenerować niczego spoza katalogu
- **JSON Pointer state binding** — props używają `{"$state": "/path"}` do deklaratywnego bindowania danych. Resolve'owane server-side przy renderowaniu
- **Edit-as-regenerate** — edycja nie jest incrementalna. Cały obecny dokument serializowany do prompt, model regeneruje z modyfikacjami. Trade-off: więcej tokenów, ale prostsza i poprawniejsza logika
- **Cycle detection + depth limit (40)** w tree rendererze — `renderElement()` trackuje visited IDs w Set, odrzuca cykle

### 03_05_awareness — Agent "świadomy"

**Delta:** Agent z metacognitive tools, temporal grounding, sub-agent delegation, workspace jako knowledge graph.

- **Tool `think`** — zero I/O, wymusza metarefleksję. Samoistnie zmienia zachowanie modelu jako cognitive scaffold
- **Tool `recall`** — deleguje do scout sub-agenta z własną sesją, system promptem i MCP file tools. Scout czyta workspace index → wybiórczo czyta pliki → zwraca kontekst. Sesja scouta persistuje across turns
- **Workspace jako cognitive memory** — episodic (E001), factual (F001, F002), procedural (P001), profile, system index. Inspirowane psychologią poznawczą
- **Template z YAML frontmatter** — parametryzacja agentów (model, narzędzia) w markdown: `---\nname: awareness\nmodel: gpt-5.2\ntools: [think, recall]\n---`
- **Backup/restore demo** — `demo.ts` kopiuje pliki workspace, seeduje dane demo, uruchamia scripted conversation, przywraca oryginały. Idempotentne, bezpieczne
- **Weather integration jako contextual enrichment** — scout fetchuje pogodę tylko gdy goal/message zawiera weather keywords. Lazy, nie eager
