# S04E04 — Projektowanie własnej bazy wiedzy dla AI

## Baza wiedzy jako "kod źródłowy" agenta

Poza kontekstem programowania agent nie ma dostępu do żadnego naturalnego źródła kontekstu. Jego użyteczność zależy wyłącznie od instrukcji systemowej, bieżącej interakcji i tego, co sam potrafi znaleźć. Baza wiedzy wypełnia tę lukę — staje się ekwiwalentem kodu źródłowego, tyle że dla domeny pozaprogramowej.

To przesunięcie perspektywy: zamiast pytać "jak budować bazę wiedzy?", pytamy "co agent musi wiedzieć, żeby być użyteczny w MOIM kontekście?". Odpowiedź jest indywidualna i wymaga iteracji — nie da się jej zaplanować z góry.

## Odkrywanie zakresu

Zamiast zakładać, że AI zautomatyzuje wszystko, warto przeskanować rzeczywiste otoczenie: regularnie odwiedzane domeny, powtarzalne aktywności, narzędzia, z których się korzysta. Lista okaże się krótsza, niż się wydaje. Następnie podzielić na: obszary do automatyzacji, obszary do kontekstowego wsparcia (agent czyta, nie działa), obszary wyłączone z AI.

Krytyczny wniosek: "najlepsze pomysły nie są powodem wykonanej pracy, lecz jej rezultatem". Trzeba budować, eksplorować i popełniać błędy, żeby dotrzeć do zastosowań, których planowanie nie wyprodukuje.

## Struktura: pięć sekcji

| Sekcja | Właściciel | Zawartość | Przykład |
|---------|-----------|-----------|---------|
| **Me** | Człowiek | Profil, wartości, preferencje, nawyki | Moje cele, mój system pracy |
| **World** | Współdzielone | Ludzie, miejsca, narzędzia, źródła | Opisy osób, konfiguracja narzędzi |
| **Craft** | Współdzielone | Projekty, eksperymenty, materiały | Notatki z projektów, szkice |
| **Ops** | Agenci | Procesy, dane z wykonywanych zadań | Research, raporty, logi |
| **System** | Człowiek | Agenci, szablony, reguły, metadata | Definicje agentów, templatki |

Podział jest funkcjonalny, nie techniczny. Kluczowe jest rozróżnienie: kto pisze (człowiek vs agent) i kto czyta (człowiek, agent, obaj).

## Markdown — zalety i granice

**Dlaczego markdown:** AI natywnie generuje i rozumie ten format. Pliki tekstowe = swobodna transformacja, wyszukiwanie, wersjonowanie. YAML frontmatter dodaje maszynowo-czytelny layer metadata (status, tagi, uprawnienia, przypisanie). Obrazy muszą być linkami zdalnymi — lokalne pliki są "niewidoczne" dla agenta.

**Kiedy coś innego:** praca wieloosobowa na jednym dokumencie w czasie rzeczywistym, złożone uprawnienia → Notion/Google Docs. Konwersja między formatami traci informacje — lepiej podjąć jedną decyzję per obszar niż konwertować.

## Notatki samowystarczalne

Zasada: **notatki piszemy tak, jakby czytelnik nie miał żadnego dodatkowego kontekstu.** To fundamentalna zmiana nawyku — z kodowania implikacji do ich jawnej ekspozycji.

Typowe luki zabijające skuteczność agenta:

- **Niejasne referencje** — nazwy projektów, imiona bez powiązania do konkretnych encji
- **Skrócone linki** — URL nie sugeruje zawartości, agent nie wie, czy warto kliknąć
- **Relatywne odsyłacze** — "ostatnia rozmowa", "poprzednia wersja" — agent nie ma chronologii
- **Powściągliwość w powtórzeniach** — agent wczytuje fragment dokumentu; jeśli powiązanie jest w innym fragmencie, go nie widzi
- **Nadpisywanie** — nowa wersja niszczy dostęp do starej; agent nie wie, która jest aktualna

Rozwiązanie: jawne linkowanie, pełne nazewnictwo, notatki indeksowe (MoC), zasady wersjonowania.

## Balans człowiek–AI w zarządzaniu bazą

Podział ról: **człowiek odpowiada za treść i zasady, AI za organizację.**

| Aktywność | Rola AI | Uzasadnienie |
|-----------|---------|-------------|
| Transformacja | Formatowanie, korekta, OCR | Źródło = człowiek, AI tylko przetwarza |
| Szablony | Egzekwowanie struktury | Człowiek definiuje formę, AI ją aplikuje |
| Organizacja | Sugestie lokalizacji | Agent wie co już istnieje, proponuje gdzie umieścić |
| Linkowanie | Łączenie powiązań | Skalowalne tylko z pomocą AI |
| Walidacja | Sprawdzanie spójności | Reguły zdefiniowane przez człowieka, egzekwowane przez AI |
| Indeksowanie (MoC) | Generowanie map treści | MoC = notatka-indeks dla obszaru treści |
| Audytowanie | Wykrywanie braków, archiwizacja szumu | CIągła weryfikacja jakości |

Granica: AI nie tworzy treści notatek w sekcjach ludzkich. Gdyby tak było, tracimy sens budowania bazy — gubimy się w treści, której nie napisaliśmy.

## Szablony notatek

Szablon zdejmuje z agenta decyzję o strukturze i lokalizacji. Agent nie wymyśla format — czyta szablon z `system/templates/` i go replikuje. Każdy szablon deklaruje sekcję docelową (Person → World/People, Idea → Craft/Ideas), więc lokalizacja jest deterministyczna.

Dzięki temu jedna prośba ("zapisz informacje o tej osobie") uruchamia: czytanie `index.md` → czytanie odpowiedniego szablonu → utworzenie notatki we właściwym miejscu. Koszt: kilkanaście wywołań LLM na jedną notatkę. Zysk: spójność struktury bez udziału człowieka.

## Workflow-y jako dane

Procesy agentowe definiuje się w `ops/` jako zestaw plików markdown — nie jako kod. Każdy workflow ma `_info.md` (metadata, źródła, tematy, przypisanie faz) + pliki fazowe (np. `01-research.md`, `02-assemble.md`, `03-deliver.md`).

Kluczowa decyzja projektowa: **instrukcje preskryptywne, nie goal-oriented.** Każda faza zawiera dokładną listę kroków + listę "Do NOT". Vague cele dają nieprzewidywalne rezultaty; tight playbook = przewidywalność.

Podział ról między agentami: orchestrator deleguje sekwencyjnie, jedna faza na raz. Każdy agent dostaje swój plik fazowy jako instrukcję.

## Start praktyczny

Nie trzeba budować całej struktury naraz. Wybrać jeden obszar lub jedną aktywność — najlepiej coś, co się podoba, niekoniecznie coś "użytecznego". Można zacząć bez kodu: podłączyć katalog markdown bezpośrednio do agenta (Claude Code), ukształtować pierwsze notatki i procesy, a dopiero potem myśleć o runtime, MCP i logice.

## Przykłady kodu z `04_04_system`

### 1. Agent jako plik markdown (`workspace/system/agents/alice.md`)

Agent = YAML frontmatter (model, narzędzia) + treść (system prompt). Zmiana zachowania agenta = edycja markdown, nie dotykanie kodu. Parser:

```javascript
const { data, content } = matter(rawMarkdown);
// data.model → "gpt-5.4", data.tools → ["files", "sum"]
// content → system prompt (treść notatki)
```

Czego uczy: separacja identity od runtime. Kod nie wie, kim jest agent — tylko ładuje markdown i wykonuje.

### 2. Agentic tool-calling loop (`src/agent.js`)

Pętla bounded: LLM → tool_calls → wykonanie → wyniki → z powrotem do LLM. Tool calls i wyniki dopinane bezpośrednio do conversation array (wymóg Responses API). `MAX_STEPS` zapobiega nieskończonej pętli. Kończy się, gdy response nie zawiera `function_call`.

Czego uczy: to jest skeleton każdego agenta z tool-calling. Wzorzec uniwersalny, niezależny od frameworka.

### 3. Rekurencyjna delegacja ze strukturalnym limitem (`src/tools/delegate.js`)

Agent może delegować do sub-agenta. Limit głębokości egzekwowany **strukturalnie** — przy `MAX_DEPTH` tool `delegate` jest fizycznie usuwany z listy dostępnych narzędzi. Agent nie może złamać reguły, bo jej nie widzi.

Czego uczy: constraint enforcement przez architekturę, nie przez prompt. Prompt instructions można zignorować; brak narzędzia — nie.

### 4. MCP — runtime discovery + namespace (`src/mcp.js`)

Serwery narzędziowe jako child processes przez stdio. Narzędzia odkrywane runtime: `client.listTools()`. Namespace collision prevention: `files__fs_read`, `web__search`. Scoped access: filesystem MCP rootowany w `./workspace` — agent nie może wyjść poza ten katalog.

Czego uczy: narzędzia są pluginami, nie częścią kodu. Dodanie nowego serwera = jedna linijka w `mcp.json`.

### 5. Workflow jako preskryptywne fazy (`workspace/ops/daily-news/`)

Trzy pliki fazowe, każdy z dokładnymi krokami i "Do NOT" listą. Metadane w `_info.md` z `depends_on` deklarującym zależności między fazami. Orchestrator ładuje `_info.md`, deleguje fazy sekwencyjnie — każda faza to osobny agent z własnym plikiem instrukcji.

Czego uczy: "prescriptive phase files beat goal-oriented ones" — konkretna lista kroków + lista zakazów daje przewidywalność, której ogólnikowe cele nie dają.

### 6. Dual-mode prompting (`workspace/system/agents/alice.md`)

Alice ma dwa tryby działania z osobną inicjalizacją: workflow mode (czyta tylko pliki procesu) i note-creation mode (ładuje index, szablony, reguły linkowania). Każdy tryb czyta inne pliki — unika niepotrzebnych odczytów i scope creep.

Czego uczy: jeden agent może obsługiwać wiele trybów, ale inicjalizacja musi być rozdzielona. Agent w trybie workflow nie powinien ładować szablonów notatek — to szum.
