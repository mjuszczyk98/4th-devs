# S01E03 — Projektowanie API dla efektywnej pracy z modelem

## Audyt API przed budową narzędzi

Przed projektowaniem narzędzi AI dla istniejącego API zweryfikuj:

- **Brakujące akcje** — czy API pozwala na wszystkie operacje potrzebne agentowi (CRUD)
- **Sposób adresowania zasobów** — identyfikator vs nazwa. Model musi wiedzieć, że "etykieta Priorytet" wymaga resolucji na ID
- **Niespójności nazewnictwa** — `content` vs `body` w różnych endpointach dezorientuje model
- **Odpowiedzi API** — samo `201 Created` bez zwrócenia obiektu jest niewystarczające dla modelu
- **Złożone relacje** — jedno zadanie użytkownika wymagające wielu wywołań API (np. promocja = stwórz + przypisz)
- **Polling, rate limit, paginacja** — obsługuj programistycznie po stronie kodu, nie zmuszaj agenta do retry/oczekiwania

Najlepsze podejście: wgraj SDK/dokumentację API do kontekstu agenta kodującego i wspólnie zaplanuj strukturę narzędzi.

## Redukcja liczby narzędzi przez grupowanie

Oficjalny Filesystem MCP: 13 narzędzi. Zoptymalizowana wersja: 4.

| Narzędzie | Zastępuje |
|---|---|
| `fs_read` (mode: file/list) | read_text_file, read_media_file, read_multiple_files, list_directory, list_directory_with_sizes |
| `fs_write` (operation: create/update) | write_file, edit_file |
| `fs_search` (match: name/content) | search_files, directory_tree, get_file_info |
| `fs_manage` (operation: mkdir/move/rename/delete) | create_directory, move_file, list_allowed_directories |

Cel nie jest minimalizacja narzędzi sama w sobie — to balans między liczbą schematów a zdolnością modelu do ich poprawnej obsługi. Parametr `mode`/`operation` konsoliduje powiązane akcje; wspólna struktura parametrów (część ignorowana w zależności od trybu) upraszcza schemat.

## Projektowanie interfejsu narzędzi

Trzy perspektywy kształtują schemat:

1. **API** — fundamentalne możliwości i ograniczenia
2. **LLM** — przejrzystość schematów, ryzyko halucynacji, limity kontekstu
3. **Use case** — co użytkownik faktycznie potrzebuje

Zasady implementacji:

- Proste nazwy właściwości, minimalna ilość danych
- Opcja `details: true/false` zamiast osobnych endpointów verbose/compact
- Automatyczna resolucja (np. nazwa pliku → pełna ścieżka) gdy model poda niepełne dane
- Ograniczenia uprawnień (np. `FS_ROOT`) egzekwowane programistycznie, nie przez prompt

### Zabezpieczenia dla akcji destrukcyjnych

- **Checksum** na `fs_write` — agent nie nadpisze pliku zmienionego w międzyczasie
- **dryRun** — podgląd wyniku operacji bez jej wykonania
- **Historia zmian** — przywracanie błędnych modyfikacji bez angażowania modelu
- **Usuwanie** ograniczone do pojedynczych plików i pustych katalogów

## Dynamiczne odpowiedzi (hints)

Narzędzia AI powinny zwracać wskazówki kontekstowe — nie tylko wynik operacji. Kategorie:

- **Błąd + recovery**: "Treść pliku została zaktualizowana. Przeczytaj go ponownie."
- **Status zasobu**: "Dokument istnieje, ale jest chroniony przed zapisem."
- **Sugestie dalszych kroków**: "Znaleziono 3 dokumenty. Przed edycją wczytaj ich treść."
- **Dostępne opcje**: "Błędny rodzaj etykiety. Dostępne: X, Y, Z."
- **Korekty automatyczne**: "Żądano linii 48-70, dokument ma 59 linii. Wczytano 48-59."

Te hinty mają sens nawet przy zaawansowanych modelach — niosą informację kontekstową (np. ograniczenia uprawnień), której model nie może wydedukować samodzielnie.

## Model Context Protocol — architektura

### Trzy role

- **Host** — aplikacja tworzącą połączenia (Claude Desktop, Cursor, ale też backend bez UI)
- **Client** — zarządza połączeniem z serwerem, część hosta
- **Server** — dostarcza narzędzia, zasoby, prompty

### Dwa transporty

| | STDIO | Streamable HTTP |
|---|---|---|
| Mechanizm | Subprocess, stdin/stdout | HTTP endpoint |
| Użycie | Lokalne narzędzia, desktop apps | Zdalne serwery, wielu użytkowników |
| Sesje | Nowy proces per połączenie | Wielosesyjny |
| Auth | Klucze API w env | OAuth 2.1 |
| Deployment | Lokalnie | VPS, Cloudflare Workers |

STDIO dla interakcji z lokalnym systemem (pliki, ffmpeg). Streamable HTTP jako domyślny wybór przy budowie serwerów.

### Pięć capability

| Capability | Kierunek | Opis |
|---|---|---|
| **Tools** | klient → serwer | Akcje wywoływane przez LLM |
| **Resources** | klient → serwer | Dane read-only (statyczne lub dynamiczne) |
| **Prompts** | klient → serwer | Szablony wiadomości z parametrami, aktywowane przez użytkownika |
| **Sampling** | serwer → klient | Serwer prosi klienta o wywołanie LLM (serwer nie potrzebuje własnych kluczy API) |
| **Elicitation** | serwer → klient | Serwer prosi klienta o input użytkownika (formularz, potwierdzenie) |

Sampling i Elicitation odwracają kierunek komunikacji — serwer inicjuje request do klienta. Klient deklaruje obsługiwane capability przy połączeniu.

## MCP nie zastępuje natywnych narzędzi

MCP tools i natywne function-calling tools to ten sam format z perspektywy modelu. Można je łączyć w jedną listę. MCP definiuje jedynie sposób dostarczania i odkrywania narzędzi w spójnej formie — raz stworzony serwer podłącza się do dowolnego hosta.

## Routing narzędzi z wielu serwerów

Przy wielu serwerach MCP nazwy mogą kolidować. Rozwiązanie: host prefikuje nazwy narzędzi nazwą serwera.

```
files__fs_read, uploadthing__upload_files
```

Routing: split po `__`, pierwsza część = serwer docelowy, reszta = oryginalna nazwa narzędzia.

## Bezpieczeństwo i prywatność

Wewnętrzne narzędzia: kontrolujesz zasoby, akcje, użytkowników, przepływ danych.

Publiczne serwery MCP: nie znasz hosta, pozostałych narzędzi, procesów agenta, danych użytkownika. Stosuj programistyczne ograniczenia — blokady dostępu, limity zapytań, walidacje, anonimizację. Niektóre zasoby/akcje nie powinny być udostępniane agentom AI.

Prompt injection pozostaje otwartym problemem bez uniwersalnego rozwiązania.

## Generowanie serwerów MCP z szablonu

Proces z agentem AI:

1. Pobierz szablon (np. Streamable MCP Template)
2. Stwórz `API.md` z dokumentacją docelowego API
3. Agent czyta README.md + manual.md szablonu
4. Agent proponuje listę narzędzi → zredukuj/pogrupuj wg zasad z lekcji
5. Zaprojektuj input/output schematów
6. Implementacja → weryfikacja → iteracja

Mały model lokalny (Qwen, GPT-OSS 20B) jako smoke test — jeśli poprawnie obsługuje narzędzia, są dobrze zaprojektowane.

## Przykłady kodu — progresja

### `01_03_mcp_native` — fundament

Pokazuje: MCP tools i natywne JS tools na jednej liście. Model nie widzi różnicy.

- **InMemoryTransport** — serwer i klient w jednym procesie (najprostsza forma MCP)
- **Unified handler map** — `{ execute, label }` dla obu źródeł narzędzi
- Konwersja schematów MCP → OpenAI function-calling format (`mcpToolsToOpenAI`)
- Agentic loop: chat → extract tool calls → execute parallel → append results → repeat

### `01_03_mcp_core` — delta: pełne capability MCP

Nowe względem `mcp_native`:

- **StdioTransport** — serwer jako subprocess (produkcyjny wzorzec)
- **Resources** — statyczne (`config://project`) i dynamiczne (`data://stats`, zmienia się per request)
- **Prompts** — szablony z parametrami i Zod schema (`code-review` z `language`, `focus`)
- **Elicitation** — serwer wysyła JSON Schema formularza, klient zwraca wypełnione wartości. Demo: auto-akceptacja z inferred defaults
- **Sampling** — serwer prosi klienta o LLM completion przez `extra.sendRequest()`. Klient konwertuje format MCP → Responses API. Separacja: serwer = logika, klient = infrastruktura AI
- Narzędzie `summarize_with_confirmation` łączy elicitation + sampling w jednym flow

### `01_03_mcp_translator` — delta: realny use case na backendzie

Nowe względem `mcp_core`:

- **Zewnętrzny serwer MCP** (`files-mcp`) zamiast własnego — klient łączy się z gotowym serwerem narzędzi
- **Konfiguracja z `mcp.json`** — deklaratywna definicja serwerów (command, args, env)
- **File watcher** — polling katalogu co 5s przez MCP tools, detekcja nowych plików
- **HTTP API** — serwer HTTP jako alternatywny interfejs do agenta (`/api/translate`, `/api/chat`)
- **Token tracking** — kumulacja `input_tokens`, `cached_tokens`, `output_tokens` z Responses API
- **Instrukcje z chunking strategy** — system prompt definiuje schemat pracy (scan → plan → translate → verify), ale agent decyduje o wykonaniu
- `strict: false` na tool definitions — zewnętrzne schematy MCP mogą nie spełniać wymagań strict mode

### `01_03_upload_mcp` — delta: multi-server, multi-transport

Nowe względem `mcp_translator`:

- **Dwa serwery MCP jednocześnie**: `files` (stdio) + `uploadthing` (StreamableHTTP)
- **Prefixowanie nazw narzędzi** — `server__tool` eliminuje kolizje, routing po `split("__")`
- **`{{file:path}}` resolver** — model używa placeholdera zamiast czytać/kodować pliki. Przed wywołaniem MCP tool, resolver rekurencyjnie zamienia placeholdery na base64 z dysku
- **Fallback na raw request** — gdy `listTools()` failuje na niestandardowym `outputSchema`, odpytuje bez Zod walidacji
- **Walidacja konfiguracji HTTP** — `ConfigurationError` z czytelnymi instrukcjami naprawczymi
