# S01E01 — Programowanie interakcji z modelem językowym

## Fundamenty LLM

- Model to funkcja: tekst wejściowy → prawdopodobieństwa kolejnych tokenów → wybór jednego (z losowością) → powtórz
- **Autoregresja**: każdy nowy token zależy od wszystkich poprzednich — dane wejściowe, historia wygenerowana przez AI i instrukcja systemowa wpływają na wynik
- **Niedeterminizm**: te same dane wejściowe mogą dać różny wynik — programowanie generatywne = deterministyczny kod + niedeterministyczny LLM
- **Bezstanowość API**: każde żądanie musi zawierać komplet danych. Historia rozmowy to odpowiedzialność klienta, nie API
- **Okno kontekstowe**: limit tokenów obejmuje wszystko — prompt systemowy, historię, input i output

## Sterowanie modelem przez kod

Kluczowa zasada: **zarządzanie kontekstem = sterowanie zachowaniem modelu**. Możliwości:
- kontrola treści zapytania
- łączenie wyników wielu zapytań
- budowanie wieloetapowych pipeline'ów (wynik jednego LLM jako input kolejnego)

## Structured Outputs

Dwa sposoby uzyskania JSON: **Structured Outputs** (JSON Schema) i **Function Calling**.

Przy projektowaniu schematu:
- `strict: true` + `additionalProperties: false` = gwarancja struktury
- nazwy i opisy pól są generowane przez model — muszą być precyzyjne i zwięzłe
- **uwzględniaj wartości neutralne** (`null`, `"unknown"`, `"mixed"`) — zmniejsza halucynacje
- **kolejność pól ma znaczenie** — wcześniejsze tokeny wpływają na późniejsze (reasoning przed sentiment przed confidence)

## Renderowanie odpowiedzi

Interakcja z LLM to nie pytanie-odpowiedź, to **seria zdarzeń** (tokeny rozumowania, tool calls, tekst, błędy, obrazy). Implikacje:
- nie przesyłaj surowego tekstu LLM do UI wprost
- używaj **semantycznych zdarzeń** z metadanymi (ID, typ, dodatkowe właściwości)
- rozbudowa systemu = dodanie właściwości do zdarzenia, nie migracja całej architektury

## Strategie doboru modeli

1. **Jeden model** — najprostsze
2. **Główny + alternatywny** — duży/skuteczny vs mały/szybki/tani
3. **Główny + specjalistyczne** — różne modele do różnych formatów/zadań
4. **Zespół małych modeli** — dekompozycja + głosowanie, rzadkie w praktyce

Jedyną miarodajną metodą wyboru jest **testowanie na własnych zadaniach**. Unikaj vendor lock-in — abstrahuj providera od logiki.

## Natywne funkcje API (OpenAI / Anthropic / Gemini)

- **Multimodalność** — Gemini Interactions API, OpenAI Responses API
- **Natywne narzędzia** — web search, file search, code execution: łatwe, ale słabo konfigurowalne
- **Cache** — kluczowy dla skali; obniża koszty i latencję przy powtarzających się promptach
- **Rate limit** — problem na nowych kontach; rozwiązanie: OpenRouter

## Organizacja promptów

| Podejście | Kiedy |
|---|---|
| Inline | Proste, rzadko zmieniane |
| Oddzielne pliki z kompozycją | Najczęstsze, elastyczne |
| Zewnętrzny system (Langfuse) | Wersjonowanie, monitoring poza kodem |
| **Pliki Markdown z frontmatter** | **Preferowane** — dostępne dla agentów jako narzędzie, dynamicznie modyfikowalne |

## Optymalizacja instrukcji

**Generalizowanie generalizacji** — zamiast dodawać regułę dla każdego błędu, zbuduj zgeneralizowany proces myślowy obejmujący całą kategorię błędów. Przykład: zamiast "użyj narzędzia X gdy..." → "zastanów się głośno nad wyborem narzędzia, określ pewność, zapytaj gdy niepewny".

Elementy dobrego promptu systemowego:
- tożsamość (kieruje uwagę, nie nadaje magicznych umiejętności)
- limity i ograniczenia (zmniejszają ryzyko, nie gwarantują bezpieczeństwa)
- styl i format
- adaptacja (co robić gdy brak danych/narzędzi)
- kalibracja (konkretne zasady formatowania)

## In-context learning (few-shot / many-shot)

Model uczy się wzorców z przykładów w kontekście. Sekcja `<examples>` w promptie > opis zasad słowami. Obecnie trend: zamiast statycznych przykładów w prompcie — dynamiczne wczytywanie przez narzędzia agenta.

## Struktury danych

**Chatbot**: tabele `conversations` + `messages` — wystarczy dla prostych interakcji.

**System wieloagentowy**:
- `sessions` — sesja + agent nadrzędny
- `agents` — instancje agentów, statusy, powiązania z sesją i rodzicem
- `items` — wiadomości, tool calls, załączniki

## Modele open-source

- **Formaty**: GGUF (llama.cpp, Windows/macOS/Linux), MLX (Apple Silicon)
- **Kwantyzacja**: Q4/Q5 = dobry balans jakość/rozmiar; Q2 = szybko/słabo
- **Wymagania**: model powinien zmieścić się w VRAM/Unified Memory; od ~32GB RAM wzwyż
- Praktyka: większość firm pozostaje przy komercyjnych API

## Przykłady kodu

### `01_01_interaction` — bezstanowość i zarządzanie historią

Ręczne budowanie kontekstu przy kolejnym zapytaniu:
```js
const secondQuestionContext = [
  { type: "message", role: "user", content: firstQuestion },
  { type: "message", role: "assistant", content: firstAnswer.text }
];
await chat(secondQuestion, secondQuestionContext);
```
Model nie pamięta — to Twój kod utrzymuje historię. `reasoningTokens` w odpowiedzi = koszt chain-of-thought.

### `01_01_structured` — JSON Schema i ekstrakcja danych

Minimalna demonstracja Structured Outputs: nieustrukturyzowany tekst → gwarantowany JSON.
```js
text: {
  format: {
    type: "json_schema",
    strict: true,
    schema: {
      properties: {
        name: { type: ["string", "null"], description: "Full name. Use null if not mentioned." },
        // ...
      },
      additionalProperties: false
    }
  }
}
```
Nowe względem poprzedniego: `text.format`, `strict: true`, `JSON.parse(outputText)`, wartości `null` dla nieznanych pól.

### `01_01_grounding` — wieloetapowy pipeline z web searchem

4-etapowy pipeline: **extract → dedupe → search → generate HTML**. Każdy etap = osobny LLM call ze swoim promptem i schematem.

Nowe względem poprzednich:
- **Równoległość** — `Promise.all()` w batchach po 5 akapitów jednocześnie
- **Caching przez hashe** — każdy etap zapisuje wynik do JSON i sprawdza `sourceHash`/`conceptsHash`/`dedupeHash`; zmiana upstream automatycznie unieważnia downstream
- **Natywny web search** — `tools: [{ type: "web_search" }]` (OpenAI) lub suffix `:online` w nazwie modelu (OpenRouter)
- **Prompty jako moduły** — `buildExtractPrompt({ paragraph, paragraphType, index, total })` — kompozycja z dynamicznymi sekcjami
- **Retry z exponential backoff** — `api.js` obsługuje 429/500/503 automatycznie
- **Abstrakcja providera** — ta sama logika działa dla OpenAI i OpenRouter dzięki warunkowemu budowaniu requestu

Wzorzec: fragment dokumentu jako jednostka przetwarzania nawet gdy cały dokument zmieściłby się w kontekście — **skupienie uwagi modelu** na mniejszym zakresie = wyższa skuteczność + możliwość użycia tańszego modelu.
