# S03E01 — Obserwowanie i ewaluacja

## Evals ≠ testy E2E

Ewaluacja LLM weryfikuje **stopień dopasowania** rezultatu do założeń, nie deterministyczną poprawność. 100% pass rate nie jest celem — celem jest wystarczający poziom na ustalonych wskaźnikach. Evals **nie gwarantują** poprawnego działania — aplikacja nadal może zachowywać się niezgodnie z oczekiwaniami.

Częściowa zgodność jest normą. "Mam na imię Adam" vs "Jestem Adam" — ten sam sens, różne cechy wypowiedzi. Ocena może być stopniowana, nie binarna.

Trzy poziomy oceny: **programistyczna** (regex, JSON schema), **LLM-as-judge** (model ocenia model), **człowiek** (manualna weryfikacja). Różne typy anomalii wymagają różnych poziomów — nie wszystko potrzebuje LLM.

Ewaluacja jest żmudna, wymaga pracy manualnej. Czasem to nie system, lecz **testy wymagają optymalizacji** — wynik oceniony gorzej może w praktyce być lepszy.

## Observability — fundament przed evals

Monitorowanie musi być wdrożone **przed** ewaluacją. Dwie przyczyny: treść logowanych zdarzeń jest materiałem do evals, a bez widoczności zachowania agenta nie wiadomo co testować.

Zastosowania observability: **analiza błędów** (agent przeszukuje niewłaściwy obszar z powodu pokrywających się opisów — problem niewidoczny w kodzie), **estymacja kosztów** (tokeny z instrukcji, narzędzi, kontekstu konwersacji, dynamicznych danych — przybliżone estymacje bywają niezwykle trudne), **wydajność** (niektórzy użytkownicy osiągają limity szybciej bez intencjonalnego działania — błąd po naszej stronie), **obsługa klienta** (bezpośredni wgląd w aktywności zgłaszającego problem).

### Hierarchia zdarzeń

| Typ | Zakres | Odpowiednik Langfuse |
|-----|--------|---------------------|
| Session | Wątek czatu / zadanie agenta | Session |
| Trace | Pojedyncza interakcja użytkownika | Trace |
| Span | Czas trwania akcji (np. gromadzenie kontekstu) | Span |
| Generation | Zapytanie do LLM z pełnym kontekstem i ustawieniami | Generation |
| Tool | Wywołanie narzędzia (input/output) | Tool |
| Event | Zdarzenie aplikacji niezwiązane z LLM | Event |

Kluczowe: nie logujemy "co się stało", ale **w jakim kontekście** — userId, sessionId, agentId, metadata (interfejs, wersja aplikacji, środowisko, specyficzne dane jak licencja, uprawnienia, preferencje, typ zapytania np. CRON).

Span (czas trwania akcji) i Event (zdarzenie niezwiązane z LLM) to nasza inicjatywa — możemy z nich całkowicie zrezygnować. Po **minimalnej** integracji szybko ujawniają się brakujące i nadmierne logi.

### Architektura integracji

Centralny punkt instrumentacji — adapter factory z dekoratorem. Agent i adapter nie wiedzą o tracingu, tracing jest nakładany transparentnie. Wymaga zaplanowania na etapie architektury, nie retrofitu.

Graceful degradation — system działa identycznie bez poświadczeń platformy observability. Każda funkcja tracingowa sprawdza aktywność przed działaniem.

## Debugowanie agentów — nie kod

Zmiana instrukcji lub opisów narzędzi może naprawić problem w jednym przypadku, ale psuje w innych. **Poprawa nie jest punktem końcowym** — trzeba weryfikować wpływ na resztę systemu.

Platformy (Langfuse, Confident AI) oferują **Playground** do re-execucji zapytań z modyfikowanymi parametrami i porównywania modeli. Wymaga przesyłania instrukcji systemowej, historii i listy narzędzi — nie tylko prompt-użytkownika.

Ciekawostka: pytanie modelu o **uzasadnienie** decyzji (np. wyboru narzędzia) często zawiera wskazówki pomocne w diagnozie, choć stoi w kontrze z badaniami nad interpretacją wnętrza modeli.

## Wersjonowanie instrukcji systemowych

Prompty w kodzie = automatyczna kontrola wersji przez Git, ale to niewystarczające — potrzebne statystyki, historia uruchomień, powiązanie z evals.

**Jednostronna synchronizacja** (aplikacja → Langfuse): prompty zostają w kodzie, wersjonowane przez content-hash. Tracimy łatwe przełączanie wersji z UI, ale zachowujemy kompletność statystyk i spójność z kodem.

## Ewaluacja — struktura

Eval składa się z trzech elementów:

- **Zadanie** — dane wejściowe + oczekiwane dane wyjściowe. Od pojedynczego promptu po pełną interakcję
- **Dane** — przykłady syntetyczne i/lub z produkcji. Syntetyczne = dobry punkt startowy
- **Ocena** — wynik 0–1, deterministyczny (kod) lub generowany przez LLM

### Offline vs Online

**Offline eval** — przed publikacją, w CI/CD lub dev. Weryfikuje skuteczność promptów, agentów, RAG na zdefiniowanych datasetach.

**Online eval** — w trakcie działania aplikacji. Interakcje oceniane automatycznie pod kątem naruszeń, których nie wyłapały wbudowane mechanizmy.

## Projektowanie datasetów testowych

Trzy zasady: **pokrycie** (pozytywne i negatywne scenariusze), **różnorodność** (szeroki obraz warunków działania), **balans** (nierozstrzelona dystrybucja uwagi między narzędzia/scenariusze).

Odpowiedzi narzędzi szybko stają się większością promptu agenta w interakcji — Braintrust zwraca uwagę, by uwzględniać je w evals, nie tylko prompt wejściowy.

Datasety mają charakter iteracyjny. LLM pomocny przy generowaniu scenariuszy i weryfikacji poprawności, ale człowiek weryfikuje rezultaty.

## Kryteria oceny

Deterministyczne: contains, is-json, equals, regex, programistyczne (JS/Python). Oparte o model: llm-rubric (ocena wg kryteriów w języku naturalnym), conversation-relevance, context-recall.

Obszary aplikacji kryteriów:

| Obszar | Co weryfikuje | Typowa metoda |
|--------|--------------|---------------|
| Skuteczność promptów | Jakość instrukcji, umiejętności agenta | LLM-as-judge |
| Wybór narzędzi | Czy agent sięga po właściwe narzędzia | Deterministyczna |
| Obsługa narzędzi | Poprawność użycia, liczba powtórzeń/błędów | Deterministyczna + LLM |
| Satysfakcja użytkownika | Aktywny feedback (łapki) | Człowiek |
| Naruszenia | Dane wejściowe i odpowiedzi modelu | Deterministyczna + LLM |

## Koszty i optymalizacja

Koszty ewaluacji = koszty API. LLM-as-judge generuje tokeny na każdą ocenę. Deterministyczne asercje (regex, porównania) są darmowe i często wystarczające.

Strategia podziału pracy: dane strukturalne/numeryczne → **kod** (darmowe, deterministyczne), język naturalny/subiektywne oceny → **LLM** (ale zoptymalizowane: batch, deduplikacja identycznych inputów, minifikacja outputu).

Promptfoo płaci więcej za output niż input — więc wysyłaj dużo danych (input), ale wymagaj minimalnej odpowiedzi (output).

## Anonimizacja danych

Obserwowanie zachowań agentów = przetwarzanie danych użytkowników. Nawet przy self-hostingu — wymaga anonimizacji: nazwy własne, adresy, dane kontaktowe, dane osobowe. Dane te są istotnym źródłem do evals, więc nie można ich pominąć.

## Decyzja strategiczna — czy evals w ogóle?

Evals nie gwarantują poprawności, kosztują czas i tokeny. Na rynku skrajne opinie: "evals are the real moat" (YCombinator) vs "no evals" (twórca Claude Code). Analogia do testów E2E — część zespołów uważa za krytyczne, inni całkowicie opcjonalne.

Decyzja zależy od **tempa zmian** i **priorytetów**. Na wczesnych etapach rozwiązania zmieniają się zbyt szybko by uzasadnić inwestycję w evals. Przy produkcie gdzie stabilność lub efektywność kosztowa są priorytetem — trudno wyobrazić rozwój bez.

Bezwzględnie warto uwzględnić observability i evals na etapie planowania architektury, nawet jeśli same evals wprowadzisz później.

## Guardrails

Uzupełnienie observability i evals — moderacja, filtrowanie, blokowanie niepożądanych zapytań. Niezależna warstwa od ewaluacji, ale często korzystająca z tych samych sygnałów.

## Narzędzia

Langfuse i Promptfoo — polecane. Confident AI, Braintrust, Grafana — alternatywy. Proste wewnętrzne skrypty wystarczają gdy potrzeby są ograniczone. Eksperymenty konfigurowalne z poziomu UI i API — można projektować w kodzie i tam analizować.

Integracja z platformami może być zrealizowana przez AI, ale **detale zależą od nas**: kontekst specyficzny dla projektu (licencje, uprawnienia, preferencje, typ zapytania) i metadata (interfejs, wersja, środowisko). Struktura obserwacji też — Span i Event są opcjonalne.

AI odnajdzie się też w **analizie wyników** — w Langfuse można wyeksportować JSON interakcji lub eksperymentu i omówić z modelem.

## Przykłady kodu

### `03_01_observability` — Agent z Langfuse tracing

Minimalny agent HTTP (Hono) z dwoma narzędziami (`get_current_time`, `sum_numbers`), pełną instrumentacją Langfuse i wersjonowaniem promptów.

**Delta względem S01E05 (`01_05_agent`):** S01E05 pokazywało Langfuse jako subscriber eventów w architekturze event-driven. Tutaj Langfuse jest zintegrowany bezpośrednio przez OpenTelemetry SDK + dekorator adaptera.

**Dekorator `withGenerationTracing`** — opakowuje `Adapter.complete()` w generation span. Agent i adapter nie wiedzą o tracingu. Fabryka adapterów aplikuje dekorator warunkowo (graceful degradation).

**OTel + Langfuse** — OpenTelemetry `NodeSDK` z `LangfuseSpanProcessor`. Tracing inicializowany na starcie, grzecznie degraduje bez poświadczeń.

**ReAct agent loop** — pętla do `MAX_TURNS=8`, wyjście gdy model nie produkuje tool calls (ma końcową odpowiedź). Usage akumulowany przez wszystkie tury. Hierarchia spanów: trace (chat-request) → agent (alice) → generation#N / tool#N.

**AsyncLocalStorage** — propaguje kontekst agenta (name, turnNumber, toolIndex) przez async granice bez parametrów. Pozwala generować hierarchiczne span names (`alice/generation#2`) z poziomu funkcji tracingowych, nie logiki biznesowej.

**Prompt sync** — content-hash w `.langfuse-prompt-state.json`. Push do Langfuse tylko gdy hash się zmieni. Jednostronna synchronizacja: kod → platforma.

**Result type** — `Result<T, E>` z discriminated union zamiast wyjątków. `if (result.ok)` daje type narrowing. Mapowanie błędów OpenAI SDK na typed `CompletionError` z kodami (`RATE_LIMITED`, `AUTHENTICATION_ERROR`, itd.).

### `03_01_evals` — Ewaluacja agenta

Tożsama struktura `src/` + katalog `experiments/` z frameworkiem ewaluacyjnym. Migracja z Chat Completions na Responses API.

**Delta względem `03_01_observability`:** dodaje eval framework — dataset management, experiment runner, deterministyczne scorers.

**Dataset CRUD** — `ensureDataset()` (get-or-create) + `syncDatasetItems()` (upsert by ID). Idempotentne — bezpieczne do re-run. Dane z plików `.synthetic.json`.

**Tool-use eval** — 6 testów oceniających 4 wymiary: decyzja (czy użyć narzędzia), wymagane (czy wszystkie), zakazane (czy żadne niechciane), call count (czy w zakresie). Całkowicie deterministyczna — extractToolNames z sesji, porównanie z oczekiwaniami. Zero LLM calls na ocenę.

**Response correctness eval** — 3 typy asercji: `exact_number` (ekstrakcja liczb z tolerancją 0.01), `contains_iso_timestamp` (regex), `relevance` (słowa kluczowe + greeting detection). Deterministyczna.

**Safety gate** — `confirmExperiment()` wymaga interaktywnego "yes" przed API calls. Koszty ewaluacji = koszty tokenów na wykonanie zadań.

**Run-level evaluators** — `createAvgScoreEvaluator` agreguje per-item scores w średnią dla całego runu. Langfuse `dataset.runExperiment()` wspiera zarówno per-case jak i run-level scoring.
