# S05E02 — Zestaw narzędzi

## Markdown UI — fundament interfejsu agenta

Rendering streamowanego Markdown w czasie rzeczywistym to **pięć warstw**:

1. **remend** — naprawia niekompletną składnię (niezamknięte code fences, partial emphasis) zanim cokolwiek trafi do parsera
2. **marked (Lexer)** — tokenizuje narastający tekst na niezależne bloki (akapit, nagłówek, lista). Bez tego — pełny re-render co token
3. **markdown-it** — zamienia tokeny w HTML
4. **highlight.js** — kolorowanie bloków kodu
5. **DOMPurify** — usuwa złośliwy HTML wygenerowany przez LLM. Niewyłączny element pipeline'u

Kolejność jest istotna: `remend → marked → markdown-it → highlight.js → DOMPurify`. Pominięcie którejś warstwy daje konkretne awarie — bez remend pękają code blocki mid-stream, bez DOMPurify masz XSS.

**Streamdown** (Vercel) — nowsza, potencjalna alternatywa całego pipeline'u. Nieprzetestowana produkcyjnie.

## Streaming — event sourcing na froncie

SSE eventy są źródłem prawdy. Bloki UI (text, thinking, tool_call, artifact, error) to **projekcja** — materializowana dwiema ścieżkami:

- **Pełna rekonstrukcja** z historii — przy ładowaniu konwersacji
- **Przyrostowa** `applyEvent` — O(1) na event podczas streamingu

Batchowanie przez `requestAnimationFrame` — bufersz eventy, flush raz na klatkę. Bez tego: render thrashing przy szybkim strumieniu tokenów.

Inkrementalny rendering dzieli markdown na:
- **Committed segments** — cachowane, nie przebudowywane
- **Live tail** — przebudowywany co nowy token

## Wirtualizacja długich konwersacji

Chunkowa (12 wiadomości na chunk), nie per-element. ResizeObserver mierzy wysokość, `scrollTop` korygowany gdy chunk powyżej viewportu zmienia wymiary (scroll anchoring).

Pin-to-bottom z histerezą — dwa progi: release (np. 80px od dna) i reacquire (4px). Zapobiega oscylacji.

CSS triki uzupełniające wirtualizację JS:
- `content-visibility: auto` — przeglądarka pomija rendering elementów poza viewportem
- `overscroll-behavior: contain` — blokuje scroll chaining do rodzica
- `scrollbar-gutter: stable` — eliminuje layout shift przy pojawianiu się scrollbar

## Typewriter i UX streamingu

Animacja pisania z trzema prędkościami (off/fast/normal/slow). Kluczowy **gating** — kolejny blok UI pojawia się dopiero gdy poprzedni skończy animację. Bez tego interfejs "wyskakuje" zamiast płynnie odkrywać treść.

Niuans z zaznaczeniem tekstu — jeśli użytkownik zaznaczył tekst w "live tail", aktualizacja render jest odkładana (`pendingRenderState`) by nie zepsuć selekcji.

## Voice agents — dwie architektury

| | STT → LLM → TTS | Realtime (Gemini Live) |
|---|---|---|
| **Modele** | 3 osobne | 1 natywny |
| **Latencja** | Wyższa (pipeline) | Najniższa |
| **Modalności** | Tylko tekst + audio osobno | Audio/tekst/obraz/wideo jednocześnie |
| **Koszt** | Niższy | Wyższy |
| **Kiedy** | Proste interakcje, polecenia | Zaawansowane interakcje wielomodalne |

Dla prostych zastosowań Realtime jest overkill — koszt i złożoność bez proporcjonalnej korzyści.

### LiveKit Agents — wzorzec prewarm/entry

- **prewarm** — ładuje ciężkie zasoby raz: Silero VAD (neural network do detekcji końca mowy), połączenia MCP, modele
- **entry** — uruchamiana per sesja użytkownika, korzysta z załadowanych zasobów (warm start)

WebRTC jako transport — token JWT krótkożyciowy (10 min) z grantami join/publish/subscribe.

Browser blokuje autoplay audio — `room.startAudio()` musi być wywołane w click handlerze.

### Custom TTS — pipeline konwersji

ElevenLabs zwraca MP3, LiveKit wymaga PCM. Wzorzec: WebSocket → ffmpeg stdin → PCM stdout → AudioByteStream. Klasyczny process pipeline w Node.js.

Multi-provider auto-detekcja: Gemini (jeśli klucz) > OpenAI+ElevenLabs > OpenAI fallback. Wybór determinowany dostępnością kluczy, nie konfiguracją.

## Wyszukiwanie — drabina decyzyjna

Od najprostszego do najbardziej zaawansowanego. Każdy poziom usprawiedliwiony konkretną potrzebą:

1. **Brak wyszukiwania** — dokumenty w kontekście konwersacji. Prawdopodobna mechanika pamięci ChatGPT: szybkość > skuteczność
2. **Pliki + grep/ripgrep** — agent nawiguje po FS. Wystarczające dopóki treści są w jednym języku i formacie
3. **Hybrydowe (FTS + semantyczne)** — pełnotekstowe jako baza, semantyczne jako rozszerzenie. **Same bazy wektorowe — nierekomendowane**
4. **Grafy** — mapowanie relacji + eksploracja. Najwyższa skuteczność, najwyższy koszt

Claude Code = tylko grep. Cursor = grep + semantyczne. Nie ma uniwersalnego podejścia.

**Rozszerzenia vs dedykowane bazy:**

- SQLite fts5 + sqlite-vec lub PostgreSQL — wystarczają na większość projektów
- Qdrant, Chroma — dopiero przy skali, wydajności, zaawansowanym filtrowaniu
- Algolia, Elasticsearch — gdy potrzebujesz pełnoprawnego silnika wyszukiwania

Rozszerzenia i dedykowane rozwiązania mają niemal identyczny proces, ale drugie to nieporównywalnie większa złożoność architektury. Musisz mieć realny powód.

## Budowanie własnych komponentów

Powtarzalne schematy w pracy z AI, które warto zamknąć w reużywalne formy:

- **Prompty** — biblioteka instrukcji przypisanych do skrótów/makr/skills. Nawet kilka шаблонów zwraca inwestycję czasu
- **CLI/MCP** — integracje z codziennymi narzędziami. Zamknięte w standardową formę = łatwe przenoszenie między agentami i udostępnianie. Zacznij od jednej integracji
- **Zarządzanie plikami** — FS, Notion, Google Drive. Agent bez dostępu do dokumentów jest drastycznie ograniczony
- **Dostęp do chmury** — upload/download obrazów i plików. Uploadthing lub własny serwer
- **Sandbox** — izolowane środowiska dla ryzykownych akcji. Alternatywa: dedykowany fizyczny komputer
- **Generowanie dokumentów** — szablony auto-uzupełniane przez LLM (tabele, wizualizacje, kreacje)
- **Przetwarzanie media** — obraz/audio/wideo. Integracje nie muszą być skomplikowane (np. Audiopen)
- **Interfejs** — własny chat UI jako panel zarządzania. Sensowny gdy potrzebna wysoka personalizacja

## Przykłady kodu

### 05_02_ui — Chat UI z event sourcing

**Delta względem poprzednich lekcji:** Pierwszy przykład pokazujący produkcyjny streaming UI z pełnym pipeline'em Markdown.

Kluczowe nauki:
- Event sourcing jako architektura — eventy są source of truth, bloki to projection. Pozwala na niezależny rozwój backendu (event generator) i frontendu (materializer)
- Mock-first development — 4 deterministyczne scenariusze z fluent builderem. UI rozwijane bez dostępu do API, zerowy koszt tokenów
- Agent loop po stronie serwera: query → text + tool_calls → execute tools → results → continue (max 6 kroków). `parallel_tool_calls: true`
- Inkrementalny Markdown: committed segments cachowane, live tail przebudowywany. `remend` naprawia niekompletny markdown zanim trafi do parsera
- Chunkowa wirtualizacja (po 12) + scroll anchoring + pin-to-bottom z histerezą —实测 działa dla 1500 wiadomości
- Typewriter z gating — bloki pojawiają się sekwencyjnie, nie "wyskakują"
- Serwer na czystym `node:http` — bez frameworka, routing po pathname + method, SSE z abort/close

### 05_02_voice — Voice agent z LiveKit

**Delta względem 05_02_ui:** Przejście z tekstowego SSE na audio WebRTC. Kompletnie inna modalność interakcji.

Kluczowe nauki:
- Pętla agenta nie jest już w kodzie — delegowana do frameworku LiveKit Agents. `prewarm` ładuje VAD + MCP, `entry` uruchamia sesję
- Integracja MCP dynamiczna — narzędzia z serwerów MCP mapowane do narzędzi LLM. JSON Schema → Zod (rekursywna konwersja obsługująca enum, anyOf/oneOf, nested objects). Prefiksowanie `serverName__toolName` — bezkolizyjne scalenie z wielu źródeł
- `Promise.allSettled` — graceful degradation: jeden MCP pada, agent działa dalej bez jego narzędzi
- Custom TTS plugin: ElevenLabs MP3 → ffmpeg pipe → PCM → AudioByteStream. Wzorzec process pipeline
- Hack na Gemini Live: `autoToolReplyGeneration` powoduje deadlock → workaround przez nadpisanie gettera `capabilities`. Ilustruje, że integracje z beta API wymagają takich sztuczek
- `AbortSignal.timeout()` — nowoczesny wzorzec timeout bez ręcznych timerów
- Browser: `room.startAudio()` musi być w click handler (autoplay policy). Canvas + Web Audio API do wizualizacji fali — 5 warstw sinusoid modyfikowanych amplitudą z AnalyserNode
