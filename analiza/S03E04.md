# S03E04 — Budowanie narzędzi na podstawie danych testowych

## Dlaczego własne narzędzia, nie gotowe MCP/CLI

Gotowe integracje (oficjalne MCP, CLI) dają dostęp do **pełnego** API. To problem, nie zaleta. Agent obsługujący wybraną kategorię maili nie potrzebuje `gmail_search` — potrzebuje `gmail_search_support` z domyślnym zawężeniem. Agent generujący grafiki nie potrzebuje pełnego MCP Replicate — potrzebuje spersonalizowanego narzędzia ze ściśle określonymi ustawieniami.

Pełne API = pełna powierzchnia ataku. Zawężone narzędzie = ograniczona przestrzeń na błędy agenta.

## LLM jako współtwórca narzędzi

Agenci AI mogą **samodzielnie budować narzędzia** (autonomiczne generowanie kodu integracji), ale proces ten jest trudny do kontrolowania i nie nadaje się na produkcję. Zamiast tego: LLM jako partner w procesie kształtowania — od koncepcji, przez strukturę, implementację, po testowanie i optymalizację. Człowiek kieruje, AI wykonuje i proponuje.

## Iteracyjne projektowanie schematów z LLM

LLM generuje **użyteczny punkt startowy (60-70%)**, ale pierwsza wersja jest iluzją poprawności. Typowe braki: brak stronicowania, niewystarczające pola (snippet za krótki), brak metadanych (labels, attachments, read status), fundamentalne błędy (base64 załączników zabijające context window).

**Metoda:** przekazujesz listę zasad/wskazówek → model rozszerza je na kontekst całej integracji. Nie musisz iterować po każdym narzędziu osobno. Kilka ogólnych zasad generuje spójną implementację we wszystkich schematach jednocześnie.

**Zmiana taktyki gdy brakuje pomysłów:** zamiast pytać o zasady → poproś o **wygenerowanie przykładowych interakcji**. Scenariusze użycia często ujawniają problemy, których nie widać w abstrakcyjnych schematach.

## Tool Envelope `{ data, hint }`

Narzędzia nie powinny zwracać "gołych" danych. Standaryzowana koperta odpowiedzi:

```
{ data: T, hint: ToolHint }
```

Gdzie `ToolHint` zawiera:

| Pole | Znaczenie |
|------|-----------|
| `status` | `success` / `empty` / `partial` / `error` |
| `reasonCode` | `OK`, `NO_RESULTS`, `AUTH_REQUIRED`, `RATE_LIMITED`, `NOT_FOUND`, `INVALID_ARGUMENT`, `POLICY_BLOCKED`, `TRANSIENT_FAILURE` |
| `summary` | Opis wyniku czytany przez LLM |
| `nextActions` | Proponowane kolejne kroki (tool + args + confidence) |
| `recovery` | `{ retryable, backoffMs, maxAttempts }` — czy i jak ponawiać |
| `diagnostics` | Scope (tool/global), httpStatus, rawMessage |

**Dlaczego to działa:** LLM dostaje nie tylko dane, ale kontekst decyzyjny. Agent nie zgaduje co zrobić po pustym wyniku — dostaje propozycję szerszego zapytania. Nie zgaduje czy błąd to auth — dostaje `reasonCode: AUTH_REQUIRED` i instrukcję "uruchom auth". Zmniejsza to halucynacje i bezcelowe tury.

## Next Actions — nawigacja agenta

Każdy hint może zawierać `nextActions` — listę proponowanych kolejnych wywołań narzędzi z gotowymi argumentami i scoringiem confidence. Przykłady:

- Po `gmail_search` z wynikami → zaproponuj `gmail_read` z ID pierwszej wiadomości (confidence 0.76)
- Po `gmail_search` z pustym wynikiem → zaproponuj `gmail_search` z szerszym zapytaniem (confidence 0.63)
- Po `gmail_read` z załącznikiem → zaproponuj `gmail_attachment` (confidence 0.72)
- Po `gmail_send` → zaproponuj `gmail_read` do weryfikacji (confidence 0.58)

To jest **guided exploration** — agent nie błądzi, dostaje mapę.

## Error classification — standaryzacja obsługi błędów

Błędy klasyfikowane przez pattern matching na wiadomości + HTTP status:

- `AUTH_REQUIRED` — patterny: `invalid_grant`, `unauthorized`, status 401/403 → "uruchom `bun run auth`"
- `RATE_LIMITED` — patterny: `rate limit`, `quota exceeded`, status 429 → retry z 3s backoff, max 3 próby
- `TRANSIENT_FAILURE` — patterny: `timeout`, `econnreset`, status 5xx → retry z 1.5s backoff, max 2 próby
- `NOT_FOUND` — status 404 → "nie znaleziono, sprawdź ID"
- `INVALID_ARGUMENT` — patterny: `invalid`, `required`, `must be`, `mutually exclusive` → komunikat walidacji

Klasyfikacja następuje w jednym miejscu (`buildErrorHint`), każda warstwa walidacji toola (registry → JSON parse → Zod → handler) używa tego samego mechanizmu.

## 4-warstwowa walidacja narzędzi

Każde wywołanie toola przechodzi przez 4 warstwy, każda z własnym errorem:

1. **Registry** — czy narzędzie istnieje (scope: `global`)
2. **JSON parse** — czy argumenty to poprawny JSON (scope: `global`)
3. **Zod validate** — czy dane pasują do schematu (scope: `tool`)
4. **Handler** — czy wykonanie się powiodło (scope: `tool`)

Każda warstwa zwraca ustandaryzowany `{ data: null, hint: buildErrorHint(...) }`. Agent nigdy nie dostaje surowego exception.

**DRY schemas:** Zod schema definiowana raz (input validation) automatycznie konwertowana do JSON Schema (OpenAI function calling) przez `z.toJSONSchema()`. Nie ma dwóch definicji parametrów — Zod jest single source of truth.

## Auto-detekcja typu zasobu

Agent nie powinien decydować czy ID to message czy thread — implementacja powinna rozwiązać to programistycznie. W przykładzie: najpierw `messages.get` → jeśli 404 → `threads.get`. Jeden parametr `id` obsługuje oba przypadki.

## Kontrola szczegółowości odpowiedzi

Narzędzia search/read przyjmują parametr `details` (default false). Bez niego — compact metadata (subject, sender, date). Z nim — pełne dane (recipients, labels, attachments). Cel: domyślnie oszczędność kontekstu, szczegóły tylko gdy użytkownik pyta konkretnie. Agent nie musi wczytywać 50 linii metadanych by odpowiedzieć "masz 3 nieprzeczytane maile".

## Safety guardrails — whitelist na destrukcyjne akcje

Wysyłka maili wymaga weryfikacji adresatów. Mechanizm:

- Whitelist = właściciel skrzynki + `GMAIL_SEND_WHITELIST` z env
- Każdy adresat poza whitelistą → **automatyczny draft** zamiast wysyłki, `policy.enforcedDraft=true`
- Agent nie może obejść tego przez prompt — blokada jest w kodzie handlera, nie w instrukcjach systemowych
- Hint z `POLICY_BLOCKED` jasno komunikuje: "Saved as draft because recipients are outside whitelist"

Zasada: **AI proposes, code decides**. Prompt może prosić o wysyłkę, ale kod decyduje czy pozwoli.

## Modify zwraca stan po zmianie

Narzędzie modify nie mówi tylko "success" — zwraca **zmienione właściwości** (`isRead`, `inInbox`, `inTrash`, `isDraft`). Agent natychmiast wie czy akcja przyniosła efekt, bez konieczności ponownego odczytu. Wzorzec: zwracaj w odpowiedzi to, co zmieniłeś, nie potwierdzenie że zmiana nastąpiła.

## Mock → Real strategy pattern

Jedna zmienna env (`EVAL_MOCK_GMAIL`) przełącza między realnym API a in-memory mockiem. Każda funkcja clienta:

```
if (EVAL_MOCK_GMAIL) return mockFunction(params);
// ... real API call
```

Mock implementuje ten sam interfejs z deterministycznymi danymi. Pozwala na ewaluację bez dostępu do backendu, ale też na testowanie edge casów (brak wyników, błędy auth) bez dotykania prawdziwej skrzynki.

## Generowanie i kategoryzacja danych testowych

LLM generuje syntetyczne dane testowe na podstawie kodu źródłowego narzędzi. Dwa wymiary kategoryzacji:

| | Pojedyncze interakcje | Scenariusze wieloetapowe |
|---|---|---|
| **Happy path** | tools/gmail_search, tools/gmail_send | scenarios/actions (szukaj → odpowiedz) |
| **Edge case / error** | tools/errors | scenarios/safety (external → enforced draft → user naciska → nadal draft) |
| **Readonly** | tools/gmail_read, tools/gmail_attachment | scenarios/readonly |

Kategorie muszą być jawne od początku — bez nich ewaluacja jest niezarządzalna. LLM domyślnie generuje proste, płytkie testy, które niewiele mówią. Trzeba pokierować: wymusić podział na kategorie, dodawać edge case'y, weryfikować czy testy nie są trywialnie zaliczalne.

Nie warto dopracowywać każdego testu na etapie tworzenia — użyteczność ujawnia się dopiero przy uruchamianiu. Scenariusze testowe mogą dodatkowo służyć do **oceny samych testów** — model na ich podstawie ocenia nie tylko wyniki, ale też jakość i kompletność zestawu testowego.

## Ewaluacja Promptfoo — architektura

**Promptfoo** = ewaluacja "offline" (development/publikacja). **Langfuse** = ewaluacja "online" (obserwacja/produkcja). Można integrować Promptfoo z Langfuse, ale nie jest to wymagane. Rola: Promptfoo jako narzędzie developerskie, Langfuse jako obserwator runtime.

**Dwa custom Providery:**
- **Stateless** (`provider.mjs`) — każde zapytanie uruchamia subprocess `bun src/evals/run-agent-eval.ts` z pełnym agentem
- **Stateful** (`provider.stateful.mjs`) — serializuje historię sesji do plików JSON z SHA256 key (`evaluationId:conversationId:model`), kolejkując zapytania z tego samego `conversationId` sekwencyjnie

**Warstwy asercji:**
1. `is-json` — poprawność JSON
2. `valid_eval_payload` — struktura wyniku (finalText, turns, reachedMaxTurns, toolCalls)
3. `required_tools` / `forbidden_tools` — czy właściwe narzędzia zostały użyte/zakazane
4. `expected_primary_tool` — czy kluczowe narzędzie było wywołane
5. `expected_outcome` — semantyczny wynik (sent/drafted/enforced_draft/modified/search)
6. `turn_budget` — czy agent zmieścił się w limicie tur
7. `llm-rubric` — LLM ocenia jakość odpowiedzi agenta na podstawie kryteriów

**Interpretacja wyników przez LLM:** po wykonaniu ewaluacji na wielu modelach, wyniki (HTML/JSON) przekazuje się do LLM z prośbą o interpretację. Agent ułatwia analizę dużych zestawów danych — prezentując różnice między modelami na różne sposoby. Ujawnia niuanse (np. styl wypowiedzi, niepełne odpowiedzi) niewidoczne w surowych liczbach pass/fail.

Stateful provider wymusza sekwencyjność przez **promise chain** — `sessionChains` Map gwarantuje, że step_02 nie startuje przed zakończeniem step_01.

## Reasoning models w OpenAI Responses API

Modele reasoning (gpt-5, o1, o3, o4) emitują reasoning items z server-side IDs. Przy `store=false` te IDs nie mogą być replikowane w kolejnych turach → 404 na turn 2+. Rozwiązanie:

```typescript
reasoning ? { reasoning: { effort: 'high' },
  include: ['reasoning.encrypted_content' as ResponseIncludable] } : {}
```

Non-reasoning modele (gpt-4.1, gpt-4o) odrzucają te parametry — więc rozróżnienie jest konieczne.

## Testowanie wielu modeli

Posiadanie ewaluacji pozwala na systematyczne porównywanie modeli:

| Model | Charakterystyka z ewaluacji |
|-------|---------------------------|
| gpt-5.2 | Perfekcyjna skuteczność, ale najwolniejszy, wypowiedzi zbyt techniczne (używa identyfikatorów) |
| gpt-5-mini | Środek — zaliczył testy, rozsądny balans |
| gpt-4.1 | Najmniej kroków, profesjonalne wypowiedzi, ale niepełne odpowiedzi i problemy stabilności |

Zaliczenie testów nie jest wystarczającym sygnałem. GPT-4.1 zaliczył wszystko, ale ujawnił problemy, które na produkcji by się pogłębiły. Ewaluacje muszą być na tyle wnikliwe, żeby wyłapać te różnice.

Wybór modelu musi brać pod uwagę szerszy kontekst — jeśli zakres agenta wykracza poza jedną integrację albo złożoność zadań wymaga przetwarzania rozbudowanego kontekstu, mniejsze modele odpadają. Jeśli integracja jest prosta (pojedyncze akcje), można rozważyć jeszcze tańsze alternatywy (OpenRouter, modele open-source jak Qwen czy GLM). Ewaluacje pozwalają to sprawdzić natychmiast.

## Optymalizacja schematów pod mniejsze modele

Nawet jeśli docelowo używamy najsilniejszego modelu — optymalizacja interfejsu narzędzi pod mniejsze modele jest opłacalna. Lepsze opisy, ściślejsze schematy, bardziej precyzyjne hinty = lepsza skuteczność z każdym modelem, nie tylko tańszym.

Po ustabilizowaniu ewaluacji możliwa jest **automatyzacja weryfikacji modeli** — agenci czytają pliki narzędzi + testy → wykonują ewaluację → interpretują wyniki → sugerują usprawnienia. Kluczowe: ten proces uruchamiamy dopiero gdy znamy i rozumiemy ewaluacje na tyle dobrze, żeby wyłapać błędy logiczne AI (testy zaprojektowane tak, że nie da się ich nie zaliczyć).

## Testowanie bez system prompt — jako miara jakości schematów

Wieloetapowe ewaluacje scenariuszowe uruchamiane są **niemal bez system prompt**. Cel: sprawdzenie, jak model radzi sobie opierając się wyłącznie na **opisach narzędzi** i ich **schematach**. Jeśli agent bez dodatkowych instrukcji obsługuje narzędzia poprawnie, po dodaniu wyspecjalizowanego promptu poradzi sobie lepiej. Testy bez promptu = dolna granica jakości interfejsu.

## Kontekst z repozytorium do ewaluacji

Pobranie repozytorium Promptfoo do zasobów projektu + zapytanie LLM o dostępne typy asercji i interfejsy. LLM znają Promptfoo ogólnie, ale kontekst z repozytorium zapewnia dostęp do najnowszych funkcjonalności, co podnosi jakość sugestii.

## Prompt systemowy jako lista zasad, nie narracja

W przykładzie Gmail system prompt to 19 linii — kompaktowa lista 5 narzędzi z jednozdaniowymi opisami + 9 konkretnych zasad routingu i obsługi błędów. Kluczowe: prompt **referuje do mechanizmu hintów** ("Each tool response is shaped as `{ data, hint }`") zamiast dublować logikę. Zasady opisują **kiedy** użyć narzędzia, nie **jak** działa.

## Przykłady kodu — progresja wiedzy

### Przykład 1: `03_04_gmail` — Tool Envelope z Hint system

**Delta:** Narzędzia nie zwracają surowych danych. Każda odpowiedź to `{ data, hint }` z `status`, `reasonCode`, `summary`, `nextActions[]`, `recovery`, `diagnostics`.

Implementacja: `src/hints/index.ts` definiuje `ToolHint` i `ToolEnvelope<T>`, `withHint()` tworzy kopertę, `buildErrorHint()` klasyfikuje błędy. Tool-hints (`src/hints/tool-hints.ts`) budują per-narzędzie hinty z kontekstowymi propozycjami kolejnych akcji.

### Przykład 2: `03_04_gmail` — Mock Client z mutowalnym stanem

**Delta:** Mock nie jest statycznym fixturem — to w pełni funkcjonalny in-memory Gmail z parserem zapytań, paginacją cursorową, mutowalnymi labelkami i generowanymi ID. Stan zmienia się między wywołaniami (wysłany mail dodaje entry, modify mutuje labelIds).

Implementacja: `src/gmail/mock-client.ts` — `createState()` inicjalizuje 15 wiadomości, `parseQuery()` tokenizuje Gmail query syntax, `matchesQuery()` filtruje, `applyActionToLabels()` modyfikuje stan. Przełączenie na mock przez `EVAL_MOCK_GMAIL=1`.

### Przykład 3: `03_04_gmail` — Stateful eval z sesjami

**Delta:** Multi-turn ewaluacja z persystencją historii między krokami scenariusza. Ten sam `conversationId` = ta sama sesja agenta.

Implementacja: `provider.stateful.mjs` — `resolveSessionKey()` komponuje klucz z `evaluationId:conversationId:model`, `runInSessionOrder()` gwarantuje sekwencyjność przez promise chain, `loadSessionHistory()` / `saveSessionHistory()` persystują historię do `.sessions/{sha256}.json`. Eval runner (`run-agent-eval.ts`) ładuje zapisaną historię do `AgentSession`, uruchamia agenta, zapisuje z powrotem.

### Przykład 4: `03_04_gmail` — Send whitelist jako guardrail

**Delta:** Blokada destrukcyjnej akcji w kodzie handlera, nie w prompcie. Agent nie może obejść policy przez prompt injection.

Implementacja: `src/gmail/client.ts:637-645` — `getEffectiveSendWhitelist()` pobiera email właściciela + env whitelist, każdy adresat sprawdzany przez `normalizeRecipientEmail()` + `whitelistSet.has()`, `blockedRecipients.length > 0` wymusza `shouldDraft=true`. Wynik: `policy.enforcedDraft=true` w `GmailSendResult` + `POLICY_BLOCKED` w hint.
