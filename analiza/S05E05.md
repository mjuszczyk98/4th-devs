# S05E05 — Nowa Rzeczywistość

## Odwołenie ról: my wspieramy AI, nie odwrotnie

Pisania kodu ręcznie → IntelliSense → AI Completion → AI generujące 90%+ kodu z człowiekiem jako supervisorem. Dziś kluczowe kompetencje to: **dostarczanie kontekstu, kierunek rozwoju, decyzje architektoniczne, quality assurance**. Nie pisanie logiki.

Konsekwencja: to, co kiedyś było "projektem po godzinach" na miesiące, dziś jest realizowalne w popołudnie. Ale szybkość generowania kodu nie równa się szybkości **ukształtowania** go — Wonderlands powstał w kilkanaście dni przy pomocy AI, ale na bazie 3 lat doświadczeń z podobną aplikacją. AI kompresuje czas implementacji, nie czas zrozumienia domeny.

## Technologia bez procesu = imponujący chatbot

System agentowy z MCP, sandboxem, przeglądarką i multi-agent delegation nie ma wartości bez **zdefiniowanych procesów**. Podłączenie Google Calendar by zapytać o dodanie wpisu jest mniej efektywne niż zrobienie tego ręcznie. Wartość pojawia się dopiero przy:
- spersonalizowanych **procedurach** opisujących jak agent ma używać narzędzi
- **skryptach** automatyzujących powtarzalne sekwencje
- **cyklicznych wyzwalaczach** uruchamiających agentów w tle

Lekcja: zacznij od **jednego** powtarzalnego procesu, który działa. Nie od architektury wszechświata.

## Cyfrowy ogród jako baza wiedzy agenta

Koncepcja: strona www generowana z systemu plików tekstowych (markdown + frontmatter + wikilinks) stanowi jednocześnie:
- **Wiedzę długoterminową** agenta — notatki, procedury, skille
- **Publikację** — wybrane treści dostępne jako strona www
- **Obszar roboczy** — agent czyta i modyfikuje pliki, tworzy nowe notatki
- **Organizację** — tagi, katalogi, wikilinks łączą wiedzę w sieć

Pliki z `visibility: private` w frontmatter są ukryte w menu i chronione hasłem. Folder workspace kompatybilny z Obsidian — edycja tekstowa i agentowa uzupełniają się.

## Daily Ops — wzorzec orkiestracji asynchronicznej

Schemat: agenci uruchamiani **równolegle** cyklicznie, każdy w swojej sesji, każdy zbiera dane z własnych integracji:

| Agent | Źródło | Akcja |
|-------|--------|-------|
| Calendar | Google Calendar | Zgromadzić wydarzenia nadchodzącego dnia |
| Tasks | Linear | Nieukończone + aktywne zadania |
| Mail | Gmail | Kluczowe powiadomienia wymagające uwagi |
| Newsfeed | Firecrawl / YouTube | Filtrowane informacje wg kryteriów |

Wyniki → jeden folder → agent agregujący → transkrypt audio → urządzenie mobilne. Agenci nie mają między sobą zależności, więc uruchamia się ich równolegle na godzinę przed deadline.

Uruchomienie: `POST /v1/sessions/bootstrap` z API key + tenant ID + initialMessage kierujący agenta do konkretnych plików opisujących proces. Brak wbudowanego crona — harmonogram z zewnątrz (cron, GitHub Actions, dowolny scheduler).

## Instrukcje dla agentów — jednorazowa inwestycja

Pułapki oczekiwań: za ogólne instrukcje, założenie bezbłędności, brak doprecyzowania + oczekiwanie jakości, długie interakcje (koszty tokenów!), oczekiwanie determinizmu.

Praktyki:
- Wskazuj konkretne **pliki** i **nazwy narzędzi** w instrukcjach — nawet kategorie lub foldery
- Utrzymuj interakcje **tak krótkie jak możliwe** — token costs w własnych systemach agentowych są realne i znacznie przewyższają subskrypcje Claude Code
- Procedury powinny być proste i **łączyć się** ze sobą przez wikilinks — agent odkrywa je stopniowo
- Doprecyzowanie to jednorazowy wysiłek zwracający się wielokrotnie

## Nawyk > technologia

Techniczne możliwości nie mają znaczenia bez nawyku korzystania. Klucz: **dopasowanie kanału do istniejących zachowań**:
- Discord → tam trafiają powiadomienia
- Telefon → prosta aplikacja mobilna / shortcut do czatu
- Poranny spacer → audio wygenerowane przez system ładuje się automatycznie przy rozłączeniu z Wi-Fi

Wyrabianie nawyku = łączenie nowej aktywności z tym, co już robimy.

## Code Mode — narzędzia przez kod, nie function calling

Gdy sandbox aktywny, agent nie ładuje definicji narzędzi do kontekstu. Zamiast tego **pisze i wykonuje kod**, który odkrywa i wywołuje MCP tools dynamicznie. Definicje narzędzi nie zajmują kontekstu — agent szuka ich na żądanie.

Konfigurowalne per agent: wyspecjalizowany agent z małą liczbą narzędzi powinien mieć je w kontekście od razu (function calling). Agent z wieloma integracjami — Code Mode.

## Referencje — przepływ artefaktów między agentami

Technika z wcześniejszych lekcji, tu połączona w pełny przepływ: agent A generuje obraz → przekazuje referencję do agenta B → agent B nakłada znak wodny → publikuje w cyfrowym ogrodzie. Referencje integrują też sandbox z systemem plików — wynik wykonania kodu staje się plikiem dostępnym dla innych agentów.

## Kierunki rozwoju — co dodawać do systemu

| Obszar | Co brakuje |
|--------|------------|
| Cron / zdarzenia | Wbudowany harmonogram + nasłuchiwanie na zewnętrzne zdarzenia |
| Otoczenie | Context awareness (S04E03/S03E05) — agenci reagują na dane z otoczenia |
| Wzmocnienie pętli | Retry z zachowaniem payload — agent modyfikuje istniejący payload przy błędzie, nie generuje od nowa |
| Głębsze integracje | Lepsze uprawnienia lokalne (z ostrożnością przy ekspozycji na zewnątrz) |
| Aplikacja mobilna | Głosowa interakcja z agentami, dostęp do wiedzy z cyfrowego ogrodu |
| Panel zarządzania | Aktywny dashboard z interakcją, nie pasywny podgląd |
| Artefakty | Dynamiczne generowanie + zapis + przywoływanie (S03E05) |
| Pamięć | Profile agenta/użytkownika, częściowo statyczne, częściowo dynamiczne, dedykowane narzędzia pamięci |

## Przykład kodu — delta względem S05E04

### `05_05_wonderlands` — Od produkcyjnego runtime do pełnej platformy agentowej

**Delta względem `05_04_api` + `05_04_ui`:** od produkcyjnego backendu + frontendu do **monorepo ze współdzielonymi kontraktami, sandboxem, przeglądarką, cyfrowym ogrodem i setup wizard**.

- **Monorepo z contracts jako source of truth** — `packages/contracts` publikuje surowe `.ts` przez `exports` map (zero build step). Zod schemas → inferred types → `Replace<>` utility overlayuje string na branded IDs. Runtime validation + compile-time safety z jednego źródła.
- **45+ typów eventów** — rozszerzenie o tool confirmation lifecycle (`requested/granted/rejected`), child run delegation (`child_run.created/completed`), memory (`observation/reflection started/completed`), thread naming, garden builds. Eventy niosą CQRS metadata: `aggregateId`, `causationId`, `traceId`, `eventNo`.
- **MCP Code Mode** — agent discovers MCP tools w runtime, pobiera TypeScript bindings, wywołuje z sandbox script przez file-based IPC bridge (`mcp-bridge.mjs`). Write request JSON → poll response JSON. Zero network, zero sockets.
- **Dual sandbox** — Node.js (local dev) + `lo` runtime z `just-bash`. Własny module resolver w `lo` (resolve bare specifiers z `package.json` exports, CJS interop). Virtual filesystem z read-only roots i path traversal protection.
- **Write-back z human approval** — sandbox proponuje zmiany plików → user review'uje → commit. Dwufazowy commit jako safety mechanism.
- **Garden system** — static site builder zintegrowany z platformą: strony, buildy, deploy, preview. Agent zarządza treścią, system generuje HTML.
- **Setup wizard** — interaktywne CLI z TTY raw mode (password masking), spinner, structured prompts. Generuje `.env`, seed'uje DB, konfiguruje API keys.
- **Coordinator pattern na froncie** — chat store (~2000 linii) komponuje ~20 fokusowanych modułów (run-lifecycle, run-bootstrap, leases, event-engine, materialize, rendered-message-state). Dependencje wstrzykiwane przez config objects (getter functions), nie importowane bezpośrednio.
- **Streaming markdown — committed/live tail split** — `streaming-markdown.ts` dzieli markdown na finalized segments (pełne renderowanie z syntax highlighting) i live tail (lekkie). Re-render tylko taila.
- **Block settling** — w trakcie streamingu bloki renderują się flat. Po zakończeniu: faza `settling` → bloki grupowane w `ToolChain` (sekwencyjne) i `ToolGroup` (równoległe) z animacją collapse.
