# S04E01 — Wdrożenia rozwiązań AI

## Punkt startowy wdrożenia AI

Brak ustalonych najlepszych praktyk — model AI, API i techniki zmieniają się w tempie miesiący, nie lat. Oznacza to, że każde wdrożenie jest de facto eksperymentem z ograniczoną zdolnością prognozowania długoterminowych skutków.

Dobra strategia: **zbudować rozwiązanie, z którego sam korzystasz**. Bezpośrednie doświadczenie problemów i ograniczeń buduje intuicję skalowalną na rozwiązania dla innych.

Fundament projektu nie musi być kompletny — celowo minimalny (`04_01_garden` nie ma UI, kompresji kontekstu, mało narzędzi). Ma być **punktem odniesienia** do iteracji.

## Cyfrowy Ogród — model architektury

Trzy zintegrowane subsystemy: **Vault** (Markdown), **Agent** (OpenAI Responses API + Daytona Sandbox), **Grove** (custom static site generator). Agent traktuje vault jednocześnie jako pamięć i przestrzeń roboczą.

Kluczowe ograniczenia celowe:
- Dane publiczne by design — brak problemu prywatności
- Treść tworzy człowiek, AI ją **wzbogaca** — nie odwróconie
- Format Markdown, konwersja do HTML wyłącznie przez kod

## Synchroniczna vs asynchroniczna współpraca z AI

Dwie różne kategorie problemów projektowych, nie tylko tryby pracy.

| Wymiar | Synchroniczna | Asynchroniczna |
|---|---|---|
| Interfejs | Centralny element systemu | Pominięty lub minimalny |
| Personalizacja | Workflow, skille, rozszerzenia na żywo | Zdefiniowane z góry procesy i harmonogramy |
| Uprawnienia agenta | Większe (nadzór człowieka) | Ograniczone do zdefiniowanych ścieżek |
| Komunikacja | Dwukierunkowa, feedback w locie | Jednokierunkowa raporty, min. zaangażowanie człowieka |
| Zmienność | Elastyczna, iteracyjna | Modyfikacje i dopasowanie do zmieniającego się otoczenia |

Hybryda jest możliwa: system działa w tle, użytkownik ma dostęp do edycji i prostych poleceń. Agent nie potrzebuje czatu — wystarczy API do zapisu/odczytu.

## Mapowanie procesów — ramy decyzyjne

Przy projektowaniu trzeba odpowiedzieć na trzy pytania naraz: **co robimy**, **czego nie robimy**, **jak to robimy**. AI nie odpowie na nie automatycznie. Decyzja o **niestosowaniu AI** tam, gdzie korzyści są marginalne, jest równie ważna.

Kryteria kształtujące architekturę:
- **Użytkownik** i jego kompetencje → zakres swobody agenta (sandbox, terminal, code mode)
- **Źródło treści** → kto tworzy, kto wzbogaca, w jakiej proporcji
- **Format danych** → Markdown jako single source of truth, HTML generowany
- **Integracje** → Skills + Code Mode dają dostęp do CLI, MCP, natywnych narzędzi
- **Publikacja** → GitHub Pages + Actions (push = deploy)
- **Dostępność** → API wymusza zdalny serwer + synchronizację (np. Mutagen)

## Balans kod vs AI

Kilka miesięcy temu: 90/10 lub 80/20 (kod/AI). Obecnie proporcje się odwracają w wybranych scenariuszach — logika agentów zwiększa rolę modeli. Elementy tradycyjnej aplikacji nadal stanowią większość architektury, ale granica się przesuwa.

## Szybkie weryfikowanie założeń

Każdy założony proces trzeba przetestować wcześnie — zachowania modeli są trudne do przewidzenia, koszty API mogą zaskoczyć, a najnowsze modele mogą nagle umożliwić to, co wcześniej było niemożliwe.

Przykłady testów z projektu:
- **Wzbogacanie**: model może nie wiedzieć, co dodać → frontmatter z instrukcją dla agenta
- **Budowanie**: przesłanie URL → agent sam decyduje o umiejscowieniu i opisie → ewaluacja na zestawie danych
- **Dostępność**: chatbot nie jest jedyną opcją — wystarczyć może wyszukiwarka lub aplikacja mobilna

AI zmienia etap prototypowania: z sekwencyjnego testowania pomysłów na **równoległe** testowanie wielu wariantów w krótszym czasie.

## Wzorce implementacyjne z `04_01_garden`

### Agentic Loop

Bounded turn-based pętla (max 20 tur). Każda tura: completion → paralelne wykonanie tool calli → wyniki z powrotem do LLM. Wyjście gdy model nie woła narzędzi. `previous_response_id` zachowuje kontekst server-side, redukując zużycie tokenów.

### Template-Driven Agent

Instrukcje, model i narzędzia zdefiniowane w `main.agent.md` z YAML frontmatter — agent konfigurowalny z Markdown, nie z kodu.

### Skill Plugin System

`SKILL.md` z frontmatter (metadata, allowed-tools, runtime-scripts) + Markdown body (instrukcje). Code-as-skill: deterministyczne transformacje danych delegowane do skryptu, nie do LLM step-by-step.

### Lazy Sandbox + Bidirectional Sync

Daytona VM tworzona dopiero na `get()`. Snapshot diffing (`size:mtimeMs`) — tylko zmienione pliki są przesyłane. `syncInFlight` flag zapobiega race conditions.

### Code Mode

Dynamicznie generowany program TypeScript z typed helper API (`codemode.vault.*`, `codemode.runtime.*`, `codemode.output.*`). Komunikacja sandbox → host przez marker-based IPC: `__CODE_MODE_RESULT__=` / `__CODE_MODE_ERROR__=`.

### Tool Registry

Map<string, Tool> z jednolitym interfejsem. Nowe narzędzie = obiekt z `definition` (OpenAI schema) + `handler`. Rejestr filtruje narzędzia per skill.

## Progresja wiedzy — delta implementacji

### Delta 1: Agentic Loop z response chaining

```typescript
const response = await completion({ ..., previousResponseId });
previousResponseId = response.id;
```

Server-side kontekst przez `previous_response_id` — zamiast wysyłać pełną historię rozmowy, referencja do poprzedniej odpowiedzi. Nowe względem klasycznego approachu z pełnym `messages` array.

### Delta 2: Skill invocation jako routing

```typescript
const match = trimmed.match(/^\/([a-z0-9][a-z0-9-]*)(?:\s+([\s\S]*))?$/i);
```

Skill ogranicza dostępne narzędzia do `allowed-tools` zdefiniowanych w SKILL.md — agent nie może użyć narzędzia spoza scope'u skilla.

### Delta 3: Frontmatter jako konfiguracja runtime

```yaml
---
model: gpt-5.2
tools: [terminal, code_mode, git_push, web_search]
---
```

Markdown + YAML frontmatter = konfiguracja agenta bez zmiany kodu. Oddziela definicję zachowania od implementacji.

### Delta 4: Snapshot-based incremental sync

```typescript
const signature = `${stat.size}:${stat.mtimeMs}`;
if (previousSnapshot.get(rel) === signature) continue;
```

Sygnatura pliku = rozmiar + timestamp. Przesyłane tylko delty. Prostsze niż hashowanie zawartości, wystarczające do detekcji zmian.

### Delta 5: Marker-based IPC

```typescript
console.log(`__CODE_MODE_RESULT__=${JSON.stringify(result)}`);
console.log(`__CODE_MODE_ERROR__=${JSON.stringify({ message, stack })}`);
```

Protokół tekstowy do komunikacji sandbox → host. Prostszy niż RPC/IPC, wystarczający gdy output to stdout.
