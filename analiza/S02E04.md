# S02E04 — Organizowanie kontekstu dla wielu wątków

## Architektury wieloagentowe

| Architektura | Komunikacja | Zależności | Przykład |
|---|---|---|---|
| **Pipeline** | Jednokierunkowa, sekwencyjna | Wynik poprzedniego = wejście następnego | ETL, generowanie + redakcja |
| **Blackboard** | Wspólny stan, agenci niezależni | Odczyt/zapis współdzielonej przestrzeni | Researcherzy z S02E03 |
| **Orchestrator** | Koordynator centralny deleguje i odbiera | Manager steruje przepływem | Claude Code, Daily Ops |
| **Tree** | Orchestrator + managerowie pośredni | Hierarchia delegowania | Złożone procesy wieloetapowe |
| **Mesh** | Adresowana: agent wie, do kogo pisze | Luźne powiązania, routing | Upload → File Manager |
| **Swarm** | Rozproszona, self-organizing | Selekcja/agregacja wyników | Rzadko spotykane w LLM |

Mesh i Swarm — trudne do debugowania i kontroli. W praktyce dominują pierwsze cztery, nierzadko łączone w jednym systemie.

## delegate i message — dwa fundamenty komunikacji

**delegate** — otwiera nowy wątek: nowa instrukcja systemowa + nowy zestaw narzędzi + niezależna pętla agentowa. Wynik sub-agenta staje się wynikiem narzędzia `delegate` dla agenta nadrzędnego.

**message** — dwukierunkowa komunikacja: pętla sub-agenta **wstrzymana** do czasu dostarczenia danych. Realizowalne przez generatory (yield = czekaj, next() = dostarcz dane). Używane gdy sub-agent nie ma wystarczających informacji i musi zapytać wstecz.

Kluczowa różnica: `delegate` = fire-and-collect, `message` = dialog z wstrzymaniem.

## Architektury oparte na zdarzeniach

Powiązania między agentami mogą być **bezpośrednie** (A deleguje do B) lub **pośrednie** — przez event bus. Przykład: `user.message` → Intent Agent klasyfikuje → `ticket.classified` → Tracker i Credit reagują niezależnie → Draft Agent czeka na komplet danych (w tym z `customer.lookup_requested`) → szkic → weryfikacja przez człowieka.

Zdarzenia adresują problem: agent A nie musi wiedzieć, że agenty C i D zareagują na jego wynik. Rozbijamy sprzężenie. Koszt: złożoność debugowania i trudniejszy tracing przepływu.

## Globalny kontekst — konflikty i strategie

Ten sam agent uruchomiony wielokrotnie (równolegle) = system wieloagentowy na jednym szablonie. Dwa wystąpienia zapisujące tę samą notatkę → ostatni zapis wygrywa, dane agenta A tracone.

Pięć strategii zarządzania:

| Strategia | Mechanizm | Kiedy |
|---|---|---|
| **Wykrywanie konfliktów** | Checksumy/hash'e linii między odczytem a zapisem | Dwa agenty modyfikują ten sam plik |
| **Unikanie konfliktów** | Izolacja zasobów per sesja, uprawnienia RO, przynależność | Projektowanie architektury |
| **Agent zarządzający** | Memory Manager z pełnym wglądem w historię + kontakt z człowiekiem | Pamięć długoterminowa |
| **Historia zmian** | Append-only zamiast overwrite — Observational Memory | Wiedza temporalna |
| **Interwencja człowieka** | Jak merge conflict w Git — człowiek rozstrzyga | Gdy automatyzacja zawodzi |

Zewnętrzny kontekst (baza wiedzy, pamięć) powinien być **odseparowany od logiki agentów**. Dokumenty mówią *co* — system decyduje *kiedy i kto*. Dostęp per agent + per użytkownik, nie globalny.

## Wyzwania współdzielenia kontekstu

1. **Sesja vs pamięć** — informacja tymczasowa (sesja) vs trwała (pamięć). Kto decyduje co utrwalić? Agenci w tle potrzebują więcej autonomii w tej decyzji, balansowanej hardkodowanymi ograniczeniami (np. dostęp do katalogów).

2. **Degradacja komunikacji** — każdy przekaz między agentami = potencjalna utrata/zniekształcenie. Instrukcje narzędzi `delegate` i `message` muszą być precyzyjne. System musi zakładać niekompletne dane na wejściu sub-agenta.

3. **Własna interpretacja** — te same dane, inna interpretacja. Ryzyko minimalne dla zadań zamkniętych ("zaktualizuj dane X"), rośnie dla otwartych ("znajdź wszystko o X").

4. **Utrata kontekstu oryginalnego** — notatka "Anna" stworzona w kontekście rozmowy o Ani K. może zostać pomylona z Anną M. w innej sesji. Metadane (źródło, data, powiązania) zapobiegają temu.

5. **Duplikowanie informacji** — nieuniknione, ale reaktywne. Mniejsze modele mogą skanować zmiany i wykrywać duplikaty (technikę widzieliśmy przy grafach w S02E03).

6. **Metadane** — wykorzystywane nie tylko w UI, ale **w komunikacji między agentami**. Pytanie "o czym rozmawialiśmy w drodze do Warszawy" wymaga metadanych o czasie i lokalizacji, nie tylko treści.

## Kiedy agenci, a kiedy kod

Decyzja w każdym przypadku osobno. Agenci uzasadnieni gdy:

- **Zadania otwarte** — jasny cel, ale potrzeba reagowania na dane z otoczenia w trakcie
- **Dynamiczne dane** — struktura input/output nie jest z góry określona lub wymaga transformacji poza możliwościami kodu
- **Dynamiczne zależności** — powiązania między danymi wykrywalne tylko na poziomie języka i znaczenia
- **Iterowanie** — kryteria zapisane językiem naturalnym, kroki niemożliwe do zdefiniowania z góry
- **Elastyczna architektura** — nowe obszary bez zmian w logice
- **Dopasowanie wyniku** — personalizacja wykraczająca poza szablony

Agenci **nie** uzasadnieni gdy: koszty zero, szybki czas reakcji, pełna przewidywalność. Tam gdzie wystarczy deterministyczny kod — on jest jedyną opcją.

## Rola agenta zarządzającego

Zwykle brak narzędzi operacyjnych (poza `delegate`, `message`, `search_memory`), ale najszerszy dostęp do informacji i uprawnień. Odpowiedzialności:

- **Wiedza o systemie** — role agentów, zakres odpowiedzialności, dane użytkownika
- **Dostęp do informacji** — wyniki agentów + przestrzeń robocza + pamięć (przynajmniej RO)
- **Delegowanie** — główny wątek, plan działania, postępy w kontekście
- **Transport wiedzy** — przekazywanie rezultatów między agentami, dostarczanie brakujących danych
- **Decyzyjność** — limity uprawnień, kiedy eskalować do człowieka
- **Weryfikacja** — ocena rezultatów przed akceptacją (lub delegacja weryfikacji)

Ograniczanie roli agenta zarządzającego jest konieczne (modele zawodzą), ale z umiarem — każdy dodatkowy etap komunikacji komplikuje zarządzanie kontekstem. Część błędów jest niezauważalna od razu (np. pominięcie etapu z powodu braku danych, nie zatrzymanie systemu).

Człowiek pozostaje kluczowym koordynatorem. Panele zarządzania (statystyki, aktywne sesje, harmonogram, obszary wymagające uwagi) stają się nowym typem interfejsu — okno czatu nie wystarcza.

## Przykład kodu: `02_04_ops`

### Architektura: agent loop + rekurencyjna delegacja

Minimalny system: 4 pliki TypeScript (~270 linii) + workspace z plikami markdown/JSON.

Agent loop w `agent.ts` — pętla z limitem 15 tur: LLM → tool_calls → wykonaj → wynik do messages → powtórz. Bez tool_calls = odpowiedź końcowa.

Delta względem S02E01 (`02_01_agentic_rag`): agent loop bez MCP. Narzędzia zdefiniowane inline w `tools.ts`, nie pobierane z zewnętrznego servera. Celowo prostsze — pokazuje czystą mechanikę delegacji.

### Agent-as-Markdown (frontmatter pattern)

Definicja agenta = plik `.agent.md` z YAML frontmatter (model, narzędzia) + treść (system prompt):

```markdown
---
model: openai:gpt-4.1-mini
tools: [get_mail]
---
You scan the inbox and return a structured summary.
```

Nowy agent = nowy plik, zero zmian w kodzie. Konfiguracja i instrukcja oddzielone od logiki uruchamiania. `gray-matter` parsuje frontmatter, treść staje się system promptem.

### delegate jako narzędzie interceptowane

`delegate` w `tools.ts` to tylko definicja JSON Schema — handler zwraca surowe args. Rzeczywista logika w `agent.ts:112-116`:

```typescript
if (name === 'delegate') {
  result = await runAgent(agent, delegatedTask, depth + 1)
}
```

Rekurencyjne wywołanie `runAgent` z depth + 1. Limity: `MAX_DEPTH = 3`, `MAX_TURNS = 15`. Sub-agent dostaje własną konwersację z własnym system promptem i własnym zestawem narzędzi. Wynik sub-agenta = wynik narzędzia dla orchestratora.

Brak `message` — komunikacja jednokierunkowa. Sub-agent nie może zwrócić się z pytaniem.

### Workflow-as-data

Orchestrator dostaje zadanie: "przeczytaj `workflows/daily-ops.md` i wykonaj". Sam odczytuje instrukcje przez `read_file`, interpretuje kroki i egzekwuje. Zmiana workflow = zmiana markdowna, bez rekompilacji.

### Sandbox plików

`isPathSafe()` sprawdza czy resolved path nie wychodzi poza `workspace/`:

```typescript
const rel = relative(workspaceResolved, fullPath)
return !rel.startsWith('..') && rel !== '..'
```

LLM generuje ścieżki — walidacja konieczna. Bez niej: path traversal (`../../etc/passwd`, `../../.env`).

### Kontekst warstwowy

Workspace dzieli dane na cztery kategorie dostępne przez `read_file`:
- **goals/** — długoterminowe cele (alignment decyzji z priorytetami)
- **history/** — wczorajszy output (dedup + escalation z licznikiem dni)
- **memory/** — preferencje usera (format, styl, reguły priorytetyzacji)
- **sources/** — dane wejściowe (mock API, statyczne JSON)

Agent sam decyduje co przeczytać, w jakiej kolejności, i jak połączyć. To nie RAG — to agent-driven context assembly.

### Uproroszczenia (świadome, PoC)

- Źródła danych: statyczne JSON, nie API
- Delegacja: sekwencyjna, nie równoległa (`Promise.all` by pryszyło koszty)
- Historia: hardkodowana data `2026-02-12.md`, nie dynamiczna
- Brak weryfikacji outputu przez orchestratora
- Brak conflict detection przy zapisie
- Wszystkie agenty na `gpt-4.1-mini` mimo mechanizmu pozwalającego na różne modele
