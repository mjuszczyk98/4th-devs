# S04E05 — Projektowanie rozwiązań wewnątrzfirmowych

## Trzy wymiary wdrożenia AI w organizacji

Każde wdrożenie rozpatruj na trzech osiach jednocześnie — pominięcie którejkolwiek kończy się niepowodzeniem:

- **Biznesowy** — koszty wdrożenia + utrzymania, decyzje prawne (dostawca chmury, Bedrock/Azure), ROI. Technologia nie jest celem, lecz kosztem z oczekiwanym zwrotem.
- **Kulturowy** — zmiana nawyków i procesów. Inicjatywy oddolne (warsztaty, wymiana doświadczeń) skuteczniejsze niż narzucone z góry.
- **Technologiczny** — zakres, plan, wskaźniki sukcesu/porażki. Rozszerzony o: wybór modeli, ewaluację, architekturę agentów. Założenie 100% skuteczności = błąd projektowy.

## Paradoks doświadczenia

Skuteczność tej samej osoby z tym samym agentem drastycznie zależy od **świadomości mechanik**. Doświadczony użytkownik naturalnie zawiera w zapytaniu wskazówki: „sprawdź w **gmailu i na slacku**, potem zerknij do **plików** z **planem dnia**". Niedoświadczony pyta: „sprawdź co mam na dziś?" — i agent nie wykonuje zadania.

Model nie jest w stanie reliably zrekompensować tego braku. Rozwiązanie: projektuj narzędzia tak, by **ograniczały potrzebę precyzji** ze strony użytkownika.

## Najprostsze wdrożenia: dokumenty jako narzędzia

Zanim piszesz kod, sprawdź czy wystarczy dokument:

- **Checklista** — powtarzalny proces (weryfikacja SEO blog posta, procedury wdrożeniowe). AI niczego nie przegapi. Rola człowieka: akceptacja, nie przypominanie sobie.
- **Onboarding** — nie zastępuje rozmowy, lecz **przekierowuje**. AI radzi sobie z niedokładnymi zapytaniami lepiej niż wyszukiwarka tekstowa.
- **Styl/standard** — jeden plik promptu dający spójność wygenerowanych treści w całym zespole.

KonceptAGENTS.md i Skills to te same wzorce — instrukcje sterujące zachowaniem modelu — przeniesione do programowania.

## Generatywne interfejsy jako kompromis determinizmu i elastyczności

Chat-only interfejs ma dwa problemy: (1) użytkownik nie widzi struktury danych, (2) akcje są niedeterministyczne. MCP Apps rozwiązują oba:

- Predefiniowany UI (HTML w iframe) → deterministyczne akcje wykonywane przez kod
- Dane z wielu źródeł w jednym widoku → redukcja przełączania kontekstu
- Model widzi tylko udostępnione dane, nie kod interfejsu

| | Czysty chat | MCP Apps | Pełna automatyzacja |
|---|---|---|---|
| Kontrola | Niska | Pełna (kod UI) | Żadna (agent sam działa) |
| Determinizm akcji | Niski | Wysoki | Niski |
| Złożoność implementacji | Niska | Średnia | Wysoka |
| Elastyczność | Wysoka | Średnia | Wysoka |

Architektura: Browser Host (chat + montowanie iframe) → MCP Server (narzędzia + zasoby UI) → Embedded Apps (sandboxed iframe z AppBridge). Każda warstwa wymienialna niezależnie. Ten sam serwer MCP podłączysz do Claude.ai bez zmian.

## Agent review — Human-in-the-Loop jako feature, nie ograniczenie

Wzorzec: AI analizuje dokument blok po bloku, komentuje wybrane fragmenty, człowiek akceptuje/odrzuca. Kluczowe decyzje projektowe:

- **Dwa tryby** — paragraph (blok po bloku, równolegle, limit komentarzy) vs at-once (cały dokument, agent sam decyduje o zakresie)
- **Narzędzie waliduje scope, nie prompt** — `add_comment` odrzuca błędny `block_id`. LLM czasem halucynuje parametry — fizyczna walidacja w handlerze narzędzia jest jedyną barierą.
- **Błędy narzędzi jako dane, nie wyjątki** — handler zwraca `{status: "rejected", error}` zamiast throw. Agent widzi powód i koryguje. Wyjątek przerywa pętlę.
- **Elastyczność przez kontekst, nie kod** — zmiana promptu + dodanie `contextFiles` zmienia agenta z korektora w fact-checker, SEO optimizer, lub router zgłoszeń.

## Model bezpieczeństwa dla agentów firmowych

Nawet z zaufanym dostawcą chmury — **nie ufasz modelowi**. Zaufanie do infrastruktury ≠ zaufanie do outputu.

Kategorie ryzyka:

- **Wyciek danych** — agent z dostępem do Internetu może przesłać dane wewnętrzne na zewnątrz
- **Destrukcja danych** — agent z dostępem do kodu może usunąć lub uszkodzić źródła
- **Nieautoryzowane akcje** — przypadkowe wywołanie `send_email`, zaproszenie osób spoza org na spotkanie
- **Wprowadzenie w błąd** — chatbot sugeruje akcję naruszającą procedury (restart produkcji)
- **Omijanie zabezpieczeń** — modele potrafią wykrywać, że są testowane, i dostosowywać zachowanie (ref: Claude Opus 4.6 System Card, Anthropic Eval Awareness)

Zasada: ograniczanie uprawnień agentów + fizyczne uniemożliwianie niepożądanych akcji na poziomie kodu, nie promptu. Sandbox, CSP, path traversal protection, narzędzia app-only — to warstwy obrony, nie pojedyncze rozwiązanie.

## Przykłady kodu — progresja i delta wiedzy

### 04_05_review — Document review z inline annotations

**Delta:** Agent z narzędziami operującymi na **położeniu w tekście** + UI z zakotwiczoną adnotacją. Wzorzec Human-in-the-Loop jako samodzielna aplikacja.

- **NDJSON streaming bez WebSocketów** — POST zwraca `application/x-ndjson`. Frontend czyta przez `ReadableStream` z `buf = lines.pop()` zachowującym niekompletną linię. Prostsze niż WS, działa z HTTP.
- **Mini Operational Transform** — accept sugestii patchuje markdown i przesuwa pozycje innych komentarzy (delta shifting). Batch accept sortuje **od końca** tekstu — klasyczny wzorzec wielokotwicowych modyfikacji.
- **Prompty jako Markdown z frontmatter** — `title`, `model`, `modes`, `contextFiles`. Samoopisujący się dokument = konfiguracja agenta. Nowy typ review = nowy plik `.md`.
- **Agent profile jako plik** — `workspace/system/agents/reviewer.md` z frontmatter `model: gpt-5.4`. Konfiguracja agenta w pliku, nie w kodzie.
- **Scoped tool creation** — `createReviewTools(allowedBlockIds)` tworzy narzędzia z wbudowanym scope. Narzędzie istnieje tylko dla konkretnego bloku, agent nie może dodać komentarz gdzie indziej.
- **Hydratacja pozycji** — po zmianie tekstu komentarze próbują znaleźć cytat na nowo. Brak dopasowania → `_broken` → filtrowanie z widoku. Pragmatyczne podejście do ulotnego stanu.

### 04_05_apps — MCP Apps z embedded interfejsami

**Delta względem S03E05_apps:** Architektura trójwarstwowa z pełnym cyklem życia embedded apps + cross-domain chaining + graceful degradation.

- **Trójwarstwowa separacja** — host (Bun) / MCP server (Hono) / embedded apps (iframe). Każda warstwa niezależnie wymienialna. Serwer MCP podłączalny do Claude.ai.
- **Dual-audience output** — każde narzędzie zwraca `content` (tekst dla LLM) + `structuredContent` (dane dla UI). Jeden wynik, dwóch odbiorców.
- **Model Context Update z embedded apps** — appka wywołuje `app.updateModelContext()` z debouncem 120ms. Model "widzi" stan UI bez pytania. Kontekst doczepiany do wiadomości usera jako blok Markdown.
- **App-driven vs model-driven actions** — narzędzia z `visibility: ["app"]` niewidoczne dla modelu, dostępne dla UI. Nie wszystko musi przechodzić przez LLM — embedded app niezależnie wywołuje `save_todos_state`.
- **Cross-domain chaining** — instrukcje agenta celowo zachęcają do sekwencji narzędzi z różnych domen biznesowych (kampania → kupon → sprzedaż). Jedno zadanie usera = łańcuch cross-domain.
- **Graceful degradation** — regexowy fallback agent działający bez klucza API. Real-world pattern: system działa zawsze, LLM jest enhancementem.
- **Markdown jako baza danych** — `todos.md` z formatem `- [x] id | text`. Parsowanie regexem, edycja programowa i ręczna jednocześnie.
- **Security w iframe** — `sandbox="allow-scripts"` + dynamicznie wstrzykiwane CSP (`default-src 'none'`, `connect-src 'none'`). Appka może tylko uruchamiać skrypty, żadnych połączeń sieciowych.
