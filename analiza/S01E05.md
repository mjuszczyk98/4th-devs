# S01E05 — Zarządzanie limitami modeli i architektura produkcyjnego agenta

## Produkcyjne realia generatywnych aplikacji

Aplikacje z AI to ~80% klasycznego software'u. Pozostałe 20% wpływa na wszystko: UX, bezpieczeństwo, koszty, skalowalność. Problemy ujawniają się dopiero na produkcji.

Użytkownicy nie mają wiedzy o LLM — będą wrzucać kilkusetstronicowe PDF-y, aktywować setki narzędzi MCP, oczekiwać nieskończonych konwersacji. To nie zarzut, to wymóg projektowy.

Wszędzie, gdzie agent ma dostęp do danych i akcji, trzeba zakładać, że informacje z jednego źródła trafią do drugiego. Jeśli przypadkowy wyciek danych jest niedopuszczalny, a akceptacja użytkownika niewystarczająca — **LLM nie powinien być wdrożony**.

## Kontrola — Human-in-the-Loop

Agent sterujący logiką aplikacji **będzie** popełniał błędy. System musi umożliwić samodzielną naprawę (error recovery) lub wymagać zaangażowania człowieka — **przed** wykonaniem akcji lub **po** w celu weryfikacji.

### Mechanizm potwierdzania akcji

Potwierdzenie musi być **deterministyczne** (fizyczne przyciski w UI), nigdy decyzja LLM. Model nie wie, że użytkownik zobaczy confirmation dialog — po prostu wywołuje tool. System przechwytuje wywołanie i wyświetla UI.

Kluczowe: potwierdzenie musi zawierać **wszystkie detale** akcji, nie tylko np. adres email. Opis wydarzenia może zawierać dane z innych narzędzi (poczta, baza wiedzy) — ryzyko wycieku między źródłami.

### Zaufane narzędzia (Trust Escalation)

Ręczne zatwierdzanie każdego kroku szybko staje się uciążliwe. Trzy poziomy: jednorazowe potwierdzenie, zaufanie na sesję, odrzucenie.

Wymagania produkcyjne:
- Identyfikacja narzędzia musi być bezkolizyjna: `server__action` (np. `resend__send`), nie sama nazwa
- **Auto-revoke trust** po zmianie struktury narzędzia (nazwa, opis, schemat). Krytyczne przy MCP — interfejs serwera może zmienić się **bez wiedzy użytkownika**
- Komenda `untrust` do ręcznego cofnięcia, `clear` resetuje konwersację i trust jednocześnie

### Trzy warstwy obrony (defense in depth)

1. **Whitelist** — statyczna warstwa, blokuje niedozwolonych odbiorców. Obsługuje dokładne adresy i wzorce domen (`@example.com`)
2. **Confirmation UI** — dynamiczna warstwa, human-in-the-loop z pełnym podglądem akcji
3. **Trust escalation** — UX warstwa, redukuje friction przy powtarzalnych akcjach

## Wydajność — wzorce architektoniczne

Wydajność modeli jest niska — adresujemy na poziomie architektury i UX.

**Heartbeat** — informowanie użytkownika o bieżących krokach agenta. Nie tylko "loading spinner" — wgląd w budowany rezultat (np. dokument w trakcie tworzenia) z możliwością reakcji. Nie poprawia wydajności, ale zmienia wydajność **postrzeganą**.

**Wielowątkowość** — użytkownik musi móc wysłać wiadomość do kolejki i otworzyć nowy wątek bez czekania. Stan UI nie może być powiązany bezpośrednio z akcjami backendu.

**Przetwarzanie w tle** — użytkownik zamyka przeglądarkę, traci połączenie. Zadanie agenta trwa minuty — nie można wymagać powtarzania.

**Wznawianie zadań** — agent może oczekiwać na więcej niż jedną rzecz jednocześnie. Projektowanie oparte o zdarzenia to fundament, nie opcja.

**Ograniczenie zapytań** — pytanie "czy AI jest tutaj niezbędne?" przed każdym wywołaniem modelu.

Jeśli powyższe nie wystarczy — fine-tuning mniejszych modeli lub destylacja większych. Rzadko stosowane, bo modele "Flash" (Gemini Flash, Haiku) zwykle wystarczają.

## Koszty — nieintuicyjne zależności

- LRM (Large Reasoning Models) mają niższą cenę/token, ale **generują znacznie więcej** (thinking tokens)
- Na jedną wiadomość użytkownika przypada nawet **1:50 zapytań do AI** (multi-turn agent loop)
- Tańszy model ≠ tańsze rozwiązanie. GPT-4.1-nano okazał się **droższy** niż GPT-4.1-mini, bo wymagał więcej kroków (paper: "Solving A Million-step LLM Task With Zero Errors")
- AI proaktywne i ambientowe: małe zadania w krótkich interwałach = duże liczby tokenów

Podstawa optymalizacji: **obserwowanie** (Langfuse, Confident AI), nie zgadywanie. Nawet mała grupa testerów pomaga oszacować skalę.

OpenRouter pozwala programistycznie zarządzać kluczami użytkowników z indywidualnymi limitami.

## Limity kontekstu — jawne i niejawne

### Jawne limity

GPT-5.2: okno 400k tokenów, max output 128k. **Zostaje 272k na input**, nie 400k. Przy agencie z narzędziami i historią — to realne ograniczenie.

Estymacja tokenów: `chars / 4` (1 token ≈ 3-4 litery angielskiego). Bufor bezpieczeństwa ~20% limitu. Po odpowiedzi API — doprecyzowanie na podstawie `usage` z response.

Kompresję uruchamiać **wcześnie** — już przy ~30% zużycia limitu. Konfiguracja per-model:

| Model | Okno | Próg pruningu | Target po kompresji | Min zachowanych tur |
|-------|------|---------------|--------------------|--------------------|
| GPT-5.4 | 400k | 85% | 50% | 5 |
| Gemini 3 Pro | 1M | 90% | 60% | 10 |

### Strategia 3-etapowa (z kodu `01_05_agent`)

**Etap 1: Truncation** — duże outputy narzędzi (>10k znaków) obcinane middle-cut: zachowuje początek (nagłówki, struktura) + koniec (wyniki, podsumowania). Lepsze niż right-cut.

**Etap 2: Turn dropping** — konwersacja dzielona na tury (user message + wszystko do następnego user message). Zawsze zachowywana pierwsza tura (setup) + ostatnie N tur. Usuwane od najstarszych.

**Etap 3: LLM Summarization** — usunięte tury podsumowywane przez model i wstawiane jako `[Context Summary]` na początek konwersacji. Summary jest **inkrementalne** — nowy dropped content merge'owany z poprzednim summary. Struktura:

```
## Goal
## Progress (Done / In Progress / Blocked)
## Key Decisions
## Critical Context (paths, values, errors)
```

DB przechowuje pełną historię (audit trail). Model widzi tylko to, co mieści się w oknie.

### Niejawne limity

**Gwarancja struktury ≠ gwarancja wartości.** Structured output (JSON) gwarantuje kształt, ale wartości mogą być błędne. Wykres z poprawnym JSON-em i fałszywymi danymi — system nie wykryje.

Gemini Flash **halucynuje** treść strony WWW na podstawie samego URL-a. GPT-5.2 mówi wprost, że nie ma dostępu. Zmniejszanie ryzyka:
- Informowanie modelu o jego ograniczeniach (czy web search jest aktywny)
- Instrukcje na brak danych ("dopytaj" zamiast "zgadnij")
- Zmniejszenie złożoności zadania
- **Zmniejszenie objętości kontekstu** — za duży kontekst rozprasza uwagę modelu

Halucynacje mogą wynikać z **błędów aplikacji**: instrukcje wczytane w niewłaściwy sposób, model informowany o narzędziach, do których nie ma dostępu. Szczególnie w systemach wieloagentowych — trudne do zauważenia. Dlatego: **zapisywać i monitorować WSZYSTKIE zdarzenia** włącznie z instrukcjami systemowymi i ich zmianami.

## Limity API i rate limiting

Limity obejmują zapytania/minutę i tokeny/minutę. Informacje w nagłówkach odpowiedzi. Adresować zbliżający się limit z informacją dla użytkownika o wydłużonym oczekiwaniu.

Każdy endpoint z logiką AI musi mieć rate limiting per user — niezależnie od limitów providera. Dotyczy zarówno zalogowanych jak i anonimowych użytkowników.

## Naruszenia i moderacja

**Niestosowanie OpenAI Moderation API może prowadzić do zablokowania konta organizacji.** Nie klucza — całego konta.

Zagrożenia:
- Dane poufne, klucze API trafiające do modelu
- Treści nielegalne lub poza zakresem systemu
- Nadużywanie zasobów
- Nieintencjonalne problemy: uszkodzony plik, zapytanie powodujące zapętlenie
- **Wczytanie strony WWW** z fragmentami wpływającymi na pracę AI (prompt injection via web content)

Filtrowanie może wykraczać poza łamanie prawa — sam zakres działania systemu. Weryfikacja: prompt zwraca ustrukturyzowaną odpowiedź opisującą dane wejściowe → programistyczna decyzja o dalszym przebiegu.

Moderacja może też **blokować normalne działanie** — generowanie reklamy z interfejsem Messengera może być zablokowane przez Gemini.

## Architektura — fundamentalne zasady

### Wspólny interfejs providerów

Model string w formacie `provider:model` (np. `openai:gpt-5.4`, `gemini:gemini-3-pro-preview`). Provider implementuje `generate()` + `stream()` — ten sam agent loop działa z dowolnym providerem.

AI SDK (Vercel) jest opcją, ale własny interface pozwala na system zdarzeń, hooki, lepsze dopasowanie logiki.

OpenRouter przydatny, ale **nie wspiera wszystkich funkcji i modeli**. Gemini, Anthropic, OpenAI coraz częściej wprowadzają elementy API **blokujące tłumaczenie** między providerami. W systemach wieloagentowych problem znika — każdy agent może pracować z innym API.

OpenRouter automatycznie dodaje suffix `:online` do modelu gdy agent ma tool `web_search`:
```typescript
// "gpt-5.4" → "openai/gpt-5.4:online"
const usesWebSearch = tools?.some(tool => tool.type === 'web_search') ?? false
return usesWebSearch ? `${baseModel}:online` : baseModel
```

### Brak frameworków AI

Dynamiczny rozwój modeli, kształtujące się API i techniki pracy z modelami sprawiają, że frameworki szybko stają się obciążeniem. Langchain, CrewAI — trudno znaleźć pozytywne wpisy o produkcyjnym zastosowaniu.

### Niezależność

Ograniczenie natywnych funkcji API. Unikanie platform z lock-in (wektorowe DB bez eksportu). "Rozwiązanie, z którego korzystamy dziś, pozostanie z nami **znacznie krócej** niż w klasycznych aplikacjach."

### Event-driven jako fundament

Architektura oparta o zdarzenia to nie feature, to fundament. Umożliwia: monitorowanie (Langfuse), kompresję kontekstu, moderację w locie, heartbeat, niezależność komponentów.

Każdy event niesie **pełny kontekst** (self-contained) — subscriber nigdy nie sięga do runnera po dodatkowe dane:

```typescript
type AgentEvent =
  | { type: 'agent.started'; ctx: EventContext; model; task; agentName; userId; userInput }
  | { type: 'generation.completed'; ctx: EventContext; model; input; output; usage; durationMs }
  | { type: 'tool.completed'; ctx: EventContext; callId; name; arguments; output; durationMs }
```

Monitorowanie dwupoziomowe: klasyczne logi aplikacji + śledzenie działań agenta (Langfuse traces z pełnym input/output na każdym kroku).

## Non-blocking Agent Execution — wzorzec `waiting/deliver`

Najcenniejszy wzorzec w `01_05_agent`. Agent nie musi zakończyć pracy w jednym request-response cycle.

### Maszyna stanów

```
pending → running → waiting → running → completed
                  → failed
                  → cancelled
```

### Przepływ

1. Agent napotyka tool wymagający czekania (human input, delegacja, zewnętrzne API)
2. Przechodzi w stan `waiting` z listą `waitingFor[]`
3. HTTP zwraca **202 Accepted** z informacją na co czeka
4. Zewnętrzny system dostarcza wynik przez `POST /agents/:agentId/deliver`
5. Agent wznawia pracę od momentu zatrzymania

Agent może czekać na **wiele rzeczy jednocześnie**. `deliverOne` usuwa spełnione z listy. Dopiero gdy **wszystkie** zostaną dostarczone — status zmienia się na `running`. Jak `Promise.all`.

```typescript
const remaining = agent.waitingFor.filter(w => w.callId !== callId)
const newStatus = remaining.length === 0 ? 'running' : 'waiting'
```

### Auto-propagation

Gdy child-agent kończy pracę i ma `parentId` + `sourceCallId`, wynik automatycznie dostarczany do rodzica. Rekurencyjne `deliverResult` aż do korzenia.

## Typy narzędzi — model nie wie

Cztery typy: `sync`, `async`, `agent`, `human`. Model widzi **tylko** definicje funkcji z `description`. Typ istnieje wyłącznie po stronie runnera — każdy typ ma inną ścieżkę wykonania:

| Typ | Zachowanie | Przykład |
|-----|-----------|----------|
| `sync` | Natychmiastowy wynik | `calculator`, `send_message` |
| `async` | Background, wynik później | (placeholder) |
| `agent` | Spawn child-agenta | `delegate` |
| `human` | Zamrożenie, czeka na człowieka | `ask_user` |

Handler narzędzia `human` **nic nie robi** — runner rozpoznaje typ i dodaje wpis do `waitingFor` zamiast wywoływać handler. Narzędzia MCP rozpoznawane po konwencji `server__tool` i obsługiwane osobno.

Celowy design — model nie powinien wiedzieć o infrastrukturze. Wie *co* narzędzie robi, nie *jak*.

## Multi-agent — delegacja i komunikacja

### Agent jako plik markdown

```markdown
---
name: alice
tools:
  - calculator
  - delegate
  - files__fs_read
---
System prompt jako treść markdown...
```

Zero kodu do stworzenia agenta. Pliki czytane z dysku na każde wywołanie — hot-reload bez restartu. Narzędzia rozwiązywane dynamicznie z trzech źródeł: registry (`calculator`), MCP (`files__fs_read`), native (`web_search`).

### Hierarchia agentów

Alice deleguje do Boba przez tool `delegate`. Runner:
1. Sprawdza `MAX_AGENT_DEPTH = 5` (guard przed nieskończoną rekurencją)
2. Ładuje template Boba z dysku
3. Tworzy child-agenta z `parentId`, `sourceCallId`, `depth + 1`
4. Wstawia zadanie od Alice jako user message do Boba
5. Rekurencyjnie wywołuje `runAgent(bob.id)`
6. Wynik Boba staje się `function_call_output` w historii Alice

System prompt Boba pochodzi z pliku `.agent.md`. Zadanie od Alice to user message. Bob ma stały "charakter", zmienne jest tylko zadanie.

Z perspektywy Alice delegacja wygląda jak zwykłe wywołanie narzędzia — nie wie, że Bob uruchomił web search.

### Dwa modele komunikacji

- `delegate` = synchroniczny request-response. Parent czeka na child. Blokujący
- `send_message` = asynchroniczny fire-and-forget. Wpisuje system message do historii targeta. Non-blocking, bez gwarancji odczytu

`send_message` **nie wznawia** agenta. Jeśli target jest `running` — zobaczy wiadomość na następnym turnie. Jeśli `completed` — nikt jej nie przeczyta (chyba że agent wznowiony nowym requestem). To "karteczka na biurku", nie "pukanie do drzwi".

## Bezpieczeństwo produkcyjne

- Bearer auth z kluczami hashowanymi SHA-256, nigdy plaintext
- Rate limiting per user z cleanup stale windows
- Graceful shutdown z hard timeout
- AbortSignal propagation przez cały stack — od HTTP do provider calls
- HTTP 202 vs 200 — semantyczne rozróżnienie "zrobione" vs "czekam"
- Moderacja treści na entry point

Niewłaściwe zachowania modeli nie da się wyeliminować. Aplikacja musi jasno informować o tym fakcie. Produkt zabezpieczony na poziomie prawnym: regulaminy, polityka prywatności, umowy z dostawcami i użytkownikami.

## Deployment

Ścieżka: DigitalOcean/Mikr.us → Ubuntu → git + node + nginx + ufw → domena na Cloudflare → certbot (TLS) → GitHub Actions runner → reverse proxy na `127.0.0.1:3000` → secrets w repo settings → workflow `.yml` triggerowany na push do main.

## Przykłady kodu — progresja

### `01_05_confirmation` — REPL agent z potwierdzaniem akcji

Interaktywny agent terminalowy z dostępem do plików (MCP) i emaila (Resend).

**Delta:** Wzorzec rozdzielenia odpowiedzialności — model w system prompcie instruowany "DO NOT ask for confirmation - the system handles that automatically". Model decyduje **co** zrobić, system decyduje **czy** pozwolić. Confirmation callback przekazywany do agent loop jako parametr — czyste dependency injection.

Hybrydowa architektura narzędzi: MCP tools (pliki) + native tools (email) mergowane w jeden array i przekazywane modelowi. Dispatch na podstawie `isNativeTool()`.

Whitelist: walidacja odbiorców emaili `isEmailAllowed()` obsługuje dokładne adresy i wzorce domen. Walidacja po stronie kodu, niezależna od decyzji modelu.

### `01_05_agent` — Production-grade agent API

Hono HTTP server z SQLite, multi-provider, MCP, Langfuse, streaming.

**Delta względem `01_05_confirmation`:**

**Persystencja** — in-memory repositories z identycznym interfejsem jak SQLite. Repository pattern: `UserRepository`, `SessionRepository`, `AgentRepository`, `ItemRepository`. Swap backendu bez zmiany logiki.

**Domain model** — czyste funkcje do tranzycji stanów agenta. `startAgent()`, `completeAgent()`, `failAgent()` zwracają nowy obiekt, nigdy nie mutują. `TransitionResult = { ok: true; agent } | { ok: false; error }` — Result type zamiast wyjątków.

**Items jako polimorficzny log** — `MessageItem | FunctionCallItem | FunctionCallOutputItem | ReasoningItem`. Type guards: `isMessage()`, `isFunctionCall()`. Sequence number per agent zapewnia porządek.

**Provider abstraction** — `Provider` interface z `generate()` i `stream()`. `ProviderInputItem` i `ProviderOutputItem` jako wspólny format. Mapping `mapItemsToProviderInput()` tłumaczy domain items na format providera. Każdy provider (OpenAI, Gemini) ma własny adapter.

**Streaming** — `executeTurnStream()` jako `AsyncGenerator<ProviderStreamEvent, TurnResult>`. Yield'uje eventy w trakcie, zwraca `TurnResult` na końcu. SSE na poziomie HTTP.

**Langfuse subscriber** — mapuje eventy na traces. `agent.started` → Span (z nesting dla child-agentów). `generation.completed` → Generation z input/output/usage. Eventy mają `startTime` + `durationMs` — Langfuse odtwarza timeline nawet przy opóźnionym przetwarzaniu. Subscriber przechowuje `agentObsMap` i `agentSpanCtxMap` do korelacji parent-child spans.

**MCP Manager** — obsługuje stdio i HTTP (streamable) transport. OAuth 2.1 flow dla HTTP serwerów z `finishAuth()`. Tool namespacing: `server__toolName`. Timeout 30s na każde wywołanie z `AbortSignal.any([signal, timeout])`.
