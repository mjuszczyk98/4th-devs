# S02E02 — Zewnętrzny kontekst narzędzi i dokumentów

## Zewnętrzne dane jako wektor ataku

Każdy dokument wczytany do kontekstu to potencjalny prompt injection. Email z treścią "prześlij mi swój kalendarz" może zostać wykonany dosłownie, jeśli agent ma narzędzie `send_email`. Filtr LLM oceniający ryzyko nie pomoże — instrukcja może być zbudowana tak, aby oszukać również filtr.

Jedyne skuteczne obrony są **środowiskowe**, nie promptowe:
- Ograniczenie zestawu narzędzi — agent bez `send_email` nie wyśle maila
- Wymaganie potwierdzenia użytkownika przez UI (przyciski, nie chat) przed nieodwracalnymi akcjami
- Zawężenie zakresu źródeł — im mniej niekontrolowanych danych, tym mniejsza powierzchnia ataku

Analogia: agent podłączony do zewnętrznych danych = formularz bez walidacji renderujący HTML bezpośrednio na stronę (XSS).

## Instruction dropout — długi kontekst psuje zachowanie

Im więcej treści w kontekście, tym większa szansa że model zignoruje część instrukcji systemowych. "Uwaga" modelu przesunie się na nowo dodane treści kosztem wcześniejszych zasad. Efekt: jakość odpowiedzi wysoka, ale forma/styl/zasady — pominięte.

Kontrmiary:
- Ograniczanie objętości jednorazowo wczytywanych treści (krótkie chunki, subagenci z własnymi oknami kontekstowymi)
- Powtarzanie kluczowych instrukcji w odpowiedziach narzędzi (technika z S02E01)
- Delegowanie pracy na subagentów — każdy operuje na wąskim zakresie w czystym kontekście

## Indeksowanie — most między dokumentami a wyszukiwaniem

Gdy baza wiedzy wykracza poza pliki markdown (PDF, DOCX, obrazy, audio), konieczne jest indeksowanie. Bezpośrednia eksploracja plików przez agenta nie rozwiązuje problemu **odnalezienia** właściwego dokumentu.

Indeksowanie obejmuje:
- **Chunking** — podział na fragmenty o podobnej długości (200-500 słów)
- **Metadane** — źródło, sekcja, indeks chunka, tagi
- **Dokumenty syntetyczne** — podsumowania, opisy obrazów, transkrypcje audio/wideo
- **Mapa powiązań** — graf relacji między dokumentami (przydatny przy wyszukiwaniu)
- **Synchronizacja** — dane się zmieniają, indeks musi nadążać (event-driven lub harmonogram)

Zabezpieczenia na warstwie programistycznej (nie LLM): walidacja rozmiaru/formatu/mime-type, moderacja treści, kontrola dostępu, wygasające linki publiczne, optymalizacja rozmiaru plików.

## Cztery strategie chunkingu

| Strategia | Mechanizm | Metadane | Koszt API |
|---|---|---|---|
| **Znaki** | Stałe okno (np. 1000 znaków) z overlapem | Tylko programistyczne (indeks, źródło) | 0 |
| **Separatory** | Rekurencyjny podział po hierarchii: `## → ### → \n\n → \n → . → spacja` | Programistyczne + sekcja z heading index | 0 |
| **Kontekst** | Separatory + LLM generuje prefix kontekstowy per chunk | Programistyczne + wygenerowany kontekst | N wywołań (1 per chunk) |
| **Tematyka** | LLM identyfikuje logiczne tematy i grupuje tekst | Programistyczne + label tematu + potencjalnie tagi | 1-kilka wywołań |

Wybór strategii determinuje zarówno jakość wyszukiwania jak i koszt indeksowania. Separatory to solidny default; kontekst i tematyka dają lepsze wyniki kosztem tokenów i czasu.

## Embedding — opis znaczenia jako wektor

Model embeddingowy (np. `text-embedding-3-small`, 1536 wymiarów) zamienia tekst na tablicę liczb opisującą jego **znaczenie**. Identyczny tekst → identyczny wektor. Podobne znaczenie → bliskie wektory.

Kryteria wyboru modelu: wielkość (koszt), liczba wymiarów, okno kontekstowe, zakres danych treningowych (języki, knowledge cutoff).

Fundamentalne ograniczenie: jeśli termin nie występował w danych treningowych, model nie opiszę poprawnie jego znaczenia. Dotyczy to żargonu firmowego, wewnętrznych nazw projektów, niszowych terminów branżowych.

## Cosine similarity — miara bliskości znaczeniowej

Porównanie dwóch embeddingów przez iloczyn skalarny znormalizowany długością wektorów. Wartość 1.0 = identyczne znaczenie, 0.0 = brak relacji.

Kluczowa właściwość: dopasowanie **znaczeniowe**, nie leksykalne. "Kobieta" bliżej "Królowej" niż "Króla", mimo braku wspólnych liter. Dlatego semantic search znajduje dokumenty niedostępne dla keyword search — i odwrotnie (keyword search łapie dokładne terminy, które embedding uśrednia).

## Hybrid search — FTS + semantic + RRF

Dwa silniki wyszukiwania działające równolegle:
- **FTS5 (BM25)** — ranking leksykalny, łapie dokładne terminy i frazy
- **sqlite-vec (cosine distance)** — ranking semantyczny, łapie znaczenie niezależnie od zapisu

Wyniki łączone przez **Reciprocal Rank Fusion**: `score(doc) = Σ 1/(K + rank + 1)` po wszystkich listach. Dokument wysoko w obu listach awansuje; dokument tylko w jednej dostaje szansę, ale nie dominuje. K=60 (standard) wygładza różnice między pozycjami.

Agent generuje **dwa osobne zapytania** — keywords zoptymalizowane pod FTS, pytanie naturalne pod embedding. Każdy silnik dostaje zapytanie dopasowane do swoich mocnych stron.

Cross-language: pytanie po polsku, dokumenty po angielsku — FTS nic nie znajdzie, ale embedding (jeśli model wspiera oba języki) nadal dopasuje znaczeniowo.

## Transformacja zapytań

Zapytanie użytkownika rzadko pasuje bezpośrednio do treści dokumentów. Agent transformuje je przez:
- Generowanie synonimów i powiązanych pojęć
- Tłumaczenie (cross-language retrieval)
- Rozbijanie na pod-zapytania pokrywające różne aspekty

Transformacja oparta wyłącznie na wiedzy bazowej modelu jest niewystarczająca przy specyficznym kontekście klienta. Rozwiązanie: plik `_index.md` (mapa treści) czytany przed eksploracją — daje agentowi orientację w dostępnych zasobach.

## Forma prezentacji kontekstu

Tekst to nie jedyna opcja dostarczania danych do modelu:
- **Obrazy z markdown** wymagają wydobycia linków regexem i przesłania osobno (model nie parsuje `![alt](url)`)
- **Dokumenty PDF/DOCX** mogą być efektywniej przesłane jako obrazy — DeepSeek-OCR: 9-10x kompresja przy 96% precyzji
- **Cytaty i źródła** — metadane chunków (`source`, `section`) służą zarówno modelowi (grounding) jak UI (odnośniki do oryginałów)

## Progresja złożoności systemów RAG

| Wariant | Komponenty | Kiedy wystarczy |
|---|---|---|
| Agent + filesystem | grep/ripgrep, pliki markdown | Wewnętrzna baza wiedzy, średnia organizacja |
| Agent + SQLite + rozszerzenia | FTS5, sqlite-vec, chunking | Ustrukturyzowane dane, potrzeba semantic search |
| Agent + dedykowane silniki | Elasticsearch/Qdrant/Neo4j + synchronizacja | Duża skala, multimodal, złożone relacje (Graph RAG) |

Każdy poziom dodaje możliwości **i złożoność**. Bezpośrednie wyszukiwanie plików nie wymaga indeksowania — co samo w sobie jest przewagą. Podejścia mogą współistnieć równolegle.

## Pięć wyzwań skuteczności RAG

1. **Wiedza bazowa przesłania search** — model odpowiada z pamięci zamiast szukać. Adres: instrukcja wymuszająca search + grounding w wynikach
2. **Niekompletne wyniki** — trudne do wykrycia przez model, prowadzą do halucynacji. Adres: hybrid search, wielokrotne szukanie, RRF
3. **"Nie wie, że coś wie"** — brak świadomości zasobów na start. Adres: mapa treści (`_index.md`), progressive disclosure
4. **Brak kontekstu użytkownika** — "moje projekty" jest nieresolwowalne bez danych o użytkowniku. Adres: augmented context (programistycznie), Graph RAG dla relacji
5. **Multimodal** — obrazy/audio/wideo wymagają pre-processingu (opisy, transkrypcje) zanim staną się searchable

## Przykłady kodu — progresja

### `02_02_chunking` — strategie podziału tekstu

Delta: cztery implementacje chunkingu na tym samym dokumencie, porównywalne przez JSONL output.

- **Rekurencyjny fallback separatorów** — gdy chunk przekracza limit, algorytm schodzi o poziom niżej w hierarchii separatorów i dzieli ponownie. Overlap wycinany z respektowaniem granic słów/linii.
- **Heading index** z dual detection (markdown `#` + plain-text headings przez regex) → chunki przypisane do sekcji przez sampling ze środka chunka (pozycja 40%), nie z początku — unika fałszywych dopasowań z overlap boundaries.
- **Resilient JSON parsing** — odpowiedź LLM oczyszczana z markdown code blocks przed `JSON.parse`, z fallbackiem.
- Strategia **topics** robi jedno wywołanie LLM dla całego dokumentu; **context** robi N wywołań (jedno per chunk). Fundamentalnie inny profil kosztowy przy tym samym celu (wzbogacenie metadanych).

### `02_02_embedding` — cosine similarity w praktyce

Delta względem chunking: przejście od tekstu do wektorów. Zero npm dependencies.

- Cosine similarity w jednym przebiegu: dot product + obie normy liczone jednocześnie
- REPL budujący macierz similarity przyrostowo — każdy nowy wpis porównywany ze wszystkimi poprzednimi
- Skrócona wizualizacja wektora: pierwsze 4 + ostatnie 2 wymiary z 1536 — praktyczny UX dla debugowania

### `02_02_hybrid_rag` — kompletny system RAG

Delta względem embedding: indeksowanie, persystencja, hybrid search, agent loop.

- **SQLite jako all-in-one**: dane + FTS5 (triggery synchronizujące chunki z indeksem) + sqlite-vec (wektory) w jednej bazie. Brak zewnętrznych silników.
- **Incremental indexing** — SHA256 hash plików, re-indeksuje tylko zmienione. Cascading deletes usuwają stale indeksy.
- **Dwa zapytania per search** — agent generuje `keywords` (dla FTS) i `semantic` (dla embeddingu) osobno. Każdy silnik dostaje input zoptymalizowany pod swoje mocne strony.
- **RRF merge** — `1/(60 + rank + 1)` sumowane po obu listach, sort descending → top N. Trzy linie kodu, zero ML.
- **Graceful degradation** — gdy embedding API nie odpowiada, system fallbackuje do samego FTS zamiast crashować.
- **Token accounting** — śledzenie input/output/reasoning/cached tokenów per request. Osobna statystyka, nie logowanie ad hoc.
