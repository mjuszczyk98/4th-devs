# S04E02 — Aktywna współpraca z AI

## Wybór interfejsu jako decyzja architektoniczna

Interfejs determinuje resztę projektu — model, integracje, personalizację, autonomię. Nie ma jednego uniwersalnego rozwiązania, ale raczej kompozycję dopasowaną do scenariusza.

| Scenariusz | Optymalny interfejs | Uzasadnienie |
|---|---|---|
| Indywidualna praca developera | CLI (Claude Code, Open Code) | Pełna personalizacja, subagenci, skróty, hooki |
| Osoby nietechniczne | Web (Claude.ai, ChatGPT) | Niski próg wejścia, Artifacts, MCP Apps |
| Kontekst zespołowy | Komunikator (Slack, Discord) | Użytkownicy zostają w znanym środowisku |
| Pełna kontrola UX | Dedykowane rozwiązanie | Brak ograniczeń platformy, ale znaczny koszt budowy |

Kluczowe: te opcje się nie wykluczają. System produkcyjny to często **kompozycja** kilku z nich.

## Ograniczenia gotowych platform

Przykład integracji MCP z Claude.ai obnaża konkretne luki:

- **Brak samplingu** — serwer MCP nie może zainicjować komunikacji z modelem po stronie klienta. Koszty API poza subskrypcją.
- **Ograniczona personalizacja instrukcji** — system prompt sterowany głównie przez opisy narzędzi i ich wyniki, nie przez dedykowany panel konfiguracji.
- **Brak dwukierunkowej komunikacji z użytkownikiem** — agent nie może prosić o dodatkowe informacje ani raportować postępu w tle.
- **Uprawnienia** — rozbudowane reguły dostępu (np. różne kampanie, różni klienci) są trudne do zaimplementowania w ramach gotowego klienta MCP.
- **Pełne wsparcie protokołu MCP jest rzadkie** — z [listy klientów MCP](https://modelcontextprotocol.io/clients) pełne wsparcie deklaruje niewiele pozycji.

Decyzja: podłączyć się do istniejącego ekosystemu (szybkie wdrożenie, ograniczenia) czy zbudować własne (pełna kontrola, koszt budowy).

## Cztery filary personalizacji interfejsu

Każdy interfejs AI potrzebuje mechanizmów personalizacji wykraczających poza sam czat:

1. **Profile (subagenci)** — podział na wyspecjalizowane konteksty z własnymi ustawieniami, modelem, narzędziami. Przełączanie przez menu, skróty, mention (`@`), automatyczne przez AI.
2. **Umiejętności (skills)** — predefiniowane instrukcje wstrzykiwane do konwersacji na żądanie użytkownika lub decyzją modelu. Wymagają wyszukiwania i grupowania przy większej liczbie.
3. **Narzędzia (tools)** — MCP + natywne integracje. Kluczowa nie jest sama funkcjonalność, ale pełny cykl: prezentacja danych wejściowych → potwierdzenie → postęp → błędy → wynik. Plus możliwość wstrzymania/anulowania.
4. **Workflow** — powtarzalne sekwencje akcji, hooki, zaplanowane zadania. Obecnie wspierane przez pojedyncze interfejsy.

Jakość implementacji tych mechanik (nie ich obecność) decyduje o użyteczności produktu.

## Mikro-akcje — niedoceniana przestrzeń

Pojedyncze akcje przypisane do skrótów klawiszowych, gestów czy wyzwalaczy — najszybszy ROI z wdrożenia AI. Zwykle wystarcza prosty skrypt + skrót klawiszowy.

| Typ akcji | Przykład | Wymagania |
|---|---|---|
| Czytanie zaznaczenia | TTS na zaznaczonym tekście | Model audio (może lokalny) |
| Wyjaśnienie zaznaczenia | Definicja/kontekst słowa kluczowego | LLM lub wyszukiwarka |
| Transformacja zaznaczenia | Tłumaczenie, korekta, parafraza, ekstrakcja | LLM + kontekst |
| Transformacja kontekstowa | Auto-dopasowanie do domeny (np. GitHub → styl dokumentacji) | Detekcja kontekstu + LLM |
| Wizualizacja zaznaczenia | Tekst → diagram/obraz w wybranym stylu | Model obrazowy |
| Powiązania z bazą wiedzy | Zaznaczenie → zapytanie → deep-link do notatki | Własna baza + x-scheme-url |

Koncepcja rozszerza się na automatyzacje mobilne (Siri Shortcuts), watchery katalogów, integracje z natywnymi API urządzeń.

## Meta-prompt jako wzorzec projektowy

Meta-prompt to instrukcja dla modelu, której celem jest **wygenerowanie innej instrukcji** poprzez rozmowę z użytkownikiem. Nie jest to autonomiczny proces — rola człowieka jest fundamentalna.

Struktura meta-promptu składa się z trzech warstw:

- **Dane** — kategorie informacji do zebrania: cel, zakres, styl, narzędzia, wzorce, modele mentalne, ograniczenia, format, wyjątki. Wszystko, czego model nie wie z natury.
- **Generator** — wiedza o dobrych praktykach tworzenia promptów: struktury, zasady specjalizacji/generalizacji, przydatne wyrażenia. Model stosuje tę wiedzę automatycznie.
- **Rezultat** — szablon końcowej instrukcji ze stałymi sekcjami: tożsamość, proces myślowy, zasady i ograniczenia, wiedza zewnętrzna, styl wypowiedzi.

### Sekcje implementacyjne meta-promptu

| Sekcja | Rola |
|---|---|
| **Proces** | Informuje model o celu: przeprowadzenie użytkownika przez pytania do instrukcji |
| **Strategia** | Zasady realizacji celu z dopasowaniem do sytuacji |
| **Dopasowanie** | Reguły specjalizacji promptów pod domenę (kod, marketing, support...) |
| **Format** | Szablon + komponenty końcowej instrukcji + sposób prezentacji |
| **Zasady** | Conducting rozmowy: czekaj na odpowiedź, pogłębiaj niejasności, nie zgaduj |
| **Natywne funkcjonalności** | Generowanie nie tylko promptu, ale też ustawień agenta (model, narzędzia) |
| **Generowanie** | Selekcja — brać pod uwagę wyłącznie elementy istotne dla bieżącego kontekstu |
| **Krytyczne zasady** | Podkreślenie najważniejszych reguł na końcu instrukcji |

### Scenariusze zastosowań meta-promptów

- **Onboarding** — automatyczne generowanie instrukcji dopasowanych do kontekstu użytkownika na podstawie jego danych (opisy produktów, zdjęcia, profil).
- **Generowanie obrazów** — dynamiczne instrukcje stylu dopasowane do produktu i tonu marki.
- **Agenci i subagenci** — Claude Code już używa meta-promptów do generowania skilli i subagentów.
- **Optymalizacja** — automatyczne doszlifowywanie promptów agentów klasyfikujących, transformujących itd.

Pomimo rosnących capabilities modeli, jakość promptów wciąż bezpośrednio przekłada się na skuteczność systemu. Meta-prompt **automatyzuje proces optymalizacji**, ale nie eliminuje potrzeby wiedzy o tym, jak dobre prompty konstruować.
