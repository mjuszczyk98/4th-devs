# S03E03 — Kontekstowy feedback wspierający skuteczność agentów

## Wyzwalacze autonomicznych działań agenta

Agent może reagować na pięć typów triggerów: **wiadomości** (człowiek/agent), **hooki** (zdarzenia wewnętrzne aplikacji), **webhooki** (zdarzenia zewnętrznych usług), **cron** (czas/daty) i **heartbeat** (regularne sprawdzanie stanu).

Wszystkie mogą trafiać do **jednego punktu wejścia** — agent dynamicznie dopasowuje zachowanie do zadania i otoczenia. Każda akcja to domyślnie nowa sesja, ale główna sesja użytkownika może być "nieskończona" z kompresją kontekstu (Observational Memory z S03E02).

Heartbeat z proaktywnym czytaniem `tasks.md` — agent sprawdza czy wymaga działania. Jeśli nie, nic się nie dzieje (invisible). Jeśli tak — wykonuje proaktywnie na bazie bieżącego kontekstu (czas, lokalizacja, aktywność użytkownika).

## Od reaktywnego do proaktywnego

L02E01: otoczenie wpływa na agenta **tylko podczas trwającej interakcji** z użytkownikiem. S03E03: pełna integracja z otoczeniem + **uniezależnienie od bezpośrednich działań człowieka**. Obecność mechanizmów proaktywnych nie wyklucza interakcji z człowiekiem.

## Stan otoczenia: aktywny vs pasywny discovery

Agent odkrywa otoczenie na dwa sposoby:
- **Aktywnie** — wywołuje narzędzia (przeglądarka, search, API)
- **Pasywnie** — otrzymuje metadane programistycznie wstrzykiwane do wiadomości

Metadane mogą obejmować: czas, lokalizację, pogodę, status urządzenia, otwarte aplikacje. Klucz: **nie wysyłać wszystkiego za każdym razem** — scope'ować do potrzeb bieżącej interakcji.

## Feedback w pętli narzędzi

Najważniejszy koncept lekcji: **informacja zwrotna wewnątrz tool loop podnosi skuteczność agenta zarówno w obrębie sesji, jak i między sesjami.**

Trzy warstwy, rosnąco zaawansowane:

### 1. Statyczne sugestie po błędach

Po porażce narzędzia — wstrzyknięcie contextual hintu do wyniku narzędzia. Nie zmienia promptu, wzbogaca output:

```
Error: querySelector returned null
[feedback] The expected DOM element is missing. The page structure may have changed.
```

### 2. Dynamiczne interwencje

Gdy 2+ porażki z rzędu → wstrzyknięcie **dodatkowej wiadomości użytkownika** do konwersacji, zachęcającej do zmiany strategii (screenshot, inny selector). Gdy agent odzyska po porażkach → sugestia zapisania discoveries do pliku.

To nie jest prompt engineering — to **runtime intervention** modyfikujący konwersację na podstawie obserwowanego zachowania.

### 3. Hook lifecycle z guard railsem

| Hook | Sytuacja | Akcja |
|------|----------|-------|
| `beforeToolCall` | Przed wywołaniem narzędzia | Nagranie stanu (np. ścieżka pliku audio) |
| `afterToolResult` | Po wyniku narzędzia | Aktualizacja flag postępu, detekcja błędów |
| `beforeFinish` | Agent chce zakończyć | **Blokada** jeśli wymagane kroki nie ukończone |

Pełna lista hooków (AI SDK): `onStart → onStepStart → onToolCallStart → onToolCallFinish → onStepFinish → onFinish`. Projekt Pi rozszerza o streaming, context building, session compression, user activity, agent statuses. Najprostszy przykład: generowanie nazwy sesji (jak auto-rename "New Chat" → opisowy tytuł w ChatGPT).

`beforeFinish` to deterministyczny guard nad niedeterministycznym agentem — wymusza wykonanie pełnego flow (listen → feedback → save session → update profile) przed zakończeniem. Wyjątek: błędy narzędzi lub przekroczenie max turns.

## Persistent learning między sesjami

Dwa komplementarne wzorce:

**Instruction files** — agent zapisuje "discoveries" (co zadziałało na danej stronie/domenie) do plików `.md`. Przy kolejnym uruchomieniu system prompt mówi: "najpierw sprawdź czy istnieje instruction file". Agent uczy się struktury świata, nie historii konwersacji.

**Profile accumulation** — profil ucznia z `weakAreas` akumuluje odkryte `trait_id` między sesjami. Profil pozostaje mały (brak sessions array), pełne sesje idą do osobnych plików w `sessions/`. Agent personalizuje feedback na podstawie historycznych słabości.

## Narzędzia wywołujące dodatkowe API

Narzędzia nie są tylko proxy do danych — mogą **samoistnie wywoływać dodatkowe zapytania do modeli**. Przykłady: narzędzie `listen` wywołuje osobne zapytanie Gemini (ASR + analiza), narzędzie `feedback` wewnętrznie wywołuje `speak` (TTS).

MCP przewiduje to przez **Sampling** — serwer MCP może odwrócić komunikację i poprosić klienta o wywołanie API. Rzadko wspierane, ale adresuje dokładnie ten przypadek.

## Fazowanie agenta z różnymi narzędziami

Agent z fazami: każda faza z **własnym promptem i zestawem narzędzi**. Model dostaje tylko to co potrzebuje — redukcja complexity i halucynacji.

Faza 1 (tworzenie eventów): narzędzia do kontaktów, miejsc, web search, kalendarza. Faza 2 (powiadomienia): narzędzia do kalendarza, tras, powiadomień. **Brak overlapu** = model nie próbuje wysyłać powiadomień w fazie tworzenia.

## OpenAI Responses API vs Gemini Interactions API

| Cecha | OpenAI Responses | Gemini Interactions |
|-------|-----------------|---------------------|
| Kontynuacja sesji | `previous_response_id` | `previous_interaction_id` |
| Wymuszanie schema | `strict: true` na tool params | `response_format` jako JSON Schema na całą odpowiedź |
| Audio output | Nie natywnie | `response_modalities: ['AUDIO']` — base64 PCM |
| Thinking control | Brak | `thinking_level: 'low'/'medium'/'high'` |
| Typ autentykacji | `apiKey` w konstruktorze | `x-goog-api-key` header |
| Output types | `text`, `function_call` | `text`, `function_call`, `thought`, `audio` |
| Store | `store: true/false` | Brak odpowiednika |

`previous_response_id` / `previous_interaction_id` — kontekst trzymany po stronie API. Klient nie wysyła całej historii — API pamięta. Drastyczna redukcja token usage.

`response_format` w Gemini wymusza strict JSON schema na **całej odpowiedzi modelu**, nie tylko argumentach narzędzi. Gwarantuje strukturę np. `{ transcript, confidence, strengths, issues, segments }`.

Gemini TTS: model `gemini-2.5-flash-preview-tts` z `speech_config: { voice, language }`. Zwraca `{ type: 'audio', data: '<base64>', mime_type: 'audio/pcm' }`. PCM → WAV konwersja w kodzie.

## Defensywne parsowanie outputów modelu

`jsonrepair` — auto-naprawa niepoprawnego JSON z argumentów narzędzi. `parseJsonLoose()` — wyciąganie JSON z markdown fences, fragmentów tekstowych, błędnych struktur. Obie techniki to **must-have** w każdym agentic system — model regularnie generuje niepoprawny JSON.

## Proaktywne zachowanie: architektura

Nieskończona sesja + heartbeat → agent czyta `tasks.md` → jeśli zadanie wymaga działania → wykonuje je w kontekście bieżącego otoczenia (lokalizacja, kalendarz, pogoda). Użytkownik dostaje spersonalizowane powiadomienie ("wyjdź za 12 min, weź tramwaj #3, pada deszcz") bez żadnego działania z jego strony.

## Web scraping: zmiana narracji

Narracja przesuwa się od blokowania agentów (AI Labyrinth) do traktowania "na równi" (Markdown for Agents, WebMCP). Nadal jednak powszechne są strony bez API (np. Goodreads — zamknięte rejestracje developerów).

**Gradacja automatyzacji:** powtarzalne zadania na dużej skali → rozważyć klasycznego bota (bez LLM). Gdy to nie wystarcza → agent AI z Playwright/Puppeteer. Na większą skalę → Browserbase, kernel.sh.

## "Connecting the dots" = wartość biznesowa

Agent łączy **szczątkowe, rozproszone informacje** (imię, "cafe on Planty", "tomorrow") w pełny event z adresem, gośćmi, opisem. Wzbogacenie oryginalnej wiadomości = ustandaryzowanie i podniesienie jakości procesów (np. sprzedażowych, marketingowych).

Bezpieczeństwo: przy tworzeniu eventów agent może pomylić adresy / wpuścić poufne dane do opisu. Rozwiązanie: deterministyczne zarządzanie kontaktami — ten etap nie przez agenta.

## Ofensywne projektowanie

Zamiast pytać tylko "gdzie system ma problemy" — pytać "jakie opcje przeniosą go na wyższy poziom". Przykład: interfejs głosowy (nawet prosty STT→LLM→TTS) w proaktywnym systemie okazuje się bardzo użyteczny.

## Content overflow → file save

Gdy treść strony przepełnia context window — zapisywana do pliku, agent przeszukuje przez `fs_search`. Alternatywa: subagent.

## Rola człowieka

Agent działający w tle musi mieć interfejs dla sytuacji gdy akcja autonomiczna niemożliwa. Konflikty nazw, brakujące dane, błędy integracji — wymagają rozstrzygnięcia.

Hooki mogą obejmować **weryfikację po stronie człowieka** — nie tylko potwierdzenie niezaufanej akcji, ale uzupełnienie brakujących informacji. System może też sam przeanalizować podjęte działania i zdecydować o potrzebie wsparcia.

Problemy z agentami często leżą po stronie użytkowników. Adresowanie tego na poziomie komunikacji produktu / onboardingu = wpływ na skuteczność porównywalny z optymalizacją architektury.

## Przykłady kodu — progresja wiedzy

### 03_03_calendar — Fazowanie + symulowane dane + webhooki

**Delta względem S03E02:** Agent reagujący na webhooki zewnętrznych usług, z fazową architekturą i w pełni symulowanym otoczeniem.

Nowe koncepty:
- **Dual-phase agent** — dwie fazy, każda z własnym promptem i zestawem narzędzi. Model dostaje tylko narzędzia potrzebne w danej fazie.
- **Symulowane dane relacyjne** — kontakty z preferences, miejsca z tags i hours, trasy z 3 opcjami (walk/drive/transit), pogoda po godzinach. Zero zewnętrznych API — pełne testowanie end-to-end.
- **Mutowalny environment state** — `setTime()` + `setUserLocation()` symulują upływ czasu i zmianę lokalizacji między krokami. `buildMetadata()` generuje blok `<metadata>` wstrzykiwany do każdej wiadomości.
- **Fuzzy search ze scoringiem** — `find_event` weighted: title match (token overlap + substring) + time proximity (odległość w minutach). Handles rozbieżności między webhook title a calendar title.
- **Webhook payloads jako trigger** — `event.upcoming` payload z `eventTitle`, `startsAt`, `minutesUntilStart` → agent sam decyduje o trasie i pogodzie → wysyła powiadomienie.
- **Rich terminal observability** — ~290 linii loggera z kolorami, tabelami, ikonami. Nie cosmetic — debugowalność systemu agentowego w batch mode.

### 03_03_browser — Feedback-driven interventions + self-learning

**Delta względem `calendar`:** Agent z runtime feedback system, interwencjami, i persistent learning między sesjami.

Nowe koncepty:
- **Feedback tracker** — per-tool history sukcesów/porażek. `consecutiveFailures()` + `generateHints()` produkują contextual porady na podstawie typu błędu (timeout, null selector, invalid JSON).
- **Runtime interventions** — dynamiczne wstrzykiwanie wiadomości do konwersacji: screenshot tip po 2+ porażkach, discovery tip po odzyskaniu. Nie modyfikują promptu — modyfikują input do modelu.
- **Instruction files jako pamięć** — `instructions/{site}.md` z gotowymi recipes (JS snippets), `instructions/{site}-discoveries.md` zapisywane przez agenta po odzyskaniu z porażek. System prompt: "zanim nawigujesz, sprawdź czy istnieje instruction file".
- **Structural DOM dump** — po nawigacji zapisywane dwa pliki: pełny tekst z numeracją linii + uproszczona struktura DOM (tag, klasy, data-testid, role, aria-label). Agent szuka selectorów przez `fs_search` na `.struct.txt` zamiast zgadywać z HTML.
- **Session persistence (Playwright)** — login flow z `headless: false` → `context.storageState()` → zapis do JSON. Kolejne uruchomienia w headless z załadowanymi cookies. Custom User-Agent + disable automation detection.
- **`jsonrepair`** — auto-naprawa niepoprawnego JSON z argumentów narzędzi przed wykonaniem. Model regularnie generuje zepsuty JSON (trailing commas, markdown fences).
- **MCP integration z tool coalescing** — narzędzia z MCP file server (`fs_read`, `fs_search`, `fs_write` via stdio) wrzucane do tego samego tool registri co browser tools. Model nie wie o źródle narzędzi.
- **`previous_response_id` chaining** — kontynuacja konwersacji bez resendowania historii. API trzyma kontekst.

### 03_03_language — Hook lifecycle + multimodalność + Gemini API

**Delta względem `browser`:** Agent z hook-based lifecycle control, multimodalnym I/O (audio in → text+audio out), i Gemini Interactions API.

Nowe koncepty:
- **Hook lifecycle z phase tracking** — `beforeToolCall` nagrywa stan, `afterToolResult` aktualizuje flagi (`listen_done`, `feedback_done`, `session_saved`), `beforeFinish` wymusza pełny flow. `tryCompletePhase()` snapshottuje ukończoną fazę i resetuje state machine.
- **Gemini Interactions API** — raw fetch do `/v1beta/interactions`. `response_format` jako JSON Schema wymuszający strukturę całej odpowiedzi. `response_modalities: ['AUDIO']` dla TTS. `thinking_level` kontrolujący głębokość reasoning.
- **`response_format` jako schema enforcement** — nie tylko na argumentach narzędzi (jak `strict: true` w OpenAI), ale na **całej odpowiedzi modelu**. Narzędzie `listen` dostaje schemat `{ transcript, confidence, strengths, issues, segments }` i model zwraca dokładnie tę strukturę.
- **Gemini TTS** — model `gemini-2.5-flash-preview-tts` z `speech_config: { voice: 'kore', language: 'en-us' }`. Output: raw PCM → ręczna konwersja do WAV (44-bajtowy header budowany w kodzie).
- **Tool-chaining w narzędziach** — `feedback` wewnętrznie wywołuje `speak` (kolejne zapytanie API). Narzędzia orkiestrują inne narzędzia, nie tylko proxy'ują dane.
- **Trait taxonomy** — ustandaryzowana klasyfikacja błędów: `grammar.articles`, `grammar.verb_tense`, `pronunciation.stress`, `pronunciation.vowels`, `fluency.fillers`. Umożliwia akumulację `weakAreas` w profilu między sesjami.
- **`parseJsonLoose()`** — wyciąganie JSON z: markdown fences, fragmentów tekstowych, błędnych struktur. Kolekcja kandydatów (full string, substring między `{` a `}`, substring między `[` a `]`) → parse first success.
- **Profile accumulation** — po sesji agent aktualizuje `weakAreas` w `profile.json` z odkrytych `trait_ids`. Profil małe (role, goals, weakAreas), sesje w osobnych plikach `sessions/*.json`.
- **Multi-phase w jednej sesji** — po ukończeniu fazy (listen → feedback → save), state machine się resetuje i agent może analizować kolejny plik audio. Aktualizacja profilu raz na koniec wszystkich faz.
