# S02E03 — Dokumenty i pamięć długoterminowa jako narzędzia

## Baza wiedzy tworzona vs podłączona

Standardowe podejście: agent przeszukuje istniejące dokumenty (pisane dla ludzi) przez chunki i embeddingi. Problem — agent **ma szansę** trafić na właściwe dokumenty, ale nie ma gwarancji.

Odwrócone podejście: baza wiedzy **zbudowana dla agentów**. Agent nie szuka — wie, gdzie patrzeć. Nie generyczna instrukcja "przeszukaj bazę", lecz "instrukcje obsługi narzędzi znajdziesz w `./workflows`". Dokument prowadzi do kolejnego dokumentu — agent podąża za wskazówkami jak za importami w kodzie.

Konsekwencje:
- Eliminacja problemu **odnajdywania** informacji — każdy agent widzi tylko to, czego potrzebuje
- Brak problemów z formatami i rozmiarem — dokumenty generowane są pod agenta
- Kompozycja — zmiana jednego wpisu propaguje się przez cały proces
- Wymaga dyscypliny w utrzymaniu struktury (AI może pomagać, ale nie przejmuje procesu)

## Cztery wymiary nawigacji po wiedzy

Kod źródłowy naturalnie posiada te cechy. Dokumenty biznesowe — prawie nigdy.

| Wymiar | Opis | W kodzie | W dokumentach |
|--------|------|----------|---------------|
| **Perspektywa** | Widok z lotu ptaka | `ls`, struktura katalogów | Rzadko istnieje |
| **Nawigacja** | Przeszukiwanie nazw i treści | `grep`, search | Wymaga RAG |
| **Powiązania** | Odnośniki między zasobami | Importy, ścieżki | Prawie brak |
| **Szczegóły** | Oryginalna treść | Odczyt pliku | Odczyt dokumentu |

Wniosek: skuteczna ekspozycja kontekstu opiera się o **informacje zawarte wewnątrz dokumentów** (linki, referencje), nie o zewnętrzny indeks. Wyjątki: Wikipedia, Digital Garden (Obsidian/Logseq) — bogate linkowanie wewnętrzne.

## Observational Memory

Podejście Mastra.ai osiągające 94.87% na LongMemEval z gpt-5-mini — bez wektorów, bez grafów, bez wyszukiwania semantycznego.

### Mechanika

Okno kontekstowe podzielone na dwa bloki:
- **Observations block** — skompresowane obserwacje z przeszłych interakcji
- **Raw messages block** — nowe, nieprzetworzone wiadomości

Dwa agenty zarządzają pamięcią:
- **Observer** (próg: 30k tokenów raw messages) — kompresuje wiadomości do zwięzłych obserwacji, czyści raw block, appenduje do observations
- **Reflector** (próg: 40k tokenów obserwacji) — garbage collection obserwacji, usuwa nieistotne

Obserwacje to sformatowany tekst z:
- **Temporal model** — trzy daty per obserwacja (data obserwacji, data referencyjna, data relatywna)
- **Priorytetyzacja** — `🔴` critical, `🟡` potentially important, `🟢` informational
- Grupowanie po dacie, inline timestamps

### Prompt caching jako ukryty killer feature

- Wiadomości appendują do stałego prefixu → **full cache hit** przez większość czasu
- Po Observer: nowe obserwacje appendują do istniejących → **partial cache hit**
- Cache invalidation tylko przy Reflector (rzadko)

Grafowy RAG generuje nowy kontekst per zapytanie — nie korzysta z prompt cache.

### Filozofia

> "Text is the universal interface"

Obserwacje to tekst, nie structured objects. LLM lepiej operuje na tekście niż na grafach/wektorach. Kompresja = utrata informacji, ale wyniki benchmarkowe pokazują, że utrata jest akceptowalna dla pamięci konwersacyjnej.

## Graph RAG — kiedy uzasadniony

Bazy grafowe (Neo4j) łączą: indeksowanie treści + wyszukiwanie pełnotekstowe (BM25) + semantyczne (vector) + nawigację po relacjach. Hybrydowy RAG w logice agentów.

Graf właściwości Neo4j:
- **Wierzchołki (nodes)** — obiekty z właściwościami i etykietami
- **Krawędzie (edges)** — kierunkowe połączenia z typem i właściwościami, łączą dokładnie dwa wierzchołki

Uzasadnione zastosowanie: **wielopoziomowe powiązania** między informacjami rozproszonymi w wielu dokumentach. Koszt (infrastruktura, złożoność, czas odpowiedzi) sprawia, że nie jest to domyślny wybór.

### OM vs Graph RAG — kiedy co

| Kryterium | Observational Memory | Graph RAG |
|-----------|---------------------|-----------|
| Infrastruktura | Zero (tylko LLM) | Neo4j + embedding API + LLM |
| Powiązania | Implicit (w tekście) | Explicit (krawędzie) |
| Skalowalność wiedzy | Ograniczona (kompresja = utrata) | Nieograniczona (graf rośnie) |
| Prompt caching | Natywne | Brak |
| Pamięć konwersacyjna | Doskonała | Nie do tego służy |
| Multi-document navigation | Słaba | Doskonała |
| Temporal reasoning | Wbudowane | Brak |

Nie wykluczają się — OM dla pamięci konwersacyjnej, graf dla współdzielonej bazy wiedzy.

## Dekompozycja na sesje

Wzorzec: ten sam agent uruchomiony w **oddzielnych sesjach** z różnymi źródłami, ale tą samą instrukcją. Wyniki zapisywane do wspólnego katalogu. Kolejny agent czyta output poprzedniego.

Zalety:
- Agent skupiony na jednym zadaniu → wyższa jakość
- Optymalne kosztowo — płacimy za wielokrotne wczytanie instrukcji, nie za ogromny kontekst
- Naturalna kompozycja — każdy etap niezależny, łatwy do debugowania i wymiany

Krytyczne ograniczenie: treści generowane dla zewnętrznych odbiorców (newsletter, kampanie mailingowe) **muszą** być weryfikowane przez człowieka. Agent = tworzenie szkicu, nie wysyłka.

## Deep Research / Deep Action

Wzorzec iteracyjnego pogłębiania:
1. Doprecyzowanie zapytania (pytania + parafraza + wstępne przeszukiwanie)
2. Dekompozycja na podzadania
3. Pętla: szukaj → analizuj → identyfikuj braki → szukaj głębiej
4. Synteza finalnego dokumentu

Zmiana nazwy z "deep research" na **"deep action"** rozszerza zastosowania — audyty, generowanie kodu, analiza logów. Wszystko, co wymaga eksploracji przez pogłębianie zapytań i nie musi być natychmiastowe.

## Przykład kodu: `02_03_graph_agents`

### Pipeline indeksowania

Dokument → chunking (4000 znaków, 500 overlap, sekcje markdown) → embeddingi (text-embedding-3-small, batch po 20) → ekstrakcja encji przez LLM → embeddingi encji → zapis do Neo4j w jednej transakcji.

Skip-if-unchanged: hash SHA256 treści porównywany z hashem w grafie. Zmieniony dokument → usunięcie starego + pełne reindeksowanie.

### Deduplikacja encji — wielowarstwowa

LLM generuje warianty tego samego konceptu ("LLM", "Large Language Model", "large language models"). Bez deduplikacji graf staje się szumem.

**Warstwa 1 — normalizacja per chunk:** title case z zachowaniem akronimów, clamp typów do dozwolonego zbioru, filtracja self-ref i wiszących krawędzi.

**Warstwa 2 — globalna deduplikacja across chunks:**

```js
const dedupeKey = (name) => singularize(name.trim().toLowerCase());
// "Large Language Models" → "large language model"
// Grupowanie po kluczu, keep longest name + longest description
```

Remap wszystkich relacji i referencji na kanoniczne nazwy. Deduplikacja identycznych krawędzi po kluczu `source→type→target`.

**Warstwa 3 — Neo4j MERGE:**

```js
MERGE (e:Entity {name: $name, type: $type})
ON MATCH SET e.description = CASE WHEN size(e.description) < size($description)
                                  THEN $description ELSE e.description END
```

Encja z dłuższym opisem wygrywa. Constraint `(name, type) UNIQUE` jako ostatnia linia obrony.

**Warstwa 4 — runtime kuracja:** `audit` wykrywa duplikaty substring matchingiem i sieroty. `merge_entities` transakcyjnie rewiruje krawędzie i kasuje duplikat.

### Hybrydowy search z RRF

Dwa kanały równolegle: BM25 (full-text na Chunk.content) + vector (cosine na Chunk.embedding). Fuzja przez Reciprocal Rank Fusion:

```js
score += 1 / (RRF_K + rank + 1)  // RRF_K = 60
```

Graceful degradation — błąd vector search nie crashuje systemu, FTS działa samodzielnie.

Po search: entity enrichment — chunki wzbogacane o encje przez krawędzie `MENTIONS`. Agent dostaje zarówno tekst (jak klasyczny RAG), jak i "uchwyty" do dalszej nawigacji po grafie (`explore`, `connect`).

### Narzędzia agenta — pełny zestaw

Retrieval: `search` (hybrid), `explore` (sąsiedzi encji), `connect` (shortest path), `cypher` (read-only, z walidacją na write keywords).

Curation: `learn` (indeksuj plik lub tekst), `forget` (usuń dokument + orphaned entities), `merge_entities`, `audit`.

Safety: Cypher odrzuca zapytania z CREATE/MERGE/DELETE/SET/DROP. Appenduje LIMIT jeśli brak. Agent nie może modyfikować grafu surowym Cypherem.
