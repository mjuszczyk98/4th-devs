# S04E03 — Kontekstowa współpraca z AI

## API jako punkt wejścia do integracji

Wspólny mianownik wszystkich narzędzi nadających się do integracji z AI: **dostępność API**. Decyzja o integracji zaczyna się od weryfikacji: czy usługa wystawia endpointy, webhooki, x-scheme-url, CLI, czy pozwala na scrapowanie treści. Struktura zwracanych danych bywa niekompletna lub zbyt ograniczona — to trzeba sprawdzić przed projektowaniem scenariusza.

Oto katalog narzędzi z perspektywy **trybu integracji**:

## Zasada izolacji agentów działających w tle

Główny błąd przy projektowaniu systemów wieloagentowych: **nadmiar zależności** powstający samoistnie w miarę rozbudowy. Problemy ujawniają się z opóźnieniem, przez nawarstwianie informacji.

Projektuj tak, aby **konflikty nie powstawały**, a nie by były rozwiązywane:

- Agent klasyfikujący zgłoszenia nie musi wiedzieć o agencie robiącym zestawienia.
- Agent weryfikujący treści nie musi znać agentów robiących research.
- Pełna izolacja nie zawsze możliwa — ale dążenie do niej jest domyślnym trybem.
- Gdy współpraca agentów jest niezbędna, stosuj mechaniki z S02E04 (orkiestracja z zaangażowaniem człowieka).

Komplikacja rośnie nieliniowo z liczbą połączeń między agentami. Każde połączenie to potencjalna pętla, wyścig, lub niespójność.

## Self-observing systems

Agenci mogą **obserwować skuteczność działania systemu** — analogia do LLM-as-a-judge z ewaluacji:

- Newsletter, którego nikt nie czyta → wyłącz lub zmień częstotliwość.
- Źródło danych niedostępne od tygodnia → oznacz i wyrejestruj.
- Agent generujący powiadomienia bez reakcji użytkownika → zmniejsz częstotliwość lub podnieś próg istotności.

To mechanizm **samooczyszczania** systemu — bez niego automatyzacje degradują w szum.

## Kontekst urządzenia jako sygnał sterujący

Agent z dostępem do stanu urządzenia (aktywna aplikacja, lokalizacja, tryb DND) może **dopasować formę komunikacji**:

- Praca w konkretnym programie → auto-włącz DND, blokuj powiadomienia.
- Sytuacja krytyczna mimo DND → eskalacja przez SMS lub telefon.
- Lokalizacja jako metadane notatek → automatyczna organizacja pamięci agenta.

Wymaga to natywnych aplikacji (Swift / Electron / Tauri / React Native) — generowanych z pomocą AI, niekoniecznie publikowanych.

## Dwa pytania filtrujące

Przy każdym potencjalnym zastosowaniu AI:

1. **Jak AI może tu pomóc?** — czy zmniejsza zaangażowanie, podnosi jakość, lub odkrywa to, czego byśmy nie zauważyli?
2. **Czy powinniśmy tu angażować AI?** — czy automatyzacja nie zniszczy relacji (sprzedaż), nie wygeneruje szumu (powiadomienia), lub nie wprowadzi ryzyka (prompt injection, wyciek danych).

Modele są obecnie **słabymi doradcami** w identyfikacji scenariuszy — ich sugestie są generyczne. Inicjatywa musi przyjść od człowieka, najlepiej z obserwacji własnych procesów roboczych.

## Scenariusze kontekstowe — wzorce

| Wzorzec | Mechanika | Kluczowe ryzyko |
|---|---|---|
| Katalogi aktywne | Plik w folderze → trigger → transformacja → przeniesienie | Założenia niepokrywające się z rzeczywistością (np. klient rezerwuje wiele sesji) |
| Przekierowania | Formularz → LLM klasyfikuje → przekierowanie do właściwej osoby | Niepełne dane wejściowe, błędna klasyfikacja priorytetu |
| Monitorowanie wskaźników | Agent czyta analitykę + feedback → priorytetyzuje problemy | Fałszywe pozytywy, ignorowanie kontekstu biznesowego |
| Szablony projektowe | Framework + lista aktywności → auto-przydzielanie zadań | Sztywność szablonu w zderzeniu z nietypowym projektem |
| Sugestie wydarzeń | Notatki + obserwacja otoczenia → propozycje w osobnym kalendarzu | Przeładowanie propozycjami, niska precyzja |
| Kontrola jakości | Treść w kolejce publikacji → agent weryfikuje linki, poprawność, spójność | Fałszywe negatywy blokujące publikację |
