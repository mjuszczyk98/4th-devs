# S01E02 — Techniki łączenia modelu z narzędziami

## Function Calling — mechanizm

- Model nigdy nie wykonuje kodu — generuje JSON z `name` + `arguments`, aplikacja wykonuje akcję i zwraca wynik
- Każde użycie narzędzia = minimum 2 zapytania do LLM (żądanie → wykonanie → odpowiedź)
- Definicje wszystkich narzędzi trafiają do kontekstu przy **każdym** zapytaniu — nawet gdy nie są używane
- Historia konwersacji rośnie: `[...messages, ...toolCalls, ...toolResults]` — wpływa na koszt i limit kontekstu
- `strict: true` w schemacie narzędzia wymusza, żeby model nie dodawał pól spoza definicji

## Projektowanie narzędzi

**Nie mapuj API 1:1.** API projektowane jest dla programistów z dostępem do dokumentacji. Narzędzia LLM muszą być zrozumiałe bez żadnego zewnętrznego kontekstu.

Trzy pytania przy projektowaniu schematu:
- Co model **musi** uzupełnić?
- Co powinno być uzupełnione **programistycznie** (np. `user_id` z sesji)?
- Czego model **nie może** uzupełnić (uprawnienia, identyfikatory kontroli dostępu)?

Zasady:
- Nazwa unikalna, niska szansa kolizji (`send_email` > `send`)
- Opis: wysoki signal-to-noise, tylko to co zwiększa szansę na wybór w odpowiednim momencie
- Łącz zasoby API gdy model potrzebuje danych z kilku endpointów naraz (`workspace_metadata` zamiast 3 osobnych calls)
- Pomijaj pola które model nie może sensownie uzupełnić (hashe, wewnętrzne flagi, IDs kontroli dostępu)
- Używaj `recursive: true`, domyślnych wartości i aliasów — wybaczaj potknięcia tam gdzie to możliwe

## Odpowiedzi i błędy narzędzi

Output narzędzia = minimum niezbędnych informacji. `stat()` zwraca kilkanaście pól — handler zwraca 4.

Błąd **nie przerywa pętli** — jest serializowany do JSON i zwracany modelowi jak każdy wynik:
```js
catch (error) {
  return { type: "function_call_output", call_id: call.call_id, output: JSON.stringify({ error: error.message }) };
}
```

Błędy opisowe z kontekstem:
- `"team_id nieprawidłowy. Wskazówka: pobierz przez workspace_metadata"` — model wie co zrobić dalej
- `"Access denied: path "../config.js" is outside sandbox"` — model wie że nie może, nie że się posypało

## Pętla agenta

```js
while (stepsRemaining > 0) {
  const response = await requestResponse(conversation);
  const toolCalls = getToolCalls(response);
  if (toolCalls.length === 0) return getFinalText(response);
  conversation = await buildNextConversation(conversation, toolCalls, handlers);
  stepsRemaining--;
}
throw new Error("Max steps reached");
```

- `MAX_TOOL_STEPS` / `MAX_TOOL_ROUNDS` — guard obowiązkowy, bez niego agent może kręcić się w nieskończoność
- Rzuć błąd po wyczerpaniu limitu — nie zwracaj cicho, model nie dowie się co poszło nie tak
- `Promise.all` na `toolCalls` — parallel function calling bez dodatkowego kodu

## Bezpieczeństwo i izolacja

Path traversal blokowany na poziomie kodu, nie promptu:
```js
const rel = relative(sandbox.root, resolved);
if (rel.startsWith("..")) throw new Error(`Access denied: path "${relativePath}" is outside sandbox`);
```

Zasada: każde uprawnienie i każda granica dostępu musi być wymuszona przez kod — model nie może sam ustalać `user_id`, ścieżek poza sandbox, ani zakresu uprawnień.

Akcje nieodwracalne (email, delete) wymagają potwierdzenia przez **UI (przyciski)**, nie przez wiadomość do modelu — model może zignorować lub błędnie zinterpretować tekst użytkownika.

## Zarządzanie kontekstem

- **Prompt cache = priorytet** — dynamiczne dane w `system prompt` (nawet bieżąca data) niszczą cache przy każdym zapytaniu
- Zewnętrzny kontekst wczytywany przez narzędzia trafia naturalnie do końca wątku — nie psuje cache systemu
- System plików jako zewnętrzna pamięć: narzędzia zapisują wyniki do plików, agent wczytuje tylko potrzebne fragmenty
- Limit narzędzi: max 10-15 na agenta. Dynamiczne wczytywanie bez utraty cache dostępne tylko u Anthropica
- Auto-compact (Claude Code, Cursor) = kompresja z utratą informacji — zapisuj ważne wątki do plików

## API vs CLI vs Function Calling

| | API bezpośrednie | CLI | Function Calling / MCP |
|---|---|---|---|
| Kontrola | brak | interfejs CLI | pełna (kod) |
| "Dokumentacja" dla modelu | wymaga eksploracji | `--help` | JSON Schema + opis |
| Skalowalność | trudna | trudna poza lokalnie | dobra |
| Dostęp do terminala | tak | tak | nie |

## Prompt Injection

Problem **otwarty** — brak skutecznych technik obrony. Agent przeglądający email może dosłownie wykonać instrukcje zawarte w treści wiadomości. Jedyne skuteczne mitygacje to ograniczenia środowiskowe i nieudostępnianie agentom zasobów wrażliwych.

## Przykłady kodu

### `01_02_tools` — minimalna architektura Function Calling

Kompletny flow w ~50 liniach: definicje narzędzi → pętla z guardem → `Promise.all` na tool calls → sklejanie konwersacji.

Uczy: separacji definicji (co model widzi) od implementacji (co kod robi), pętli z `MAX_TOOL_STEPS`, parallel execution przez `Promise.all`.

Mockowane handlery — realne byłyby API calls. `webSearch: true` to natywne narzędzie OpenAI doklejone jedną linią obok własnych tools.

### `01_02_tool_use` — delta względem `01_02_tools`

**Architektura:** podział na `definitions.js` / `handlers.js` / `index.js` — skalowalne przy 10+ narzędziach. Zmiana narzędzi nie dotyka `executor.js` ani `app.js`.

**Sandbox guard** — blokuje path traversal zanim dotrze do systemu plików. `app.js` zawiera wprost test `"Try to read ../config.js"` jako demonstrację.

**Błąd jako dane** — `catch` serializuje wyjątek do `output` zamiast przerywać pętlę. Model dostaje `ENOENT: no such file or directory` i może podjąć decyzję.

**Celowe filtrowanie outputu** — `stat()` zwraca ~15 pól, handler zwraca 4. Reszta świadomie odrzucona.

**`TOOLS.md`** — dokumentacja dla developera (nie dla modelu) z pełnym schematem input→handler→output→errors. Wzorzec do powielenia w każdym projekcie z narzędziami.
