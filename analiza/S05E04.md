# S05E04 — Produkcja

## Przepaść MVP ↔ produkcja w gen-AI

W klasycznych aplikacjach produkcja różni się skalą, monitoringiem, CI/CD. W gen-AI różni się **fundamentalnie** — detale, które w MVP są niewidoczne, determinują użyteczność i bezpieczeństwo na produkcji. Modele kodują szybko, ale **pomijają** to, czego nie znają z doświadczenia. Różnicę robi to, co zostanie pominięte.

## Konwersacje są strukturą delikatną

- **Delete wiadomości** — Anthropic/Gemini wymagają user message jako ostatniej. Usunięcie ze środka łamie invarianty API. ChatGPT, Claude, Cursor nie oferują delete. Zamiast tego: **rozgałęzianie** (branching) i **przywracanie** do punktu.
- **Edit wiadomości agenta** — otwiera many-shot jailbreak: użytkownik może zasugerować agentyowi, że wykonał coś, czego nie powinien. Odpowiedź: agent messages nieedytowalne.
- **Long paste w prompcie** — textarea z wielkim tekstem zabija wydajność edycji. Rozwiązanie: detekcja rozmiaru → automatyczna konwersja do file attachment.

Każda z tych trzech rzeczy to "drobny detail". Łącznie definiują czy produkt jest użyteczny.

## Halucynacje w niszowych domenach

Whisper halucynuje na ciszy ("Thanks for watching!", "Subtitles by the Amara.org community") — prawdopodobnie artefakt treningu na napisach filmowych. Mieszanie języków w jednym zdaniu ("Turn left and walk straight down **Piotrkowska** Street"). Nie jest to błąd do naprawienia promptem — to ograniczenie modelu wymagające świadomego projektowania pipeline'u audio.

## Narzędzia wspomniane, ale nieaktywne

Samo wspomnienie o możliwościści w system prompt, bez aktywacji narzędzia, wystarczy by model **symulował** jego użycie. Ryzyko: niejawne halucynacje funkcjonalne — model opisuje wynik akcji, która nigdy nie nastąpiła.

## Logika agenta na produkcji — runtime pipeline

Wiadomość użytkownika traktowana jak **zadanie** (job), nie event. Job → Run (próba wykonania). Oddzielenie "co zrobić" od "kto próbuje" umożliwia retry, delegację, monitoring.

Cykl: `Request → Persist → Queue → Schedule → Claim → Turn Loop → [Done | Delegate | Wait] → Persist → Events`

### Readiness engine

Deterministyczny scheduler odpytujący DB o pary job/run. Priorytetyzacja **stała**:

1. Dostarczenie wyników od child agents
2. Wznowienia po wait
3. Odzyskiwanie po crashu (stale recovery)
4. Nowe zadania

Root runs mają priorytet nad child runs. Gwarantuje to, że zagnieżdżone delegacje nie zagładzą nowych interakcji.

### Durable-first execution

**Persist first, execute second.** Komenda zapisuje stan do DB *zanim* wywoła model. Konsekwencje: model fail → work queued (nie lost), crash recovery z durable state, route handler zwraca persistent data, nie ephemeral output.

### Lease-based crash recovery

```
Worker claims run → heartbeat co X ms
  → crash = heartbeat stopuje → claim expires
  → scheduler detect stale → requeue z exponential backoff
  → po maxRecoveries → permanent fail
```

`staleRecoveryCount` inkrementuje per run. Opóźnienie: `baseDelayMs * 2^(count-2)`. Zapobiega rapid re-crash loops.

### Delegacja jako tool call

`delegate_to_agent` — natywne narzędzie. LLM sam decyduje o delegacji. Mechanika: child run + child job w jednej transakcji → parent → `waiting` → dependency edge → child complete → readiness engine delivers result → parent resumes. Zagnieżdżenie wielopoziomowe (Alice → Tony → Claire).

Wzorzec: delegacja jest narzędziem, więc infrastruktura orkiestracji jest taka sama jak dla innych tool calls.

### Event sourcing z outbox

Domain events → `event_outbox` (SQLite) → dedykowane workery rozsyłają do SSE (realtime), projections (read models), Langfuse (traces). Rozdziela writes od side effects, gwarantuje at-least-once delivery.

## Bezpieczeństwo — konkretne ataki i mitigacje

| Wektor | Mitigacja |
|--------|-----------|
| Many-shot jailbreak przez edycję | Agent messages nieedytowalne w UI |
| Narzędzie wspomniane, nieaktywne | Tool availability per-run: `isAvailable(context)` |
| MCP STDIO na serwerze | Dopuszczalne **tylko** na desktop, proces lokalny użytkownika |
| Timing attack na hasła | scrypt + timingSafeEqual |
| Sekrety OAuth w DB | AES-256-GCM encryption at rest |
| Double execution na retry | Idempotency keys z progress snapshots |

Brak skutecznej obrony przed prompt injection — świadomość tego faktu jest częścią architektury.

## Frontend produkcyjny dla agentów

### Streaming

rAF batching — SSE eventy nie trigerują re-render indywidualnie. `requestAnimationFrame` batch = jedno DOM update na klatkę. Incremental markdown: committed blocks cached, re-render tylko "live tail".

### Stan

Three-layer message projection: `durableMessages` (backend confirmed) + `optimisticMessages` (pending user) + `liveAssistantMessage` (streaming) → scalane w projected view. Eliminuje flicker przy stream→durable transition.

### Virtual scrolling

Chunkowanie wiadomości po 12, overscan 3 chunks, `ResizeObserver` do pomiaru rzeczywistych wysokości, canvas text measurement do estymacji.

### Złożoność kompozytora

Tiptap v3 z custom extensions: `@` mention agentów, `#` mention plików, image paste, inline markdown. Command palette z provider-based architekturą — każdy feature dostarcza itemy. Keyboard shortcut scoping przez layer stack.

## Sugestie ogólne — zmiana perspektywy

- **Jakość > szybkość** — gdy każdy może generować kod, wyróżnikiem jest dbałość o detale i jakość.
- **Użytkownik nie wie czym jest LLM** — projektuj dla minimalnej wiedzy. Skuteczny agent jest niewidoczny.
- **Agent bez czatu** — reagowanie na akcje (upload, klik) daje większą kontrolę niż open-ended chat. Czat sensowny przy złożonych akcjach lub skali.
- **Budowanie dla agentów** — aplikacje muszą być dostępne nie tylko dla ludzi, ale i dla agentów działających w ich imieniu. To trend, który dopiero się zaczyna.
- **Kwestionuj wzorce** — po poznaniu zasad, łam je celowo. Koszt eksperymentu jest dziś minimalny.

## Przykłady kodu — progresja i delta wiedzy

### `05_04_api` — Produkcyjny backend multi-agent runtime

**Delta względem `05_01_agent_graph`:** od grafu w pamięci do **durable runtime z crash recovery, multi-tenancy i event sourcing**.

- **Result<T, E> zamiast wyjątków** — każda operacja domenowa zwraca discriminated union. Wyjątki lecą dopiero w warstwie HTTP. Wymusza jawne error handling na każdym call site.
- **Branded IDs** — `asRunId("run_abc")` waliduje prefix + daje compile-time type safety. Brak możliwości przekazania ThreadId gdzie oczekiwano RunId.
- **Command pattern (CQRS-inspired)** — komendy (`createStartThreadInteractionCommand`) enkapsulują całe operacje biznesowe: walidacja (Zod) → auth → DB writes w transakcji → domain events → typed result.
- **MCP Gateway z workspace isolation** — discovery (static + dynamic), OAuth z encryption, path rewriting per run (izolacja filesystem). Serwery MCP scope'owane per workspace.
- **Context budget calibration** — `usage_ledger` śledzi tokeny per thread. Estymacje budżetu kalibrowane na podstawie rzeczywistych zużyć z poprzednich tur.
- **23 migracje SQLite** — WAL mode, foreign keys, busy timeout. Pełny schema: accounts, tenants, sessions, threads, messages, agents, revisions, runs, jobs, claims, dependencies, items, tool executions, MCP servers, files, events outbox, usage ledger, idempotency keys.
- **Hono** zamiast Express — lightweight, edge-compatible. API versioned `/v1` ze spójnym envelope: `{ ok: true, data, meta: { requestId, traceId } }`.

### `05_04_ui` — Produkcyjny frontend Svelte 5

**Delta względem `05_02_ui`:** od streaming chat UI do **kompletnego interfejsu agentowego z virtual scrolling, event sourcing i multi-tenancy**.

- **Svelte 5 runes w praktyce produkcyjnej** — `$state`, `$derived`, `$effect` w 15k+ linii. Factory functions z DI: `createChatStore(deps)` pozwala testować bez mock frameworka. Context API jako lightweight DI przez `setContext`/`getContext`.
- **`shared/chat.ts` (~1100 linii)** — wspólny kontrakt frontend↔backend. 30+ typów eventów, branded IDs, API shapes. Frontend i backend mówią tym samym językiem typów.
- **Typewriter effect z gatingiem** — bloki odkrywane po jednym z konfigurowalną prędkością. Buforowanie: nowe bloki z backendu czekają aż obecny się wyświetli. Durable handoff: gdy stream się skończy, typewriter pomija replay.
- **Projected sync batching** — batch/transaction system dla state updates. `projectedSyncBatchDepth` liczy zagnieżdżenie, flush na depth=0. Eliminuje redundant DOM updates przy wielu zmianach w jednym ticku.
- **SSE z reconnect** — exponential backoff (250ms base, 2s max, 6 attempts), event ID tracking dla cursor-based resume, abort signal forwarding.
- **Submit button jako state machine** — 4 stany (idle/active/cancelling/finishing) przez CSS grid stacking. Wszystkie labelki na tej samej komórce, widoczność przez opacity.
