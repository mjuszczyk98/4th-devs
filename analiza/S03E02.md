# S03E02 — Ograniczenia modeli na etapie założeń projektu

## Projektowanie roli AI w systemie

**Dedykowane narzędzia > all-in-one agenci.** Pełna automatyzacja procesów przez AI to obecnie zły pomysł niezależnie od obszaru. Zamiast tego: narzędzia wspierające ludzi, nie zastępujące ich.

Typowe pułapki i alternatywy:

| Pułapka | Dlaczego nie działa | Alternatywa |
|---------|---------------------|-------------|
| RAG na "całą bazę wiedzy" | Koszt zebrania danych z różnych źródeł przewyższa wartość | Zweryfikuj tezy prototypem → zawęź zakres (onboarding, sprzedaż, managerowie) |
| Czatbot na stronę | Budowanie relacji przez bota = wizerunkowe ryzyko | Agenci w tle zintegrowani z panelem obsługi — pracownik nie musi uczyć się AI |
| Auto-email marketing | Skuteczność niska, ryzyko wizerunkowe wysokie | Wsparcie istniejących procesów, nie ich automatyzacja |
| Agent na całą organizację | Prompt injection + halucynacje = brak kontroli | Wyspecjalizowane narzędzia per obszar |

**Kluczowa umiejętność:** określanie, co w danej chwili nie ma znaczenia. Regularnie aktualizuj przekonania — to co niemożliwe dziś, jutro może być trywialne.

**Przepaść autonomy:** odchylenia i błędy tolerowane gdy człowiek jest w pętli → przekreślają sens istnienia systemu autonomicznego. Błąd co piąty raz przy ustalonym procesie = system bezużyteczny. Przejście do autonomicznych agentów to nowa klasa problemów programistycznych, nie inkrementalny wzrost trudności.

## Podział odpowiedzialności: model vs kod

Model nie musi nieść całej logiki. Może lepiej zarządzać **deterministyczną logiką** przez routing zapytań i referencje do dokumentów.

- **AI proposes, system decides** — model proponuje akcje, ale security properties (uprawnienia, scope danych, klasyfikacja kontaktów) determinuje kod. Zapobiega to privilege escalation przez prompt injection.
- Jeśli model musi "przeliczać w pamięci" — przenieś to do kodu. Obliczenia w sandboxie są deterministyczne, w kontekście LLM — nie.
- Czasem właściwa odpowiedź to **wycofanie się z pomysłu** i oddanie procesu ludziom.

## Architektura bezpieczeństwa

**Zasada zero-trust:** nie ufamy agentom w zakresie udostępniania informacji i pracy na zewnętrznych źródłach. Każdy punkt kontaktu ze światem zewnętrznym = niebezpieczny.

### Warstwy obrony

1. **Prompt systemowy = publiczny** — żadnych sekretów, bo zostanie wydobyty
2. **Programistyczna kontrola dostępu** — zmiana zachowania agenta przez input nie może prowadzić do eskalacji uprawnień
3. **Capability stripping** — agent dostaje narzędzia TYLKO na czas ich potrzeby. Faza kategoryzacji = pełne narzędzia, faza szkicowania = zero narzędzi.
4. **Izolacja danych** — scope'owanie kontekstu przez warstwy:
   - Layer 1: izolacja konta (tylko `shared` + matching account)
   - Layer 2: scope'owanie contact type (`internal` > `trusted_vendor` > `client` > `untrusted`)
5. **Runtime mutex** — blokada KB na konkretne konto na czas sesji draft, z `try/finally` gwarantującym zwolnienie
6. **Audit trail** — logowanie każdego dostępu do danych (co zwrócone, co zablokowane, dlaczego)

### Filtrowanie prompt injection

Oddzielny prompt oceniający intencje inputu, całkowicie odseparowany od głównego wątku. Rezultat: konkretny ciąg znaków ("bezpieczne"/"niebezpieczne"), weryfikowany programistycznie. Atakujący nie zna fraz-kluczy. Ominięcie możliwe, ale trudne dla typowego użytkownika.

## Heartbeat jako wzorzec orkiestracji

Zamiast sztywnego DAG-a — **pull-based task queue** z round-based reconciliacją.

**Trzy komponenty:**
- **Kontrakty** — struktura planu, zadań, zależności, statusów, wymaganych capabilities
- **Heartbeat** — manager przydzielający zadania, aktualizujący stan po każdym cyklu
- **Pamięć** — filesystem-based, działa w ramach jednego agenta i jako kanał komunikacji między nimi

**Task state machine:**
```
open → in-progress → done
                   → blocked → open (retry)
                   → waiting-human → open (z human_answer)
```

Każdy cykl: reconcile dependencies → resolve waiting tasks → dla każdego agenta: claim task → run session → persist status → flush events.

**Przykładowy flow etapów (events):** przeszukanie źródeł → pogłębienie wyszukiwania → wstępne notatki → pierwsza wersja → raport finalny. Agenci mogą tworzyć nowe zadania w trakcie (np. potrzebne dodatkowe dane), a nawet prosić o kontakt z człowiekiem gdy plan wymaga rozstrzygnięcia.

**Dlaczego nie DAG:** proces jest dynamiczny. Zadania mogą wymagać dodatkowych kroków, nowe zadania mogą powstać w trakcie. Heartbeat daje elastyczność przy zachowaniu kontroli.

## Observational Memory

Problem: długotrwałe sesje agentowe przepełniają context window.

**Observer/Reflector cycle:**
- **Observer** — ekstrahuje ustrukturyzowane obserwacje z conversation history (>30K nieobserwowanych tokenów). Oznacza priorytety (red/yellow/green).
- **Reflector** — kompresuje i reorganizuje obserwacje (>40K tokenów). Progresywny poziom kompresji.

To zapobiega context window exhaustion bez utraty kluczowych informacji. Działa per-agent-session, nie globalnie.

## Sandbox + kod = obejście ograniczeń modeli

Pięć obszarów optymalizacji: tokeny wejściowe, cache, tokeny wyjściowe, liczba zapytań, mniejsze modele. Kluczowe narzędzie: **generowanie kodu w sandboxie**.

**Kiedy**: dane zbyt duże na kontekst LLM (np. 240 plików, 150K linii). Agent generuje kod, który przetwarza dane deterministycznie — 6-10 kroków zamiast setek.

**Architektura 3-procesowa:**
```
Proces główny (agent loop) ↔ MCP server (STDIO) ↔ Sandbox (Deno)
```

**HTTP Tool Bridge** — sandbox nie ma bezpośredniego dostępu do narzędzi hosta. Obejście: lokalny HTTP server mapujący tool names na endpointy. Prelude wstrzykiwany przed kodem sandboxa daje czysty async API (`tools.fs_read({path: ...})`).

**Ostrzeżenie:** kalkulacje w kodzie są poprawne, ale poprawność wczytania danych nie jest gwarantowana. Agenci pracujący na krytycznych danych = ścisły nadzór + deterministyczne procesy wszędzie gdzie możliwe.

**Skalowanie sandboxów:** architektura 3-procesowa (główny + MCP STDIO + sandbox) jest wyzwaniem na produkcję z tysiącami użytkowników. Rozwiązania rosną: Cloudflare Sandbox, Daytona. Problem zaczyna być adresowany przez ekosystem.

## Knowledge files jako anty-wzorce

Zamiast uczyć AI "jak zrobić", powiedz mu "co idzie źle". Pliki knowledge zawierają wyselekcjonowane zasady będące wynikiem trial-and-error (np. "blank pages w PDFKit", "cursor drift po positioned blocks"). Prompt wymusza czytanie tych plików jako pierwszy krok.

To jest RAG, ale retrieval domain to **anty-wzorce i edge cases**, nie surowa wiedza.

## Człowiek jako final gate

Agent nigdy nie wykonuje nieodwracalnych akcji. Wysyłanie maili, publikowanie dokumentów, postowanie — to domena człowieka. Agent kategoryzuje, etykietuje, szkicuje, filtruje, wzbogaca. Człowiek decyduje.

Etykieta przypisana przez agenta = zdarzenie obserwowalne → wyzwalacz dla innych agentów lub akcji deterministycznych (np. label "Bug Report" → tworzenie issue w Jira). System eventów rozszerza wartość bez zwiększania ryzyka.

## Przykłady kodu — progresja wiedzy

### 03_02_code — Sandbox execution

**Delta:** Agent z narzędziami MCP + wykonanie kodu w piaskownicy Deno.

Nowe koncepty:
- **HTTP Tool Bridge** — sandbox komunikuje się z hostem przez HTTP localhost. Prelude generuje obiekt `tools` z async metodami proxy'ującymi do MCP.
- **4 poziomy uprawnień** (`safe`/`standard`/`network`/`full`) mapowane na flagi Deno. Sandbox dostaje tylko uprawnienia niezbędne dla danego poziomu.
- **Deterministyczny generator danych** — FNV-1a + mulberry32 PRNG seedowany stringiem. Identyczne dane co uruchomienie.
- **Knowledge-guided generation** — wymuszenie czytania plików z anty-wzorcami przed generowaniem kodu.
- **Dual runtime** — Bun jako proces główny, Deno jako sandbox. Osobne zarządzanie zależnościami (package.json vs Deno cache).

### 03_02_email — Izolacja danych + capability stripping

**Delta:** Dwufazowa architektura z programistyczną kontrolą dostępu i ewaluacją.

Nowe koncepty (nieobecne w `code`):
- **Phase-based capability stripping** — Faza 1 (triage): 12 narzędzi, KB unlocked. Faza 2 (draft): 0 narzędzi, KB locked na konkretne konto. AI nie może sięgnąć po więcej danych niż upieczone w prompcie.
- **Runtime mutex na KB** — `lockKnowledgeToAccount()` z `try/finally`. Moduł-level state z error throwing na naruszenia.
- **System-determined classification** — `mark_for_reply` to wirtualne narzędzie: AI mówi "odpowiedz", SYSTEM determinuje `contactType` i scope KB. AI nie może eskalować uprawnień.
- **Adwersarz testowy** — celowy email social engineering w skrzynce. Agent musi odmówić — bo runtime guard blokuje dostęp do danych drugiego konta.
- **Snapshot-diff state tracking** — checkpoint-based diff system dla before/after wizualizacji.
- **Ewaluacja Langfuse** — 4 suite'y testowe (triage, izolacja, język, malicious) z konkretnymi metrykami.
- **Emergent language matching** — KB zawiera politykę językową ("odpowiadaj w języwie nadawcy"), eval weryfikuje. Bez library do detekcji języka — czysto emergent behavior.
- **Privacy by design** — `store: false` na wszystkich wywołaniach AI. Konwersacje nie persistowane na serwerach OpenAI.
- **Responses API** zamiast Chat Completions.

### 03_02_events — Multi-agent autonomy

**Delta:** Autonomiczny system multi-agentowy z heartbeat loop, memory management i goal contracts.

Nowe koncepty (nieobecne w `email`):
- **Goal Contract** — `goal.md` z YAML frontmatter (cel, wymagania, budżety, zakazy) → LLM planner → strukturyzowany plan z zadaniami. Jeśli cel nieosiągalny → `no-go.md` z powodami.
- **Heartbeat loop** — pull-based: agenci claimują zadania, nie są do nich przypychani. Round-based reconciliacja zależności.
- **Agent Templates jako `.agent.md`** — deklaratywne definicje agentów w markdown z YAML frontmatter (model, tools, capabilities, system prompt).
- **Capability-based task assignment** — zadania deklarują `required_capabilities`, agenci deklarują `capabilities`. Claiming matchuje capabilities, nie nazwy agentów.
- **Observational Memory** — observer/reflector cycle zapobiegający context overflow w długich sesjach. Progresywna kompresja z priorytetyzacją.
- **Event sourcing** — JSONL + per-round markdown. Pełna audytowalność.
- **Bounded replanning** — system proponuje patch'e (add/split/reassign/cancel task) ale tylko w ramach budżetu.
- **Task files jako source of truth** — filesystem-based kolejka zadań. Pliki `.md` z YAML frontmatter — czytelne, wersjonowalne, debugowalne bez bazy danych.
- **Token estimation z kalibracją** — estymacja `chars/4` autokalibruje się na bazie rzeczywistego API-reported usage. Precyzja rośnie w czasie.
- **No-go handling** — gdy planner uznaje cel za nieosiągalny, zapisuje `no-go.md` z powodami, brakującymi capabilities i sugerowanymi akcjami. Zapobiega cichym fiaskom.
- **Plan validation z repair** — wygenerowany plan walidowany (deps, capabilities, forbidden phrases, cycle detection). Niepoprawny → do 2 prób naprawy przez LLM.
