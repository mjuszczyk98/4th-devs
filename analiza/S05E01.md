# S05E01 — Architektura

## Prymitywy, nie funkcjonalności

Projektowanie na prymitywach to główna idea architektoniczna tej lekcji. Zamiast budować struktury pod konkretną funkcjonalność, buduje się je pod **możliwość przekształcenia**.

| Funkcjonalność (sztywna) | Prymityw (elastyczny) |
|--------------------------|-----------------------|
| Wiadomości user ↔ asystent | Zdarzenia między aktorami (user, agent, system) |
| Osobne tabele na pliki/obrazy/dokumenty | Artefakty — metadane dowolnej treści |
| Tabela `messages` z polem `text` | Tabela `items` z typami (message, tool_call, invocation, result) |

Różnica: "prosty czatbot" szybko staje się agentem, potem systemem wieloagentowym. Prymitywy pozwalają rozbudowę bez przebudowy architektury. Zdarzenia nie muszą dotyczyć tylko wiadomości — mogą obejmować reasoning modelu, kompresję kontekstu, prośbę o potwierdzenie subagenta. Aktor nie musi być użytkownikiem — może być agent lub system.

## 5 obszarów decyzyjnych architektury gen-AI

- **Gateway** — scentralizowana komunikacja z AI. Musi umożliwiać swobodne przełączanie modeli i providerów (AI SDK / LiteLLM / własny adapter).
- **API** — wyspecjalizowane endpointy (np. `/product/review`), nie bezpośredni dostęp do modelu przez `/api/chat`. Kontakt z modelem ograniczony już na poziomie API.
- **System plików** — scope uprawnień dla agentów. Agent może przypadkowo podjąć destrukcyjną akcję (np. usunąć katalog) — to wymaga innych zabezpieczeń niż klasyczna aplikacja.
- **Baza danych** — dodatkowe struktury: interakcje, zaplanowane zadania, definicje agentów i narzędzi.
- **Zależności** — ewaluacja, obserwowalność, transformacja dokumentów, SSE, RAG.

### Pewniki (niemal zawsze):

- Centralizacja interakcji z AI (budowanie zapytań z wielu miejsc utrudnia zarządzanie)
- Otwartość na wielu providerów (lepsze modele pojawiają się u różnych dostawców)
- Strumieniowanie zdarzeń (postępy + czas reakcji)
- Furtka na multimodalność (struktury DB gotowe na obraz/audio nawet jeśli teraz tylko tekst)
- Struktura `items` zamiast `messages` — monitorowanie akcji **między** wiadomościami
- Obsługa zadań w długim horyzoncie (użytkownik zamyka kartę, limity czasu połączenia)

## System powinien wzmacniać się z rozwojem modeli

Decyzje architektoniczne muszą uwzględniać, że modele będą lepsze. Konsekwencje:

- Nie buduj na obszarach, gdzie LLM szybko dogoni funkcjonalność — produkt stanie się niepotrzebny przed zwrotem z inwestycji.
- Architektura musi być bardziej podatna na modyfikacje niż w klasycznych systemach — dinamika iteracji jest rzędu tygodni, nie kwartałów.
- Ostrożność przy frameworkach AI — ich fundamenty wciąż się zmieniają, oparcie na nich całej aplikacji = ryzyko blokady.

## Integracja wielu providerów

Problem pojawia się, gdy **różni providerzy w tej samej logice** (np. agent z OpenAI, subagent z Anthropic). Wymaga warstwy tłumaczeń mapującej:

- **Strukturę API:** system message — OpenAI (w wiadomościach) / Anthropic (oddzielne pole) / Gemini (`system_instruction`)
- **Ustawienia:** reasoning — OpenAI (`reasoning_effort`) / Anthropic+Gemini (`budget_tokens`)
- **Thought Signatures:** Gemini 3+ wymaga ich przy tool calls w bieżącej turze. Anthropic ma analogiczny mechanizm. Brak obsługi = utrata reasoning tokens.
- **Ograniczenia:** Anthropic nie pozwala na wiadomość asystenta jako pierwszą w historii.

| Podejście | Kiedy | Ryzyko |
|-----------|-------|--------|
| **OpenRouter** | Tylko LLM, bez zaawansowanych funkcji API | Brak wsparcia dla nowszych funkcji, sporadyczne błędy mapowania sygnatur |
| **Biblioteki** (AI SDK, LiteLLM) | Provider dobrze wspierany, potrzebny快速 start | Blokada dostępu do najnowszych API, bugi z niskim priorytetem naprawy |
| **Własna logika + SDK** | Większość projektów produkcyjnych | Pełna kontrola, ale pełen ciężar. W erze AI budowanie własnej logiki jest opłacalne |

Rekomendacja lekcji: własna logika z domyślnym formatem **Responses API** (najpopularniejszy). Kluczowe decyzje przy budowie: (1) domyślny kształt zapytania/odpowiedzi/streamingu, (2) mechanizm wyboru providera + mapowanie ustawień, (3) zakres obsługiwanych ustawień.

## Architektura agenta — łączenie wzorców

`05_01_agent_graph` łączy 4 wzorce w jednym systemie:

**Orchestrator** — agent z `delegate_task` i `create_actor`. Nie wykonuje pracy. Tworzy aktorów dynamicznie (nie predefiniowanych). Reguła: nie blokuj żeby czekać — scheduler wznowi automatycznie.

**Blackboard** — współdzielony stan (sesja, zdarzenia, zadania, relacje, artefakty). Agenci nie komunikują się bezpośrednio — komunikują się przez stan za pomocą narzędzi (np. `write_artifact`).

**DAG** — relacje `depends_on` tworzą graf. Scheduler deterministycznie rozwiązuje zależności: promuje gotowe zadania, wstrzymuje czekające, wznawia po zakończeniu dzieci.

**Events** — każda zmiana stanu → event → SSE. Użyteczne: observability, ewaluacja, guardrails, UI.

### Delta vs S03E02 (heartbeat)

W `03_02_events` plan zadań był statyczny — określony z góry. Tutaj plan jest **kształtowany dynamicznie** przez agenta zarządzającego. Elastyczność rośnie, przewidywalność maleje.

### Cykl scheduler'a

1. `findReadyTasks` — zadania bez zależności lub z Dependencies spełnionymi
2. Status → `in_progress`, przekazanie do pętli aktora
3. Aktor kończy: `done` lub `blocked`. Status `waiting` ustalany automatycznie gdy aktor skończył ale scheduler widzi aktywne subagenty
4. Po zakończeniu pętli aktorów → kolejna runda

## Przykłady kodu

### `05_01_agent_graph` — system wieloagentowy z grafową orkiestracją

**Delta względem poprzednich lekcji:** po raz pierwszy scheduler zarządza cyklem życia zadań na dynamicznym DAG-u, a pamięć jest kompresowana dwufazowo. Agent to dane, nie klasa.

**Agent jako dane, nie klasa.** Agent = obiekt konfiguracyjny (`AgentDefinition` — instrukcje + narzędzia + maxSteps). Zachowanie wyłania się z konfiguracji + uniwersalnego runtime'u. Różnicowanie agentów: **tylko zestaw narzędzi** (researcher nie ma `read_artifact`, writer nie ma `web_search`).

**Observational Memory — 2-fazowa kompresja kontekstu.** Observer konwertuje surową historię na priorytetyzowane obserwacje XML (🔴🟡🟢). Reflector — gdy za dużo — kompresuje przez LLM z poziomami agresywności. Kluczowy niuans: `splitByTailBudget` gwarantuje, że invocation+result nigdy nie są rozdzielone podczas kompresji.

**Trójwarstwowa strategia kontekstu LLM.** Skompresowana pamięć (sealed) + aktualny snapshot grafu + aktywny ogon. LLM nigdy nie dostaje pełnej historii.

**Graf zadań nad flat store.** Relacyjna warstwa nad FileStore (JSON). Zamiast dedykowanej bazy grafowej — zbiór relacji `Relation`, zapytania grafowe jako filtry. Demonstruje, że task DAG nie wymaga Neo4j.

**Artifact placeholders.** Agenci referencjonują artefakty przez `{{file:path}}` zamiast kopiować treść. Oszczędność tokenów + unikanie niespójności.

**Trójpoziomowa obsługa błędów.** API (retry z exponential backoff) → Actor (`RecoverableActorError`) → Task (recovery state z rozróżnieniem `explicit_block` / `llm_transient` / `runtime_error`).

**Prompt caching per task-actor.** SHA-256 z `session:task:actor` — maksymalne wykorzystanie prompt caching w OpenAI Responses API.

**Write queue — sekwencyjne zapisy bez mutexów.** Łańcuch Promisów: `this.writeQueue = this.writeQueue.then(async () => ...)`. Gwarantuje kolejność zapisów do pliku bez blokad.

**Brak frameworka agentowego.** ~1500 linii TS, zero LangChain/AutoGen, czysty kod + OpenAI SDK. Separation of concerns: domain (typy) → store (persystencja) → runtime (kontekst) → tools (narzędzia) → scheduler (orkestracja) → memory (kompresja) → agents (konfiguracja).
