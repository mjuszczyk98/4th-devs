# S05E01 — Architektura

## Prymitywy architektoniczne

- Zamiast projektować wokół **funkcjonalności** (np. "czat"), projektować wokół **prymitywów** — najprostszych elementów z których buduje się struktury wyższego rzędu.
- Przykład: "wiadomości między użytkownikiem a asystentem" → "zdarzenia między aktorami". Zdarzenia mogą dotyczyć nie tylko tekstu, ale też wywołań narzędzi, reasoningu, kompresji kontekstu. Aktor może być użytkownikiem, agentem, systemem.
- Artefakty = metadane reprezentujące treść dowolnego typu (plik, obraz, interaktywny interfejs ze stanem). Zamiast osobnych struktur per typ — jeden prymityw z polem `kind`.

## Decyzje architektoniczne (obecne w każdym projekcie)

| Obszar | Kluczowa decyzja |
|---|---|
| **Gateway** | Scentralizowana komunikacja z AI. Swobodne przełączanie modeli/providerów przez AI SDK, LiteLLM lub własny format API. |
| **API** | Klient nie ma bezpośredniego dostępu do modelu. Wyspecjalizowane endpointy (`/product/review`) zamiast generycznych (`/api/chat`). |
| **System plików** | Uprawnienia i zakresy dostępu dla agentów. Ryzyko nieprzewidzianych akcji (np. usunięcie katalogu) wymaga sandboxingu. |
| **Baza danych** | Dodatkowe struktury: interakcje, zaplanowane zadania, definicje agentów i ich narzędzi. |
| **Zależności** | Biblioteki: ewaluacja, observability, transformacja markdown, wyszukiwanie semantyczne, renderowanie streamów do HTML, frameworki AI. |

## Pewniaki (zawsze warto zaimplementować)

- Centralizacja interakcji z AI — rozproszone wywołania blokują zarządzanie ustawieniami i przełączanie modeli.
- Obsługa wielu providerów — otwartość na lepsze modele innych dostawców.
- Strumieniowanie (SSE) — informowanie o postępach, mniejszy czas reakcji.
- Multimodalność — projektować struktury danych tak, by dodanie obrazu/audio było trywialne, nawet jeśli początkowo przetwarzany jest tylko tekst.
- Logika agentów od dnia zero — np. tabela `items` zamiast `messages`, by przechowywać akcje między wiadomościami.
- Zdarzenia długoterminowe — użytkownik zamknie kartę, czas przekroczy limit połączenia.

## Elastyczność architektury w dobie AI

- System powinien być zaprojektowany tak, by **rozwój modeli wzmacniał jego możliwości** — nie budować na funkcjach, które LLM szybko opanuje samodzielnie.
- Dynamika zmian wymaga szybkich iteracji — zakres modyfikacji obejmujący to, co normalnie zajmowało kwartał.
- Ostrożność przy frameworkach AI — ich fundamenty wciąż się zmieniają. Oparcie całej aplikacji o rozwiązanie niestabilne = ryzyko blokady rozwoju.

## Integracja z wieloma providerami

Problem: więcej niż jeden provider **w tej samej logice** (np. agenci z różnymi modelami). Wymaga warstwy tłumaczeń, bo:

- Różnice w strukturze API: system prompt w OpenAI = element listy wiadomości, w Anthropic = oddzielne pole `system`, w Gemini = `system_instruction`.
- Różnice w ustawieniach: Anthropic/Gemini używają `budget_tokens`, OpenAI — `reasoning_effort`.
- Thought Signatures — wymagane przez Gemini 3+ przy narzędziach w bieżącej turze; Anthropic ma analogiczny mechanizm.
- Ograniczenia specyficzne: np. Anthropic nie przyjmuje wiadomości asystenta jako pierwszej.

| Podejście | Zalety | Wady |
|---|---|---|
| **OpenRouter** | Wygoda, jeden endpoint | Tylko podstawowe funkcje API, sporadyczne błędy mapowania/sygnatur |
| **Biblioteki (AI SDK, LiteLLM)** | Abstrakcja nad providerami | Blokują dostęp do najnowszych funkcji, istotne błędy o niskim priorytecie |
| **Własna logika** | Pełna kontrola, najlepsze dopasowanie | Cały ciężar utrzymania; z AI do kodowania — koszt akceptowalny |

Własna logika: określić domyślny format (obecnie: Responses API), sposób routingu na provider, zakres obsługiwanych ustawień.

## Architektura systemu wieloagentowego (05_01_agent_graph)

Łączy cztery wzorce:

1. **Orchestrator** — agent z narzędziami `create_actor` i `delegate_task`, zarządza zadaniami i subagentami.
2. **Blackboard** — współdzielony stan (sesja, zdarzenia, zadania, relacje, artefakty), którym agenci zarządzają przez narzędzia.
3. **DAG** — relacje zależności między zadaniami. Deterministyczny scheduler rozwiązuje zależności, promuje gotowe zadania, wstrzymuje oczekujące.
4. **Zdarzenia** — każda zmiana stanu emituje event przez SSE. Używane do wizualizacji, observability, guardrails.

### Scheduler — cykl życia zadania

Stany zadania: `todo → in_progress → done | waiting | blocked`

- **todo**: brak zależności lub zależności spełnione → scheduler promuje do `in_progress`.
- **waiting**: actor kończy pracę, ale scheduler wykrywa aktywne subzadania → czeka na ich zakończenie, potem promuje z powrotem do `todo`.
- **blocked**: nieusuwalna przeszkoda (błąd runtime, actor bez narzędzi) lub limit prób.
- Recovery: transient errors (rate limit, 5xx) → automatyczny retry z exponential backoff (max 3 próby). Zapisywane jako `TaskRecoveryState` na zadaniu.

### Przepływ rundy

```
while (ready_tasks exist):
  for each ready_task:
    set in_progress → find assigned actor → runActorTask()
    → completed? set done, unblock parents
    → waiting?    keep waiting (children still active)
    → blocked?    set blocked with recovery info
```

### Actor loop (runActorTask)

```
for step 1..maxSteps:
  processTaskMemory()        // compress history if needed
  buildTaskRunInput()         // assemble prompt from prefix + memory + snapshot + raw items
  generateToolStep()          // LLM call with tools
  if no tool calls:
    if has unfinished children → return waiting
    else → return completed/blocked
  for each tool call:
    execute → record invocation + result as Items
    if terminal outcome (complete_task/block_task) → return
```

### Kontekst aktora

Prompt składa się z:
1. **Prompt prefix** — stały: rola, zasady, rejestr agentów, instrukcje użycia narzędzi.
2. **Task memory** — skompresowane obserwacje z poprzednich kroków (sealed history).
3. **Task snapshot** — aktualny stan: lista aktorów, wszystkie zadania w sesji, zależności, artefakty.
4. **Raw items** — nieobserwowane zdarzenia (wiadomości, decyzje, invocations, results) od `lastObservedSeq`.

### Pamięć (Observational Memory)

Dwufazowy cykl, adaptacja [Mastra Observational Memory](https://mastra.ai/blog/observational-memory):

**Observer** — uruchamiany gdy raw items przekroczą `observationThresholdTokens` (800). Dzieli items na *head* (do skompresowania) i *tail* (ostatnie ~30% zachowane jako raw). Head → LLM → strukturalne obserwacje w XML z priorytetami (🔴🟡🟢).

**Reflector** — uruchamiany gdy skumulowane obserwacje przekroczą `reflectionThresholdTokens` (1500). Kompresuje do `reflectionTargetTokens` (800) przez wielopoziomową kompresję (level 0–2, coraz bardziej agresywna).

Stan pamięci trzymany na obiekcie `Task.memory: MemoryState` — observations, lastObservedSeq, observationTokens, generation.

### Model domenowy

| Encja | Kluczowe pola |
|---|---|
| **Session** | id, title, status (active/paused/done), usage |
| **Actor** | id, session_id, type (user/agent), name, status (active/idle), capabilities (tools, instructions) |
| **Task** | id, session_id, parent_task_id, owner_actor_id, title, status, priority, memory, recovery |
| **Item** | id, session_id, task_id, actor_id, type (message/decision/invocation/result), content, sequence |
| **Artifact** | id, session_id, task_id, kind (file/plan/diff/image), path, version |
| **Relation** | id, from_kind, from_id, relation_type (depends_on/assigned_to/produces/uses), to_kind, to_id |

### Rejestr narzędzi

| Narzędzie | Rola | Wynik |
|---|---|---|
| `create_actor` | Utworzenie/aktualizacja aktora z rejestru lub ad-hoc | `continue` |
| `delegate_task` | Utworzenie sub-zadania z przypisaniem aktora + opcjonalne `dependsOnTaskIds` | `continue` |
| `complete_task` | Zamknięcie zadania | `completed` (terminal) |
| `block_task` | Zablokowanie zadania z powodu | `blocked` (terminal) |
| `write_artifact` | Zapis artefaktu do FS + metadane w store | `continue` |
| `read_artifact` | Odczyt artefaktu po ID lub ścieżce | `continue` |

Narzędzia terminalne (`complete_task`, `block_task`) przerywają actor loop w danej turze. Pozostałe zwracają `continue` — LLM dostaje wynik i decyduje o kolejnym kroku.

### Event system

`emit(type, data)` → rozsyła do listenerów + buforuje (max 500). Używane przez SSE endpoint (`/events`) do streamowania do dashboardu oraz przez logger terminalowy. Replay na połączeniu — nowy klient dostaje całą historię.

## Uzupełnienia

### Abstrakcja providera w praktyce (config.js + client.ts)

Lekcja omawia warstwę tłumaczeń między providerami. W kodzie zaimplementowano ją jako:
- Pojedynczy klient OpenAI SDK z konfigurowalnym `baseURL`, `apiKey` i `headers` — pozwala przełączać się między OpenAI a OpenRouter bez zmiany kodu.
- `resolveModelForProvider()` mapuje nazwę modelu na format wymagany przez danego providera.
- Automatyczne wykrywanie modeli reasoningowych (`isReasoningModel` — dopasowanie `/^o\d/i`) i warunkowe dodawanie parametru `reasoning: { effort, summary: 'auto' }`.
- Retry z exponential backoff na poziomie klienta HTTP (`MAX_RETRIES = 2`) dla błędów 429/5xx i błędów sieciowych.

### Web search — dwa mechanizmy zależne od providera

W `client.ts` web search realizowany jest różnie w zależności od providera:
- **OpenAI**: dodanie `{ type: 'web_search_preview' }` do tablicy narzędzi API.
- **OpenRouter**: dopisanie suffiksu `:online` do nazwy modelu (`model:online`).
- To konkretny przykład różnic API między providerami, o których mówi lekcja.

### Prompt caching

Actor generuje stabilny klucz cache'owania: `sha256(session_id:task_id:actor_id).slice(0,48)` i przekazuje go jako `prompt_cache_key` do Responses API. Dzięki temu powtórne wywołania tego samego aktora korzystają z cache'u kontekstu (śledzone przez `cachedTokens` w `TokenUsage`). Cache hit rate jest raportowany w podsumowaniu sesji.

### Stale task recovery

Na początku `processSession()` uruchamiana jest funkcja `recoverStaleTasks()` — sprawdza zadania ze statusem `in_progress` i resetuje je do `todo`. Odpowiada za sytuację, gdy proces ulegnie awarii w trakcie wykonywania zadania. To uzupełnienie recovery opisanego w distillacji (transient LLM errors + auto-retry).

### Agent registry — wzorzec rejestracji agentów

Agenci definiowani są jako obiekty `AgentDefinition` z polami: `name, type, tools, instructions, webSearch?, maxSteps?`. Rejestr (`agents/index.ts`) mapuje nazwę → definicję. Tylko agenci z `bootstrapAgents` (`['orchestrator']`) są tworzeni przy starcie sesji. `create_actor` najpierw szuka w rejestrze — jeśli agent istnieje, pobiera jego tools/instructions automatycznie; jeśli nie, wymaga podania ich jawnie. Pozwala to na tworzenie specjalistów ad-hoc bez predefiniowanej konfiguracji.

### Dynamiczne vs statyczne planowanie

Lekcja kontrastuje scheduler z przykładu `05_01_agent_graph` ze "heartbeat" z `03_02_events`. W tamtym przypadku cały plan zadań i zależności był określany z góry. Tutaj plan jest **kształtowany dynamicznie** przez agenta zarządzającego — to daje większą elastyczność, ale mniejszą przewidywalność.

### Cykl życia statusu aktora

Po zakończeniu lub zablokowaniu zadania, `updateActorStatus()` sprawdza czy aktor ma jeszcze aktywne zadania (`status != done && status != blocked`). Jeśli tak → `active`, jeśli nie → `idle`. Zapobiega przypisywaniu zadań do nieaktywnych aktorów.

### Akumulacja użycia tokenów

`TokenUsage` jest akumulowany na poziomie sesji (`accumulateSessionUsage`) z dwóch źródeł: (1) actor loop (kumulatywne za wszystkie kroki), (2) operacje pamięci (observer + reflector). W podsumowaniu wyświetlany jest cache hit rate: `cachedTokens / inputTokens * 100%`.

## Przykłady kodu

### 1. Domain — typowe encje i relacje

Delta: schema domenowy z relacjami opartymi o generyczny `Relation` z `from_kind/from_id/relation_type/to_kind/to_id` — zastępuje specjalistyczne tabele joinujące. Jeden typ relacji obsługuje wszystkie powiązania między encjami.

```ts
export type RelationType = 'depends_on' | 'assigned_to' | 'produces' | 'uses'
export type EntityKind = 'session' | 'actor' | 'task' | 'item' | 'artifact'

export interface Relation {
  id: string; session_id: string
  from_kind: EntityKind; from_id: string
  relation_type: RelationType
  to_kind: EntityKind; to_id: string
  created_at: string
}
```

Czego uczy: generyczny model relacji (polimorficzny join) pozwala zapytania typu "znajdź wszystkie depends_on dla taska X" bez dedykowanych tabel per związek.

### 2. Graph queries — deterministyczna logika zależności DAG

Delta: scheduler nie jest LLM — to czysty kod sprawdzający warunki na stuctured data. `findReadyTasks` filtruje po statusie + rozwiązuje zależności, `unblockParents` kaskadowo promuje zadania po zakończeniu dzieci.

```ts
const findReadyTasks = async (sessionId: string): Promise<Task[]> => {
  const candidates = await rt.tasks.find(
    t => t.session_id === sessionId && (t.status === 'todo' || t.status === 'waiting' || t.status === 'blocked'),
  )
  const ready: Task[] = []
  for (const task of candidates) {
    if (task.status === 'todo' && !await areDependenciesMet(task)) continue
    if (task.status === 'waiting' && await areDependenciesMet(task) && !await hasUnfinishedChildren(task)) { ... }
    if (task.status === 'blocked' && shouldAutoRetryTask(task)) { ... }
    ready.push(task)
  }
  return ready.sort((a, b) => a.priority - b.priority)
}
```

Czego uczy: rozdzielenie deterministycznej logiki (scheduler) od niedeterministycznej (LLM actor) — scheduler zarządza cyklem życia zadań, actor wykonuje pracę. Maksymalnie 20 rund, po czym pętla się kończy.

### 3. Actor loop — agent z narzędziami w strukturze grafowej

Delta względem poprzednich agent-loop: kontekst budowany dynamicznie ze snapshotu całej sesji (aktorzy, zadania, artefakty, zależności) + observational memory. Tool calls zapisywane jako Items z sekwencją. Narzędzia terminalne przerywają loop.

```ts
for (let step = 1; step <= maxSteps; step++) {
  await processTaskMemory(task.id, rt)
  const input = await buildTaskRunInput(task, rt, graph, promptPrefix)
  const response = await generateToolStep({ instructions, input, tools, promptCacheKey })
  // record decision text as Item
  // execute each tool call → record invocation + result as Items
  // if terminal outcome → return
}
```

Czego uczy: actor nie zarządza swoim kontekstem — scheduler wstrzykuje pełny snapshot sesji. Narzędzia mają statusy terminalne (`completed`/`blocked`) vs kontynuujące (`continue`). Prompt cache key stabilizuje cache'owanie przez `sha256(session:task:actor)`.

### 4. Observational Memory — observer + reflector

Delta: dwufazowa kompresja historii zadania z zachowaniem ciągłości. Observer zamienia raw items na ustrukturyzowane obserwacje z priorytetami. Reflector kompresuje narastające obserwacje wielopoziomowo.

```
raw items (sequence > lastObservedSeq)
  → estimate tokens → above threshold?
    → split into head (compress) + tail (keep raw, ~30%)
    → Observer: LLM extracts 🔴🟡🟢 observations
    → merge with existing observations
    → above reflection threshold?
      → Reflector: multi-level compression (0=gentle, 1=aggressive, 2=heavy)
    → persist on Task.memory
```

Czego uczy: pamięć jest per-task-thread, nie per-session. Pozwala każdemu zadaniu mieć własną skompresowaną historię. Split head/tail chroni ostatnie interakcje przed kompresją. Generation counter śledzi ile razy reflektor był uruchamiany.
