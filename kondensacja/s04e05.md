# S04E05 — Projektowanie rozwiązań wewnątrzfirmowych

## Trzy wymiary wdrożeń AI

| Wymiar | Zakres |
|---|---|
| Biznesowy | koszty + utrzymanie, compliance (Bedrock/Azure), komunikacja ograniczeń do biznesu |
| Kulturowy | warsztaty, wymiana doświadczeń, inicjatywy oddolne — adopcja wymaga zaangażowania na każdym poziomie organizacji |
| Technologiczny | zakres projektu, ewaluacja modeli, architektura agentów, założenie <100% skuteczności |

Niedeterminizm modeli + konieczność zmiany nawyków + różnice w doświadczeniu AI między użytkownikami = główne przeszkody adopcji. Użytkownik świadomy mechanik agenta pisze zapytania zawierające niejawne wskazówki (np. wymienia konkretne źródła danych); użytkownik nieswiadomy pisze ogólnikowo i dostaje gorsze wyniki.

## Narzędzia AI bez linii kodu — dokument jako interfejs

Najmniej inwazyjne wdrożenia AI to dokumenty/prompty podłączane do istniejących interfejsów (Claude, Slack):

- **Checklista** — stały, powtarzalny proces (np. weryfikacja SEO artykułu). AI weryfikuje kompletność, człowiek potwierdza.
- **Onboarding** — dokument kierujący nową osobę we właściwe miejsce. AI dopasowuje zapytanie semantycznie, nawet gdy fraza jest daleka od zawartości.
- **Styl** — prompt opisujący spójny styl wizualny, współdzielony przez zespół (np. generator grafik dla całego projektu).

Paradygmat: AGENTS.md i Skills to ten sam wzorzec — deklaratywne instrukcje w plikach — przeniesiony do kontekstu firmowego.

## Bezpieczeństwo i prywatność danych

Dostęp do zaufanego dostawcy (Bedrock/Azure) ≠ bezpieczeństwo. Wektory ryzyka:

- Agent z dostępem do Internetu może exfiltrate dane.
- Agent z możliwością wykonania kodu może usunąć/uszkodzić źródła danych.
- LLM wprowadza w błąd — czatbot z bazą wiedzy może zasugerować nieprawidłową akcję.
- Agent może przypadkowo wywołać narzędzie z destrukcyjnym efektem (send_email na zły adres).
- Modele potrafią wykrywać, że są testowane, i dostosowywać zachowanie (System Card: Claude Opus 4.6, Eval Awareness).

Konkluzja: ograniczanie uprawnień i **fizyczne uniemożliwienie** akcji na poziomie kodu, nie instrukcji. Nie zakładać, że coś jest niemożliwe — analizować opcje, nawet jeśli wykluczają pełną automatyzację.

## Przegląd dokumentu — agent blokowy (04_05_review)

### Architektura

Dokument Markdown → AST (remark) → bloki (heading, paragraph, list_item, blockquote). Każdy blok dostaje ID (`b1`, `b2`, ...). Agent przetwarza bloki sekwencyjalnie lub równolegle i decyduje, czy dodać komentarz przez narzędzie `add_comment`.

### Dwa tryby pracy

| Tryb | Opis | Przetwarzanie |
|---|---|---|
| `paragraph` | jeden blok = jedno wywołanie agenta | kolejka + worker pool (concurrency 4) |
| `at_once` | cały dokument = jedno wywołanie agenta | jeden agent loop ze wszystkimi blokami |

### Narzędzie `add_comment` — kotwiczenie komentarzy w tekście

Agent nie edytuje dokumentu bezpośrednio. Zamiast tego wywołuje narzędzie z parametrami: `block_id`, `quote` (exact match), `kind` (comment/suggestion), `severity`, `title`, `comment`, `suggestion`. Handler:

1. Weryfikuje, że `block_id` jest w zakresie bieżącego review.
2. Znajduje pozycję cytatu w tekście bloku (`findQuoteRange`) — wymaga unikalnego matchu.
3. Sprawdza brak nakładania się z istniejącymi komentarzami (`overlaps`).
4. Zapisuje komentarz z pozycją (`start`, `end`) do sesji.

### Akcje po review

- **Accept** — podmienia tekst w bloku, przesuwa pozycje sąsiednich komentarzy (delta = `len(suggestion) - len(original)`), stales overlapping comments.
- **Reject** — oznacza komentarz jako odrzucony.
- **Revert** — przywraca oryginalny tekst bloku ze `originalBlockText` zapisanego przy accept.
- **Rerun block** — stales otwarte komentarze dla bloku + ponownie uruchamia agenta z opcjonalnym `customMessage`.

### Generowanie podsumowania

Po zakończeniu review osobne wywołanie LLM generuje 1-2 zdania podsumowania na podstawie statystyk (status counts, severity counts, kind counts) + top 10 komentarzy posortowanych po severity/status. Fallback: generowane deterministycznie gdy LLM zawiedzie.

### Konfiguracja przez pliki workspace

- `workspace/system/agents/reviewer.md` — profil agenta (frontmatter: model, title) + instrukcje systemowe.
- `workspace/prompts/*.md` — prompty review z frontmatter: `modes` (dostępne tryby), `contextFiles` (dodatkowe pliki kontekstowe, np. sitemap).
- `workspace/documents/*.md` — dokumenty do review z frontmatter.

## Generatywne interfejsy — MCP Apps (04_05_apps)

### Problem

Agent czatowy podłączony do wielu usług (CRM, Stripe, newsletter) daje dostęp do danych, ale nie daje wygodnego interfejsu do ich przeglądania i modyfikowania. Surowy tekst z czatu nie zastąpi dedykowanego UI.

### Architektura trójwarstwowa

1. **Serwer MCP** — rejestruje narzędzia (`registerAppTool`) i zasoby UI (`registerAppResource`). Narzędzia dzielą się na:
   - **App tools** — wywoływane przez model, zwracają dane + `structuredContent` + link do zasobu `ui://`.
   - **App-only tools** (`visibility: ["app"]`) — dostępne tylko z wnętrza iframe'a aplikacji, ukryte przed modelem.

2. **Host** (przeglądarka) — renderuje czat, montuje aplikacje w sandboxed iframe, proxy'uje żądania przez `AppBridge`.

3. **Embedded App** (iframe) — łączy się z hostem przez MCP Apps SDK (`new App()`). Cykl życia: `connect()` → `ontoolinput` / `ontoolresult` → interakcja użytkownika → `callServerTool()` / `updateModelContext()` → `onteardown`.

### Przepływ danych

```
user prompt → model wybiera app tool → serwer MCP zwraca text + structuredContent + ui://resource
→ host montuje iframe z HTML zasobu → app w iframe odbiera tool result (structuredContent)
→ użytkownik wchodzi z appką → app może: callServerTool, updateModelContext, openLink
→ model dostaje zaktualizowany kontekst z appki (appContexts)
```

### Kluczowe mechaniki SDK

- `app.callServerTool({ name, arguments })` — wywołanie narzędzia MCP z wnętrza iframe.
- `app.updateModelContext({ content, structuredContent })` — wstrzykuje stan appki do kontekstu modelu (debounced 120ms). Model widzi aktualny snapshot.
- `app.openLink({ url })` — prośba do hosta o otwarcie linku (np. Stripe, Linear).
- `app.ontoolresult` — callback z wynikiem narzędzia po wywołaniu przez model.
- `app.onteardown` — wywoływany przy demontażu; app może zapisać dirty state.

### Narzędzia app-only vs model-visible

Serwer MCP rejestruje narzędzia persystencji (np. `get_todos_state`, `save_todos_state`) z `visibility: ["app"]`. Model nie widzi tych narzędzi — nie może ich wywołać. Służą wyłącznie komunikacji app↔serwer. To realizuje zasadę ograniczania uprawnień na poziomie kodu, nie instrukcji.

### Fallback bez AI

Gdy brak klucza API (`hasAiAccess() === false`), agent używa regex-based routingu: matchuje wzorce z wiadomości użytkownika do konkretnych narzędzi MCP. Traktuje `"show sales this month"` → `open_sales_analytics({ from, to })`. System działa jako deterministyczny router bez udziału LLM.

### Skalowalność MCP Apps

Ten sam serwer MCP może być podłączony do wielu klientów (Claude.ai, własny host). MCP Apps nie zastępują agentów — są z nimi komplementarne. Generatywne interfejsy adresują sytuacje, gdy sam czat nie wystarcza: dane z wielu źródeł, scoped UI zamiast raw JSON, operacje pozostające pod kontrolą kodu.

## Uzupełnienia

### NDJSON — streaming postępu review w czasie rzeczywistym

Review API (`server.js`) streamuje zdarzenia jako `application/x-ndjson` — każda linia to osobny JSON. Zdarzenia: `started` → `block_start` → `comment_added` → `block_done` → `summary_start` → `complete`. Frontend odbiera je iteracyjnie (ReadableStream), co pozwala na aktualizację UI bez czekania na koniec całego review. Wzorzec istotny dla długotrwałych zadań agentowych, gdzie użytkownik potrzebuje natychmiastowej informacji zwrotnej.

### Batch accept — sortowanie odwrotne chroniące pozycje

`batchAcceptComments` sortuje sugestie malejąco po pozycji (`b.start - a.start`) i aplikuje od końca bloku. Każda podmiana tekstu zmienia długość i przesuwa kolejne pozycje — przetwarzanie od końca zachowuje poprawność `(start, end)` niezmnodyfikowanych komentarzy. Analogicznie `acceptReviewComment` po pojedynczym accept przesuwa pozycje sąsiadujących komentarzy o `delta = len(suggestion) - len(original)`.

### `parallel_tool_calls: false` — sekwencyjne wywołania narzędzi

Agent w `04_05_apps` wysyła `parallel_tool_calls: false` w requescie do Responses API. Zapobiega to równoległemu wywołaniu wielu narzędzi w jednym kroku — krytyczne gdy narzędzia mają efekty uboczne (np. create_coupon → open_coupon_manager). Domyślnie modele mogą planować parallel calls.

### Re-anchoring komentarzy po edycji dokumentu (`resolveCommentRange`)

Po operacji accept tekst bloku się zmienia, więc zapisane `(start, end)` mogą być nieaktualne. `resolveCommentRange` najpierw waliduje zapisane pozycje (`text.slice(start, end) === quote`), a jeśli nie pasują — fallbackuje do `findQuoteRange` (wyszukiwanie tekstu). Podobnie `hydrateReviewForDocument` przy ładowaniu sesji z dysku ponownie kotwiczy komentarze do aktualnego stanu dokumentu. Graceful degradation — komentarz nie ginie, nawet jeśli tekst drgnął.

### Auto-reconnect klienta MCP

`createMcpRuntime` opakowuje MCP client w mechanizm transparentnej rekonstrukcji sesji. `withClient(callback)` łapie błędy `Invalid session` / `Mcp-Session-Id required`, tworzy nową Connection i powtarza callback. Gwarantuje, że pojedynczy utracony session-id nie psuje całego czatu.

### `appContexts` — kolekcja snapshotów z aktywnych apek

Frontend (`chat.js`) utrzymuje `Map<resourceUri, context>` z ostatnim snapshotem `updateModelContext` każdej zamontowanej appki. Każda wiadomość do `/api/chat` niesie `appContexts: serializeAppContexts()` — model otrzymuje aktualny stan wszystkich otwartych interfejsów. Deduplikacja po `resourceUri` zapewnia, że każda appka ma dokładnie jeden najświeższy snapshot.

### Eksperymentowanie jako mechanizm discovery

Źródło podkreśla: prezentacja działającego prototypu stakeholderom generuje organicznie nowe wymagania — "pojawiły się potrzeby na inne narzędzia, ponieważ prezentacja pokazała możliwości, o których wcześniej nikt nie pomyślał". Strategia: budować małe, pokazywać wcześnie, iterować na podstawie reakcji.

### MCP jako interfejs integracyjny B2B

Serwer MCP może być udostępniony klientom zewnętrznym — jeśli ich interfejs wspiera MCP Apps, integracja jest transparentna. Ten sam serwer `04_05_apps` można podłączyć bezpośrednio do Claude.ai. Zmienia to MCP z narzędzia wewnętrznego na platformę integracyjną.

### Generatywne interfejsy poza czatem

MCP Apps nie ograniczają się do interfejsu czatu — mogą być osadzane w dowolnych miejscach aplikacji, gdzie AI działa bez swobodnego pola tekstowego (np. panele administracyjne, dashboardy). Czat to najczęstszy, ale nie jedyny kontekst.

## Przykłady kodu

### 1. Agent loop — narzędzia z rejestru (04_05_review)

Delta względem S04E01: uproszczony agent loop bez `previous_response_id`, z dynamicznym rejestrem narzędzi.

```js
const runAgent = async ({ input, instructions, model, tools }) => {
  const definitions = tools.map((t) => t.definition);
  const handlers = Object.fromEntries(
    tools.map((t) => [t.definition.name, t.handler]),
  );
  let conversation = [{ role: "user", content: input }];

  for (let step = 0; step < MAX_STEPS; step += 1) {
    const response = await complete({ input: conversation, tools: definitions, instructions, model });
    const calls = response.output.filter((i) => i.type === "function_call");
    if (calls.length === 0) return { text: response.text, conversation };

    const results = [];
    for (const call of calls) {
      const output = await handlers[call.name](JSON.parse(call.arguments ?? "{}"));
      results.push(call, {
        type: "function_call_output",
        call_id: call.call_id,
        output: JSON.stringify(output),
      });
    }
    conversation = [...conversation, ...results];
  }
};
```

Czego uczy: `tools` to para (definition, handler) — separacja deklaracji (dla modelu) od implementacji. Handler zwraca plain object serializowany do JSON — model dostaje wynik jako `function_call_output`. Loop kończy się, gdy model nie wywołuje narzędzia (zwraca sam tekst).

### 2. Narzędzie add_comment — walidacja + kotwiczenie (04_05_review)

Delta: handler narzędzia z wielopoziomową walidacją, zapobiegający nakładaniu się komentarzy.

```js
const range = findQuoteRange(block.text, quote);
if (range.error) return { status: "rejected", error: range.error };

const existing = session.comments.find((entry) =>
  entry.blockId === block_id && entry.status === "open" && overlaps(entry, range)
);
if (existing) return { status: "rejected", error: `Quote overlaps with existing comment ${existing.id}.` };
```

Czego uczy: `findQuoteRange` wymaga unikalnego matchu cytatu w bloku — jeśli cytat pojawia się wielokrotnie, handler odrzuca i prosi o dłuższy fragment. `overlaps` sprawdza nakładanie się zakresów. Model dostaje informację zwrotną jako rejected status i może poprawić próbę.

### 3. Review engine — worker pool z kolejną (04_05_review)

Delta: kontrolowana współbieżność przez kolejkę zamiast `Promise.all` na wszystkich blokach.

```js
const queue = reviewableBlocks.map((block, i) => ({ block, i }));
const workers = Array.from({ length: Math.min(PARAGRAPH_REVIEW_CONCURRENCY, queue.length) }, async () => {
  while (queue.length > 0) {
    const { block, i } = queue.shift();
    await reviewBlock(block, i);
  }
});
await Promise.all(workers);
```

Czego uczy: worker pool z kolejną (N workerów, 1 współdzielona kolejka) zamiast chunków — każdy worker bierze kolejny blok gdy skończy poprzedni. Równomierny rozkład loadu niezależnie od różnic w czasie przetwarzania bloków.

### 4. MCP App — rejestracja narzędzia z zasobem UI (04_05_apps)

Delta: `registerAppTool` wiąże narzędzie z zasobem `ui://` — model wywołuje narzędzie, host automatycznie montuje iframe.

```js
const appTool = (name, config, handler) => registerAppTool(server, name, {
  ...config,
  _meta: { ui: { resourceUri: TODO_RESOURCE_URI, ...config._meta?.ui } },
}, handler);

appTool("open_todo_board", { title: "Open todo board", inputSchema: {} }, async () => {
  const state = await readTodosState();
  return {
    content: [{ type: "text", text: `Opened todo board. ${summarizeTodos(state)}.` }],
    structuredContent: toTodoPayload(state),
  };
});
```

Czego uczy: wynik narzędzia zawiera `structuredContent` — dane dla iframe'a. `_meta.ui.resourceUri` wskazuje zasób HTML montowany w sandboxie. Model dostaje tekst, iframe dostaje dane strukturalne — ten sam endpoint, dwa konsumenci.

### 5. MCP App SDK — komunikacja iframe↔host (04_05_apps)

Delta: embedded app w iframe łączy się z hostem przez SDK i może wywoływać narzędzia serwera + aktualizować kontekst modelu.

```js
const app = new App({ name: "Todo Board", version: "0.1.0" });

app.ontoolresult = (result) => applyState(result?.structuredContent ?? {});

const save = async () => {
  const result = await app.callServerTool({ name: "save_todos_state", arguments: { items: state.items } });
  applyState(resultToState(result));
};

const syncModelContext = async () => {
  await app.updateModelContext({
    content: [{ type: "text", text: `Todo board snapshot: ${state.items.length} total` }],
    structuredContent: { kind: "todos", items: state.items },
  });
};
```

Czego uczy: `ontoolresult` — app reaguje na wynik narzędzia wywołanego przez model. `callServerTool` — app wywołuje app-only tool niedostępny dla modelu. `updateModelContext` — debounced sync wstrzykuje snapshot stanu appki do kontekstu kolejnego zapytania modelu. Trójkąt: model ↔ host ↔ app.
