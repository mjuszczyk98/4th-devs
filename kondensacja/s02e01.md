# S02E01 — Zarządzanie kontekstem w konwersacji

## Instrukcja systemowa jako mapa, nie encyklopedia

- Rola promptu systemowego spada — agenci dynamicznie odkrywają potrzebny kontekst. Instrukcja daje **świadomość otoczenia**, nie kompletne informacje.
- Umieszczaj w prompcie **zgeneralizowane** opisy ról narzędzi/pamięci, a nie szczegóły ich działania. Narzędzie może być podpięte do różnych agentów z różną rolą.
- Pytanie projektowe: **o czym agent musi wiedzieć PRZED uruchomieniem narzędzi**, aby skrócić ścieżkę do celu?

**Cztery warstwy promptu systemowego:**

| Warstwa | Typ treści | Przykład |
|---|---|---|
| Uniwersalne instrukcje | Generalizowany opis ról (pamięć, osobowość) | "Posiadasz pamięć długoterminową — profile osób, preferencje" |
| Otoczenie | Środowisko, metadata, interfejs | OS, typ interakcji (głos/CLI/CRON), zasoby |
| Sesja | Stan bieżącej interakcji, kompresja historii | "Oto podsumowanie wcześniejszej rozmowy…" |
| Zespół agentów | Współdzielone zasady, placeholder'y | Reguły komunikacji, współdzielenie plików |

## Sygnał vs szum — warunki, nie kontrola

- Nie da się w pełni kontrolować proporcji sygnału. Cel: **stworzyć warunki** maksymalizujące sygnał poprzez detale systemu.
- Granica między logiką w kodzie a logiką w AI przesuwa się — kod maleje ilościowo, ale musi być bardziej dopracowany.
- Błędy w głównym systemie agenta nie są do ominięcia (w przeciwieństwie do błędów obsługi narzędzi).

**Co buduje sygnał:**
- Poprawne, kompletne dane wejściowe (konflikty narzędzi, mylne API — częste źródła problemów)
- Dopracowane instrukcje jako **komponenty** pojawiające się dynamicznie, nie tylko statyczny prompt
- Generyczne mechanizmy (kompresja kontekstu, planowanie, monitorowanie)
- Przestrzeń na interwencję człowieka/agenta — dostarczenie sygnału gdy agent utknie

## Agentic Search — kontekst przez obserwację

- Agent domyślnie **nie wie, o czym wie**. Musi być pokierowany do sięgania po zewnętrzne informacje.
- Agentic RAG: agent samodzielnie iteruje wyszukiwanie — obserwuje wyniki, koryguje zapytania, pogłębia eksplorację.
- Kluczowa sekwencja: wyszukiwanie → korekta na podstawie wyników → wykorzystanie odkrytych terminów do kolejnych zapytań.
- Brak gwarancji 100% pokrycia — ale skuteczność bardzo wysoka przy odpowiednich zasadach.

**Zasady sterujące eksploracją (fragmenty promptu):**

| Zasada | Działanie |
|---|---|
| Skanowanie | Eksploruj strukturę folderów, nazwy plików, nagłówki |
| Pogłębianie | Iteracyjne wyszukiwanie: synonimy, terminologia z odkrytych fragmentów |
| Eksplorowanie | Tropy powiązane: przyczyna/skutek, część/całość, problem/rozwiązanie |
| Weryfikacja pokrycia | Czy mam definicje, limity, wyjątki? Jeśli nie — wróć do pogłębiania |

## Generalizowanie instrukcji

- Generalizacja instrukcji = najważniejsza umiejętność przy budowie agentów. Podobna do projektowania generycznych komponentów architektonicznych.
- Brak walidatorów/promilu błędów — tylko obserwacja i iteracja.
- Modele potrafią **uzasadnić** swoje zachowanie i zasugerować zmiany, ale proponują zbyt bezpośrednie (sztywne) instrukcje. Rolą człowieka jest generalizacja.

**Proces iteracji instrukcji z LLM:**
1. **Analiza:** opisz problem, poproś o diagnozę przyczyn zachowania
2. **Generalizacja:** poproś o uniwersalne przyczyny, nie fix konkretnego przypadku
3. **Korekta:** odrzuć ~60% sugestii, doprecyzuj kierunek, podkreśl niezależność od narzędzi
4. **Iteracja:** wskazuj konkretne błędy w propozycjach — model sam wyciąga esencję

## Struktura dynamiczna — cache hit i sterowanie uwagą

- Definicje narzędzi leżą **pod** instrukcją systemową — zmiana promptu = zerwanie cache narzędzi.
- Dane dynamiczne (gałąź gita, otwarte pliki, data) → wiadomość użytkownika, nie prompt systemowy. Tagi XML-like oddzielają metadane od treści zapytania.
- W kolejnych wiadomościach — tylko zmieniające się dane. Agent odświeża resztę przez narzędzia.
- Powtarzanie kluczowych instrukcji w kolejnych wiadomościach steruje **uwagą modelu** — z czasem gubi kontekst.

## Sterowanie uwagą bez narzędzi operacyjnych

- **Lista zadań:** model zapisuje → odhacza → przepisuje resztę. Generowane przez model treści silniej wpływają na jego zachowanie (koncepcja many-shot influence).
- **Tryb planowania:** sygnał w wiadomości użytkownika zmienia zachowanie bez modyfikacji promptu systemowego. Cache bezpieczny.
- Oba mechanizmy mają odbicie w UI — nie tylko w kontekście modelu.

## Kontekst poza oknem kontekstowym — Agent Harness

- Agent Harness = cała infrastruktura wokół modelu. Sesje, pamięć, pliki, komunikacja między agentami — to wszystko działa poza bieżącym oknem.
- Pamięć długoterminowa może być budowana asynchronicznie (np. raz/dobę, Batch API = niższe koszty).
- Pliki: komunikacja między narzędziami, kolaboracja agentów, umiejętności, pamięć. Mogą powstawać bez agentów i być później wykorzystane.

## Workspace — organizacja przestrzeni agentów

- Podział na **sesje** (katalogi dat) — programistyczne ograniczenie dostępu między użytkownikami.
- Podział na role katalogów: `notes/`, `outbox/` (zapis agenta), `inbox/` (zapis root → sub-agent).
- Root przekazuje dokumenty między agentami — sub-agenci nie komunikują się bezpośrednio.

## Maskowanie kontekstu (Manus)

- Technika: uzupełnienie początkowego fragmentu wypowiedzi modelu tokenami wywołania konkretnego narzędzia → ogranicza dostępne akcje bez usuwania definicji.
- Deterministyczne wykrycie serii powiązanych akcji → prefilling tokenów narzędzia przeglądarki.
- Oznaczone jako deprecated w API Anthropic — rzadko dostępne, ale pokazuje kreatywne podejścia do sterowania kontekstem.

## Przykłady kodu — progresja

### 1. System prompt — zgeneralizowane zasady wyszukiwania

Delta: Prompt nie opisuje *co* znaleźć, lecz *jak szukać*. Sekcje `SCAN / DEEPEN / EXPLORE / VERIFY` są uniwersalne — działają niezależnie od zawartości dokumentów. Jedyny specyficzny element: informacja o języku dokumentów (polski) i źródle (AI_devs).

```
## SEARCH GUIDANCE
- Scan: explore structure first, then search
- Deepen (multi-phase): search → read → collect new terms → re-search
- Explore: follow cause/effect, part/whole, problem/solution leads
- Verify coverage: definitions? limits? edge cases? → loop back if gaps
```

### 2. Pętla agentowa — agentic loop z historią

Delta: Struktura `chat → tool_calls → results` w pętli z `MAX_STEPS`. Historia konwersacji jest przekazywana między zapytaniami — agent kontynuuje kontekst. Klucz: narzędzia MCP są mapowane do formatu OpenAI dynamicznie, model sam decyduje o ich wywołaniu.

```js
const messages = [...conversationHistory, { role: "user", content: query }];
for (let step = 1; step <= MAX_STEPS; step++) {
  const response = await chat({ input: messages, tools });
  const toolCalls = extractToolCalls(response);
  if (toolCalls.length === 0) return { response, conversationHistory: messages };
  messages.push(...response.output);
  messages.push(...await runTools(mcpClient, toolCalls));
}
```

### 3. REPL — sesja z zachowaniem kontekstu

Delta: REPL utrzymuje `conversation.history` między zapytaniami. Komenda `clear` resetuje historię i statystyki. Ilustruje podział: REPL zarządza cyklem życia sesji, agent loop zarządza pojedynczym zapytaniem, a prompt systemowy zarządza zachowaniem.
