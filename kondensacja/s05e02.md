# S05E02 — Zestaw narzędzi

## Architektura UI agenta — Event Sourcing

Agent UI oparty na **strumieniu zdarzeń** (SSE), nie na modelu wiadomości. Każdy token, wywołanie narzędzia, reasonig — to osobne zdarzenie z `id`, `seq`, `messageId`, `at`. Zdarzenia materializują się w bloki renderowe po stronie klienta.

Dwie ścieżki materializacji:
- **Pełna** (`materializeBlocks`) — O(n), hydratacja historii z serwera
- **Inkrementalna** (`applyEvent`) — O(1), mutacja istniejącej listy bloków na każde zdarzenie SSE. Klucz do wydajności przy streaming.

**Typy zdarzeń:** `text_delta`, `thinking_start/delta/end`, `tool_call`, `tool_result`, `artifact`, `error`, `complete`. Każdy typ mapuje się na dedykowany blok UI.

## Streaming Markdown — inkrementalny rendering

Problem: re-render całej wiadomości na każdy token = katastrofa wydajnościowa.

**Rozwiązanie: inkrementalny widok Markdown.** Treść dzielona na niezależne bloki (akapit, nagłówek, code fence). Podział na `committedSegments` (sfinalizowane, nie zmienią się) + `liveTail` (ostatni niekompletny blok, re-renderowany). Tylko liveTail jest parsowany na każdą deltę — finalized segmenty zostają w DOM bez dotykania.

**Łączenie (compaction):** gdy delta trafia do liveTail i `parseMarkdownIntoBlocks` dzieli go na >1 blok, pierwszy blok jest „promowany" do committedSegments. Można to odroczyć (`compactionDeferred`) gdy DOM nie jest gotowy.

## Naprawa niekompletnego Markdownu

Podczas streaming model generuje otwartą składnię — niezamknięte code fence, nieparzyste `**`, niekompletne tabele.

- **remend** — automatyczna naprawa niekompletnego MD (otykanie znaczników)
- Ręczne dodanie zamykającego ` ``` ` gdy `hasIncompleteCodeFence` wykryje problem
- Zabezpieczenie przed pomyleniem `*` na końcu elementu listy z list markerem — wstrzyknięcie zero-width space

## Renderowanie Markdown

**markdown-it** + **highlight.js** + **DOMPurify** — warstwa: tokenizacja → kolorowanie składni → sanityzacja HTML. Krytyczne: DOMPurify usuwa złośliwe tagi z outputu LLM (XSS prevention).

Optymalizacje:
- Cache renderowania — `Map<string, string>` z limitem 400 wpisów, klucz = `highlight:appendCaret:content`
- Podczas streaming: `highlight: false` (taniejsze), pełne highlight dopiero po zakończeniu
- Zaznaczenie kursora (`injectCaret`) — wstrzyknięcie `<span class="caret-blink">` przed ostatni zamykający tag blokowy

## Typewriter — kontrola tempa strumienia

Cztery tryby (`off/fast/normal/slow`) definiujące `burst` (tokenów na flush) i `interval` (ms). „Off" = instant, bez sztucznego opóźnienia.

## Wirtualizacja listy wiadomości

Chunk-based virtualization zamiast per-message. Wiadomości grupowane po `CHUNK_SIZE` (12). Chunk jest jednostką montowania/demontażu w DOM.

**Mechanika:**
- `ResizeObserver` mierzy wysokość każdego chunka w locie
- Niezmierzone chunki estymowane na podstawie średniej z dotychczasowych pomiarów
- `visibleRange` — które chunki są w viewport + overscan (3 chunki marginesu)
- Top/bottom spacer — absolutne pozycjonowanie, `overflow-anchor: none`
- **Scroll anchoring:** gdy chunk powyżej viewport zmienia wysokość, `scrollTop` kompensuje deltę

**Pin/unpin do dołu:** asymetryczne progi — `PIN_RELEASE_DISTANCE` (80px) żeby odpinać, `PIN_REACQUIRE_DISTANCE` (4px) żeby przypiąć z powrotem. `scrollGuardUntil` i `bottomLockUntil` — czasowe blokady chroniące przed wyścigami.

## Batchowanie SSE — requestAnimationFrame

Zdarzenia SSE trafiają do bufora `pending[]`, flush raz na `requestAnimationFrame`. Zapobiega render thrashing — DOM aktualizuje się maks. 60fps, niezależnie od tempa streamingu.

## Agent server — agentic loop ze streaming

Serwer utrzymuje konwersację jako tablicę `ConversationMessage` z eventami. Na żądanie:
1. Konwersja historii → `input` dla Responses API
2. Streaming w pętli (max 6 kroków, tool use w każdym)
3. Każdy krok: parsowanie strumienia OpenAI → emitowanie własnych `StreamEvent`ów
4. Po wywołaniu narzędzi: tool output → kolejny krok loopa
5. `parallel_tool_calls: true` — równoległe wywołania narzędzi w jednym kroku

**Konwersja zdarzeń:** `response.output_text.delta` → `text_delta`, `response.reasoning_summary_text.delta` → `thinking_delta`, `response.function_call_arguments.done` → `tool_call`. Serwer tłumaczy format providerowy na własny, zunifikowany model zdarzeń.

## Uzupełnienia

### parseMarkdownIntoBlocks — edge cases

`marked` Lexer tokenizuje MD na bloki, ale same tokeny nie wystarczą. Trzeba scalić sąsiadujące bloki w przypadkach:

- **Zagnieżdżone tagi HTML** — `htmlStack` śledzi otwarte tagi; bloki są scalane dopóki stos się nie opróżni
- **Niezamknięte bloki matematyczne `$$`** — parzystość `$$` decyduje czy blok jest kompletny; nieparzysta liczba = scal z następnym
- **Footnotes** (`[^id]` / `[^id]:`) — zawsze traktowane jako jeden blok (ref + definicja muszą być w jednym drzewie parsowania)

Bez tych reguł inkrementalny rendering rozbiłby struktury wymagające kontekstu.

### Deduplikacja zdarzeń SSE po stronie klienta

`chat-store` utrzymuje `eventIdsByMessageId: Map<string, Set<string>>` oraz `lastSeq` per wiadomość. Zdarzenie jest ignorowane gdy `knownIds.has(event.id)` lub `event.seq <= message.lastSeq`. Chroni to przed podwójnym renderowaniem przy retransmisjach lub wyścigach sieciowych.

### `store: false` w Responses API

Serwer wysyła `store: false` w żądaniu do OpenAI Responses API — konwersacja nie jest persystowana po stronie providera. Pełna historia utrzymywana wyłącznie w `ConversationStore` na serwerze, co daje pełną kontrolę nad kontekstem.

### Prewarm w agencie głosowym

`defineAgent({ prewarm })` ładuje model VAD (Silero) i inicjalizuje MCP connections **zanim** uczestnik dołączy do pokoju. Eliminuje latencję startową przy pierwszym połączeniu.

### autoToolReplyGeneration — workaround deadlock'u Gemini

Domyślnie `google.beta.realtime.RealtimeModel` ma `autoToolReplyGeneration: true`, co powoduje deadlock w pętli speech-queue drain frameworka Livekit przy Gemini Live. Wymagany override:

```js
Object.defineProperty(realtimeModel, "capabilities", {
  get: () => ({ ...nativeCaps, autoToolReplyGeneration: false }),
})
```

Ręczne generowanie tool-reply (analogiczne do ścieżki OpenAI) omija problem.

### Token server — JWT dla LiveKit

Osobny mikroserwer (Hono) generuje `AccessToken` (JWT) z grantem `roomJoin + canPublish + canSubscribe`. Frontend pobiera token, łączy się z LiveKit server przy użyciu JWT. `room` i `identity` generowane per sesja.

### Mock/demo mode — testowanie UI bez LLM

Serwer wspiera tryb `mock` z predefiniowanymi scenariuszami (sales, email, artifact, research). `ScenarioBuilder` generuje sekwencję `StreamEvent`ów z konfigurowalnymi opóźnieniami. `detectScenario()` dopasowuje prompt regexem do scenariusza. Umożliwia testowanie UI z 1500+ wiadomościami bez kosztów API.

### Brakujące elementy UI agenta (źródło)

Lekcja wymienia funkcje których brakuje w przykładowym UI, a które są potrzebne w produkcji: załączniki (obrazy/dokumenty), kopiowanie z formatowaniem, rozgałęzianie/edycja wiadomości, usuwanie, wstrzymywanie generacji, przeszukiwanie konwersacji, wysyłanie podczas inferencji, LaTeX (KaTeX), formatowanie tekstu (TipTap), diagramy (Mermaid), mapy myśli (Markmap), udostępnianie wątków, subagenci, skróty klawiszowe.

## Przykłady kodu

### 1. Inkrementalna materializacja bloków z eventów

Delta: zamiast przebudowywać całą listę bloków na każde zdarzenie SSE, bloki są mutowane in-place. `toolIndexById` pozwala O(1) resolwację tool_result → istniejący ToolBlock.

```ts
// materialize.ts
export const applyEvent = (blocks: Block[], event: StreamEvent, toolIndexById: Map<string, number>): void => {
  switch (event.type) {
    case 'text_delta': {
      const lastBlock = blocks[blocks.length - 1]
      if (lastBlock?.type === 'text') {
        lastBlock.content += event.textDelta
        updateTextRenderState(lastBlock)
        break
      }
      blocks.push(createTextBlock(event.id, event.at, event.textDelta))
      break
    }
    case 'tool_call': {
      toolIndexById.set(event.toolCallId, blocks.length)
      blocks.push({ type: 'tool_interaction', status: 'running', ... })
      break
    }
    case 'tool_result': {
      const existingBlock = blocks[toolIndexById.get(event.toolCallId)!]
      existingBlock.status = event.ok ? 'complete' : 'error'
      break
    }
  }
}
```

Czego uczy: event sourcing po stronie klienta — zdarzenia z serwera materializują się w bloki UI przez czystą redukcję z indeksem, bez frameworków stanu.

### 2. Inkrementalny streaming Markdown

Delta: treść nie jest re-renderowana w całości. `committedSegments` + `liveTail` = tylko ogon jest parsowany na deltę. Bloki promowane z liveTail do committedSegments są zamrażane.

```ts
// streaming-markdown.ts
export const syncIncrementalMarkdownView = (current, { content, streaming }) => {
  const delta = content.slice(current.processedContent.length)
  const candidateTail = current.liveTail + delta
  const blocks = parseMarkdownIntoBlocks(candidateTail)
  const { finalizedSources, liveTail } = splitBlocks(blocks, streaming)

  return {
    committedSegments: current.committedSegments.concat(toSegments(blockId, current.nextSegmentIndex, finalizedSources)),
    liveTail,
    processedContent: content,
    nextSegmentIndex: current.nextSegmentIndex + finalizedSources.length,
  }
}
```

Czego uczy: podział na frozen/live segmenty — fundament wydajnego renderingu streaming Markdown w przeglądarce.

### 3. SSE consumer z batchowaniem rAF

Delta: SSE events buforowane i flushowane synchronizowane z `requestAnimationFrame`, nie na każdy event z sieci.

```ts
// chat-store.svelte.ts
const pending: StreamEvent[] = []
let rafScheduled = false

const flushBatch = () => {
  rafScheduled = false
  const batch = pending.splice(0)
  for (const event of batch) ingestEvent(event)
}

await consumeSse(response, (event) => {
  pending.push(event)
  if (!rafScheduled) {
    rafScheduled = true
    requestAnimationFrame(flushBatch)
  }
})
```

Czego uczy: buforowanie zdarzeń sieciowych + flush na ramce animacji = stabilny 60fps niezależnie od tempa streaming.

### 4. Agent głosowy z trybami fallback

Delta: jedna baza kodu, trzy ścieżki (Gemini Realtime / OpenAI+ElevenLabs / OpenAI). Mode resolve z env → wybór komponentów STT/LLM/TTS. MCP narzędzia konwertowane dynamicznie.

```js
// agent.js
const createSessionConfig = (vad) => {
  const mode = resolveVoiceMode()
  if (mode.id === "gemini") {
    return { sessionOptions: { vad, llm: new google.beta.realtime.RealtimeModel({...}) } }
  }
  if (mode.id === "elevenlabs") {
    return { sessionOptions: { vad, stt: new openai.STT(), llm: new openai.LLM(), tts: new ElevenLabsTTS({...}) } }
  }
  return { sessionOptions: { vad, stt: new openai.STT(), llm: new openai.LLM(), tts: new openai.TTS() } }
}
```

Czego uczy: strategia wyboru voice stack na podstawie dostępnych kluczy API — graceful degradation z Realtime (single model) przez hybrydowe (STT+LLM+TTS) do podstawowego.

### 5. ElevenLabs TTS streaming przez WebSocket + ffmpeg

Delta: custom adapter TTS dla Livekit — WebSocket do ElevenLabs zwraca MP3 base64, przepuszczany przez ffmpeg do PCM (s16le), feedowany do `AudioByteStream` Livekit.

```js
// elevenlabs_tts.js — ElevenLabsSynthesizeStream.run()
const socket = new WebSocket(`wss://api.elevenlabs.io/v1/text-to-speech/${voiceId}/stream-input?...`)
const ffmpeg = spawn(ffmpegPath, ["-i", "pipe:0", "-f", "s16le", "-ar", "24000", "-ac", "1", "pipe:1"])

socket.on("message", (raw) => {
  const d = JSON.parse(raw.toString())
  if (d.audio) ffmpeg.stdin.write(Buffer.from(d.audio, "base64"))
})

ffmpeg.stdout.on("data", (pcmChunk) => {
  for (const frame of byteStream.write(pcmChunk.buffer)) { /* queue frames */ })
})
```

Czego uczy: wzorzec integracji TTS z agent voice frameworkiem — pipe WebSocket → ffmpeg transcode → PCM frames → audio pipeline. `FLUSH_SENTINEL` wymusza wysłanie tekstu oczekującego w buforze.
