# S03E03 — Kontekstowy feedback wspierający skuteczność agentów

## Wyzwalacze autonomii

Agent może zostać aktywowany przez: wiadomości (od człowieka lub innych agentów), hooki wewnętrzne, webhooki zewnętrzne, cron (harmonogram), heartbeat (regularna kontrola stanu). Kluczowa różnica względem klasycznego event-driven programming: system agentowy może mieć **jeden punkt wejścia**, który dynamicznie interpretuje zadanie w języku naturalnym i dobiera akcję do bieżącego stanu otoczenia.

## Proaktywność i nieskończona sesja

Sesja użytkownika może być utrzymywana przez kompresję kontekstu (np. Observational Memory). Heartbeat cyklicznie sprawdza listę zadań (np. `tasks.md`) — jeśli nic nie wymaga akcji, agent się nie udźwięknia. Jeśli tak, proaktywnie wznawia sesję. Decyzja zależy od metadanych: godziny, lokalizacji, pogody, statusu komputera.

Zasada: **niewymagane zdarzenia nie powinny zanieczyszczać kontekstu**. Sesje są domyślnie niezależne — współdzielenie ich jest decyzją architektoniczną, nie domyślnością.

## Enrichment — łączenie rozproszonych danych

Agent działa jako węzeł łączący szczątkowe informacje z wielu źródeł (kontakty, miejsca, kalendarz, mapa, pogoda). Oryginalna wiadomość użytkownika jest **wzbogacana** o dane z narzędzi — adresy, maile, trasy, warunki pogodowe — zanim trafi do dalszego przetwarzania. Wartość biznesowa leży w standaryzacji procesów (np. sprzedażowych), nie w pojedynczych interakcjach.

## Feedback i nauka między sesjami

Agent przeglądarkowy nie odkrywa struktury strony za każdym razem od nowa — korzysta z plików instrukcji w katalogu `instructions/`. Kiedy popełnia błąd, system generuje **sugestie** (np. zrób zrzut ekranu, zapisz notatkę). Po wyjściu z awarii agent zapisuje odkrycia (`{site}-discoveries.md`) do ponownego wykorzystania w przyszłych uruchomieniach.

Mechanizm: `FeedbackTracker` rejestruje każdy wynik wywołania narzędzia (sukces/porażka), liczy kolejne porażki, generuje hinty specyficzne dla typu błędu. Po osiągnięciu progu porażek do konwersacji wstrzykiwana jest interwencja (np. sugestia screenshotu). Po odzyskaniu — sugestia zapisu odkryć.

## Hooki cyklu życia

Hooki to funkcje wywoływane na etapach pętli agenta: `onStart`, `onStepStart/Finish`, `onToolCallStart/Finish`, `onFinish`. W AI SDK to podstawowy zestaw — w projektach produkcyjnych hooki obejmują też strumieniowanie, kompresję sesji, aktywność użytkownika.

Trzy role hooków:
- **Przechwycenie** — modyfikacja danych wejściowych/wyjściowych narzędzia przed przekazaniem dalej (np. zapis ścieżki pliku audio)
- **Śledzenie stanu** — flagi fazowe (`listen_done`, `feedback_done`, `session_saved`) aktualizowane po każdym wywołaniu narzędzia; automatyczny reset po ukończeniu fazy
- **Strażnik procesu** — `beforeFinish` sprawdza czy wszystkie wymagane kroki zostały wykonane; jeśli nie, wstrzykuje komunikat zmuszający agenta do dokończenia (z wyjątkiem: błędy narzędzi lub przekroczenie limitu kroków)

## Narzędzia wywołujące dodatkowe API

Narzędzia `listen` i `feedback` wewnętrznie uruchamiają osobne zapytania do modeli specjalizowanych (ASR, TTS). Agent orkiestruje **kiedy** je wywołać, ale same narzędzia realizują własne pipelines. W MCP odpowiada za to mechanizm **Sampling** — odwrócona komunikacja, gdzie serwer MCP prosi klienta o wywołanie API.

## Projektowanie defensywne vs ofensywne

Zamiast tylko planować punkty awarii (brakujące dane, konflikty nazw, błędy integracji), projektuj **co dodatkowego można zrobić** — np. interfejs głosowy, akcje deterministyczne tam gdzie LLM nie jest potrzebny. Nie każdy agent musi być uniwersalny — wysoka specjalizacja jest akceptowalna.

Uwzględnij też jakość inputu użytkownika — problemy z agentami często leżą po stronie człowieka. Onboarding i komunikacja produktu to element architektury na równi z kodem.

## Uzupełnienia

### Utrzymanie sesji przeglądarki (browser agent)

Agent może poruszać się po stronach wymagających logowania. Logika: osobny skrypt `npm run login` otwiera przeglądarkę z GUI, użytkownik loguje się ręcznie, po zamknięciu przeglądarki sesja (cookies/localStorage) jest zachowana na dysku. Agent uruchamiany później (`npm run start`) dziedziczy tę sesję i działa w imieniu zalogowanego użytkownika. Techniką jest Playwright z `storageState` — stan autoryzacji jest zapisywany do pliku JSON i wczytywany przy każdym uruchomieniu agenta.

### Auto-naprawa argumentów JSON (`jsonrepair`)

LLM regularnie generuje niepoprawny JSON w argumentach narzędzi (trailing commas, brakujące cudzysłowy, markdown fences). Browser agent implementuje dwuetapowe parsowanie: najpierw `JSON.parse`, a przy porażce — `jsonrepair(raw)` jako fallback. Jeśli naprawa się powiedzie, do wyniku narzędzia doklejany jest hint: _"JSON was auto-repaired before executing. Keep arguments strict JSON next time."_. Wzorzec ten zapobiega zbędnym błędom i skraca pętlę odzyskiwania.

### Obsługa przepełnienia kontekstu przez treść strony

Gdy narzędzie `navigate` pobiera treść strony przekraczającą limit okna kontekstowego, agent zapisuje pełny tekst do pliku (`pages/*.txt`) i strukturę DOM do `pages/*.struct.txt`. Zamiast ładować całą treść do konwersacji, agent przeszukuje pliki narzędziami `fs_search` (grep) lub `fs_read` z `offset/limit`. Alternatywą jest uruchomienie subagenta dedykowanego do ekstrakcji danych ze strony. Zasada: **nigdy nie ładuj pełnego tekstu strony do konwersacji**.

### Bezpieczeństwo: deterministyczne interfejsy dla wrażliwych operacji

Agent kalendarzowy tworzy wydarzenia łącząc dane z kontaktów i miejsc — tu ryzyko pomylenia adresów lub wycieku poufnych informacji w opisie. Lekcja rekomenduje: dla operacji niosących ryzyko bezpieczeństwa informacji, wyodrębnić deterministyczny interfejs (np. programistyczne zarządzanie kontaktami), zamiast pozostawiać to agentowi. Agent zarządza logiką, ale wrażliwe dane przechodzą przez kod.

### Narzędzie `feedback` jako wewnętrzny orkiestrator wielu wywołań API

Narzędzie `feedback` w agencie językowym nie jest prostym proxy — wewnętrznie realizuje pipeline: (1) generuje tekst feedbacku przez Gemini, (2) wywołuje narzędzie `speak` do generowania audio, (3) agreguje wyniki obu wywołań w jedną odpowiedź. Agent widzi tylko końcowy wynik, nie detaly orkiestracji. Wzorzec: **jedno narzędzie może enkapsułować cały sub-pipeline** modeli i API, ukrywając złożoność przed pętlą agenta.

### `previousResponseId` — ciągłość konwersacji na poziomie API

Agent językowy korzysta z Gemini Interactions API, które oferuje `previous_interaction_id` — identyfikator poprzedniej odpowiedzi przesyłany w kolejnym zapytaniu. Dzięki temu API samo zarządza historią konwersacji po stronie serwera, bez konieczności budowania tablicy wiadomości w kodzie. To alternatywa dla ręcznego akumulowania `InputItem[]` — API przechowuje kontekst, a hook `beforeFinish` wstrzykuje komunikat jako nowe zapytanie tekstowe.

### Konkretny mechanizm heartbeat: `tasks.md`

Proaktywny agent okresowo otrzymuje dodatkową wiadomość z prośbą o przeczytanie pliku `tasks.md` i wykonanie zawartych w nim poleceń. Plik zawiera najważniejsze aktywności wymagające kontekstu z głównego wątku sesji. Jeśli żadne zadanie nie wymaga akcji — agent pomija dalsze wykonanie i użytkownik nic nie zauważa. Treść `tasks.md` może być ustalana ręcznie lub zarządzana przez agenta w imieniu użytkownika.

### Zmiana krajobrazu automatyzacji webowej

Lekcja zaznacza przesunięcie w traktowaniu agentów w sieci: Cloudflare wprowadził [Markdown for Agents](https://blog.cloudflare.com/markdown-for-agents/) (strukturyzowana treść dla agentów zamiast blokad), a Google Chrome proponuje [WebMCP](https://developer.chrome.com/blog/webmcp-epp) (natywny protokół komunikacji agentów ze stronami). To zmniejsza potrzebę scrapowania i nawigacji browserowej, ale nie eliminuje jej — platformy bez API (np. Goodreads po zamknięciu rejestracji developerów) nadal wymagają agenta z przeglądarką.

## Przykłady kodu — progresja wiedzy

### 1. Calendar — wstrzykiwanie metadanych środowiska

Delta: statyczny prompt + dynamiczny blok `<metadata>` doklejany do wiadomości użytkownika. Agent nie odkrywa otoczenia sam — otrzymuje je programistycznie.

```ts
export const buildMetadata = (): string => {
  const weather = getWeatherAt(date, hour);
  const lines = [
    `<metadata>`,
    `Current time: ${dayName}, ${date} ${timeStr} CET`,
    `Location ID: ${state.userLocation.placeId}`,
    `Location: ${state.userLocation.name}`,
    `Weather: ${weather.description} — ${weather.tempC}°C`,
    `</metadata>`,
  ];
  return lines.join('\n');
};
```

Dwie fazy agenta (add-events, notifications) z **osobnymi zestawami narzędzi** i promptami — jedna sesja nie musi widzieć narzędzi drugiej.

### 2. Browser — feedback tracker i interwencje

Delta: stanful tracking wywołań narzędzi + dynamiczne wstrzykiwanie sugestii do konwersacji na podstawie wzorców porażek.

```ts
const generateHints = (tool: string, outcome: ToolOutcome, error?: string): string[] => {
  if (tool === 'click' && error?.includes('timeout'))
    return ['The element may not be visible. Try scrolling first.'];
  if (recentOf(tool, 3).filter(e => e.outcome === 'fail').length >= 3)
    return ['Multiple failures detected. Consider changing strategy.'];
  return [];
};
```

Interwencje wstrzykiwane jako dodatkowe wiadomości `role: 'user'` po wynikach narzędzi — agent traktuje je jako kontekstowe podpowiedzi, nie jako błędy systemu. Pliki instrukcji w `instructions/` eliminują potrzebę odkrywania struktury strony za każdym razem.

### 3. Language — hooki jako strażnik procesu

Delta: pełny cykl hooków zarządzających fazami analizy audio z wymuszaniem ukończenia wszystkich kroków przed zakończeniem.

```ts
beforeFinish: (finalText) => {
  const missing: string[] = [];
  if (!state.phase_flags.listen_done) missing.push('listen to audio');
  if (!state.phase_flags.feedback_done) missing.push('generate feedback');
  if (!state.phase_flags.session_saved) missing.push('save session');
  if (!state.profile_updated) missing.push('update profile.json');
  if (missing.length === 0) return { allow: true, missing: [] };
  return {
    allow: false,
    missing,
    inject_message: 'You must complete these before finishing:\n'
      + missing.map(m => `- ${m}`).join('\n'),
  };
},
```

Kluczowy wzorzec: `afterToolResult` śledzi flagi faz i automatycznie resetuje stan po ukończeniu (`tryCompletePhase`), pozwalając na wiele tur analizy w jednej sesji. Narzędzia `listen`/`feedback` wywołują dodatkowe API wewnętrznie — agent nie widzi tego detalu, dostaje gotowy wynik.
