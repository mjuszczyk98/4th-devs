# S02E04 — Organizowanie kontekstu dla wielu wątków

## Architektury systemów wieloagentowych

| Architektura | Mechanizm koordynacji | Kiedy stosować |
|---|---|---|
| **Pipeline** | Sekwencyjne przekazywanie rezultatu dalej, bez powrotu | Przekształcanie danych w stałych krokach |
| **Blackboard** | Współdzielony stan, niezależni agenci czytają/zapisują | Równoległe gromadzenie danych z wielu źródeł |
| **Orchestrator** | Agent-koordynator zleca, odbiera, decyduje | Złożone wieloetapowe zadania wymagające koordynacji |
| **Tree** | Orchestrator + managerowie pośredni | Duża skala, hierarchiczne podziały odpowiedzialności |
| **Mesh** | Komunikacja adresowana między agentami | Agenci ze stałymi rolami komunikujący się bezpośrednio |
| **Swarm** | Komunikacja rozproszona, selekcja/agregacja wyników | Eksploracyjne, конкурентные zadania |

W praktyce — pierwsze cztery, często łączone. Mesh/Swarm rzadziej produkcyjnie z LLM (trudność debugowania).

## Narzędzia komunikacji między agentami

- **delegate** — zlecenie zadania innemu agentowi. Otwiera nowy wątek z własnym system promptem i zestawem narzędzi. Wynik sub-agenta = wynik wywołania narzędzia.
- **message** — dwukierunkowa komunikacja. Sub-agent wstrzymuje pętlę do czasu dostarczenia odpowiedzi (scenariusz dla generatorów JS).

Dwukierunkowość pozwala sub-agentowi poprosić o doprecyzowanie → nadrzędny agent kontaktuje użytkownika → przekazuje odpowiedź → praca sub-agenta zostaje wznowiona.

## Architektury zdarzeniowe

Zamiast bezpośrednich wywołań — zdarzenia (np. `user.message`, `ticket.classified`, `tracking.found`). Agenci nasłuchują na zdarzenia i reagują. Pozwala na:
- Wielu agentów reagujących na to samo zdarzenie niezależnie
- Zależności: agent może wysłać zdarzenie żądania danych i zawiesić się do ich nadejścia
- Naturalną integrację człowieka w proces (weryfikacja, decyzje)

## Globalny kontekst — konflikty współbieżne

Już jedna instancja agenta uruchomiona wielokrotnie = system wieloagentowy. Problem: agent A czyta plik → agent B zapisuje plik → agent A zapisuje plik = utrata danych B.

| Strategia | Mechanizm |
|---|---|
| **Wykrywanie** | Checksumy / hash'e linii między odczytem a zapisem |
| **Unikanie** | Izolacja zasobów, uprawnienia read-only, przynależność do sesji |
| **Agent zarządzający** | Dedykowany agent z pełnym wglądem w historię + kontakt z człowiekiem |
| **Historia zmian** | Append-only zamiast overwrite — rzadziej konflikty |
| **Manualne** | Eskalacja do człowieka gdy auto-rozwiązanie nie wystarcza |

Zewnętrzny kontekst powinien być **odseparowany od logiki agentów** — dokumenty opisują wiedzę, nie zachowania. Dokumenty dostępne także dla ludzi = kolaboracja, nie tylko automatyzacja.

## Pułapki współdzielonego kontekstu

1. **Sesja vs pamięć** — kto decyduje co trafia do pamięci długoterminowej? Agenci w tle potrzebują zgeneralizowanych zasad, balansowanych z deterministycznymi ograniczeniami w kodzie.
2. **Degradacja komunikacji** — każdy przekaz między agentami = potencjalna utrata/zniekształcenie. Instrukcje narzędzi delegate/message muszą być precyzyjne. System musi zakładać niekompletne dane.
3. **Interpretacja** — agent z kompletnymi danymi może je zinterpretować po swojemu. Ryzyko rośnie z otwartością zadania.
4. **Kontekst informacji** — notatka wyrwana z konwersacji traci kontekst (np. „Anna" z rozmowy o innej Annie).
5. **Duplikowanie** — nieuniknione, ale wykrywalne przez mniejsze modele skanujące zmiany.
6. **Metadane** — źródło, data, tags wykorzystywane nie tylko w UI, ale podczas komunikacji między agentami.

**Zasada:** projektować system tak prosty jak to możliwe i utrzymywać w tej formie najdłużej.

## Agent zarządzający — rola i ograniczenia

Minimalna liczba narzędzi (delegate/message + recall/search_memory), ale szeroki dostęp do informacji. Odpowiada za:
- Rozbijanie zadań, kształtowanie planu, monitorowanie postępów
- Wiedzę o systemie: role agentów, dostęp do pamięci (min. read-only), dane użytkownika
- Weryfikację rezultatów (samodzielnie lub przez delegację)
- Decyzyjność: kiedy sam podejmuje decyzję, kiedy eskaluje do człowieka
- Transport wiedzy między agentami

Obecnie: ograniczać rolę, ale z umiarem — każdy dodatkowy etap komunikacji komplikuje zarządzanie kontekstem.

## Kiedy agenci, a kiedy kod

| Kryterium | Agenci uzasadnieni |
|---|---|
| Zadania otwarte | Reagowanie na dane z otoczenia w trakcie realizacji |
| Dynamiczne dane | Input/output nie z góry określony, transformacje wykraczające poza kod |
| Dynamiczne zależności | Zależności na poziomie semantyki, nie struktury danych |
| Iterowanie | Kryteria w języku naturalnym, kroki niemożliwe do zdefiniowania z góry |
| Elastyczna architektura | Nowe obszary, wysoka elastyczność logiki |
| Dopasowanie wyniku | Personalizacja wykraczająca poza szablony |

Gdzie wymagania wykluczają LLM (koszt zero, ms response, pełna przewidywalność) — tylko kod.

## Przykłady kodu — Daily Ops multi-agent

### Progresja przykładu

`02_04_ops` — uproszczony system Daily Ops: orchestrator czyta workflow z pliku, deleguje do 4 agentów specjalistycznych, czyta historię/cele/preferencje, syntonizuje wynik do pliku markdown.

### Szablony agentów jako markdown frontmatter

```
---
name: mail
model: openai:gpt-4.1-mini
tools:
  - get_mail
---
[system prompt]
```

- Jeden plik `.agent.md` = pełna definicja agenta (model, narzędzia, instrukcje)
- `gray-matter` parsuje frontmatter, treść = system prompt
- Podmioty — osobne szablony, osobne narzędzia, osobne modele

### Delegowanie = rekurencyjne runAgent()

```ts
if (name === 'delegate') {
  result = await runAgent(agent, delegatedTask, depth + 1)
}
```

- `delegate` jako narzędzie — wywołanie `runAgent()` z inkrementowanym `depth`
- Ograniczenie rekurencji: `MAX_DEPTH = 3`, `MAX_TURNS = 15`
- Wynik sub-agenta wraca jako `content` do narzędzia w pętli nadrzędnego agenta

### Workflow jako plik markdown

Orchestrator dostaje zadanie: „przeczytaj `workflows/daily-ops.md` i wykonaj". Workflow definiuje kroki sekwencyjnie — agent interpretuje go jako instrukcję. To oddziela logikę procesu od kodu agenta.

### Bezpieczeństwo ścieżek

```ts
function isPathSafe(path: string): boolean {
  const fullPath = resolve(join(WORKSPACE, path))
  const rel = relative(resolve(WORKSPACE), fullPath)
  return !rel.startsWith('..') && rel !== '..'
}
```

Wszystkie operacje plikowe scoped do katalogu `workspace/`. Path traversal zablokowany.

### Czego uczy ten przykład

1. **Workflow w pliku, nie w kodzie** — zmiana procesu bez redeploymantu, agent interpretuje markdown jako instrukcję.
2. **Rekurencyjna delegacja z guardami** — depth + turns limitują złożoność i koszt.
3. **Podział odpowiedzialności przez narzędzia** — sub-agent widzi tylko swoje narzędzie (np. `get_mail`), orchestrator widzi `delegate` + `read_file` + `write_file`.
4. **Synteza = rola orchestratora** — nie wystarczy zebrać dane, trzeba je skorelować z historią, celami i preferencjami.
