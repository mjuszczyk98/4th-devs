# S02E02 — Zewnętrzny kontekst narzędzi i dokumentów

## Bezpieczeństwo zewnętrznego kontekstu

- Zewnętrzne dane dostarczane przez narzędzia mogą zmienić zachowanie agenta — równoważne brakowi walidacji formularza renderowanego jako HTML (XSS).
- Filtrowanie prompt injection przez LLM nie jest gwarancją bezpieczeństwa — atak może przejść przez filtr.
- Skuteczna mitigacja: ograniczenie zestawu narzędzi (agent nie może wysłać maila, jeśli nie ma takiego narzędzia) + jawna zgoda użytkownika na akcje.
- Walidacja plików: rozmiar, format, mime-type, źródło. Moderacja treści. Linki tymczasowe, trudne do odgadnięcia.
- Zabezpieczenia realizowane programistycznie — agent otrzymuje tylko błędy i wskazówki.

## Indeksowanie dokumentów

Indeksowanie to proces przekształcenia surowych plików w przeszukiwalną strukturę. Obejmuje: chunking, metadane, transformacje (podsumowania, transkrypcje), grafy powiązań, synchronizację.

### Strategie chunkingu

| Strategia | Mechanizm | Metadane | Koszt |
|-----------|-----------|----------|-------|
| Znaki | Stałe okno + overlap | pozycja, długość | 0 |
| Separatory | Rekurencyjny podział: nagłówki → akapity → zdania → słowa | sekcja, źródło | 0 |
| Kontekst | Separatory + LLM generuje 1-2 zdania kontekstu per chunk | jak wyżej + kontekst | tokeny |
| Tematy | LLM identyfikuje granice tematów, dzieli od podstaw | temat, sekcja, źródło | tokeny |

- Długość chunka: 200–500 słów / 500–4000 tokenów.
- Separator-based to kompromis — zachowuje strukturę dokumentu bez kosztu LLM. Context-enriched (inspiracja Anthropic) dodaje prefiks sytuujący chunk w dokumencie. Topic-based daje najlepszą semantykę, ale wymaga pełnego przetworzenia przez model.

## Embedding i wyszukiwanie semantyczne

- Embedding = tablica liczb (wektor) opisująca znaczenie tekstu. Model `text-embedding-3-small` generuje 1536 wymiarów.
- Wymiar wektora w bazie **musi** zgadzać się z liczbą wymiarów modelu.
- Cosine similarity mierzy podobieństwo znaczeniowe — "Kobieta" jest bliżej "Królowej" niż "Króla".
- Model embeddingu nie opisze poprawnie terminów spoza swoich danych treningowych.
- Embedding działa wielojęzycznie — polskie zapytanie znajdzie angielski dokument semantycznie, nawet gdy FTS zawiedzie na dopasowaniu słów.

## Hybrydowe wyszukiwanie

- **FTS5 (BM25)** — dopasowanie leksykalne, szybkie, deterministyczne.
- **Wektorowe (cosine)** — dopasowanie znaczeniowe, wymaga embeddingu.
- **RRF (Reciprocal Rank Fusion)** — scalanie dwóch list wyników na podstawie pozycji w rankingu, nie wyników surowych:

```
score = Σ  1 / (k + rank)   gdzie k = 60
```

Dokument wysoko w jednej liście + nisko w drugiej może awansować dzięki sumie rang.

- Agent generuje **dwa zapytania**: lista słów kluczowych (FTS) + zdanie w języku naturalnym (semantic). Jedno narzędzie `search` przyjmuje oba parametry.
- Degradacja elegancka: gdy embedding API niedostępne, system wraca do FTS-only.

## Architektura — od prostego do złożonego

1. **System plików + grep/ripgrep** — brak indeksowania, agent sam pisze zapytania. Dobre dla małych baz markdown.
2. **SQLite + FTS5** — full-text bez wektorów. Jedna baza, niska złożoność.
3. **SQLite + FTS5 + sqlite-vec** — hybryda w jednej bazie. Optymalny stos na małą/średnią skalę.
4. **Elasticsearch / Qdrant / Algolia** — dedykowane silniki, wymagają synchronizacji z bazą danych.

Kluczowe pytanie: nie "które podejście jest najlepsze?", lecz "które jest najlepsze dla tego problemu?"

## Prezentacja treści w kontekście

- Zewnętrzne treści trafiają do kontekstu jako **wyniki narzędzi** — techniki ze S01E02/S01E03 obowiązują.
- Dwa narzędzia nawigacji: `search` (wyszukaj) + `read` (przeczytaj). Agent nie widzi oryginalnych plików bezpośrednio.
- Obraz jako format kontekstu: DeepSeek-OCR pokazuje 9-10x kompresję przy 96% precyzji. Sensowne nawet dla tekstu.
- Obrazy w markdown (`![](link)`) wymagają wydobycia przez regex i osobnego przesłania do modelu.
- Każdy chunk w wynikach powinien zawierać: źródło (plik), sekcję, pozycję — dla modelu (uzasadnienie) i UI (cytaty, odnośniki).

## Wyzwania skuteczności RAG

- **Konflikt wiedzy** — jeśli informacja jest w danych treningowych, agent może pominąć dokumenty.
- **Niekompletność** — dotarcie do 100% informacji przez wyszukiwanie jest bardzo trudne. Model nie zauważy braków.
- **Świadomość zasobów** — agent musi wiedzieć, co jest dostępne, by celowo przeszukiwać.
- **Kontekst użytkownika** — "moje projekty" wymaga połączenia z tożsamością. Semantyka częściowo pomaga, Graph RAG lepiej.
- **Rozbududzony kontekst** — negatywnie wpływa na instruction following. Agent "zapomina" instrukcje systemowe.
- **Multimodalność** — obrazy, audio, wideo trudniejsze w indeksowaniu i wyszukiwaniu niż tekst.

Mitigacja: zawężenie zakresu źródeł, dekompozycja (subagenci), UI z cytatami i odnośnikami.

## Przykłady kodu — progresja

### 1. Chunking: cztery strategie na tym samym dokumencie

Każda strategia to funkcja przyjmująca tekst, zwracająca `[{ content, metadata }]`.

**Characters** — najprostsze, zero kontekstu:

```js
const chunkByCharacters = (text, size = 1000, overlap = 200) => {
  const chunks = [];
  let start = 0;
  while (start < text.length) {
    chunks.push(text.slice(start, start + size));
    start += size - overlap;
  }
  return chunks.map((content, i) => ({ content, metadata: { strategy: "characters", index: i } }));
};
```

Delta **Separators**: rekurencyjny podział przez hierarchię separatorów `["\n## ", "\n### ", "\n\n", "\n", ". ", " "]` + overlap dobierany na granicy słowa/linii + metadane sekcji z indeksu nagłówków.

Delta **Context**: separatory + LLM generuje 1-2 zdania kontekstu per chunk — enrichChunk woła `chat()` z chunk.content i prosi o kontekst.

Delta **Topics**: LLM dostaje cały dokument, zwraca `[{ topic, content }]` — chunki generowane od podstaw na podstawie struktury tematu. Wymaga parsowania JSON z outputu modelu.

### 2. Embedding: REPL z macierzą podobieństwa

Pojedynczy endpoint, zawsze zwraca wektor o tej samej długości (1536), inne wartości dla różnych tekstów:

```js
const cosineSimilarity = (a, b) => {
  let dot = 0, normA = 0, normB = 0;
  for (let i = 0; i < a.length; i++) {
    dot += a[i] * b[i];
    normA += a[i] * a[i];
    normB += b[i] * b[i];
  }
  return dot / (Math.sqrt(normA) * Math.sqrt(normB));
};
```

Uczy: identyczne teksty → identyczny wektor. Podobne znaczeniowo → wysoki cosine score. embedding API wspiera batch (tablica inputów).

### 3. Hybrid RAG: pełny pipeline w SQLite

**Delta względem poprzednich**: łączy chunking + embedding + FTS5 + sqlite-vec + agent z narzędziem search.

Schemat bazy — cztery tablice powiązane:

```
documents (id, source, content, hash)
  → chunks (id, document_id, content, chunk_index, section, chars)
    → chunks_fts (FTS5 virtual table, synced via triggers)
    → chunks_vec (vec0 virtual table, embedding float[1536])
```

Kluczowe mechanizmy:

- **Reindeksowanie**: hash SHA256 pliku — pomija niezmienione, usuwa nieistniejące.
- **FTS triggers**: `AFTER INSERT/DELETE/UPDATE` na chunks utrzymują chunks_fts w synchronizacji.
- **Embedding batch**: 20 chunków na zapytanie API.
- **Float32 → Buffer**: sqlite-vec wymaga `Buffer.from(new Float32Array(arr).buffer)`.

Narzędzie search agenta — dwa parametry wejściowe:

```js
{ keywords: "autoregresja model", semantic: "Czym jest autoregresja w kontekście modeli językowych?" }
```

RRF w `search.js`: każdy chunk dostaje score = suma `1/(60+rank)` z obu list, top N po posortowaniu. Degradacja: gdy embed() rzuci błąd → wyniki FTS-only.
