# S01E02 — Techniki łączenia modelu z narzędziami

## Function Calling — mechanizm

- LLM nie ma dostępu do otoczenia. Function Calling to most: model generuje JSON z nazwą narzędzia i argumentami, a kod aplikacji uruchamia właściwą funkcję i zwraca wynik do kontekstu.
- Jedna interakcja to minimum **dwa zapytania do LLM**: pierwsze — model decyduje o wywołaniu narzędzia, drugie — model generuje odpowiedź na podstawie wyniku.
- Pętla agenta: zapytanie → tool call → wynik → kolejne zapytanie, aż model zwróci finalną odpowiedź lub wyczerpie limit kroków.
- Schematy narzędzi są dołączane do **każdego** zapytania — zużywają kontekst nawet nieużywane.

## Projektowanie narzędzi

- **Nie mapuj API 1:1.** API jest dla programistów z dokumentacją; narzędzia są dla modelu bez kontekstu. Filtruj, łącz, upraszczaj.
- Ogranicz liczbę narzędzi do 10-15 per agent. Powyżej — rozpraszają model i zużywają kontekst.
- Połączenie wielu zasobów API w jedno narzędzie (np. `workspace_metadata`) redukuje liczbę kroków agenta.
- Pomijaj właściwości nieistotne dla modelu: wewnętrzne ID, hashe, flagi implementacyjne.

### Nazwy, opisy, schematy

- Nazwa unikatowa, niskie ryzyko kolizji. `send` < `send_email`.
- Opis = wysoki signal-to-noise. Zwięzły, zwiększający szansę poprawnego wyboru narzędzia. Testuj na 10-30 przykładowych zapytaniach.
- Dziel właściwości na: **obowiązkowe dla modelu**, **ustawiane programistycznie** (np. user_id), **niedostępne dla modelu**.
- Zezwalaj na wiele form tej samej wartości (np. identyfikator **lub** nazwa etykiety).

### Domyślne wartości

- Ustalaj sensowne defaulty, ale informuj model w opisie o ich istnieniu.
- Autoryzowany użytkownik → domyślne przypisanie do niego, bez wymogu podania ID.
- Defaulty filtrowania/sortowania = minimum informacji, z opcją rozszerzenia.

### Walidacja i błędy

- Błąd "coś poszło nie tak" = ślepy zaułek dla agenta. Zawsze konkretna informacja + wskazówka naprawcza.
- Komunikaty o błędach powinny sugerować następny krok: `"team_id jest nieprawidłowy. Pobierz go przez 'workspace_metadata'."`
- Reaguj na literówki/warianty: `"czy chodziło o 'completed'?"`.
- Standard obsługi błędów dla narzędzi LLM musi być **wyższy** niż dla zwykłego API — model może wysłać cokolwiek.

## Natywne vs własne narzędzia

- Natywne (web search, code execution) — wygodne, ale ograniczona konfiguracja. Logika po stronie providera.
- Własne — pełna kontrola. Można łączyć oba typy równocześnie w jednym zapytaniu.

## API vs MCP vs CLI

| Aspekt | API | CLI | Function Calling / MCP |
|--------|-----|-----|----------------------|
| Kontrola | praktycznie brak | ograniczona do udostępnionego interfejsu | pełna |
| Dokumentacja | model musi eksplorować | `--help` dostępne lokalnie | wbudowana w schematy |
| Skalowalność | dobra | trudna poza lokalnym urządzeniem | dobra |
| Elastyczność | pełna (curl, skrypty) | pełna (terminal) | ograniczona do zdefiniowanych akcji |

Proxy (warstwa między API a LLM) to sposób na adaptację istniejącego API bez jego modyfikacji. MCP to jeden z wzorców takiego proxy.

## Augmented Function Calling

- Dodatkowe instrukcje kontekstowe wpływające na zachowanie modelu przy wywołaniu narzędzia, np. stały styl generowanych obrazów.
- Trzy tryby dołączania: **statyczny** (użytkownik), **dynamiczny** (model decyduje na podstawie nazwy/opisu), **hybrydowy**.
- Może obejmować sekwencje akcji, aktywowanie/dezaktywowanie skilli, ich tworzenie.

## Workflow vs Agent

| | Workflow | Agent |
|---|---------|-------|
| Struktura | sztywna, z góry ustalone etapy | dynamiczna, model decyduje o krokach |
| Pętla LLM | sekwencyjna, ograniczona | pętla z wieloma turami narzędzi |
| Przewidywalność | wysoka | niższa, ale elastyczniejsza |

- Jeśli proces jest ustrukturyzowany i rzadko się zmienia → workflow.
- Jeśli potrzebujesz reakcji na zmiany i otwartych problemów → agent, ale najpierw spróbuj zamknąć problemy.
- 100% skuteczności wymaga nadzoru człowieka — LLM sam go nie zagwarantuje.

## Reasoning w agentach

- Wbudowany reasoning modeli (LRM) poprawia skuteczność, ale prostym zadaniom może szkodzić.
- Dodatkowe techniki sterowania refleksją:
  - **Planowanie** — lista zadań jako narzędzie; aktualizacja listy = przypomnienie o wątkach.
  - **Odkrywanie** — model "nie wie, że wie"; prowadź go przez proces eksploracji pamięci/bazy wiedzy.
  - **Przekierowanie** — zarządzanie uwagą na podstawie stanu (np. otwarta przeglądarka → focus na jej obsłudze).
  - **Uśrednianie** — wiele modeli → perspektywy → głosowanie/uśrednienie.

## Transformacja zapytań

- Zapytanie użytkownika nie zawsze bezpośrednio pasuje do zasobów. Transformacja przez synonimy/powiązane zagadnienia zwiększa szansę trafienia.
- Model potrzebuje ogólnej wiedzy o zawartości bazy wiedzy (np. plik `_index.md`) — bez tego nie nawiguje skutecznie.
- Należy rozpoznawać, czy pytanie w ogóle dotyczy bazy wiedzy — zachęcaj agenta do pytań doprecyzowujących.

## Optymalizacja szybkości i skuteczności

- **Prompt cache = priorytet.** Stabilna instrukcja systemowa + stabilny początek wątku = automatyczny cache (Gemini, OpenAI). Zmiana nawet jednej linijki system promptu niszczy cache.
- Każdy krok agenta = kolejne zapytanie + transfer kontekstu + czas zewnętrznych usług. Minimalizuj kroki.
- Równoległe wywołania narzędzi (`parallel function calling`) dla niezależnych akcji.
- Projektuj narzędzia na operacje batchowe (edycja wielu rekordów naraz).
- Zmieniaj model na mniejszy tam, gdzie to możliwe.
- Ogranicz długość kontekstu i generowane tokeny. Przekazuj dane między narzędziami przez pliki (nie przez output modelu).

## Zarządzanie kontekstem

- Dynamiczne dane w instrukcji systemowej (data, godzina) niszczą prompt cache. Umieszczaj je ostrożnie.
- Pływające okno kontekstu (usuwanie najstarszych wiadomości) jest gorsze niż utrzymanie pełnego wątku z cache'em.
- Przy zbliżaniu się do limitu: kompresja (auto-compact) + zapis kluczowych informacji do plików dostępnych dla agenta.
- Wyniki narzędzi można zapisywać jako pliki — agent wczytuje tylko istotne fragmenty, nie cały wynik naraz.

## Dynamiczne listy narzędzi

- Dynamiczne dodawanie narzędzi bez psucia prompt cache — obecnie tylko Anthropic.
- Alternatywy: sub-agenci z osobnymi oknami kontekstu, code execution sandbox (narzędzia jako pliki, tylko podstawowe w kontekście — "Progressive Disclosure").
- Agent musi mieć wskazówki o dostępnych zasobach — inaczej "nie wie, że wie".

## Bezpieczeństwo

- **Prompt injection** = problem otwarty, bez skutecznej obrony. Agenci przetwarzający niezaufane dane (email, web) mogą zostać manipulowani.
- Ograniczaj agentów na poziomie środowiskowym: sandbox, whitelist adresów, izolacja kontekstu.
- Akcje nieodwracalne (wysyłka maila) → potwierdzenie przez UI (formularz/przycisk), nie przez wiadomość do modelu.
- Dane wrażliwe (user_id, uprawnienia) → kontrola w kodzie, model nie ma prawa ich ustalać.
- Tryb **dry-run** — agent testuje akcję przed wykonaniem, ocenia poprawność, potem wykonuje z tymi samymi argumentami.

## Przykłady kodu

### Delta 1: Minimalna pętla tool-callingu

`01_02_tools/app.js` — najprostsza realizacja: definicje narzędzi jako JSON Schema + handlery jako zwykłe funkcje + pętla z limitem kroków.

```js
const handlers = {
  get_weather({ location }) { return { temp: -2, conditions: "snow" }; },
  send_email({ to, subject, body }) { return { success: true, status: "sent" }; },
};

while (stepsRemaining > 0) {
  const response = await requestResponse(conversation);
  const toolCalls = getToolCalls(response);
  if (toolCalls.length === 0) return getFinalText(response);
  conversation = await buildNextConversation(conversation, toolCalls, handlers);
}
```

Uczy: separacja definicji (co model widzi) od implementacji (co kod wykonuje). Model nigdy nie widzi handlera.

### Delta 2: Sandboxing + wiele narzędzi + conversation state

`01_02_tool_use/` — dodaje warstwę bezpieczeństwa (`resolveSandboxPath` blokuje path traversal) i pełny executor zarządzający stanem rozmowy.

```js
export const resolveSandboxPath = (relativePath) => {
  const resolved = resolve(sandbox.root, relativePath);
  const rel = relative(sandbox.root, resolved);
  if (rel.startsWith("..") || resolve(rel) === resolved) {
    throw new Error(`Access denied: path "${relativePath}" is outside sandbox`);
  }
  return resolved;
};
```

Uczy: narzędzia muszą być izolowane programistycznie — model może (celowo lub przez halucynację) próbować wyjść poza wyznaczony zakres. Błąd wraca do modelu jako informacja, nie jako crash.

### Delta 3: Kompozycja narzędzi w executorze

`01_02_tool_use/src/executor.js` — conversation to tablica: `[user, ...toolCalls, ...toolResults]` — model widzi pełną historię wywołań.

```js
conversation = [...conversation, ...toolCalls, ...toolResults];
```

Uczy: wyniki narzędzi i same wywołania stają się częścią kontekstu. Każde wywołanie = koszt tokenów. Dlatego minimalizacja kroków i zwięzłe odpowiedzi z narzędzi mają znaczenie.
