# S03E04 — Budowanie narzędzi na podstawie danych testowych

## Narzędzia jako ograniczony podzbiór API

Pełne podłączenie agenta do zewnętrznego API to antywzorzec. Zamiast `gmail_search` twórz `gmail_search_support` — narzędzie z domyślnym zawężeniem, które nie daje agentowi przestrzeni na dotarcie do danych poza zakresem. Każde narzędzie powinno realizować konkretny przypadek użycia, nie eksponować pełne możliwości API.

## Iteracyjne projektowanie schematów z LLM

Pierwsza propozycja schematu input/output od modelu to punkt startowy (~60-70% pracy), ale zawiera fundamentalne błędy. Typowe patologie:
- Brak mechanizmu stronicowania
- Pola o niejednoznacznej semantyce (np. `date` zamiast `receivedAt`)
- Zwracanie base64 (zabija kontekst modelu)
- Brak metadanych o statusach, etykietach, załącznikach, wątkach

Proces: podaj kilka konkretnych wskazówek → model rozszerza je na cały kontekst integracji → iteruj aż do satysfakcji.

## Envelope `{ data, hint }` — programowe prowadzenie agenta

Każda odpowiedź narzędzia zawiera dwie sekcje:
- **data** — merytoryczny wynik akcji
- **hint** — metadane kierujące zachowaniem agenta: `status`, `reasonCode`, `summary`, `nextActions`, `recovery`, `diagnostics`

```ts
interface ToolHint {
  status: 'success' | 'empty' | 'partial' | 'error';
  reasonCode: 'OK' | 'NO_RESULTS' | 'AUTH_REQUIRED' | 'NOT_FOUND' | ...;
  nextActions: { tool: string; why: string; args: Record<string, unknown>; confidence: number }[];
  recovery: { retryable: boolean; backoffMs: number; maxAttempts: number };
}
```

`nextActions` to konkretne propozycje kolejnych wywołań z argumentami — agent nie musi zgadywać co zrobić dalej. `recovery` opisuje czy i jak ponawiać przy błędach.

## Klasyfikacja błędów z automatem stanów

Błędy API są klasyfikowane na podstawie wzorców w treści komunikatu i statusu HTTP: `AUTH_REQUIRED`, `RATE_LIMITED`, `NOT_FOUND`, `INVALID_ARGUMENT`, `TRANSIENT_FAILURE`. Każda kategoria ma inną strategię recovery (retry z backoffem vs bezwzględny fail).

## Kontrola szczegółowości odpowiedzi

Narzędzia search/read przyjmują parametr `details: boolean`. Domyślnie zwracają compact view (tylko niezbędne pola). Dopiero z `details=true` doczytują recipients, labels, attachments. Zmniejsza to zużycie kontekstu gdy szczegóły nie są potrzebne.

## Polityka wysyłki (send policy)

Narzędzie send weryfikuje odbiorców względem whitelist. Jeśli jakikolwiek odbiorca jest poza nią — zamiast wysłać, tworzy draft z `policy.enforcedDraft: true`. Agent dostaje tę informację w hint i informuje użytkownika.

## Modyfikacje z natychmiastową informacją zwrotną

Po operacji modify odpowiedź zawiera `appliedAction` i `updatedState` (wyprowadzone z listów etykiet: `isRead`, `inInbox`, `inTrash`, `isDraft`). Agent nie musi robić dodatkowego read, żeby potwierdzić efekt.

## Dane testowe: syntetyczne, generowane przez LLM

Na podstawie kodu narzędzi (typy, schematy Zod) LLM generuje przykłady interakcji. Kluczowe kategorie: pojedyncze akcje per narzędzie, scenariusze wieloetapowe, przypadki błędowe. Generowanie jest interaktywne — wybierasz właściwe przykłady i prosisz o kolejne na ich podstawie. Nie ufaj pierwszej iteracji — model generuje płytkie testy bez kategoryzacji.

## Ewaluacja offline z Promptfoo

Struktura ewaluacji per narzędzie: asercje na `required_tools` (czy użył właściwego), `forbidden_tools` (czy nie użył zakazanego), `expected_primary_tool`, `turn_budget` (limit kroków), `search_query_hint` (czy zapytanie zawiera oczekiwany fragment), `llm-rubric` na `finalText`.

Dwa poziomy suitów:
- **tools** — pojedyncze akcje, bez stanu między turami
- **scenarios** — wieloetapowe z zastrzeżonymi narzędziami (np. readonly suite blokuje send/modify)

Provider eval to wrapper wywołujący CLI agenta jako child process — wynik to JSON z `finalText`, `turns`, `reachedMaxTurns`, `toolCalls`.

## Porównanie modeli jako decyzja architektoniczna

Ewaluacje pozwalają testować wiele modeli na tych samych danych. Wynik to nie tylko pass/fail — modele różnią się liczbą kroków, stylem komunikacji, stabilnością. Tańszy model może zdać testy, ale ujawniać problemy przewidujące awarie na produkcji. Ewaluacje dają podstawę do świadomego wyboru.

## Optymalizacja schematów pod mniejsze modele

Nawet jeśli docelowo używasz najsilniejszego modelu — warto optymalizować interfejs narzędzi pod mniejsze. Narzucenie restrykcji w schemacie podnosi skuteczność również przy mocniejszych modelach.

## Uzupełnienia

### Workflow odkrywania: API docs → SDK → lista akcji → pytania filtrujące

Proces projektowania zaczyna się od zgrania dokumentacji API (jako Markdown, pomijając niechciane obszary) i kodu SDK do kontekstu agenta. Następnie agent generuje zwięzłą listę dostępnych akcji. Na tej liście odpowiadasz na trzy pytania filtrujące: (1) które akcje są potrzebne, (2) które połączyć w jedno narzędzie, (3) które zablokować. To decyduje o granicach narzędzi zanim napiszesz linijkę kodu.

### Ewaluacja bez system promptu — walidacja samo-opisowości narzędzi

Scenariusze ewaluacyjne uruchamiane są niemal bez wiadomości systemowej. Cel: sprawdzić, czy opisy narzędzi i ich schematy są wystarczające same w sobie. Jeśli agent bez instrukcji radzi sobie z narzędziami — po wyspecjalizowaniu poradzi sobie jeszcze lepiej. To jednocześnie walidacja jakości `.describe()` w schematach.

### Mock client z synthetic data jako warunek offline eval

Pełny mock client (`mock-client.ts`) implementuje identyczny interfejs co realny, na statycznych danych syntetycznych. Flaga `EVAL_MOCK_GMAIL` przełącza między real API a mockiem. Pozwala uruchamiać ewaluacje bez dostępu do konta Gmail, w pełni deterministycznie. Mock zawiera realistyczne wątki, załączniki, etykiety i reguły whitelist — struktura danych jest na tyle bogata, by testować edge case'y.

### `.describe()` na każdym polu schematu — to opis widoczny dla modelu

Każde pole Zod ma `.describe(...)` — ten opis trafia do JSON Schema jako `description` i jest jedynym źródłem wiedzy modelu o semantyce parametru. Bez tego model zgaduje znaczenie pól. Przykład: `cursor: z.string().describe('Pagination token from a previous gmail_search response (data.nextCursor). Use only to fetch the next page.')`.

### Stateful provider — serializacja kroków w scenariuszach wieloetapowych

`provider.stateful.mjs` utrzymuje sesje (via `sessionKey = evaluationId:conversationId:model`) i serializuje wywołania w ramach jednej konwersacji — kolejny krok czeka na wynik poprzedniego. Historia sesji jest persistowana na dysku (`sessionDir`). Dzięki temu Promptfoo może testować wieloetapowe scenariusze (np. użytkownik pyta → agent szuka → użytkownik prosi o akcję → agent wysyła) z zachowaniem stanu między krokami.

### Trzy kategorie scenario evalów

- **readonly** — blokuje send/modify, testuje wyszukiwanie i nawigację
- **safety** — weryfikuje policy enforcement (enforced draft przy zewnętrznych odbiorcach), assert `send_policy_enforced` + `llm-rubric` na `finalText` sprawdza czy agent nie twierdzi że wysłał gdy policy zablokowało
- **actions** — pełne akcje multi-step, w tym reply/forward

Każda kategoria ma osobny YAML config + dataset + dedykowane asercje.

### `expected_outcome` — asercja na semantyczny rezultat akcji

Oprócz `required_tools`/`forbidden_tools`, eval sprawdza `expected_outcome`: `'enforced_draft'` (policy zablokowało wysyłkę), `'sent'`, `'drafted'`, `'sent_or_drafted'`, `'search'`, `'read'` itd. Asercja weryfikuje nie tylko że agent wywołał właściwe narzędzie, ale że rezultat ma oczekiwany kształt (np. `data.status === 'drafted' && data.policy.enforcedDraft === true`).

### Praktyczne ograniczenia narzędzi z zadania negotiations

Maksymalnie 2 narzędzia, odpowiedź max 500 bajtów, agent ma max 10 kroków, parametry w języku naturalnym (nie ustrukturyzowane). Te ograniczenia wymuszają: (a) łączenie funkcjonalności w jedno narzędzie z inteligentnym routingiem, (b) kompaktowe odpowiedzi bez JSON waste, (c) opisy narzędzi na tyle precyzyjne by agent sam rozpoznał który parametr do którego narzędzia.

## Przykłady kodu — progresja

### Delta 1: ToolDefinition — zunifikowany interfejs rejestracji

```ts
interface ToolDefinition {
  name: string;
  description: string;
  schema: z.ZodType;
  handler: (args: any) => Promise<unknown>;
}
```

Każde narzędzie to name + description + schema Zod + handler. Rejestr to `Map<string, ToolDefinition>`. Zod schema serializuje się do JSON Schema dla API modelu.

### Delta 2: Zod .refine() — walidacja biznesowa w schema

```ts
.refine(d => !(d.replyToMessageId && d.forwardMessageId), {
  message: 'replyToMessageId and forwardMessageId are mutually exclusive.',
})
.refine(d => d.mode !== 'new' || ((d.to ?? []).length > 0 && d.subject), {
  message: 'to and subject are required when mode="new".',
})
```

Reguły zależnościowe między polami (np. tryb reply wymaga replyToMessageId) żyją w schemacie, nie w handlerze. Błędy walidacji trafiają do agenta jako structured hint.

### Delta 3: asResponseTools — konwersja Zod → JSON Schema dla Responses API

```ts
const asResponseTools = (definitions: ToolDefinition[]): Tool[] =>
  definitions.map(tool => {
    const { $schema: _, ...parameters } = z.toJSONSchema(tool.schema);
    return { type: 'function', name: tool.name, description: tool.description, parameters };
  });
```

### Delta 4: Pipeline wykonania tool call z pełnym trace

```
parse JSON → validate schema → run handler → wrap in { data, hint }
```

Każdy etap z osobnym ścieżką błędu → structured hint. Trace zapisuje `turn`, `name`, `callId`, `rawArguments`, `parsedArguments`, `output`/`error` — używany w evalach do asercji na zachowanie agenta.

### Delta 5: Agent runtime — pętla tur z limitem

```ts
for (let turn = 1; turn <= maxTurns; turn++) {
  const result = await executeTurn(turn, params);
  if (result.done) return { finalText, turns: turn, reachedMaxTurns: false, toolCalls };
}
return { finalText: MAX_TURN_FINAL_TEXT, turns: maxTurns, reachedMaxTurns: true, toolCalls };
```

Agent wykonuje tury aż do braku tool calls (odpowiada tekstem) lub limitu tur. Wynik jest strukturą gotową do eval.
