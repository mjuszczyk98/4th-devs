# S03E02 — Ograniczenia modeli na etapie założeń projektu

## Nie automatyzuj, wspieraj

Pełna automatyzacja procesów przez AI to obecnie zły pomysł w większości przypadków. Zamiast tego buduj narzędzia wspierające ludzi — etykietowanie, priorytetyzację, filtrowanie, szkicowanie. Błędy w trybie wsparcia mają ograniczone konsekwencje, bo człowiek pozostaje w pętli.

Specjalistyczne, wąskie narzędzia > all-in-one agenty dla całej organizacji. Prompt injection i halucynacje uniemożliwiają obecnie bezpieczne autonomiczne systemy o szerokim zakresie.

## Bezpieczeństwo przez ograniczenia programistyczne

Prompt systemowy = publiczny. Nigdy nie umieszczaj w nim sekretów.

Dostęp agenta do zasobów i akcji **musi** być kontrolowany kodem, nie promptem. Kluczowe zasady:
- Agent nie powinien móc wysyłać wiadomości, publikować treści ani wykonywać nieodwracalnych akcji
- Zewnętrzne źródła danych = potencjalny wektor ataku — limituj dostęp lub weryfikuj output
- Filtr prompt injection: osobny prompt w izolowanym zapytaniu zwraca "bezpieczne"/"niebezpieczne", weryfikowane programistycznie. Atakujący nie zna frazy kluczowej.

## Izolacja kontekstu (wzorzec scoped access)

Agent pracujący na danych wielu kont/projectów musi mieć programistycznie wymuszone ograniczenia widoczności wiedzy w momencie generowania treści. Przykłady:

- **Kategoryzacja** — brak ograniczeń, agent widzi wszystko
- **Szkicowanie odpowiedzi** — dostęp do KB zablokowany do konta + kategorii dopasowanych do typu kontaktu nadawcy

Lock jest binarny i egzekwowany kodowo — agent nie może go ominąć przez prompt.

## Przenoszenie obliczeń do kodu (sandbox)

LLM nie powinien "przeliczać w pamięci". Dla danych przekraczających możliwości kontekstu: agent generuje kod, kod wykonuje się w izolowanym środowisku.

To redukuje: halucynacje, koszty tokenów, czas wykonania. 150k+ linii danych → 6-10 kroków agenta zamiast setek zapytań.

Architektura trzech procesów: **Proces Główny** (agent LLM) → **MCP** (narzędzia plikowe) → **Sandbox** (wykonanie kodu). Sandbox komunikuje się z hostem przez HTTP bridge, dostając prelude z wygenerowanymi funkcjami proxy do narzędzi.

## Zarządzanie trudnością zadań (heartbeat + kontrakty)

Systemy agentowe wykonujące wieloetapowe procesy potrzebują struktury: **kontrakty**, **heartbeat**, **pamięć**.

**Kontrakt celu** (GoalContract) definiuje: objective, must_have, forbidden, budżet kroków, budżet replanowania, max zadań, warunki wymagające aprobaty człowieka.

**Heartbeat** = pętla managera: sprawdza stan zadań → przydziela gotowe do realizacji → agenci wykonują → stan aktualizowany programistycznie → kolejny cykl. Zadania mają zależności, statusy, przypisanych agentów. Mogą być realizowane równolegle lub sekwencyjnie.

**Planowanie**: LLM generuje plan na podstawie celu + mapy możliwości agentów → walidacja strukturalna → repair loop (max 2 próby naprawy) → materializacja jako pliki markdown. Jeśli nie wykonalne → no-go z uzasadnieniem.

**Replanowanie**: w trakcie wykonania agent może dojść do wniosku, że potrzebuje dodatkowych zadań. ReplanPatch: add_task, split_task, reassign_owner, change_dependencies, de_scope_task, cancel_open_task. Ograniczone budżetem.

## Observational Memory

Dla długotrwałych sesji agentowych — kompresja historii konwersacji do ustrukturyzowanych obserwacji z priorytetami (🔴/🟡/🟢). Obserwacje zastępują pełną historię w kolejnych zapytaniach. Zastosowane per-agent, umożliwia komunikację między agentami przez system plików.

## Przykłady kodu — progresja

### 1. `03_02_code` — Sandbox z wykonywaniem kodu

**Czego uczy:** Przeniesienie ciężaru obliczeniowego z LLM do deterministycznego kodu w izolowanym środowisku.

Delta: Agent ma dwa narzędzia — `fs_read`/`fs_write` (przez MCP) i `execute_code` (sandbox Deno). Kod generowany przez LLM działa na surowych danych, nie w kontekście promptu.

Klucz: Poziomy uprawnień sandboxa (`safe` → `standard` → `network` → `full`) kontrolują co wygenerowany kod może zrobić. Bridge HTTP pozwala kodowi w sandboxie wywoływać narzędzia hosta.

### 2. `03_02_email` — Dwufazowy agent z izolacją kontekstu

**Czego uczy:** Rozdzielenie operacji na "bezpieczne" i "wymagające izolacji" + programistyczny lock wiedzy.

Delta względem code: Zamiast sandboxa — kontrola dostępu do wiedzy. Dwie fazy:
- **Triage**: pełny dostęp, etykietowanie maili, planowanie odpowiedzi
- **Draft**: `lockKnowledgeToAccount(account)` — KB zablokowany do jednego konta + kategorii dopasowanych do typu kontaktu. Lock w kodzie (`access-lock.ts`), nie w prompcie.

Agent nie może wysyłać maili — szkic wraca do człowieka. Eval suite weryfikuje izolację, język, odporność na malicious emails.

### 3. `03_02_events` — Multi-agent z heartbeat, kontraktami i memory

**Czego uczy:** Orkiestracja wielu wyspecjalizowanych agentów przez pętlę heartbeat z dynamicznym planowaniem.

Delta względem email: Od pojedynczego agenta do systemu. Trzy nowe mechaniki:
- **GoalContract** (`goal.md` z frontmatter) → plan generowany przez LLM → walidacja → materializacja jako pliki zadań
- **CapabilityMap** skanuje szablony agentów (`*.agent.md`) budując mapę umiejętności potrzebnych do przypisania zadań
- **Observational Memory** kompresuje historię sesji każdego agenta — umożliwia długotrwałą pracę bez przepełnienia kontekstu

Agent może wywołać `request_human` — system pauzuje i czeka na decyzję człowieka (human-in-the-loop).
