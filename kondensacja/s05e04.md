# S05E04 — Produkcja

## Różnica między MVP a produkcją

Budowanie narzędzi AI na własne potrzeby jest proste — nawet kilkanaście minut wystarczy na stworzenie rozwiązania przynoszącego realną wartość. Można przy tym skorzystać z technologii, którymi nie posługujemy się na co dzień (np. Swift), a największym wyzwaniem jest sam pomysł.

Nawet przy pierwszych wdrożeniach pojawia się **przepaść** dzieląca środowisko developerskie/MVP od środowiska produkcyjnego. W aplikacjach generatywnych problem ten jest jeszcze bardziej widoczny.

## Detale mają znaczenie

Generatywne aplikacje muszą być **bardziej dopracowane** niż klasyczne. Nie chodzi tylko o architekturę — chodzi o drobne rzeczy, na które normalnie nie zwrócilibyśmy uwagi. Modele językowe niemal zawsze będą je pomijać — to **my musimy zwrócić na nie uwagę**. Umiejętność rozpoznawania takich detali bierze się niemal wyłącznie z doświadczenia.

### Usuwanie i edytowanie wiadomości w czacie

- Wiele interfejsów pozwala na usuwanie wiadomości, ale tej funkcji nie znajdziemy w ChatGPT, Claude, Claude Code czy Cursor.
- Częściej spotykana jest opcja **rozgałęzienia** konwersacji lub **przywrócenia** jej do wskazanego punktu.
- Usuwanie wiadomości ze środka zaburza historię, może negatywnie wpływać na zachowanie modelu, mechanizm zarządzania kontekstem, a nawet uniemożliwić prowadzenie dalszej konwersacji.
- **Ograniczenia API:** API Anthropic i Gemini wymagają, aby wiadomość użytkownika była ostatnia — w przeciwnym razie zwrócą błąd.
- Edytowanie treści wiadomości modelu nie powinno być możliwe — agent posługuje się narzędziami, a edycja może otworzyć ścieżkę do **many-shot jailbreaking**.

### Problemy z przetwarzaniem audio (dyktowanie)

- Niektóre modele (np. Whisper) są prawdopodobnie trenowane na napisach filmowych, co prowadzi do halucynacji: cisza interpretowana jako "Thanks for watching!" lub "Subtitles by the Amara.org community".
- Modele wielojęzyczne mają problemy przy mieszaniu języków.

### Wzmianki o niedostępnych możliwościach

Jeśli w instrukcji systemowej pojawi się jakakolwiek wzmianka o możliwościach, które w rzeczywistości **nie są dostępne** (np. przeszukiwanie Internetu bez aktywowanego narzędzia), model i tak **może** zachować się tak, jak gdyby było inaczej.

### Wydajność interfejsu przy długich treściach

Pole tekstowe po wklejeniu dużej ilości tekstu staje się niemal niemożliwe do edycji z powodu spadku wydajności. W rozmowach z LLM bardzo często wkleja się duże ilości treści. Rozwiązanie: wykrywanie zbyt dużej ilości treści i obsługiwanie jej tak, jak gdyby został dołączony plik tekstowy.

### Kluczowe właściwości

- Współdzielenie zasobów między konwersacjami, rozgałęzianie wątków, udostępnianie między użytkownikami
- Duża kontrola nad tym, co dzieje się na poszczególnych etapach działania agentów
- Kontrola daje możliwości pracy asynchronicznej — oczekiwanie na wiele zależności i wznawianie
- Interakcje nie muszą odbywać się tylko między użytkownikiem a agentem — zlecenia mogą tworzyć inni agenci lub zdarzenia zewnętrzne
- Agenci mają "rewizje" z ustawieniami w chwili zapytania — pomocne przy analizie aktywności

## Agent Runtime — główna logika

Z perspektywy użytkownika agent odpisuje na wiadomości i w razie potrzeby wzywa innych agentów. W tle wiadomość jest traktowana jako **zadanie do wykonania**, które można wykonać, wznowić, zatrzymać, a także przekazać do narzędzi lub innych agentów.

### Przepływ

1. **Żądanie** — HTTP z wiadomością i ustawieniami. Uwierzytelnienie i walidacja.
2. **Inicjalizacja** — Utworzenie wpisów w bazie: sesja, wątek, wiadomość, job, run. **Wyłącznie zapis danych** — żadne dodatkowe akcje.
3. **Kolejka** — Job `queued`, run `pending`. Przygotowanie do późniejszego wznowienia w razie błędu.
4. **Scheduler** — Cykliczne odpytywanie o pary job/run wymagające działania. **Stała kolejność:** dostarczanie wyników od dzieci → wznowienia → odzyskiwanie po awariach → nowe zadania.
5. **Claim** — Worker rezerwuje run z czasem wygaśnięcia. Uruchamiany **heartbeat** — cykliczne odświeżanie potwierdzające, że proces działa.
6. **Pętla Agenta** — Gdy model nie wskaże narzędzi → odpowiedź zapisana, run zakończony. W przeciwnym razie → wykonanie narzędzi i kolejna iteracja.
7. **Delegacja** — Gdy model wywołuje `delegate_to_agent`, tworzony prywatny **child run** z własnym agentem, narzędziami i kontekstem. Parent przechodzi w **waiting**. Delegacja może być zagnieżdżona wielopoziomowo — jak stos wywołań.
8. **Dostarczenie wyniku** — Gdy child run zakończony, scheduler dostarcza wynik do parenta jako odpowiedź narzędzia. Zależność rozwiązana, parent wraca do kolejki.
9. **Zapis** — Wiadomości wątku oraz items (reasoning, wywołania narzędzi, wyniki, delegowanie).
10. **Odzyskiwanie** — Scheduler w każdym cyklu sprawdza "martwe" runy. Przywraca do `pending` i ponawia próbę z exponential backoff.

### Kluczowe mechanizmy

- **Heartbeat + Claim z wygaśnięciem** — jeśli proces upadnie, rezerwacja wygasa automatycznie
- **Hierarchia delegacji** — child → parent, wielopoziomowo, przezroczyste dla użytkownika (child run nie jest przypisany do widocznego wątku)
- **Automatyczne odzyskiwanie** — wykrywanie martwych runów i ponawianie z backoff

## Lekcje z produkcji

### Mamy nowe możliwości, ale zasady są takie same

Nawet gdy 100% kodu generujemy z AI (pod nadzorem), pytania "co ma znaczenie?", "jaki problem rozwiązujemy?", "czy można lepiej?", "co nam umyka?" pozostają tak samo trudne. Generowanie rozbudowanych aplikacji kilkoma promptami **nie rozwiązuje wszystkich problemów**.

### Wszyscy eksplorujemy

Im bardziej rosną możliwości i popularność LLM, tym więcej nowych pytań. Wszystkie przekonania na temat AI należy **bardzo często aktualizować** lub wprost zakładać, że mogą być niepoprawne.

### Zrozumienie zasad tylko po to, aby je łamać

Gdy poznamy zasady pracy z modelami — warto je **kwestionować** i szukać własnych ścieżek. Koszt sprawdzenia nowej technologii/strategii, a nawet zbudowania całego produktu od podstaw, jest zupełnie inny niż kilkanaście miesięcy temu.

### Poziom zrozumienia AI przez użytkowników

Jedno z największych wyzwań. Stworzenie rozwiązań łatwych w obsłudze dla osoby z **minimalną lub zerową wiedzą o AI** — to ogromne wyzwanie. **Zawsze zakładaj, że użytkownicy nie wiedzą nic o modelach.**

### Rola jakości

W świecie gdzie każdy może wygenerować złożone narzędzie, ogromnym wyróżnikiem jest **jakość**. Warto korzystać z AI nie tylko by pracować szybciej, ale przede wszystkim by **podnosić jakość swojej pracy**.

### Agent bez czatu

Spojrzenie na systemy agentowe poza interfejsem czatu otwiera zupełnie nowe zastosowania. W kontekście produkcyjnym znacznie łatwiej zbudować logikę korzystającą z modeli **bez okna czatu** — wystarczy obserwowanie zachowań użytkownika, reagowanie na akcje i uzupełniane zasoby (wciśnięcie przycisku, wgranie pliku).

### Budowanie "dla agentów"

Dotychczas tworzyliśmy aplikacje z myślą o użytkownikach. Coraz częściej powinniśmy robić to z myślą o **agentach działających w imieniu użytkowników**.

## Przykłady kodu

### 1. Readiness Engine — deterministyczny scheduler (05_04_api)

Delta: pure function — pobiera snapshot stanu z DB i deterministycznie zwraca kolejną decyzję. Priorytetyzacja jest stała i jawnie zdefiniowana.

```typescript
// Kolejność priorytetów:
// 1. deliver_resolved_child_result — dostarcz wyniki zakończonych runów-dzieci
// 2. recover_timed_out_wait — timeout'y oczekiwań
// 3. requeue_waiting_job — odblokuj joby, których dependencje się spełniły
// 4. resume_waiting_run — wznow runy czekające
// 5. requeue_stale_running_run — recoveruj runy z expired lease (crash recovery)
// 6. execute_pending_run — rozpocznij nowy run (root przed child)
// Sortowanie wewnątrz grup: priorytet → queuedAt → runCreatedAt → runId
```

Czego uczy: rozdzielenie deterministycznej logiki (scheduler) od niedeterministycznej (LLM). Stała kolejność priorytetów zapewnia przewidywalność.

### 2. Claim & Heartbeat — fault tolerance (05_04_api)

Delta: mechanizm lease'ów z heartbeat zapewniający, że martwy proces nie blokuje zadań. Stale recovery z exponential backoff.

```typescript
// claim w run_claims: runId → workerId, acquiredAt, expiresAt, renewedAt
// heartbeat co leaseTtlMs/3
// Jeśli proces umrze → heartbeat wygasa → scheduler wykrywa stale running run
// → requeue z exponential backoff: staleRecoveryBaseDelayMs * 2^(count-2)
// Limit: maxStaleRecoveries (domyślnie 5), po czym run = failed
```

Czego uczy: klasyczny wzorzec distributed systems — lease + heartbeat + exponential backoff. Wymagany dla każdego systemu z długotrwałymi zadaniami.

### 3. Delegation — tworzenie child run (05_04_api)

Delta: delegacja jako transakcja bazodanowa tworząca pełną infrastrukturę dla child runa — prywatny wątek, własne narzędzia, zależności, eventy.

```typescript
// delegation-service.ts — w jednej transakcji:
// 1. Walidacja, że parent revision ma alias subagenta
// 2. Pobranie child agent i jego aktywnej rewizji
// 3. Utworzenie child Run (threadId=null — prywatny!)
// 4. Utworzenie job dependency edge (parent → child)
// 5. Utworzenie run dependency (parent wait na child)
// 6. Linkowanie plików z parenta do child run
// 7. Utworzenie handoff item + task item w items childa
// 8. Emisja eventów: run.created, child_run.created, delegation.started
```

Czego uczy: child run jest prywatny (`threadId: null`) — tylko root run pisze do widocznego threada. Zagnieżdżone delegacje są przezroczyste dla użytkownika.

### 4. Event Sourcing + Outbox Pattern (05_04_api)

Delta: każda zmiana domeny emituje event zapisywany w tej samej transakcji co dane. Background worker dostarcza asynchronicznie.

```typescript
// domain_events — event sourcing z event_no, aggregateId, aggregateType, payload
// event_outbox — outbox pattern z asynchroniczną dostawą
// Dostawa do: Realtime relay (SSE), Langfuse (telemetria), Projection dispatcher (read models)
```

Czego uczy: outbox pattern gwarantuje, że eventy nie zostaną utracone nawet przy awarii procesu między zapisem danych a wysyłką eventów.

### 5. Result Monad + Branded Types (05_04_api)

Delta: domena zwraca `Result` zamiast rzucać wyjątki. Każdy ID ma swój typ.

```typescript
// Result — błędy są wartościami, nie przepływem sterowania
export type Result<TValue, TError> =
  | { ok: true; value: TValue }
  | { ok: false; error: TError }

// Branded types — kompilator nie pozwala pomylić RunId z JobId
export type Brand<TValue, TName extends string> = TValue & { readonly __brand: TName }
export type AccountId = Brand<string, 'AccountId'>
export type RunId = Brand<string, 'RunId'>

// Prefixed IDs — każdy ID ma prefiks: acc_, run_, job_, ses_, thr_, msg_, agt_, ten_
export const createPrefixedId = <TPrefix extends string>(prefix: TPrefix) =>
  `${prefix}_${crypto.randomUUID().replace(/-/g, '')}`
```

Czego uczy: Result monad wymusza obsługę błędów na każdym kroku. Branded types eliminują całą klasę błędów na poziomie kompilacji. Prefiksowane ID czynią logi natychmiast czytelnymi.

### 6. Svelte 5 Runes — reaktywny stan (05_04_ui)

Delta: `$state` może być używane poza komponentami Svelte (w `.svelte.ts`), co pozwala tworzyć reactive stores jako zwykłe obiekty TS.

```typescript
// chat-store.svelte.ts — stan w zwykłym module TS
const state = $state<ChatState>({
  messages: [],
  isStreaming: false,
})

// Trzy warstwy wiadomości:
let durableMessages: UiMessage[] = $state([])      // z backendu (REST)
let optimisticMessages: UiMessage[] = $state([])   // natychmiast po wysłaniu
let liveAssistantMessage: UiMessage | null = $state(null) // streaming z SSE

// buildProjectedMessages() łączy trzy warstwy w finalną listę dla UI
```

Czego uczy: trzy warstwy wiadomości (durable/optimistic/live) to wzorzec optimistic UI. Gdy serwer zwróci trwałą wiadomość, optimistic jest usuwana, live zastępowane przez durable.

### 7. rAF Batching + Event Sourcing (05_04_ui)

Delta: SSE events buforowane i flushowane na requestAnimationFrame — gwarantuje co najwyżej jeden update na klatkę animacji (~16ms).

```typescript
// with-raf-batching.ts
export const withRafBatching = <T>(onBatch: (batch: T[]) => void) => {
  let pending: T[] = []
  let scheduled = false
  const flush = () => {
    scheduled = false
    if (pending.length === 0) return
    const batch = pending
    pending = []
    onBatch(batch)
  }
  return {
    push(item) {
      pending.push(item)
      if (scheduled) return
      scheduled = true
      requestAnimationFrame(flush)  // + setTimeout(100) fallback
    },
  }
}

// materialize.ts — applyEvent implementuje event sourcing
export const applyEvent = (blocks: Block[], event: BackendEvent, ...) => {
  switch (event.type) {
    case 'stream.delta':     // append text to current text block
    case 'tool.called':      // push new tool_interaction block
    case 'tool.completed':   // update existing tool block with output
    case 'reasoning.summary.delta': // upsert thinking block
  }
}
```

Czego uczy: event sourcing po stronie klienta — zdarzenia z serwera materializują się w bloki UI przez czystą redukcję z indeksem. rAF batching zapobiega render thrashing.

### 8. Streaming Markdown z inkrementalnym rendering (05_04_ui)

Delta: treść nie jest re-renderowana w całości. committedSegments + liveTail = tylko ogon jest parsowany na deltę.

```typescript
export const syncIncrementalMarkdownView = (current, { content, streaming }) => {
  const delta = content.slice(current.processedContent.length)
  const candidateTail = current.liveTail + delta
  const blocks = parseMarkdownIntoBlocks(candidateTail)
  const { finalizedSources, liveTail } = splitBlocks(blocks, streaming)
  return {
    committedSegments: current.committedSegments.concat(
      toSegments(blockId, current.nextSegmentIndex, finalizedSources)
    ),
    liveTail,
    processedContent: content,
  }
}
```

Czego uczy: podział na frozen/live segmenty — fundament wydajnego renderowania streaming Markdown. Zatwierdzone segmenty zostają w DOM bez dotykania.

## Uzupełnienia

### Clean Architecture w agentic runtime

Projekt rygorystycznie dzieli kod na warstwy: `app/` (konfiguracja), `adapters/` (HTTP, AI, MCP), `application/` (komendy, runtime), `domain/` (encje, repozytoria), `db/` (Drizzle ORM), `shared/` (typy). Adaptery zależą od domeny, nie odwrotnie — wymiana komponentów infrastruktury bez dotykania logiki biznesowej.

### SQLite z WAL mode jako baza dla agent runtime

SQLite z pragmami: `journal_mode = WAL`, `foreign_keys = ON`, `synchronous = NORMAL`, `busy_timeout = 5000`. WAL mode + synchroniczny dostęp = zero latency na zapis, pełna trwałość, brak zewnętrznego serwera DB. Świadoma decyzja — dla single-process runtime, SQLite jest optymalny.

### Tenant-scoped Foreign Keys

Foreign keys zawsze obejmują `(id, tenant_id)` — gwarantuje na poziomie bazy, że wiersz z jednego tenanta nigdy nie odwoła się do wiersza z innego tenanta.

### Tool Registry — Plugin Architecture

Narzędzia mogą być: **native** (delegate_to_agent, suspend_run), **MCP** (dynamicznie z serwerów z fingerprint'ing i approval workflow), **provider** (web_search), **system** (internal). Każde narzędzie ma `isAvailable` callback dostający `ToolContext` — może sprawdzić czy agent ma dostęp w kontekście konkretnego runa.

### Virtual message list z height estimation

Wiadomości dzielone na chunki po 12. Tylko widoczne chunki (z overscan ±3) renderowane w DOM. `@chenglou/pretext` do pre-layout text measurement — oblicza ile linii zajmie tekst bez renderowania DOM, umożliwia przewidywanie wysokości przed montażem.

### Flash-less theme switching (FOUC prevention)

Theme aplikowany synchronicznie w `<head>` zanim jakikolwiek CSS się załaduje — eliminuje całkowicie FOUC. CSS containment (`contain: layout style`) izoluje layout scrollowalnej listy do własnej warstwy GPU.

### Command Palette z Provider Pattern

Command palette jest generyczna — funkcjonalność wstrzykiwana przez "providers" (konwersacje, workspace'y, agenci, MCP servers, tool profiles). Shortcut Layer Stack — gdy palette otwarty, skróty globalne dezaktywowane.

### Optimistic UI z stable UI keys

Gdy użytkownik wysyła wiadomość, natychmiast pojawia się jako "optimistic" z lokalnym ID. Kiedy serwer odpowie z prawdziwym ID, store mapuje stare na nowe przez `stableUiKeyByMessageId` — zapobiega remounting komponentu i utracie stanu scroll.

### Tool outcomes mogą być `waiting`

Narzędzie nie musi zwrócić natychmiastowego wyniku. Może zwrócić `{ kind: 'waiting', wait: {...} }` — run jest zawieszany (suspend). Obsługuje asynchroniczne interakcje: czekanie na człowieka, MCP, upload.

### Agent Markdown jako source of truth

Agenci definiowani jako Markdown z frontmatter (gray-matter). Frontmatter: model, tools, memory policy, workspace policy, subagent links. Każda zmiana tworzy nową rewizję z checksum SHA256.

### Współdzielenie root job między turami

`reopenThreadRootJobForNewMessage` — ten sam root job jest reopenowany dla kolejnych wiadomości, z nowym currentRunId. Job jest jednostką szeregowalną, nie jednorazową.

### agents.html — interaktywna wizualizacja

Self-contained HTML z animowaną wizualizacją sekwencji działania runtime. Cztery scenariusze: simple reply, tool use z follow-up, nested delegation, crash recovery. Każdy krok pokazuje warstwę, aktora, mutacje obiektów i hooki kodu.
