# S01E01 — Programowanie interakcji z modelem językowym

## Model językowy jako funkcja w kodzie

- LLM jest dostępny przez API (OpenAI, Anthropic, Gemini, OpenRouter), co daje programistyczną kontrolę nad kontekstem zapytania i zachowaniem modelu.
- Generowanie jest **autoregresywne**: model przewiduje kolejny token na podstawie wejścia + dotychczas wygenerowanej treści. Wygenerowanego tokenu nie można usunąć.
- API jest **bezstanowe** — każde żądanie musi zawierać kompletny kontekst (prompt systemowy + historia wiadomości). Obowiązuje limit okna kontekstowego wyrażony w tokenach.
- Token = fragment tekstu (część słowa, słowo, znak). Język polski generuje ~50-70% więcej tokenów niż angielski, co zwiększa koszt i czas.

## Sterowanie zachowaniem przez kontekst

- Kluczowa zasada: **sterowanie zachowaniem modelu = zarządzanie kontekstem**. Można tworzyć wiele zapytań, gdzie wynik jednego kształtuje wejście drugiego.
- Przykład: pierwsze zapytanie klasyfikuje intencję użytkownika, drugie dostosowuje prompt do wykrytej kategorii. Z perspektywy użytkownika — prosta interakcja; z perspektywy kodu — routing kontekstu.
- Kosztem większej kontroli jest wydłużony czas reakcji i wzrost kosztów (więcej zapytań API).

## Structured Outputs

- Mechanizm wymuszający odpowiedź modelu w formacie JSON zgodnym z dostarczonym JSON Schema (tryb `strict` gwarantuje strukturę).
- Dwa wymiary schema: (1) **struktura** — gwarantowana w trybie strict, (2) **wartości** — generowane na podstawie treści wejściowej oraz **nazw i opisów** pól schema. Nazwy i opisy muszą być zrozumiałe i zwięzłe.
- Kolejność właściwości w schema ma znaczenie: tokeny generowane wcześniej wpływają na kolejne (np. `reasoning` przed `sentiment` przed `confidence`).
- Uwzględnianie wartości "nieznany"/"neutralny" zmniejsza ryzyko halucynacji — nie zmuszamy modelu do decyzji gdy nie ma podstaw.

Typowe transformacje realizowane przez Structured Outputs: ekstrakcja, klasyfikacja, kompresja, wzbogacenie, synteza, tłumaczenie, weryfikacja, parafraza, generowanie.

## Semantyczne zdarzenia

- Interakcja z agentami = seria **zdarzeń**, nie prosta para pytanie-odpowiedź. Zdarzenia obejmują: tekst, wywołania narzędzi, żądania potwierdzenia, błędy, obrazy.
- Oddzielenie trzech warstw: (1) LLM generuje tekst/JSON, (2) aplikacja reaguje na zdarzenia i zarządza stanem, (3) użytkownik widzi reprezentację stanu — odległą od surowego outputu modelu.
- Komunikacja oparta o semantyczne zdarzenia (z ID, typem, metadanymi) zamiast surowego tekstu — ułatwia rozbudowę bez migracji danych.

## Strategie doboru modeli

| Strategia | Opis | Kiedy stosować |
|---|---|---|
| Główny | Jeden model dla wszystkiego | Proste aplikacje |
| Główny + Alternatywny | Duży model do trudnych zadań, mały do reszty | Optymalizacja kosztu/szybkości |
| Główny + Specjalistyczne | Różne modele do różnych typów zadań | Maksymalna skuteczność per zadanie |
| Zespół małych | Wiele małych modeli + dekompozycja/głosowanie | Niski koszt, eksperymenty |

- Pytanie "jaki model jest najlepszy?" jest błędne — właściwe: "jaki model jest najlepszy **w tej sytuacji?**". Jedyny sposób wyboru: testy na własnych zadaniach.
- Unikać vendor lock-in — aplikacja nie powinna być ściśle powiązana z jednym providerem.
- Frameworki (AI SDK itp.) obecnie nerekomendowane — narzucają ograniczenia bez adekwatnej wartości.

## Kluczowe funkcjonalności API providerów

- **Multimodalność** — Gemini Interactions API i OpenAI Responses API prowadzą; warto projektować architekturę na wielu providerów.
- **Natywne narzędzia** (web search, file search, code execution) — proste w użyciu, ale ograniczone w konfiguracji i silnie uzależniające od platformy.
- **Cache** — krytyczny mechanizm dla agentów; ogranicza "przeliczanie" danych wejściowych, obniża koszt i czas.
- **Rate limit** — limity szybko stają się wąskim gardłem na produkcji; warto planować ich zwiększenie z wyprzedzeniem.

## Organizacja promptów w kodzie

| Technika | Zalety | Wady |
|---|---|---|
| Inline | Prostota | Brak kompozycji, trudna konserwacja |
| Oddzielne pliki | Kompozycja, zmienne | Wymaga monitorowania finalnego promptu |
| Systemy zewnętrzne (Langfuse) | Wersjonowanie, monitoring poza kodem | Dodatkowa zależność |
| **Pliki markdown + YAML frontmatter** | Dostępne z FS w runtime, edytowalne przez agentów | Brak wbudowanego wersjonowania |

- Markdown + frontmatter = preferowany format. Łączy zalety pozostałych podejść i daje agentom dostęp do instrukcji w runtime (np. dynamiczne generowanie umiejętności).
- Prompty można też projektować deklaratywnie przez sygnatury (`input → output`) w narzędziach takich jak DSPy czy Ax.

## Projektowanie instrukcji systemowych

Elementy składowe promptu systemowego:

- **Tożsamość** — nadaje kontekst, kieruje uwagę modelu ku skojarzeniom (nie daje magicznych zdolności).
- **Strukturyzacja** — tagi XML-like wyznaczają granice sekcji.
- **Limity** — zmniejszają ryzyko niepożądanych zachowań, ale nie dają gwarancji.
- **Ograniczenia** — warunkowe reguły działania (np. "edytuj zadanie dopiero po jego odnalezieniu").
- **Styl** — zwięzły styl wypowiedzi przyspiesza działanie, ale "modele myślą poprzez generowanie tokenów" — nie należy z tym przesadzać.
- **Adaptacja** — instrukcje dla sytuacji, w których model naturalnie sobie nie poradzi.
- **Kalibracja** — konkretne wskazówki formatowania (np. "używaj Mermaid dla diagramów").

**Generalizowanie generalizacji**: zamiast dodawać specyficzną regułę per problem, kształtuj proces myślowy modelu. Przykład: zamiast "nie używaj add_event gdy użytkownik prosi o zadanie" → "zastanów się głośno nad wyborem narzędzia, określ pewność, poproś o doprecyzowanie w razie wątpliwości". Taka reguła adresuje kategorię problemu, nie pojedynczy przypadek.

- Instrukcje systemowe **nie są zabezpieczeniem** — można je ominąć (jailbreaking). Bezpieczeństwo implementuje się na poziomie kodu i założeń projektowych.

## In-context learning

- Model wykorzystuje wzorce widoczne w treści zapytania. Wystarczy umieścić wiedzę w kontekście — model z niej skorzysta.
- Few-shot / many-shot: prezentowanie przykładów interakcji pozwala modelu rozpoznać schematy klasyfikacji, formatowania itp.
- Obecnie preferowany pattern: instrukcja systemowa zawiera zgeneralizowane zasady, specyficzne przykłady i dokumenty wczytywane dynamicznie przez narzędzia.

## Przetwarzanie wieloetapowe (wzorce z Grounding)

- Tekst przetwarzany w **fragmentach** (akapity) nawet gdy całość mieści się w oknie kontekstowym — skupienie uwagi modelu na mniejszym zakresie zwiększa skuteczność i pozwala używać mniejszych modeli.
- Zapytania wysyłane **równolegle w grupach** — krótszy czas wykonania + unikanie rate-limit.
- Wyniki pośrednie **zapisywane do plików** — możliwość wznowienia po błędzie bez powtarzania etapów.
- **Cache**: jeśli pliki dla danego wejścia istnieją, pomijaj regenerację.

## Przykłady kodu

### 1. Interaction — wieloturnowa rozmowa przez Responses API

Delta: podstawowa interakcja z API, budowanie historii kontekstu między zapytaniami.

```js
const response = await fetch(RESPONSES_API_ENDPOINT, {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    Authorization: `Bearer ${AI_API_KEY}`,
  },
  body: JSON.stringify({
    model: MODEL,
    input: [...history, toMessage("user", input)],
    reasoning: { effort: "medium" },
  }),
});
```

Czego uczy: każde zapytanie zawiera pełną historię (`input: [...history, nowy_msg]`). Model nie pamięta poprzednich zapytań — my manually dołączamy je jako kontekst. Odpowiedź zwraca `usage.output_tokens_details.reasoning_tokens` — miara "wysiłku" modelu.

### 2. Structured — wymuszony format JSON przez JSON Schema

Delta względem #1: dodanie `text.format` ze schema gwarantuje, że odpowiedź jest parsowalnym JSON-em o określonej strukturze.

```js
const personSchema = {
  type: "json_schema",
  name: "person",
  strict: true,
  schema: {
    type: "object",
    properties: {
      name: { type: ["string", "null"], description: "Full name. Null if not mentioned." },
      skills: { type: "array", items: { type: "string" }, description: "List of skills. Empty array if none." },
    },
    required: ["name", "skills"],
    additionalProperties: false,
  },
};

// W wywołaniu API:
body: JSON.stringify({ model: MODEL, input: "...", text: { format: personSchema } }),
// Odpowiedź:
const person = JSON.parse(extractResponseText(data));
```

Czego uczy: typy `["string", "null"]` + opisy z instrukcją "use null if..." realizują wzorzec obsługi nieznanych wartości. `additionalProperties: false` + `strict: true` = gwarancja struktury. Odpowiedź to string do sparsowania, nie gotowy obiekt.

### 3. Grounding — pipeline czterostopniowy ze Structured Outputs

Delta względem #2: wieloetapowe przetwarzanie, każda stage ma własny schema + prompt, etapy łączą się kaskadowo.

**Etap 1 — Extract**: z każdego akapitu wydobywa koncepty ze schema określającym `label`, `category`, `needsSearch`, `searchQuery`, `surfaceForms`. Jeden akapit = jedno zapytanie API.

**Etap 2 — Dedupe**: łączy synonimiczne koncepty w grupy (`canonical`, `ids`, `aliases`). Schema wymusza uzasadnienie (`rationale`) grupowania.

**Etap 3 — Search**: dla każdego unikalnego konceptu wywołuje API z web search (`tools: [{ type: "web_search_preview" }]`), schema wymusza `summary`, `keyPoints`, `sources`.

**Etap 4 — Ground**: generuje HTML z `<span class="grounded" data-grounding="...">` owijający frazy z oryginalnego tekstu.

Czego uczy:
- Pipeline = seria zapytań z kaskadowymi zależnościami — wynik etapu N jest wejściem etapu N+1.
- Każdy etap ma dedykowany prompt i schema — separacja odpowiedzialności.
- Równoległe batchowanie: `Promise.all` na grupach akapitów/konceptów.
- API `extractSources()` parsuje odpowiedź i wyciąga źródła web search z `web_search_call` i `url_citation` — przykład wydobywania metadanych z odpowiedzi API poza samym tekstem.
