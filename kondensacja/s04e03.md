# S04E03 — Kontekstowa współpraca z AI

## Perspektywa integracji AI z codziennością

- Przesunięcie z "interakcji z AI" na "współpracę kontekstową" — AI działa w tle, reaguje na zdarzenia, nie wymaga ciągłej uwagi.
- Urządzenia, aplikacje i usługi projektowano dla ludzi — teraz interakcja coraz częściej obejmuje boty. Skutek: wzrost znaczenia API, scrapowania, deep-linków, uprawnień.
- Modele nie są dobrymi doradcami w zakresie *gdzie* zastosować AI — ich sugestie są generyczne. Własna obserwacja procesów + pytania "jak AI może pomóc?" i "czy powinniśmy tu angażować AI?" to właściwa metoda odkrywania przypadków użycia.

## Obszary integracji

| Obszar | Mechanizm integracji | Przykładowy scenariusz agentowy |
|---|---|---|
| OS (Mac/Win/Linux) | CLI, skrypty, AppleScript, deep-linki, harmonogramy | Manager schowka, sterowanie DND, dostęp do urządzeń |
| Mobile | API czujników, lokalizacja, automatyzacje | Odczyt lokalizacji do organizacji notatek |
| Komunikatory | API wiadomości, obserwowanie wybranych kanałów | Monitorowanie aktywności, feedback, wybiórcze reakcje (np. tylko na wiadomości z określoną emotikonką) |
| Kalendarz | API zdarzeń, webhooki | Sugestie wydarzeń, time blocking, przegląd harmonogramu |
| E-mail | API, etykiety, przekierowania | Dodatkowy adres dla agenta, read-only dostęp, filtrowanie, wzbogacanie kontekstu |
| Internet | Scraping (Firecrawl/Jina), RSS, API (YouTube, X) | Nasłuchiwanie sygnałów, monitorowanie changelogów, trending topics |
| Zarządzanie zadaniami | API (Linear, Todoist, ClickUp), webhooki | Inicjowanie procesu kreatywnego, sugestie technik, automatyczne powiązania z dokumentami |
| Repozytoria | GitHub API, Actions | Code review, rozwiązywanie ticketów — ale ryzyko prompt injection realne (np. wykradanie tokenów) |
| CRM / Sprzedaż | API (Attio i in.) | Research, lejki, oferty — największy bezpośredni ROI, ale ryzyko zniszczenia relacji |
| Narzędzia graficzne | API modeli obrazu, Figma, Blender | Generowanie grafik dopasowanych do kampanii |
| Bazy danych | Convex/Supabase — natychmiastowy dostęp API | Stan aplikacji, konfiguracje, dane sesji |

## Wspólny mianownik integracji

- **API** — kluczowy wymóg. Zwrócić uwagę na kompletność endpointów i zwracanych danych — mogą być niekompletne z perspektywy agenta.
- **Webhooki** — mechanizm reaktywny: agent nie odpytuje, tylko odpowiada na zdarzenia.
- **Deep-linki** (x-scheme-url) — URL-y wywołujące akcje w aplikacjach desktopowych bez API.
- **Powiadomienia mailowe** — jako wyzwalacze akcji (np. parsowanie powiadomienia o nowym zgłoszeniu).

## Scenariusze kontekstowe — wzorce projektowe

### Katalogi aktywne
Koncepcja folderów, których zawartość uruchamia automatyzację. Np. `concept/ → review/ → ready/ → published/` — dokument wstawiony do `concept/` jest transformowany przez agenta i przenoszony dalej. Podobny wzorzec: `inbox/ → processing/ → archive/`.

### Izolacja agentów
- Agenci działający w tle powinni być **izolowani** — agent klasyfikujący zgłoszenia nie musi wiedzieć o agencie generującym zestawienia.
- Pełna izolacja nie zawsze możliwa — wtedy zaangażowanie człowieka bywa potrzebne częściej, a efektywność systemu spada, gdy czeka na manualne decyzje.
- Problem zależności między agentami ujawnia się często dopiero po czasie — nawarstwienie informacji lub rozbudowa systemu odkrywa ukryte sprzężenia.

### Samoobserwacja systemu
- Agenci mogą cyklicznie weryfikować skuteczność przepływu informacji (analogia do LLM-as-a-judge).
- Jeśli generowane raporty/digesty przestają być czytane — system powinien to wykryć i zasygnalizować.
- Źródła danych stające się niedostępne powinny być automatycznie oznaczane i usuwane (lub flagowane do weryfikacji).

### Zjawisko zapętlania logiki
- Zbyt elastyczne automatyzacje mogą doprowadzić do zapętlania lub nieoczekiwanych interakcji między agentami.
- Projektować tak, aby problem konfliktów **w ogóle nie występował** — prewencja przez prostotę, nie rozwiązywanie po fakcie.

### Kontekst urządzenia → adaptacja zachowania
- Agent z dostępem do statusu urządzenia (otwarta aplikacja, lokalizacja, tryb DND) może dynamicznie dobierać kanał komunikacji (SMS, powiadomienie, cisza).
- Wymaga natywnego dostępu — możliwe przez własne mini-aplikacje (AI potrafi wygenerować aplikacje w Swift/Rust/C#).

## Przepływ danych między aplikacjami

- **Szablony projektowe** — frameworki aktywności + szablony + integracja z narzędziami do zadań. Agent unika "efektu pustej kartki" i monitoruje standardy.
- **Przekierowania** — zgłoszenia natychmiast kierowane do właściwych osób na podstawie klasyfikacji LLM z dostępem do bazy wiedzy (nawet pojedynczych plików).
- **Optymalizacja workflow** — agenci obserwują skuteczność własnych procesów (skanowanie logów, odczyt rezultatów z aplikacji) i generują rekomendacje.
- **Monitorowanie wskaźników** — MRR, Churn, NPS: wiele czynników pochodzi z feedbacku (ankiety, wiadomości, spotkania) — agenty ustalają priorytety i wychwytują powtarzające się wzorce.
- **Raporty** — LLM nie powinny generować kompleksowych raportów dla kluczowych decyzji. Wartość: transformacje i klasyfikacje niedostępne przez kod + równoległa analiza z odpowiednim kontekstem.

## Kluczowe napięcia projektowe

- **Ograniczenie zaangażowania człowieka** vs **rozsądek w uprawnieniach** — dawać agentom minimum potrzebnych uprawnień.
- **Elastyczność automatyzacji** vs **ryzyko zapętlenia** — klasyczne automatyzacje były zbyt sztywne, ale LLM-owe bywają zbyt elastyczne.
- **ROI z automatyzacji** vs **ryzyko zniszczenia relacji** — szczególnie w sprzedaży i komunikacji z klientami.
- **Autonomia agenta** vs **oczekiwanie na decyzje człowieka** — każda manualna akcja obniża efektywność systemu w tle.

## Uzupełnienia

### Kontrola jakości treści przez agentów
Treści trafiające do kolejki publikacji (np. newsletter) mogą być weryfikowane przez agentów: wykrywanie uszkodzonych/niewłaściwych linków, poprawność językowa, poprawność merytoryczna. Samo wykrycie jednego złego linku w newsletterze to potencjalnie duży zysk. Wzorzec: agent weryfikujący stoi na końcu pipeline'u publikacyjnego, przed zatwierdzeniem przez człowieka.

### Tworzenie powiązań w bazie wiedzy
Z czasem w bazach wiedzy pojawia się duplikacja informacji i potrzeba powiązania wpisów. Agenci obecnie nie radzą sobie z tym wystarczająco skutecznie, by oddać im proces w całości, ale generowanie sugestii powiązań na podstawie najnowszych zmian w repozytorium ma duży sens.

### Panel zarządzania (Second Brain)
Koncepcja spersonalizowanej „ściany" agregującej kontekst zewnętrzny (nowości branżowe) i wewnętrzny (obszary wymagające uwagi). Poprzedniczka LLM-era: „Second Brain". Przegląd materiałów z tego obszaru daje mnóstwo inspiracji rozszerzalnych przez AI.

### Nasłuchiwanie sygnału — ranking częstościowy
Agenci monitorujący źródła (YouTube, X, newslettery, blogi) mogą rankingować wydarzenia po częstotliwości pojawiania się w danym okresie. Przykład: narzędzie wspomniane wielokrotnie w ciągu miesiąca = sygnał, że warto zwrócić na nie uwagę. To konkretna technika priorytetyzacji, nie tylko „monitorowanie".

### Projektowanie pod przypadki brzegowe — przykład konsultacji
Źle zaprojektowany system agentowy przynosi więcej problemów niż wartości. Przykład z lekcji: proces konsultacji (rezerwacja → przygotowanie → realizacja → rozliczenie) — klient chcący zarezerwować wiele sesji i opłacić z góry rozbija założenia uproszczonego flow. Analogia do programowania: założenia adresowane w projekcie często okazują się niewystarczające w zderzeniu z rzeczywistością.

### Manager Schowka — wymóg prywatności
Obserwowanie schowka systemowego pod kątem zasobów wiedzy, wydarzeń, zadań, notatek (np. konfiguracji narzędzi). Kluczowe ograniczenie: zawartość schowka to ostatnia rzecz, którą chcemy przesyłać na serwery dostawców. Wymaga lokalnych modeli i odpowiedniego sprzętu.

### Sugestie wydarzeń — wzorzec „oddzielnego kalendarza"
Agent tworzy sugestie w oddzielnym kalendarzu (nie głównym), opierając się na notatkach i obserwowaniu otoczenia. Dopiero po akceptacji przenoszone do głównego. Rozwiązuje problem „zaśmiecania" kalendarza propozycjami.

### Dodatkowy e-mail agenta — wzorzec bezpieczeństwa
Agent posiada własny adres e-mail odbierający newslettery, powiadomienia systemowe i przekierowane wątki (auto-etykiety sygnalizujące potrzebę wzbogacenia kontekstu). Ograniczenia bezpieczeństwa: możliwość wysyłki wyłącznie do właściciela, dostęp „read only" do bazy wiedzy. Na główny adres trafiają wyłącznie najistotniejsze informacje lub ich zbiorcze wersje.
