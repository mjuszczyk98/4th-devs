# S04E04 — Projektowanie własnej bazy wiedzy dla AI

## Architektura bazy wiedzy — podział na strefy odpowiedzialności

Pięć katalogów z różnym właścicielem i celem:

| Katalog | Właściciel | Cel |
|---|---|---|
| `me/` | człowiek | tożsamość, preferencje, proces osobisty |
| `world/` | człowiek + agent | ludzie, miejsca, narzędzia, źródła, zdarzenia |
| `craft/` | człowiek + agent | idee, projekty, wiedza, eksperymenty, publikacje |
| `ops/` | agent | workflowy, research, dane operacyjne |
| `system/` | człowiek | profile agentów, szablony, reguły |

Kluczowa zasada: **człowiek odpowiada za treść i zasady, AI za organizację**. Granica musi być jawna — agent nie przekracza strefy człowieka bez wyraźnego pozwolenia.

## Markdown jako format wiedzy — zalety i ograniczenia

Markdown jest AI-native (modele generują i przetwarzają go naturalnie), programistycznie transformowalny (zwykły tekst) i edytowalny przez agentów w runtime. Wzbogacanie o strukturę: YAML frontmatter jako warstwa metadanych (publish, status, access, tags, attention).

Ograniczenia: Markdown przegrywa z Notion/Docs przy jednoczesnej edycji wielu osób, różnym poziomie uprawnień i zaawansowanym formatowaniu. Konwersja Markdown ↔ Notion traci informacje — lepiej wybrać format per obszar, nie konwertować.

Obrazy: zawsze jako zewnętrzne URL, nigdy lokalne pliki — agent musi móc cytować link w odpowiedzi, a lokalne pliki to uniemożliwiają.

## Baza wiedzy vs pamięć długoterminowa

W architekturze z lekcji to ta sama przestrzeń — pamięć długoterminowa i baza wiedzy pokrywają się, bo funkcjonują w tych samych plikach. Treść jest dynamiczna, tworzona przez człowieka i AI.

**Problem luk kontekstowych** — notatki pisane „dla siebie" są nieczytelne dla agenta:

| Luk | Przykład | Rozwiązanie |
|---|---|---|
| Niejawne referencje | „ostatnia rozmowa", „w poprzedniej wersji" | jawny link do powiązanej notatki |
| Nierozpoznawalne linki | skrócone URL bez opisu treści | pełne URL z opisem |
| Nadpisane wersje | edycja usuwa poprzednią wiedzę | wersjonowanie lub explicit „zastąpiono przez X" |
| Brak powtórzeń | powiązanie istnieje tylko w jednym dokumencie | linki w obu kierunkach |

Zasada: **notatki muszą być pisane tak, jakby czytelnik nie miał żadnego dodatkowego kontekstu.**

## Frontmatter — wieloosiowa kontrola metadanych

Osy niezależne od siebie:
- **publish** (`draft` → `review` → `live` → `updated`) — cykl publikacji, niezależny od statusu treści
- **status** (`seed` → `growing` → `evergreen` → `archived`) — dojrzałość treści
- **access.read / access.write** — dziedziczone z sekcji, nadpisywane per notatka
- **attention** (`who` + `reason`) — sygnał handoffu między człowiekiem a agentem
- **tags** — płaskie (bez hierarchii `ai/llm`), opisują **co**, nie **gdzie** (folder to załatwia)

Domyślnie wszystko jest `seed` + `draft` + dostęp zgodny z sekcją. Jawne ustawienie tylko przy nadpisaniu.

## Role AI w zarządzaniu bazą wiedzy

| Rola | Zakres | Uwagi |
|---|---|---|
| Transformacja | formatowanie, korekta, przepisywanie z audio/obrazu | człowiek = źródło, AI = formatter |
| Szablony | utrzymanie struktury notatki | szablony w `system/templates/` |
| Organizacja | weryfikacja placement'u, sugestia zmiany | nie automatyczna akcja |
| Linkowanie | wikilinkowanie między powiązanymi notatkami | wymaga reguł w `system/rules/` |
| Walidacja | sprawdzanie zgodności z zasadami | struktura, tagi, linki |
| Indeksowanie | generowanie Map of Content | programistycznie lub przez agenta |
| Audytowanie | wykrywanie braków, szumu, brakujących linków | dodatkowa weryfikacja |

## Szablony notatek — deterministyczna struktura

Szablony w `system/templates/` definiują: docelowy katalog, konwencję nazwy pliku, sekcje frontmatteru, sekcje treści. Agent przed utworzeniem notatki: czyta index → czyta szablon → decyduje o strukturze i lokalizacji → tworzy plik **lub aktualizuje istniejący**.

Efekt: im większa baza, tym większa wartość szablonów — spójność struktury notatek bez ręcznej dyscypliny. Jedna notatka = kilkanaście zapytań do LLM (orientacja + szablon + decyzja + zapis).

## Workflow jako pliki Markdown — procesy wieloagentowe

Workflow = katalog w `ops/` z plikami: `_info.md` (konfiguracja, źródła, fazy) + `01-*.md`, `02-*.md` ... (fazy). Każdy plik fazy ma frontmatter z `agent` (kto wykonuje) i `depends_on` (zależności).

**Proces daily-news:**
1. Alice czyta `_info.md` → deleguje fazę 1 do Ellie (research)
2. Alice czeka na wynik → deleguje fazę 2 do Tony'ego (assembly)
3. Alice czeka na wynik → deleguje fazę 3 do Rose (delivery)

Fazy **sekwencyjne** — nigdy równoległe, bo każda zależy od plików wygenerowanych przez poprzednią. Pliki fazowe są preskryptywne: dokładne kroki + lista „Do NOT" → beat goal-oriented instrukcje.

## Profile agentów jako pliki Markdown

Agent = plik `system/agents/<name>.md` z frontmatterem `{title, description, model, tools}` + treść jako system prompt. Loader parsuje gray-matter i zwraca obiekt konfiguracji.

Każdy agent ma: ograniczony zestaw narzędzi (np. Ellie: `files, web`; Tony: `files`; Rose: `files, send_email`), instrukcje z jawnymi ograniczeniami („read only those files", „do NOT browse workspace tree").

## Delegacja z kontrolą głębokości

`delegate` tool pozwala agentowi przekazać zadanie innemu agentowi. Dziecko dostaje własny model, narzędzia, instrukcje — działa niezależnie, zwraca tekstowy wynik. **Depth limit** (`MAX_DEPTH = 2`): agenci na maksymalnej głębokości nie widzą toola `delegate`, więc nie mogą spawnować kolejnych dzieci. Zapobiega nieskończonej rekurencji.

## MCP — namespacing i dynamiczna rejestracja

Narzędzia MCP łączone przez stdio, konfigurowane w `mcp.json`. Nazwy tooli prefiksowane serwerem: `files__fs_read`, `web__search`. Rejestracja dynamiczna: lokalne toolle + MCP toolle łączone w jeden registry. Filtrowanie po prefiksie serwera — agent deklaruje `tools: [files, web]`, dostaje wszystkie toolle z tych serwerów.

## Wzorzec ścieżki docelowej

Agenci delegowani otrzymują **dokładne ścieżki** do odczytu i zapisu — nie eksplorują filesystemu samodzielnie. Zmniejsza to liczbę kroków, eliminuje błędy nawigacji i utrzymuje przewidywalność. Orchestrator (Alice) jest jedynym agentem z orientacją w strukturze vaultu.

## Uzupełnienia

### Filozofia wdrożenia: iteracja od jednego punktu

Lekcja kładzie silny nacisk na to, że bazy wiedzy NIE buduje się od razu w pełnej strukturze. Kluczowe zasady:

- Wybierz **jeden obszar** lub nawet **jedną aktywność** i ją zaadresuj. Przykłady: spersonalizowany newsletter, podcast, planowanie dnia, hobby.
- Zacznij od czegoś, co się **podoba**, a dopiero potem od tego, co jest użyteczne. "Najlepsze pomysły nie są powodem wykonanej pracy, lecz jej rezultatem" — trzeba budować, eksperymentować i popełniać błędy.
- Na etapie kształtania struktury **nie trzeba pisać kodu** — katalog z bazą wiedzy można podłączyć bezpośrednio do Claude Code i wspólnie z agentem ukształtować pierwsze procesy, notatki i szablony. Kod pojawia się dopiero gdy procesy wymagają automatyzacji na zdalnym serwerze.

### Obszary wyłączone z automatyzacji ale obecne w bazie

Lekcja wyróżnia obszary, których **nie chcemy automatyzować**, ale które warto uwzględnić w bazie wiedzy. Przykład: tworzenie treści — człowiek pisze, ale agent ma dostęp do materiałów i może się do nich odwołać bez ręcznego przekazywania kontekstu. To trzecia strefa obok "nasza" i "agentowa": wiedza będąca kontekstem dla agentów, ale wnosząca wartość dla nas.

### Map of Content (MoC)

Notatki pełniące rolę indeksu/mapy dla wybranych obszarów treści. Mogą być generowane programistycznie lub przez agenta. W kodzie — `workspace/index.md` i `workspace/<sekcja>/index.md` pełnią tę funkcję: linkują dzieci, nie zawierają samodzielnej treści.

### mind.md — pełny zakres konstytucji bazy wiedzy

Distillation wspomina `mind.md` w 6 linijkach. Plik ma 258 linii i zawiera:

- **15 scenariuszy agentów** pokazujących jak każdy agent nawiguje po vaultcie (np. Ellie: `Ops/Research` → `World/Sources` → `Craft/Knowledge` → `Craft/Projects`; Rose: `Ops/Newsletter` → `Me/Preferences` → `Craft/Shared` → `World/Sources`). Te scenariusze są praktyczną dokumentacją przepływów między sekcjami.
- **Dziedziczenie uprawnień per sekcja** — np. `Craft/Knowledge` → write: `[adam, ellie, tony]`; `Craft/Lab` → write: `[adam, tony]`; `Craft/Shared` → write: `[adam, rose]`. Nie trzeba powtarzać w każdej notatce.
- **Wikilink conventions**: ścieżki vault-root absolutne (`[[Craft/Knowledge/AI/transformers]]`, nie `[[../../transformers]]`), aliasy przez pipe (`[[path|display text]]`).
- **Tag reuse**: agenci muszą sprawdzać tagi w sąsiednich notatkach przed tworzeniem nowych.

### Inżynieria promptów agentów — dwa tryby pracy

Każdy agent ma dwa tryby działania z osobną logiką:

1. **Workflow mode** — agent dostaje ścieżkę do pliku fazy, czyta TYLKO ten plik i wykonuje jego kroki. Nie czyta templates, indexów, nie eksploruje filesystemu. Plik fazy ma priorytet nad defaultowym procesem agenta.
2. **Standalone mode** — agent działa samodzielnie (np. Ellie czyta template knowledge.md, researchuje temat, zapisuje w odpowiednim katalogu).

Każdy agent ma w prompcie sekcję **"Do NOT"** — listę zakazanych akcji zapobiegającą niepotrzebnej eksploracji (np. "Do NOT browse the workspace tree, read templates, scrape full articles").

### Wzorzec scatter/gather w researchu

Ellie przy web scrapingu używa `outputMode: "file"` — wyniki zapisywane jako pliki, nie inline w kontekście. Następnie czyta tylko potrzebne fragmenty z zapisanych plików. Kontrola wielkości kontekstu: pełne artykuły nigdy nie trafiają do conversation history.

### Wielo-turnowa akumulacja konwersacji

`demo-example-queries.js` pokazuje wzorzec wielo-turnowej konwersacji stanowej: po każdej odpowiedzi conversation jest aktualizowana o `[...result.conversation, { role: "assistant", content: result.text }]` — agent pamięta poprzednie notatki w tej samej sesji i może się do nich odwołać przy kolejnych zapytaniach. To kontrastuje z single-shot `app.js` gdzie każde zapytanie zaczyna nową konwersację.

### dryRun przy aktualizacji

Agent prompts rozróżniają tworzenie i aktualizację plików: "When creating new files, write directly — never use dryRun. Only use dryRun when updating existing files." — mechnizm bezpieczeństwa przed nadpisaniem istniejącej wiedzy.

## Przykłady kodu

### 1. Loader — profil agenta z pliku Markdown

Delta: definicja agenta jako plik markdown, nie kod. Frontmatter = konfiguracja, treść = prompt.

```js
export const loadAgent = async (name) => {
  const raw = await readFile(join(AGENTS_DIR, `${name}.md`), "utf-8");
  const { data, content } = matter(raw);
  return {
    name: data.title ?? name,
    model: data.model ?? "gpt-4.1-mini",
    toolNames: data.tools ?? [],
    instructions: content.trim(),
  };
};
```

Czego uczy: agent = plik konfiguracyjny, nie klasa. Łatwo dodawać nowych agentów bez zmiany kodu — wystarczy nowy plik `.md` w katalogu agentów.

### 2. Delegate — tool przekazujący zadanie innemu agentowi

Delta względem zwykłych tooli: handler nie wykonuje logiki, tylko uruchamia **osobny agent loop** z własnym modelem i instrukcjami.

```js
export const createRun = (depth, chatFn) => async ({ agent: name, task }) => {
  const child = await loadAgent(name);
  const { text } = await chatFn(
    [{ role: "user", content: task }],
    { ...child, depth: depth + 1 },
  );
  return { agent: child.name, result: text };
};
```

Czego uczy: factory pattern z `depth` — każdy poziom delegacji zwiększa depth, a tool `delegate` jest niedostępny gdy `depth >= MAX_DEPTH`. Dziecko zaczyna świeżą konwersację (samo `user` message).

### 3. Agent loop z delegacją i filtracją tooli

Delta względem prostych loopów z S04E01: depth-aware tool availability, `Promise.all` na parallel tool calls, conversation accumulowana jako spread `[...current, ...calls, ...results]`.

```js
const canDelegate = depth < MAX_DEPTH;
const availableTools = canDelegate
  ? [...tools(toolNames), delegateDefinition]
  : tools(toolNames);

for (let step = 0; step < MAX_STEPS; step++) {
  const response = await complete(current, availableTools, { instructions, model });
  const calls = response.output.filter((item) => item.type === "function_call");
  if (calls.length === 0) return { text: response.output_text, conversation: current };
  const results = await Promise.all(calls.map(executeTool));
  current = [...current, ...calls, ...results];
}
```

Czego uczy: conversation rośnie o obiekty `function_call` i `function_call_output` — model widzi pełną historię wywołań i wyników. Brak `previous_response_id` — pełny kontekst re-sendowany co turn.

### 4. MCP registry — dynamiczne łączenie narzędzi z namespacing

Delta: narzędzia nie są statycznie zdefiniowane w kodzie, ale wczywtywane z serwerów MCP i prefiksowane.

```js
export const registerMcpTools = (mcpTools, mcpClients) => {
  for (const tool of mcpTools) {
    handlers[tool.name] = (args) => callTool(mcpClients, tool.name, args);
  }
  allDefinitions = [...localTools.map((t) => t.definition), ...toDefinitions(mcpTools)];
};

export const tools = (names) => {
  if (!names || names.length === 0) return allDefinitions;
  const nameSet = new Set(names);
  return allDefinitions.filter((def) =>
    nameSet.has(def.name) || nameSet.has(def.name.split("__")[0]),
  );
};
```

Czego uczy: prefix matching (`name.split("__")[0]`) pozwala agentowi deklarować `tools: [files]` i automatycznie dostać wszystkie toolle z serwera `files` (`files__fs_read`, `files__fs_write`, itd.).

### 5. Workflow phase file — proces jako instrukcja w Markdown

Delta: definicja procesu operacyjnego jako plik markdown z frontmatterem `{agent, depends_on}` i preskryptywnymi krokami.

```markdown
---
title: "Phase 1 — Research"
agent: ellie
depends_on: []
---
## Steps
1. Run `web__search` with queries `"{topic} news {yyyy-mm-dd}"` for each topic...
2. Read the first 50 lines of each search result file...
3. For each topic with results, write a note to `ops/daily-news/{yyyy-mm-dd}/{topic}.md`...
Do NOT: browse the workspace tree, read templates, scrape full articles...
```

Czego uczy: preskryptywne instrukcje (dokładne kroki + zakazy) dają bardziej przewidywalne rezultaty niż goal-oriented („zbierz newsy"). `Do NOT` lista = negatywny prompt zapobiegający eksploracji. Agent dostaje ścieżki, nie musi szukać.

### 6. mind.md — konstytucja bazy wiedzy

Delta: dokument definiujący pełną architekturę vaultu w jednym pliku — struktura katalogów, frontmatter defaults, access control per section, agent scenarios.

Czego uczy: jeden plik referencyjny z którego agent czerpie zasady organizacji wiedzy. Frontmatter defaults dziedziczone per section (np. `Craft/Knowledge` → write: `[adam, ellie, tony]`) — nie trzeba ich powtarzać w każdej notatce. Tagi opisują co, nie gdzie — folder determinuje lokalizację, tagi treść.
