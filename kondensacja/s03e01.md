# S03E01 — Obserwowanie i ewaluacja

## Observability — fundament przed evals

- Monitoring nie jest opcjonalny przy systemach agentowych — złożoność dynamicznego kontekstu, wielokrotnych narzędzi i rozgałęzień przekracza możliwości analizy samego kodu.
- Obserwowalność nie ogranicza się do logowania zdarzeń. Wymaga **grupowania i zagnieżdżania** zdarzeń z zachowaniem pełnego kontekstu: użytkownik, sesja, agent, prompt.
- Architektura: scentralizowana logika interakcji z LLM API + narzędzia, z jedną rurą monitorującą. Brak centralizacji = fragmentaryczne dane.

## Hierarchia obserwacji (Langfuse)

| Typ | Zakres | Kluczowe dane |
|---|---|---|
| Session | Wątek / czat | Id sesji, użytkownik |
| Trace | Pojedyncza interakcja usera | Input, output, metadane |
| Span | Czas trwania akcji | Durations, fazy przetwarzania |
| Generation | Wywołanie LLM | Model, prompt, tools, usage, output |
| Agent | Działanie agenta w interakcji | Task, wynik |
| Tool | Wywołanie narzędzia | Input/output narzędzia |
| Event | Zdarzenia aplikacji | Niezależne od LLM |

Zasada: każde zdarzenie niesie **kontekst** (kto, w jakiej sesji, który agent), nie tylko **fakt** (co się stało).

## Tracing — implementacja

- `AsyncLocalStorage` propaguje kontekst (agent, turn, tool index) przez call stack bez jawnych parametrów.
- Funkcje wrapperowe (`withTrace`, `withAgent`, `withTool`, `startGeneration`) opakowują logikę biznesową — gdy tracing wyłączony, wywołują fn bez overheadu.
- Adapter LLM dekorowany przez `withGenerationTracing` — loguje input (messages + tools), output (text + tool_calls), usage. Jeden punkt przechwytu, zero zmian w logice agenta.

## Wersjonowanie promptów

- Prompt systemowy = część kodu (git trackuje), ale to nie wystarcza. Potrzebna korelacja treści promptu ze statystykami i historią uruchomień.
- Jednostronna synchronizacja: kod → platforma. Hash SHA256 treści promptu determines czy pushować nową wersję. Stan lokalny w `.langfuse-prompt-state.json`.
- Wynik: każda generacja na platformie ma referencję do konkretnej wersji promptu (PromptRef: name + version).

## Debugowanie agentów ≠ debugowanie kodu

- Zmiana instrukcji naprawiająca jeden przypadek może psuć inne. Brak determinizmu = brak gwarancji regresji.
- Playground na platformie (Langfuse, Confident AI) pozwala odtworzyć interakcję z pełnym kontekstem (system prompt + messages + tools) i manipulować parametrami.
- Przydatna technika: poproszenie modelu o **uzasadnienie** wyboru narzędzi — często ujawnia przyczyny problemu.

## Eval — struktura i zasady

Eval = ustrukturyzowany test oceniający zachowanie modelu/agenta pod kątem metryk.

**Składniki:**
- **Dane wejściowe** (input) + **oczekiwane wyjście** (expected output)
- **Dataset** — syntetyczny + produkcyjny, iteracyjnie rozwijany
- **Evaluator** — funkcja `(input, output, expectedOutput) → score[]` — deterministyczna lub LLM-based

**Offline vs Online eval:**
- Offline: przed publikacją, w CI/CD, w dev env
- Online: w trakcie działania, automatyczna ocena interakcji

## Projektowanie datasetów testowych

| Zasada | Opis |
|---|---|
| Pokrycie | Wszystkie kategorie zachowań — pozytywne i negatywne |
| Różnorodność | Różne warunki, style zapytań, edge case'y |
| Balans | Równomierne rozłożenie scenariuszy — skrzywienie zaburza wyniki |

Anatomia promptu w kontekście eval: odpowiedzi narzędzi szybko stanowią większość kontekstu — uwzględniać je w testach.

## Kryteria oceny — deterministyczne vs model-based

**Deterministyczne:** `contains`, `is-json`, `equals`, regex, programistyczne asercje (JS/Python).
**Model-based:** `llm-rubric` (ocena wg kryteriów w języku naturalnym), `conversation-relevance`, `context-recall`.

**Obszary ewaluacji agentów:**
- Skuteczność wyboru narzędzi (czy agent sięga po właściwe)
- Skuteczność obsługi narzędzi (poprawność argumentów, liczba retry)
- Poprawność odpowiedzi (exact match, format, relevance)
- Wykrywanie naruszeń (input + output) — safety guardrail

## Implementacja eval — wzorzec

Proces: parse dataset → seed do Langfuse → `dataset.runExperiment({ task, evaluators, runEvaluators })`.

- **Task** = uruchomienie agenta z inputem, zwraca strukturę wynikową
- **Evaluator per item** = porównuje output z expected, zwraca tablicę score'ów (0–1)
- **RunEvaluator** = agreguje per-item score'e na poziomie całego eksperymentu

Ewaluator deterministyczny dekomponuje ocenę na niezależne wymiary (decision accuracy, required tools, forbidden tools, call count) i uśrednia.

## Koszty i pragmatyzm

- Ewaluacja kosztuje tokeny — eksperyment z N przypadkami = N wywołań agenta. Potwierdzenie przed uruchomieniem to dobra praktyka.
- Anomalię kosztów najłatwiej wyłapać przez monitoring: twarde limity + alerty na anomalię usage.
- Evals mogą być tymczasowe — np. test porównawczy modeli przed decyzją o downgrade.

## Przykłady kodu — progresja

### 1. Observability — hierarchia spanów z AsyncLocalStorage

Delta: Kontekst tracingowy (agentName, turnNumber, toolIndex) propagowany przez `AsyncLocalStorage.run()`. Zero jawnego przekazywania — wrapper `withAgent` / `withTool` same rejestrują span na platformie i udostępniają kontekst potomnym wywołaniom.

```ts
// context.ts — bezparametrowa propagacja kontekstu
export const withAgentContext = (name, id, fn) =>
  storage.run({ agentName: name, agentId: id, turnNumber: 0, toolIndex: 0 }, fn);

export const formatToolName = (tool) =>
  `${storage.getStore().agentName}/${tool}#${nextToolIndex()}`;
```

### 2. Decorator adaptera — przechwyt generacji

Delta: `withGenerationTracing` opakowuje `Adapter.complete()` — loguje messages + tools jako input, text + tool_calls jako output, usage per generation. Jeden dekorator, pełna widoczność bez modyfikacji logiki agenta.

```ts
// adapter.ts — decorator pattern
export const withGenerationTracing = (adapter: Adapter): Adapter => ({
  complete: async (params) => {
    if (!isTracingActive()) return adapter.complete(params);
    const gen = startGeneration({ model, input: buildGenerationInput(params, ...) });
    const result = await adapter.complete(params);
    result.ok ? gen.end({ output, usage }) : gen.error({ code, message });
    return result;
  },
});
```

### 3. Sync promptów — hash-gated push

Delta: SHA256 treści promptu determines czy wysłać nową wersję. Lokalny stan w JSON trzyma mapping name→{hash, version}. Wynik: PromptRef (name + version) dołączany do każdej generacji.

```ts
// prompts.ts — jednostronna synchronizacja
const contentHash = createHash('sha256').update(source.content).digest('hex');
if (cached?.contentHash === contentHash) { /* skip, use cached ref */ return; }
const version = await client.prompts.create({ name, prompt: content, labels: ['production'] });
state.prompts[name] = { contentHash, version };
```

### 4. Tool-use evaluator — dekompozycja na wymiary

Delta: Ewaluator nie daje jednego binarnego verdictu. Rozbija ocenę na 4 niezależne wymiary (decision, required, forbidden, call count), każdy 0 lub 1. Średnia = overall score. Pozwala precyzyjnie zidentyfikować typ błędu.

```ts
// tool-use.ts — wielowymiarowy evaluator
const decision = expected.shouldUseTools ? (count > 0 ? 1 : 0) : (count === 0 ? 1 : 0);
const required  = expected.requiredTools.every(n => unique.has(n)) ? 1 : 0;
const forbidden = expected.forbiddenTools.every(n => !unique.has(n)) ? 1 : 0;
const callCount = inRange(count, expected) ? 1 : 0;
return { overall: (decision + required + forbidden + callCount) / 4 };
```

### 5. Response-correctness — typowane kryteria oceny

Delta: Dataset definiuje `expect.type` — evaluator dispatchuje po typie. `exact_number` weryfikuje regexem, `contains_iso_timestamp` pattern-matchuje, `relevance` sprawdza keywords. Każdy typ = inna deterministyczna strategia oceny, zero LLM w ewaluatorze.

```ts
// response-correctness.ts — dispatch po typie oczekiwanego wyniku
if (expected.type === 'exact_number')  return scoreExactNumber(response, expected.value);
if (expected.type === 'contains_iso_timestamp') return scoreTimestamp(response);
if (expected.type === 'relevance')     return scoreRelevance(response, expected.topic);
```
