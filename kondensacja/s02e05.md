# S02E05 — Projektowanie agentów

## Architektura instrukcji systemowej

Instrukcja agenta to warstwy odrębnej odpowiedzialności, nie jeden monolityczny blok tekstu:

| Sekcja | Cel | Co zawiera |
|---|---|---|
| **Identity** | Persona — charakter, styl, zachowanie | Cechy, ton, skojarzenia słowne, "instynkty". Brak detali technicznych. |
| **Protocol** | Zasady działania w środowisku | Zarządzanie kontekstem, wspomnieniami, reakcje na błędy, prośby o pomoc. Może zawierać ścieżki plików, ale bez ścisłego wiązania z narzędziami. |
| **Voice** | Ton wypowiedzi | Przykłady wyrażeń, antywzorce, few-shot, dynamiczne dostosowanie do sytuacji. Rozbudowana — LLM szybko zapominają o stylu bez przypomnień. |
| **Tools** | Dostępne narzędzia i agenci | Niemal w pełni generowana dynamicznie. Opisy narzędzi opisują się same — nie dubluj ich w instrukcji. Wyjątek: zachowania specyficzne (np. nie mylić powiadomienia z komunikacją). |
| **Workspace** | Dynamiczny kontekst sesji | Observational memory, dane runtime. Zmienna per-request. |
| **CTA** | Zakończenie promptu | Jedno zdanie sygnalizujące koniec instrukcji — zapobiega "doklejaniu" przez model. |

**Kluczowa zasada:** obecność instrukcji nie gwarantuje jej przestrzegania. Projektuj tak, by model pozytywnie zaskoczył, nie by wykonał dokładnie to, co chcesz.

## Sterowanie zachowaniem przez skojarzenia

Słowa o zabarwieniu emocjonalnym ("instynkt", "wyczucie") w sekcji identity to nie ozdobnik — one realnie fokusują uwagę modelu na określonych wzorcach zachowania. To forma promptingu przez **skojarzenia i halucynację na swoją korzyść**, a nie przez bezpośrednie instrukcje.

Profil agenta działa jak "nie myśl o niebieskich motylach" — cechy wprofilu kierują uwagę modelu niezależnie od jego "inteligencji".

## Narzędzia — zasady przypisywania

- Nie ma sztywnej reguły liczby narzędzi. 10–15 to mit — złożone API (GitHub) może być jednym narzędziem CLI, prosty system może wymagać 3 osobnych funkcji.
- **Współdzielenie narzędzi** między agentami zmniejsza wymianę informacji, ale tworzy ryzyko: agent z ograniczonym dostępem może uznać brak wyniku za brak danych.
- **Progressive disclosure** — agent nie zna wszystkich narzędzi na start. Odkrywa je dynamicznie, oszczędzając okno kontekstowe.
- Pytaj: **czy system stanie się lepszy wraz z rozwojem modeli?** Jeśli nie — prawdopodobnie budujesz niewłaściwą rzecz.

## Ryzyko połączeń między narzędziami

Agent z dostępem do filesystemu i Jiry może obejść ograniczenia — utworzyć notatkę w sekcji nieobjętej restrykcjami. Każde połączenie między narzędziami to wektor problemów. Restrykcje mogą zmniejszyć użyteczność do zera — dlatego sandbox.

## Architektura sandbox z progresywnym odkrywaniem

Sandbox daje agentowi ogromną swobodę przy zachowaniu kontroli. Modelowany w przykładzie `02_05_sandbox`:

1. Agent startuje z **4 narzędziami metody**: `list_servers`, `list_tools`, `get_tool_schema`, `execute_code`
2. Odkrywa serwery MCP, wczytuje schematy TypeScript **na żądanie**
3. Generuje kod JS i uruchamia go w **QuickJS** — dane nie trafiają do kontekstu agenta, funkcjonują jako zmienne w sandboxie
4. Kod wywołuje narzędzia MCP synchronicznie — asyncified host functions wyglądają synchronicznie z perspektywy QuickJS

Zalety: agent łączy narzędzia elastycznie, operuje na dużych zbiorach danych bez konsumpcji tokenów, ograniczony dostęp do świata zewnętrznego.

## Organizacja wiedzy w systemie wieloagentowym

| Kategoria | Zakres | Udostępnianie |
|---|---|---|
| **Dokumenty sesji** | Załączniki + pliki z bieżącej interakcji | Tylko agenci w sesji |
| **Wiedza publiczna** | Pamięć długoterminowa, współdzielona | Agenci + użytkownicy |
| **Wiedza prywatna** | Kontekst użytkownika, jego dokumenty | Per-user |
| **Wiedza agentów** | Instrukcje, wspomnienia, obserwacje agentów | Per-agent |
| **Pamięć podręczna** | Wyniki wyszukiwań, treści stron | Tymczasowa, między sesjami |
| **Runtime** | Sesje, interakcje, harmonogram | Niewidoczna dla agentów |

Klasyfikacja danych jest dynamiczna — ta sama informacja (np. "nowy projekt") może być prywatna, publiczna, lub dotyczyć agenta, zależnie od kontekstu. Trudno oczekiwać poprawnej autoklasyfikacji. **Dlatego: proste struktury, minimalne reguły.**

## Observational Memory — wzorzec Observer/Reflector

Agent `02_05_agent` implementuje kompresję kontekstu wieloetapową:

```
┌─────────────────────────────────────────────────┐
│  System prompt + Observations (skompresowane)   │
│  ─────────────────────────────────────────────── │
│  Surowe wiadomości (ogon — tylko najnowsze)     │
└─────────────────────────────────────────────────┘
```

**Observer** — gdy nieobserwowane wiadomości przekroczą próg tokenów:
- Serializuje wiadomości z głowy (starsze) do tekstu
- LLM wyciąga strukturyzowane obserwacje (XML z tagami źródłowymi `[user]`, `[assistant]`, `[tool:name]`)
- Priorytetyzacja: 🔴 fakty użytkownika > 🟡 aktywna praca > 🟢 detale
- Ogon (30% budżetu) pozostaje surowy — zapewnia ciągłość

**Reflector** — gdy obserwacje rosną ponad próg:
- Kompresuje obserwacje przez LLM z rosnącymi poziomami agresywności
- Zachowuje priorytet: `[user]` nigdy nie usuwane, `[assistant]` kondensowane jako pierwsze
- Generacja (generation count) rośnie — śledzi ile razy pamięć była kompresowana

**Kalibracja** — system estymuje tokeny lokalnie i porównuje z rzeczywistym użyciem API. Kalibracja poprawia estymaty w czasie, zmniejszając odchylenia.

## Szablony agentów — frontmatter Markdown

Definicja agenta = plik `.agent.md` z frontmatterem:

```yaml
---
name: alice
model: gpt-5.4-mini
tools:
  - read_file
  - write_file
---
Treść instrukcji systemowej...
```

Parsowane przez `gray-matter` — model, narzędzia i prompt w jednym pliku. Umożliwia dynamiczne ładowanie i łatwą iterację.

## Progresja przykładów

### `02_05_agent` — kompresja kontekstu
Delta: Agent HTTP z mechaniką Observational Memory. Cykl observer → reflector utrzymuje stałe okno kontekstowe przy nieograniczonej długości rozmowy. Sesja in-memory, persystencja logów do `workspace/memory/`.

### `02_05_sandbox` — progresywne odkrywanie + wykonanie kodu
Delta: Agent startuje bez znajomości narzędzi biznesowych. Odkrywa je przez MCP, wczytuje schematy TypeScript, generuje kod JS i uruchamia w izolowanym QuickJS. Dane operacyjne nigdy nie trafiają do kontekstu LLM — są zmiennymi w sandboxie. Kluczowe: host functions są asyncified, więc z perspektywy kodu w sandboxie wywołania wyglądają synchronicznie.
