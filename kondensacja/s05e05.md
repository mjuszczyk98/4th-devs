# S05E05 — Nowa Rzeczywistość

## Zmiana paradygmatu w programowaniu

- Coraz więcej osób programuje we współpracy z AI, generując ponad 90% kodu. Role się odwróciły — **to my wspieramy AI w generowaniu kodu**, nadzorując kierunek, dostarczając kontekst, podejmując kluczowe decyzje i dbając o jakość.
- Jeszcze niedawno AI było jedynie wsparciem (IntelliSense → AI Completion). Dziś w dużej mierze to człowiek pełni rolę wsparcia dla AI.
- Ta zmiana nie jest jeszcze powszechna, ale adaptacja postępuje niezwykle szybko.
- Wszyscy **eksplorujemy**, nikt nie wie na pewno co nadchodzi, ale wiemy dużo o tym co dzieje się **teraz** — i to jest przestrzeń do działania.

## Nowe możliwości, nowe wyzwania, stare zasady

Budowanie systemów agentowych łączy nowe możliwości i wyzwania ze znanymi zasadami i procesami. Trudność polega na tym, że będziemy doświadczać **podważania tego, co uznajemy za słuszne**, a przekonania muszą ulegać dopasowaniu do zmieniającego się otoczenia.

Przykład: rozwijanie złożonego systemu wyłącznie na własne potrzeby — jeszcze niedawno kojarzyło się z długimi miesiącami pracy "po godzinach". Dziś pomysł na aplikację mobilną można zrealizować nawet w ciągu jednego popołudnia.

## Projekt "Wonderlands" — przypadek referencyjny

### Koncepcja

Agent, którego bazą wiedzy jest **cyfrowy ogród** — strona www generowana na podstawie systemu plików tekstowych połączonych katalogami, tagami i wikilinkami. W trakcie rozmowy można poprosić o zapisanie/wczytanie informacji. Agent korzysta z tej bazy przy posługiwaniu się narzędziami.

### Główne elementy

| Element | Opis |
|---|---|
| **API** | Interakcja z agentem — czat i zapytania przetwarzane w tle |
| **Czat** | Zaawansowany interfejs ze strumieniowaniem, blokami, prompt cache, referencjami plików, interakcjami między agentami |
| **Agent** | Logika oparta na współpracy między agentami, zdarzeniach, zarządzaniu zadaniami. Observational Memory |
| **Narzędzia** | Natywne (web search, delegate) + MCP w Function Calling i Code Mode |
| **Obrazy** | Generowanie przez OpenAI / Gemini / OpenRouter, edycja plików wgranych przez użytkownika |
| **Sandbox** | Dwa tryby: proces Node.js lub narzędzie [lo](https://github.com/just-js/lo), dostęp do FS przez [just-bash](https://github.com/vercel-labs/just-bash). Alternatywa: Daytona |
| **Przeglądarka** | Dostęp przez lokalnie uruchomiony [kernel.sh](https://kernel.sh/). Na produkcję: browserbase |
| **Garden** | Strona generowana z FS użytkownika. Pliki z `visibility: private` zablokowane hasłem |

### Zespół agentów

Całość można zorganizować w zespół powiązanych agentów z dostępem do wybranych obszarów systemu plików i narzędzi. Można skonfigurować system adresujący różne obszary: nauka, budowanie nawyków, hobby, projekty zawodowe. Baza wiedzy może zostać opublikowana jako strona www.

### Działanie w tle

System może działać **w tle**, otrzymując zapytania z zewnętrznych źródeł. Uruchomione zadania wyświetlają się w historii konwersacji — można do nich wrócić lub zareagować.

### Wartość domyślna

Wszystkie możliwości nie dają realnej wartości "out of the box" — domyślnie system będzie jedynie imponującym chatbotem. Trzeba zaprojektować zespół agentów, ich narzędzia i procedury. **Najlepiej zacząć od czegoś prostego.**

### Architektura

Podział client/server (jak S05E04), ale jako jedna aplikacja ze współdzielonymi kontraktami opisującymi komunikację opartą na zdarzeniach. Moduły odseparowane w głównej logice, ale połączone tak lekko jak to możliwe.

### Kluczowe detale mechaniczne

- **Kontekst:** dodatkowe informacje o narzędziach, agentach, metadanych "cyfrowego ogrodu", bieżąca data. Przestrzeń na rozbudowanie o dane z otoczenia.
- **Code Mode:** gdy sandbox aktywny, agent posługuje się narzędziami przez pisanie i wykonywanie kodu — definicje narzędzi **nie są** wczytywane do kontekstu. Wyspecjalizowany agent z małą liczbą narzędzi prawdopodobnie powinien mieć je w kontekście od razu.
- **Referencje:** agenci przekazują pliki między sobą i przy wywołaniu narzędzi stosują referencje. Wykorzystano to też w generowaniu/edycji obrazów oraz połączono z sandboxem i systemem plików.
- **Dodatkowe zdarzenia:** ukierunkowane na obsługę błędów i informowanie o statusie wątków w tle.

### Wyjątki produkcyjne

- **STDIO:** rejestracja serwerów STDIO nie może być dostępna w produkcyjnej aplikacji webowej (chyba że aplikacja desktopowa z bezpośrednim kontaktem z komputerem użytkownika)
- **Sandbox:** na produkcji można mieć więcej niż jeden — np. just-bash do nawigacji w FS, lo do wykonywania kodu
- **Uprawnienia:** przed publikacją dodatkowa weryfikacja — mimo zarządzania uprawnieniami, sandboxa i wymogu potwierdzeń

## Możliwe zastosowania

Same narzędzia podłączone do agentów nie wnoszą dużo bez konkretnego pomysłu. Użyteczność rośnie, gdy **spersonalizujemy** sposób użycia poprzez opisy procesów, procedury, a niekiedy skrypty.

Jeden schemat — wiele obszarów: nauka, treningi, praca, hobby, research, klasyfikacja zgłoszeń, wiadomości, archiwizacja danych (często niemal całkowicie w tle).

### Przykładowy schemat — Daily Ops

Agenci wywoływani cyklicznie z wiadomościami kierującymi do plików opisujących element procesu:

1. **Agent Calendar** — zgromadzić wydarzenia z nadchodzącego dnia i kluczowe z kolejnych dni
2. **Agent Tasks** — zebrać nieukończone wpisy z ostatnich dni oraz aktywne/nadchodzące
3. **Agent Mail** — zgromadzić kluczowe powiadomienia/wiadomości wymagające uwagi
4. **Agent Newsfeed** — zapisać informacje spełniające kryteria ważności, zestawić z wiadomościami z ostatnich dni
5. Wszystko trafia do jednego folderu → agent opracowuje transkrypt + generuje wersję audio → trafia na urządzenie mobilne

### Uruchamianie sesji w tle

curl -X POST http://127.0.0.1:3000/v1/sessions/bootstrap \
    -H "Authorization: Bearer sk_local_API_TOKEN" \

Zapytanie realizowane w tle, wyświetlane na liście konwersacji. Gdy akcja wymaga interwencji — pojawia się w "Activity Bar".

### Organizacja czasowa

Zadania w tle mogą być pogrupowane czasowo, bez budowania zależności. Jeśli newsletter o 5:01, agenci zbierający informacje powinni być wywołani przynajmniej godzinę wcześniej i **równolegle**.

**Taki struktury nie buduje się "w jeden wieczór"** — warto zacząć od czegoś prostego. Jeśli po uruchomieniu agenci zaczną realizować **jeden powtarzalny proces** — to już sukces.

## Współpraca z agentem — praktyczne wskazówki

### Typowe wygórowane oczekiwania

- Instrukcje zbyt ogólne — zakładamy, że agent "domyśli się"
- Zakładamy bezbłędne działanie bez doprecyzowania wymagań
- Oczekujemy jakości bezbłędnej, nawet jej nie doprecyzowując
- Chcemy szybkości, ale nie inwestujemy w opracowanie i weryfikację procesów
- Oczekujemy deterministycznego wykonania promptów "linia po linii"

### Praktyczne porady

- **Wskaż ścieżki:** używaj "#" do wywołania palety z konkretną ścieżką
- **Doprecyzuj zadania automatyczne:** wskaż konkretne pliki, nazwy narzędzi, foldery
- **Krótkie interakcje:** nawet przy logice kompresji, interakcje powinny być **tak krótkie jak to możliwe** — przekłada się na skuteczność i koszty (praca z własnymi agentami = płatność za zużyte tokeny)
- **Proste opisy workflow/skilli:** pamiętaj o ograniczonej zdolności modeli do utrzymania uwagi. Notatki mogą się łączyć — agent może je stopniowo odkrywać
- **Dopracowanie instrukcji to jednorazowy wysiłek**, który z dużym prawdopodobieństwem zwróci się wielokrotnie

### Nawyk pracy z agentem

Sama jakość interakcji to nie wszystko — istotny jest **nawyk pracy** z treściami, które system generuje. Łatwo skonfigurować codzienny newsletter, ale po co, jeśli nie będziemy go czytać? Wyzwaniem nie jest wygenerowanie serwerów MCP, ale to, że skorzystanie z nich tylko raz nie wniesie wartości.

### Budowanie nawyków

Najlepiej sprawdza się **dopasowanie otoczenia** i łączenie nowych aktywności z tym, co już robimy:

- Jeśli spędzamy czas na Discordzie — tam powinny trafiać powiadomienia
- Jeśli dużo na telefonie — prosta aplikacja mobilna lub skrót do czatu na www
- Jeśli rano trenujemy — w chwili wychodzenia z domu wczytujemy nagranie audio z przeglądem newsów

## Przykłady kodu

### 1. Monorepo ze współdzielonymi kontraktami

Delta: projekt jest monorepo (apps/client + apps/server + packages/contracts + packages/sandbox-runtime-lo). Kontrakty API współdzielone przez Zod schemas i branded types.

```
wonderlands/
├── apps/
│   ├── client/          # Svelte 5 + Vite frontend
│   └── server/          # Hono + TypeScript backend
├── packages/
│   ├── contracts/       # Shared API contract types (Zod schemas)
│   └── sandbox-runtime-lo/  # Custom sandbox runtime (just-bash based)
├── scripts/             # Dev scripts
└── setup/               # Setup wizard
```

Czego uczy: współdzielone kontrakty (`packages/contracts`) eliminują dryfowanie typów między front-endem a back-endem. Branded IDs zadeklarowane raz, używane wszędzie.

### 2. Agenci jako Markdown z pełnym DSL konfiguracyjnym

Delta: znacznie bardziej zaawansowane niż proste `.agent.md` — pełny DSL z wersjonowaniem rewizji, checksumami SHA-256 i sandbox/kernel/memory/garden policy.

```yaml
# agent-markdown.ts — walidowany przez Zod (schema: agent/v1)
slug: alice
kind: primary
visibility: account_private
model:
  provider: openai
  alias: gpt-5.4
  reasoningEffort: medium
sandbox:
  mode: enabled
  runtime: lo
  networkAccess: false
  mountGarden: readWrite
kernel:
  enabled: true
  viewport: "1280x720"
memory:
  observationScope: thread_shared
  reflectionSourceTokenThreshold: auto
garden:
  preferredSlugs: [my-garden]
subagents:
  - alias: researcher
    mode: async_join
    slug: researcher
tools:
  mcpMode: code    # direct | code
  nativeTools: [web_search, delegate_to_agent]
```

Czego uczy: agent nie jest tylko promptem — to pełna konfiguracja runtime z bezpieczeństwem, pamięcią, narzędziami i relacjami. Każda zmiana tworzy nową rewizję.

### 3. Digital Garden — statyczny generator stron wbudowany w serwer

Delta: całkowicie nowy subsystem. Markdown → HTML z sidebar, TOC, listing pages, pagination, Pagefind search i protected access.

**Build pipeline** (`build-site.ts`, 1438 linii):
1. **Collect** — skanuj `.md` pliki
2. **Parse** — frontmatter (`gray-matter`) → exposure: public/protected/private/hidden
3. **Rewrite links** — `[[slug]]` → relative hrefs, wykrywanie broken links
4. **Render** — HTML z inline CSS, sidebar, TOC, listing, pagination
5. **Search** — Pagefind indeksy (osobne public + protected)

**Auto-build worker** — fingerprintuje source (SHA-256 hash wszystkich plików), debounce po ostatniej zmianie, automatycznie publikuje po udanym build.

**Agent integration** — agenty dostają garden context w system prompt: gdzie są pliki, jak używać sandbox, jak manipulować zawartością garden.

### 4. Context Compaction z Boundary Integrity

Delta: nowy mechanizm kompakcji kontekstu zachowujący integralność par tool call/response i pending waits.

```typescript
// context-compaction.ts — adjustTailStartForBoundaries
// Iteracyjnie dostosowuje granicę tail aby zachować:
// 1. Tool call + response pairs razem
// 2. Pending wait call IDs w tail
```

Czego uczy: kompakcja nie może ciąć w środku interakcji narzędzia — model straciłby kontekst trwającej operacji.

### 5. Sandbox Writeback jako Explicit Consent Model

Delta: sandbox edits są domyślnie ephemeral — muszą być jawnie zatwierdzone przez `commit_sandbox_writeback`. Dwa tryby wykonania: `bash` (just-bash inline) i `script` (JavaScript ES module).

Czego uczy: warstwa bezpieczeństwa między "agent wykonuje kod" a "agent modyfikuje pliki" — sandbox może działać, ale zmiany nie trwają bez wyraźnej zgody.

### 6. Polling Worker — generyczny wzorzec z wake support

Delta: zamiast skomplikowanych message queues, proste polling workers z adaptacyjnym opóźnieniem.

```typescript
createPollingWorker<T>({
  computeNextDelay: ({ result, wakeRequested }) =>
    wakeRequested || result ? 0 : pollIntervalMs,
  runOnce: async () => { ... },
  supportsWake: true,
})
```

Czego uczy: jeśli była praca → natychmiast sprawdź ponownie (delay=0). Jeśli nie → czekaj `pollIntervalMs`. `wake()` przerywa oczekiwanie. Wystarczające dla single-server SQLite deployment.

### 7. Multi-Lane Event Outbox

Delta: każdy event dispatchowany do 4 niezależnych lanes z osobną retry/quarantine logiką.

- **realtime** — SSE do klientów
- **projection** — read model rebuild
- **background** — side effects (garden build, email)
- **observability** — Langfuse tracing

Quarantine logic — failed observability events po max retries idą do quarantine. Payload sidecars — duże payloady kompresowane i przechowywane osobno.

### 8. Delegation Handoff Envelope

Delta: strukturalny kontekst przekazania między agentami — child dostaje pełny envelope z informacjami o parent, target, plikach.

```typescript
interface DelegationHandoffEnvelope {
  kind: 'delegation_handoff'
  parent: { agentId, agentRevisionId, runId, sourceCallId }
  target: { agentId, agentName, agentRevisionId, alias, delegationMode, inputFileIds, runId }
  version: 1
}
```

Czego uczy: delegacja to nie tylko przekazanie zadania — to pełny kontekst relacji, wersji agenta i plików wejściowych.

## Uzupełnienia

### Agent Markdown — pełny cykl życia

Create z Markdown → Parse z walidacją Zod → Revision z checksum SHA-256 → Activate (activeRevisionId) → Delegation przez `delegate_to_agent`. Każda zmiana agenta tworzy nową rewizję — pozwala śledzić ewolucję konfiguracji.

### Kernel Browser Sandbox

Docker-based Chromium z CDP i Playwright API. Local mode z `docker-compose.kernel.yml`, Cloud mode z `api.kernel.sh`. Policy-driven: viewport, recording, output types, network access. Session artifacts: zrzuty ekranu, PDFy, recordingi.

### MCP Code Mode

Gdy `mcpMode: code`, sandbox może wywoływać narzędzia MCP z wewnątrz sandbox execution — agent pisze skrypt który używa narzędzi MCP. Definicje narzędzi **nie są** wczytywane do kontekstu modelu.

### Full-text search z Pagefind

Garden generuje osobne indeksy Pagefind dla publicznych i chronionych stron. Strony z `visibility: private` nie są indeksowane.

### Garden Protected Access

Strony chronione hasłem z sesjami TTL. `_garden.yml` konfiguruje sekcje, nawigację, listing i publishing roots.

### Realtime Event Relay — In-Memory Pub/Sub

In-memory subscriptions z SSE fan-out. Klient na połączeniu dostaje hydratację z durable messages + run graph. Klucz: browser jest tylko observer plus control surface — server owns progress once the turn has been accepted. Klient nie musi być online aby run się zakończył.

### 60+ event types

Pełny lifecycle: runy (created, started, waiting, completed, failed, cancelled), joby, delegacje, memory (observed, reflected), garden (build started, completed), workspace, sandbox execution, tool executions, file operations.

### Readiness Engine — heterogenne decyzje

Zamiast hardcoded DAG walker, readiness engine podejmuje heterogenne decyzje w jednym unified loop: resume waits, requeue stale runs, start sandbox executions, queue pending jobs. Priorytetyzacja deterministyczna.

### Stale run reconciliation na startupie

Na startupie `reconcileOnStartup()` sprawdza uruchomione runy z `claim_expired` i requeue'uje je — recovery po restarcie serwera, nie tylko po crashu worker'a.
