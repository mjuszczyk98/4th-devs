# S01E05 — Zarządzanie limitami modeli w środowisku produkcyjnym

## Kontrola nad akcjami agenta

- LLM sterujący logiką aplikacji **będzie popełniał błędy**. System musi umożliwiać samodzielną naprawę (error recovery) lub wymagać zatwierdzenia człowieka **przed** lub **po** akcji.
- Potwierdzenia akcji muszą być **deterministyczne** (kod, nie LLM) i zawierać **pełne detale** (np. cały szkic emaila, nie sam adres).
- Mechanizm zaufanych narzędzi ("trust") redukuje uciążliwość potwierdzeń. Krytyczne zasady:
  - Identyfikacja narzędzia: `serwer__akcja` (np. `resend__send`).
  - Narzędzie **musi zostać odznaczone jako zaufane** przy jakiejkolwiek zmianie jego nazwy, opisu lub schematu — szczególnie przy serwerach MCP, które mogą się zmienić bez wiedzy użytkownika.
- Jeśli przypadkowe przecieki danych są niedopuszczalne, a potwierdzenie użytkownika nie wystarcza — **LLM nie powinien tam być wdrożony**.

## Zarządzanie kontekstem i tokenami

- Limit okna kontekstu = input + output. GPT-5.2 (400k okno, 128k max output) → realnie 272k na input.
- Estymacja tokenów: **chars / 4** + bufor ~20%. Po zapytaniu — korekta na podstawie `usage` z odpowiedzi API.
- Kompresję/przycinanie kontekstu uruchamiać **wcześnie**, już przy ~30% zużycia limitu.
- Użytkownicy nie znają limitów LLM. Trzeba zakładać: PDF-y na setki stron, setki aktywnych narzędzi MCP, oczekiwanie nieskończonych rozmów.

## Architektura zdarzeniowa (event-driven)

- Pętla agenta emituje zdarzenia: `agent.started`, `turn.started/completed`, `tool.called/completed/failed`, `agent.waiting/resumed/completed/failed/cancelled`.
- Każde zdarzenie niesie **EventContext** z `traceId`, `sessionId`, `agentId`, `rootAgentId`, `depth` — umożliwia korelację w systemach wieloagentowych.
- Subskrypcja zdarzeń otwiera przestrzeń na: monitorowanie (Langfuse), moderację, kompresję kontekstu, blokady.
- Zdarzenia pozwalają na **wznowienie** agenta po dostarczeniu wyniku (`deliverResult`) — agent przechodzi w stan `waiting`, a po otrzymaniu danych wraca do `running`.

## Abstrakcja nad providerami

- Format `provider:model` (np. `openai:gpt-4.1`, `gemini:gemini-3-flash-preview`) rozdziela wybór modelu od implementacji.
- Adapter per provider tłumaczy wspólny interfejs (wzorowany na Responses API) na natywne API danego dostawcy.
- OpenRouter wystarczy jako uproszczenie, ale **nie wspiera wszystkich funkcjonalności** — dedykowane połączenia i tak będą potrzebne.
- Providerzy coraz częściej wprowadzają API-specyficzne funkcje blokujące pełną translację. W systemach wieloagentowych każdy agent może używać innego providera.

## Typy narzędzi i ich wykonanie

- **Sync tools** — wykonywane natychmiast w pętli agenta (np. kalkulator).
- **MCP tools** — identyfikowane po prefiksie `serwer__narzędzie`, wykonywane przez klienta MCP (stdio lub streamable HTTP).
- **Agent tools (delegation)** — spawnują agenta-dziecko z guarding głębokości (`MAX_AGENT_DEPTH`). Wynik dziecka automatycznie propaguje się do rodzica.
- **Deferred tools** — brak zarejestrowanego handlera → agent przechodzi w stan `waiting`, czeka na dostarczenie wyniku przez API (`POST /deliver`).

## Halucynacje i niejawne ograniczenia

- **Gwarancja struktury ≠ gwarancja wartości.** Poprawny JSON z błędnymi danymi nie zostanie wykryty przez system.
- Modele Flash potrafią **w pełni halucynować** treść strony www na podstawie samego URL, podczas gdy mocniejsze modele przyznają się do braku dostępu.
- Model może "domyślić się" brakujących parametrów (np. email na podstawie nazwiska) zamiast poprosić o doprecyzowanie.
- Redukcja ryzyka: informowanie modelu o jego ograniczeniach, instrukcje zachowania przy braku danych, zmniejszenie złożoności zadania, zmniejszenie objętości kontekstu.

## Wydajność i UX

- **Heartbeat** — informowanie o postępie zmienia postrzeganą wydajność, nawet jeśli realna się nie poprawia.
- **Wielowątkowość** — długie zadania wymagają: kolejki wiadomości, możliwości otwarcia nowego wątku, oddzielenia stanu UI od back-endu.
- **Przetwarzanie w tle** — użytkownik może zamknąć aplikację; wyniki muszą być persystowane.
- **Wznawianie** — agent może czekać na wiele rzeczy jednocześnie; wznowienie wymaga dostarczenia wyników do wszystkich oczekujących `callId`.
- **Zasada:** "Czy AI jest tutaj niezbędne?" — jeśli zadanie da się zrealizować kodem, zrób to kodem.

## Koszty i limity API

- LRM (Large Reasoning Models) — niższa cena/token, ale **znacznie więcej tokenów**. Tańszy model może być droższy w użyciu, bo wymaga więcej kroków (GPT-4.1-mini vs GPT-4.1-nano).
- Na jedno zdarzenie użytkownika może przypadać 1:50 zapytań do AI.
- Nagłówki odpowiedzi HTTP zawierają informacje o limitach RPM/TPM — programistycznie reaguj na zbliżający się limit.
- OpenRouter pozwala na programistyczne zarządzanie kluczami użytkowników z indywidualnymi limitami wydatków.

## Decyzje architektoniczne

- **Brak frameworków AI** — dynamiczny rozwój API i technik sprawia, że frameworki (Langchain, CrewAI) szybko stają się obciążeniem.
- **Niezależność** — ograniczenie natywnych funkcjonalności API i platform z utrudnionym eksportem danych. Rozwiązania zmieniają się szybciej niż w klasycznych aplikacjach.
- Agenty definiowane jako **pliki markdown** (frontmatter: model, narzędzia; body: system prompt) — czytane z dysku przy każdym requeście, bez restartu.
- Autentykacja: klucze API hashowane (SHA-256), przypisane do użytkowników w bazie.
- Baza: SQLite (Drizzle ORM) z tabelami `users`, `sessions`, `agents`, `items` — sesja persystuje historię rozmowy, agent — stan wykonania.

## Moderacja treści

- OpenAI Moderation API — nie tylko klasyfikacja, ale też zgodność z polityką. **Brak moderacji może prowadzić do zablokowania konta organizacji.**
- Moderacja może błędnie blokować legalne akcje (np. mock interfejsu Messenger w reklamie).
- Zakres działania systemu weryfikuje się przez prompt zwracający ustrukturyzowaną odpowiedź, a potem **programistycznie** decyduje o dalszym przebiegu — nie zostawiając decyzji modelowi.

## Przykłady kodu — progresja

### 1. Whitelist + confirmation (01_05_confirmation)

**Czego uczy:** Separacja walidacji od LLM. Agent nie decyduje kto może dostać email — robi to deterministyczny kod.

```
TOOLS_REQUIRING_CONFIRMATION = new Set(["send_email"]);

const isEmailAllowed = (email, whitelist) => {
  const domain = email.split("@")[1];
  return whitelist.some(p => p.startsWith("@") ? domain === p.slice(1) : email === p);
};
```

Agent wywołuje `send_email` → system przechwytuje → sprawdza whitelist → wyświetla UI potwierdzenia z pełnymi detalami → deterministyczna akcja (Y/T/N).

### 2. Trusted tools — delta: redukcja frakcji

**Czego uczy:** Pamięć zaufania narzędzi w sesji z możliwością resetu.

```
const createConfirmationHandler = (rl, trustedTools) => async (toolName, args) => {
  if (trustedTools.has(toolName)) return true;  // auto-approve
  // ... show confirmation UI
  if (choice === "t") trustedTools.add(toolName);  // remember trust
};
```

Kluczowe: `trustedTools.clear()` przy "clear" i "untrust" — zaufanie jest sesyjne, nie permanentne.

### 3. Pętla agenta ze zdarzeniami i delegacją (01_05_agent)

**Czego uczy:** Produkcyjna pętla agenta z pełnym zarządzaniem stanem, przerwaniem, wznowieniem i wielopoziomową delegacją.

```
while (agent.status === 'running' && agent.turnCount < maxTurns) {
  if (signal?.aborted) { agent = cancelAgent(agent); return; }
  // prepareTurnInput: load items → prune if needed → summarize dropped → map to provider format
  // executeTurn: provider.generate() → handleTurnResponse()
  // handleTurnResponse: store output → execute sync/MCP/agent tools → collect waitingFor
  if (waitingFor.length > 0) return { status: 'waiting', waitingFor };
}
```

Delta względem prostego agenta: **kontekst jest persystowany w bazie**, agenty mają stany (`pending` → `running` → `waiting` → `completed`/`failed`/`cancelled`), a delegacja agent-dziecko z depth-guardingiem automatycznie propaguje wynik do rodzica.

### 4. Pruning kontekstu — delta: zarządzanie oknem

**Czego uczy:** Kiedy i jak ucinać kontekst.

```
if (needsPruning(items, task, contextWindow, threshold)) {
  const pruneResult = pruneConversation(items, task, contextWindow, pruning);
  if (pruning.enableSummarization) {
    summary = await generateSummary(provider, model, pruneResult.droppedItems, session.summary);
    session = { ...session, summary };
  }
  prunedItems = pruneResult.items;
}
if (session.summary) input.unshift({ role: 'system', content: `[Context Summary]\n${summary}` });
```

Dropped items → generowanie podsumowania przez LLM → podsumowanie ląduje jako wiadomość systemowa na początku inputu. Sesja accumuluje summary kaskadowo.
