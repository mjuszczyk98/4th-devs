# S01E04 — Multimodalność i załączniki w agentach AI

## Problem załączników w API

LLM odbiera obrazy/audio jako Base64 lub URL, ale **nie widzi adresu URL** — nie może się do niego odwołać w narzędziach ani przekazać innym agentom. Dotyczy to wszystkich typów załączników.

**Rozwiązanie:** w wiadomości użytkownika, oprócz treści i pliku, umieścić dodatkowy element z odnośnikiem do pliku (np. tag `<media>`), który agent może przekazać do narzędzi. Instrukcja systemowa powinna wyjaśniać, jak agent ma się do plików odwoływać. Programistycznie odnośnik jest zamieniany na Base64 lub publiczny URL.

## Workflow vs Agent

| Kryterium | Workflow | Agent |
|---|---|---|
| Klasyfikacja | Prosta, statyczna | Dynamiczna, niepewna |
| Dane | Znane z góry, stałe | Zmienne, rozbudowane |
| Instrukcja | Sztywny proces krok po kroku | Cel + limity + uniwersalne zasady |
| Skuteczność | Wysoka dla przewidywalnych zadań | <100%, ale elastyczna |

Agent nie może mieć instrukcji zależnych od **zestawu danych** — może mieć zależne od **klasy problemów**. Znaczną część logiki przenosimy na model, dając przestrzeń do działania. Koszt: ryzyko błędów, które wymaga weryfikacji przez człowieka.

## Vision — narzędzie analityczne w pętli agenta

Agent nie widzi obrazów, które sam generuje. Dlatego wyposaża się go w **narzędzie do analizy obrazu** (vision) — oddzielne narzędzie natywne, które przyjmuje `imageBase64 + mimeType + question` i zwraca tekstową odpowiedź modelu wizyjnego.

Kluczowy wzorzec z przykładów: **pętla generuj→analizuj→popraw**. Agent generuje obraz, wywołuje `analyze_image` z opisem aspektów do sprawdzenia (prompt adherence, artefakty, anatomia, text rendering, spójność stylu, kompozycja), otrzymuje werdykt ACCEPT/RETRY i ewentualnie ponawia próbę.

## Szablony JSON jako prompty do generowania obrazów

Strukturyzowanie promptu jako obiektu JSON daje:
- **precyzyjną edycję** jednej sekcji (np. `subject`) bez przepisywania całości
- **kopiowanie szablonu** (`template.json`) do nowego pliku i modyfikacja tylko potrzebnego fragmentu
- **referencje** — przekazanie nazwy pliku szablonu zamiast pełnego promptu

```
Klonuj: template.json → prompts/scene_{timestamp}.json
Edytuj: zmień tylko sekcję "subject"
Generuj: przekaż pełny JSON jako prompt do create_image
```

Agent może też generować szablon JSON na podstawie obrazów referencyjnych, "ucząc się" stylu.

## Obrazy referencyjne — sterowanie kompozycją i pozą

Do narzędzia generującego obraz można przekazać **obrazy referencyjne** jako tablica `reference_images`. Przydatne do:
- zachowania pozy postaci (np. referencyjna sylwetka chodzącej postaci)
- in-painting / out-painting
- spójności postaci między scenami

W przykładzie `image_guidance` agent łączy JSON prompt z referencyjną grafikią pozy — JSON określa styl i scenę, referencja determinuje kompozycję.

## Generowanie dokumentów (HTML → PDF)

Agent z dostępem do systemu plików + generowania obrazów + konwersji HTML→PDF (Puppeteer) potrafi:
1. Czytać `template.html` i `style-guide.md`
2. Klonować szablon HTML i modyfikować `<body>`
3. Generować obrazy i osadzać je w HTML (ścieżki absolutne)
4. Konwertować gotowy HTML do PDF

Kluczowe: szablon to **master reference** — nigdy nie edytowany bezpośrednio, zawsze klonowany.

## Audio — Gemini jako backend multimodalny

Trzy operacje audio przez Gemini (`gemini-2.5-flash`):
- **Transkrypcja** z diaryzacją, timestampami, detekcją emocji, tłumaczeniem
- **Analiza** (muzyka/mowa/dźwięki/ogólna) z ustrukturyzowanym wyjściem JSON (response_schema)
- **TTS** (`gemini-2.5-flash-preview-tts`) — single i multi-speaker (max 2), kontrola stylu przez naturalny język w tekście

Pliki >20MB wymagają uploadu przez resumable upload API Gemini (`/upload/v1beta/files`). YouTube URL-e obsługiwane bezpośrednio jako `file_uri`.

## Wideo — analiza i generowanie

**Analiza wideo** (Gemini): bezpośrednie przesyłanie pliku wideo lub YouTube URL, bez konieczności podziału na klatki. Obsługa `video_metadata` (fps, start_offset, end_offset). Cztery tryby: analiza, transkrypcja, ekstrakcja (sceny/keyframe/obiekty/tekst), custom query.

**Generowanie wideo** (Kling przez Replicate): text-to-video i image-to-video. Kluczowa technika — **klatka startowa + końcowa** dla maksymalnej kontroli:
1. Wygeneruj klatkę startową z JSON prompt
2. Wygeneruj klatkę końcową z referencją do startowej (spójność postaci)
3. Wyślij obie do `image_to_video` z promptem opisującym ruch

Model Kling ograniczony do 10s, ale **ostatnia klatka segmentu = pierwsza klatka następnego**, co pozwala łączyć segmenty.

## Architektura narzędzi — wzorzec z przykładów

Wszystkie 8 przykładów współdzielą tę samą architekturę:

```
agent.js  → pętla: chat → tool calls → results (MAX_STEPS)
api.js    → Responses API client
config.js → model, instrukcje systemowe, konfiguracja backendów
mcp/      → klient MCP (system plików przez Files MCP)
native/   → narzędzia natywne (vision, generowanie obrazów, audio, wideo)
helpers/  → logger, stats, shutdown
repl.js   → interaktywna pętla REPL (z historią konwersacji)
```

**Narzędzia natywne** (definiowane jako OpenAI function format) = wszystko, czego MCP nie zapewnia: vision, generowanie obrazów, audio, wideo, HTML→PDF. Rozróżnienie: `isNativeTool(name)` → natywne, reszta → MCP.

Agent łączy narzędzia MCP (dostęp do systemu plików) z narzędziami natywnymi (multimodalność) w jednej pętli. Instrukcja systemowa określa **kiedy** i **jak** ich używać, nie zaś sztywny proces.

## Halucynacje wizualne

Najnowsze modele generują obrazy z 95% poprawnych detali i 5% subtelnych błędów (np. błędny napis na ostatniej klatce komiksu). Halucynacje dotyczą nie tylko tekstu — też obrazów. Wymaga to szczególnej uwagi przy ocenie wygenerowanych treści.

## Uzupełnienia

### Workflow może być narzędziem agenta

Workflow i agent nie są decyzją typu "albo-albo". Workflow może stać się jednym z narzędzi w zestawie agenta. Gdy agent dysponuje wieloma stylami generowania i musi wybrać odpowiedni na podstawie kontekstu — logika agenta jest uzasadniona. Gdy proces jest prosty i przewidywalny — wystarczy workflow. Ale można też mieć agenta, który deleguje do workflow jako jednego ze swoich narzędzi.

### Problem załączników nie ma standardu branżowego

Żadne z dostępnych API nie adresuje problemu przekazywania załączników między agentami/narzędziami wprost. Nie istnieją też publiczne materiały o dobrych praktykach w tym obszarze. Podejście z tagiem `<media>` jest autorską konwencją — warto mieć świadomość, że jest to otwarty problem projektowy.

### Style-guide.md jako konfiguracja runtime'owa

Agent nie ma zasad stylu zakodowanych w instrukcji systemowej — zamiast tego czyta `style-guide.md` z systemu plików przed pierwszą akcją. To oddziela reguły stylu od kodu agenta i pozwala modyfikować jego zachowanie bez zmian w instrukcjach. Ten sam wzorzec dotyczy szablonów HTML (`template.html`) i JSON (`template.json`). Użytkownik może dostosować output agenta edytując pliki markdown/HTML/JSON, bez dotykania kodu.

### Generowanie obrazów przez kod

Agenci mogą tworzyć grafiki nie tylko przez modele image-generation, ale też **pisząc kod** — np. generując wykresy, diagramy, infografiki. Obraz jest wtedy wynikiem działania narzędzia (execute code), a nie modelu wizyjnego. Ten sam problem braku widoczności outputu dotyczy obu ścieżek — agent nie widzi wyniku, chyba że wyposażymy go w narzędzie vision.

### Asymetria PDF: generowanie vs przetwarzanie

Generowanie PDF przez HTML→PDF (Puppeteer) jest relatywnie proste. Natomiast **przetwarzanie istniejących PDF-ów** (czytanie, nawigacja, ekstrakcja) to poważny problem biznesowy. Providerzy udostępniają specjalne API (np. Claude PDF support), a narzędzia takie jak LlamaIndex adresują ten problem. To ważna asymetria: generowanie dokumentów jest znacznie łatwiejsze niż ich późniejsze rozumienie.

### UX odpowiedzi audio

Gdy agent odpowiada w formie audio, należy unikać: dyktowania URL-i, tabel, zaawansowanego formatowania tekstu. Treść odpowiedzi musi być dostosowana do medium — dźwięk nie przenosi formatowania wizualnego. To dotyczy zarówno TTS, jak i interfejsów głosowych w czasie rzeczywistym.

### Filozofia instrukcji agenta klasyfikującego

Przykład `image_recognition` demonstruje wzorzec instrukcji opartej na **regułach rozumowania**, nie na krokach procesu. Instrukcja definiuje: jak zbierać dowody (tylko obserwowalne cechy), jak dopasowywać (profile = wymagania minimalne, nie wyczerpujące), jak radzić sobie z niejednoznacznością (brak dopasowania → unclassified), jak obsługiwać kompozyty (oceniać każdy obiekt osobno). To jest realizacja zasady "instrukcja zależna od klasy problemów, nie od zestawu danych".
