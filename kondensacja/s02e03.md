# S02E03 — Dokumenty i pamięć długoterminowa jako narzędzia

## Observational Memory — kompresja zamiast wyszukiwania

- Kluczowy problem RAG: agent trafia na dokument, który wygląda na kompletny, ale w rzeczywistości wymaga pogłębienia w powiązanych źródłach. Prompt nie rozwiąże tego — pełny kontekst może wymagać dziesiątek dokumentów.
- Observational Memory (Mastra.ai) eliminuje problem wyszukiwania poprzez kompresję bieżącej konwersacji do dziennika logów:
  - **Observer** — uruchamiany po przekroczeniu 30k tokenów w sesji. Otrzymuje dotychczasowy dziennik + najnowsze wiadomości. Generuje zaktualizowany dziennik. Przetworzone wiadomości zostają usunięte, dziennik trafia do system message.
  - **Reflector** — uruchamiany gdy dziennik przekracza 60k tokenów. Kompresuje jego treść.
- Dziennik przetrwa pojedynczą sesję — nieskompresowane wiadomości + najnowsza wersja dziennika przechodzą do nowej sesji.
- Wynik: ~95% na LongMemEval z gpt-5-mini. Koszt: stopniowa utrata szczegółów (kompresja stratna).
- **Wniosek:** zamiast rozwiązywać problemy wyszukiwania, można je wyeliminować przez odpowiednią architekturę pamięci.

## Baza wiedzy budowana dla agentów, nie podłączana do nich

- **Istniejąca KB** → agent **ma szansę** trafić na właściwe dokumenty. Wymaga chunkingu, RAG, rozwiązywania problemów wyszukiwania.
- **Budowana KB** → agent **wie**, gdzie są dokumenty. Struktura katalogów i wewnętrzne odnośniki w dokumentach prowadzą go krok po kroku.
- Agent nawiguje po dokumentach na 4 poziomach:
  - **Perspektywa** — spojrzenie z lotu ptaka (np. `ls`)
  - **Nawigacja** — przeszukiwanie nazw plików i treści (np. `grep`)
  - **Powiązania** — odnośniki między plikami (np. importy w kodzie, linki wewnętrzne)
  - **Szczegóły** — odczyt pełnej treści dokumentu
- Kod źródłowy jest naturalnym przykładem — agenci do kodowania skutecznie nawigują po projektach korzystając tylko ze ścieżki + podpowiedzi w plikach (nazwy funkcji, importy, ścieżki API).
- Ekspozycja kontekstu opiera się na **informacjach wewnątrz dokumentów**, nie na zewnętrznej strukturze. Wyświetlanie drzewa katalogów pomaga tylko gdy jest stałe i agent na nim operuje.

## Dekompozycja wieloetapowa — osobne sesje, współdzielona KB

- Złożone procesy dekomponuje się na osobne sesje agentów. Każdy agent czyta instrukcje i zapisuje wyniki do współdzielonego systemu plików.
- Przykład: Researcher (3 sesje gromadzenia) → Writer (synteza) → Sender (dystrybucja). Instrukcje określają zasady zapisu, agenci niezależnie produkują spójny wynik.
- **Kluczowa zasada bezpieczeństwa:** agent generujący treść do wysyłki powinien mieć uprawnienia tylko do tworzenia szkicu. Halucynacje + prompt injection = ryzyko przekierowania odbiorców.

## Grafy jako pamięć — hybrydowy Graph RAG

- Neo4j — grafy właściwości: wierzchołki (node) z etykietami i właściwościami + krawędzie (edge) z kierunkiem, typem i właściwościami. Każda krawędź łączy dokładnie dwa wierzchołki.
- **Kiedy grafy:** priorytetowe wielopoziomowe powiązania między informacjami rozproszonymi w wielu dokumentach + hybrydowe wyszukiwanie (pełnotekstowe, semantyczne, grafowe).
- **Kiedy nie:** wysoka złożoność, koszt utrzymania, długi czas odpowiedzi. Uzasadnione tylko w specyficznych przypadkach.

## Deep Research / Deep Action — iteracyjna eksploracja

- Przepływ: doprecyzowanie zapytania → parafraza → dekompozycja → wyszukiwanie → analiza → identyfikacja braków → kolejna iteracja → synteza.
- Możliwe wzbogacenie: wstępne przeszukiwanie do zawężenia kontekstu przed właściwym zapytaniem.
- Zastosowania wykraczają poza research — audyty, generowanie kodu, każdy proces wymagający eksploracji przez pogłębianie zapytań.

## Przykłady kodu — Graph RAG Agent

### Progresja przykładu

Przykład `02_03_graph_agents` realizuje pełny pipeline Graph RAG: indeksowanie dokumentów → ekstrakcja encji i relacji → hybrydowe wyszukiwanie → eksploracja grafu przez agenta z narzędziami.

### Architektura grafu (schema)

Trzy typy węzłów, trzy typy relacji:

| Węzeł | Klucz | Relacja | Opis |
|---|---|---|---|
| `Document` | `source` (unique) | `HAS_CHUNK` | Dokument źródłowy |
| `Chunk` | `source + chunkIndex` | `MENTIONS` | Fragment tekstu z embeddingiem |
| `Entity` | `name + type` (unique) | `RELATED_TO` | Encja wyekstrahowana przez LLM |

Indeksy: full-text na `Chunk.content` i `Entity.name`, wektorowe na `Chunk.embedding` i `Entity.embedding`.

### Pipeline indeksowania

```
chunk → embed chunks → extract entities/rels via LLM → embed entities → write to Neo4j
```

- Hash SHA256 pozwala pominąć niezmienione dokumenty. Zmiana = usunięcie starego + reindeksacja.
- Ekstrakcja (LLM, gpt-5-mini): 3-15 encji per chunk, z góry zdefiniowane typy encji (concept/person/technology/organization/technique/other) i relacji (relates_to/uses/part_of/created_by/influences/contrasts_with/example_of/depends_on).
- **Dedyplikacja globalna** — normalizacja nazw (Title Case + akronimy), singularizacja, merge po kluczu `lowercase+singularized`. Dłuższa nazwa i dłuższy opis wygrywają.

### Hybrydowe wyszukiwanie — RRF Fusion

Dwa niezależne strumienie: BM25 (full-text) + cosine (vector). Wyniki łączone przez **Reciprocal Rank Fusion**:

```
score(chunk) = Σ 1/(k + rank + 1)  // k=60, sumowane po obu strumieniach
```

Kluczowanie po `source::chunkIndex`. RRF degraduje graceful — brak vector search = tylko FTS.

### Narzędzia agenta — dwie kategorie

| Kategoria | Narzędzie | Rola |
|---|---|---|
| Retrieval | `search` | Hybrydowy retrieval + odkrywanie encji w chunkach |
| | `explore` | Sąsiedztwo encji w grafie |
| | `connect` | Najkrótsza ścieżka między encjami |
| | `cypher` | Read-only zapytania strukturalne |
| Curation | `learn` | Indeksowanie pliku lub tekstu |
| | `forget` | Usunięcie dokumentu z grafu |
| | `merge_entities` | Merge duplikatów encji |
| | `audit` | Diagnostyka: orfany, duplikaty, statystyki |

**Strategia retriewalowa** zdefiniowana w instrukcjach agenta: zawsze `search` → potem `explore` dla ciekawych encji → `connect` dla relacji między dwiema encjami → `cypher` tylko dla zapytań strukturalnych. Agent nigdy nie wyszukuje dla small talk.

### Pętla agenta

Maksymalnie 30 kroków: `user input → LLM → [tool calls → results → LLM]* → tekstowa odpowiedź`. Parallelne wykonanie tool calls (`Promise.all`). Historia konwersacji zachowana między zapytaniami w ramach jednej sesji REPL.

### Czego uczy ten przykład

1. **Graf jako pamięć agenta** — nie tylko chunki z embeddingami, ale sieć encji z relacjami umożliwiająca nawigację po powiązaniach.
2. **RRF jako prosty fuse** — bez wagi, bez trenowania, degraduje graceful gdy jeden strumień niedostępny.
3. **Ekstrakcja ustrukturyzowana** — LLM wydobywa encje/relacje z ograniczeniami (zdefiniowane enumy, brak self-refs, wymóg obecności source+target w liście encji).
4. **Narzędzia kuracji** — graf żyje: `learn`/`forget` do aktualizacji, `merge_entities`/`audit` do utrzymania jakości.
5. **Bezpieczeństwo zapytań** — `cypher` odrzuca write keywords, dodaje `LIMIT` gdy brak.
