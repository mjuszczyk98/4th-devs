# S01E03 — Projektowanie API dla efektywnej pracy z modelem

## Audyt API przed projektowaniem narzędzi

Zanim napiszesz pierwsze narzędzie, zweryfikuj:
- **Kompletność akcji** — czy API pozwala na tworzenie/odczyt/aktualizację/usuwanie zasobów
- **Identyfikacja zasobów** — model musi wiedzieć, że "etykieta Priorytet" = ID 42; mapowanie nazwa→ID leży po stronie narzędzia
- **Spójność kontraktu** — mieszanie `content`/`body`, `id`/`ID` w jednym API wymaga normalizacji
- **Kompletność odpowiedzi** — sam status `201 Created` bez zwrotki zasobu zmusza model do dodatkowego GET
- **Złożoność wieloetapowa** — jedno zadanie użytkownika wymagające 3+ wywołań API kandydatem na agregację w jedno narzędzie
- **Asynchroniczność** — polling i webhooki adresuj kodowo, nie zostawiaj ich agentowi
- **Rate limit** — kolejki i retry po stronie kodu, nie pętli agenta
- **Paginacja i wyszukiwanie** — agenci potrzebują precyzyjnych filtrów, nie "pobierz wszystko"

## Konsolidacja narzędzi

Zamiast dublować 1:1 endpointy API, grupuj po domenie działania. Przykład: 13 narzędzi systemu plików → 4:

| Narzędzie | Odpowiedzialność | Kluczowe mechaniki |
|-----------|-----------------|-------------------|
| `fs_search` | Przeszukiwanie plików i katalogów | Wzorzec nazwy + treść, drzewo katalogów, metadane |
| `fs_read` | Odczyt treści i listowanie | Tekst/binarne/wiele plików, zakres linii, automatyczne rozwiązywanie ścieżek |
| `fs_write` | Zapis i edycja | Checksum weryfikacji współbieżności, dryRun, historia zmian |
| `fs_manage` | Struktura katalogów, przenoszenie, usuwanie | Usuwanie ograniczone do pustych katalogów/pojedynczych plików, koncepcja "kosza" |

Cel: balans między liczbą schematów a przejrzystością. Zbyt mało narzędzi = wielkie schematy. Zbyt dużo = zaśmiecenie kontekstu.

## Projektowanie interfejsu z perspektywy LLM

Pięć wymiarów wpływających na kształt narzędzia:
1. **Struktura API** — możliwości i ograniczenia podstawowe
2. **Przejrzystość** — proste nazwy, czytelne opisy, minimalna liczba parametrów
3. **Złożoność akcji** — dopasowana do możliwości modelu (unikaj wieloetapowych orkiestracji w jednym handlerze)
4. **Ograniczenia modelu** — halucynacje, zapominanie kontekstu → walidacja i checksumy po stronie kodu
5. **Przypadki użycia** — adresują potrzeby użytkownika, nie możliwości API

### Dynamiczne odpowiedzi i wskazówki

Każda odpowiedź narzędzia (sukces i błąd) powinna nieść informację kontekstową:
- **Błąd → co zrobić**: "Plik został zmieniony. Przeczytaj go ponownie."
- **Status ograniczony → komunikat**: "Dokument chroniony przed zapisem."
- **Sukces → sugestia**: "Znaleziono 3 pliki. Wczytaj je przed edycją."
- **Nieprawidłowa wartość → dostępne opcje**: "Dostępne etykiety: X, Y, Z."
- **Korekta → informacja**: "Żądano linii 48–70, plik ma 59. Wczytano 48–59."

Mechanizmy bezpieczeństwa dla operacji destrukcyjnych: checksum (wykrywanie współbieżnych zmian), dryRun (podgląd), historia wersji (rollback bez udziału modelu).

## Model Context Protocol — architektura

### Trzy role

| Rola | Definicja | Przykład |
|------|-----------|----------|
| **Host** | Aplikacja tworząca połączenia | Claude Desktop, back-end agenta |
| **Client** | Połączenie zarządzane przez Host | Instancja MCP Client w kodzie |
| **Server** | Proces udostępniający capabilities | files-mcp, uploadthing MCP |

MCP nie jest alternatywą dla natywnych narzędzi — jest komplementarny. Z perspektywy modelu źródło narzędzia jest nieistotne: schematy MCP i natywne scalane są w jedną listę Function Calling.

### Komponenty MCP

| Komponent | Kierunek | Opis |
|-----------|----------|------|
| **Tools** | Model → Server | Akcje wywoływane przez LLM |
| **Resources** | Model ← Server | Dane statyczne/dynamiczne do odczytu |
| **Prompts** | Użytkownik → Server | Szablony wiadomości z parametrami |
| **Sampling** | Server → Model | Żądanie completion od modelu (wymaga akceptacji użytkownika) |
| **Elicitation** | Server → Użytkownik | Żądanie danych od użytkownika (formularz) |
| **Apps** | Model → UI | Interaktywne interfejsy w odpowiedzi agenta |

### Transporty

| Typ | Zastosowanie | Cechy |
|-----|-------------|-------|
| **STDIO** | Procesy lokalne, desktop | Jeden użytkownik na proces, dostęp do fs/narzędzi lokalnych |
| **Streamable HTTP** | Serwery zdalne, produkcja | Multi-user, OAuth 2.1, Cloudflare Workers / VPS |

### Routing wielu serwerów

Przy podłączeniu wielu serwerów MCP do jednego agenta, nazwy narzędzi prefiksowane nazwą serwera:

```
files__fs_read, files__fs_write, uploadthing__upload_files
```

Agent rozpakowuje prefix i kieruje wywołanie do odpowiedniego clienta. Host powinien też eliminować konflikty nazewnictwa między serwerami (np. `gmail__search` vs `resend__search`).

## Agent vs Workflow

| Wymiar | Workflow | Agent |
|--------|----------|-------|
| Kontrola | Pełna — zdefiniowane kroki | Ograniczona — model decyduje o kolejności |
| Elastyczność | Niska — nie zareaguje na niespodziewany błąd | Wysoka — model koryguje, weryfikuje, poprawia |
| Ryzyko | Niskie | Wyższe, ale z szansą na lepsze rezultaty |

Kryterium wyboru: jeśli proces **zawsze** musi wykonać się według ustalonych kroków → workflow. Jeśli kolejność i zakres akcji zależą od kontekstu → agent.

## Bezpieczeństwo

- Narzędzia dla systemów wewnętrznych: wysoka kontrola (znane procesy, ograniczony przepływ danych)
- Serwery MCP publiczne: brak widoczności otoczenia → programistyczne ograniczenia (blokady, limity, anonimizacja, walidacja)
- OAuth w MCP: odkrywanie metadanych → PKCE → wymiana kodu na tokeny RS → szyfrowanie → auto-refresh
- Modele lokalne (Qwen, GLM) jako test quality — jeśli mały model obsługuje narzędzia poprawnie, interfejs jest dobrze zaprojektowany

## Budowa serwerów MCP — proces

1. Pobierz szablon (np. Streamable MCP Template)
2. Utwórz `API.md` z dokumentacją integracji
3. Poproś AI o przeczytanie README + manual.md szablonu
4. AI sugeruje listę narzędzi → zastosuj zasady konsolidacji
5. Zaprojektuj input/output z perspektywy LLM (hints, walidacja, sugestie)
6. Zaimplementuj, zweryfikuj, usuń nieużywane elementy szablonu

Publikacja: VPS + nginx (jeden endpoint `/mcp`) lub Cloudflare Workers (KV + wrangler deploy). Format MCPB pakietuje serwer STDIO w jeden plik z auto-konfiguracją.

## Uzupełnienia

### Sampling = delegacja dostępu do AI

Sampling pozwala serwerowi MCP korzystać z LLM **bez własnych kluczy API** — to klient (Host) posiada relację z providereem AI. Serwer wysyła żądanie `sampling/createMessage` z komunikatami i ograniczeniami, klient realizuje completion i zwraca wynik. Ten sam mechanizm działa dla elicitation — serwer prosi klienta o dane od użytkownika bez wiedzy o jego interfejsie. Kluczowa korzyść: serwery MCP mogą leverage'ować AI bez konfiguracji providerów.

### Deklaratywna konfiguracja serwerów (`mcp.json`)

Produkcyjny wzorzec: konfiguracja połączeń MCP w jednym pliku JSON, wspierający mieszane transporty (stdio + HTTP) z definicją komend, argumentów, zmiennych środowiskowych i URL-i:

```json
{
  "mcpServers": {
    "files": { "transport": "stdio", "command": "npx", "args": ["tsx", "path/to/server.ts"], "env": { "FS_ROOT": "./workspace" } },
    "uploadthing": { "transport": "http", "url": "http://localhost:3000/mcp" }
  }
}
```

Klient wczytuje config, tworzy odpowiedni transport per serwer i prefixuje nazwy narzędzi nazwą serwera.

### Architektura agenta autonomicznego z MCP

Przykład translatora demonstruje wzorzec agenta operującego w tle: **źródło zdarzeń** (polling katalogów) → **klient MCP** (narzędzia do fs) → **pętla agenta** (LLM decyduje o kolejności akcji). Agent otrzymuje cel ("przetłumacz plik"), ale sam decyduje jak: wczytuje, dzieli, tłumaczy, weryfikuje, poprawia. Kluczowe elementy: limit kroków (max iterations jako safety guard), historia tool calls, niezależne zarządzanie stanem per plik (Set inProgress).

## Przykłady kodu — progresja wiedzy

### 1. `mcp_core` — pełny obieg protokołu MCP

Uczy: MCP to więcej niż tools. Serwer rejestruje tools, resources, prompts. Klient deklaruje capabilities (sampling, elicitation) i obsługuje żądania zwrotne od serwera.

Kluczowy wzorzec — serwer pyta klienta o completion (sampling) i potwierdzenie użytkownika (elicitation) w jednym handlerze:

```js
const confirmation = await extra.sendRequest(confirmationForm(text), elicitationResponseSchema);
if (confirmation.action !== "accept") return textResult("Cancelled");
const result = await extra.sendRequest(samplingMessage(text, style), samplingResponseSchema);
```

### 2. `mcp_native` — unifikacja narzędzi MCP i natywnych

Delta: narzędzia MCP i natywne JS są izomorficzne z perspektywy agenta. Wzorzec: jedna mapa handlerów, jedna lista schematów.

```js
const handlers = Object.fromEntries([
  ...mcpTools.map(t => [t.name, { execute: (args) => callMcpTool(client, t.name, args) }]),
  ...Object.entries(nativeHandlers).map(([name, fn]) => [name, { execute: fn }])
]);
const tools = [...mcpToolsToOpenAI(mcpTools), ...nativeTools];
```

### 3. `mcp_translator` — agent operujący na filesystem przez MCP

Delta: agent z pętlą agentic (max 80 kroków), konfiguracja serwera z pliku `mcp.json`, automatyzacja — file watcher wyzwala tłumaczenie. Konfiguracja STDIO transportu ze zmiennymi środowiskowymi:

```json
{ "mcpServers": { "files": { "command": "npx", "args": ["tsx", "../mcp/files-mcp/src/index.ts"],
  "env": { "FS_ROOT": "./workspace" } } } }
```

### 4. `upload_mcp` — wiele serwerów MCP + rozwiązywanie referencji plików

Delta: jednoczesne połączenie STDIO (files) i HTTP (uploadthing). Routing po prefiksie nazwy serwera. Wzorzec `{{file:path}}` — model deklaruje intencję odczytu pliku, resolver podmienia placeholder na base64 przed wywołaniem MCP.

```js
// Prefiksowanie: files__fs_read → serwer "files", narzędzie "fs_read"
const [serverName, ...toolParts] = name.split("__");
const client = clients[serverName];
const result = await client.callTool({ name: toolParts.join("__"), arguments: args });
```
