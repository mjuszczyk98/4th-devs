# S03E05 — Niedeterministyczna natura modeli jako przewaga

## Paradoks determinizmu w LLM

- LLM nie jest ani deterministyczny (dla tych samych danych wejściowych daje różne wyniki), ani wystarczająco losowy (często powtarza te same wzorce). Ograniczenie `temperature`/`top_p` nie rozwiązuje żadnej z tych stron.
- Zachowanie modelu jest silnie zdeterminowane przez **poprzedzającą treść**. Jeśli kontekst się nie zmienia, odpowiedzi będą bardzo podobne. Różnorodność wzrasta, gdy do gry wchodzi pamięć, otoczenie i stan sesji.
- Wniosek: zamiast walczyć z niedeterminizmem — **zaprojektować przestrzeń**, w której staje się on zaletą.

## Od poleceń do warunków — zmiana paradygmatu projektowania agentów

- Klasyczny agent: użytkownik → polecenie → narzędzie. Relacja 1:1 między intencją a akcją.
- Agent "świadomy": użytkownik → kontekst → model sam decyduje o akcjach. Brak bezpośrednich poleceń, model sam odkrywa luki i sięga po informacje.
- Kluczowa różnica: **oddelegowanie decyzji** o tym *kiedy* i *jak* działać, przy zachowaniu **wyznaczonej przestrzeni** (narzędzia, dane, metadane). Agent porusza się po wyznaczonym obszarze, ale sam wybiera ścieżkę.
- Ryzyko "utraty kontroli" jest pozorne — model operuje w ramach zdefiniowanych mechanik, narzędzi i danych. Przestrzeń jest większa, ale nie jest nieograniczona.

## Cztery obszary oddelegowane do modelu

| Obszar | Istota oddelegowania |
|---|---|
| **Proaktywność** | Model sam określa kiedy działać i jakiej wiedzy potrzebuje — na podstawie podejrzeń, nie poleceń |
| **Synteza** | Łączenie informacji bez sztywnych reguł "jeśli X to Y" — dynamiczne dopasowanie do sytuacji |
| **Wnioskowanie** | Narzędzia (`think`, `recall`) tworzą przestrzeń na refleksję; sama ich obecność wpływa na zachowanie modelu |
| **Dopasowanie** | Model decyduje o fokusie uwagi, stylu, formie — mechaniki wzmacniają kategorie zachowań, nie konkretne akcje |

## Architektura warstwowa agenta "świadomego"

- **Tożsamość i samowiedza:** instrukcje zarysowują przestrzeń (możliwości, ograniczenia), zaznaczają konieczność odróżniania wiedzy bazowej od zewnętrznego kontekstu.
- **Zdolności poznawcze:** instrukcje wpływające na zachowanie przez zadawanie pytań — nie przez reguły.
- **Inteligencja emocjonalna:** odczytywanie stanu rozmówcy, dopasowanie do celu (nawet gdy użytkownik go jeszcze nie zna).
- **Sposoby ekspresji:** dopasowanie stylu sterowane **metadanymi** dołączanymi do wiadomości użytkownika.
- **Mechaniki wzmacniające:** w metadanych i wynikach narzędzi — postawy i schematy myślowe, nie konkretne akcje.

## Sterowanie rozumowaniem przez zgeneralizowane instrukcje

- Instrukcje **kierują "snem" modelu** bez bezpośrednich poleceń. Przykłady: agent wie, że posiada osobowość (ale musi ją odkryć); wie, że informacje pochodzą z zewnątrz (nie z jego bazowej wiedzy); ma pozwolenie na luźne łączenie faktów.
- Cecha krytyczna: instrukcje są **zgeneralizowane** — opisują postawy i zasady, nie procedury. Modele słabiej radzą sobie z takim stopniem abstrakcji (nawet modele z rozwiniętym PE).
- Rola człowieka: projektowanie warunków powstawania zachowań, nie samych zachowań. LLM jest przydatny jako partner myślowy, ale generalizacja pozostaje zadaniem projektanta.

## Narzędzie `think` — sztuczna pauza na refleksję

- `think` nie pobiera danych — zmusza model do sformułowania pytań do samego siebie. Zwrot z inwestycji: model zauważa luki **zanim** odpowie.
- Wynik narzędzia wraca do kontekstu jako `function_call_output`, wpływając na kolejne kroki. To mechanizm typu "nudge" — obecność narzędzia zmienia zachowanie, nawet gdy nie zostanie wywołane w danej turze.

## Narzędzie `recall` i wzorzec scout-subagenta

- `recall` deleguje wyszukiwanie do **oddzielnego agenta** (scout), który ma dostęp do systemu plików przez MCP. Główny agent nie wie, jakie pliki istnieją — scout odkrywa je na podstawie celu (goal).
- Scout trwałą sesję — kolejne wywołania `recall` budują na tym, co już znalazł. Parametr `new_session` resetuje kontekst przy zmianie tematu.
- Kluczowy wzorzec architektoniczny: **oddzielenie wnioskowania (main agent) od eksploracji danych (scout)**. Scout jest celowo ograniczony w turach (max 8) i czyta tylko to, co bezpośrednio odpowiada na cel.

## Metadane temporalne jako kontekst situacyjny

- Każda wiadomość użytkownika jest opakowana w blok `<metadata>` z: `now_iso`, `weekday`, `local_time`, `timezone`, listą `recallable`, nudge ("think before you respond; recall when topic shifts").
- Metadane sterują uwagą modelu bez modyfikacji promptu systemowego — bezpieczne dla cache, zmieniają się co turę.

## Trzy strategie generowania interfejsów przez LLM

| Podejście | Mechanizm | Kontrola | Błędy |
|---|---|---|---|
| **Artifacts** | LLM generuje pełny HTML/CSS/JS | Niska — model ma pełną swobodę | Wyższe ryzyko, ale możliwości rosną z modelami |
| **Render (JSON spec)** | LLM generuje JSON z predefiniowanych komponentów | Średnia — ograniczony katalog, walidacja schematu | Niższe — server-side rendering deterministyczny |
| **MCP Apps** | LLM wywołuje narzędzie → serwer uruchamia gotowy UI | Wysoka — pełny determinant host, dwukierunkowa komunikacja | Minimalne — UI jest predefiniowane |

### Artifacts — pełna swoboda generatywna

- Model generuje `{title, html}` — pełny dokument HTML. Capability packs (Preact, Chart.js, D3, Tailwind, Zod, day.js, PapaParse) ładowane do iframe jako prelude.
- Edycja przez **search/replace** na HTML — model nie przepisuje całego dokumentu, tylko aplikuje operacje. Podobny wzorzec jak agenty do kodu.
- CSP izoluje artefakt: `default-src 'none'`, brak dostępu do sieci. Tailwind v3 celowo zamiast v4 — modele lepiej go znają.
- Kluczowe: informowanie modelu o dostępności packs w prompcie → model **sam wybiera** które wykorzystać.

### Render — deterministyczny rendering ze specyfikacji

- Model generuje **strukturalny JSON** (`{title, spec, state}`) zamiast HTML. Spec definiuje drzewo komponentów (root + elements), state dostarcza dane.
- Komponenty z katalogu: Stack, Grid, Card, Heading, Metric, LineChart, BarChart, Table, Alert, Accordion, Input, Select, itd.
- Walidacja: sprawdzanie typów komponentów, limit elementów (120), detekcja cykli, głębokość rekurencji. Nieznane komponenty → odrzucenie.
- **Wiązanie danych przez JSON Pointer:** `{"$state": "/kpis/0/value"}` — propsy resoluowane server-side. Model nie pisze logiki renderowania.
- Edycja przez regenerację całego dokumentu z kontekstem bieżącego stanu.

### MCP Apps — dwukierunkowa komunikacja przez protokół

- Model nie widzi kodu UI — wywołuje narzędzia (`manage_lists`, `get_lists_state`, `save_lists_state`), które są obsłużone przez serwer MCP z wbudowanym interfejsem (registerAppResource).
- UI jest predefiniowany, ale interaktywny — akcje użytkownika w przeglądarce zapisują stan do plików markdown, które agent może potem odczytać.
- Wzorzec: host zarządza UI, agent zarządza logiką. Komunikacja przez structuredContent — model otrzymuje wyniki, nie HTML.

## Wybór bibliotek pod kątem LLM, nie "najlepszych praktyk"

- Przy artifacts nie stosujemy najnowszych wersji bibliotek — wybieramy te, które modele **najlepiej znają** (Tailwind v3, nie v4).
- Capability packs rozdzielone na kategorie — model prosi o potrzebne, nie ładuje wszystkiego. Katalog packs jest jawny w prompcie systemowym.

## Progresja wzorców w przykładach

1. **Awareness** — agent z dwoma narzędziami metapoznawczymi (think, recall), scout sub-agent, template YAML+MD. Brak narzędzi operacyjnych — model sam decyduje o potrzebie kontekstu.
2. **Artifacts** — agent generujący pełny HTML z capability packs. Edit przez search/replace (delta, nie regeneracja). Live preview przez WebSocket.
3. **Render** — agent generujący JSON spec z katalogu komponentów. Server-side rendering deterministyczny. Stan zapisywalny i odczytywalny.
4. **MCP Apps** — agent przez protokół MCP steruje gotowym UI. Model izolowany od kodu interfejsu. Dwukierunkowa synchronizacja stanu.

Każdy krok ogranicza niedeterminizm w inny sposób: awareness — przez generalizację instrukcji, artifacts — przez capability packs, render — przez katalog komponentów i walidację, apps — przez predefiniowany UI.

## Uzupełnienia

### Teoretyczne kotwice

- **"Śniące maszyny" (Karpathy):** LLM jako maszyna snująca — wszystko jest "halucynacją", która czasem nam sprzyja. To przesuwa perspektywę z "eliminuj halucynacje" na "kieruj snem modelu".
- **Cognitive Architectures for Language Agents** (arxiv 2309.02427): praca wprowadzająca pojęcie architektury kognitywnej dla agentów językowych. Przejście od "spełniania założeń" do "stwarzania warunków" powstawania zachowań.
- **Theory of Mind** (arxiv 2505.00026): modele od GPT-4 demonstrują rozwinięte zdolności wnioskowania o stanach mentalnych innych — to podstawa mechanik inteligencji emocjonalnej agenta awareness.

### Ograniczenia modeli przy generalizacji instrukcji

- Nawet modele z rozwiniętymi umiejętnościami Prompt Engineeringu (np. Claude Opus 4.6) **zawodzą przy silnej generalizacji** instrukcji agenta awareness — trudno im wyjść poza konkretne reguły. Są za to dobrym partnerem w procesie projektowania takich instrukcji.
- Konkluzja: rola człowieka jako projektanta warunków (nie zachowań) pozostaje kluczowa.

### Trzytypowy model pamięci (workspace structure)

Workspace agenta awareness dzieli pamięć na trzy kategorie, analogicznie do kognitywnych systemów pamięci:

| Kategoria | Lokalizacja | Zawartość |
|---|---|---|
| **Epizodyczna** | `memory/episodic/` | Oznaczone czasem snapshoty interakcji |
| **Faktyczna** | `memory/factual/` | Trwałe fakty persystentne między sesjami |
| **Proceduralna** | `memory/procedural/` | Reguły behawioralne wyuczone w czasie |

Ponadto `profile/user/` (identity, preferences, important-dates) oraz `profile/agent/` (persona) stanowią warstwę tożsamości. Workspace Index (`system/index.md`) to mapa nawigacyjna dla scouta — agent główny nigdy nie widzi struktury plików bezpośrednio.

### External signal enrichment — wzorzec automatycznego wzbogacania kontekstu

- Scout agent integruje **pogodę z Open-Meteo API** — gdy cel (goal) lub wiadomość użytkownika zawiera słowa kluczowe typu "weather", "outside", "tonight", scout automatycznie pobiera pogodę na podstawie lokalizacji z `profile/user/identity.md`.
- Wzorzec: **scout nie tylko czyta pliki, ale też wzbogaca kontekst o sygnały z zewnętrznych API**, jeśli cel tego wymaga. Wynik wstrzykiwany jest jako `Weather hint` do kontekstu scouta.

### Szablony agentów z YAML frontmatter

- Szablony agentów (`.agent.md`) używają **gray-matter** — YAML frontmatter definiuje `name`, `model`, `tools`, a treść markdown jest `systemPrompt`. Wzorzec: agent to plik markdown, nie kod.
- Narzędzia deklarowane we frontmatter są **resolve'owane** — tylko wymienione w szablonie trafiają do modelu. Dla awareness: `think` + `recall`. Dla scout: brak narzędzi deklaratywnych (dostaje narzędzia MCP dynamicznie).

### Kluczowe parametry konfiguracyjne

- Agent awareness: **GPT-5.2**, `reasoning_effort: high`, `parallel_tool_calls: true`, `maxTurns: 10`.
- Scout: max **8 tur**, trwała sesja z resetem przez `new_session`.
- Artifacts/Render: `reasoning_effort` konfigurowalny przez env.

### Walidacja outputu LLM przez Zod (render pipeline)

- Render pipeline waliduje JSON od modelu przez `z.object()` schema (title, spec, state). Niezgodne payloady są odrzucane.
- Dodatkowo: ekstrakcja JSON z markdown code fences (`extractJsonCandidates`) — model czasem zwraca JSON w blokach ```json```, pipeline radzi sobie z oboma formatami.
- **Fallback document:** gdy API key brakuje, render generuje lokalny dokument z syntetycznymi danymi — system degraduje gracefully, nie crashuje.

### MCP Apps — szczegóły implementacyjne

- `registerAppTool` i `registerAppResource` z `@modelcontextprotocol/ext-apps/server` — rozszerzenie SDK do rejestracji narzędzi połączonych z UI.
- Pole `_meta.ui.resourceUri` w definicji narzędzia łączy tool z interfejsem graficznym. `visibility: ['app']` oznacza tool dostępny tylko z poziomu UI, nie z CLI agenta.
- **structuredContent**: narzędzia MCP Apps zwracają `structuredContent` (dane dla UI) **obok** `content.text` (dla agenta). Agent widzi tekst, UI widzi struktę.
- `registerAppResource` rejestruje HTML jako resource z `RESOURCE_MIME_TYPE`, UI wczytywane przez `prefersBorder: true`.

### Observability — trace JSONL

- `responses-loop` zapisuje trace do `workspace/traces/{date}.jsonl` — każdy request, response, tool result z timestampem, modelem, statusem i preview.
- Wzorzec: pełna trasa decyzji agenta do debugowania i analizy zachowania. Trace label odróżnia sesje agenta (`agent`) od scouta (`scout`).

### ArtifactKit — client-side state bridge

- Capability pack `core` wstrzykuje `window.ArtifactKit` z `createStore` (reaktywne store z subscribe), `q` i `qa` (querySelector helpers). Pozwala artefaktom zarządzać stanem bez frameworka.
- Modele mogą generować interaktywne UI używając ArtifactKit do lokalnego zarządzania stanem w izolowanym iframe.
